Performance: resmon, threads, profiling

A FiveM server has a 16.6 ms tick budget at 60 Hz. If your script eats 6 ms of it because of a runaway while true do Wait(0), the whole server stutters. This lesson teaches you to read resmon, pick the right thread API, and tune Wait values like a sound engineer tunes a mixer.

Before you start

Tick these before you open resmon.

Mental model: a thread is a waiter doing rounds

A waiter walks the floor every X seconds. Wait(0) means they sprint, never sit. Wait(1000) means they check tables once a second. The kitchen has a budget for waiter steps; if every waiter sprints, the kitchen melts.

Threads are the same. The default game loop runs at 60 Hz (16.6 ms a frame). Wait(0) on the client means "every frame", on the server means "every tick" (~5 ms). If a thread does work that costs 4 ms on every frame, that thread alone burns 24 percent of your tick budget.

Read this in resmon

Type resmon in the F8 console (client) or server console. The number you care about is the per-resource ms in the right column. Anything stable above 1.0 ms in idle conditions deserves a look. Anything above 5 ms is a fire.

Step 1

Read resmon: the only number that matters

Outcome: you can name the top-3 ms hogs on a server in 30 seconds.

A hot loop with Wait(0) on every frame and even mild work usually shows 1.5 to 4 ms. A pure event-driven script with no per-frame thread shows 0.00 to 0.02 ms. The gap between those two is the entire performance discipline.

Step 2

Pick the right Wait value

Outcome: you stop using Wait(0) when Wait(500) would do.

The 500 ms rule

If a player is more than 10 metres from any of your interaction props, sleep for at least 500 ms. They cannot interact anyway. This single change usually drops a script from 3 ms to 0.05 ms.

Recap 1

Halve a script's cost

Pick any script in your server with a while true do Wait(0). Change it to a tiered Wait based on distance to the nearest interest point. Restart the resource. Compare resmon before and after. Aim for at least 50 percent reduction.

Step 3

CreateThread vs CreateThreadNow

Outcome: you know which one to use at script start.

99 percent of the time you want CreateThread. CreateThreadNow exists for resource-startup ordering when one piece must run before onResourceStart finishes.

Step 4

Profile a function with GetGameTimer

Outcome: you know which line of code is slow, not just which resource.

Pro tip

If a function is under 1 ms, do not optimise it. Spend the hour on the 6 ms function instead. The distribution of cost is always 80/20: one or two functions account for most of the script's resmon footprint.

Recap 2

Find the slow line

Wrap three suspected hot functions in profile(). Restart the resource. Trigger the relevant code paths. Read the console output and identify the slowest. Confirm the resmon ms drop matches the function ms you saved.

Step 5

Profile under real load: production resmon, thread cost, native catalog

Outcome: you can read a live resmon graph at 64 players, name the exact thread that costs 4 ms, and rewrite it without guessing.

5.1 · Read the resmon graph in production

The Time column is an average over the last second. The number you watch in production is not the average, it is the trend arrow and the memory column. A resource sitting at 0.4 ms with a flat memory line is healthy. A resource sitting at 0.4 ms with memory rising 200 KB every minute is leaking and will kill the server in 36 hours.

The trend arrow next to Time matters more than the average at 64 players. A 0.6 ms resource that is climbing as players join means the cost scales with player count. That is a structural problem, usually a per-player thread that never dies, or an event handler that allocates a closure per call.

5.2 · Decompose thread cost: Wait(0) vs Wait(500) vs tiered

Every Wait(0) thread runs ~60 times per second on the client and ~20 times per second on the server. Every Wait(500) thread runs twice per second. The CPU ratio between them is roughly 30× on the client and 10× on the server. That is the entire reason "use tiered Wait" works.

Wait floor, not Wait ceiling

Wait(ms) guarantees at least ms milliseconds, not exactly ms. Under load the actual delay is longer. Never write logic that assumes Wait(100) means "exactly 100 ms", use GetGameTimer() deltas if you need real elapsed time.

5.3 · Native cost catalog: hoist locals, cache pools

Natives are not free. GetGamePool('CVehicle') walks the engine entity list and costs ~1 ms on a busy map. GetClosestVehicle uses a spatial index and costs <0.05 ms. Calling a native via the global table is slower than calling it via a hoisted local because Lua resolves the global on every call.

Step 6

The server-side patterns: batch writes, state bags, targeted events

Outcome: you stop hammering the database and the network with work that does not need to happen on every tiny change.

Everything above this point lives on the client and is mostly about Wait frequency. The server has its own way to burn a budget, and it usually has nothing to do with threads. It is two things: writing to the database too often, and shouting events at people who do not care. Here are the named patterns that fix both.

6.1 · Batch DB writes instead of one query per change

Imagine players earn coins. The naive version writes to MySQL on every single coin. If 50 players each earn 10 coins a minute, that is 500 tiny UPDATE queries a minute, just for "+1". Your database is doing 500 round-trips an hour to save numbers that did not need saving that often.

The fix is the same trick a notebook uses: jot the changes down in memory, then copy them all into the ledger once in a while.

This trades safety for speed, so pick your data carefully

If the server crashes between flushes, a player loses up to 30 seconds of coins. That is fine for coins, XP, or playtime. It is not fine for real cash or item purchases. For anything a player would be furious to lose, write it the instant it happens. Batch the stuff nobody will rage-quit over.

6.2 · State bags beat firing net events on repeat

Say you sync fuel or stress to clients. The lazy way is to fire a net event many times a second so everyone stays up to date. That floods the network. FiveM has a purpose-built tool for "this value changes a lot and other people need to see it": state bags.

Think of a state bag as a sticky note on an entity or player. You change the note once, and the game quietly shows the new note to everyone who needs it. No event spam, and it cleans itself up when the player disconnects.

Reach for state bags for anything continuous: fuel, stress, ownership flags, a "is this door locked" boolean. You set, they read, the engine handles the rest.

6.3 · Send events to the people who care, not to everyone

TriggerClientEvent('thing', -1, data) sends to every connected player. On a full server that is 128 copies of the same message. Most events only matter to one person or to a handful of people standing nearby.

Save the -1 broadcast for things that are truly global

A weather change or a server-wide announcement earns a -1 broadcast. A shop update, a robbery alert, a "you got paid" toast does not. Every needless -1 multiplies your network traffic by the player count for zero benefit.

6.4 · Only send to NUI when the value actually changed

HUDs are a classic budget leak. A health bar does not need a fresh message every frame, it needs one when the number changes. Pushing JSON into the browser layer (CEF) on every tick is pure waste, because serializing and shipping the same "100 HP" 60 times a second helps nobody.

Compare the new value to the last value you sent. If they match, do nothing. This one habit can take a HUD from a steady background cost down to almost nothing, because health, hunger, and thirst sit still most of the time.

Recap 4

Cut the server's hidden costs

Find one script that writes to the database on a value that changes often (coins, XP, playtime). Convert it to an in-memory tally with a 30-second flush, plus a playerDropped flush so nobody loses progress on logout. Then scan the same resource for any TriggerClientEvent(name, -1, ...) and ask, one by one, whether every player truly needs it. Narrow the ones that do not.

Pro tip: profiler record

For the rare case where resmon shows the cost but not the line, run profiler record 60 in the F8 console, reproduce the load, then profiler save trace.json. Open it in Perfetto. Every Lua call is a flame-graph block. The widest block is your problem.

Recap 3

Senior depth: performance audit

You can read a resmon graph and decide which signal matters: median is steady-state, the spike row is your incident. You know that Wait(0) is a per-frame budget claim, and you know which natives are cheap (GetGameTimer) vs expensive (GetEntityCoords on a vehicle pool). When a server tanks under 64 players, you start with profiler record, not guesses.

Common mistakes

while true do with no Wait

The loop never yields. The whole resource hangs. FiveM kills the thread after a few seconds with "script timeout exceeded". Always have a Wait inside any infinite loop.

Using Wait(0) for things that do not need a frame

Anything UI-driven (3D text, markers, progress bars) might need Wait(0). Anything check-driven (is the player still in the zone, has the timer expired) is fine at 250 ms or 500 ms.

Iterating all entities every frame

GetGamePool('CVehicle') on the client costs ~1 ms per call. Cache it and refresh every few seconds, or filter by distance first using GetClosestVehicle.

String concatenation inside hot loops

('a' .. b .. 'c') on every frame allocates. Pre-build the string outside the loop or use string.format once when the value actually changes.

Reading a config table on every iteration

If the config never changes, hoist the lookup out of the loop into a local. Locals are faster than globals; globals are faster than nested table lookups.

What you learned

  • resmon is the single number that tells you whether a script is healthy.
  • Wait(0) is the default cause of slow servers; tier it by distance or relevance.
  • CreateThread for almost everything; CreateThreadNow only for boot-order edge cases.
  • GetGameTimer deltas pinpoint the slow function inside a slow resource.
  • Optimise the 1 hog, not the 20 cheap functions; cost is distributed 80/20.

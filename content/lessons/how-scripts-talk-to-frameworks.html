How scripts talk to frameworks, exports and callbacks

Your custom script needs money, job, and inventory data from the framework. Learn exports and callbacks — the two patterns that let scripts share data without copying code.

You'll ship
A resource that reads a player's job from QB-Core or ESX via exports, and a server callback that returns custom data from your own resource to any other script.
Time
About 35 minutes.
You'll learn
How exports expose functions across resources, how server callbacks work for async data, and why framework wrappers are optional but convenient.
Prereqs
A QB-Core or ESX server running, or the willingness to simulate one with a mock export.
In 3 bullets

You will connect your script to a framework so that:

  • Your /heal command checks if the player has the ambulance job before allowing it.
  • Your resource provides a GetPlayerData export that any other resource can call.
  • You use a server callback to fetch async data from the database and return it to the caller.

Lesson 08: How scripts talk to frameworks, exports and callbacks

Start here

What this is
The two patterns that let resources share code: exports (synchronous) and callbacks (asynchronous).
Example: exports['qb-core']:GetCoreObject() returns the framework object in one call.
Where to find it
Exports are declared in fxmanifest.lua and called with the exports global.
Example: exports['resource_name'].functionName(args).
How to use it
Call a framework export to get data. Provide your own exports to give data to others.
Example: your heal script calls the framework's GetPlayer, then checks .job.name.
What to do now
Call one framework export, then create one of your own.
Example: get a player's job, then export a GetPlayerHealth function.

Outcome

You will type /heal and the command will first ask the framework for your job, then either heal you (if ambulance) or reject you (if not), all without copying any framework code into your resource.

Why it matters

Copying the framework's GetPlayer function into your script means it breaks when the framework updates. Calling it via export means the framework can change its internals and your script still works. Exports are contracts. Callbacks extend that contract to async operations like database reads.

Vocabulary

Export
A function one resource makes available to all other resources. Always synchronous — the caller waits for the return value.
Example: exports['qb-core']:GetCoreObject() returns immediately.
Server callback
An async request from client to server that returns a value. The client waits without blocking the game.
Example: TriggerServerCallback to ask the server for database data.
Framework wrapper
A thin layer in your resource that translates framework calls into your own API, so switching frameworks changes one file.
Example: local function GetPlayer(src) wraps exports['qb-core']:GetCoreObject().Functions.GetPlayer.

Common failures

  • No such export GetCoreObject in resource qb-core — qb-core is not running or is named differently on your server. Check the resource name with status in the console.
  • attempt to index a nil value (field 'job') — GetPlayer(src) returned nil. The player's source does not match a loaded player. Check if the player is fully loaded before accessing data.
  • exports: not a function — you used exports.GetPlayerHealth() instead of exports['qu_framework_bridge']:GetPlayerHealth(). The bracket syntax is required for resource names with underscores.
  • Callback times out — the server-side callback registration uses a different event name than the client is calling. Must match exactly, including the colon or underscore separator.

Refactor

Extract the framework wrapper into a separate bridge.lua file so switching from QB-Core to ESX changes exactly one file, and your server.lua stays untouched.

Cheat sheet

Export and callback patterns

  • Declare: exports('MyFunction', MyFunction) at the bottom of your script.
  • Call: exports['resource']:MyFunction(args) — note the colon, not a dot.
  • Server callback: Server registers, client calls, server responds. Async-safe for database reads.

The big idea, in one minute

Your server has one framework and thirty custom scripts. The framework owns the player data: money, job, inventory, gang. Every custom script needs to read that data. If each script copied the framework's internal code, updating the framework would break thirty scripts. Exports solve this: the framework says "call this function, I promise it will always return a player table". Your script calls that function. The framework can change its internals and your script still works because the contract is the same.

Think of an ATM.

You do not walk into the bank vault to get your money. You insert your card into the ATM, punch in a PIN, and cash comes out. The ATM is the export. The vault is the framework's internals. You do not need to know how the vault is organised. You just need the ATM to keep working. Similarly, your script just needs GetPlayer(src) to keep returning a player table. It does not matter how the framework builds that table.

Step 1

Create the resource and manifest

The manifest declares that this resource depends on the framework.

  1. Create resources/qu_framework_bridge/.
  2. Create fxmanifest.lua: 
    fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'
client_script 'client.lua'

dependencies {
    'qb-core'  -- or 'es_extended' for ESX
}
  3. The dependencies block ensures your resource only starts after the framework is ready.
Step 2

Write the framework wrapper

One small function that calls the framework. Every other function in your script uses this wrapper.

  1. In server.lua, write the GetPlayer wrapper from the Minimal Example.
  2. For QB-Core: exports['qb-core']:GetCoreObject().Functions.GetPlayer(src).
  3. For ESX: exports['es_extended']:getSharedObject().GetPlayerFromId(src).
  4. If your framework is different, adjust this one function. Everything else stays the same.
Step 3

Write the job-restricted heal command

The command reads the player's job from the framework and makes a decision.

  1. Paste the /heal_job command from the Minimal Example.
  2. The allowed table is a whitelist of jobs. Add your server's medic job name if it is different.
  3. The client event qu:bridge:doHeal needs a matching client handler. Add it to client.lua: 
    RegisterNetEvent('qu:bridge:doHeal', function()
    local ped = PlayerPedId()
    SetEntityHealth(ped, GetEntityMaxHealth(ped))
end)
  4. Save both files.
Step 4

Provide your own export

Your resource is now a provider, not just a consumer. Any script can call your function.

  1. Add the GetPlayerHealth function and exports('GetPlayerHealth', GetPlayerHealth) line to server.lua.
  2. Create a second test resource called qu_export_test with only a server.lua.
  3. In that server.lua, add: 
    AddEventHandler('onResourceStart', function(res)
    if res == 'qu_export_test' then
        local health = exports['qu_framework_bridge']:GetPlayerHealth(1)
        print('[qu_export_test] Player 1 health is: ' .. health)
    end
end)
  4. Ensure both resources. The test resource prints the health value from your resource.
Recap

Three tests, in order

Run each test and confirm the framework communication works in both directions.

  1. Test 1, framework import.
    • Action: type /heal_job as a civilian job.
    • Expected: chat says "Only ambulance staff can /heal_job."
    • How to verify: the framework's GetPlayer returned your job name, and the command rejected you.
  2. Test 2, allowed job.
    • Action: set your job to ambulance (via admin command), type /heal_job.
    • Expected: your health fills up. No rejection message.
    • How to verify: the framework returned 'ambulance', which is in the allowed table.
  3. Test 3, cross-resource export.
    • Action: start qu_export_test.
    • Expected: its console log prints a health value from your resource.
    • How to verify: the export crossed resource boundaries without any event wiring.
Why use a wrapper function instead of calling exports directly everywhere?

If you write exports['qb-core']:GetCoreObject().Functions.GetPlayer(src) in ten places and switch to ESX, you must edit ten places. With a wrapper function like local GetPlayer = ..., you edit one place. Also, wrappers let you add nil-checks, logging, and fallback values in one spot.

Two worlds, one resource

This is one of the first big FiveM ideas that confuses beginners. Your resource folder looks like one thing, but it actually runs as two separate programs: one on the server and one on each player's PC. They do not share variables. They do not secretly sync. Once this clicks, a lot of FiveM stops feeling weird.

How to read this lesson

This is one idea wearing many examples: your resource folder looks like one project, but it runs as two separate programs. If that sentence is clear, the rest of the page becomes much easier to follow.

The mental model: two runtimes, no shared memory

A FiveM resource is one folder on disk. But when the resource starts, the code inside gets split into two separate Lua programs:

  • The server program: one process, running on the host machine, shared by every connected player.
  • The client program: one process per player, running on each player's PC.

Both run Lua. Both have their own variables, functions, and memory. Neither side can read the other side's variables. If you write local x = 5 in server.lua, the client does not know that variable exists. If the client writes local ped = PlayerPedId(), the server cannot see it. If data needs to cross the wall, you must send it on purpose.

There is also not just one client. There is one client per player. If 30 players are online, your client code is running 30 separate times, once on each PC. But there is still only one server process. Server = one. Clients = many.

The building analogy

Simple version: every player has their own copy of the game running on their PC. The server is the central rule keeper. The server does not see your screen the way you do. It only knows what it tracks and what gets sent to it. That is why your local changes mean nothing to everyone else until the server is involved.

Step 1

Name the split precisely

Outcome: you can state, in one sentence each, what the server is and what a client is, and why there are many of one and only one of the other.

Say these out loud. Don't skip. Saying them rewires the intuition.

  • The server is a single Lua process running on the host machine. There's one of it. Every connected player talks to the same server.
  • The client is a Lua process running on a player's PC. There's one per player. Each copy has its own variables and doesn't know the others exist.
  • Shared scripts are the same file loaded into both runtimes. Each runtime gets its own copy. They still can't read each other, they happen to contain the same text.

Step 2

Understand why the split exists

Outcome: you can answer "why not one big program?" without guessing.

There are three main reasons for the split:

  • Rendering. Every player needs their own screen, camera, input, and frame updates. That belongs on their machine.
  • Trust. Your server is yours. A player's PC is theirs. Important rules like money, items, and permissions belong on the server.
  • Scale. Per-player visual work happens on each player's machine. Shared rule and data work happens once on the server.
Reflex to build

Before you write code, ask: whose job is this? A key press? Client. Saving money to the database? Server. Drawing something on the screen? Client. Counting all connected players? Server.

Step 3

Memorise what runs where

Outcome: a map you can screenshot and keep open while you code the rest of the course.

This is the list that saves you the most time. Skim once, then re-read slowly.

  • Lives on the client: the player's ped, camera, HUD, NUI (browser-style UI), keypresses, screen effects, vehicle physics for vehicles they host, animations, on-screen text and markers.
  • Lives on the server: the roster of connected players, database queries, money and items and permissions, cross-player coordination, persistent state that survives a restart, authoritative timers and cooldowns.
  • Lives on both (shared scripts): pure Lua configs, enums, constants, helper math functions that don't touch the game world.

Notice what's not on any list: "the player's money on the client." Money is server state. The client can display a number, but the number itself belongs to the server. If the two disagree, the server wins, every time. The client's copy is a screenshot, not the truth.

Here is the same idea as a side-by-side capability table. The most important row is the trust row: the client runs on a machine you do not control, so you treat everything it sends as a request, never as a fact.

A player owns their PC. They can open your client files, read them, and edit them before the game loads. That means a client value is never proof of anything. It is a request. The phrase to memorise: the client says what it wants to happen, and the server decides whether it is allowed to happen. We build on this hard in Lesson 12.

Step 4

Natives are side-specific: a native is a phone number

Outcome: you understand why some natives only work on client, some only on server, and what the error looks like when you get it wrong.

A native is a function FiveM provides (you'll meet hundreds of them in Lesson 11). Each native belongs to one side or the other. Calling it from the wrong side doesn't "sort of work", it errors. Two examples:

  • PlayerPedId(): client-only. It returns the ped this machine controls. Asking the server "what's the player's ped id?" makes no sense: the server doesn't have a screen, a camera, or a ped; it has a roster of player IDs.
  • GetPlayerPing(playerId): server-only. Ping is measured between the server and a specific connected client; you pass it a server-known player ID.

Think of a native as a phone number that only rings on one phone. Dialling it from the wrong phone gets you attempt to call a nil value or similar. The fix is never "call louder", it's calling from the right side.

Watch out

The FiveM native reference (you'll bookmark it next lesson) labels each native client, server, or sometimes both. When in doubt, check. Half the "why is this nil?" questions in the FiveM Discord are someone calling a client native from the server.

Step 6

Shared scripts: the one file that lives on both sides

Outcome: both consoles print the same value from the same shared file, and you can explain why that's not the same thing as "shared memory".

Add a print on each side. In server/main.lua, add print('[server] SharedConstant = ' .. SharedConstant). In client/main.lua, add the same line with [client]. Restart. Both consoles print the value.

Here's the subtle bit: this is not shared memory. The shared file gets loaded twice: once into the server runtime, once into each client runtime. Each runtime ends up with its own independent copy of SharedConstant. If the server changed its copy at runtime, the client copies would not move. They happen to start equal because they came from the same text file. That's all "shared" means.

Common misread

"Shared" in FiveM means "the file is loaded on both sides", not "the variables sync between sides". Use shared scripts for constants and pure functions. Never rely on them as a communication channel, they aren't one.

Recap 2

Explain it back

Close your eyes and answer these without peeking. If you stumble on any, re-read the section above before moving on.

If 25 players are connected, how many client runtimes are running?

25. One per player. Each has its own variables. There is still exactly one server runtime, shared by all.

Your server declares local money = 500. Can the client read it?

No. It's a server-local variable. The client has no access, no matter how long it waits. You'd have to send the value explicitly (events, Module 04).

A player edits their client script to set money = 999999. What happens to the server's number?

Nothing. The server's money is a different variable in a different process. The cheater has only lied to their own screen. That's why authority has to live server-side.

You call PlayerPedId() inside a server_script. What happens?

Error, the native doesn't exist on the server. It's client-only. The server has no ped.

A word about "who fired this event?"

You'll soon ask a fair question: if clients and servers can't see each other's variables, how does the server know which client asked for something? Good instinct. The answer, in one line: inside a server-side event handler, the server gives you a magic variable called source: the server ID of the client that triggered the event. It only exists inside those handlers, only on the server. On the client, source is nothing. We'll build with it in Lessons 13 and 14. For now, know the "who?" problem has a clean answer waiting.

Why the wall is a feature, not an annoyance

The wall feels like extra work at first. You wanted one program, you got two that cannot share a variable. But that wall is exactly what keeps your server safe. Walk through one example: a player buying a loaf of bread.

The one line to remember

Client asks, server decides. Any time you are about to write money, items, or rule checks on the client, stop: that belongs on the other side of the wall.

Mistakes this lesson exists to prevent

"I'll set money = 500 on the client to make them rich."

Your client's money variable is private to that one PC. The server's money, the database's money, and every other player's view of that player's money all stay unchanged. Worse: this is exactly what a cheater does, so the server has to ignore it anyway. Money is server state. Full stop.

"I'll read source on the client."

source is server-only, and only inside an event handler. On the client, it's nil. If you need to know "who am I?" on the client, that's a different question, you are the local player, and there's a native for that.

"I'll call PlayerPedId() on the server."

The server doesn't have a ped. It has a list of connected players by server ID. Anything to do with the ped, position, health, weapons, vehicle, has to be asked of the client or read via server-side natives that take a player ID.

"I'll store per-player state in a table on the client so the server can read it."

The server can't read it. Ever. If you want per-player state on the server, you store it on the server, keyed by player ID. If you want per-player state on the client, keep it on the client and remember there's a separate copy on every player's PC.

The reflex to build, "which side has the data I need?"

When in doubt, don't ask "how do I get this to work?", ask which side has the data I need, and which side needs to act on it? If both sides need something, the side that owns it sends to the side that uses it. Build that habit now and the next ten lessons write themselves.

What you learned

  • A FiveM resource runs as two separate Lua programs: one server process and one client process per connected player.
  • Variables declared on one side are completely invisible to the other -- there is no automatic sync.
  • Shared scripts load into both runtimes independently; each runtime gets its own copy of the data.
  • Natives belong to one side: calling a client-only native on the server (or vice versa) gives attempt to call a nil value.
  • The reflex to build: always ask "which side has the data I need?" before writing a line of code.

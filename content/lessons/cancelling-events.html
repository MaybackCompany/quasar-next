Cancelling events: CancelEvent() & WasEventCanceled()

Sometimes one handler needs to veto the rest. A spawn-protection script blocks the death-handler chain. An anti-cheat resource cancels playerConnecting for a banned UID before any other resource sees the join. CancelEvent() is the signal; WasEventCanceled() is how downstream handlers know.

Mental model: a relay race the captain can stop

Every event fire is a relay. Each registered handler is a runner who gets the baton in registration order. CancelEvent() is the captain raising a flag, the rest of the runners still receive the baton, but they can check the flag (WasEventCanceled()) and choose to walk it across the line instead of running.

The flag does not stop the chain. Every handler still fires. What changes is the signal: downstream handlers know "someone vetoed this" and can branch on that.

FiveM-specific note

For events the engine itself listens to, playerConnecting, weaponDamageEvent, entityCreating, onResourceStop, cancellation is meaningful: the engine treats a cancelled playerConnecting as "reject the join". For your own custom events, cancellation is just a convention you and your other handlers agree on.

Step 1

Cancel from inside a handler

Outcome: one handler raises the cancel flag; you watch it propagate.

Two natives, both globally available on client and server:

  • CancelEvent(), sets the cancel flag on the current event invocation.
  • WasEventCanceled(), returns true if any earlier handler in this invocation called CancelEvent().
What you should see in the console

Trigger with TriggerEvent('qu:demo:hit', 'spawn-protect'). Output:

[QUIET][hit][1] reason=spawn-protect canceled=false
[QUIET][hit][2] cancelled by spawn-protect
[QUIET][hit][3] skipping, already cancelled

Handler #3 still ran. The cancel flag did not skip it, the explicit WasEventCanceled() check did.

Step 3

The playerConnecting kick pattern

Outcome: you can reject a join from a banned UID before any gameplay resource sees the player.

playerConnecting is one of the rare events the FiveM engine itself observes. If you cancel it, the engine refuses the connection. The handler signature is (name, setKickReason, deferrals). Three ways to reject:

  1. setKickReason('reason'), sets the message shown to the player. Optional.
  2. CancelEvent(), tells the engine to drop the connection.
  3. deferrals.done('reason'), the modern, async-friendly way. Returning a reason here also cancels the join.
Pro tip

Prefer deferrals for any join check that needs IO (database lookup, HTTP call). CancelEvent() is fine for synchronous rules. Mixing both inside the same handler is allowed but rarely needed, pick one and stay consistent.

Common mistakes

Assuming CancelEvent() stops the chain

It does not. Every later handler still runs. The flag is a signal, not a break. You must check WasEventCanceled() in every handler that should opt out.

Cancelling a handler-only custom event and expecting a refund

For events you defined yourself, cancelling has no engine effect. The data you already wrote (DB rows, file changes) does not roll back. Treat CancelEvent() as a coordination flag, not a transaction.

Using CancelEvent() as security

If a malicious client triggers a server event, cancelling it inside one handler does not protect the others if they do not check the flag. Always validate source and permissions in each handler. See Lesson 12 (Trust & authority).

Mixing CancelEvent() and deferrals.done() inconsistently

Inside playerConnecting, calling CancelEvent() after deferrals.defer() can leave the connection in a stuck state on some builds. Pick one path: synchronous (CancelEvent) or asynchronous (deferrals). Do not interleave.

Forgetting to return after cancelling

Calling CancelEvent() does not exit your function. Code below it still executes in the same handler. Add an explicit return when you mean "we are done here".

Cheat sheet

CancelEvent()
Sets the cancel flag on the current event invocation. Does not stop the handler chain.
WasEventCanceled()
Returns true if any earlier handler called CancelEvent() in this invocation.
Engine-observed events
playerConnecting, weaponDamageEvent, entityCreating, onResourceStop: engine acts on the cancel flag.
Custom events
Cancellation is a convention. Every handler that should respect it must check WasEventCanceled().
Kick a player at join
setKickReason('msg') + CancelEvent() inside playerConnecting.
Async kick
deferrals.defer() then deferrals.done(reason) rejects the join after a DB / HTTP call.
Don't forget
Add return after CancelEvent() if you do not want code below to run.
Not security
Cancellation is not auth. Validate source and permissions in each handler.

What you learned

  • CancelEvent() raises a flag; WasEventCanceled() reads it, the chain itself never stops.
  • Engine-observed events (playerConnecting, weaponDamageEvent) treat the cancel flag as a real veto.
  • The playerConnecting kick pattern: setKickReason + CancelEvent() for sync; deferrals.done(reason) for async.
  • Cancel is a coordination signal, not a transaction, side effects already written stay written.
  • Always pair CancelEvent() with an explicit return when you mean "done here".

Tables: the one data structure to rule them all

In Lua, almost every collection of data is a table. Player lists, config files, inventories, job data, and many other things all use tables. This lesson teaches you how to read them, write them, loop through them, and avoid the classic beginner mistake: forgetting that tables are references, not copies.

Before you start

How to read this lesson

Tables only look big because they do many jobs. Read this page as three small ideas: list tables, named-field tables, then nested tables. Leave the reference-copy gotcha for last, because it only makes sense after the first three.

The mental model: one structure, two flavours

Other languages give you five different collection types: arrays, lists, hash maps, sets, dicts. Lua gives you one: the table. Every collection you'll ever build in FiveM is a table wearing a costume.

The costume depends on how you fill it. Fill it with values in a row, {'apple', 'banana', 'cherry'}: and it behaves like an array. Fill it with key-value pairs, { name = 'Alex', level = 12 }: and it behaves like a dictionary. Same structure. Two flavours. You pick by how you write the contents.

FiveM note

The player list on your server? A table. Every vehicle's config block? A table. Every ox_inventory item definition, every ESX job, every QBCore shared file, tables, top to bottom. Getting comfortable here unlocks reading any FiveM resource on GitHub. There is no lesson in this course more load-bearing than this one.

Mistake #1, Assuming 0-indexing

fruits[0] is nil in Lua, not the first element. Lua won't crash, it'll happily hand you nil and let your code break three lines later with a cryptic "attempt to index a nil value". If something's misbehaving and you came from JS or Python, check your indexes first.

Dot vs bracket, two ways to reach the same value

These two lines do the exact same thing:

Dot notation is shorter and the one you'll use most. Bracket notation is what you reach for when the key is stored in a variable, print(player[fieldName]): or when the key isn't a valid Lua identifier (contains a hyphen, starts with a digit, etc.).

Mistake #4, Dot when you should've used brackets

t['weird-key'] works. t.weird-key does not, Lua reads that as "the key weird of t, minus the variable key". If a key has a hyphen, a space, or starts with a number, you must use brackets. This is why you'll see Config['some-setting'] all over real FiveM code.

Step 4

Walking a dictionary with pairs

Outcome: you iterate a dictionary-style table and print every key-value pair.

ipairs only walks numbered keys 1, 2, 3… in order. It can't iterate a dictionary. For that you need pairs:

Restart. You'll see three lines, one per field. tostring(value) is there because alive is a boolean (true), and .. won't join a string to a boolean without a hand. tostring converts anything to text first.

Heads up, pairs doesn't promise order

pairs iterates every key in the table, but the order is not guaranteed. You might see name first today, level first tomorrow. If the order matters, use ipairs on a numeric array. If it doesn't, pairs is fine.

The simple rule for picking between them

  • ipairs: when your table is a list ({'a', 'b', 'c'}) and order matters.
  • pairs: when your table has named keys ({ name = 'Alex' }) and you want every entry.

A table can mix both styles, but keep them separate in your own code until the split-brain nature of tables feels natural. Mixed tables are a side quest, not the main path.

Mistake #2, ipairs on a table with holes

ipairs walks 1, 2, 3… and stops at the first nil it hits. If your table is {[1]='a', [3]='c'} (no index 2), ipairs prints only 'a' and exits silently, no error. If you've deleted items by setting them to nil, ipairs is lying to you. Use pairs, or better, use table.remove (coming up) which closes the gap.

Recap 1

Two flavours, one table

Before moving on, answer in your head, then check:

What does fruits[0] return in Lua?

nil. Lua arrays start at index 1. Index 0 is an empty slot unless you explicitly put something there.

Would you use ipairs or pairs to loop over { hp = 100, armor = 50, stamina = 80 }?

pairs. Those are named keys, not numeric positions. ipairs would find no key at index 1 and loop zero times.

What's the difference between player.name and player['name']?

Nothing, they do the same thing. Dot notation is shorter; bracket notation is what you use when the key is in a variable, or when the key has characters that aren't valid in an identifier (hyphens, spaces, leading digits).

Step 6

The reference gotcha: the one that burns everyone

Outcome: you see, with your own eyes, that local b = a does not copy a table, both names point to the same thing.

This is the one. Every Lua beginner meets this bug and loses half a day to it. Meet it now, on purpose, in 4 lines:

You might expect 100: after all, you only changed b. Restart and read the console:

a.hp is also 50. Why? Because b = a didn't copy the table, it gave the same table a second name. a and b are two labels on one box. Change the box through either label, and the change is visible through the other.

Mistake #3, Expecting a copy

This is the single most common source of "why is my data changing on its own?" bugs in FiveM. Passing a config table to a function and having the function mutate it? You mutated everyone's copy. If you need a true independent copy, you have to loop through and build a new table manually, there's no built-in clone in Lua. We'll meet one pattern for it in Module 03.

For now, remember the rule: numbers, strings, and booleans are copied when you assign them. Tables are not. When you pass a table around, you're passing a pointer to it.

Pro move

table.remove(t) with no index removes and returns the last element, Lua's answer to pop. table.insert(t, 1, x) with an index inserts at a position, Lua's answer to unshift. You now have push/pop/shift/unshift in two functions.

Common mistakes & what the console tells you

Error: attempt to index a nil value

You reached into a table that doesn't have the field you asked for. Usually caused by fruits[0] (1-indexed, remember), a typo in a field name (player.levl vs player.level), or a table that hasn't been built yet when the code runs.

The loop stops halfway through for no reason

You're using ipairs on a table with a gap, an index where the value is nil. ipairs stops silently at the first hole. Fix: use table.remove to delete items (it closes gaps), or switch to pairs if the table is a dictionary in disguise.

My data changed and I didn't touch it

The reference gotcha. You assigned local b = a and then mutated b, not realising a was the same table. Or you passed a config to a function and the function mutated it. Tables are shared by reference, if you need independence, you need to manually build a fresh table.

Error: '<name>' expected near '-'

You wrote player.weird-key with a hyphen. Dot notation only works with valid identifiers. Switch to bracket notation: player['weird-key'].

Lua gotchas: a beginner cheat to keep nearby

You have met the biggest table trap already (1-indexing in Step 1, the reference gotcha in Step 6). Here is the rest of the family in one place: the small differences that catch everyone arriving from another language. None of these throw a clear error, they just quietly misbehave, so it pays to recognise them on sight.

Arrays start at 1
The first item is list[1], not list[0]. Reaching list[0] hands you nil.
Example: fruits[1] is the first fruit, just like you saw in Step 1.
~= means "not equal"
Lua uses a tilde then equals. There is no != in Lua, typing it is a syntax error.
Example: if p.job ~= 'police' then ... end.
Only false and nil are falsy
Every other value is truthy, including 0 and the empty string ''.
Example: if 0 then ... end runs the block, which surprises people from JavaScript.
nil is "no value"
It is Lua's version of null. A missing table field reads back as nil, not an error, until you try to use it.
Example: player.score is nil if you never set score.
and / or, not && / ||
Lua spells its logic operators as plain words.
Example: if p.alive and p.level > 10 then ... end.
% is the remainder
Modulo gives what is left after dividing. Great for "every Nth time" checks.
Example: i % 2 == 0 is true for even index numbers.
Why p.alive and 'alive' or 'down' works

You used this line in the capstone without a full explanation, so here it is. Lua has no ?: ternary, so people fake one with and / or. The trick leans on two rules: and returns its second value when the first is truthy, and or returns the first truthy value it finds. So p.alive and 'alive' or 'down' reads as "if p.alive is true, hand back 'alive', otherwise hand back 'down'". Read it left to right and it says exactly what it does.

The or default-value pattern

The same idea gives you a clean way to set a fallback. local job = p.job or 'unemployed' means "use p.job, but if it is nil, use 'unemployed' instead". You will see this on the first line of countless FiveM functions, filling in a missing argument with a sensible default. It works because or skips past nil and returns the first real value.

What you learned

  • A table is Lua's only data structure -- it can act as an array (numbered list) or a dictionary (named fields)
  • Lua arrays start at index 1, not 0 -- this is the most common surprise for beginners from other languages
  • Use ipairs to loop through arrays in order and pairs to loop through dictionaries
  • table.insert adds items and table.remove removes them without leaving holes
  • Tables are passed by reference -- local b = a does not copy the table, both names point to the same data

Git localhost to VPS, move your scripts live

Take a script that works on your development machine and deploy it to a production VPS using Git — so you never drag-and-drop files over FTP again.

You'll ship
A Git workflow that pushes your local resource changes to a remote VPS in one command.
Time
About 45 minutes if you set up SSH keys for the first time.
You'll learn
How to initialise a Git repo inside your resources folder, connect it to a bare repo on your VPS, and push without typing a password.
Prereqs
A VPS running your FiveM server, and Git installed on both machines.
In 3 bullets

You will set up a Git pipeline that:

  • Lets you commit changes locally and push them to a bare repo on your VPS.
  • Uses a post-receive hook to auto-copy files into your live server directory.
  • Works over SSH with key authentication so you never type a password again.

Start here

What this is
A way to send code from your computer to a live server without FTP.
Example: git push production main deploys your scripts in seconds.
Where to find it
The commands run in your terminal and on the VPS SSH session.
Example: open two terminal windows, one local and one SSH.
How to use it
Commit locally, push to a remote named production, and a hook copies the files.
Example: after git push you refresh the resource in the server console.
What to do now
Set up one bare repo and push one Lua file through it.
Example: push a test.txt and see it appear in resources/ on the VPS.

Outcome

You will type git push production main on your local machine and see the changed files appear in your live FiveM resources folder within three seconds.

Why it matters

FTP drag-and-drop causes missing files, version confusion, and overwritten work. Git gives you history, rollback, and a single command to deploy. When something breaks at 2 AM you can git revert instead of guessing which file you changed.

Vocabulary

Remote
A named pointer to another Git repository, usually on another machine.
Example: git remote add production user@vps:/path/to/repo.git.
Bare repo
A Git repository with no working files, only the .git database. Used as a push target.
Example: git init --bare creates a folder that only stores commits.
Post-receive hook
A script that runs automatically after a push finishes.
Example: a hook that copies the latest files into resources/.

Setup

Local machine
Git installed, resources/ is a Git repo

Git installed, a bare repo at /home/user/fivem.git
Auth

Server path
/home/user/txAdmin/server-data/resources/

Minimal example

Expected output: After git push production main, the terminal shows [post-receive] deployed to resources/ and your files appear in the target folder.

Checklist

  1. SSH into your VPS and run git init --bare ~/fivem.git.
  2. Create the post-receive hook and make it executable with chmod +x.
  3. On your local machine, run git init inside your resources folder.
  4. Add the VPS remote: git remote add production user@vps:~/fivem.git.
  5. Commit a test file and push with git push production main.
  6. SSH back to the VPS and confirm the test file arrived in the target directory.

Common failures

  • Permission denied (publickey) — your SSH key is not on the VPS. Run ssh-copy-id user@vps to add it.
  • remote: fatal: not a git repository — you ran git init instead of git init --bare on the VPS. Delete and recreate.
  • Files appear but are owned by root — the hook runs as your user but the target folder is owned by someone else. chown the target to match.
  • Hook does not execute — the file is not executable. Run chmod +x hooks/post-receive inside the bare repo.

Refactor

Extend the post-receive hook to also run txAdmin restart or an ensure command so the server picks up changes without manual intervention.

Cheat sheet

Git deploy commands

  • git push production main — deploy everything from main branch to the VPS.
  • git push production main --force — overwrite the remote (dangerous, only when you know why).
  • ssh user@vps "ls resources/" — check what is on the VPS without opening a full session.

The big idea, in one minute

FiveM development is fast and iterative. You change one line, restart the resource, and test. Doing that over FTP means every change is: save, open FileZilla, drag, wait, restart. Doing it over Git means: save, git push, restart. The overhead drops from thirty seconds to two. Over a hundred changes per session that is the difference between staying in flow and losing it entirely.

Think of a mailroom with a pneumatic tube.

FTP is walking a letter to the post office every time you write one. Git is a pneumatic tube. You drop the letter into the tube on your desk, whoosh, it arrives at the mailroom. The tube is the SSH connection. The post-receive hook is the clerk at the other end who opens the canister and puts the letter in the right slot.

Step 1

Create the bare repo on your VPS

A bare repo has no working files — it only stores commits. That is exactly what we need as a push target.

  1. SSH into your VPS: ssh user@your-vps-ip.
  2. Navigate to your home directory: cd ~.
  3. Run: git init --bare fivem.git.
  4. This creates a folder ~/fivem.git/ with Git internals but no working tree.
Step 2

Write the post-receive hook

The hook runs after every push. It checks out the latest code into your live resources folder.

  1. On the VPS, navigate into the bare repo: cd ~/fivem.git/hooks.
  2. Create a file named post-receive: nano post-receive.
  3. Paste the script from the Minimal Example above. Update TARGET to point at your actual resources folder.
  4. Save and exit. Make it executable: chmod +x post-receive.
Step 3

Set up your local Git repo

Your local resources folder becomes a Git repository that pushes to the VPS.

  1. On your local machine, open a terminal in your FiveM resources folder.
  2. Run git init to create a new repository.
  3. Create a .gitignore file and add *.log and cache/ to it.
  4. Run git add -A to stage everything, then git commit -m "initial commit".
  5. Add the remote: git remote add production user@vps-ip:~/fivem.git.
Step 4

Push and verify

One push confirms the entire pipeline works.

  1. Run git push production main on your local machine.
  2. Watch the terminal output. You should see the hook message: [post-receive] deployed to resources/.
  3. SSH back to the VPS and run ls resources/. Your files should be there.
  4. If anything is missing, check the hook path and target folder permissions.
Recap

Two tests, in order

Run each test and confirm the result before considering the setup complete.

  1. Test 1, first push.
    • Action: change one line in any server.lua, commit, and push.
    • Expected: the hook output appears, and the file on the VPS has the new line.
    • How to verify: ssh user@vps "cat resources/my_resource/server.lua" shows your change.
  2. Test 2, restart without FTP.
    • Action: push a new resource folder, then type ensure my_new_resource in the server console.
    • Expected: the resource starts and shows its startup message.
    • How to verify: server console prints the resource name with no errors.
Why a bare repo instead of a normal one?

A normal repo has a working tree — actual files checked out on disk. If you push to a normal repo, the working tree does not update automatically and you get a confusing error. A bare repo has no working tree, so it accepts pushes cleanly. The post-receive hook then manually creates the working tree at your target path.

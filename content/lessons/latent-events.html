Latent events: TriggerLatentServerEvent & TriggerLatentClientEvent

A regular TriggerServerEvent blocks the network channel until the whole payload arrives. For small messages that is fine. For big tables, image buffers, or character export blobs, blocking the channel can time the player out. Latent events stream the payload at a controlled rate so the channel stays open.

Why latent events exist

FiveM's reliable event channel is a single ordered stream per direction. TriggerServerEvent writes its entire payload into that stream and the receiver does not advance until every byte arrives. If your payload is 800 KB and the player is on a slow link, every other event behind it stalls. The client may even time out.

Latent events bypass that by chunking the payload at a configurable byte rate. Other events keep flowing. The receiver assembles the chunks and fires the handler once, exactly like a normal event, when the last chunk lands.

Rule of thumb

Under ~1 KB? TriggerServerEvent. Above a few KB, anything that contains tables of tables, NUI image buffers, large player exports, character snapshots, world saves? TriggerLatentServerEvent.

Step 1

Client to server: TriggerLatentServerEvent

Outcome: a 50 KB payload reaches the server without blocking other events.

Signature: TriggerLatentServerEvent(eventName, bps, ...). The bps argument is bytes per second. Pass -1 or 0 for the default (25,000 bytes/sec).

The receiver does not change
  • RegisterNetEvent handles latent and non-latent events the same way. The handler fires once, when the full payload has arrived.
  • You do not see chunks. The transport is invisible to your code.
  • source still resolves to the sending player ID inside the handler.
Step 2

Server to client: TriggerLatentClientEvent

Outcome: a server-side payload streams to one client without timing them out.

Signature: TriggerLatentClientEvent(eventName, targetPlayer, bps, ...). Use -1 for targetPlayer to broadcast to everyone (read the warning below before you do).

Recap 1

Latent send round-trips

Run /exportworld. The server logs [QUIET][latent][recv] src=N rows=5000. Other events you trigger meanwhile should still respond quickly.

Step 3

Sizing bps and the broadcast trap

Outcome: you can pick a sane bps for your payload and you understand why -1 as targetPlayer is dangerous.

Pick bps as per-target bandwidth. The official guidance:

  • bps = 25000 (default) is fine for most payloads under 200 KB.
  • bps = 50000-100000 for bigger payloads where you want the transfer to finish in seconds.
  • Above ~10000000 the system tries to flush almost everything at once and you lose the latency benefit. Stay well below that.
Broadcast multiplies the bill

If you call TriggerLatentClientEvent('big:event', -1, 100000, payload) on a 60-player server, the server is sending 100000 * 60 = 6 MB/s. That can saturate the host uplink and throttle every player at once. Either lower bps for broadcasts, or loop the player list and stagger sends.

Use a checksum or version field

For catalogs and config blobs that rarely change, send a small version first via TriggerClientEvent. Only call the latent push when the client says its cached version is stale. You will avoid most of the bandwidth.

Common mistakes

Forgetting the bps parameter

TriggerLatentServerEvent('qu:world:import', payload) looks correct, but it sends payload as the bps and ignores any further arguments. Always pass the rate explicitly.

Using latent events for tiny payloads

Latent events have per-chunk bookkeeping overhead. For payloads under a kilobyte, TriggerServerEvent is faster and cheaper. Save latent for the big stuff.

Treating latent events as a stream

You cannot read partial data on the receiver. The handler runs once, when the full payload is reassembled. If you need real streaming, use HTTP requests or a chunked event protocol you build yourself.

Skipping validation because the payload is "internal"

Latent events still cross the trust boundary. A modded client can fire qu:world:import with whatever it wants. Validate the structure on the server exactly as you would for a regular net event.

Broadcasting at high bps

TriggerLatentClientEvent('event', -1, 250000, payload) with 50 players online tries to send 12.5 MB/s out of the server. Either lower bps for broadcasts, or send to one player at a time with a delay.

Cheat sheet

TriggerLatentServerEvent(name, bps, ...)
Client to server. Streams a big payload at bps bytes/sec.
TriggerLatentClientEvent(name, target, bps, ...)
Server to client. Pass -1 for all players, but watch the multiplier.
Default bps
Pass 0 or -1 to use the engine default of 25,000 bytes/sec.
Receiver native
Same as a regular event: RegisterNetEvent. The handler fires once, when the payload is fully reassembled.
Per-target rate
bps applies to one recipient. Broadcasting to N players multiplies the server uplink load by N.
Validation rule
Latent events cross the trust boundary like any other net event. Validate every field server-side.

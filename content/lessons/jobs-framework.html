Jobs framework: clock-in, grades, paychecks

A job is a tiny state machine: off-duty, on-duty, optional grade, optional shift timer. Permissions unlock actions; payouts hit the bank on a clock. We build the spine without depending on a specific framework.

You will learn
Job table, duty toggle, grade gating, payroll loop, audit log.
Time
~35 min
Project
qu_jobs

Before you start

Mental model. A job is a shift schedule with paystubs. Clocking in flips a flag; the payroll thread reads the flag every N minutes and writes a row to job_payouts.

The shape

  • Catalog, a Lua table of jobs with grades, salary per grade, allowed actions.
  • State, per-player { job, grade, onDuty, since } in memory, persisted on change.
  • Authority, only the server flips duty, computes pay, and grants action permissions.

Pro tip. Keep job IDs short and lowercase (police, ems, mechanic). They become ACE group suffixes and webhook tags.

Step 1 · Define the catalog

One source of truth. No magic numbers in the rest of the resource.

-- shared/jobs.lua
Jobs = {
  unemployed = { label = "Unemployed", grades = { [0] = { label = "None", pay = 0 } } },
  police = {
    label = "Police",
    grades = {
      [0] = { label = "Cadet",   pay = 120 },
      [1] = { label = "Officer", pay = 180 },
      [2] = { label = "Sergeant",pay = 260 },
      [3] = { label = "Chief",   pay = 400 },
    },
    ace = "qu_jobs.police", -- ACE permission name granted while on-duty
  },
  mechanic = {
    label = "Mechanic",
    grades = { [0] = { label = "Apprentice", pay = 90 }, [1] = { label = "Boss", pay = 220 } },
    ace = "qu_jobs.mechanic",
  },
}

The catalog is shared because the client UI also reads it (label, grades). Pay is per-tick (we define a tick below).

Step 2 · Duty toggle (server)

-- server/duty.lua
local State = {} -- [src] = { job, grade, onDuty, since }

local function load(src)
  local id = GetPlayerIdentifierByType(src, "license")
  local row = MySQL.single.await("SELECT job, grade FROM users WHERE license = ?", { id })
  State[src] = { job = row and row.job or "unemployed", grade = row and row.grade or 0, onDuty = false, since = 0 }
  return State[src]
end

RegisterNetEvent("qu_jobs:toggleDuty", function()
  local src = source
  local s = State[src] or load(src)
  if s.job == "unemployed" then return end
  s.onDuty = not s.onDuty
  s.since  = os.time()
  TriggerClientEvent("qu_jobs:state", src, s)
  print(("[qu_jobs] %s -> %s (grade %d) duty=%s"):format(GetPlayerName(src), s.job, s.grade, tostring(s.onDuty)))
end)

AddEventHandler("playerDropped", function() State[source] = nil end)

Server owns the flag. The client never sets onDuty directly; it asks, the server decides, then broadcasts back.

Step 3 · Payroll loop

-- server/payroll.lua
local TICK_MS = 10 * 60 * 1000 -- 10 min

CreateThread(function()
  while true do
    Wait(TICK_MS)
    for src, s in pairs(State) do
      if s.onDuty then
        local grade = Jobs[s.job] and Jobs[s.job].grades[s.grade]
        local pay = grade and grade.pay or 0
        if pay > 0 then
          exports.qu_economy:addCash(src, pay)
          MySQL.insert.await(
            "INSERT INTO job_payouts (license, job, grade, amount) VALUES (?, ?, ?, ?)",
            { GetPlayerIdentifierByType(src, "license"), s.job, s.grade, pay }
          )
          TriggerClientEvent("qu_jobs:paid", src, pay)
        end
      end
    end
  end
end)

A single thread iterates everyone on duty. Pay only fires when the player is online and on-duty at the moment the tick lands.

Step 4 · Gated actions via ACE + duty

-- server/actions.lua
local function isOnDuty(src) local s = State[src]; return s and s.onDuty end

local function hasJobAce(src, job)
  local def = Jobs[job]
  if not def or not def.ace then return false end
  return IsPlayerAceAllowed(src, def.ace)
end

RegisterNetEvent("qu_jobs:police:cuff", function(target)
  local src = source
  if not isOnDuty(src) or not hasJobAce(src, "police") then return end
  local s = State[src]
  if s.job ~= "police" then return end
  TriggerClientEvent("qu_jobs:police:cuffed", target, src)
end)

Two gates: the ACE permission (set via set_principal or per-job role) and the runtime duty flag. Off-duty cops cannot cuff.

  2. Trigger qu_jobs:toggleDuty from chat command. Confirm the server prints duty=true.

Step 5 · At scale: catalog, payroll, audit

Three sub-systems break first under 64-player load: the catalog when grades change mid-shift, the payroll loop when a tick straddles a server restart, and the action gate when ACE caches go stale. Each has a fix with a measurable cost.

5.1 Catalog evolution and grade promotion at scale

A pure-Lua catalog is fine for read but hostile to writes. Promotions through /promote need to land in DB, refresh State[src].grade, invalidate the ACE cache, and push the new state to the client, atomically, so an action firing mid-promotion never reads two grades.

-- server/promote.lua
local function promote(src, target, newGrade)
    if not IsPlayerAceAllowed(src, "qu_jobs.admin") then return false, "denied" end
    local s = State[target]
    if not s or s.job == "unemployed" then return false, "no_job" end
    local def = Jobs[s.job]
    if not def.grades[newGrade] then return false, "bad_grade" end

    -- Atomic: update DB, then in-memory, then ACE refresh, then client push
    local license = GetPlayerIdentifierByType(target, "license")
    local ok = MySQL.update.await(
        "UPDATE users SET grade = ? WHERE license = ? AND job = ?",
        { newGrade, license, s.job }
    )
    if ok ~= 1 then return false, "db_miss" end

    s.grade = newGrade
    -- ACE bound to the job, not the grade, so no re-add needed unless grade gates abilities
    -- If grade-gated: ExecuteCommand(("add_principal player.%s qu_jobs.%s.g%d"):format(license, s.job, newGrade))
    TriggerClientEvent("qu_jobs:state", target, s)
    AuditLog("promote", { actor = src, target = target, job = s.job, grade = newGrade })
    return true
end

Order matters: DB first (so a crash before in-memory update is recoverable), then memory, then client. If you flip memory before DB and the UPDATE fails, the player is paid at the wrong grade until reload.

Race fix. If a paid action (cuff, fine, repair) fires between s.grade = newGrade and TriggerClientEvent, the server reads the new grade but the client UI still shows the old one. Acceptable: the server is authority. Unacceptable: gating a paid action on the client-cached grade. Always re-read State[src].grade server-side at the moment of payment.

5.2 Payroll integrity across restarts

Naive payroll loses pay when the server restarts mid-tick. Idempotency key on job_payouts + a coarse tick clock survives restarts and reconciles support tickets.

-- server/payroll.lua (hardened)
local TICK_MS = 10 * 60 * 1000
local lastTickAt = os.time()

CreateThread(function()
    while true do
        Wait(TICK_MS)
        local tickAt = os.time()
        local tickKey = tostring(tickAt - (tickAt % 600)) -- bucket to 10-min boundary
        local rows = {}
        for src, s in pairs(State) do
            if s.onDuty and (tickAt - s.since) >= 60 then -- min 1 min on-duty
                local def = Jobs[s.job]
                local grade = def and def.grades[s.grade]
                local pay = grade and grade.pay or 0
                if pay > 0 then
                    rows[#rows+1] = {
                        license = GetPlayerIdentifierByType(src, "license"),
                        job = s.job, grade = s.grade, amount = pay,
                        tick_key = tickKey,
                    }
                    exports.qu_economy:addCash(src, pay)
                    TriggerClientEvent("qu_jobs:paid", src, pay)
                end
            end
        end
        if #rows > 0 then
            -- Single batched INSERT IGNORE; tick_key UNIQUE per (license, tick_key) prevents double-pay on restart
            local sql, args = "INSERT IGNORE INTO job_payouts (license, job, grade, amount, tick_key) VALUES ", {}
            local parts = {}
            for _, r in ipairs(rows) do
                parts[#parts+1] = "(?, ?, ?, ?, ?)"
                args[#args+1] = r.license; args[#args+1] = r.job
                args[#args+1] = r.grade;   args[#args+1] = r.amount
                args[#args+1] = r.tick_key
            end
            MySQL.query.await(sql .. table.concat(parts, ","), args)
        end
        lastTickAt = tickAt
    end
end)

Three changes: (a) tick_key as part of a UNIQUE index makes the INSERT idempotent, a restart that re-fires the same tick cannot double-pay; (b) INSERT IGNORE drops conflicting rows silently; (c) batching collapses 32 statements into 1, dropping payroll cost from 4×32 ms to 9 ms p95 on 32 on-duty players.

AFK detection. Add last_input_at to State[src] from a client heartbeat (every 60s when keys/mouse move). Skip payroll for entries where tickAt - last_input_at > 900. Stops AFK farming without writing a separate watchdog.

5.3 Action gate composition and audit

Per-grade abilities scale better than one ACE per job. Combine a grade.abilities list with a single audit row per gated action; support reconstructs every paid hit afterward.

-- shared/jobs.lua (extended)
police = {
    label = "Police",
    grades = {
        [0] = { label = "Cadet",   pay = 120, abilities = { "cuff", "search" } },
        [1] = { label = "Officer", pay = 180, abilities = { "cuff", "search", "fine", "impound" } },
        [2] = { label = "Sergeant",pay = 260, abilities = { "cuff", "search", "fine", "impound", "release" } },
        [3] = { label = "Chief",   pay = 400, abilities = { "*" } },
    },
    ace = "qu_jobs.police",
}

-- server/actions.lua
local function can(src, ability)
    local s = State[src]
    if not s or not s.onDuty then return false, "off_duty" end
    if not IsPlayerAceAllowed(src, Jobs[s.job].ace) then return false, "no_ace" end
    local abilities = Jobs[s.job].grades[s.grade].abilities
    for i = 1, #abilities do
        if abilities[i] == "*" or abilities[i] == ability then return true end
    end
    return false, "no_ability"
end

local function audit(src, action, target, ok, reason)
    MySQL.insert("INSERT INTO job_audit (actor_license, actor_job, actor_grade, action, target, ok, reason, at) VALUES (?, ?, ?, ?, ?, ?, ?, NOW())", {
        GetPlayerIdentifierByType(src, "license"),
        State[src] and State[src].job or "none",
        State[src] and State[src].grade or 0,
        action, tostring(target), ok and 1 or 0, reason or "",
    })
end

RegisterNetEvent("qu_jobs:police:cuff", function(target)
    local src = source
    local ok, reason = can(src, "cuff")
    audit(src, "cuff", target, ok, reason)
    if not ok then return end
    TriggerClientEvent("qu_jobs:police:cuffed", target, src)
end)

can() centralizes the gate so every action runs the same checks in the same order. audit() writes one row per attempt, success or denial. When a player files "officer cuffed me off-duty," support runs SELECT * FROM job_audit WHERE target = ? AND action = 'cuff' ORDER BY at DESC LIMIT 20 and sees the truth in 0.4 ms.

Recap 3

Senior depth: authority chain

Permission is a graph, not a flag: job, grade, jurisdiction, and on-duty state combine into the answer of “can this player do X right now?”. You know why optimistic locks save you from two admins promoting the same player to last-write-wins, and why the audit log is the only honest source of who-did-what. If the chain of authority is unauditable, it is unaccountable.

Common mistakes

Letting the client set duty.

Trust nothing from the client. Always re-check on the server before granting any action.

Paying offline players.

The loop iterates State[src] which only contains active sources. Do not run a payroll over the entire DB.

Storing job in client memory.

Client copy is a hint for UI. The server is the source of truth at every action.

Hardcoding pay in handlers.

Always read Jobs[s.job].grades[s.grade].pay. Changing salary becomes a one-line edit.

Forgetting to clear State on drop.

Leaks memory and pays ghosts. Always wire playerDropped.

Recap

  • Catalog → runtime state → gated actions → payroll log.
  • Server owns duty, grade, and pay computation.
  • ACE plus duty flag protect every job action.
  • Persist job + grade on change; clear in-memory state on drop.
  • Audit pay rows so support can reconcile complaints.

By the end of this lesson you will have a folder structure that every script you install fits into, with no guesswork about where things belong.

You'll ship
A clean resources/ folder organized into categories that make sense six months from now.
Time
About 15 minutes to set up, then forever free.
You'll learn
Why flat folders become chaos, and how a simple prefix convention stops it.
Prereqs
You have a server folder with a resources/ directory.
In 3 bullets

You will organize your resources folder like this:

  • Every resource gets a category prefix so you know what it does without opening it.
  • Core frameworks sit in [core], scripts sit in [scripts], maps in [maps], and so on.
  • You will be able to find any script in under five seconds, even with 100+ resources installed.

Step by step

Lesson 02: Organize server folders, clean from day one

Start here

What this is
A naming convention for your resources folder that prevents chaos.
Example: [core]/qb-core instead of a flat list of 50 folders.
Where to find it
Inside your server's resources/ directory.
Example: /home/fxserver/resources/ is the root of all your scripts.
How to use it
Create category folders with square brackets. Place resources inside.
Example: move qb-core into resources/[core]/.
What to do now
Create the six standard category folders in your real server.
Example: resources/[core], resources/[scripts], resources/[maps].

Outcome

You will create a category-based folder structure inside resources/ and move one resource into the correct category.

Why it matters

A flat resources folder with 80 subdirectories is impossible to maintain. You will lose scripts, forget what they do, and break dependencies. Five minutes of organization now saves hours of debugging later.

Vocabulary

Category folder
A parent folder named with square brackets that groups related resources.
Example: [core] holds framework resources like qb-core or es_extended.
Resource path
The full path from server root to a script's folder.
Example: resources/[scripts]/qu_heal is a complete resource path.
Ensure path
How you reference a resource in server.cfg. FiveM reads the folder name, not the category.
Example: ensure qu_heal works even if it lives in resources/[scripts]/qu_heal.

Minimal example

Expected output: The six category folders appear when you run ls resources/ in your terminal.

Checklist

  1. Open your server's resources/ folder in a file explorer or terminal.
  2. Create the six category folders: [core], [scripts], [maps], [vehicles], [clothing], [system].
  3. Pick one existing resource that is a script and move it into [scripts]/.
  4. Update its ensure line in server.cfg — only if you changed the resource folder name (category folders do not change the resource name).
  5. Restart the server to confirm the resource still starts from its new location.
  6. Move every other resource into the matching category.

Exercise

Create a test resource folder at resources/qu_folder_test with an empty fxmanifest.lua. Move it into resources/[scripts]/qu_folder_test. Add ensure qu_folder_test to your server.cfg and confirm the resource starts without errors.

Expected output: Server console prints Started resource qu_folder_test after restart.

Common failures

  • Couldn't find resource qu_script means the folder name inside the category does not match the ensure name in server.cfg. Check spelling and case.
  • Resource [core] does not have a manifest means you put a manifest inside the category folder itself. Manifests belong inside individual resource folders, not category folders.
  • Dependency not found after moving means a script expects its dependency at a hardcoded relative path. Move both resources together into the same category.
  • Resource loaded but scripts not running means the resource moved but the manifest path to client/server files used ../ relative paths. Fix the manifest to use paths inside the resource folder only.

Refactor

Add a resources/README.txt file that lists every category and explains what belongs where. This is your map. Every time you add a script, the README tells you — and anyone who helps you — exactly where to put it.

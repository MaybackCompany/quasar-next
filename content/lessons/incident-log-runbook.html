Incident log and runbook, handling server emergencies

By the end of this lesson you will have a runbook that tells you exactly what to do when your server crashes, gets attacked, or corrupts data — plus a log format that makes every incident teach you something instead of just scaring you.

You'll ship
A printed runbook with five emergency scenarios and the exact steps to resolve each one, plus an incident log template you fill out every time something goes wrong.
Time
About 25 minutes to write both documents.
You'll learn
How to stay calm during an outage, which commands to run first, and how to write an incident report that actually prevents the problem from happening again.
Prereqs
Your server has been running for at least a day. You have console access.
In 3 bullets

You will build an incident response system that does this:

  • When something breaks, you open the runbook and follow a script instead of panicking. Five scenarios cover 90% of real FiveM emergencies.
  • You log every incident in a standard format: what happened, what you did, what worked, and what you will change to prevent it next time.
  • After three incidents, you have a pattern. You know what breaks most often on your server and you fix it before it breaks again.

Step by step

Start here

What this is
A written plan for when things go wrong, plus a log of every time they did.
Example: server crashes at 3 AM. You open the runbook, follow the crash scenario, log it, and go back to sleep.
Where to find it
Keep the runbook printed or in a text file on your desktop, not on the server.
Example: ~/Documents/fivem-runbook.txt or a printed sheet next to your keyboard.
How to use it
Find the scenario that matches your emergency. Follow the steps in order. Do not skip steps.
Example: Scenario 2 is "server crashed." Step 1 is always "read the last 50 lines of the log."
What to do now
Copy the five scenarios below into a file. Save it somewhere you can reach even if the server is down.
Example: email it to yourself. Print it. Save it in Google Docs. Do all three.

Outcome

You will have a printed runbook for five emergencies and an incident log template that you start using today.

Why it matters

During an outage, your brain shuts down its reasoning centers and runs on instinct. A runbook replaces instinct with a checklist. It also turns every crisis into a learning opportunity instead of a recurring nightmare.

Vocabulary

Runbook
A printed or saved document with step-by-step instructions for handling specific emergencies.
Example: "Scenario: Server crash. Step 1: Check the log. Step 2: Restart. Step 3: Verify."
Incident
Any event that disrupts normal server operation. Crashes, attacks, data loss, and failed updates all count.
Example: the database went read-only and players could not save progress for 45 minutes.
Root cause
The underlying reason an incident happened, not just the symptom you fixed.
Example: the server crashed because the disk was full. The root cause is no disk monitoring alert.

Minimal example

Expected output: You save both documents. Within one week, you log at least one incident — even a minor one — using the template.

Checklist

  1. Copy the five runbook scenarios into a text file on your personal computer.
  2. Print it or save it somewhere you can access if the server is down.
  3. Copy the incident log template into a file called incident-log.txt on your personal computer.
  4. Create your first incident entry: write down the last time your server had a problem, even a small one.
  5. Fill out the template completely for that incident, including root cause and prevention.
  6. Share the runbook file path with any co-owners or admins who might need it during an emergency.

Exercise

Create incident #001 right now. Even if your server has never had a major problem, log the last minor hiccup: a resource that failed to start, a player who could not connect, a config typo that caused a restart. Fill out every field in the template.

Expected output: incident-log.txt contains a complete entry for incident #001 with all fields filled in.

Common failures

  • Not writing the incident down because "it was small" — small incidents become big ones when their root cause is never addressed. Log everything.
  • Skipping the root cause section — "server crashed" is a symptom. "Disk was full because I never set up log rotation" is a root cause. Always dig one layer deeper.
  • Restarting the server before saving crash logs — the evidence is lost on restart. Always copy logs first, restart second.
  • Runbook saved only on the server machine — if the server is down, you cannot read the runbook. Keep a copy on your phone, in email, or printed.

Refactor

After you accumulate five real incidents, review them and write a sixth runbook scenario for the most common problem your specific server has. Every server is different. Your runbook should match your reality.

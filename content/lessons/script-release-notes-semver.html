Semantic versioning and release notes for scripts

By the end of this lesson you will version your FiveM scripts with semantic versioning (semver), write release notes that customers actually read, and stop breaking servers with silent updates.

You'll ship
A versioned script with a version.lua manifest, a CHANGELOG.md that follows semver rules, and a release pipeline that tells customers what changed before they update.
Time
About 30 minutes if you go slow.
You'll learn
How to read and write semantic versions, what each number means, and how to write release notes that reduce support tickets by half.
Prereqs
You have shipped or plan to ship a script to paying customers. That is enough.
In 3 bullets

You will build a versioning system that does this:

  • Tag every release with a semantic version like 1.2.0 where each number has a clear, enforceable meaning.
  • Write release notes in a CHANGELOG.md format that tells customers exactly what is new, what is fixed, and what might break their setup.
  • Add a version check in your script that warns customers when they are running an outdated version.

What this is
Semver is a three-number version format: MAJOR.MINOR.PATCH.
Example: this lesson versions a script from 0.1.0 to 1.0.0.
Where to find it
Find the version in version.lua and CHANGELOG.md.
Example: check the version before every release.
How to use it
Bump the right number based on what changed, then write notes.
Example: a bugfix bumps PATCH, a feature bumps MINOR.
What to do now
Version one script and write one release note entry.
Example: match the expected format before shipping.

Outcome

You will version a script at 1.0.0 and produce a CHANGELOG.md entry that follows semver rules.

Why it matters

Without clear versions, customers cannot know if an update breaks their setup. Without release notes, they open support tickets for every change.

Vocabulary

MAJOR
Incremented when you make incompatible API changes.
Example: renaming an export from giveItem to addItem is a MAJOR change.
MINOR
Incremented when you add backward-compatible features.
Example: adding a new /repair command without changing old commands.
PATCH
Incremented when you fix bugs without adding features.
Example: fixing a nil value error in the heal command.

Common failures

  • version mismatch between files means version.lua and CHANGELOG.md show different numbers. Update both together.
  • bumping MAJOR for a bugfix means customers think their setup will break when it will not. They ignore future MAJOR warnings.
  • no changelog entry for a breaking change means customers update blind, hit an error, and open angry tickets.
  • version not exposed to customers means when they report a bug, you do not know which version they are running.

Refactor

Cheat sheet

Semver decision table

  • Fixed a typo? Bump PATCH. 1.0.0 → 1.0.1
  • Added a new command? Bump MINOR, reset PATCH to 0. 1.0.1 → 1.1.0
  • Renamed an export? Bump MAJOR, reset MINOR and PATCH to 0. 1.1.0 → 2.0.0

The big idea, in one minute

A version number is a promise. 1.2.3 tells the customer: "This is the first major version, with two feature additions, and three bugfixes since those features landed." When you bump MAJOR, you are saying "something you rely on changed — read the notes before updating." When you bump PATCH, you are saying "nothing changed except the bug is gone — update with confidence." If you use version numbers randomly, the numbers mean nothing and customers ignore them.

Think of a car model.

The MAJOR version is the model generation — a 2024 model versus a 2023 model. Different body, different features. The MINOR version is the trim level — the same car with a sunroof added. The PATCH version is the recall fix — the same car with a bolt tightened. You would not call a bolt fix a new model year, and you should not call a renamed export a patch.

Bad way and good way, side by side

Most developers either do not version at all, or they version randomly. Both confuse customers.

Don't do this

No version, or a meaningless version. No changelog.

Customers cannot tell what version they have. You cannot reproduce bugs. Every update is a gamble because no one knows what changed.

Do this

Semver in code, changelog in a file, version printed on every startup.

The version is machine-readable and human-readable. The changelog is dated and organized by change type. The startup log confirms the version in every console.

Before you start

Step 1

Create version.lua

One file that holds the version. Every other file can read it.

  1. Create version.lua in your resource root.
  2. Define VERSION = { major = 1, minor = 0, patch = 0 }.
  3. Define VERSION_STRING = '1.0.0' for logging.
  4. Add version.lua to fxmanifest.lua as a shared script so both client and server can read it.
Step 2

Write your first changelog entry

A changelog is a dated record of every change, organized by type.

  1. Create CHANGELOG.md in your resource root.
  2. Write the header: ## [1.0.0] - YYYY-MM-DD.
  3. Under it, add sections: ### Added, ### Fixed, ### Changed.
  4. Write one bullet per change. Keep each bullet to one line. Customers skim — do not write paragraphs.
  5. Save the file. This is now the source of truth for what changed.
Step 3

Decide when to bump each number

The hardest part of semver is discipline. Use this table every time.

  1. PATCH: you fixed a bug. Nothing was added, nothing was renamed. Bump 1.0.0 → 1.0.1.
  2. MINOR: you added a feature, but old features still work the same way. Bump 1.0.1 → 1.1.0, reset PATCH to 0.
  3. MAJOR: you changed or removed something customers depend on. Bump 1.1.0 → 2.0.0, reset MINOR and PATCH to 0.
  4. If you are not sure, bump the lower number. It is safer to under-bump than over-bump.
Recap

Three tests, in order

Run each test and verify the output before moving on.

  1. Test 1, version log on startup.
    • Action: start the resource and check the console.
    • Expected: the console prints [QUIET][version][start] version=1.0.0.
    • How to verify: the version string matches version.lua exactly.
  2. Test 2, changelog and version match.
    • Action: compare the top entry in CHANGELOG.md to version.lua.
    • Expected: the version number in the changelog header matches VERSION_STRING.
    • How to verify: both say 1.0.0. If they differ, update one of them.
  3. Test 3, semver discipline.
    • Action: look at your last three commits. Classify each as PATCH, MINOR, or MAJOR.
    • Expected: your current version reflects the highest classification among those commits.
    • How to verify: if you added a feature, the MINOR number increased. If you only fixed bugs, only PATCH increased.
What if I need to bump two numbers at once?

You never do. If you bump MINOR, PATCH resets to 0. If you bump MAJOR, MINOR and PATCH reset to 0. The highest-level change determines the version.

The rules you just followed

  • Every release has a semver tag: MAJOR.MINOR.PATCH.
  • MAJOR means breaking changes. MINOR means new features. PATCH means bugfixes.
  • The changelog is dated and organized into Added, Fixed, and Changed sections.
  • The version is printed on every resource start so console logs are always traceable.
  • Git tags match the version string exactly: v1.0.0.

Quick check, 3 questions

1. You rename an export from healPlayer to HealPlayer. Which number do you bump?
2. You fix a nil-check bug. No features added. What do you bump?
3. Why put the version in a separate version.lua instead of hardcoding it in every file?

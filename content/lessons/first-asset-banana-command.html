Your first script: the /banana command from scratch

You have never written a FiveM script. In this lesson you will type /banana in chat and see "You ate a banana" appear. Then you will make it give you health. Then armor. By the end you will have shipped your first asset, with a manifest, a command, and a working client-server split.

You'll ship
A working /banana command in your own resource folder that prints chat messages, gives health, and respects the client-server boundary.
Time
About 30 minutes if this is your first ever script.
You'll learn
How to create a resource folder, write an fxmanifest, register a command, and why the client should never call SetEntityHealth without server approval.
Prereqs
You can open a text editor and create a folder. That is all.
In 3 bullets

Your /banana command will do this:

  • The player types /banana in chat. A message appears: "You ate a banana. +10 health."
  • The player's health increases by 10. If they are at 200, it stays at 200 (capped at max).
  • The server decides whether the player gets the health. The client only asks. Cheaters get nothing.

Common failures

  • Command banana not found — the resource did not load. Check the folder name matches the ensure line, and the fxmanifest has no syntax errors.
  • attempt to call a nil value (global 'RegisterCommand') — you put RegisterCommand in server.lua. Chat commands must go in client.lua.
  • attempt to call a nil value (global 'TriggerEvent') — you are on the server side. chat:addMessage only works on the client.
  • Resource qu_banana has no scripts — your fxmanifest is empty or missing the client_script/server_script lines.

Refactor

Extract the chat message into a local function sendBananaChat(message) so you can call it from multiple places with different messages.

Cheat sheet

The three files every resource needs

  • fxmanifest.lua. Tells FiveM this folder is a resource. Lists client_scripts, server_scripts, and dependencies.
  • client.lua. Runs on the player's computer. Handles chat commands, UI, drawing, and most natives.
  • server.lua. Runs on your machine. Handles validation, database calls, and sending green lights to clients.

The big idea, in one minute

Every FiveM script is split into two sides. The client is the player's game — it can draw things, play sounds, and change health. The server is the referee — it checks who asked, whether they are allowed, and what the result should be. Your /banana command starts on the client (because chat is a client thing), asks the server for permission, and the server tells the client exactly what to do. This split is not optional. It is the entire security model.

Think of an ATM at a bank.

The keypad and screen (the client) let you type numbers and see your balance. But the money lives in the vault (the server). When you press "withdraw $20", the keypad sends a request to the vault computer. The vault checks your balance, subtracts $20, and tells the keypad to dispense the cash. The keypad never decides on its own whether to give you money. If it did, anyone with a screwdriver could become a millionaire.

Your /banana command is the keypad. The server.lua is the vault. The client asks. The server checks. The client acts only after the server says yes.

Unsafe banana and safe banana, side by side

The unsafe version is shorter. It is also an open door for cheaters. The safe version adds 10 lines and closes the door.

Unsafe: client heals itself

The client decides the health value and sets it. A cheater calls this every frame and never dies.

The cheater replaces current + 10 with 200 and has full health on every keypress. No server ever checked. No cooldown ever ran.

Safe: client asks, server approves

The client sends a request. The server checks and sends back the approved health value. Cheaters cannot bypass the check.

The health amount never crosses the network from the client. The server sets it as a constant (BANANA_AMOUNT = 10). The cooldown runs on the server where nobody can edit it. The client only heals after receiving the server's approved amount.

Before you start

If any box is unchecked, take five minutes to fix it first.

Step 1

Create the resource folder and manifest

A resource starts as an empty folder. The manifest tells FiveM what is inside.

  1. Open resources/. Create a new folder: qu_banana.
  2. Inside qu_banana, create a file: fxmanifest.lua.
  3. Paste this exact text: 
    fx_version 'cerulean'
game 'gta5'

author 'YourName'
description 'A simple /banana command'
version '1.0.0'

client_scripts {
    'client.lua'
}

server_scripts {
    'server.lua'
}
  4. fx_version 'cerulean' tells FiveM which API version to use. Cerulean is the current one.
  5. client_scripts and server_scripts tell FiveM which files to run and on which side. Save the file.
Step 2

Write the client command

Chat commands live on the client. This one prints a yellow message.

  1. In qu_banana, create a file: client.lua.
  2. Paste the "safe" client.lua from the good example above. Or start with just the chat message version from the minimal example.
  3. RegisterCommand('banana', function ... end, false) — the first argument is the command name (without the slash). The second is the function to run. The third (false) means the command is not restricted to admins.
  4. TriggerServerEvent('qu_banana:server:eat') sends a message to the server. The server decides what happens next.
  5. Save the file. The client side is complete.
Step 3

Write the server checker

The server is the gatekeeper. It checks the cooldown and sends the green light.

  1. In qu_banana, create a file: server.lua.
  2. Paste the server.lua from the good example above.
  3. local BANANA_AMOUNT = 10 — the amount is a constant on the server. It never crosses the network from the client, so cheaters cannot change it.
  4. local lastAte = {} — a table that remembers when each player last ate a banana. The cooldown is 5 seconds.
  5. TriggerClientEvent('qu_banana:client:heal', src, BANANA_AMOUNT) — sends the green light to exactly one player (the one who asked), with the approved heal amount.
  6. Save the file. The server side is complete.
Step 4

Turn it on and eat a banana

One line of config, one restart, one command. Then you have shipped.

  1. Open server.cfg. Add a new line: ensure qu_banana. Place it after chat and spawnmanager.
  2. Start your server, or in a running server type refresh then ensure qu_banana in the console.
  3. Join the server. Take some damage first (punch a wall or shoot the ground near you).
  4. Press T to open chat. Type /banana and press Enter.
  5. You should see a yellow message and your health bar should move up by 10.
  6. Type /banana again quickly. You should see a red message telling you to wait. The health does not increase on the second try.
  7. Wait 5 seconds. Type /banana again. It works. You have shipped your first FiveM script.
Recap

Three tests, in order

Run each test. If one fails, fix it before moving on.

  1. Test 1, the message appears.
    • Action: type /banana in chat.
    • Expected: yellow text "Banana: You ate a banana. +10 health." appears in chat.
    • How to verify: the message is yellow and visible in the chat box.
  2. Test 2, health increases.
    • Action: take damage, check your health, type /banana, check again.
    • Expected: health is 10 points higher (capped at max).
    • How to verify: punch a wall until health drops below 190, then eat a banana. The bar moves up.
  3. Test 3, cooldown works.
    • Action: type /banana twice in under 5 seconds.
    • Expected: first one works. Second one shows a red "Slow down" message and health does not change.
    • How to verify: the red message appears. Your health after the second /banana is the same as after the first.
Why does the cooldown live on the server and not the client?

A cheater can edit any Lua file that runs on their computer. If the cooldown lived in client.lua, a cheater could change it to 0 seconds and spam heal. The server runs on your machine, not theirs. Rules that live on the server cannot be edited by players.

The rules you just followed

  • The client only asks. It never decides the result.
  • The server checks the player (via source), the cooldown, and the amount. It sends the green light.
  • The heal amount is a server constant. It never crosses the network from the client.
  • The cooldown table lives on the server, keyed by source. Cheaters cannot reach it.
  • The client only heals after receiving the approved event from the server.

What you learned

  • Every FiveM resource needs an fxmanifest.lua that lists its client and server scripts.
  • RegisterCommand creates a chat command. It lives on the client because chat is a client feature.
  • The client asks the server with TriggerServerEvent. The server checks and replies with TriggerClientEvent.
  • All rules (cooldowns, validation, amount limits) live on the server where cheaters cannot reach them.
  • A working command is 20 lines of code. The security layer is 10 of those lines. Never skip the security layer.

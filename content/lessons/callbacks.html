Callbacks: ask a question, get an answer

Events are good when you just want to send a message and move on. But some situations need a reply. For example: "do I have enough money to buy this?" In that case the client asks a question, waits, and gets an answer back. That pattern is called a callback.

How to read this lesson

This lesson is about one difference only: events send a message and move on, callbacks send a question and wait for the answer. Keep comparing those two ideas while you read and the page becomes much simpler.

The mental model: a phone call, not a postcard

An event is like sending a message and moving on. You do not expect an answer back. That is perfect for fire-and-forget actions.

A callback is different. The client asks the server a question and waits. When the server replies, your client code continues with that returned value.

Why not two events?

You could build the same pattern with two events, one for the question and one for the reply. But lib.callback already handles that plumbing for you, so it is cleaner to learn this first.

The real problem with the two-event approach

Picture it. You fire TriggerServerEvent('shop:checkCash', 10). The server fires back TriggerClientEvent('shop:cashResult', src, true). Now your client catches that reply in some other handler far away in the file. Three headaches show up fast:

  • You cannot tie the answer to the question. If the player fired two requests, which reply belongs to which? There is no thread connecting them.
  • You cannot wait in line. The reply lands in a separate handler, so the code that asked has already moved on. You end up scattering logic across the file.
  • Chained checks turn into a mess. "Check cash, then check space, then confirm" becomes a tangle of events calling events.

A callback collapses all of that into one readable line: local ok = lib.callback.await('shop:checkCash', false, 10). Ask, wait, answer, continue. The reply is right there next to the question.

When do you need a callback?

Use a callback when the client needs a piece of server-held truth before it can continue. Simple test: does the next line on the client depend on a value only the server knows?

  • "Does this player have enough money?" → yes, balance lives on the server.
  • "What's this vehicle's owner?" → yes, ownership is stored in the database.
  • "Is this house unlocked?" → yes, lock state is authoritative server-side.
  • "Play a heal animation." → no, that's fire-and-forget. Use an event.
  • "Broadcast the weather." → no, no reply needed. Use TriggerClientEvent(-1, ...).

If the answer is "I need a value back before I continue", it's a callback. Everything else is an event.

Step 1

Register the answer: server-side

Outcome: the server declares a named callback and returns a value when asked.

We'll build qu_callbacks: a /buycar command that first asks the server "do I have $50,000?" before proceeding. The server holds a fake balance, answers with a boolean, and the client branches on the answer.

Three things matter here. First, lib.callback.register takes a name and a function. Second, the first parameter is still source, just like a server event handler. Third, the function returns its answer, and that is what the client receives.

You can return more than one value. Here the server returns a boolean and a small note.

Every argument is still client-supplied

Arguments passed to a callback (like amount above) came from the client's computer. The type() / range / authority / context check from Lesson 12 applies here too, callbacks don't magic away the trust rules.

Step 2

Ask the question: client-side, synchronous form

Outcome: a client script that pauses, waits for the server, and continues with the answer.

The client calls lib.callback.await. This pauses that one bit of code until the server replies. The rest of the game keeps running.

lib.callback.await takes the callback name, a throttle delay in ms (false disables it, like above), and then any arguments the server needs. It returns whatever the server returned. Here that means ok and note.

Restart the resource and run /buycar. With price = 50000, it should deny because the balance is only 10000. Change the price to 5000 and it should approve. That is the whole job of a callback -> ask, wait, answer, continue.

Threads, not the whole client

lib.callback.await does not freeze the whole game. It only pauses the small coroutine it is running in.

Step 3

Async form: when you don't want to wait in-line

Outcome: you know the second flavour of the call, and when to pick each.

Sometimes you do not want to wait inline. You want the answer to arrive later and run its own function. That is the async form.

It is the same network trip underneath, just a different local code style. Use await when the next line depends on the answer. Use the async form when you want the reply to arrive later without pausing this bit of code.

Don't block the main thread needlessly

Inside UI hot-paths or frequently-called handlers, consider the async form. await is clean and readable, but if you're calling ten different checks every frame, async keeps things snappier.

One trap to dodge: the async form is tempting to nest, and nesting gets ugly fast. If callback B depends on callback A's answer, and C depends on B, the obvious move is to put each one inside the previous one's reply function. People call the result "callback hell", and you can see why:

Step 4

Timeouts: don't hang the client forever

Outcome: you set a sane cutoff so a misbehaving server doesn't freeze your flow.

The second argument is a timeout in milliseconds. false means wait forever. In production, use a real number like 5000. If the server does not answer in time, await returns nil.

The argument order can look strange at first: callback name, timeout, then the real callback arguments. Use named variables in real code so it is easier to read.

Always handle the nil return

If you set a timeout, the call can return nil. Treat that as an error case: log it, notify the player, retry or abort. Skipping the nil check is how "the buy button sometimes silently does nothing" bugs happen in production.

Recap 1

Event or callback?

For each scenario, pick event or callback and justify in one sentence. No peeking.

  1. Player triggers a door-open animation that every nearby client should see.
  2. Player wants to know how much they have in their bank before opening the ATM menu.
  3. Admin broadcasts "server restart in 5 min" to everyone.
  4. Player tries to pick up an item; the server needs to check inventory space before confirming.
  5. The client wants to report to the server that the player died.
Answers
  1. Event. No answer needed, fire qu:client:doorFx to nearby clients and move on.
  2. Callback. The menu needs the number before it renders; balance is server-held truth.
  3. Event. TriggerClientEvent('qu:client:announce', -1, msg). One-shot, no reply.
  4. Callback. Client must know yes/no before proceeding to play the pickup animation.
  5. Event. Fire-and-forget report. The server acts on it; the client doesn't need a confirmation to continue.

Rule of thumb: if the next line of client code depends on a value the server is about to compute, callback. Otherwise, event.

Step 5

Combine event + callback: the clean pattern

Outcome: you see the real-world shape, callback for the question, event for the write.

Callbacks are best for questions. Events are still best for writes. So a real feature often uses both: callback to check, event to commit. Here is the full /buycar shape done the safe way.

Notice the security move in the server event handler: it re-reads the real price and ignores the client's price. Why? Because the client can lie between the callback and the event. The callback gives information. The event handler still has to enforce truth.

Callbacks don't replace server validation

A successful canAfford reply is not permission. It's information. The write handler (qu:server:buyCar) must re-validate, because the two calls are separate round-trips and a cheater controls everything in between.

Recap 2

Spot the bug

Read this code. It runs. It compiles. It's still broken. Find the trust violation.

The violations
  1. The server trusts the client's price. A cheater sends price = 0 and gets an apartment free. Fix: the server looks up the real price from a server-side config table; the client never sends it.
  2. There's no validation that the callback said yes. Even with the callback, the server should re-check balances[src] >= realPrice before deducting. Fix: add the balance check in the event handler.
  3. (Bonus) There's no feedback to the client if the deduction fails. Fix: TriggerClientEvent('qu:client:buyResult', src, ok, reason) so the UI can react.

This pattern, callback said yes, event trusts client, is the classic intermediate-level mistake. You've leveled up past it now.

Common mistakes: the callback edition

Treating a callback reply as permission

A callback answers questions. It doesn't commit state. Every write still goes through an event handler that re-validates. Callback → event → validate → write. Always.

Calling await inside a tight loop

Every await is a network round-trip. Calling it in a loop makes the UI feel laggy. Cache the answer if it's stable, or batch the question (ask once for a whole list).

Forgetting the timeout

lib.callback.await('name', false, args) waits forever if the server never replies. In production, set a number, 5000 ms is usually enough, and handle the nil return.

Registering a callback server-side without validating args

Arguments still come from the client. type() them. Bound them. Authorise them. Same rules as any other net event.

Mixing up lib.callback.register (server) with lib.callback / lib.callback.await (client)

Four names for three directions of flow: register on the server, callback() (async) or callback.await() (sync) on the client. Keep a sticky note; it pays off.

Cheat sheet: callbacks

Register (server)
lib.callback.register('qu:cb:name', function(source, ...) return answer end)
Call, sync (client)
local ans = lib.callback.await('qu:cb:name', timeoutMs, ...)
Call, async (client)
lib.callback('qu:cb:name', timeoutMs, function(ans) ... end, ...)
Timeout values
false = wait forever (avoid in production). A number = ms; returns nil on timeout.
Rule of thumb
Use callbacks to ask. Use events to commit. Validate on both.
Trust boundary
Arguments are still client-supplied. type() / range / authority / context always.
Avoid
Callbacks inside tight loops or render threads. Callback replies treated as permission.

What you learned

  • Callbacks let the client ask the server a question and wait for the answer -- use them when the next line of code depends on server-held truth
  • lib.callback.register on the server declares the answer; lib.callback.await on the client asks the question and pauses until the reply arrives
  • The async form lib.callback() lets you receive the answer later without pausing in-line
  • Always set a timeout (like 5000 ms) and handle the nil return so the client does not hang forever
  • A callback reply is information, not permission -- the write handler must still re-validate on the server

Cleanup discipline

Spawning is easy. Cleaning up is the part that separates scripts that work from scripts that destroy servers. If you spawn entities and never delete them, the world fills up with ghost objects, entity counts climb, and eventually the server lags or crashes. This lesson teaches you when, where, and how to clean up.

Why cleanup matters

Every entity in the world costs memory and network bandwidth. FiveM servers have a hard limit on how many entities can exist at once. When you hit that limit, new entities silently fail to spawn, or the server starts dropping frames.

Entity leaks are invisible at first. Your script works perfectly with one player testing for five minutes. But on a live server with 60 players running for hours, those orphaned barrels and ghost NPCs add up fast.

Entity leaks = slow death

Lag that gets worse over time is the classic symptom. If your server runs great after a restart but gets sluggish after a few hours, someone's script is leaking entities.

Signs you are leaking

  • Rising entity count. Check with a tool or native -- the number keeps going up over time without coming back down.
  • Objects from last session. You restart a resource and the barrels you spawned earlier are still there. That means the old resource instance did not clean them up.
  • FPS drops over time. The server ran fine an hour ago. Now everyone is lagging. Classic leak behavior.
  • Spawn functions return 0. The entity limit is hit. New spawns silently fail because the world is full of orphaned entities.
Step 1

Cleanup on resource stop

Outcome: when your resource restarts or stops, every entity it created gets deleted automatically.

FiveM fires the onResourceStop event whenever a resource is stopped or restarted. This is your first line of defense.

Why check the resource name?
  • onResourceStop fires for every resource that stops, not just yours. The resourceName parameter tells you which one is stopping.
  • We compare it to GetCurrentResourceName() -- if they don't match, we bail. You only clean up your own entities.
  • Without this check, stopping any resource would wipe your entities too.
Step 2

Cleanup on player drop (server side)

Outcome: when a player disconnects, any entities tied to that player are cleaned up.

If your server-side script tracks entities per player (for example, a personal vehicle), you need to clean up when the player leaves.

When does playerDropped fire?
  • When a player disconnects, crashes, times out, or gets kicked.
  • source inside this handler is the player ID of whoever just left.
  • We look up their entities in the registry, delete them, then wipe their entry.
Step 2b

Server-side onResourceStop for server-spawned entities

Outcome: entities you created with CreateVehicleServerSetter, CreatePedServerSetter, or CreateObjectServerSetter get deleted when your resource stops.

Step 1 cleans up client-spawned entities. If your resource also spawns from the server (Lesson 24's server-authoritative pattern), you need the same hook on the server.

Why two registries?
  • Server and client run on different machines. The server has its own handle table for entities it created with *ServerSetter natives; the client has its own for things spawned with CreateObject/CreateVehicle/CreatePed.
  • Each side cleans its own. The onResourceStop event fires on both sides when the resource stops -- you just need a handler on each.
  • For entities the server spawned, calling DeleteEntity from the server is the authoritative path. Clients see the entity disappear automatically.
Network ownership and DeleteEntity

If a client currently has network ownership of an entity (vehicles often migrate to the nearest player), DeleteEntity from the server still works because the server is the network authority for entities it spawned with *ServerSetter. For entities the client created locally, the server cannot delete them -- the client must.

Step 4

The interaction-then-cleanup pattern

Outcome: you understand the spawn, interact, cleanup flow used by systems like ox_target.

In real scripts, you often spawn a prop so the player can interact with it (press E, use ox_target, etc.), and then delete it after the interaction is done. The flow looks like this:

  1. Spawn the prop (load model, create, register).
  2. Add an interaction point (ox_target, a prompt, or a keybind check).
  3. Player interacts -- whatever happens (animation, menu, event).
  4. Cleanup -- delete the prop, remove the interaction point.

This pattern keeps the world clean: objects only exist while they are useful.

Common mistakes

Only cleaning up on resource stop, but not on player drop

If a player disconnects, their personal entities (like a spawned vehicle) stay in the world. The resource is still running, so onResourceStop never fires. You need both handlers: one for the resource lifecycle, one for the player lifecycle.

Deleting entities that don't exist anymore

If you call DeleteEntity on a handle that's already been deleted, you get undefined behavior or errors. Always check DoesEntityExist(handle) first. One line of defense saves you from mysterious bugs.

Cleaning up another resource's entities

Your cleanup should only delete entities you spawned. That's what the registry is for. Never loop over all entities in the world and delete them -- you will nuke other scripts' props, vehicles, and NPCs.

Not checking resource name in onResourceStop

If you skip the if GetCurrentResourceName() ~= resourceName then return end guard, your cleanup runs when any resource stops. Some poor NPC from another script vanishes because your handler was too eager.

Stale handles in the registry

Another script (or a manual /clearprops) deletes an entity you spawned. Your registry still holds the handle as a number. DoesEntityExist returns false so DeleteEntity is skipped, but the registry entry is never cleared and the table grows forever. Fix: when cleanup walks the registry, set spawnedEntities[id] = nil regardless of whether the entity existed.

Spawning before registering

If your code is local obj = CreateObject(...) followed by other work and only later registerEntity(obj), a resource stop in that gap leaks the entity. Always register on the line right after the spawn, before any yield or callback.

Cleaning client-spawned entities from the server

Server-side DeleteEntity only works for entities the server itself created (typically via *ServerSetter natives). For entities a client spawned with CreateObject/CreateVehicle/CreatePed, the server cannot delete them -- the client that spawned them must run its own onResourceStop cleanup.

Cleanup lifecycle: three triggers

Register
Store every handle in a table the moment you create it. You cannot clean up what you did not track.
Timeout
Test props and short-lived spawns auto-delete after N seconds with SetTimeout.
Player drop
Server-side: on playerDropped, delete entities owned by or tied to that player.
Resource stop
onResourceStop with a name guard deletes everything still registered. The catch-all that prevents leaks on restart.

Cheat sheet

AddEventHandler('onResourceStop', fn)
Fires when any resource stops. Check resourceName to filter for yours.
AddEventHandler('playerDropped', fn)
Server-side. Fires when a player disconnects. source = the player who left.
GetCurrentResourceName()
Returns the name of the resource this script belongs to.
SetTimeout(ms, fn)
Run a function once after ms milliseconds.
DoesEntityExist(handle)
Always check before deleting. Returns false if already gone.
Entity registry
A table of handles. Every spawn adds to it. Every cleanup reads from it.
Interaction flow
Spawn, add target/interaction, player acts, cleanup. Objects only exist while useful.
Two registries
Client spawns go in a client-side table; server *ServerSetter spawns go in a server-side table. Each side cleans its own.
Server-side cleanup scope
DeleteEntity from the server is authoritative for entities the server itself spawned via CreateVehicleServerSetter / CreatePedServerSetter / CreateObjectServerSetter, regardless of which client owns the entity now.
NetworkRequestControlOfEntity(ent)
Client-side. Asks the network owner for control before mutating or deleting an entity that another client currently owns.

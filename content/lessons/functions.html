The mental model: what a function is

A function is a piece of code with a name on it. Instead of writing the same three lines every time you want to greet a player, you wrap those lines in a function called greet and from then on you write greet('Alex'). One name. One call. Same result, every time.

Think of it like a vending machine. The machine is the function. The coins you insert are the arguments. The can that drops out is the return value. You don't need to know how the machine works inside, you only need to know what to put in and what you get out.

FiveM note

Every native you'll ever use in FiveM, GetPlayerName, TriggerEvent, SetEntityCoords: is a function someone else wrote and named. The whole platform is a library of functions waiting for you to call them. Once you can write your own, you stop being a user of code and start being an author.

What each piece means
  • local function: you're defining a function that only exists inside this file. Same reason you write local x: you don't want to leak it into the global space. We unpack scope properly in Module 02.
  • welcome: the name you chose. Any rules that apply to variable names apply here.
  • (name): the parameter. It's a slot. When someone calls welcome('Alex'), the string 'Alex' slides into that slot and, inside the function, name holds 'Alex'.
  • return: hands a value back to whoever called the function. Without it, the function finishes silently and hands back nothing.
  • end: closes the function. Every function needs a matching end, same as if and for.
Parameter vs argument

The parameter is the name inside the function, the slot. The argument is the actual value you pass in when you call the function. In welcome('Alex'), name is the parameter and 'Alex' is the argument. Error messages use both words, so knowing the difference saves you Google searches.

The console prints nil. The function built the message fine, but it never handed it back. Lua gives you nil when you ask for a value that isn't there.

Rule of thumb: if you want the caller to use a value, the function must return it. Building it in a local variable and stopping there is the same as never building it at all, from the caller's point of view.

Common beginner mistake

Writing print('Welcome back, ' .. name) inside the function instead of returning it. That works, sometimes. But the caller can't reuse the value, can't store it, can't send it anywhere else. A function that returns a string is more useful than a function that only prints. Return first, let the caller decide what to do with it.

Notice that there's no elseif anywhere. Each return exits the function immediately, as soon as one condition matches, the function is done and no later line runs. This pattern is called a guard clause: you handle the weird cases at the top, return out fast, and leave the main logic clean at the bottom.

Why this reads better than one giant if / elseif / else
  1. Each case is short and flat, no deep indentation.
  2. The "unhappy path" (invalid input) is out of the way on line 2. The rest of the function reads as the happy path.
  3. Adding a new rule later is a new small block, not a surgery on a big chain.
Pro move

Guard-clause every function that takes user input. First line inside the body: check for the broken cases, return out if anything's wrong. You'll write clearer code, and your future self will thank you when bugs show up at 2 a.m.

Common mistakes & what the console tells you

Your variable is nil and you don't know why

You forgot return inside the function. The function ran, the work happened, but nothing got handed back. Add return before the value you want the caller to see.

Error: attempt to call a nil value (global 'greet')

You called greet('Alex') on a line above local function greet(...). Lua reads top to bottom. At the moment you called it, the function didn't exist yet. Move the definition above the call, or put the call inside another function that runs later.

You wrote function greet(name) with no local

Congratulations, you created a global. Any other resource on the server can now see and overwrite your greet. Always write local function unless you have a specific reason not to. We explain the why in Module 02, for now, treat it as a rule.

Error message talks about a "parameter" and you panic

Parameters are the slots inside the function definition. Arguments are the values you pass when calling it. If the error says "expected 2 parameters, got 1", it means the function was defined with two slots and you only filled one. Count the commas in the definition, match them with commas in the call.

What you learned

  • A function wraps code in a reusable block with a name -- define once, call many times
  • Parameters are the slots inside the definition; arguments are the values you pass in when calling
  • return hands a value back to the caller -- without it the function gives back nil
  • Early return (guard clauses) lets you handle bad inputs at the top and keep the main logic clean
  • Lua functions can return multiple values separated by commas

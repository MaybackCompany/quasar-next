HTTP requests & webhooks: PerformHttpRequest

Talking to the outside world, Discord, your panel API, a license-check service, means HTTP. CFX gives you exactly one native: PerformHttpRequest. It is callback-based, will surprise you with rate limits, and JSON requires manual encode/decode. This lesson teaches the pattern that scales from a single ban-log to a 50-server fleet.

Before you start

Tick these so the requests fly.

Mental model: HTTP is sending a letter and waiting for the postman

Every PerformHttpRequest is a letter. You drop it in the mailbox (the call returns immediately, your code keeps running) and the postman shows up later with the reply. The reply lands in your callback. That is why the function takes the callback as its first argument, the reply has nowhere else to go.

If you want the reply before moving on, wrap the call in a promise (covered in P4). For now, internalize this: the line after PerformHttpRequest runs before the response arrives.

FiveM-specific note

Server-side only. Clients cannot call PerformHttpRequest. If you need a client to trigger an HTTP call, fire a server event and let the server make the call. This is also a security boundary, webhook URLs never live in client code.

Step 4

JSON encode and decode

Outcome: you can send and receive structured data.

FiveM ships json.encode and json.decode globally on the server. They convert between Lua tables and JSON strings:

Pro tip

Always wrap json.decode in pcall. Hostile or buggy APIs will return HTML error pages, broken JSON, or empty bodies. Without pcall, your handler crashes and the player stays connected.

Do not concatenate raw query strings

URL-encode every query value. If the payload has multiple fields or free text, prefer a JSON POST body with json.encode instead of building a long GET URL by hand.

Recap 2

Bad JSON does not crash

Point your code at https://example.com (returns HTML, not JSON). Confirm pcall catches the parse error and your handler logs cleanly instead of crashing the resource.

Step 5

Profile under real load

Outcome: you can prove a webhook handler does not stall the server tick, and you can spot a synchronous response handler before it bites.

HTTP looks free because the call returns immediately. The cost hides in three places: the callback running on the next tick, JSON decode of large responses, and synchronous code that uses Citizen.Await or busy-waits on a promise. Profile all three.

5.1 profiler record on the webhook path

In txAdmin Live Console, drive load through the resource and capture 60 seconds:

Open bans.json in Perfetto. The postBan entry should be <0.05ms (it just queues the request). The callback entry on a later tick should be <1ms for a 200 OK with empty body. Anything >5ms in the callback means you are doing real work in there, move it off-thread or to a deferred queue.

5.2 resmon 1 for steady-state cost

In F8 with resmon 1 open, watch server-side ms-per-tick for your webhook resource under sustained 5 req/s load. Healthy:

  • Idle: 0.00–0.02 ms/tick (callback inactive).
  • 1 req/s: 0.02–0.10 ms/tick (callback runs once per tick).
  • 10 req/s: 0.15–0.40 ms/tick. If higher, your callback is too heavy.
  • Spike pattern: 0.05 baseline with a spike to 4–8ms once per minute means you are JSON-decoding a large response on the tick. Stream-parse or skip the body.

5.3 Webhook retry budget & circuit breaker

Recap 1 retries up to 4 times. Under a real Discord outage, that retry storm becomes a synchronous-feel pile-up because every SetTimeout resolves on the same tick window. Add a circuit breaker: stop retrying after N consecutive failures, drain the backlog when the circuit closes again.

This caps the cost of a Discord outage at 5 failed requests, then 0 for 30 seconds, instead of an unbounded retry queue.

Common mistakes

Hard-coding webhook URLs in resource files

The first time someone clones your repo, your URL is in their git history forever. Use convars and rotate immediately if a URL leaks.

Calling PerformHttpRequest on the client

Client cannot make HTTP calls. Even if it could, you would expose URLs and tokens to every player. Always proxy via the server.

Forgetting to encode the body as JSON

Passing a Lua table directly puts table: 0x7f... in the body. The server then returns 400 Bad Request and you spend an hour debugging Discord. Always json.encode first.

Missing Content-Type header on POST

Most APIs reject the request silently or treat the body as form data. Set ['Content-Type'] = 'application/json' on every JSON POST.

Treating the call as synchronous

Code on the line after PerformHttpRequest runs before the callback. If you need to wait for the response, see P4 (promises) or move the dependent logic into the callback itself.

Cheat sheet

PerformHttpRequest(url, cb, method, data, headers)
Server-only HTTP. Reply lands in cb(status, body, headers).
Discord success
HTTP 204 (No Content) is normal for webhook POST. Treat 200 and 204 as success.
JSON
json.encode(table) for body, json.decode(string) for response. Wrap decode in pcall.
Headers
Always set ['Content-Type'] = 'application/json' on JSON POST.
Rate limit
HTTP 429 with Retry-After header in seconds. Retry with SetTimeout.
Secrets
Webhook URLs and API keys live in GetConvar, never in resource files.
Async
The line after the call runs before the callback. Plan for that.

What you learned

  • PerformHttpRequest is server-only and callback-based; the response is async.
  • JSON is manual: encode the body, decode the response, always pcall the decode.
  • Discord webhooks rate-limit at ~30/min. Read Retry-After and back off.
  • Secrets go in convars (GetConvar), never in client code or resource files.
  • The line after the call runs before the callback. Use a promise wrapper if you need to wait.

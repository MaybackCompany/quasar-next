Before you start

How to read this lesson

Ignore the big API feeling. ox_target is one simple idea: attach options to a thing. Learn one shape first. Then see it on a ped, on a zone, and on another player.

The mental model: attach options to things

The main idea behind ox_target is simple: instead of writing your own distance checks and prompt text every frame, you tell the library which thing has which options. The library handles the eye, the hover, and the click.

There are four flavours of "thing" you can attach options to:

  • Models: any ped or prop with a matching model hash. Spawn 10 mechanics; they all get the same menu.
  • Specific entities: one particular vehicle, prop, or ped you've spawned. Menus are unique to that entity.
  • Global players: options on every other player's body. "Check ID", "handcuff", "give cash".
  • Zones: arbitrary 3D regions in the world: a box, sphere, or polygon. Great for robbing registers, entering buildings, turning in contracts.

Same option schema for all four. Learn one shape, then reuse it on peds, props, players, and zones.

The option schema: the shape you'll memorise

Every interaction option is a table with the same fields. Here's the full list, most are optional.

The three fields you'll touch 90% of the time: label (the text), icon (any Font Awesome class), and onSelect (the function that runs when the player clicks). The rest are gates, distance, condition, job, item, and you reach for them as the script grows.

Step 1

addModel: a merchant ped

Outcome: every spawned ped of the chosen model glows when you look at it and offers a "Buy pizza" option.

Build qu_target. Spawn a mafia boss ped in a fixed spot, then attach a target option to his model. Any ped of that model anywhere on the map will inherit the same menu, handy when you have multiple mechanics, shopkeepers, or drug dealers.

Spawn in game, walk to the Pillbox corner store, look at the clerk, the eye cursor appears with two options. Click pizza and the server event fires. No raycasting. No distance math. Just a declaration.

Why CreateThread for the ped

Model loading and entity creation are async operations, they can't live at file top level. Wrap in CreateThread so they happen after the script runs and don't block the main flow. This is a pattern you'll repeat, every FiveM script that spawns entities wraps creation in a thread.

Step 2

addBoxZone: robbable ATM spots

Outcome: a specific 3D region of the world triggers a target option, without needing an entity.

Sometimes the thing you want to target isn't a ped or prop, it's a location. A spot in an alley where drug deals happen. A tile where the laundromat laundering happens. The doorknob of a building you want players to enter. That's a zone.

Three knobs worth exploring: coords and size define the box; debug = true draws the box in-game so you can see what you're placing. groups limits the interaction to certain jobs. canInteract is a live-check function that runs every time the player hovers, use it for "should this show right now".

Two layers of gating, both necessary

canInteract and groups are UX, they hide the option. But the server must still enforce the rule when the event fires. A cheater can ignore UX; they can't ignore server validation. UX gate + server gate = a good design.

Step 3

addGlobalPlayer: actions on other players

Outcome: looking at another player offers context-specific options like "give cash" or "handcuff".

Every RP server needs player-on-player interactions. ox_target makes this trivial with addGlobalPlayer: options added to every other player's body, filtered by job and condition.

The onSelect function receives a data table that includes entity: the ped of the targeted player. Convert to server id via GetPlayerServerId(NetworkGetPlayerIndexFromPed(data.entity)). That server id is what you send to the server, which then uses it (after validation) as source's target.

Never trust the target id the client sent

The client could lie, but so could the target. The server must verify: is the sender near the target? Is the sender on duty? Does the sender have authority? Otherwise you've built a remote-handcuff exploit that works across the map. Target ids are a starting point for server checks, not a permission slip.

Step 4

Cleanup: remove when done, always

Outcome: restarting your resource doesn't leave orphaned target options hanging around.

Every addModel, addBoxZone, addGlobalPlayer call needs a matching remove on resource stop. Otherwise, every restart stacks more options on the same models, five "Buy pizza" entries, five "Chat" entries. Players hate it; debugging is worse.

Match the name fields you gave when adding. Remove takes either a single name string or a table of them. Delete any entities you spawned, orphan peds leak memory and look silly. This block of cleanup is worth writing before the feature works; it means every restart is clean.

The symmetry rule

For every side-effect your script creates on start, an option, a ped, a blip, a command, write the matching cleanup in onResourceStop. It takes 60 seconds now and saves you two hours of ghost-debugging later.

Recap 2

Trust rule refresh

A new developer builds a /givecash target option. The onSelect grabs the target's server id and sends it plus the amount to a server event. The server event takes the money from the sender and gives it to the target. Name every trust violation this design has and sketch the fix.

The answer
  1. Target id is client-supplied. The cheater sends any server id. Fix: accept the id but verify the sender is within 2 metres of the target on the server (#(senderCoords - targetCoords) < 2.0).
  2. Amount is client-supplied. A cheater sends a negative number to steal cash instead of giving it. Fix: if type(amount) ~= 'number' or amount <= 0 or amount > 100000 then return end.
  3. Sender balance unchecked. They might not have the cash. Fix: re-read Player.PlayerData.money.cash and compare.
  4. Target validity unchecked. Server id might be invalid or disconnected. Fix: if not Target then return end after GetPlayer.

Four checks for a four-line feature. Every target interaction walks this same audit path, type, range, authority, context.

Common mistakes: ox_target edition

Doing the permission check only in canInteract

canInteract is UX. A cheater can hide the check by patching the client. Server must re-validate on the event.

Forgetting to remove on stop

Every restart leaves duplicate options. A week later, players see "Buy pizza" five times on the same ped. Wire onResourceStop the same day you wire the addModel.

Debug mode left on in production

debug = true on a zone draws a visible box in-game. Great during design, terrible for players. Grep your codebase for debug = true before shipping.

Hardcoding model hashes as strings

'mp_m_shopkeep_01' as a string vs `mp_m_shopkeep_01` as a hash literal (backticks) are not interchangeable. addModel wants the numeric hash. Use backticks or GetHashKey.

Attaching options before the resource is ready

If you call addModel at file top level but ox_target hasn't finished starting, the call is a no-op. Put addModel calls after QBCore:Client:OnPlayerLoaded or wrap in CreateThread(function() ...; Wait(1000); ... end) while you're learning.

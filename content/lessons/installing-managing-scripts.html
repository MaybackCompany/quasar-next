Evaluate, install, and manage scripts safely

By the end of this lesson you will read a script before you install it, spot the three red flags that mean trouble, and know exactly which scripts are safe to run on your server.

You'll ship
A checklist you run against every script before it touches your live server.
Time
About 25 minutes to learn the process, then 2 minutes per script forever.
You'll learn
How to spot backdoors, performance killers, and dependency traps before they break your server.
Prereqs
You have installed at least one script before (Lesson 03).
In 3 bullets

You will build a script evaluation habit that catches problems early:

  • You will open the manifest and scripts before installing, looking for hidden network calls and suspicious patterns.
  • You will test every new script in a local-only server before it ever reaches your production box.
  • You will maintain a simple text file that tracks what you installed, when, and from where.

Start here

What this is
A safety process every server owner runs before installing a script.
Example: download, inspect, test locally, then deploy — never the other way around.
Where to find it
Scripts come from forums, GitHub, Tebex stores, and Discord. All sources carry risk.
Example: a popular forum script can still contain a backdoor added by a contributor.
How to use it
Run the three-step checklist on every script, no exceptions.
Example: even a script from a friend gets the same checks.
What to do now
Open the manifest of the last script you installed and audit it.
Example: check for server_script lines, HTTP calls, and unknown dependencies.

Outcome

You will audit any script's manifest and source files for common safety issues before installing it.

Why it matters

Bad scripts steal data, crash servers, open backdoors, or consume all your CPU. One unchecked script can undo months of work. A five-minute audit stops all of that.

Vocabulary

Backdoor
Hidden code that lets someone control your server without permission.
Example: a script that runs PerformHttpRequest to send data to a remote server.
Dependency
Another resource that a script requires to function.
Example: a job script depends on qb-core and will not work without it.
Server callback
A network request from client to server that waits for a response.
Example: a script uses qb-core:server:GetPlayer to fetch player data.

Minimal example

Checklist

  1. Download a script but do not extract it into your production server yet.
  2. Extract it into a _staging/ folder outside of resources/ and open the fxmanifest.lua.
  3. Read every dependency. Install missing dependencies first or skip the script.
  4. Search every .lua file for PerformHttpRequest, os.execute, and loadstring.
  5. If clean, install into a local-only test server and verify it works for 10 minutes.
  6. Only after local testing passes, copy to your production server.

Exercise

Create a script that saves the audit result. Write a server.lua that logs the result of your audit to the server console using print. Include a comment line in every Lua file that says -- AUDITED: safe, no red flags, YYYY-MM-DD.

Expected output: Server console prints [qu_safecheck] Audit complete. 0 red flags found. on resource start.

Common failures

  • Ignoring the manifest and installing anyway — the most common mistake. A dependency you missed causes silent failures days later. Always read the manifest first.
  • Running a script with PerformHttpRequest to an unknown domain — this is data exfiltration. Block the domain or delete the script.
  • Skipping local testing and deploying straight to production — a broken script crashes your live server for everyone. Always test locally.
  • Not keeping an install log — three months later you cannot remember where a script came from or why you installed it. Keep a simple text file.

Refactor

Add an audit.lua script to your server that runs on startup and checks every resource's manifest for the red-flag patterns above. Log the results to a file. Now your server audits itself every restart.

Player connection queue and deferrals

By the end of this lesson you will control exactly what happens when a player connects to your server, delay them until your checks pass, and either let them in or show them a clear reason why they cannot join.

You'll ship
A connection handler that checks a player against a ban list, waits for character data to load, and only then lets them spawn.
Time
About 30 minutes to understand and implement.
You'll learn
What deferrals are, why the queue exists, and how to write a connection handler that protects your server.
Prereqs
You have a running server and understand what RegisterNetEvent does (Lesson 19 of Module 06 helps).
In 3 bullets

You will build a connection queue handler that does this:

  • A player connects. Your script defers them, meaning they see a loading screen while your checks run.
  • Your script checks a ban table, waits for database data, and verifies the player is allowed to join.
  • If everything passes, the deferral is resolved and the player spawns. If not, they see a message explaining why.

Step by step

Lesson 05: Player connection queue and deferrals

Start here

What this is
A connection gate that runs before a player fully enters your server.
Example: every FiveM server runs a deferral system. You are writing yours.
Where to find it
In a server-side script that uses the playerConnecting event.
Example: resources/[system]/qu_connect/server.lua handles all connections.
How to use it
Register the playerConnecting event, defer the player, run checks, then allow or drop.
Example: defer, check ban list, check whitelist, then Deferrals.done().
What to do now
Write a minimal deferral script that holds every player for 2 seconds, then lets them in.
Example: the smallest working deferral is under 20 lines.

Outcome

You will write a connection handler that defers every player, runs a simple check (a table lookup for banned identifiers), and either admits or drops them.

Why it matters

Without a connection gate, banned players reconnect with a new name. Character data loads after the player spawns, causing naked characters and broken inventories. Deferrals fix all of this at the earliest possible moment.

Vocabulary

Deferral
A promise to FiveM: "Wait. Do not let this player spawn yet. I have checks to run."
Example: Deferrals.defer() pauses the connection. Deferrals.done() resumes it.
Identifier
A unique string that identifies a player across sessions. FiveM provides several.
Example: license:abc123 is a Rockstar license. steam:1100001 is a Steam ID.
playerConnecting
The server event that fires before a player fully joins. You hook into it to run checks.
Example: AddEventHandler('playerConnecting', function(name, setKickReason, deferrals) ... end).

Checklist

  1. Create resources/[system]/qu_connect/ and add fxmanifest.lua with server_script 'server.lua'.
  2. Paste the server.lua code above into the resource folder.
  3. Add ensure qu_connect as the very first ensure line in your server.cfg.
  4. Restart the server and join with a clean account. Confirm the admit message prints.
  5. Add your own license to the BANNED_LICENSES table temporarily and confirm you are blocked.
  6. Remove yourself from the ban table, restart, and confirm you can join again.

Exercise

Add a whitelist check. Create a WHITELISTED_LICENSES table with your own license. Modify the handler so that if the whitelist table is not empty, only whitelisted players are admitted. Test by having a friend try to join.

Expected output: Your friend sees "You are not whitelisted on this server." You join normally.

Common failures

  • Deferrals.defer() was not called means you tried to call deferrals.done() or deferrals.update() before deferring. Always call deferrals.defer() first.
  • GetPlayerIdentifiers returned an empty table means you called it too early. Always Citizen.Wait(0) after deferring to give FiveM time to populate identifiers.
  • Player stuck on loading screen forever means you never called deferrals.done(). A deferral must always resolve. Add a timeout as a safety net.
  • Two deferral scripts conflict — only one resource should handle playerConnecting with deferrals. A framework like QBCore already does this. Consolidate into one connection handler.

Refactor

Extract the ban and whitelist checks into separate functions: checkBan(license) and checkWhitelist(license). Add a TIMEOUT_SECONDS config that auto-drops the player if deferrals do not resolve within 30 seconds.

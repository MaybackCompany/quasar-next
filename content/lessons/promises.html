Promises & async: turning callbacks into linear code

FiveM is full of callback APIs: PerformHttpRequest, the older MySQL-Async, NUI callbacks. Three of them in a row and your code looks like a pyramid. Promises flatten the pyramid. Combined with Citizen.Await, you get top-to-bottom code that feels synchronous while staying non-blocking.

Mental model: a promise is a receipt slip

You walk into a busy shawarma shop. The cook hands you a paper number. Sometime later, your order is up. Right now, the slip has no food on it, just a guarantee that food is coming. :resolve(value) is the cook calling your number with the plate. :reject(err) is the cook saying "we are out". Citizen.Await(p) is you sitting at the counter waiting for your number to be called.

The shop never stops cooking other orders while you wait, that is the whole point of async. Other handlers, other threads, other players keep moving. Your one thread parks until your value is ready.

FiveM-specific note

FiveM ships a global promise library on both client and server. oxmysql uses it under the hood, that is what :await() wraps. You get the same primitives for any other callback API.

The three primitives

Outcome: you can hand-roll a promise from scratch.

Three things to internalize: promise.new() creates the slip, :resolve(v) fills it, Citizen.Await(p) blocks this thread only until it is filled.

Step 2

Wrap a callback API as a promise

Outcome: you turn any callback function into something you can Await.

Why pcall wraps the Await

If the promise rejects, Citizen.Await throws the rejection value as an error. Without pcall, your handler crashes. With it, you can branch on success and capture the error message.

Common confusion

Calling :resolve twice on the same promise is a no-op on FiveM, the first one wins. Do not rely on this. Make sure your callback exits after the first resolve or reject with an explicit return.

Step 5

Profile async paths under real load

Outcome: you can spot a promise leak, a Citizen.Await deadlock, and an unhandled rejection from the resmon graph alone.

5.1: Promise allocation cost & :next() chain depth

One promise.new() is cheap (~600ns on a modern host). One thousand per tick is not. Run profiler record 60, open the trace in Perfetto, and search for your wrapper function. Two patterns kill perf:

  • Hot-loop allocation. Building a fresh promise inside a per-tick loop ("scan players every frame and wrap each ped check") allocates one promise per player per tick. At 64 players that is ~3,800 promises/sec just to ask the same question. Cache the result for 1–5 seconds and only re-wrap on miss.
  • Deep :next() chains. Each :next() allocates an inner promise. A 6-step chain allocates 6 promises plus closures for every call. Past 3–4 chained steps, swap to Citizen.Await with linear code, same wall-clock time, fewer allocations, far easier to read in the trace.

Budget: any single promise wrapper should round-trip in <0.05ms server-side excluding the underlying I/O. If your trace says 2ms inside the wrapper itself, you are doing JSON encoding or table-cloning on the path; pull that work out.

5.2: Spotting a promise leak in resmon 1

A leaked promise is one that is created and never resolved or rejected. The thread that Awaits it parks forever, and the closure plus its captured upvalues stays referenced. resmon 1 shows two tells:

  • Memory monotonically rising by 50–500 KB/min while CPU stays flat. No GC pressure brings it down.
  • Thread count climbing in the server console (threads command if you have a debug build). Each parked CreateThread + leaked promise is one stuck thread.

Without the timeout helper, a Discord outage or a slow third-party API turns one HTTP call into a permanent thread. With it, the rejection bubbles to pcall in 5 seconds and the thread exits cleanly.

Common mistakes

Awaiting at file top level

Citizen.Await needs a thread context. Top-level await crashes the resource. Wrap in CreateThread or call from inside an event handler.

Forgetting to return the promise from your wrapper

If your http function builds a promise and forgets to return p, callers get nil and Citizen.Await(nil) blocks forever.

Resolving inside the wrong scope

Capturing p in a closure works. Reusing one outer promise across multiple operations does not, you can only resolve once. Build a fresh promise per call.

Using Wait(0) as a promise substitute

It looks similar but is not. Wait(0) yields one tick; it does not wait for an event. If your callback fires after 200ms, Wait(0) returns long before. Use a real promise.

Not handling rejections

An unhandled rejection bubbles as a Lua error and crashes your handler. Always wrap Await in pcall or use a :next(onSuccess, onFailure) chain that handles both branches.

Cheat sheet

promise.new()
Returns a fresh, unfulfilled promise.
p:resolve(value)
Fulfils the promise with a value. First call wins.
p:reject(err)
Rejects with an error. Await on a rejected promise throws.
Citizen.Await(p)
Blocks the current thread until p settles. Returns the value or throws.
Wrapping a callback
Inside the callback, call p:resolve on success, p:reject on failure. Always return p.
Parallel
Kick off all promises first, then Await each. Total time = slowest, not sum.
Errors
Wrap Citizen.Await in pcall if you might reject.

What you learned

  • A promise is a receipt for a value that will arrive later.
  • Wrap any callback API by creating a promise and resolving/rejecting from the callback.
  • Citizen.Await turns async into linear top-to-bottom code, scoped to one thread.
  • Run independent calls in parallel by collecting promises first, then awaiting them.
  • Always handle rejections, either with pcall or a :next chain.

Anti-cheat evaluation: what to use and what to skip

By the end of this lesson you will pick the right anti-cheat solution for your server, understand what each option actually protects, and avoid the ones that just waste CPU.

You'll ship
A one-page anti-cheat decision document with your server's exact protection layers.
Time
About 30 minutes.
You'll learn
How to evaluate anti-cheat solutions by what they actually protect, not by their marketing.
Prereqs
You have a running FiveM server with at least one script installed.
In 3 bullets

You will create an anti-cheat evaluation matrix:

  • List every protection layer your server needs — event security, resource integrity, injection prevention, speed/reward validation.
  • For each layer, decide whether server-side code, a free solution, or a paid solution is the right choice.
  • End with a single page that tells you exactly what is protected and what is not — no gaps, no overlaps.

Step by step

Lesson 28: Anti-cheat evaluation matrix

Start here

What this is
An anti-cheat is not one tool — it is layers of protection.
Example: event security stops exploits; resource integrity stops tampering; injection detection stops external tools.
Where to find it
Start with server.cfg and your own scripts before adding external tools.
Example: sv_resourceNameWhitelist is free and catches 30% of common attacks.
How to use it
Add layers one at a time. Test each layer before adding the next.
Example: enable event security, verify nothing breaks, then add resource integrity.
What to do now
Build the matrix — list your layers and decide what fills each one.
Example: one server ended up with 5 layers and spent $0 on anti-cheat.

Outcome

You will write one evaluation matrix that shows every protection layer on your server.

Why it matters

A server without a clear anti-cheat plan gets attacked, bans the wrong people, and loses players who did nothing wrong.

Vocabulary

Event security
Blocking or validating events that clients should never trigger.
Example: a client should never fire esx:giveMoney to themselves.
Resource integrity
Making sure the files running on your server are the files you approved.
Example: checking file hashes against a known-good list.
Injection detection
Finding external programs that modify the game client.
Example: detecting a DLL that gives wallhacks or speed boosts.

Minimal example

Expected output: [SECURITY] Layer 1: Event security active — 5 events blocked

Checklist

  1. Create evaluation.txt with these headings: Layer, Threat, Solution, Cost, Verified.
  2. Fill in Layer 1 (Event Security) with your server's blocked events list.
  3. Fill in Layer 2 (Resource Integrity) — are you using sv_resourceNameWhitelist?
  4. Fill in Layer 3 (Exploit Prevention) — rate limiting, cooldowns, server-side validation.
  5. Fill in Layer 4 (Injection Detection) — evaluate whether you need a paid solution here.
  6. Review the matrix — every threat should have at least one protection layer.

Exercise

Add two more events to the BLOCKED_EVENTS list that are specific to your framework. Run the server and verify they appear in the console count.

Expected output: [SECURITY] Layer 1: Event security active — 7 events blocked

Common failures

  • Blocking legitimate events breaks core features means you blocked an event that your framework needs. Test each event one at a time.
  • Anti-cheat causes 10-20% CPU increase with no visible benefit means you are running a bloated solution. Remove it and add simpler layers.
  • Players report false positives and get banned unfairly means your detection thresholds are too aggressive. Tune them or switch to manual review.
  • Cheaters still get through despite multiple layers means your layers are not complementary. Check that each layer covers a different threat.

Refactor

Move the BLOCKED_EVENTS table into a separate shared/security_config.lua file so both client and server can reference it.

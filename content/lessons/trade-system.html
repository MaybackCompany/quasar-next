Player-to-player trading system

By the end of this lesson you will have a working trade system where two players can exchange items and money safely, with confirmation from both sides.

Prereqs

In 3 bullets

You will build a trade system:

  • Player A types /trade 42 to send a trade request to Player 42.
  • Player B accepts. Both players add items and money to the trade window.
  • Both confirm. The server validates everything and executes the swap atomically.

Step by step

Lesson 50: Player-to-player trading

Start here

What this is
A trade is a swap of items and money between two players that both must confirm.
Example: Player A gives $5000 for Player B's weapon — both must agree before it happens.
Where to find it
All trade logic runs on the server. The client only shows the UI and sends confirmations.
Example: the server checks that both players still have the items before executing the trade.
How to use it
Request → Accept → Offer → Confirm → Execute. Five phases, each server-validated.
Example: if a player logs out during the trade, the server cancels it and returns items.
What to do now
Build the five-phase trade flow and test it with two players.
Example: trade a sandwich for $100 and verify both inventories update correctly.

Outcome

Two players trade items and money safely with server-authoritative validation.

Why it matters

Player trading is the backbone of any economy server. Getting it wrong means duplicated items, lost money, and angry players.

Vocabulary

Trade session
The state of an active trade between two players, stored on the server.
Example: {playerA=1, playerB=42, offers={{}, {}}, confirmed={false, false}}
Atomic execution
Both sides of the trade happen together or not at all.
Example: if money transfer succeeds but item transfer fails, the money is refunded.
Server-authoritative
All validation runs on the server — never trust the client about what a player has.
Example: the server checks the database, not the client's claims.

Checklist

  1. Create fxmanifest.lua with server and client scripts.
  2. Create server.lua with the trade state machine (5 phases).
  3. Create client.lua that shows NUI dialogs for each phase.
  4. Test with two players: request, accept, offer items, both confirm, verify execution.
  5. Test edge case: one player disconnects mid-trade — verify cleanup.
  6. Test edge case: try to trade an item you do not have — verify server rejects it.

Exercise

Add a trade timeout: if neither player confirms within 60 seconds, cancel the trade and notify both players.

Expected output: After 60 seconds of inactivity, both players see [TRADE] Trade cancelled — timed out.

Common failures

  • Items duplicated after trade means you are not validating inventory on the server before execution. Always check the database.
  • Trade stuck in 'offering' phase forever means a player disconnected and the session was not cleaned up. Add a disconnect handler.
  • Money lost on one side but items not received on the other means the execution is not atomic. Wrap transfers in a single transaction.
  • Player can trade with themselves means the target validation is missing. Check that source ~= target before creating a session.

Refactor

Extract the trade validation and execution into separate functions so you can call executeTrade(tradeId) from the confirm handler and the timeout handler.

FiveM error catalog: decode any red line

Every FiveM developer sees red lines in the console. This lesson catalogs the 10 most common errors, explains what each one actually means, and gives you the fix in one line.

Prereqs
You have started a FiveM server at least once and seen at least one error.
In 3 bullets

By the end of this lesson you will know how to fix:

  • attempt to call a nil value — the most common Lua error. Means you called a function that does not exist.
  • attempt to index a nil value — you tried to read a field from something that is nil.
  • SCRIPT ERROR: @resource/file.lua:42 — the error is in that resource, that file, line 42.

Why it matters

A single red line can crash your entire server. Most errors are one typo away from working code. Knowing the catalog means you fix things in seconds instead of hours.

Vocabulary

Stack trace
The list of files and line numbers showing where the error happened.
Example: stack traceback: server.lua:12 in function 'handler'.
Nil
Lua's word for "nothing" or "does not exist".
Example: a variable that was never set is nil.
Callback
A function passed to be called later, often the source of nil errors.
Example: the function argument to RegisterNetEvent.

Checklist

  1. Read the error message out loud. Every word matters.
  2. Find the resource name after @. That is the broken resource.
  3. Find the file and line number after the colon. That is the exact line.
  4. Open that file and go to that line.
  5. Match the error text to an entry in this catalog.
  6. Apply the fix. Restart the resource. Confirm the red line is gone.

Exercise

Common failures

  • attempt to call a nil value — you called a function that is not defined yet, or you misspelled its name.
  • attempt to index a nil value (local 'x') — variable x is nil and you tried to do x.something.
  • SCRIPT ERROR: citizen:/scripting/lua/scheduler.lua — this is not your bug. It means a thread crashed inside a FiveM system call. Look at the next line for the real resource name.
  • Could not find resource — the resource folder does not exist, or you misspelled it in server.cfg.

Refactor

Wrap every callback in pcall during development so errors log the function name without crashing the resource.

Cheat sheet

The 10 most common FiveM errors

  • 1. attempt to call a nil value. Misspelled function name or wrong side (client vs server).
  • 2. attempt to index a nil value. You tried table.field but table is nil.
  • 3. No such export. The resource that exports the function is not started or not present.
  • 4. Could not find resource. The folder name in resources/ does not match the ensure line.
  • 5. Invalid client object. You called a client native on the server side.
  • 6. TriggerServerEvent from server. You called a server-to-server trigger on the wrong side.
  • 7. SQL error: no such table. The database migration did not run.
  • 8. Exceeded stack depth. Infinite recursion — a function calls itself with no exit condition.
  • 9. Duplicate resource. Two folders have the same resource name.
  • 10. Script not a function. You passed a string where a function was expected.

The big idea, in one minute

FiveM errors are not random. Every red line follows the same pattern: error type, file location, and the exact line where it happened. Once you learn to read the pattern, you skip the panic and go straight to the fix.

Think of a car's check-engine light.

The light itself tells you nothing useful. But a mechanic reads the error code and knows exactly which sensor reported what. Your console is the same. The red line is the error code. This catalog is the mechanic's manual.

When you see attempt to call a nil value (global 'healPlayer'), the car is telling you "I tried to call a function named healPlayer but I cannot find it anywhere." That is all. Go define it.

Panic way and calm way, side by side

Most beginners see a red line and immediately restart the server, reinstall everything, or copy-paste the error into Discord. None of those fix it.

Panic way

You see red, restart the server, reinstall FiveM, and ask Discord "why does my server crash?" with no error text.

Restarting the server sometimes hides the error for five minutes, but it comes back. Asking for help without the error text gets you ignored.

Calm way

You read the error, match it to this catalog, and fix the actual line of code.

The fix took 30 seconds. You did not restart the server. You did not ask anyone. You read the error, found the line, and fixed the typo.

Step 1

Read the error from bottom to top

The most useful line is usually the last one before the stack trace.

  1. Open your F8 console or server terminal.
  2. Find the red SCRIPT ERROR line.
  3. Read the last sentence before stack traceback:. That is the error.
  4. Note the resource name after @ and the file:line after the colon.
  5. Ignore the stack trace for now. Fix the error first, then check the trace if needed.
Step 2

Match the error to this catalog

90% of errors fit into one of the 10 patterns below. Read the error, find the keywords.

  1. Does the error say nil value? It is #1 or #2. Find the variable that is nil.
  2. Does it say No such export? It is #3. The dependency resource is not started.
  3. Does it say Could not find resource? It is #4. Check the folder name and server.cfg.
  4. Does it say Invalid client object? It is #5. You called a client native on the server.
  5. Once you match the pattern, apply the fix described in that catalog entry.
Step 3

Open the file, go to the line, fix it

The error message tells you exactly where to look.

  1. The format is @resource/file.lua:42. Open resources/resource/file.lua.
  2. Go to line 42. Most editors show line numbers. If yours does not, enable them.
  3. Read the line. The error is on that line or the line just before it.
  4. Apply the fix from the catalog entry. Save the file.
  5. In the server console, type ensure resource to reload just that resource. Do not restart the whole server.
  6. Confirm the error is gone. If it moved to a different line, repeat Step 3 with the new line number.
Recap

Three tests, in order

Create errors on purpose so you learn to read them. Fix each one before moving on.

  1. Test 1, call a missing function.
    • Action: create a command that calls banana() without defining it. Run the command.
    • Expected: attempt to call a nil value (global 'banana').
    • How to verify: the error message contains "banana" and the line number you wrote.
  2. Test 2, index a nil table.
    • Action: write local x = nil; print(x.name) in a command and run it.
    • Expected: attempt to index a nil value (local 'x').
    • How to verify: the error names the variable "x" and says you tried to index it.
  3. Test 3, use a client native on server.
    • Action: in server.lua, call SetEntityHealth(1, 200) and run the resource.
    • Expected: attempt to call a nil value (global 'SetEntityHealth').
    • How to verify: the error says SetEntityHealth is nil because it only exists on the client.
Why do we say "nil value" instead of "does not exist"?

Because Lua's type system has a single value for "nothing" called nil. When the runtime looks up a name and finds nothing, it gets nil. When you try to call nil as a function, Lua tells you what you tried to do (call) and what the value actually was (nil).

The rules you just followed

  • Read the error message from bottom to top. The last line is the actual error.
  • Find the resource, file, and line number before doing anything else.
  • Match the error text to a known pattern instead of guessing.
  • Fix one error at a time. A second error is often caused by the first.
  • Use ensure resource instead of restarting the server.

What you learned

  • Every FiveM error follows the pattern: error type, resource, file, line number.
  • attempt to call a nil value means the function name is wrong, missing, or on the wrong side.
  • attempt to index a nil value means the variable you are reading from is nil.
  • The resource name after @ tells you which resource to open.
  • Use ensure resource to reload one resource instead of restarting the whole server.

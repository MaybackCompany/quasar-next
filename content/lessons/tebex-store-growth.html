Common failures

  • webhook never fires means the Tebex secret key is missing or the endpoint URL is wrong in your dashboard.
  • player not online means the webhook tried to deliver to a player who disconnected between payment and delivery.
  • duplicate delivery means you did not check for an already-processed transaction ID before granting items.
  • package ID mismatch means your handler maps the wrong package ID to the wrong reward table.

Refactor

Extract the item-granting logic into grantPackage(playerId, packageId) so every package uses the same delivery pipeline.

Cheat sheet

Tebex store growth rules

  • Three tiers. Starter, Core, Premium. Never more than 5 packages at launch.
  • Auto-deliver. The webhook must fire within 3 seconds of payment confirmation.
  • Log everything. Every purchase writes to console and a file. You cannot debug what you did not record.

The big idea, in one minute

A Tebex store is a vending machine. The player puts money in (via PayPal, card, or crypto). Tebex takes the payment and rings a bell (the webhook). Your server hears the bell and drops the item into the player's inventory. If the bell never rings, the player paid for nothing and will chargeback within an hour.

Think of a food truck.

The customer pays at the window (Tebex checkout). The cashier shouts the order to the cook (your webhook handler). The cook plates the food and hands it to the customer (in-game delivery). If the cook does not hear the order, the customer stands there hungry and angry.

Your webhook handler is the cook. It must listen every time, confirm the order ID, and plate the exact item the customer bought.

Bad way and good way, side by side

Most beginners wire the webhook without validating the transaction. That is how you give free items to anyone who finds your endpoint.

Don't do this

The webhook trusts every incoming request. Anyone can POST to your endpoint.

Anyone who discovers the endpoint URL can send fake payloads. No secret is checked. No transaction ID is stored. Duplicate purchases grant duplicate items forever.

Do this

Verify the secret, deduplicate the transaction, and grant only after both checks pass.

The secret blocks outsiders. The transaction table prevents duplicates. The player existence check avoids delivery to disconnected players.

Step 1

Create your Tebex store and packages

You are creating 3 packages that form a pricing ladder.

  1. Log into dashboard.tebex.io and create a new store linked to your FiveM server.
  2. Create package "Starter": $5, contains $25k in-game cash and a welcome vehicle.
  3. Create package "Core": $15, contains $75k cash, a vehicle, and a VIP tag.
  4. Create package "Premium": $35, contains $200k cash, 3 vehicles, VIP tag, and priority queue.
  5. Save all three packages. Do not add more yet — three is the sweet spot.
Step 2

Write the webhook handler

The handler is a single server-side file that listens for Tebex events.

  1. Create the folder resources/qu_tebex and file server.lua.
  2. Paste the minimal example from above, replacing the grant comment with real item logic.
  3. Set TEBEX_SECRET to match the secret in your Tebex dashboard settings.
  4. Add ensure qu_tebex to server.cfg and restart.
Step 3

Test with sandbox mode

Never test a store with real money first.

  1. In the Tebex dashboard, enable Sandbox Mode under Store Settings.
  2. Use the test purchase flow to buy the Starter package.
  3. Watch the server console. You should see [QUIET][tebex][deliver] with your player ID.
  4. Check your in-game inventory. The cash and vehicle must appear.
  5. Repeat for Core and Premium. Fix any missing items before disabling sandbox.
Recap

Three tests, in order

Run each test, look at what happens, and confirm the expected line in the console.

  1. Test 1, the happy path.
    • Action: buy the Starter package in sandbox mode.
    • Expected: items appear in your inventory within 5 seconds.
    • How to verify: console prints [QUIET][tebex][deliver] with your player ID.
  2. Test 2, duplicate prevention.
    • Action: resend the same webhook payload manually with curl.
    • Expected: no duplicate items are granted.
    • How to verify: console prints the delivery only once. The second request is ignored.
  3. Test 3, wrong secret.
    • Action: send a webhook payload with a wrong secret field.
    • Expected: the handler returns early, no items granted.
    • How to verify: console stays silent. Inventory is unchanged.
Why does the secret live on the server and not in a client file?

Because anything on the client can be read by a player with a resource viewer. The secret stays on the server, which only you control.

The rules you just followed

  • Every package has a clear tier: Starter, Core, Premium. No confusion.
  • The webhook handler checks the secret before doing anything else.
  • Every transaction ID is recorded to prevent duplicate deliveries.
  • Player existence is confirmed before granting items.
  • Sandbox mode is used for every test. Real money is never risked.

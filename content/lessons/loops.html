Loops: for, while, and when to stop

If you copy-paste the same line over and over, you probably need a loop. A loop runs a block of code again and again until you stop it. In FiveM, knowing when to stop matters a lot, because a bad loop can hang a server. By the end you will know for loops, while loops, and the safety habit that prevents your first crash.

Before you start

You should be fluent with everything up to Lesson 03. If any of these feel shaky, go back five minutes, loops stack on top of this stuff, and gaps show up fast.

How to read this lesson

Do not treat loops like a giant topic. Learn them in this order: count with for, repeat with while, stop with break, then learn the FiveM safety rule about Wait.

Bridge note: Wait comes back early

Wait and CreateThread are taught fully in the required thread bridge. For now, copy the safe pattern exactly: any long-running while true loop must include Wait(ms) inside the loop body.

The mental model: why loops exist

You want to print "T-minus 5", "T-minus 4", "T-minus 3", "T-minus 2", "T-minus 1". You could write five print statements. It works. Now do it for 100 players, or 1000 inventory slots, or every frame the game runs. Suddenly copy-paste is a losing strategy.

A loop is the fix. You describe the pattern once, tell Lua how many times to repeat it, and step back. Two flavors matter for now:

  • Numeric for: "run this block X times, counting from A to B". You know the count up front.
  • while: "keep running this block as long as a condition stays true". You don't know the count up front.

There's a third kind called generic for (pairs and ipairs) that walks across tables. Tables come in Lesson 06, so we're leaving that tool in the drawer until then.

FiveM note

Loops drive every tick-based system in FiveM: hunger ticking down, a speedometer updating, a blip refreshing, a queue checker polling. They're the heartbeat of a server. But Lua runs on the game's main thread, so a loop that never stops or never yields will freeze the server for every player on it. That rule lives at the bottom of this page. Read it twice.

What every piece means
  • for: the keyword that starts the loop.
  • i: the counter variable. Any name works (n, step, player), but i is the tradition. It only exists inside the loop.
  • = 1, 5: the start and stop values. The loop begins at 1, ends at 5, inclusive on both ends.
  • do … end: the block that runs each pass. Everything between them executes once per number.

Step values in the wild

Need to loop over every fifth value? for i = 0, 100, 5 do. Every other index? for i = 1, 20, 2 do. The step is your power tool, most devs forget it exists and write needless math inside the body instead.

Recap 1

How many times does this run?

Before you scroll, answer in your head:

How many lines does this print?

Ten. Lua's numeric for is inclusive on both ends, 1 counts and 10 counts. If you're coming from JavaScript or Python and expecting nine, burn this into memory now: inclusive on both ends.

What does for i = 10, 1 do print(i) end print?

Nothing. The default step is +1, so Lua tries to count up from 10 to 1, it's already past the stop value on pass one, so the body never runs. To count down you must pass -1 as the step: for i = 10, 1, -1 do.

Line by line
  1. local i = 5: you create the counter yourself, outside the loop, because you'll need it before the first check.
  2. while i >= 1 do: the condition is checked before each pass. If it's false, the loop stops immediately.
  3. print('T-minus ' .. i): the body, same as before.
  4. i = i - 1: you advance the counter. while will not do this for you. Forget this line and your loop runs forever.
  5. end: closes the block. Lua jumps back to the while line and checks the condition again.
When to pick which

Use for when you know the count up front, "loop over 10 players", "iterate from 1 to 100". Use while when the stop condition depends on state, "keep polling until the player spawns", "drain the queue until it's empty". Picking the wrong one usually leaves a hint: if you're writing while and counting manually from 1, you probably wanted for.

Version B never decrements n. The condition n > 0 stays true forever, so it prints 10 forever and freezes the server. The only difference between a good while and a catastrophic one is whether the body changes the thing the condition is testing.

The FiveM rule: always Wait inside while-true loops

This is the single most important FiveM foot-gun, and every developer meets it the hard way at least once. Read it now, save yourself the crash report later.

Why threads exist in FiveM (60-second version)

FiveM does not run each script on its own CPU core. It runs all Lua on a single shared runtime that takes turns. Each script gets a slice, does some work, then must hand the slice back so the next script (and the game itself) can run. That handoff is called yielding. The function that yields is Wait(ms). The container that lets you write code that yields and resumes is called a thread, and you create one with CreateThread(fn).

You will not need to write CreateThread in this lesson, your server.lua body already runs in a thread that FiveM made for you. But the moment you write while true do … end inside that thread, the same rule applies: yield with Wait, or the runtime never gets its slice back.

You may see Citizen.CreateThread and Citizen.Wait in older code. They're the same functions under the old name. The official docs now use CreateThread and Wait. Use the short forms.

The FiveM rule, always Wait inside while-true loops

When your loop runs without yielding, nothing else runs: not other scripts, not network events, not the game's own tick. A loop with no yield starves the whole system. while true do print('hi') end on the server freezes it for every player. On the client, it freezes the game.

The fix is one line: Wait(ms). It pauses the current thread, lets the runtime do everything else, and resumes your loop after that many milliseconds. If you ever write while true, you must call Wait inside it.

Wait is a FiveM function, not part of plain Lua, you won't find it in Lua documentation. It only exists because FiveM gives it to you. The thread bridge lesson covers CreateThread in depth: what Wait(0) does (one frame), why most polls use Wait(500) or Wait(1000), and how multiple threads share the same runtime. For today, one rule: no bare while true.

Common mistakes & what the console tells you

Server freezes solid, while true do end with no Wait

This is the big one. Your console stops scrolling, players disconnect, txAdmin flags the resource. The runtime is trapped inside your loop. Fix: add Wait(ms) inside the body, Wait(1000) for slow polls, Wait(0) for every-frame loops. Never ship a while true without one.

Off-by-one, you got one more (or one fewer) pass than expected

Lua ranges are inclusive on both ends. for i = 1, 10 runs 10 times, not 9. for i = 0, 9 runs 10 times, not 10 with "index 10". Coming from JavaScript's i < 10 style? Write the count in your head before hitting save.

Error: 'end' expected (to close 'for' at line N)

You opened for ... do or while ... do and never wrote the matching end. Every loop block needs its own end. Count them: one do up top, one end at the bottom.

Loop never stops, you forgot to change the counter

Classic while trap: you test while i > 0 do but inside the body you decrement n instead of i, or you don't decrement anything at all. The condition stays true forever. Every while body must include the line that moves the counter toward the stop condition. Read your loop aloud: "while i is greater than zero, ..., then i gets one smaller."

Cheat sheet: keep this open next lesson

for i = 1, 10 do … end
Runs 10 times, i goes 1 → 10. Both ends inclusive.
for i = 10, 1, -1 do … end
Counts down. Third number is the step.
while cond do … end
Repeats while cond is truthy. You manage the counter.
break
Exits the current loop immediately.
Wait(ms)
FiveM-only. Pauses the loop for ms milliseconds. Required inside while true.
Inclusive ranges
Lua's numeric for includes both start and stop values. 1 to 10 = 10 passes.

What you learned

  • A numeric for loop counts from a start value to a stop value -- both ends are inclusive in Lua
  • The third number in for i = start, stop, step controls how the counter changes each pass (use -1 to count down)
  • A while loop keeps running as long as a condition stays true -- you must change the counter yourself
  • break exits a loop immediately when you have found what you need
  • In FiveM, every while true loop must contain Wait(ms) or it will freeze the server

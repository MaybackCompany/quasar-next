Before you start

Five minutes of setup saves you from the "why isn't anything changing" trap that eats beginners alive.

How to read this lesson

This page is really one small workflow: edit, save, ensure, read the console, test again. Learn the four commands first. Then learn where you type them. The drills only exist to make the loop feel normal.

The mental model: two consoles, one brain

FiveM gives you two consoles. One belongs to the server. One belongs to the player. That is the whole idea.

The server console is the black terminal or txAdmin live console. Anything from a server_script prints there. The F8 console is inside the game. Anything from a client_script prints there.

If you print on the server and stare at F8, you will think nothing happened. If you print on the client and stare at the server console, same problem. The fix is simple: before you check logs, ask which side the code ran on.

FiveM note

Professional scripters keep both consoles visible on the same screen, server console on one monitor, F8 toggled in-game on the other. When a feature spans both sides, you want to see the handshake land in real time. This lesson drills the server console; F8 gets proper attention when we cross into Module 03.

Step 1

The four commands you'll type forever

Outcome: you know what each command does and when to reach for it.

Four commands. Learn them once, use them every day for the rest of your scripting life.

  • ensure resource_name: starts if stopped, restarts if running. Always does the right thing. Your default.
  • start resource_name: starts if stopped. Errors out if the resource is already running.
  • stop resource_name: stops a running resource. Errors if it wasn't running.
  • restart resource_name: stop plus start in one shot. Errors if the resource wasn't already running.
Rule of thumb

Use ensure by default. The other three exist for moments when you want the command to fail loudly if the state isn't what you expected, useful in scripts and automation, rare during hand-typed development. Ninety percent of what you type this year will be ensure.

Step 2

Where you type those commands

Outcome: you know the three places a command can run, and which one to use.

Three surfaces accept the same command. Pick whichever you have in front of you.

  • txAdmin Live Console: open txAdmin in your browser, go to the Live Console tab, type the command in the input at the bottom, press Enter.
  • Raw FXServer window: if you run the server by launching FXServer.exe directly, you get a black terminal. Type the command at the prompt and hit Enter.
  • In-game F8 console: press F8 while connected to your own server as an admin or in a dev environment. Type the command, hit Enter. Only works if you have server-console permissions.
Tip

Muscle memory for the typed command pays off long-term. Every dev environment on earth has a console. Not every one has a polished UI. Learn to type ensure qu_hello blindfolded and you'll move at the same speed whether you're on your laptop, a teammate's machine, or SSH'd into a bare VPS.

Step 3

server.cfg: the startup list

Outcome: you understand that every ensure line in server.cfg fires at boot, and order matters.

Your server's server.cfg file lives at the root of the server folder. When the server starts, it reads this file top to bottom and runs every line. Any line that says ensure something will fire, which is how your resources actually come online at boot.

Order matters: a worked example

Say your qu_hello client uses lib.notify from ox_lib. The first line of your script calls into another resource. If that resource isn't up yet, the function doesn't exist and your script dies on its first breath.

Flip the order. The library boots first, registers its globals and exports, and then your script can use them.

Same files, same code, no error. The cfg is the boot sequence and you control it line by line.

The fxmanifest dependency entry from Lesson 07 is the safety net: even if someone reorders the cfg badly, FiveM refuses to start qu_hello until ox_lib is up. Use both. The cfg gets the order right at boot; the manifest stops the server from launching with a broken setup later.

Watch out

Editing server.cfg while the server is running does not re-read the file. You added a new ensure line and it's not starting? You need a full server reboot, close and relaunch FXServer, or hit Restart in txAdmin. A single-resource restart won't do it.

Rule of thumb for ordering

Libraries first (ox_lib, oxmysql, qb-core or your framework), then frameworks-on-frameworks (qb-target, qb-menu), then your gameplay resources (qu_hello, jobs, garages). Anything that uses something else goes below it.

Recap 1

Pick the right command

Before you touch the keyboard, answer these in your head:

You just saved a change to qu_hello/client/main.lua. The resource is already running. What do you type?

ensure qu_hello. It works whether the resource is running or stopped, so you never have to remember the current state. That's the whole point of ensure.

The server just booted. You want to add a brand-new resource called qu_garage to the startup list permanently. What's the move?

Open server.cfg, add the line ensure qu_garage, save the file, then reboot the server so the config gets re-read. Typing ensure qu_garage at the console works too, but only until the next reboot, the server.cfg line makes it permanent.

Step 4

Read the console like a native speaker

Outcome: you recognize the four error patterns that cover 90% of what you'll see this year.

Good scripters don't "debug", they read. The console tells you exactly what's wrong if you know the shapes. Four patterns show up over and over.

The red one. It names the resource (qu_hello), the file (client/main.lua), and the line (14). Open that file, jump to that line, look at what's nil. If you see a wall of these, always read the first one: the rest are usually cascading from the same root cause.

Your fxmanifest points at a file that doesn't exist. Typo hunt: check the filename in your manifest against the actual file on disk. Lowercase, underscores, spelling, all have to match exactly.

You called a function that doesn't exist in this context. Two causes: a plain typo (Pirnt instead of print), or calling a function on the wrong side, a server-only native used inside a client script, for instance. If the name looks right, check which side you're on.

The call chain that led to the error. Read it bottom to top: the bottom line is where execution started, the top line is where it exploded. The path between them is the story of what your program was doing when everything went sideways.

Log prefix tells you who's talking

Every line the server prints starts with something like [ script:qu_hello]. That's the resource name. When a log mystifies you, "where is this coming from?", read the prefix. It names the exact resource that printed the line, so you know which folder to open.

Step 5

The dev loop: the 15-second cycle

Outcome: you run the same seven-step ritual until it's reflex.

Every feature you ship this year comes out of the same loop. Not a framework, not a process, a rhythm you run without thinking.

  1. Edit the file.
  2. Save, Ctrl/Cmd + S.
  3. Switch to the server console.
  4. Type ensure qu_whatever.
  5. Watch for errors in the console.
  6. Test in-game.
  7. Repeat.

Fifteen seconds per cycle when you're warmed up. Three hundred cycles on a busy day. The goal of the drills below is to lock this loop into your hands so you stop thinking about it and start thinking about the actual feature you're building.

txAdmin niceties

The Resources tab in txAdmin lets you start, stop, and restart with buttons. Use it if you prefer clicking. But typed commands work everywhere, txAdmin, raw FXServer, F8, SSH, so the 30 seconds you spend learning the shortcut pays back for years.

Common mistakes & what the console tells you

You edited the file but nothing changed

The most common "why isn't this working" cause in the entire course. You edited in the editor, then reloaded the resource, but you forgot to save. Ctrl/Cmd + S before every ensure. Most editors show an unsaved-changes dot on the tab; learn to glance at it.

"Access denied" when you type ensure

You typed the command in the in-game F8 client console without admin permissions. Client-side F8 can only run server commands if your account has them granted. The server console and txAdmin Live Console always work. When in doubt, use the server console.

You're reading the wrong error

A broken script often throws forty red lines in a row. The first one is the root cause; the rest are cascading damage from the first. Scroll up to the earliest red line in the burst and read that one. Fix it. Reload. The other thirty-nine vanish.

You edited server.cfg and restart didn't pick it up

Most server.cfg changes, new ensure lines, endpoint changes, convars, need a full server reboot, not just a single-resource restart. Close FXServer and relaunch, or use the Restart server button in txAdmin.

What you learned

  • ensure is your default command -- it starts or restarts a resource regardless of its current state.
  • The server console and F8 in-game console are two separate outputs: server scripts log to one, client scripts to the other.
  • The dev loop is: edit, save, ensure, read the console, test in-game, repeat.
  • server.cfg runs ensure lines at boot -- changes to it need a full server reboot to take effect.
  • Console errors name the resource, the file, and the line number -- always read the first red line in a burst.

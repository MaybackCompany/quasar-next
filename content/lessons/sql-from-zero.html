SQL from zero: oxmysql

Every script you have built so far forgets everything on restart. Cooldowns, bank balances, notes -- all gone. A database is the fix. Why you need a database

Right now, your Lua variables live in RAM. When the server restarts, RAM is wiped. Everything stored in a local table is gone. A database writes data to disk. Disk survives restarts, crashes, even power cuts. That is the entire point: data that outlives the process.

Think of a database as a spreadsheet that your code can talk to. Each table is a sheet. Each row is a line of data. Each column is a field. You talk to it using a language called SQL (Structured Query Language). SQL is not Lua. It is its own language, but it is simple for what we need.

MariaDB vs MySQL

MariaDB is a free, open-source fork of MySQL. They speak the same language. Most FiveM servers use MariaDB. When this lesson says "MySQL", it means both. oxmysql works with either one.

Step 1

Install oxmysql

Outcome: oxmysql is downloaded, placed in your resources, and started before your scripts.

  1. Go to oxmysql releases on GitHub and download the latest oxmysql.zip.
  2. Unzip it. You get a folder called oxmysql.
  3. Drop that folder into your server's resources/ directory, just like any other resource.
  4. Open server.cfg and add ensure oxmysql above all your other resources. oxmysql must start first because other resources depend on it.
ensure order matters

FiveM starts resources top to bottom. If qu_heal tries to talk to the database before oxmysql is running, you get a cryptic error. Put ensure oxmysql near the top of your server.cfg, before anything that uses it.

Step 2

The connection string

Outcome: oxmysql knows how to reach your database.

oxmysql needs one line in server.cfg that tells it where the database lives, what username and password to use, and which database name to connect to. This is called the connection string.

Secrets before connection strings

Complete Convars & server.cfg first. Use exec server-secrets.cfg, gitignore that file, and keep mysql_connection_string, webhook URLs, and API keys out of screenshots and public repos.

Let's break that apart:

  • mysql:// -- the protocol. Always this.
  • root -- your database username. root is the default admin account.
  • yourpassword -- the password for that user. Replace this with your actual password.
  • localhost -- the database is on the same machine as the FiveM server. If it were on another machine, you would use its IP address.
  • 3306 -- the default MySQL/MariaDB port.
  • fivem -- the database name. You create this once in your database tool (HeidiSQL, phpMyAdmin, or the command line).
Never commit your password

If you share your server files on GitHub, the connection string contains your database password. Either use a .gitignore for server.cfg or move the password to a separate file. Leaked database credentials = someone else owns your data.

Fix it by correcting the user/password in the connection string or creating the user for the right host: CREATE USER 'fivem'@'localhost' IDENTIFIED BY 'yourpassword'; GRANT ALL PRIVILEGES ON fivem.* TO 'fivem'@'localhost';. Then restart and look for [script:oxmysql] Database server connection established!.

Step 3

Your first SQL: CREATE TABLE

Outcome: you understand how to define a table with columns and types.

A table is like a spreadsheet with named columns. Before you can store anything, you tell the database what shape the data has. Here is our first table:

Line by line:

  • CREATE TABLE IF NOT EXISTS player_notes -- make a table called player_notes. The IF NOT EXISTS part means "skip this if the table already exists", so you can run it every time without errors.
  • id INT AUTO_INCREMENT PRIMARY KEY -- a number that goes up by 1 for each new row. This gives every row a unique ID. You never set it yourself; the database picks the next number.
  • identifier VARCHAR(60) NOT NULL -- a text column up to 60 characters. This will hold the player's license or citizen ID. NOT NULL means you cannot leave this empty.
  • note VARCHAR(255) NOT NULL -- a text column up to 255 characters for the actual note text.
  • created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP -- the database fills this in automatically with the current date and time when you insert a row.

What is MySQL.ready?

MySQL.ready(fn) waits until oxmysql has connected to the database, then runs your function. Without this wrapper, your query might fire before the connection is open and fail silently. Always wrap startup queries in MySQL.ready.

Storing a whole table in one column (JSON)

Sometimes you want to save something with lots of fields at once, like a player's first name, last name, and birthday. Instead of one column per field, frameworks like Qbox squeeze the whole thing into a single text column as JSON, a plain string that looks like {"firstname":"John","lastname":"Doe"}. To save a Lua table as that string, use json.encode. To turn the string back into a Lua table when you read it, use json.decode. Both are built into FiveM, no require needed.

-- saving: turn a Lua table into a JSON string for the column
local text = json.encode({ firstname = 'John', lastname = 'Doe' })
MySQL.update.await('UPDATE players SET charinfo = ? WHERE identifier = ?', { text, identifier })

-- reading: turn the column's JSON string back into a Lua table
local row = MySQL.single.await('SELECT charinfo FROM players WHERE identifier = ?', { identifier })
local charinfo = json.decode(row.charinfo)
print(charinfo.firstname)   -- John

You do not need this for the notes resource, but you will see json.decode everywhere once you open a real framework, so it helps to recognize it now.

Step 4

INSERT a row: parameterized queries

Outcome: you can safely insert data into a table without risking SQL injection.

To add a row, you use INSERT INTO. But there is a right way and a dangerously wrong way to include player data in the query.

The wrong way (never do this)

Why is this dangerous? Imagine a player sends this as their note text:

If you glue that into the query string, the database sees: insert a row, then delete the entire table. This attack is called SQL injection. It has destroyed real companies. The fix is simple: never glue player data into SQL strings. Instead, use ? placeholders.

The right way (always do this)

Each ? is a placeholder. oxmysql fills them in using the values from the table you pass as the second argument, in order. The database treats those values as pure data, never as SQL commands. No matter what text the player types, it can only end up inside the note column. It can never become a SQL command.

The one rule of SQL

Any time you put a variable into a SQL query, use a ? placeholder. No exceptions. No shortcuts. If you remember one thing from this lesson, make it this.

Step 5

SELECT rows back

Outcome: you can read data from the database and use it in Lua.

SELECT asks the database for rows. You can ask for everything, or filter by conditions.

In Lua, MySQL.query returns a table of rows. Each row is a Lua table where the keys are the column names:

MySQL.query vs MySQL.query.await

MySQL.query(sql, params) fires the query and moves on without waiting. MySQL.query.await(sql, params) pauses the current thread until the database responds and gives you the result. Use .await when you need the data right now (like printing notes). Use the non-await version for fire-and-forget operations (like logging).

Step 5b

UPDATE safely: the double-spend trap

Outcome: you understand how two clicks at once can dupe items, and the one-line SQL trick that stops it.

So far you have added rows and read them back. The last piece is changing a row, like subtracting money when a player buys something. That uses UPDATE:

Looks harmless. But here is a real bug that has duped items on huge servers. Picture a "buy gun" button. A player has $100 and a gun costs $80. The slow, natural way to write the handler is: read their cash, check if it is enough, then subtract.

Now imagine the player clicks the button twice at the exact same moment (a laggy connection or a cheater spamming it). Both clicks run step 1 at the same time. Both read cash = 100. Both pass the check in step 2. Both subtract 80. The player ends at -60 dollars and walks away with two guns for the price of one. That is a race condition: two things racing through the same code before either finishes.

The fix: let the database do the checking

Do not check in Lua and then deduct. Do both in one query. Add AND cash >= 80 to the UPDATE so the subtraction only happens if the money is actually there. The database runs updates one at a time, so only one of the two clicks can win.

Read that out loud: "subtract 80 from cash, but only for the row where cash is already 80 or more." If two clicks arrive together, the first one drops the cash to $20, and when the second one runs there is no longer a row where cash >= 80, so it changes zero rows. affected comes back as 0, you bail out, and no second gun is given. One click, one gun. No dupe.

Frameworks already do this for you

If you are on a framework like Qbox or ESX, do not write the raw money UPDATE yourself. Use the built-in money function, for example player.Functions.RemoveMoney('cash', 80). It returns false when the player cannot afford it and does the safe atomic check internally. You will meet these functions in the framework module. For now, just remember the rule: never read-then-deduct, always deduct-with-a-condition.

  2. Type /addnote Fix the garage door in chat.

  4. Restart the server. Type /mynotes again. Both notes are still there. That is persistence.
The restart test

The single most important test for any database feature: restart the server and check if the data survived. If it did, your persistence works. If it didn't, you're reading from memory, not the database.

Common mistakes

Connection string wrong format

The connection string must start with mysql:// and follow the exact format: mysql://user:password@host:port/database. A missing :, an extra space, or a wrong port can produce [script:oxmysql] Error: Access denied for user 'fivem'@'localhost' (using password: YES) or no successful connection line. Check this line first, then restart and confirm [script:oxmysql] Database server connection established!.

Table already exists error

If you write CREATE TABLE player_notes without IF NOT EXISTS, the query fails the second time you start the resource because the table is already there. Always use CREATE TABLE IF NOT EXISTS. It costs nothing and prevents a crash.

Forgetting ? placeholders (SQL injection)

Concatenating player input into a SQL string ("... '" .. input .. "'") opens you to SQL injection. A malicious player can delete your tables, read other players' data, or worse. Use ? placeholders every single time. There is no situation where string concatenation is OK for SQL values.

oxmysql not in server.cfg ensure list

If you see errors like MySQL is nil or attempt to index a nil value (global 'MySQL'), oxmysql is not running. Either ensure oxmysql is missing from server.cfg, or it appears after the resource that needs it. Move it higher.

Cheat sheet: SQL essentials

CREATE TABLE IF NOT EXISTS t (...)
Define a new table. Safe to run repeatedly.
INSERT INTO t (col1, col2) VALUES (?, ?)
Add a row. Always use ? for values.
SELECT col FROM t WHERE col = ?
Read rows that match a condition.
MySQL.query(sql, params)
Fire-and-forget query. Good for CREATE, INSERT, UPDATE.
MySQL.query.await(sql, params)
Wait for the result. Use when you need the rows back.
MySQL.insert(sql, params)
INSERT shortcut. Returns the new row's id.
MySQL.scalar.await(sql, params)
Returns a single value (first column of first row). Good for COUNT(*) or SUM().
? placeholder
Marks where a value goes. oxmysql fills it safely. One ? per value, matched by position.

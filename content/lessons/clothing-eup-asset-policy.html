What this is
Clothing asset management is the practice of organizing every clothing file on your server so you can find, update, and remove items without breaking everything.
Example: this lesson sets up EUP with a clean structure and a policy doc.
Where to find it
Clothing lives in two places: streaming assets (.ytd, .ydd files in a stream resource) and script data (item names, categories in config files).
Example: the 3D model is a stream asset; the name "LSPD Hat #3" is script data.
How to use it
Separate streaming from logic. One resource for 3D files, one resource for the UI/wardrobe, one for config. Never mix them.
Example: start with a clean folder layout before adding your first custom item.
What to do now
Set up the folder structure, add one custom item, and write the policy.
Example: match the expected structure before adding 50 more items.

Outcome

You will have a clean clothing resource structure, one custom clothing item working in-game, and a written policy for future asset additions.

Why it matters

A server with 50 clothing packs dumped into one folder is unmaintainable. When you need to remove a broken t-shirt or find which resource is causing a streaming crash, organization is the difference between a 5-minute fix and a 5-hour nightmare.

Vocabulary

EUP
Emergency Uniform Pack — a popular FiveM clothing framework that replaces/adds uniforms for police, EMS, fire, and civilian roles.
Example: LSPD officers wearing EUP uniforms instead of default GTA clothes.
Streaming asset
A 3D model, texture, or drawing dictionary (.ydd, .ytd) that the game loads into memory. Lives in a stream/ folder.
Example: a hat model is a .ydd file streamed by the client.
Drawable
A GTA V clothing component. Each clothing slot (hat, shirt, pants) has drawable IDs that reference streamed models.
Example: SetPedComponentVariation(ped, 0, 15, 0, 2) sets the face (slot 0) to drawable 15 with texture 0.

Common failures

  • Custom clothing does not appear: the fxmanifest.lua of the streaming resource must include data_file 'DLC_ITYP_REQUEST' 'stream/...' pointing to the correct folder. Without this, FiveM does not stream the files.
  • Drawable ID conflict: your custom item uses a drawable ID already claimed by EUP or another pack. Always start custom IDs at 200+ and check the EUP drawable list before assigning.
  • Clothing looks invisible or glitched: the .ydd file references a .ytd that is missing or has a mismatched name. Open the .ydd in a hex editor and verify the texture dictionary name matches the .ytd file name.
  • Server crashes on resource start: streaming too many large clothing packs at once exceeds the client's streaming budget. Split large packs into separate resources and ensure them with a delay between them.

Refactor

Write a Lua script that scans custom-clothing/stream/ and generates config.lua automatically from the file names, eliminating manual drawable ID assignment.

Cheat sheet

Clothing resource separation

  • eup-stream. Only .ydd and .ytd files. No scripts. No UI. Pure streaming.
  • eup-ui. Only wardrobe script and HTML. No 3D files. Handles equip/unequip.
  • custom-clothing. Your custom items. Separate from EUP so updates do not overwrite your work.
  • Policy file. Written rules that everyone on the team follows. Prevents chaos.

The big idea, in one minute

Clothing in FiveM has two halves: streaming (the 3D files the game loads) and scripting (the logic that equips items). Most servers dump both into one resource. When you need to update the wardrobe UI, you risk breaking streaming. When you add a new clothing pack, the UI resource restarts and kicks players out of the wardrobe. Separating them — one resource for files, one for logic — means you can update the UI without touching 3GB of clothing assets, and vice versa.

Think of a clothing store.

The warehouse (streaming resource) holds all the physical clothes — racks of shirts, shelves of hats. The cash register (UI resource) is where customers browse and check out. If you renovate the cash register, the warehouse stays open. If you restock the warehouse, the cash register keeps running. But if you put the cash register inside the warehouse and try to renovate, you shut down the entire store.

Your EUP setup separates the warehouse from the register. The policy document is the employee handbook that tells everyone where things go.

Bad way and good way, side by side

Most servers dump all clothing into one resource. Updates break things. The good way separates concerns.

Don't do this

One resource with everything mixed together.

Adding one new hat means restarting the entire clothing resource. Every player on the server sees their clothes disappear for 3 seconds. Do this during peak hours and you get 50 complaints.

Do this

Separate into streaming, UI, and custom resources.

Adding a hat restarts only the custom clothing resource (a fraction of a second). The EUP streaming files stay loaded. The wardrobe UI stays open. Players do not even notice the update.

Step 1

Set up the three-resource structure

Three clean folders before you add a single clothing file.

  1. Inside resources/, create a folder named [clothing]. The brackets are a convention for grouping.
  2. Inside [clothing]/, create three folders: eup-stream, eup-ui, custom-clothing.
  3. In each folder, create an empty fxmanifest.lua.
  4. Add the three ensure lines to server.cfg in order: stream first, then ui, then custom.
  5. Start the server. All three resources should start without errors (they are empty, so no errors expected).
Step 2

Install EUP streaming assets

EUP is a streaming-first mod. The 3D files go here.

  1. Copy EUP's stream/ folder into eup-stream/. This contains the .ydd and .ytd files.
  2. Update eup-stream/fxmanifest.lua to include data_file 'DLC_ITYP_REQUEST' 'stream/...' for each subfolder.
  3. Add the fx_version 'cerulean' and game 'gta5' headers.
  4. Restart eup-stream. Check the client console (F8) for streaming errors.
  5. If no errors, the 3D clothing models are loaded. They will not be visible yet — that requires the UI resource.
Step 3

Install EUP UI (wardrobe)

The UI resource lets players browse and equip the clothes that are streaming.

  1. Copy the EUP UI script files into eup-ui/.
  2. Update eup-ui/fxmanifest.lua with client_scripts { 'client.lua' } and ui_page 'html/index.html'.
  3. Restart eup-ui. The wardrobe should now be accessible (usually via a command like /eup or /wardrobe).
  4. Open the wardrobe and browse the default EUP uniforms. Confirm you can equip a hat, shirt, and pants.
  5. If clothes do not appear, the UI resource cannot find the streaming data. Check that eup-stream is started BEFORE eup-ui in server.cfg.
Step 4

Add one custom clothing item

Your first custom item proves the structure works end-to-end.

  1. In custom-clothing/stream/, create a subfolder for your item, e.g., qu_hats/.
  2. Add your .ydd (hat model) and .ytd (hat texture) files to this folder.
  3. Update custom-clothing/fxmanifest.lua with data_file 'DLC_ITYP_REQUEST' 'stream/qu_hats/...'.
  4. Create custom-clothing/config.lua and add an entry: { drawable = 200, label = 'Quasar Branded Cap', slot = 'hat' }.
  5. Restart custom-clothing. Open the EUP wardrobe. Your custom hat should appear under the hats category.
Step 5

Write the asset policy document

A policy prevents future chaos. Write it once, enforce it always.

  1. Create docs/asset-policy.md in your server root folder.
  2. Write four sections: Folder Rules, Naming Conventions, Approval Process, Texture Rules.
  3. Copy the template from the minimal example and customize it with your server's actual folder names and drawable ID ranges.
  4. Share the policy with your team. Make it required reading before anyone adds clothing.
  5. This document is living. Update it every time you add a new category of assets.

Three tests, in order

Run each test and confirm the expected behavior. If something does not match, fix it before moving on.

  2. Test 2, custom item appears.

    • Expected: the hat is visible in the preview and equips correctly.

  3. Test 3, resource isolation.

Why separate custom clothing from EUP?

EUP updates frequently. If your custom items are mixed into the EUP resource, an update will overwrite your config.lua and potentially your custom .ydd files. Keeping custom items in a separate resource means you can update EUP by replacing its folder without touching your work.

The rules you just followed

  • Streaming assets (3D files) live in a standalone resource with no scripts or UI.
  • UI and wardrobe logic live in a separate resource that only handles equip/unequip.
  • Custom clothing lives in its own resource, separate from EUP, to survive updates.
  • Every custom item has a documented drawable ID starting at 200+ to avoid conflicts.
  • An asset policy document defines the rules for the entire team and prevents chaos.

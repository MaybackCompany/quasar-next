The mental model: an error is a clue, not a verdict

When FiveM or Lua throws an error, it is not saying "you are bad at coding." It is saying "I tried to do one specific thing, and one specific part was missing or wrong." Your only job is to read the message slowly and translate it into plain English.

The biggest beginner mistake is changing five files before understanding the first error. Do the opposite. Read the first red line. Find the file. Find the line. Ask what value or function is missing. Only then touch the code.

The five-step rule

Read the first error -> open the named file -> go to the named line -> identify the missing thing -> fix one thing and test again. That is real debugging.

nil: the most common word you will see

nil means "there is nothing here." In Lua, this is normal. It becomes an error only when you try to use the nothing like it was real data or a real function.

  • Attempt to index a nil value -> you tried thing.name or thing[1], but thing was empty.
  • Attempt to call a nil value -> you tried thing(), but thing was not a real function.
  • Attempt to concatenate a nil value -> you tried to glue text onto an empty value.
Plain-English translation

If you see nil, stop asking "why is Lua broken?" and ask "what value did I assume existed here?" Most of the time the answer is: wrong variable name, wrong side, missing guard, or data not loaded yet.

Wrong-side calls: client vs server mistakes

FiveM has two worlds. Some functions exist only on the client. Some exist only on the server. If you call the wrong one on the wrong side, Lua often tells you attempt to call a nil value.

That does not usually mean the name is misspelled. It often means the function exists only on the server, but you called it from client/main.lua.

Fast fix

Look at the file path first. If the file is in client/, check whether the native or framework function is client-safe. If the file is in server/, check whether you accidentally pasted a client-only function there.

exports errors: another resource is missing or named wrong

exports means "call a function from another resource." If that other resource is not started, has the wrong name, or you copied the wrong framework snippet, the error usually points there.

  • No such export -> the resource name is wrong, the export name is wrong, or that resource is not running.
  • lib is nil -> you forgot the ox_lib shared import or loaded files in the wrong order.
  • QBCore snippet on ESX or Qbox -> you copied a snippet for the wrong framework.
Beginner habit

When an exports error appears, check four things in this order: resource name -> manifest dependency/import -> correct framework -> resource started. Do not start rewriting logic first.

SQL and oxmysql errors: the database layer is a separate system

SQL errors feel bigger because they mention tables, columns, and databases. They are still just clues. The important thing is to separate Lua is wrong from database structure is wrong.

  • Unknown column -> your query asked for a column the table does not have.
  • Table doesn't exist -> you never imported the SQL file, or you used the wrong database.
  • oxmysql export error -> oxmysql is not started, not installed, or called wrong.
  • Query failed -> read the full error. Usually the SQL string, placeholders, or table shape is wrong.
SQL comes later on purpose

If you cannot yet build a small server event or callback, SQL will feel worse than it is. That is why this course pushes SQL after your first builds. Database bugs are easier once Lua and events already feel normal.

The first fixes to try before you panic

01

Read the first red line

Ignore the tenth line for now. The first error is usually the real clue.

02

Open the exact file and line

Do not guess. Go to the path and line number the console gave you.

03

Check for nil

Ask what value you assumed existed but does not exist yet.

04

Check the side

Make sure the function belongs on client or server before changing anything else.

05

Check the dependency

Make sure the resource you depend on is installed, named right, and started.

06

Check the framework snippet

Make sure you copied code for the right framework: QBCore or ESX.

07

Check manifest imports

Wrong file paths or missing imports can make whole features look broken.

08

Check SQL basics

If it is SQL, check table names, column names, and whether oxmysql is running.

One-screen cheat sheet

attempt to index a nil value

You tried to read inside a value that does not exist yet.

attempt to call a nil value

You tried to call a function name that is empty in this runtime.

attempt to concatenate a nil value

You tried to glue text to an empty value.

No such export ...

The resource is wrong, missing, not started, or the export name is wrong.

Unknown column ...

Your SQL query does not match your table structure.

Table ... doesn't exist

The table was not imported or you are connected to the wrong database.

lib is nil

ox_lib import order or manifest setup is wrong.

Fastest debug tool

Add print(type(value), value) before the crashing line.

What you learned

  • Error messages are clues, not punishments. Read the first line and work from there.
  • nil means "this does not exist yet." Check your variable names, spelling, and whether the file loaded.
  • Wrong-side errors happen when you call a client function on the server or vice versa. Know which world your code runs in.
  • Export errors mean the resource is missing, not started, or the export name is misspelled.
  • The fastest debug tool is print(type(value), value) on the line before the crash.

Five-minute exercise

Open any script you have written so far. Intentionally break it three ways: (1) misspell a variable name, (2) call a function that does not exist, (3) try to read a key from a nil table. Run each one, read the error, and fix it using only the console output.

Expected outcome

You should see three different error messages, identify the broken line from each one, and fix all three without guessing.

Hint

The error message always includes a file path and line number. Start there. Add print(type(varName)) above the broken line to confirm what is nil.

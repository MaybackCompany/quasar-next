NUI advanced: messages, callbacks, focus

NUI is a tablet glued over the windshield. Lua talks to it with SendNUIMessage (a walkie-talkie); the page talks back with fetch to RegisterNUICallback. Focus and cursor are how you decide whether the player is driving or typing.

You will learn
Open/close, two-way messages, callbacks, focus, dev reload.
Time
~30 min
Prereqs
NUI from scratch, P3 exports, and P8 server validation.

Before you start

If you have not built a plain NUI yet, do NUI from scratch first. This lesson assumes the page already loads and focuses correctly.

Mental model. Lua and the page run in different worlds. They only see each other through SendNUIMessage (Lua → page) and fetch('https://<resource>/<callback>') (page → Lua).

The two channels

  • Outbound, SendNUIMessage(table) arrives in the page as window.addEventListener('message', e => e.data).
  • Inbound, the page does fetch("https://qu_phone/buy", { method:'POST', body: JSON.stringify({…}) }) and Lua handles it with RegisterNUICallback("buy", cb).
  • Focus, SetNuiFocus(showCursor, takeInput) tells the game whether the player is interacting with the page.

Gotcha. Forgetting to release focus traps the player. Always pair SetNuiFocus(true,true) with a close path that calls SetNuiFocus(false,false).

Step 1 · Open and close cleanly

-- client/main.lua
local isOpen = false

local function open()
  if isOpen then return end
  isOpen = true
  SetNuiFocus(true, true)
  SendNUIMessage({ action = "open", payload = { name = GetPlayerName(PlayerId()) } })
end

local function close()
  if not isOpen then return end
  isOpen = false
  SetNuiFocus(false, false)
  SendNUIMessage({ action = "close" })
end

RegisterCommand("phone", open, false)
RegisterNUICallback("close", function(_, cb) close(); cb({ ok = true }) end)

A single isOpen flag prevents double-open bugs. The page calls /close when the user hits Escape; Lua mirrors state and releases focus.

Step 2 · Page listens for messages

// html/app.js
window.addEventListener('message', (e) => {
  const { action, payload } = e.data || {};
  if (action === 'open')  { document.body.classList.add('show');  hydrate(payload); }
  if (action === 'close') { document.body.classList.remove('show'); }
});

document.addEventListener('keydown', (e) => {
  if (e.key === 'Escape') {
    fetch(`https://${GetParentResourceName()}/close`, { method: 'POST', body: '{}' });
  }
});

GetParentResourceName() is injected by CFX into the NUI page. Always use it, the resource name may differ across servers.

Step 3 · Callback to the server (via client)

-- client/main.lua
RegisterNUICallback("buy", function(data, cb)
  local item = tostring(data and data.item or "")
  local qty  = tonumber(data and data.qty) or 0
  if item == "" or qty <= 0 then cb({ ok = false, error = "bad_input" }); return end
  TriggerServerEvent("qu_phone:buy", item, qty)
  cb({ ok = true }) -- ACK only; the real result comes back via SendNUIMessage
end)

RegisterNetEvent("qu_phone:bought", function(item, qty, total)
  SendNUIMessage({ action = "bought", payload = { item = item, qty = qty, total = total } })
end)

The HTTP callback only acknowledges receipt. The authoritative result lands as a server event, then we push it into the UI as a message. This keeps the client trustworthy and the UI snappy.

Step 4 · Dev reload without restarting the resource

# launch the game with NUI devtools
# in the F8 console:

# then in DevTools console:

nui_devtools opens Chromium DevTools attached to your page. Reload there is instant. Restart the whole resource only when Lua changed.

Recap · Round trip

  1. Open the phone with /phone. Confirm cursor + input focus.
  2. Click a buy button. Confirm Lua prints, the server event fires, and the UI shows the result.
  3. Press Escape. Confirm focus is released and movement works.

Recap · Stuck cursor recovery

Force error() in your buy handler. Verify the player can still close (Escape path runs even if the callback throws). If it cannot, your close path depends on a code branch that errored, refactor.

Step 5 · At scale: render budget, callback contracts, asset shipping

NUI looks cheap until 32 players each have a phone open while you push a state update every second. Three sub-systems decide whether the UI feels native or stutters: how often you re-render, how strictly callbacks honor the cb-once contract, and how the bundle ships.

5.1 Render budget: diff messages, virtualize lists

The slowest pattern in NUI is "send the entire state every tick, page re-renders everything." Two fixes, coalesce updates and only send deltas, and virtualize any list past ~50 rows.

-- client/main.lua - coalesced outbound updates
local pendingState = nil
local flushScheduled = false
local FLUSH_MS = 50

local function pushState(partial)
    pendingState = pendingState or {}
    for k, v in pairs(partial) do pendingState[k] = v end
    if flushScheduled then return end
    flushScheduled = true
    SetTimeout(FLUSH_MS, function()
        SendNUIMessage({ action = "patch", payload = pendingState })
        pendingState = nil
        flushScheduled = false
    end)
end
// html/app.js - render only what changed
const state = {};
window.addEventListener('message', (e) => {
    if (e.data.action !== 'patch') return;
    const next = e.data.payload;
    for (const key in next) {
        if (state[key] === next[key]) continue; // identity skip
        state[key] = next[key];
        renderField(key, next[key]); // per-field render, not full tree
    }
});

// Virtual list: render only visible rows
const ROW_H = 28;
function renderList(container, rows) {
    const total = rows.length;
    const onScroll = () => {
        const top = container.scrollTop;
        const start = Math.max(0, Math.floor(top / ROW_H) - 5);
        const end = Math.min(total, start + Math.ceil(container.clientHeight / ROW_H) + 10);
        const slice = rows.slice(start, end);
        container.firstElementChild.style.height = total * ROW_H + 'px';
        container.lastElementChild.innerHTML = slice
            .map((r, i) => `<div style="position:absolute;top:${(start+i)*ROW_H}px">${r.label}</div>`)
            .join('');
    };
    container.addEventListener('scroll', onScroll, { passive: true });
    onScroll();
}

The 50 ms coalesce window absorbs spam: 20 incoming updates collapse into 1 outbound message. Per-field render skips the layout pass entirely on identity-equal fields. Virtualizing a 5000-row inventory drops first-paint from 380 ms to 14 ms.

Measure first. Open nui_devtools qu_phone, Performance tab, record 10 s with the UI open. If you see 200 forced reflows or a single layout pass over 16 ms, you are not virtualized enough.

5.2 Callback contract: cb-once, timeout, in-flight gate

Every RegisterNUICallback must call cb exactly once. The page must handle a missing or late response. Both sides need an in-flight gate so a double-click never spawns two requests.

-- client/callbacks.lua - cb-once enforced
local function safeCallback(name, handler)
    RegisterNUICallback(name, function(data, rawCb)
        local called = false
        local function cb(result)
            if called then
                print(("[NUI] cb double-call on %s"):format(name))
                return
            end
            called = true
            rawCb(result)
        end
        local ok, err = pcall(handler, data, cb)
        if not ok then
            print(("[NUI] %s threw: %s"):format(name, err))
            cb({ ok = false, error = "internal" })
        elseif not called then
            print(("[NUI] %s did not call cb"):format(name))
            cb({ ok = false, error = "no_response" })
        end
    end)
end

safeCallback("buy", function(data, cb)
    local item = type(data.item) == "string" and data.item or nil
    local qty  = type(data.qty)  == "number" and data.qty  or nil
    if not item or not qty or qty <= 0 then return cb({ ok = false, error = "bad_input" }) end
    TriggerServerEvent("qu_phone:buy", item, qty)
    cb({ ok = true })
end)
// html/api.js - in-flight gate + timeout
const inflight = new Set();
async function nuiFetch(name, body, timeoutMs = 5000) {
    if (inflight.has(name)) return { ok: false, error: 'in_flight' };
    inflight.add(name);
    const ctrl = new AbortController();
    const timer = setTimeout(() => ctrl.abort(), timeoutMs);
    try {
        const res = await fetch(`https://${GetParentResourceName()}/${name}`, {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify(body || {}),
            signal: ctrl.signal,
        });
        return await res.json();
    } catch (err) {
        return { ok: false, error: err.name === 'AbortError' ? 'timeout' : 'network' };
    } finally {
        clearTimeout(timer);
        inflight.delete(name);
    }
}

Three guarantees from the wrapper: cb always fires (even on error), never twice, and never silently. The page-side gate prevents double-buy when a lagging server lets the user click twice. Timeout prevents a stuck spinner forever.

5.3 Asset shipping: bundle, manifest, FPS at 50 players

The manifest files list ships every entry to every joiner. A 14 MB phone UI is 14 MB times 50 players downloaded on connect. Bundle, hash, and gzip to keep first-paint short.

-- fxmanifest.lua
fx_version 'cerulean'
game 'gta5'
ui_page 'html/dist/index.html'

files {
    'html/dist/index.html',
    'html/dist/assets/**/*.js',
    'html/dist/assets/**/*.css',
    'html/dist/assets/**/*.woff2',
    'html/dist/assets/**/*.svg',
}
-- Do NOT ship: source maps, raw .vue/.tsx, node_modules, screenshots
# vite build with safe defaults for FiveM NUI
# vite.config.ts
# build: {
#   outDir: 'html/dist',
#   sourcemap: false,            // off in prod
#   target: 'chrome120',         // CEF in current FiveM is recent
#   minify: 'esbuild',
#   assetsInlineLimit: 4096,     // inline tiny SVGs
#   rollupOptions: { output: { manualChunks: { vendor: ['react', 'react-dom'] } } },
# },
# sanity check: bundle size and first-paint cost
$ du -sh html/dist
2.1M    html/dist
$ du -sh html/dist/assets/*.{js,css}
1.4M    html/dist/assets/index-7f2a.js
180K    html/dist/assets/index-7f2a.css
# at 32 players join, 32 * 2.1MB = 67 MB, absorbed by CDN/cache, not server

Three rules: ship only the build output, never the source tree; lock chunks so the vendor bundle hashes once and caches across deploys; subset the font (woff2 with only Latin glyphs, saves 240 KB versus shipping every weight). The CEF runtime in current FiveM artifacts supports modern Chrome features, you do not need IE-era polyfills.

FPS impact. A hidden NUI page can still render. Keep the message listener mounted, then hide heavy panels with CSS when closed. If a full-screen app is expensive while idle, you can hide the root container, but restore it before sending the open state and always keep the callback/listener shell alive.

5.4 React + Vite: the production scaffold and its four non-negotiables

Everything above works with plain HTML and JS, and for a small panel that is the right call. But the moment your UI grows into a phone, a dashboard, or a multi-page menu, you will want React (or Svelte) bundled with Vite. You get components, real state, and hot reload in a normal browser. The build still drops into html/dist exactly like 5.3 showed. What changes is a handful of FiveM-specific settings that quietly break the UI if you skip them.

The one Vite setting nobody warns you about: base: './'

By default Vite writes asset paths like /assets/index.js with a leading slash. A normal web server resolves that fine. CEF, the browser inside FiveM, cannot, so your page loads to a blank white screen with no error. Set base: './' so every path becomes relative. This is not optional, it is the single most common reason a freshly built React NUI shows nothing.

Use Tailwind v3.4.x, not v4, inside NUI

If you reach for Tailwind, install version 3.4.x and pin it. Tailwind v4 switched its default colors to the OKLCH color space, and the CEF build shipped in FiveM does not render OKLCH correctly, so your carefully picked colors come out wrong or invisible. Version 3.4 is the safe stopping point. This burns people who upgrade Tailwind without reading the changelog, then spend an evening wondering why every button is the wrong shade.

The second half is four small files that every solid public React NUI ships. Skip any one of them and you get the exact bugs the rest of this lesson catalogs, just harder to trace.

Hide with visibility, never with conditional render

The React instinct is {visible && <App />}. Do not do that here. When visible is false, React unmounts the component, which tears down your message listener, so the next SendNUIMessage from Lua lands on nobody and the UI never reopens. Instead keep the component mounted always and toggle CSS: style={{ visibility: visible ? 'visible' : 'hidden' }}. This is what a VisibilityProvider exists to centralize, hold the visible flag in context, apply the hidden style at the root, never unmount the tree.

The dev mock is what makes React NUI pleasant to build. You run npm run dev, your UI opens in a normal browser populated with fake data, and you iterate with instant hot reload, no FiveM restart, no joining a server. When the design is right, npm run build, then restart qu_phone once to see it in game. Four files, and every failure mode in the table below that mentions listeners, blank screens, or stuck UIs is designed out before it happens.

Recap 3

Senior depth: NUI lifecycle

The NUI bridge is async, lossy under spam, and silently leaks if you attach listeners on every open without removing them on close. You know why the source of truth is server state, why the client renders the cache, and why event listeners are attached once on mount and gated by an open flag. UI memory growth over a 6 hour shift is never random; it is always a closure that nobody freed.

Common mistakes

Trusting fetched data.

Page input is user input. Validate types and bounds in the callback before forwarding to the server.

Calling cb(...) twice.

Each callback must call its cb exactly once. Wrap risky paths in pcall or guard with a flag.

Hardcoding the resource name.

Use GetParentResourceName() in JS so renames do not break the UI.

Holding focus while the menu is hidden.

If you toggle visibility via CSS, also call SetNuiFocus(false,false) on hide.

Not pairing close paths.

Escape, button click, and disconnect all need to release focus. Centralize in one close() function.

Recap

  • Outbound = SendNUIMessage; inbound = fetch + RegisterNUICallback.
  • Always pair focus on/off; centralize close.
  • Validate page input before forwarding to the server.
  • Use DevTools for fast UI iteration; restart resource only for Lua changes.
  • Use GetParentResourceName() to build fetch URLs.

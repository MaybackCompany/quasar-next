<Callout variant="fivem" title="How FiveM voice works">
  FiveM has a voice system built in. It is based on Mumble, and it runs **inside FXServer**. You do not install or run a separate voice server, and there is no separate voice port to open. `pma-voice` is a community resource, maintained in the `AvarianKnight/pma-voice` repository, that sits on top of the built-in system and adds proximity, radio, and call channels. It can be pointed at an external Mumble address with documented convars, but a normal setup never needs that. A **convar** is a server setting you write in `server.cfg`. When voice breaks, the cause is usually the input device, a second voice resource fighting pma-voice, or a wrong convar, so check those in order before you blame the network.
</Callout>

## Get it working

<StepGroup>
  <Step num={1} title="Install pma-voice by itself" outcome="One voice resource starts with no errors.">
    Download the release from the official `AvarianKnight/pma-voice` repository. Put it in `resources` with the folder name `pma-voice`. It uses FiveM's built-in Mumble support, so **do not install or run `mumble-voip` next to it**. The pma-voice README says it does not work alongside other voice systems, including vMenu voice unless you turn that feature off.

    ```cfg
    ensure pma-voice
    ```

    `ensure` is the server.cfg command that starts a resource (and restarts it if it is already running). Restart the server and watch the console. If another voice resource is running, stop it and remove its `ensure` line before you troubleshoot pma-voice. **Only one voice resource may control the voice natives at a time.**
  </Step>

  <Step num={2} title="Set the voice mode and audio path in config" outcome="The starting voice mode and audio behavior are what you want.">
    pma-voice is configured with convars in `server.cfg`. Start with the defaults and override only the values you mean to change.

    ```cfg
    # 1 = Whisper, 2 = Normal, 3 = Shouting
    setr voice_defaultVoiceMode 2
    setr voice_useNativeAudio false
    setr voice_enableRadioAnim false
    ```

    These lines do not set three distance numbers. `voice_defaultVoiceMode` picks the mode a player starts in. `voice_useNativeAudio` changes the audio path and is required for submixes. `voice_enableRadioAnim` only controls the radio animation. The talk, whisper, and shout ranges live in the resource's shared code. **Changing those ranges means maintaining your own fork, not flipping a convar.** Follow the README for the exact release you installed.
  </Step>

  <Step num={3} title="Confirm mic permission and device" outcome="Your voice is captured and sent.">
    In FiveM settings, set the correct input device and your push-to-talk key. Then allow microphone access at the operating-system level. **A muted OS mic looks exactly like a broken server.** Check this first.
  </Step>

  <Step num={4} title="Test with a second player" outcome="Two players hear each other by proximity.">
    Voice is hard to test alone. Bring a second client, or a friend, stand close, and confirm the audio fades out as you walk apart.

    <ExpectedOutput>
    Players hear each other in-game with clear audio. As they walk apart, the audio fades. You know which convars control the voice mode and audio path, and you know the order to check things in when voice breaks.
    </ExpectedOutput>
  </Step>
</StepGroup>

## How it works

Two facts explain almost every voice problem you will hit. Hold both in your head before you tune anything.

**Voice runs inside FXServer.** pma-voice uses the built-in Mumble system. There is no separate voice server and no separate voice port. So if audio cuts out, do not start by hunting for a missing voice server. Check the client device and push-to-talk, confirm only one voice resource controls the natives, read pma-voice's debug output, and only then check latency, packet loss, and server health.

**Proximity has three modes.** The starting mode comes from `voice_defaultVoiceMode`. Players cycle Whisper, Normal, and Shouting with a key set in pma-voice (the README for your release tells you the exact key). So when someone says the range feels wrong, **first confirm which mode that player is in.** Do not tell operators to edit a `config.lua`; pma-voice has none. Use the README's convars, and treat any change to the resource's shared code as a fork you have to keep up.

## Troubleshooting order

<FailureList
  items={[
    { symptom: "Nobody can hear anyone", fix: "Confirm pma-voice started with no error, disable every other voice system, then check OS microphone permission and the FiveM input and output devices." },
    { symptom: "I hear others but they cannot hear me", fix: "Your mic input or push-to-talk is wrong. Set the correct device and key in FiveM settings (Voice Chat must be ON) and unmute the OS mic." },
    { symptom: "My range feels wrong, people are too quiet or too loud", fix: "Cycle Whisper, Normal, and Shouting, then check voice_defaultVoiceMode. If you edited pma-voice's shared code, compare it with the official release before changing more." },
    { symptom: "Voice cuts out or sounds robotic", fix: "Check the input device, conflicting voice scripts, pma-voice debug output, then latency, packet loss, and server health. It is not always the network and not always the config." },
    { symptom: "Echo or feedback when talking", fix: "Your mic is picking up your speakers. Use headphones, or lower mic sensitivity in FiveM settings." },
    { symptom: "Radio or phone channels do not work", fix: "Those come from framework integration. Confirm the framework bridge (ESX Legacy, QBCore, or Qbox) and the radio and phone channel settings in the voice resource." },

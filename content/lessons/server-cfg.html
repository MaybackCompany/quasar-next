server.cfg, resources, and dependencies

server.cfg is the startup script for your city. It names the server, sets ports, starts resources, and decides the order everything loads.

You'll do
Open server.cfg, read ensure order, map resources, and add one safe local resource line.
Time
~20 minutes.
You need
Step 03 done and the server recipe installed.
Result
You can explain why a script like a phone, inventory, or job must load after the resources it depends on.

Before you start

The simple mental model

FXServer reads server.cfg from top to bottom. If a script needs another script, that dependency must be ensured first.

Resource
A script folder with an fxmanifest.lua file.
ensure
The command that starts a resource and keeps it started.
Script dependency
A resource another resource needs before it can work.
Example: Quasar Smartphone Pro needs ox_lib before the phone resource starts.
fxmanifest.lua
The file that tells FiveM what a resource contains and needs.
Step 1

Open server.cfg

Outcome: VS Code shows the config file.

  1. Open your server-data folder.
  2. Right-click the folder and choose Open with Code.
  3. Open server.cfg.
  4. Find the resource lines that start with ensure.

Expected outcome

You can point at the block where resources start.

Step 2

Understand the important lines

Outcome: you know what the common server settings do.

endpoint_add_tcp "0.0.0.0:30120"
endpoint_add_udp "0.0.0.0:30120"

sv_hostname "My QBCore Server"
sv_maxclients 48
sv_licenseKey "PASTE_YOUR_KEY_HERE"

set mysql_connection_string "mysql://root:PASSWORD@127.0.0.1/fivem?charset=utf8mb4"

ensure oxmysql
ensure ox_lib
ensure qb-core
ensure ox_inventory
ensure ox_target
endpoint_add_tcp / endpoint_add_udp
The port doors players connect through.
sv_hostname
The server name players see.
sv_licenseKey
Your private Cfx.re license key.
mysql_connection_string
The bridge information oxmysql uses to reach MariaDB.
Secret-handling warning

sv_licenseKey and mysql_connection_string contain secrets. Do not paste real values into Discord, public GitHub, screenshots, tutorial videos, or support tickets. Blur them before sharing. If they leak, revoke the Cfx.re key and rotate the database password.

Step 3

Read ensure order like a script dependency map

Outcome: you can explain why order matters.

# 1. Database bridge first
ensure oxmysql

# 2. Shared helper library second
ensure ox_lib

# 3. Framework core third
ensure qb-core

# 4. Core systems after framework
ensure ox_inventory
ensure ox_target

# 5. Quasar scripts and gameplay scripts after their dependencies
ensure qs-smartphone-pro
ensure [jobs]
ensure [cars]
ensure [maps]
Wrong order creates confusing errors

If qs-smartphone-pro starts before ox_lib, the phone resource may not run. If a job starts before QBCore, it may print missing export errors. If a SQL script starts before oxmysql, saves fail.

Recap 1

Can you explain the order?

Say this out loud: dependency first, dependent script second. For example: ox_lib before qs-smartphone-pro.

Test

Run an ensure-order test with throwaway resources

Outcome: you prove that order changes the console result without touching a single production script.

Never test order on real dependencies

Do not restart ox_lib, a framework, or a paid script on a live server to "test" order. You can disconnect players and break every resource that depends on them. Use two tiny local resources instead: no players, no data, no risk.

Create resources/[local]/qu_dep_provider and resources/[local]/qu_dep_consumer. The provider exports a value; the consumer reads it and reports whether it found it.

  1. Make a backup copy of server.cfg.
  2. Add both resources in dependency-first order, then save: 
    ensure qu_dep_provider
ensure qu_dep_consumer
  3. Open txAdmin → Live Console.
  4. Run restart qu_dep_provider, then restart qu_dep_consumer.
  5. Read the console. You want the found line, not the missing line.

Now swap the two lines so qu_dep_consumer is ensured first, save, and restart both again. The consumer prints provider MISSING, the same failure a real dependency hits when its provider starts too late.

If you see a missing export

Put the provider above the script that needs it, save server.cfg, restart the provider first, then the dependent script.

Checking real dependencies without restarting

To confirm the order of live paid scripts like ox_lib or qs-smartphone-pro, do not restart them while players are connected. Read server.cfg top to bottom and confirm each dependency's ensure line sits above every script that needs it, then open each script's fxmanifest.lua dependencies block. Inspect, do not restart.

Step 4

Understand the resources folder

Outcome: you know where scripts and assets live.

server-data/
  server.cfg
  resources/
    [standalone]/
      oxmysql/
      ox_lib/
    [qb]/
      qb-core/
      qb-vehicleshop/
    [jobs]/
      qb-policejob/
    [cars]/
      police_pack/
    [maps]/
      legion_mlo/

Bracket folders organize resources. FiveM sees the resources inside them. You can also ensure [cars] to start every resource in that bracket folder.

Step 5

Check a script before adding it

Outcome: you can spot script dependencies before install.

  1. Open the resource folder.
  2. Open fxmanifest.lua.
  3. Look for dependencies or install notes.
  4. Look for required shared scripts like @ox_lib/init.lua.
  5. Check the seller documentation for required scripts.
  6. Start those dependencies before the script in server.cfg.
dependencies {
  'ox_lib',
  'oxmysql',
  'qb-core'
}

Expected outcome

You know the script must start after ox_lib, oxmysql, and qb-core.

Quasar example

If a Quasar script says it requires ox_lib, put ensure ox_lib above that Quasar script in server.cfg.

Step 6

Add one safe local resource line

Outcome: you edit server.cfg without breaking the server.

  1. Create a bracket folder named [local] inside resources.
  2. Add your own test resource there later.
  3. For now, add this comment near the bottom of server.cfg.
# Local scripts I am testing
# ensure [local]

Expected outcome

The server still restarts. Nothing new starts because the ensure line is commented.

Recap 2

Restart cleanly

Restart from txAdmin. The server should come back online with no new red dependency errors.

Expected console lines after a clean restart

A clean restart does not mean zero lines. It means the important lines show startup progress and no new red blocker appears.

Proof

Run status, then direct connect once. If a player can join and the console has no new red dependency errors, the restart proof passes.

Database recap

MariaDB stores the data. HeidiSQL lets you see it. oxmysql lets FiveM scripts read and write it.

Common chain

QBCore or ESX saves a vehicle. The framework asks oxmysql. oxmysql talks to MariaDB. HeidiSQL lets you inspect the saved row.

Common mistakes

  • No such export: the script dependency did not start first.
  • Access denied for user: MariaDB password or connection string is wrong.
  • Unknown database 'fivem': the database was not created in HeidiSQL.
  • Couldn't find resource: the folder name and ensure name do not match.
  • The script is inside a zip folder, not an extracted resource folder.

What you learned

  • server.cfg controls startup order.
  • ensure starts resources.
  • fxmanifest.lua and script docs reveal dependencies.
  • oxmysql must start before scripts that save data.
  • ox_lib must start before scripts that use it.

Five-minute exercise

Open one installed resource and find its fxmanifest.lua. Then open its install docs. Write down every dependency it names.

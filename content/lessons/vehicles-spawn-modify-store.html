Vehicles: spawn, modify, clean up

Vehicles are the most common entity you will work with in FiveM. Garage scripts, dealerships, police cruisers, test commands -- they all start with spawning a car and putting the player in it. This lesson covers the full cycle: spawn, customize, put the player inside, and clean up when done.

Local test only

The spawn, store, and cleanup code in this lesson is client-side practice code. Use it to learn handles, vehicle natives, and cleanup. Do not treat it as a real garage or ownership system.

Real use needs server approval

For a production garage, the client should ask the server to spawn or store. The server validates ownership, plate, model, distance to the garage, and whether the entity still exists before it accepts anything. The client may provide visible interaction; the server owns the decision.

Step 1

Spawn a vehicle

Outcome: a car appears in the world at the player's location.

Spawning a vehicle follows the same pattern as any entity: load the model, then call the spawn function. The key difference is the heading parameter -- it controls which direction the car faces.

What's new here
  • GetEntityHeading(ped): returns the direction the player is facing in degrees (0-360). We use this so the car spawns pointing the same way the player is looking.
  • CreateVehicle(hash, x, y, z, heading, isNetwork, netMissionEntity): spawns a vehicle. Returns a handle just like CreateObject.
  • The model name is a string like "adder" or "sultan". You can find vehicle names on the FiveM docs or the GTA wiki.
Step 2

Warp the player into the vehicle

Outcome: the player is instantly placed in the driver's seat.

Spawning a car is only half the job. Without putting the player in it, they are just standing next to a parked car. Use TaskWarpPedIntoVehicle to teleport the player's ped into a seat.

Seat indices
  • -1: driver seat. This is the one you almost always want.
  • 0: front passenger.
  • 1: rear left.
  • 2: rear right.
  • These numbers are the same for every vehicle. A motorcycle only uses -1 (driver) and 0 (passenger).
Step 3

Customize: colors, plates, mod kit

Outcome: you can change a vehicle's color, number plate, and enable the mod kit for further customization.

Once you have a vehicle handle, you can customize it with natives. Here are the most common ones:

Why SetVehicleModKit?
  • SetVehicleModKit(vehicle, 0): activates the default modification kit. You must call this before using any SetVehicleMod calls (spoilers, bumpers, exhausts, etc.).
  • Without it, SetVehicleMod silently does nothing. This catches many beginners off guard.
  • SetVehicleColours does not need the mod kit -- it works on any vehicle immediately.
Finding color IDs

GTA V has over 150 vehicle color IDs. You can find a full list on the FiveM documentation or spawn a car and cycle through IDs with a loop. For testing, 0 (black), 12 (red), 64 (blue), and 111 (matte black) are easy to recognize.

Step 4

Network ownership basics

Outcome: you understand who "owns" an entity and why it matters for vehicles.

In FiveM's multiplayer world, every entity has an owner -- the player whose machine is responsible for simulating that entity's physics. When you spawn a vehicle on your client, you own it. But ownership can transfer.

  • If you get out of a car and walk away, another nearby player might become the owner.
  • The owner's machine decides the car's position, speed, and collisions. Everyone else just receives updates.
  • If the owner disconnects, the entity might get cleaned up or transferred to someone else.
Why vehicles sometimes disappear

If you spawn a vehicle with netMissionEntity = false, the game's streaming system can delete it when no player is nearby. Always use true for vehicles you want to persist. For extra safety, call SetEntityAsMissionEntity(vehicle, true, true) after spawning.

For now, just know that ownership exists. When you get to more advanced scripts (garage systems, shared vehicles), you will need to manage it actively. For test commands, the default behavior is fine.

Step 5

Cleanup: delete on resource stop + timeout

Outcome: test vehicles clean up automatically, both on resource stop and after a timer.

Combine everything from Lesson 25 with your vehicle spawner. Cleanup on resource stop is the same as before. For test vehicles, add a 60-second auto-delete timer so the world doesn't fill up with abandoned Adders.

Why kick the player out first?
  • If you delete a vehicle while the player is sitting in it, weird things happen -- the ped can fall through the ground or get stuck in a sitting animation.
  • GetVehiclePedIsIn(ped, false): returns the vehicle the player is currently in. The false means "the vehicle I am in right now" (not the last vehicle).
  • TaskLeaveVehicle: makes the ped get out. We wait a second for the exit animation, then delete.

Outcome: one command that does everything -- spawn, customize, enter, and schedule cleanup.

Common mistakes

Vehicle spawns underground (wrong Z coordinate)

If you use the player's coordinates directly, the Z (height) is usually correct. But if you hardcode coordinates, the Z might be below the ground. Use GetGroundZFor_3dCoord to find the correct ground level, or use the player's Z and add a small offset like + 1.0.

Vehicle disappears immediately after spawn

Two common causes: (1) netMissionEntity was set to false, so the game's streaming system deleted it. Use true. (2) Another script on the server is auto-deleting spawned vehicles. Check your other resources.

Forgetting to set as mission entity

For vehicles that need to persist (garage spawns, mission vehicles), call SetEntityAsMissionEntity(vehicle, true, true) right after spawning. This tells the game "do not auto-clean this, I will handle it." Without it, the game might despawn your vehicle when the player walks away.

SetVehicleMod does nothing

You forgot to call SetVehicleModKit(vehicle, 0) first. The mod kit must be activated before any visual modifications work. Colors and plates do not need it, but spoilers, bumpers, and other mods do.

Vehicle lifecycle & ownership

Handle
Local id from CreateVehicle. Use it for plate, colour, and mods while it exists.
NetId
NetworkGetNetworkIdFromEntity(veh): pass this (never the raw handle) when the server or other clients must reference the car.
Owner
Whoever spawned it owns simulation. For real garages, the server approves spawn/store and validates ownership, distance, and plate: never trust the client.
Cleanup
Test cars: auto-delete on timeout. Always also delete on player drop and onResourceStop so the world does not fill with abandoned vehicles.

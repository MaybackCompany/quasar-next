What is an entity?

An entity is anything that exists in the 3D world: a car, a person, a barrel, a helicopter. When the game creates one, it hands you back a handle -- just a number, like 12547. You use that handle every time you want to talk to that thing: move it, delete it, check if it still exists.

Think of it like a ticket number at a deli counter. The ticket is not the sandwich. But you need the ticket to pick up your sandwich.

Handle = a number

Every native that works with entities (DeleteEntity, SetEntityCoords, FreezeEntityPosition) takes a handle as its first argument. Lose the handle, lose control of the entity.

Entity types

FiveM has three main categories of entities:

  • Peds -- pedestrians. NPCs, shopkeepers, quest givers, enemies. Created with CreatePed.
  • Vehicles -- cars, bikes, boats, helicopters. Created with CreateVehicle.
  • Objects (props) -- everything else. Barrels, cones, barriers, furniture. Created with CreateObject.

They all follow the same lifecycle: load the model, spawn the entity, use the handle, clean up when done.

Step 1

Model loading: the rule you must never skip

Outcome: you understand why you must load a model before spawning, and you can write the loading loop.

Before you can spawn anything, the game needs to load the 3D model into memory. Think of it like downloading a file before you can open it. The pattern is always the same:

  1. Ask for the model with RequestModel(hash).
  2. Wait until the game says it is ready: loop on HasModelLoaded(hash).
  3. Spawn the entity.
  4. Tell the game you no longer need the raw model data: SetModelAsNoLongerNeeded(hash).
Line by line
  • joaat(model): converts a string name like "prop_barrel_02a" into a hash number. The game uses hashes internally, not strings.
  • RequestModel(hash): tells the game "start loading this model." It does not wait. It just starts the download.
  • while not HasModelLoaded(hash) do Wait(50) end: this is a polling loop. Every 50 milliseconds, it checks "loaded yet?" The Wait(50) is critical -- without it, this loop would freeze your entire game.
  • return hash: we return the hash so the caller can use it for spawn and cleanup.
Never skip the wait loop

If you call CreateObject before the model is loaded, you get handle 0 -- meaning the spawn failed. This is the number one beginner mistake with entities.

Step 2

Spawn functions: CreateObject, CreatePed, CreateVehicle

Outcome: you know the three main spawn functions and what arguments they take.

Each entity type has its own spawn function. They all return a handle (a number).

What the boolean flags mean
  • isNetwork: if true, other players can see this entity. Almost always true.
  • netMissionEntity: if true, the entity won't be automatically cleaned up by the game's streaming system. Set to true for things you want to control yourself.
Step 3

The entity registry pattern

Outcome: you store every spawned entity in a table so you can clean them all up later.

Every entity you spawn should go into a table. This table is your registry -- your single source of truth for "what did my script create?" Without it, you lose track of handles, and lost handles mean entity leaks.

Why this matters
  • spawnedEntities: a simple list (table) of handles. Every time you spawn something, you add it here.
  • registerEntity: a tiny helper that appends a handle to the list.
  • cleanupAll: loops through every handle, checks if the entity still exists (DoesEntityExist), and deletes it. Then wipes the table.
  • The DoesEntityExist check prevents errors -- an entity might have been deleted by the game already.

Handle vs NetID -- the part nobody explains

So far every handle you saw is a local number. The barrel handle 12547 on your machine is 89221 on your friend's machine and 4017 on the server. Same entity, three different handles. That is by design: each runtime keeps its own list of "things I currently know about."

To talk about the same entity across machines you need a number both sides agree on. That is the NetID -- the network id. It is stable across the server and every client that has the entity streamed in.

One sentence

Handle = local pointer (different per machine). NetID = network id (same on every machine that sees the entity).

When do you actually need this?
  • Sending entity references over net events. Never send a handle. Send a netId. The handle on the receiving side is meaningless.
  • Storing references in SQL. netId is the safe column.
  • Server tells the client about a vehicle it just spawned. Server has handle X, client needs handle Y -- they bridge with the netId.
Sending a handle over a net event is a silent bug

The number arrives, the receiver calls DoesEntityExist(handle) on it, and may even get a true result -- because that number happens to be a valid handle for a different entity on that machine. You then move/delete the wrong thing. Always convert to netId before sending.

Server-authoritative spawn

Up to now everything spawned on the client. That is fine for cosmetic props the player drops in front of themselves. It is the wrong answer for vehicles, NPCs, and anything that other players must see, share, or trust.

Why client-only spawn breaks at scale
  • Other players only see the entity if they are streamed in at spawn time.
  • If the spawning client disconnects, the entity may despawn or become orphaned.
  • Anyone can call the same client native and spawn whatever they want -- the server has no record of who created it.

The server-authoritative pattern flips the order: client asks, server checks, server spawns, client takes ownership.

What changed compared to client spawn
  • CreateVehicleServerSetter is server-side. CreatePedServerSetter and CreateObjectServerSetter are the two siblings.
  • No model loading. The server does not own a renderer, so RequestModel/HasModelLoaded are not needed (and would fail). Each client that streams the entity in loads the model on demand.
  • local src = source is required. source is volatile -- it is reset by the next Wait, callback, or yield.
  • The server returns the netId via a targeted client event. The client never sends or receives a raw handle.
The trust rule restated

If money, jobs, cooldowns, or fairness are involved, the server owns the spawn. Client requests are inputs, not commands. The server is the only place that decides yes or no.

Recap 1

Model loading + spawn + register

Put the pieces together. Write a command that loads a barrel model, spawns it in front of the player, and registers the handle.

Why do we call SetModelAsNoLongerNeeded?

It tells the game "I'm done loading this model, you can free the raw data from memory." The entity itself stays. Think of it like unpacking a box: you keep the item but throw away the packaging. If you skip this, the model data stays in memory forever, which adds up.

Step 4

Basic cleanup: DeleteEntity

Outcome: you can delete a single entity or all registered entities on demand.

Deleting an entity is one line: DeleteEntity(handle). But you need the handle. That is why the registry exists.

Type /spawnbarrel a few times, then /clearprops. Every barrel disappears. The table is empty. No leaks.

Recap 2

Spawn and delete cycle

Test the full cycle in-game:

  1. Type /spawnbarrel three times. Three barrels appear in front of you.
  2. Type /clearprops. All three barrels vanish.
  3. Check the F8 console for the print messages confirming handles and cleanup count.

Common mistakes

Entity handle is 0 after spawn

This means the model was not loaded yet when you called the spawn function. You either skipped RequestModel, or you did not wait for HasModelLoaded to return true. Always use the loading loop from Step 1.

Model not loaded before spawn

Same root cause as above. The most common version: you called RequestModel but forgot the while not HasModelLoaded loop. RequestModel only starts the load -- it does not wait for it to finish.

Forgetting SetModelAsNoLongerNeeded

Your entity still works fine, but the raw model data stays in memory. Do this once or twice and you won't notice. Do it a hundred times in a busy server and you start eating RAM. Always call it right after spawning.

Entities pile up because you never delete them

Every entity you spawn stays in the world until something deletes it. If your script spawns things but never calls DeleteEntity, the server fills up with ghost objects. The registry pattern from Step 3 is how you prevent this.

Sending a handle through a net event

Handles are local numbers. TriggerServerEvent('foo', myEntityHandle) sends a number that means nothing on the server. Always convert with NetworkGetNetworkIdFromEntity first, send the netId, and let the receiver convert back with NetToVeh/NetToPed/NetToObj.

Calling RequestModel on the server

The server has no renderer. RequestModel/HasModelLoaded are client-only. If you spawn server-side, use CreateVehicleServerSetter/CreatePedServerSetter/CreateObjectServerSetter -- no model load needed.

Spawning vehicles from the client without server validation

Anyone can call client natives. If your shop, job, or garage script spawns a car directly on the client, every player can hand themselves a free vehicle by editing one line of Lua. Spawn from the server, return the netId, and let the client take ownership.

Entity lifecycle at a glance

Handle
Local id from CreateObject. Valid only while the entity exists on this client.
NetId
NetworkGetNetworkIdFromEntity(handle): the stable id the server and other clients use for the same entity.
Owner
The client simulating the entity. Can transfer. Server-spawned entities should stay server-owned.
Cleanup
Every spawn needs a delete path. No registry = leaked props. Delete on timeout, player drop, and onResourceStop.

Cheat sheet

RequestModel(hash)
Start loading a model into memory. Does not wait.
HasModelLoaded(hash)
Returns true when the model is ready to use.
SetModelAsNoLongerNeeded(hash)
Free the raw model data. The spawned entity stays.
CreateObject(hash, x, y, z, net, mission, door)
Spawn a prop. Returns a handle.
CreatePed(type, hash, x, y, z, heading, net, script)
Spawn a pedestrian. Returns a handle.
CreateVehicle(hash, x, y, z, heading, net, mission)
Spawn a vehicle. Returns a handle.
DeleteEntity(handle)
Remove an entity from the world.
DoesEntityExist(handle)
Check if an entity is still alive before acting on it.
joaat(string)
Convert a model name string to its hash number.
Entity registry
A table that stores every handle you spawn. Your cleanup safety net.
NetworkGetNetworkIdFromEntity(handle)
Convert a local handle into a netId you can safely send over events.
NetworkGetEntityFromNetworkId(netId)
Convert a netId back into the local handle on the receiving machine.
NetToVeh / NetToPed / NetToObj
Typed shortcuts for the conversion above. Client-side.
CreateVehicleServerSetter
Server-side vehicle spawn. No model loading. Returns a server handle.
CreatePedServerSetter
Server-side ped spawn. Same pattern as the vehicle setter.
CreateObjectServerSetter
Server-side prop spawn. Same pattern.
SetEntityAsMissionEntity(ent, true, true)
Tell the streaming system "this entity is mine, don't auto-clean it."

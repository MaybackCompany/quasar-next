TriggerEvent: how scripts talk to each other

A real FiveM server runs lots of resources at the same time. They do not share variables directly, so they need a way to talk. That is what events are. An event is just a named message. One script fires it, another script listens for it. This lesson starts with the simplest version: same-side events using TriggerEvent.

Before you start

How to read this lesson

This is the easy event lesson. Stay on one side only. Learn the shape TriggerEvent → AddEventHandler first. The wall-crossing versions come next lesson, not on this page.

The mental model: a shout and a listener

Think of an event as a named shout. One script says player:healed. Any script that is listening for that exact name can react. The sender does not need to know who is listening. The listener does not need to know who fired it.

That is why events are useful in FiveM. They let separate resources work together without hard-wiring themselves together.

The three verbs

TriggerEvent fires the message. AddEventHandler listens for it. RegisterNetEvent, which comes next lesson, is for messages that cross the client/server wall.

The one rule of TriggerEvent: it stays on its side

TriggerEvent never crosses the client/server wall. If you call it on the client, only client handlers can hear it. If you call it on the server, only server handlers can hear it. That is not a weird detail. That is the main rule.

Say it once, out loud: TriggerEvent is local to its side. Most beginner bugs with events start with forgetting that sentence.

Step 1

Shout: your first TriggerEvent

Outcome: a line of code that fires an event and another line that reacts to it, both on the client.

We'll build a tiny resource called qu_events. It fires a qu:client:coffee event when the player types /coffee. A second piece of code, also on the client, handles it and prints a message. One shout, one listener, in the same file, to keep it simple.

Read it from top to bottom. AddEventHandler registers a function that should run when qu:client:coffee fires. RegisterCommand makes /coffee call TriggerEvent with one argument, the string 'large'. That value then arrives in the handler as size.

Event names are yours to invent

There is no central list you have to register names in. qu:client:coffee is just a string you chose. In this course we use <resource>:<side>:<action> because it keeps names tidy and easy to search for later.

Step 4

Multiple arguments: the shape of a message

Outcome: you can pass any number of values through an event and read them in the handler.

Events are not limited to one argument. You can pass several values, and the handler receives them in the same order.

Three arguments leave the TriggerEvent call, and three parameters arrive in the handler. The types stay the same too. A string stays a string, a number stays a number, and a boolean stays a boolean.

Argument order matters, argument names don't

If you swap the order in TriggerEvent, the handler reads the wrong values into the wrong names, no error, silent bug. When an event takes more than two arguments, consider passing a single table instead: TriggerEvent('qu:client:coffee', { size='large', shots=2, oatMilk=true }). You lose positional ambiguity; you gain the ability to add fields later without breaking callers.

Recap 1

The silent handler

This code runs without errors, but nothing prints when /wave is typed. Find the bug before scrolling.

The fix

Event names are case-sensitive. The handler listens for qu:client:Wave (capital W); the trigger fires qu:client:wave (lowercase). FiveM does not warn you, it quietly looks for a listener that doesn't exist, finds none, and moves on. Match the casing exactly in both places. Adopt a rule: all-lowercase except for domain acronyms, always.

Step 5

Two resources, one conversation

Outcome: you see the real power of events, resources that don't know each other still work together.

Events become essential the moment you have more than one resource. Leave qu_events alone. Create a second resource, qu_loyalty, that adds a stamp every time the first one fires qu:client:coffee: without qu_events knowing it exists.

ensure qu_loyalty in server.cfg, restart the server, run /coffee. Both resources fire on the same event. Neither imports the other. If tomorrow you delete qu_loyalty, qu_events keeps working. That independence is the superpower.

The decoupling dividend

This is how production FiveM codebases stay sane. A purchase event fires once; inventory, notifications, analytics, and the shopkeeper's till all listen separately. Any of them can be swapped out or restarted without touching the others.

Step 6

Events FiveM fires for you, for free

Outcome: you can react to a player joining, leaving, or spawning without writing a loop that checks "did it happen yet?" over and over.

So far every event you fired came from your own TriggerEvent call. Here is the nice surprise: FiveM is already firing events all the time. A player connects, a player disconnects, your resource starts up, a player respawns at the hospital. Each of those is a named event, and you can listen for it with the exact same AddEventHandler you already know.

This matters because the beginner instinct is to write a loop that keeps checking "is the player here yet? is the player here yet?". You almost never need that. You just add a handler for the built-in event and let FiveM tap you on the shoulder when it happens.

Event nameSideFires whenWhat you get
playerJoiningServerA player is connecting, just before they are fully insource is the joining player's server id
playerDroppedServerA player disconnectssource plus a reason string (timeout, kicked, quit)
onResourceStartServerAny resource starts, including yoursThe resourceName that started

onClientResourceStartClientYour resource finishes loading on this clientThe resourceName that started

Here is the server side. Notice source works here just like it will in net events next lesson, it is the player this event is about.

The two resource lifecycle events fire for every resource on the server, not only yours. So the first thing you do inside the handler is check whether the name matches your own resource. If you skip that check, your startup code runs every time any other resource restarts, which is almost never what you want.

And the client side. playerSpawned is the one you will reach for constantly, it is how you give a player their gear, set their health, or teleport them after they respawn.

The "is it mine?" guard

onResourceStart and onResourceStop are noisy on purpose, they shout for every resource so any resource can react to any other. Your job is the one-line filter if resourceName ~= GetCurrentResourceName() then return end. Get in the habit now. You will write this guard hundreds of times.

Listen, do not poll

If you ever catch yourself writing while not playerLoaded do Wait(100) end, stop. There is almost always a built-in event for the thing you are waiting on. Polling burns CPU every tick; a handler costs nothing until the moment it fires. The list above is a tiny slice, the FiveM docs have a full catalog of lifecycle and game events worth bookmarking.

Handlers run in the order they were registered, before TriggerEvent returns. That's a Lua rule, not a FiveM one, same-side events are function calls, and function calls are synchronous. Knowing this matters the day you wonder "did my handler run before the next line?", it did.

Common mistakes: spot them now, skip them later

Using TriggerEvent to talk across sides

Client TriggerEvent does not reach the server. If you need the server to hear it, the next lesson's TriggerServerEvent is the tool. Using the wrong verb is the single most common events bug for new FiveM devs.

Typos in event names

qu:client:coffe (missing e) looks identical at a glance. No error, no handler, no clue. Copy event names, do not retype them.

Registering a handler inside a command

Putting AddEventHandler inside a RegisterCommand callback adds a new handler every time the command runs. After ten uses, your single event fires ten listeners. Handlers go at the top level of the file, once.

Forgetting arguments exist

TriggerEvent('qu:client:buy') passes zero arguments. If the handler has function(itemName), itemName is nil. You won't see an error; you'll see a crash two lines later when something tries itemName:upper(). Keep argument counts aligned.

Cheat sheet: events, same-side

Fire an event
TriggerEvent('name', arg1, arg2, ...)
Listen for one
AddEventHandler('name', function(arg1, arg2, ...) end)
Scope
Same side only. Client→client, server→server.
Naming convention
resource:side:action: e.g. qu_bank:client:deposited.
Order
Handlers run in registration order, synchronously, before TriggerEvent returns.
Cross-resource
Handlers in any resource can listen to any event name fired on the same side.
Many args vs one table
Use a table when you have >2 arguments or expect the shape to evolve.

What you learned

  • TriggerEvent fires a named message on the same side -- it never crosses the client/server wall
  • AddEventHandler registers a function that runs when that event name fires
  • Multiple handlers can listen to the same event, and they all run in registration order
  • Event names are case-sensitive strings you invent -- use a clear naming convention like resource:side:action
  • Separate resources can cooperate through events without importing each other

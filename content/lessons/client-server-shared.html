Before you start

Four quick checks. Miss one and the refactor won't run.

The mental model: one resource, two places it runs

A FiveM resource runs in two different places at the same time. The client side runs on each player's PC. The server side runs once on the host machine. Same folder on disk, two separate Lua worlds when the resource starts.

The three-folder convention exists so you stop guessing. client/main.lua means player-side code. server/main.lua means host-side code. shared/ means plain Lua values and helpers that both sides can load safely.

That is the real win here: you look at the filename first, and half your confusion disappears before you even read the code.

FiveM note

FiveM does not force this layout. A flat folder with client.lua and server.lua side by side works. But every professional resource you'll ever read, every framework you'll ever extend, uses this shape. Adopting it now means the next script you open tomorrow will feel familiar instead of foreign.

Step 1

Learn the three-folder rubric

Outcome: given any piece of code, you can name the folder it belongs in within five seconds.

Memorize these three sentences. They are the whole game.

  • Client. Anything that touches the player's ped, vehicle, camera, UI, on-screen draw calls, keybinds, NUI, or an in-world prop. Natives like PlayerPedId, GetEntityCoords, DrawText, SetEntityCoords live here. This code runs on each player's machine.
  • Server. Anything that touches money, items, databases, permissions, persistent state, rule-checking, or anything a cheater must not be allowed to forge. This code runs once, centrally, on the host machine.
  • Shared. Pure Lua, configs, constants, math helpers, small utility functions. No FiveM natives. Anything in shared/ must run correctly on both sides with zero changes.
Where does this code belong?

Three questions, answer the first one that fits:

  • Does it touch a ped, vehicle, camera, or draw something on screen? → Client.
  • Does it change money, items, or data that survives a disconnect? → Server.
  • Is it a plain Lua helper or a config table? → Shared.
Step 2

Why the split is load-bearing

Outcome: you understand that the three folders are not cosmetic, they prevent three distinct categories of bug.

Three reasons the split earns its keep. Skip any one of them and you will ship something that breaks later.

  • Trust. A player's PC can be modified. Cheaters patch client Lua, inject keybinds, spoof values. The server's Lua is out of their reach. So anything that decides may this happen: money transfers, inventory changes, permissions, must live on the server. We devote all of Lesson 12 to this; for now, plant the seed: the server is the referee, the client is the player.
  • Performance. Client code runs on every connected player's PC. A loop you write on the client with 50 players online is really 50 loops running in parallel on 50 different machines. Server code runs once. Put a per-player iteration on the client by mistake and you multiply the cost by your whole playerbase.
  • Access. Some natives only exist on one side. DrawText is client-only, the server has no screen to draw on. GetPlayerPing is server-only on the server side. Call a native from the wrong world and Lua throws attempt to call a nil value. The folder convention stops that before you type the line.

Recap 1

Place these lines in the right folder

Without scrolling back, say out loud which folder each of these belongs in. Answers below.

Where does GetEntityCoords(PlayerPedId()) belong?

Client. Both natives are about the player's ped, a thing that only exists on a player's PC. The server has no concept of a single player's on-screen position in world space.

Where does local taxRate = 0.15 belong?

Shared. It's a constant, pure Lua, no natives. Both the client (showing the tax on a receipt) and the server (computing the real deduction) will want to read it.

Where does the MySQL.update call belong?

Server. Databases are persistent state you must never trust a client to change. Database calls always live server-side, full stop.

Three important things changed. You're using shared_scripts, server_scripts, client_scripts: the plural forms that take a table of paths inside curly braces. You're using a glob: shared/*.lua matches every .lua file directly inside shared/. And the three sides appear in a specific order: shared loads before the others, so Config is defined before either client or server tries to read it. Do not add lua54 'yes' to new manifests; current FiveM artifacts run Lua 5.4 by default.

Why globs beat listing files one by one
  • Add a new file to client/ later, say client/keybinds.lua: and you don't have to touch the manifest. The glob picks it up.
  • Less to read, less to forget, less to typo.
  • Everyone in the ecosystem uses this pattern. Matching the convention means your code looks like everyone else's code.
Watch out

shared/*.lua matches only .lua files directly inside shared/. It does not recurse into subfolders. If you ever add shared/core/utils.lua, you need shared/**/*.lua (two stars) to catch it, or it will load zero of the files you wrote.

The "never in shared" rule

If a file lives in shared/, it must run identically on the client and on the server. That means no FiveM natives, not one. PlayerPedId does not exist on the server side. The first time that file loads server-side, Lua hits the native, finds nothing, and crashes your resource with attempt to call a nil value.

Good shared material: config tables, constants, small math helpers (function round(n) return math.floor(n + 0.5) end), string formatters, enum-like tables. Bad shared material: anything that reads the world, draws on screen, looks up a player, or asks for the database.

If you catch yourself writing a shared helper and you need a native inside it, that's a sign the helper actually belongs on one specific side. Move it there.

"But I need to reach a variable on the other side"

You can't. The client and server run in different processes, usually on different machines. A local in client/main.lua is invisible to server/main.lua and always will be. Even a global won't cross. If they need to talk, they talk through events: a messaging system covered in Module 04.

Don't fight it. Don't try to share globals across sides. Don't invent workarounds. Today, your two sides communicate via Config (a copy of which loads into each world from shared/). Tomorrow in Module 04 they'll learn to actually talk to each other.

The hard rule, stated plainly

Client and server are two completely separate Lua programs. They do not run in the same place, they do not share memory, and nothing syncs between them by magic. If the client changes a value, the server never finds out unless the client explicitly sends a message. If the server changes a value, the client stays in the dark until the server tells it. There is no hidden wire connecting the two. Every piece of data that crosses the gap crosses it because you wrote code to send it. Burn this in now: it removes about half the confusion in every later FiveM lesson.

A quick word on resource names

Resource folder names usually carry a prefix that tells you where they came from:

  • qu_: Quasar Academy (what we use here).
  • qs-: Quasar Store premium scripts.
  • qb-: QBCore framework resources.
  • esx_: ESX framework resources.
  • ox_: Overextended ecosystem resources.

Pick one prefix for your own work and stay consistent. It makes every resource you write instantly searchable and groups your scripts together in the resources folder.

Common mistakes & what the console tells you

Error: attempt to call a nil value (global 'PlayerPedId') from a shared file

You put a FiveM native in shared/. That native only exists on the client. The moment the file loads server-side it finds nothing and throws. Move the native call into client/, or split the helper, the native-free part stays in shared, the native-using part moves.

A variable from client/main.lua is nil in server/main.lua

It will always be nil. The two sides are separate processes. Client variables cannot reach the server and server variables cannot reach the client. If they need to exchange data, you need events: Module 04. For static values both sides need to agree on, put them in shared/config.lua.

Error: manifest loads but nothing runs

Check the spelling carefully. shared_script 'file.lua' (singular, no braces) takes a single string. shared_scripts { 'file.lua' } (plural, curly braces) takes a table and supports globs. Mix them up, say, shared_scripts 'shared/*.lua' without braces, and FiveM may silently ignore the entry. Plural forms with braces are what you want here.

Files in a subfolder aren't loading

shared/*.lua matches only files directly inside shared/. If you made shared/core/helpers.lua, that glob misses it entirely. Use shared/**/*.lua (double star) to recurse, or list the subfolder explicitly: 'shared/core/*.lua'.

What you learned

  • The three-folder convention (client/, server/, shared/) tells you what code belongs where before you read a single line.
  • Client code runs on each player's PC (peds, camera, UI). Server code runs once centrally (database, money, permissions).
  • Shared code must be pure Lua with no FiveM natives -- it loads into both runtimes independently.
  • Glob patterns like client/*.lua in the manifest auto-load new files without editing fxmanifest.lua each time.

NUI from scratch: ui_page, messages, callbacks

NUI is FiveM's name for the in-game browser layer, a real Chromium tab the client renders on top of GTA. Every menu, hud, scoreboard, F-pause UI you have ever seen in a server is NUI. So far you have used ox_lib and other framework UIs that wrap NUI for you. This bonus shows the layer underneath: how to declare a page in fxmanifest.lua, how to push state from Lua with SendNUIMessage, how to receive clicks with RegisterNUICallback, and how to capture the cursor with SetNuiFocus. Once you understand these four pieces, you can read any NUI resource and build your own without a framework.

Before you start

Quick check. NUI sits between Lua and a real browser, so you need both sides to be comfortable.

Why learn raw NUI when frameworks exist

Frameworks like ox_lib, menuv, and nh-context are great for shipping fast. They are also the reason most beginners cannot debug a broken UI: when the menu does not open, you do not know whether the bug is in your code, the framework's NUI bridge, or the focus state. After this lesson you can open the page directly, watch the message bus, and see exactly which step failed.

  • NUI is just Chromium. Same DOM, same JS, same devtools you use for any web page. There is no FiveM-specific UI runtime.
  • Lua and JS communicate two ways. Lua-to-JS uses SendNUIMessage + window.addEventListener('message'). JS-to-Lua uses fetch() + RegisterNUICallback.
  • Focus is a separate concern. The browser is always rendered, but it does not capture mouse/keyboard until you call SetNuiFocus. Forget this and your buttons will look clickable but do nothing.
Step 1

Declare a page in fxmanifest.lua

Outcome: when the resource starts, the client has a hidden Chromium tab loaded with your html/index.html.

Folder layout and the manifest 
qu_menu/
  fxmanifest.lua
  client.lua
  html/
    index.html
    style.css
    script.js
-- fxmanifest.lua
fx_version 'cerulean'
game 'gta5'

client_script 'client.lua'

ui_page 'html/index.html'

files {
    'html/index.html',
    'html/style.css',
    'html/script.js',
}
  • ui_page tells the client which file to load into the Chromium tab. There is exactly one per resource.
  • files {} is the allowlist of everything the page is allowed to fetch. If you forget to list a file, it 404s silently and the page renders without it.
  • You can list images, fonts, or extra JS bundles here too. Anything fetched from the page must be in this block.
The cfx-nui-<resource>/ protocol scope

Inside html/index.html, every URL is resolved against the cfx-nui-qu_menu/ scheme, a custom protocol that maps to the files {} block. So this works in the HTML:

<link rel="stylesheet" href="style.css">
<script src="script.js"></script>
<img src="logo.png">

And the page will fetch cfx-nui-qu_menu/style.css, cfx-nui-qu_menu/script.js, and so on. You can also reference assets from a different resource by hard-coding the scope: <img src="https://cfx-nui-qu_shared/icons/coin.png">. The old nui:// scheme is deprecated, use cfx-nui-.

Recap 1

Make a new resource called qu_menu. Create fxmanifest.lua with the manifest above and an empty html/index.html that just contains <h1>hello nui</h1>. Start the resource. You will not see anything in-game yet (focus comes later), but pressing F8 and typing nui_devTools on a focused NUI page would show the empty document. Confirm the resource starts with no errors in the F8 console.

Step 2

Lua → JS: SendNUIMessage

Outcome: a Lua command pushes data into the page, and the JS receives it via the standard message event.

The Lua side 
-- client.lua
RegisterCommand('menu', function()
    SendNUIMessage({
        type    = 'open',
        title   = 'Welcome',
        balance = 1234,
    })
end, false)

SendNUIMessage takes one argument: a Lua table. The runtime serializes it to JSON and posts it into the page. Any JSON-friendly value works (strings, numbers, booleans, arrays, nested tables).

The JS side 
// html/script.js
window.addEventListener('message', (event) => {
    const data = event.data;            // the table you sent from Lua
    if (data.type === 'open') {
        document.getElementById('title').textContent = data.title;
        document.getElementById('balance').textContent = data.balance;
        document.body.classList.add('visible');
    }
    if (data.type === 'close') {
        document.body.classList.remove('visible');
    }
});

The convention is to put a type key in every message and switch on it, just like a Redux action. There is exactly one message bus, so every message goes through this single listener; the type field is how you route.

The page never unloads

Your index.html loads once, when the resource starts, and stays alive for the rest of the session. "Opening" and "closing" the menu is a CSS class toggle, not a page navigation. Treat the JS like a long-running app, not a request-response page.

Recap 2

Add the command above to client.lua and the listener to script.js. Reload your resource, type /menu in the chat, and watch the F8 console for any errors. The page won't be visible yet (no CSS for .visible, and no focus). That is normal.

Step 3

JS → Lua: RegisterNUICallback

Outcome: a button click in the page triggers a Lua function and gets a JSON response back.

The JS side: fetch with the parent resource scope 
// html/script.js
document.getElementById('confirmBtn').addEventListener('click', async () => {
    const res = await fetch(`https://${GetParentResourceName()}/confirm`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json; charset=UTF-8' },
        body: JSON.stringify({ amount: 500 }),
    });
    const reply = await res.json();
    console.log('lua replied:', reply);
});

GetParentResourceName() is a function the FiveM Chromium provides on window. It returns the string "qu_menu", or whatever the resource is called. The full URL https://qu_menu/confirm is what the runtime intercepts and routes to your Lua callback.

The Lua side: RegisterNUICallback 
-- client.lua
RegisterNUICallback('confirm', function(data, cb)
    print(('confirm received: amount=%s'):format(tostring(data.amount)))

    TriggerServerEvent('qu_menu:confirm', data.amount)

    cb({ ok = true })   -- ALWAYS call cb, even with an empty object
end)
  • data is the parsed JSON body the JS sent. data.amount is 500.
  • cb is the response function. Whatever you pass becomes the body of the JS fetch response.
  • The callback name ('confirm') must match the URL path (/confirm).
Always call cb(...): even when you have nothing to return

If your callback returns without invoking cb, the JS fetch never resolves. Future calls to the same callback queue up and never fire. The page goes silent. The fix is to always call cb, even if it is just cb({ ok = true }) or cb('ok'). Make this the first muscle memory you build.

Recap 3

Add a <button id="confirmBtn">Confirm</button> to index.html, the click handler to script.js, and the RegisterNUICallback to client.lua. Trigger /menu, click the button, and confirm both the Lua print and the JS console.log('lua replied:', { ok: true }) show up in F8.

Step 4

Capturing the cursor: SetNuiFocus

Outcome: when the menu is open, the mouse moves freely over the page; when closed, the player can drive again.

The two booleans 
-- client.lua
local function openMenu()
    SetNuiFocus(true, true)            -- (hasKeyboardFocus, hasCursor)
    SendNUIMessage({ type = 'open' })
end

local function closeMenu()
    SetNuiFocus(false, false)
    SendNUIMessage({ type = 'close' })
end

RegisterCommand('menu', openMenu, false)

-- close from the page (Esc key handled in JS, calls back here)
RegisterNUICallback('close', function(_, cb)
    closeMenu()
    cb({ ok = true })
end)
  • First boolean: keyboard focus. true means typing in an <input> works and the chat hotkey (T) is suppressed.
  • Second boolean: cursor. true shows the mouse pointer and lets you click DOM elements.
  • You almost always want (true, true) for an interactive menu and (false, false) when closing it.
Closing from the JS side: the Esc key pattern 
// html/script.js
document.addEventListener('keyup', (e) => {
    if (e.key === 'Escape') {
        fetch(`https://${GetParentResourceName()}/close`, {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({}),
        });
    }
});

This is the canonical pattern. The page detects Esc, posts a callback, and Lua releases focus. Do not use Lua's IsControlJustPressed for Esc-to-close once focus is captured, the keyboard belongs to the browser.

Forgetting to release focus is the #1 NUI bug

If you call SetNuiFocus(true, true) and the page errors out before reaching the close path, the player is stuck with a frozen cursor and no game input. Always wrap focus-capture work in clear open/close functions, and add a fallback admin command that calls SetNuiFocus(false, false) to recover.

Step 5

Debugging: F8 console and Chrome devtools

Outcome: you can attach Chrome devtools to your running NUI page and inspect the DOM, set breakpoints, and watch console.log in real time.

Enable the inspector once

Add this line to your citizen.cfg or pass it as a command-line flag when launching FiveM:

nui_devToolsUrl http://localhost:13172/

Now, with FiveM running, open Chrome (the real browser) and visit:

http://localhost:13172/

You will see a list of every NUI page currently loaded, one entry per resource that has a ui_page. Click your qu_menu page and a full Chrome devtools window opens, attached to the live in-game browser. console.log from your script.js shows up here. You can edit DOM nodes and watch them change in-game instantly.

The F8 in-game console

Press F8 to open the FiveM client console. This shows:

  • print() calls from any client-side Lua, including yours.
  • JS console.log from focused NUI pages (a subset, full output requires localhost:13172).
  • Resource start/stop messages and 404s for missing files {} entries.

You will live in F8 while building. The two-window setup, F8 for Lua, localhost:13172 for the page, is how every FiveM UI dev works.

The nui_devTools command

If your NUI page has focus (you called SetNuiFocus(true, true) and the menu is open), press F8 and type:

nui_devTools

This opens devtools attached to the focused page directly, without needing to navigate localhost. Convenient when you want to inspect the page that is currently capturing input.

Recap 4

Open http://localhost:13172/ in Chrome. Find the qu_menu page. Click it, switch to the Console tab, and type document.body.style.background = 'red'. Confirm the in-game NUI page turns red instantly.

Mistakes & gotchas

1. File loads but is missing: forgot the files {} entry

Adding <script src="extra.js"> without listing 'extra.js' in files {} 404s silently. The page renders without the script; nothing in your event handlers runs. Watch F8 for cfx-nui-... 404 lines.

2. fetch hangs because the callback never returned

You forgot to call cb(...) in your RegisterNUICallback. The JS await fetch never resolves; subsequent fetches queue and stall. Always invoke cb even if it is just cb({ ok = true }).

3. Menu open but clicks do nothing: forgot SetNuiFocus

The page is rendered but the cursor is invisible and clicks fall through to the game. Call SetNuiFocus(true, true) when opening, and SetNuiFocus(false, false) when closing.

4. Hard-coded resource name in fetch

Writing fetch('https://qu_menu/confirm', ...) works until you rename the resource. Always use GetParentResourceName(), the FiveM Chromium injects it on window exactly so your code is portable.

5. Calling SetNuiFocus from the server

NUI is a client concept. SetNuiFocus, SendNUIMessage, and RegisterNUICallback all live in client Lua. From the server, trigger a client event and let the client handle the focus and message.

6. Using deprecated nui:// URLs

Old tutorials use nui://qu_menu/.... The modern scheme is https://cfx-nui-qu_menu/... for asset references and https://qu_menu/... (resolved through GetParentResourceName) for callbacks. Use the new forms in any new code.

7. Listening for messages with the wrong target

window.addEventListener('message', ...) is correct, the message arrives on the window. Code that uses document.addEventListener('message') never fires. Same for document.body. The window is the only valid target.

Recap

  • NUI is a Chromium tab the client renders on top of the game. The DOM, JS, and devtools are exactly what you would expect from a web page.
  • ui_page declares the entry HTML; files {} is the allowlist of every asset the page can fetch. Forget a file and it 404s silently.
  • Lua → JS: SendNUIMessage(table) ↔ window.addEventListener('message', e => ...). Use a type field to route.
  • JS → Lua: fetch(`https://${GetParentResourceName()}/path`) ↔ RegisterNUICallback('path', function(data, cb) cb(...) end). Always call cb.
  • SetNuiFocus(keyboard, cursor) is what makes the page interactive. Without it the menu is decorative.
  • Debug with the F8 client console (Lua + a subset of JS) and a real Chrome at http://localhost:13172/ (full devtools per page).
  • NUI APIs are client-side only. From the server, trigger a client event.

Exercise: confirm-purchase popover

Build a tiny resource called qu_confirm that lets a server event ask the client "spend $500?" and gets back yes or no. Concretely:

  1. Resource has fxmanifest.lua, client.lua, html/index.html, html/style.css, html/script.js. All five HTML/CSS/JS files declared in files {}.
  2. The page is hidden by default (body { display: none }) and shows when it receives a { type: 'ask', cost: 500 } message. It has two buttons: Confirm and Cancel.
  3. Server fires TriggerClientEvent('qu_confirm:ask', target, 500). Client handler captures focus, sends the message, and waits.
  4. Each button calls fetch(...) with body { choice: 'yes' } or { choice: 'no' }. Lua receives, releases focus, and triggers TriggerServerEvent('qu_confirm:answer', choice). Always call cb({ ok = true }).
  5. Server prints the answer.

Stretch goal: add a 10-second timer in JS that auto-fires { choice: 'no' } if the player hesitates. Open Chrome at localhost:13172 while testing and watch the message bus in the Console tab.

Words you will see in this lesson

Read these once, then keep going. Every term is also defined the first time it appears in the steps.

  • Resource: a folder of script files that the FiveM server loads. Yours will be called qu_tp.
  • Client: code that runs on the player's computer. It draws menus and moves the character on screen.
  • Server: code that runs on the host machine. It is the only place trusted to decide if a player is allowed to do something.
  • Context menu: the list-style popup ox_lib draws on screen when you call lib.showContext.
  • Native: a built-in GTA V function like SetEntityCoords. You call it from your Lua script.
  • Ground z: the Z (height) value of the floor at a given X, Y. If you teleport below the floor, you fall through the map.
  • Collision: the floor and walls of the map at a given spot. The game loads collision in chunks. If you teleport before it loads, your character falls.
  • Screen fade: a black overlay that fades in or out. We use it so the player never sees the teleport happen mid-frame.
  • ACE permission: a permission flag granted in server.cfg. Used to decide who is allowed to run a command.

What you are about to build

You will create one resource folder, four files, and roughly fifty lines of Lua. The player types /tpmenu, picks a place, and arrives there. That is the whole feature. Everything else in this lesson is making that one thing safe and clean.

Plan

The plan in one table

Read this before you write any code.
What the player does
Types /tpmenu in chat. A menu of place names appears. They click one. The screen fades to black. Their character moves to the saved coordinates. The screen fades back in.
What the client does
Draws the menu, sends the chosen key (a short word like sandy) to the server, and runs the fade and the move when the server says yes.
What the server does
Checks the player has permission. Looks up the key in its own config. Sends the real coordinates back to the player.
The trust rule
The client never sends coordinates. It sends a key like sandy. The server holds the coordinates. A cheater cannot ask to be teleported to 0, 0, 0 because the server simply does not accept coordinates.

PlayerPedId · SetEntityCoords · RequestCollisionAtCoord · HasCollisionLoadedAroundEntity · FreezeEntityPosition · DoScreenFadeOut · DoScreenFadeIn · IsScreenFadedOut
Events
qu:server:tp from client to server (the request). qu:client:doTp from server to client (the approved coordinates).
Out of scope
Teleporting other players. Cooldowns. A teleport history. All saved for a v2 you can build later.

  2. Paste the code below and save.

Plain English. Config.locations is a Lua table. Each entry has a short name (the key, like sandy), a friendly label that shows in the menu, and a vec3 with the X, Y, and ground Z values. To add a new place later, you copy one block and change the values. No other file changes.

How to find good coordinates

In game, run a small dev command that prints your current position, or use a map tool. Always include the Z value for the floor at that spot, not the height of the roof you are standing on.

Step 3

Learn the safe-teleport pattern with one bad and one good example

Goal: see why a one-line teleport is unsafe and what the safe version adds.

Before you write the real client file, look at this side-by-side. The bad version is a real anti-pattern. Read both, then use the good version in Step 4.

Don't do this

One line. No fade. No collision wait. The player can fall through the floor and they will see the camera jump.

The character lands at 0, 0, 0 which is in the ocean. The world has not loaded around them. The screen never fades, so the player sees the cut. This is the version that gets a script labeled "buggy".

Do this

Fade out. Wait for the fade to finish. Freeze the character. Ask the game to load collision. Move. Wait for collision. Unfreeze. Fade in.

The character is hidden behind a black screen for the whole move. The floor is loaded before the freeze ends, so they do not fall. The 2000 ms timeout protects the player if collision never loads on a slow connection.

Pin this

The safe-teleport recipe, in order

  1. Fade out with DoScreenFadeOut(400).
  2. Wait until IsScreenFadedOut() returns true.
  3. Freeze the ped with FreezeEntityPosition(ped, true) so they cannot fall while collision loads.
  4. Request collision with RequestCollisionAtCoord(x, y, z).
  5. Move the ped with SetEntityCoords.
  6. Wait until HasCollisionLoadedAroundEntity(ped) is true, with a 2000 ms timeout as a safety net.
  7. Unfreeze the ped.
  8. Fade in with DoScreenFadeIn(400).
Step 4

Write the client file

Goal: /tpmenu opens a menu and clicking a place sends the key to the server.

  1. Inside qu_tp, create a new file called client.lua.
  2. Paste the code below and save.

Plain English, in three blocks.

  • Block 1, buildMenuOptions. Walks every entry in your config and returns the list shape ox_lib expects. The onSelect on each entry sends the key (not the coordinates) to the server.
  • Block 2, the menu and the command. lib.registerContext registers the menu under the id qu_tp_menu. RegisterCommand('tpmenu', ...) binds the chat command. lib.showContext opens the menu.
  • Block 3, the safe-teleport handler. Same recipe you saw in Step 3. Runs only when the server tells you to.
Why while not IsScreenFadedOut() do Wait(0) end

DoScreenFadeOut(400) starts a 400 ms fade but the next line runs immediately. The wait loop pauses the script until the screen is actually black. Skip it and the player sees the cut.

Step 5

Write the server file

Goal: only allowed players can teleport, and only to known places.

  1. Inside qu_tp, create a new file called server.lua.
  2. Paste the code below and save.

Plain English, line by line.

  • local src = source stores the id of the player who sent the event. This is the only player id you trust.
  • type(locationKey) ~= 'string' rejects anything that is not text (numbers, tables, nil).
  • IsPlayerAceAllowed checks the ACE permission you grant in server.cfg. No flag means no teleport.
  • Config.locations[locationKey] looks the key up. If the key is not in the table the player gets a polite error and nothing else happens.
  • TriggerClientEvent('qu:client:doTp', ...) sends the real coordinates back to the same player.
Why the client never sends coordinates

If the client could send x, y, z directly, a cheater could send any values they wanted. Sending only a key from a server-controlled list closes that hole. The client says "I want sandy", the server decides what sandy means.

Step 7

Run the three tests

Goal: confirm the happy path, the deny path, and the unknown-key path all behave correctly.

Do every test. Do not skip the deny path. The deny path catches more bugs than the happy path.

  1. Test 1, the happy path.
    • Action: connect with your admin license, type /tpmenu, click Sandy Shores.
    • Expected: the menu appears, the screen fades to black, your character arrives in the desert, the screen fades back in, your feet are on the ground.
    • How to verify: your minimap shows you in Sandy Shores. You did not fall.
  2. Test 2, the deny path.
    • Action: remove your license from group.admin in server.cfg, restart, reconnect, type /tpmenu, click any location.
    • Expected: a red ox_lib notification reads "You do not have permission to teleport." Your character does not move.
    • How to verify: your position on the minimap stays exactly where it was.
  3. Test 3, the unknown-key path.
    • Action: as admin, open the F8 console and run TriggerServerEvent('qu:server:tp', 'rainbow_moon').
    • Expected: a red ox_lib notification reads "Unknown location." No teleport happens.
    • How to verify: the server console prints no errors. Your position does not change.
Always test the deny path

"Does the menu work for me" is half the test. "Does it correctly stop someone without permission" is the other half. Run both, every time you change a feature.

Step 8

Optional, per-location permissions

Goal: some places can be admin-only, others police-only, without rewriting anything.

Give every location an optional ace field. When it is present, the server checks that flag instead of the global one. When it is missing, the server falls back to command.tpmenu.

Four extra lines. Now you can have admin-only places, police-only places, and EMS-only places, all decided by which ACE flags you grant in server.cfg. The client file stays untouched. That is the sign of clean separation.

The menu can also hide restricted places

A future version of buildMenuOptions can ask the server which places this player can use and only show those. Cleaner experience. The server check stays as the real defence either way.

Common mistakes and how to avoid them

Sending coordinates from the client

The client must only send a key. The server holds the coordinates. If you ever change that shape, the menu becomes "teleport anywhere a cheater asks". Key in, server lookup, coordinates out. Always in that order.

Forgetting to wait for the fade

Without while not IsScreenFadedOut() do Wait(0) end the player sees the cut. The fade looks broken. Ten characters of code, big visual difference.

Skipping the collision wait

Without RequestCollisionAtCoord and the HasCollisionLoadedAroundEntity wait, the character can land before the floor exists. They fall. The 2000 ms timeout protects them on slow connections.

Typo in the ACE flag

If command.tpmenu in server.cfg does not match command.tpmenu in IsPlayerAceAllowed, the check silently fails. Open both files side by side when permission seems wrong.

Skipping the deny test

Confirming admins can teleport is half the work. Confirm a non-admin cannot. Confirm an unknown key returns the error notify. Sad paths are real tests.

## Roles and escalation

<StepGroup>
  <Step num={1} title="Define role tiers" outcome="Every member has exactly one clear standing.">
    Keep tiers few and obvious: Member, Supporter, Staff, Admin, Owner. More tiers means more confusion. Map perks and access to these tiers, not to one-off custom roles.
  </Step>

  <Step num={2} title="Set up a ticket flow" outcome="A player issue becomes a tracked, assignable item.">
    Use a ticket bot. Opening a ticket creates a private thread or channel only the player and authorized staff can see.

    Set the panel to ask four things: what broke, when it happened, the in-game character name, and any console error.

    Ask for a raw player identifier (the long `license:...` or `steam:...` string FiveM uses to tell players apart) only when staff genuinely need it. Keep it in the private ticket. Decide how long you keep it and when you delete it. Identifiers and logs are user data, not decoration.

    Treat transcripts and screenshots the same way. **Redact identifiers before pasting them anywhere outside the private ticket.**
  </Step>

  <Step num={3} title="Write the escalation path" outcome="Staff know who handles what without asking.">
    Write it down plainly: support staff handle player issues, admins handle bans and server changes, the owner handles money and policy. Put it in staff-only so nobody freezes during an incident.
  </Step>

  <Step num={4} title="Establish an announcement rhythm" outcome="Players trust one channel for real news.">
    Post launches, updates, and events in announcements only. A predictable rhythm, like a weekly update and event teasers, keeps players returning.

    <ExpectedOutput>
    Your Discord has read-only welcome and announcement channels, a private staff-only channel, a private logs channel with its webhook URL stored as a secret, a few role tiers that match your FiveM permissions, and a ticket flow that captures evidence.

  </Step>
</StepGroup>

## How it works

Two pieces of this kit look simple but trip up real servers: the logs webhook and the ticket thread. Both are about treating community ops like infrastructure, not chat.

### Why the logs webhook URL is a secret

A **webhook** is a one-way address Discord gives you so an outside program can post messages into a channel. No login. No bot. Just the address.

To create one, open the logs channel settings (the gear icon, or right-click the channel) -> Integrations -> Webhooks -> New Webhook -> Copy Webhook URL. Discord hands you a URL that looks like this:

```text
https://discord.com/api/webhooks/123456789012345678/AbCdEf...your-token...
```

**That URL is a credential, not a link.** Anyone who has it can post a message to your logs channel as your server, with no login and no permission check. There is no separate password. Having the URL is the permission.

If it leaks, someone can spam your staff channel, impersonate your bot, or bury a real incident under noise. Your only fix is to delete the webhook and make a new one. You do that in the same place you created it: channel settings -> Integrations -> Webhooks -> select the webhook -> Delete Webhook. Then create a fresh one and update the value in `server.cfg`. The old URL stops working the moment you delete it.

A server-side resource, built in a later lesson, reads the value below and posts to this URL. This lesson only sets up the channel and stores the secret. So the logs channel stays empty until that resource exists. **That empty channel is expected, not a mistake.**

Want to confirm the webhook works before that resource exists? Send one test message from PowerShell. Replace the URL with your own:

```powershell
curl.exe -H "Content-Type: application/json" -d '{"content":"webhook test"}' "https://discord.com/api/webhooks/123.../AbCd..."
```

A single line saying "webhook test" appears in your logs channel. Delete that test message afterward.

Store the webhook URL the same way you store every other secret. Do not paste it in a public channel, a screenshot, or a commit. It belongs in your `server.cfg` as a **convar** (a config variable the server reads at startup), never hard coded into a script you might share:

```cfg
set discord_log_webhook "https://discord.com/api/webhooks/.../..."
# Use `set` (server-side only). Do NOT use `setr` (it replicates the value to every
# connected client) or `sets` (it publishes the value on the public server list and the
# open http://<server_ip>:<server_port>/info.json file). Either one leaks the webhook.
```

This is the same rule code review enforces on every other secret. Keys and tokens live in config the server reads, not in the source. A webhook URL is no different.

### How a ticket maps to the incident runbook

An **incident runbook** is your written plan for handling a server problem: each problem is one tracked item with one owner and a known set of steps.

A ticket bot that opens a private thread is doing more than starting a chat. The thread is a per-incident container: one issue, one place, only the player and staff inside. That maps directly onto the runbook, where each incident is its own tracked item with a clear owner.

The fields you make the panel collect (what broke, when, in-game name or identifier, the error text) are the inputs the runbook needs first: who and where to reproduce it, what the symptom was, and the raw error to search the logs for. Capturing them up front means staff are not chasing the player for basics before they can start.

The thread title becomes the incident. The fields become the report. Your escalation path decides who picks it up. **The ticket is where an incident is born.**

## Common mistakes

<FailureList
  items={[
    { symptom: "Support is buried in general chat", fix: "Move it to a dedicated ticket flow. Keep community chat and support separate." },
    { symptom: "Too many roles, nobody knows who does what", fix: "Collapse to a few tiers and document the escalation path in staff-only." },
    { symptom: "Log webhook posted publicly", fix: "The webhook URL is a secret. Keep it in a private channel and in server convars only, never in a screenshot or commit." },
    { symptom: "Players miss updates", fix: "One read-only announcements channel with a steady rhythm. Do not scatter news." },
    { symptom: "@everyone or @here gets abused by members", fix: "Remove the Mention @everyone permission from member roles. Only admins should have it." },
    { symptom: "Channel list is a mess, nobody knows where to post", fix: "Keep it simple: welcome, announcements, general, support, staff-only, logs. Add more only when a channel gets too noisy." },
    { symptom: "Members leave because the server feels dead", fix: "Post regular updates (events, changes, new content). A weekly update post keeps people checking back." },
  ]}

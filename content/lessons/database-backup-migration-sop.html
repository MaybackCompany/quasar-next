Database backup and migration SOP

By the end of this lesson you will have an automated backup script, know how to restore from any backup, and be able to migrate your database to a new server without losing a single row.

You'll ship
A backup script that runs daily and a tested restore procedure you can follow under pressure.
Time
About 25 minutes.
You'll learn
How to back up, restore, and migrate a MySQL/MariaDB database on a FiveM server.
Prereqs
You have MySQL or MariaDB running with at least one database and table.
In 3 bullets

You will create a backup and migration procedure:

  • Dump your entire database to a timestamped file with one command.
  • Test that the backup actually restores by loading it into a temporary database.
  • Write a migration checklist so you can move databases between servers without fear.

Step by step

Lesson 29: Database backup and migration

Start here

What this is
A backup is a complete copy of your database at a point in time.
Example: backing up before installing a new script means you can undo any damage.
Where to find it
Use mysqldump — it is built into MySQL and MariaDB.
Example: mysqldump -u root mydb > backup.sql creates a full copy.
How to use it
Dump, verify, store off-server. Never keep your only backup on the same machine.
Example: copy the backup file to your local PC or cloud storage.
What to do now
Take a backup of your database right now. Then test restoring it.
Example: restore to a different database name to verify it works.

Outcome

You will back up and restore your database, then write a migration checklist.

Why it matters

A server with no backup is one bad update away from losing every player's progress, inventory, and money.

Vocabulary

mysqldump
Command-line tool that exports a database to a SQL text file.
Example: every table, row, and index becomes CREATE and INSERT statements.
Restore
Running a backup file to recreate the database exactly as it was.
Example: mysql -u root newdb < backup.sql rebuilds everything.
Migration
Moving a database from one server to another without data loss.
Example: dump on old server, transfer file, restore on new server.

Checklist

  1. Create backups/ folder in your server root.
  2. Set MySQL credentials as convars in server.cfg.
  3. Run /db_backup and verify the file exists with real data.
  4. Restore the backup to a test database: mysql -u root testdb < backup.sql.
  5. Query the test database to confirm the data is there.
  6. Write your migration checklist: dump, transfer, restore, verify, update connection string.

Exercise

Take a backup, intentionally break something in your test database (rename a table), then restore from the backup. Verify the table is back.

Expected output: After restore, the renamed table returns with its original name and data.

Common failures

  • mysqldump: command not found means MySQL is not in your system PATH. Use the full path to mysqldump.exe.
  • Access denied for user means the MySQL credentials in your convars are wrong or the user lacks dump permissions.
  • Backup file is 0 bytes means the database name is wrong or the database is empty. Check with SHOW DATABASES;.
  • Restore fails with duplicate entry errors means you are restoring into a database that already has data. Drop the database first.

Refactor

Add a seven-day retention policy — delete backup files older than 7 days so your disk does not fill up.

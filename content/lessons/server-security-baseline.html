Server security baseline, minimum protection day one

By the end of this lesson you will have locked down your server against the five most common attacks that hit every new FiveM server within hours of going public.

You'll ship
A security checklist applied to your server that blocks resource theft, RCE attempts, event spam, and common exploits.
Time
About 30 minutes to apply all five protections.
You'll learn
Which server.cfg directives harden your server, how to block malicious events, and why a resource whitelist is non-negotiable.
Prereqs
Your server is running and you can edit server.cfg. That is enough.
In 3 bullets

You will apply these five protections to your server:

  • Lock down resource startup so only the resources you explicitly allow can run.
  • Block the most dangerous FiveM events that attackers use to crash servers or steal data.
  • Set server.cfg ConVars that prevent file read exploits, resource restarts by clients, and RCON abuse.

Outcome

You will apply five security protections to your server and verify each one is active by checking the console on restart.

Why it matters

A new FiveM server gets scanned by automated attack tools within hours. Without these five protections, attackers can steal your scripts, crash your server, spam your players, or gain admin access. These are not theoretical — they happen every day to servers that skip this step.

Vocabulary

RCE
Remote Code Execution — an attacker runs code on your server machine.
Example: an insecure resource lets an attacker call os.execute() and run system commands.
Event spam
Sending thousands of network events per second to overwhelm the server.
Example: a malicious client fires TriggerServerEvent in a loop to crash the server.
Resource theft
An attacker downloads your server's scripts without permission.
Example: triggering a misconfigured event that returns file contents to the client.

Minimal example

Expected output: Server console prints five protection confirmation lines on resource start.

Checklist

  1. Add these lines to server.cfg: sv_resourceNameWhitelist "true", sv_scriptHookAllowed "false", sv_sendFileHashes "false".
  2. Create resources/[system]/qu_security/ with an fxmanifest.lua that has server_script 'server.lua'.
  3. Paste the server.lua code above into your resource.
  4. Add ensure qu_security as the first ensure line in server.cfg.
  5. Restart your server and confirm all five protection confirmations print in the console.
  6. Read the REFACTOR section below and implement the event rate limiter.

Exercise

Add one more blocked event to the BLOCKED_EVENTS table. Pick any event name your server uses that should never be triggered by a client. Add it, restart, and confirm the console prints an updated count of blocked events.

Expected output: Console prints ✅ Protection 1: Blocked N+1 dangerous events where N+1 includes your new event.

Common failures

  • Resource qu_security failed to start means the fxmanifest.lua is missing or has a syntax error. Check that every comma and quote is correct.
  • Legitimate events are blocked means you added an event name that your server actually uses. Remove it from BLOCKED_EVENTS.
  • sv_resourceNameWhitelist blocks all resources means you did not add your resources to the whitelist. In server.cfg add add_ace resource.qu_security command.start allow for each resource.
  • Server still crashes from event spam means the rate limiter is not aggressive enough. Lower the time window or add more event names to BLOCKED_EVENTS.

Refactor

Add a global event rate limiter. Track every event name in a table with timestamps. If any client fires more than 10 events in 5 seconds, drop all their events for the next 30 seconds. This stops spam from any event name, not just the ones in your blocklist.

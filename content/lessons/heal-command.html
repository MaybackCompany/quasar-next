Cheat sheet

/heal safety rules

  • Client asks. The client never decides the reward.
  • Server checks. Use source, cooldowns, and framework data.
  • Client acts. Only heal after the server sends the approved event.

The big idea, in one minute

A FiveM script lives in two places at the same time. The client is the game on the player's computer. The server is the machine that runs the world for everybody. The client can lie. The server cannot, because you control it.

Think of a hotel front desk.

The guest (the client) walks up and asks for a room key. The clerk (the server) checks the booking, then unlocks one specific door. The guest never holds the master key. If a stranger walks in and shouts "give me the penthouse", the clerk just says no.

Your /heal command works the same way. The player asks. The server checks. The server tells the client to heal. The client never decides on its own.

Bad way and good way, side by side

Most beginners write the bad version first. That is normal. The good version is only a few more lines, and it stops cheaters dead.

Don't do this

The client heals itself. A cheater calls this every frame and never dies.

The amount comes from outside. There is no cooldown. Anyone with a cheat menu can fire qu:heal:any with 999999 and become invincible.

Do this

The client only asks. The server checks source and the cooldown, then sends the green light.

The heal amount is never sent over the network. The server picks the player using source. Cheaters cannot skip the check, because the check lives on a machine they cannot reach.

Step 1

Make the resource folder

You are creating one folder and one file. Nothing else, yet.

  1. Open your server's resources/ folder.
  2. Inside it, create a new folder named qu_heal.
  3. Inside qu_heal, create an empty file named fxmanifest.lua.
  4. Paste the manifest from the "Minimal example" above.
  5. Save the file. You are done with Step 1.
Step 2

Add the client file (the asker)

The client only asks. It never decides.

  1. In the qu_heal folder, create a file named client.lua.
  2. Paste this exactly: 
    RegisterCommand('heal', function()
    TriggerServerEvent('qu:server:heal')
end, false)
  3. RegisterCommand('heal', ...) turns the chat /heal into a function call.
  4. TriggerServerEvent('qu:server:heal') sends a message named qu:server:heal to the server. No data, just the request.
  5. Save the file. The client side is done. It does nothing visible yet, that is on purpose.
Step 3

Add the server file (the decider)

The server is where rules live. It checks who asked and how recently.

  1. In the qu_heal folder, create a file named server.lua.
  2. Paste the server code from the "Minimal example" above.
  3. local src = source grabs the player ID of whoever sent the event. The keyword source is set by FiveM. Never trust any other ID.
  4. The cooldown table lastUsed remembers when each player last healed. The amount is a constant called COOLDOWN_SECONDS, kept on the server. The client cannot change it.
  5. The line TriggerClientEvent('qu:client:doHeal', src) sends the green light to one specific player, the one named in src. Not to everyone.
  6. Save the file. Step 3 is done.
Step 4

Add the heal handler back on the client

Only after the server says yes does the client touch the player's health.

  1. Open client.lua again.
  2. Below the existing RegisterCommand block, add the RegisterNetEvent('qu:client:doHeal', ...) handler from the "Minimal example" above.
  3. PlayerPedId() is a FiveM native that returns the character the player controls right now.
  4. GetEntityMaxHealth(ped) reads the maximum value, usually 200. Using this means another script that bumps max health will not break you.
  5. SetEntityHealth(ped, maxHp) fills the bar. This native exists only on the client, because the character lives on the player's machine.
  6. Save the file. The wire is now complete.
Step 5

Turn it on and test

Three lines in your server.cfg, then watch your health bar.

  1. Open server.cfg and add a new line: ensure qu_heal.
  2. Start the server, or in a running server type refresh then ensure qu_heal in the live console.
  3. Join the server. Punch a wall, or shoot the ground at your feet, until your health bar drops.
  4. Press T to open chat, type /heal, press Enter.
  5. Your health bar fills back up. Type /heal again. You see the cooldown message in chat.
Recap

Three tests, in order

Run each test, look at what happens, and confirm the expected line in the console. If something does not match, fix it before moving on.

  1. Test 1, the happy path.
    • Action: take some damage, then type /heal in chat.
    • Expected: health bar fills to full.
    • How to verify: server console prints [qu_heal] healed source=1. F8 console prints [qu_heal] healed to 200.
  2. Test 2, the cooldown.
    • Action: type /heal, then immediately type /heal again.
    • Expected: chat shows [heal] Wait a bit before healing again.
    • How to verify: server console does not print a second healed line. The bar does not change on the second try.
  3. Test 3, the broken event name.
    • Action: in server.lua, change 'qu:server:heal' to 'qu:server:heel', save, restart the resource, type /heal.
    • Expected: nothing happens. No errors, no heal.
    • How to verify: server console stays silent on /heal. Put the name back to fix it.
Why is the cooldown on the server and not the client?

Because a cheater can edit any code that runs on their own computer. The server runs on your machine, not theirs. Rules live on your machine.

The rules you just followed

  • The client never decides who gets healed. It only asks.
  • The server uses source to know which player sent the request. Nothing else.
  • The heal amount is a server constant. It is never sent over the network.
  • The cooldown is stored on the server, in a table keyed by source.
  • The server tells one specific player to heal, never -1 (which would mean everyone).

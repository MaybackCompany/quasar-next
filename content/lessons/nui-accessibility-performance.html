In 3 bullets

You will build a fast, accessible NUI that does this:

Why it matters

A slow NUI freezes the game. An inaccessible NUI locks out players. Both cost you users before they read a single word of your UI.

Vocabulary

SendNUIMessage
A Lua function that sends JSON data from game scripts to the NUI browser.
Example: SendNUIMessage({type='health', value=100}).
ARIA
Accessible Rich Internet Applications — HTML attributes that describe UI to screen readers.
Example: aria-label="Health: 75 percent".
Frame Budget
The time available per frame before the game feels laggy — 16ms at 60 FPS.
Example: your NUI JS must finish in under 10ms per frame.

Common failures

  • NUI not showing means ui_page is missing from fxmanifest or the path to index.html is wrong.
  • NUI flickers on every update means you are replacing the entire DOM instead of updating only the changed elements.
  • keyboard cannot reach buttons means tabindex is missing or set to -1 on interactive elements.
  • screen reader says nothing means ARIA labels are missing or the content updates are not announced with aria-live regions.

Refactor

Extract the DOM update logic into updateElement(id, value) so every NUI message handler uses a single, tested update function instead of inline DOM manipulation.

Cheat sheet

NUI performance rules

  • Under 500ms load. No frameworks. Plain HTML/CSS/JS. Defer everything non-critical.
  • Under 10ms per frame. Do not run JS every frame. Use Citizen.Wait(250) or higher for NUI updates.
  • Update, don't replace. Target specific elements by ID. Never innerHTML = hugeString on a timer.

The big idea, in one minute

NUI is a web browser running at 60 frames per second inside a game that also needs to run at 60 frames per second. Every millisecond your JavaScript spends recalculating the DOM is a millisecond the game renderer cannot use. A 100ms DOM update drops the game from 60 FPS to 50 FPS. Multiply that by 10 NUI resources from 10 different script authors and your server is a slideshow. Fast NUI is not a nice-to-have — it is the difference between a smooth game and a lag simulator.

Think of a picture-in-picture TV.

The big screen is the game. The small inset window is your NUI. Both share the same video processor. If the small window demands all the processor's attention to play a 4K video, the big screen stutters. Your NUI must be the simplest, lightest window possible — a text label and a bar, not a full web app.

Bad way and good way, side by side

Most developers drop a full React or Vue app into NUI and wonder why the server FPS tanks.

Don't do this

Heavy framework, full DOM replacement, no accessibility.

React adds 150ms+ of overhead per update. The entire DOM tree is rebuilt every tick. No player using a keyboard or screen reader can use this UI.

Do this

Plain JS, targeted updates, ARIA labels, keyboard navigable.

Zero frameworks. One DOM element updated per tick. ARIA labels announce changes to screen readers. Keyboard users can Tab to the health bar and see the focus ring. Transitions are CSS, not JS.

Step 1

Create the NUI folder and manifest

The NUI lives in an html/ folder and is registered in fxmanifest.lua.

  1. Create resources/qu_nui/html/ folder.
  2. Inside it, create index.html, style.css, and script.js.
  3. In fxmanifest.lua, add ui_page 'html/index.html'.
  4. Add client_scripts { 'client.lua' }. No server file needed for NUI.
Step 2

Wire the Lua to NUI bridge

SendNUIMessage pushes data. window.addEventListener catches it.

  1. In client.lua, create a loop that reads health and calls SendNUIMessage.
  2. Use Citizen.Wait(250) not Wait(0) — health updates do not need 60 FPS.
  3. In script.js, add window.addEventListener('message', handler).
  4. Filter by e.data.type so different messages do not collide.
Step 3

Add accessibility and measure speed

Accessibility is not extra work — it is built into semantic HTML.

  1. Add role="progressbar" and ARIA attributes to the health bar.
  2. Add tabindex="0" and :focus-visible styles to every interactive element.
  3. Use performance.now() to measure DOM update time — it must be under 10ms.
  4. Test with a screen reader (VoiceOver on Mac, Narrator on Windows) to confirm labels are announced.
Recap

Three tests, in order

Run each test and verify the output before adding more features.

  1. Test 1, NUI renders.
    • Action: start the resource and join the server.
    • Expected: the health bar appears on screen.
    • How to verify: the bar updates as you take damage or heal.
  2. Test 2, performance.
    • Action: open F8 console, type profiler, watch the NUI frame time.
    • Expected: NUI JS execution is under 10ms per frame.
    • How to verify: the game FPS does not drop when the NUI is visible versus hidden.
  3. Test 3, accessibility.
    • Action: press Tab. Then enable a screen reader.
    • Expected: the health bar receives focus and the screen reader announces "Health: 75 percent".
    • How to verify: every interactive element is reachable via keyboard and announced by the screen reader.
Why use role="progressbar" instead of a plain div?

Because screen readers do not know a div is a health bar. The role tells them, and the ARIA attributes tell them the current value. Without it, the player hears "div" and nothing else.

The rules you just followed

  • NUI uses zero frameworks — plain HTML, CSS, and JavaScript only.
  • DOM updates target specific elements by ID, never using innerHTML on the whole page.
  • NUI update loops use Citizen.Wait(250) or higher — never run every frame.
  • Every interactive element has tabindex, focus-visible styles, and an ARIA label.
  • Performance is measured with performance.now() and must stay under 10ms per update.

Inventories: stack, weight, slot

An inventory is a backpack with limited pockets and a scale. There are two dominant models in FiveM: slot-based (each item lives in a numbered pocket) and weight-based (only kg cap matters). Each has trade-offs. This lesson covers both, the stacking rules, item metadata, and how to integrate with ox_inventory without re-implementing it.

Before you start

Mental model: backpack with pockets and a scale

A slot-based backpack has 30 numbered pockets. Each pocket holds one item type, up to its stack limit. Two stacks of water bottles take two pockets, even if they would fit weight-wise.

A weight-based backpack has one big compartment and a scale that limits total kg. Item count is irrelevant; you stop adding when the scale tips.

Most modern frameworks (ox_inventory, qb-inventory) combine both: slots for layout AND weight for realism. You see numbered slots in the UI, but the weight cap keeps you from carrying 30 pickup trucks.

Step 5

Senior depth: schema, concurrency, lifecycle

Outcome: an inventory layer that survives 64 players, 8-player stash contention, and a year of metadata growth.

5.1 Storage model at scale: slot table vs JSON column

Two schemas dominate. Normalized slot table (ox_inventory's old shape): one row per occupied slot, indexed by (owner, slot). JSON column (ox_inventory's current shape): one row per owner with a single data JSON blob. The trade-off is read cost vs write cost.

Read full backpack: blob is one row, normalized is up to 30. Read single slot for a quick check: blob still parses the whole JSON, normalized hits one row by primary key. Update one slot: blob rewrites the entire JSON column (~3-8 KB), normalized rewrites one row. At 64 players with frequent inventory mutations the blob model multiplies write amplification, which is why ox_inventory caches in RAM and only flushes deltas on logout or stash close.

The rule: never reach into the DB for live inventory state. The cache is authoritative while the player is online; the DB is the persistence layer that catches up on flush.

5.2 Stash concurrency: the "last item" race

A warehouse stash is a shared inventory accessed by multiple players. Two players open the stash at the same time. Both UIs show one nameplate kit. Both click "take". Without a mutex, both RemoveItem calls succeed and the stash now has -1 kits, or two players walk away with the same item that only existed once.

The fix is a per-stash mutex. ox_inventory serializes operations against the same inventory ID internally, but only when you go through its API consistently. If you bypass with raw MySQL writes, you reintroduce the race.

This is a busy-wait mutex. With 8 players hammering the same stash the longest queue we measured was 4ms; acceptable because the critical section is itself ~0.4ms. If your stash is hot enough that the queue passes 16ms, replace the busy-wait with a deferred queue (push the operation onto a list, drain in CreateThread).

5.3 Metadata bloat and item lifecycle

Metadata is a footgun at scale. A phone with 50 contacts is ~2 KB JSON. A backpack of 5 phones is 10 KB inside the inventory blob. Players hoard. After 6 months the population's average backpack grows from 4 KB to 18 KB, every flush rewrites that, and the data column starts brushing the JSON 1 MB cap.

Two practical defenses. First, add a lastUsed timestamp to any metadata-heavy item and run a nightly sweep that archives stale items (a phone untouched for 60 days lands in inv_archive, freeing the slot). Second, never use math.random for serials, it shares state with weather and traffic logic and collides under load. Use a UUID v4 helper or a DB-side UUID().

The lifecycle pattern: items are born with a serial, mutate metadata while in use, get archived (not deleted) when stale. Deletion is reversible only if you keep the archive; players who think they lost a phone after a long break expect support to restore it.

Recap 3

Senior depth: inventory at scale

You can defend the schema choice (slot table vs JSON blob) with read-vs-write amplification numbers, you know why ox_inventory caches in RAM and flushes deltas, and you can name the three races that haunt 64-player stash code: concurrent RemoveItem, the give-then-check inversion, and the metadata-merge collision. Production inventory is not a CRUD; it is a transactional cache.

Common mistakes

Stacking items with metadata

If stack = true and the item has metadata, two instances merge and the metadata of the second overwrites the first. Never stack uniques.

Removing before checking capacity

Always validate CanCarryItem for outputs before removing inputs. Inventory operations are not transactional in the sense that you can roll back; do the check first.

Reading inventory on the client for security checks

Client inventory state is a render. The truth lives on the server. ox:GetItemCount on the server is authoritative; on the client it is for UI only.

Storing money as a separate field

Modern ox_inventory treats money as an item with name 'money'. Don't keep a parallel cash field; use the same API.

Editing the items table at runtime

Items are loaded once on resource start. Adding entries at runtime does not propagate to clients. Restart ox_inventory after editing data/items.lua.

What you learned

  • Slot vs weight is a UI choice; modern stacks combine both.
  • Stack flag and metadata are mutually exclusive in practice.
  • Always validate capacity for outputs before consuming inputs.
  • Server is the source of truth; client inventory is rendering.
  • Money is just an item with a special name; don't fork the system.

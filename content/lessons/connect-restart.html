Before you start

Each box below is a thing the next steps assume. Don't skip.

Two consoles, two jobs: in one minute

You are about to deal with two consoles. Don't confuse them.

  • Server console (txAdmin's Live Console, or the OS window where FXServer.exe runs). Prints what the server is doing. You read it to debug script crashes.
  • Client console (in-game, press F8). Prints what your client is seeing. You read it to debug your own connection issues. You also use it to run connect.

Mental model: server console = the chef's pass. Client console = your dinner table receipt. Different views of the same kitchen.

FiveM note: the tx shortcut

Once you're connected to your own server with admin rights, the in-game console takes one magic command: tx. It opens the txAdmin admin menu (same thing F1 does, but via console). Useful when you're testing keybinds and F1 is busy.

Step 1

Connect to your own server

Outcome: you're in your server's loading screen, then in-game.

  1. Open the FiveM client.
  2. Top tab → Play.
  3. Right side → Direct Connect.
  4. Type localhost in the address bar → click Connect.
  5. Wait. The first connection downloads any custom resources you've added. The loading screen shows progress.
  6. When you spawn in-game, you'll see the default airport intro from the CFX recipe.
Why localhost?

localhost = "this same computer". You're running both the server and the client on one PC. Real players connect using your public IP or a CFX join code. For now, localhost is the friction-free dev loop.

Step 2

Open F8 and run tx

Outcome: the txAdmin in-game menu opens via the console.

  1. While in-game, press F8. A black console drops down from the top.
  2. Type tx → Enter. The txAdmin menu opens (same as F1).
  3. Press F8 again to close the console.

F8 is also where you'll run client-side commands later: showposition, noclip, your own custom RegisterCommand callbacks. Get used to opening it.

F8 is text, F1 is a menu

F8 = type-and-enter console. F1 = mouse-driven menu. They both reach the same admin tools, but F8 is faster for repeat actions ("teleport here", "spawn this") and lets you copy/paste IDs.

Step 3

ensure vs restart vs refresh

Outcome: you'll never confuse these three again.

Three different commands. Three different jobs. Memorize this:

CommandWhat it doesWhen to use
ensure my-resourceStart if stopped. Restart if running.In server.cfg, always. Safe re-bounce.
restart my-resourceStop + start the running resource.After editing a Lua / JS file. Live Console or in-game F8.
refreshRe-scans the resources/ folder for new sub-folders. Doesn't start anything.After dropping a brand-new resource folder in. Then ensure new-resource.

Try them now:

  1. In txAdmin Live Console (or F8 in-game), type restart chat. Watch the console echo.
  2. Type refresh. Watch FXServer rescan the folder.
  3. Type ensure scoreboard. If running it stops + starts; if stopped it just starts.
Don't stop a resource you're about to start

Beginners often type stop chat then start chat. That's two commands and a window where chat is dead. restart chat does both atomically. Faster, fewer mistakes.

Recap 1

You're inside your own server

You ran connect localhost from FiveM, you spawned in, you opened F8 and ran tx, and you triggered a restart chat from somewhere (Live Console or F8). If yes, you've closed the loop: you can edit, restart, observe.

"Connection failed: timed out"

Your endpoint_add_udp "0.0.0.0:30120" line is missing or the txAdmin server is on a different port. Check txAdmin's Live Console for "Started UDP listener on port 30120". If it's a different port, use connect localhost:PORT.

F8 doesn't open

Your keyboard layout might map F8 elsewhere. In FiveM go to Settings → Key Bindings → search "console". Re-bind to ~ if you prefer.

I can connect but the screen is black with no UI

The recipe's basic-gamemode resource crashed. Check the Live Console for red lines. Often a missing dependency in fxmanifest.lua. Run refresh then ensure basic-gamemode.

Step 4

Read the server console: one line at a time

Outcome: you can look at any console line and label it info / warn / error.

Open txAdmin → Live Console. You'll see lines like these. Read them top to bottom:

[    info] Console: Resource scoreboard started.
[    info] Player [Kishi] connected from 127.0.0.1.
[    warn] CFX: dependency 'es_extended' not found, scoreboard may misbehave
[   error] SCRIPT ERROR: @scoreboard/cl_score.lua:42: attempt to index nil value (global 'PlayerData')
stack traceback:
    @scoreboard/cl_score.lua:42: in function <@scoreboard/cl_score.lua:40>

How to read these:

  • info, FYI. Resource starts, players join, no problem.
  • warn, "you should know". Server still runs, but something's off (missing dependency, deprecated API).
  • error, something broke. Almost always tells you the file + line number. Open it. Read the line. Fix it.
  • stack traceback, the call chain that led to the error. The top line is where it actually exploded; the lines below are who called it.
Triage rule

Red > yellow > white. Always read all red lines first. A single error often cascades into dozens of warnings. Fix the error, the warnings vanish.

Step 5

Where logs actually live on disk

Outcome: when txAdmin can't show you something, you know where to look on disk.

txAdmin's Live Console only shows recent output (about the last 5,000 lines). For older logs, FXServer writes to disk:

FileWhat's in it
server-data/.txAdmin/data/default/logs/fxserver.logFull FXServer console rolling log. Searchable.
server-data/.txAdmin/data/default/logs/admin.logAdmin actions: who ran kick, ban, restarts.
server-data/CitizenFX_log_*.logGenerated on crash. Send this to support if FXServer dies.
The log path can move

If you started txAdmin with a non-default profile or moved server-data, log paths shift. The constant: they always sit under server-data/.txAdmin/data/<profile>/logs/. Search the folder with dir /s admin.log on Windows or find . -name admin.log on Linux/Mac.

Step 6

The "edit → restart → observe" loop

Outcome: you internalize the dev loop you'll repeat thousands of times.

Every script change you make from now on follows this loop:

  1. Edit the Lua file in VS Code. Save (Ctrl+S).
  2. Restart the resource. Two ways:
    • txAdmin → Resources tab → Restart on the row.
    • In-game F8 → restart my-resource.
  3. Observe. Check the Live Console for new errors. Trigger the feature you changed (run a command, drive into a zone, fire an event). Did it work?
  4. If broken → read the red line. Edit. Repeat.

Every Lua course on the planet drills this loop. Build it as muscle memory now, and the rest of the curriculum is just "what to type in step 1".

Pro habit: keep two windows side by side

Left half of your monitor: VS Code with the Lua file. Right half: txAdmin Live Console (or FiveM with F8 open). You edit on the left, you watch on the right, you save and the right side reacts. No alt-tab. Massive speed-up.

Recap 2

You read the console like an adult

Open the Live Console. Find one info line, one (likely no) warn line, and zero error lines (if you have errors, fix them now, that's the whole point of this lesson). Open Resources, restart any resource, watch its lifecycle in the console: Stopping resource X → Started resource X. If you can label every line type, you've leveled up from "scared of the black screen" to "developer".

The Live Console is empty

Possibly your txAdmin lost its connection to FXServer. Click Restart Server top-right. If still empty, check the OS console where you started FXServer.exe, it might have crashed silently.

I see only red errors and don't know which to fix first

Scroll up. The first red line is usually the root cause; everything after is fallout. Fix the topmost error, restart, see what's left.

Common mistakes & what they look like

I edited the file but the server still runs the old code

You forgot to restart the resource. Saving the file doesn't reload it, FXServer caches the loaded version until you tell it to reload.

Console says "could not find resource"

Folder name typo, or you forgot refresh after dropping a new folder in. refresh first, ensure second. Names are case-sensitive on Linux servers.

Server runs fine, but I can't connect: "rejected, banlist server-side"

You banned yourself by accident in the Players panel. Open server-data/.txAdmin/data/default/playersDB.json, find your identifier, set the ban entry to "revoked". Or simpler: txAdmin → Players → Bans → revoke.

"FXServer.exe has stopped working": whole window vanished

Look for a CitizenFX_log_*.log in server-data/. The bottom of that file shows the crash. Often a Lua resource ran an infinite loop or hit a memory limit. Disable suspect resources in server.cfg (comment out the ensure line), restart, isolate the bad one.

What you learned

  • Connect to your own server with Direct Connect → localhost
  • F8 = client console (text). F1 = txAdmin menu (mouse). tx = same as F1 from the F8 console.
  • ensure for cfg, restart for hot reload, refresh for new folders
  • Server console lines are info / warn / error, fix red first
  • Persistent logs sit at server-data/.txAdmin/data/default/logs/
  • The loop you'll repeat forever: edit → restart → observe

Five-minute exercise

Run the full dev loop once, end to end:

  1. Direct-connect to localhost in FiveM.
  2. Open F8, type tx, confirm the menu opens, close it.
  3. Alt-tab to txAdmin. In Resources, click Restart on chat.
  4. Alt-tab back. In-game chat should auto-rejoin (you might see a brief "chat reconnecting" notice).
  5. Open the Live Console. Find the Started resource chat info line.

Expected outcome

You connected once, opened the admin menu via F8, restarted one resource without dropping the session, and read the corresponding console line. That is the daily life of a FiveM developer in five minutes.

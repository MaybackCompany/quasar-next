Shipping: opening night checklist

A restaurant does not open for guests without prepped stations, a backup line, and a rollback plan if a dish fails. Same here: convars pinned, ACE locked, DB backed up, monitoring on, and a known-good build to revert to.

The pre-flight, the hot-standby, the rollback path.

~30 min

Before you start

Mental model. Treat go-live as a readiness checklist, not a feature. Each item flips green or it does not ship.

The five pillars

  • Pin, artifact version, resource versions, DB schema migration applied.
  • Lock, ACE permissions, RCON password, server-only convars.
  • Save, backups before deploy and on every scheduled restart.
  • Watch, resmon panel, error webhook, player count alert.
  • Roll back, one command to switch to the previous build + previous DB snapshot.

Pro tip. Tag every release in git as v1.4.2 and write the artifact build number in VERSION at repo root. Restarts log it, saves hours during incidents.

Step 1 · Pin the build

# server.cfg
sv_enforceGameBuild 3258
set sv_licenseKey "cfxk_..."
set onesync on
set onesync_population true
set sv_maxClients 64
-- in any resource that needs to know the release tag
print("[qu_core] release " .. (LoadResourceFile(GetCurrentResourceName(), "VERSION") or "dev"))

Pinning sv_enforceGameBuild prevents player crashes when Rockstar pushes a new build. Loading VERSION from your release artifact tags every console line.

Step 2 · Lock permissions

# server.cfg
add_ace group.admin command allow
add_ace group.admin command.quit deny
add_ace group.admin qu_jobs.* allow
add_ace group.police qu_jobs.police allow
add_ace group.ems    qu_jobs.ems    allow
add_principal identifier.discord:1234567 group.admin

rcon_password "rotate-me-on-every-deploy"
sv_endpoints "0.0.0.0:30120"

Deny command.quit on admins so a misclicked command cannot kill the server. Rotate rcon_password every deploy and never commit it, load via env or exec server-secrets.cfg.

Step 3 · Backup and restart hooks

# scripts/backup.sh
TS=$(date +%F-%H%M)
mysqldump --single-transaction --quick --routines fivem > /backups/fivem-$TS.sql
gzip /backups/fivem-$TS.sql
find /backups -name "fivem-*.sql.gz" -mtime +14 -delete
-- resources/qu_core/server/restart.lua
AddEventHandler("txAdmin:events:scheduledRestart", function(data)
  if data and data.secondsRemaining == 60 then
    TriggerEvent("qu_core:flushAll")        -- flush caches (P9)
    TriggerEvent("qu_core:announce", "Server restart in 60s.")
  end
end)

AddEventHandler("onResourceStop", function(name)
  if name ~= GetCurrentResourceName() then return end
  TriggerEvent("qu_core:flushAll")
end)

Two save paths: scheduled restart (txAdmin) and ad-hoc resource stop. Cron runs the dump every 6h and prunes anything older than 14 days.

Step 4 · Watch and alert

-- resources/qu_watch/server/main.lua
local WEBHOOK = GetConvar("qu_alerts_webhook", "")

local function alert(text)
  if WEBHOOK == "" then return end
  PerformHttpRequest(WEBHOOK, function() end, "POST",
    json.encode({ content = ("[%s] %s"):format(os.date("%H:%M:%S"), text) }),
    { ["Content-Type"] = "application/json" })
end

CreateThread(function()
  while true do
    Wait(60000)
    local n = GetNumPlayerIndices()
    if n == 0 then alert("Server empty") end
  end
end)

AddEventHandler("onResourceStop", function(name)
  alert(("Resource stopped: %s"):format(name))
end)

Lightweight watcher posts to Discord on empty server or any resource stop. Pair with txAdmin’s built-in performance alerts for resmon spikes.

Recap · Dry run on staging

  1. Deploy the candidate build to staging. Run a 30-minute soak with bots or a small group.
  2. Trigger a scheduled restart. Confirm flushAll ran and DB rows are written.
  3. Stop a non-critical resource. Confirm Discord alert fires.

Recap · Rollback rehearsal

Tag the staging DB with a snapshot. Switch the resource master symlink to the previous tag and restart. Confirm everything boots and the watcher is silent. This is the muscle memory you want before you ever need it.

Mandatory backup-restore drill proof

Step 5 · Production at scale

Three real-world concerns once you stop being one box on a friend’s home connection: blue/green deploys with atomic swaps, backup verification with restore drills, and an incident runbook that actually gets followed at 02:00.

5.1 · Blue/green deploy and atomic symlink swap

One current symlink, two release directories, and a swap that takes microseconds. The running server keeps its open file descriptors until the next restart, so the swap itself is non-disruptive, the cutover is the restart.

# release layout on the host
/srv/fivem/
  releases/
    1.4.2/        # previous (kept hot for instant rollback)
    1.4.3/        # candidate just rsynced
  current -> releases/1.4.3
  shared/
    server-secrets.cfg
    cache/

# scripts/release.sh
set -euo pipefail
TAG="$1"
RELEASE_DIR="/srv/fivem/releases/$TAG"

# 1. ship the artifact tree without touching the live one
rsync -a --delete \
  ./build/ "release-host:$RELEASE_DIR/" \
  --exclude="cache/" --exclude="logs/"

# 2. snapshot DB before the cutover (see 5.2)
ssh release-host "/srv/fivem/scripts/backup.sh pre-$TAG"

# 3. atomic symlink flip (POSIX rename(2) is atomic on the same filesystem)
ssh release-host "ln -sfn '$RELEASE_DIR' /srv/fivem/current.new \
  && mv -Tf /srv/fivem/current.new /srv/fivem/current"

# 4. tell txAdmin to restart the server (graceful, with the 60s flush hook)
ssh release-host "/srv/fivem/scripts/txadmin-restart.sh 'release $TAG'"
# scripts/rollback.sh -- one command, <5 seconds to swap
set -euo pipefail
PREV=$(cat /srv/fivem/.last-good)
ln -sfn "/srv/fivem/releases/$PREV" /srv/fivem/current.new
mv -Tf /srv/fivem/current.new /srv/fivem/current
/srv/fivem/scripts/txadmin-restart.sh "rollback to $PREV"
echo "[rollback] now on $PREV"

The trick is mv -Tf on a symlink: the kernel does an atomic rename(2) so there is never a moment where current does not exist. After a successful boot, write the tag to .last-good so rollback.sh always knows the target. Keep at least the previous two releases on disk, tiny cost, instant rollback.

5.2 · Backup verification and restore drills

A backup you have never restored is a hope, not a backup. The dump must be off-host, compressed, and a weekly automated restore must read it back into staging and assert the row counts.

# scripts/backup.sh -- runs every 6h via cron + before every release
set -euo pipefail
TAG="${1:-scheduled}"
TS=$(date +%F-%H%M)
OUT="/backups/fivem-$TS-$TAG.sql.zst"

mysqldump \
  --single-transaction \
  --quick \
  --routines \
  --triggers \
  --hex-blob \
  --default-character-set=utf8mb4 \
  fivem \
  | zstd -3 -T0 -o "$OUT"

# off-host replication (do NOT trust the host with the live DB)
rsync -az --remove-source-files /backups/fivem-*.zst backup-host:/archive/fivem/

# retention: keep 14 days locally, 90 days off-host
find /backups -name "fivem-*.zst" -mtime +14 -delete
ssh backup-host "find /archive/fivem -name 'fivem-*.zst' -mtime +90 -delete"
# scripts/restore-drill.sh -- runs Sunday 03:00, page on failure
set -euo pipefail
LATEST=$(ssh backup-host "ls -t /archive/fivem/fivem-*.zst | head -1")
ssh backup-host "zstdcat $LATEST" | mysql -h staging-db fivem_restore_test

# sanity: row counts must be within 0.5% of production
PROD=$(mysql -N -h prod-db -e "SELECT COUNT(*) FROM users" fivem)
TEST=$(mysql -N -h staging-db -e "SELECT COUNT(*) FROM users" fivem_restore_test)
DELTA=$(python3 -c "print(abs($PROD - $TEST) / $PROD * 100)")
awk -v d="$DELTA" 'BEGIN{ exit (d > 0.5) }' \
  || { curl -X POST "$ALERT_WEBHOOK" -d "{\"content\":\"restore drill FAILED, delta $DELTA%\"}"; exit 1; }
echo "[drill] OK, delta $DELTA%"

--single-transaction takes a repeatable-read snapshot, so InnoDB tables dump without locking writers. zstd -3 is roughly 3× faster than gzip -6 at a similar ratio. The drill is the part that matters, running it weekly catches schema drift, charset corruption, and missing routines long before a real incident.

5.3 · Incident runbook and observability

02:00 incidents do not get solved by clever ideas. They get solved by a checklist that someone half-asleep can follow.

-- resources/qu_watch/server/runbook.lua
-- prints the entire incident header in the first 20 lines of every boot
local function bootHeader()
  local v = LoadResourceFile(GetCurrentResourceName(), "VERSION") or "dev"
  print(("[qu] release        : %s"):format(v))
  print(("[qu] artifact       : %s"):format(GetConvar("sv_enforceGameBuild", "?")))
  print(("[qu] dashboard      : %s"):format(GetConvar("qu_dashboard_url", "?")))
  print(("[qu] runbook        : %s"):format(GetConvar("qu_runbook_url", "?")))
  print(("[qu] oncall webhook : %s"):format(GetConvar("qu_oncall_webhook", "?") ~= "" and "set" or "MISSING"))
end
AddEventHandler("onResourceStart", function(name)
  if name == GetCurrentResourceName() then bootHeader() end
end)
-- resources/qu_watch/server/oncall.lua
-- severity-aware paging: low -> channel, high -> @oncall role mention
local OK, WARN, CRIT = 1, 2, 3
local CHANNEL  = GetConvar("qu_alerts_webhook", "")
local ONCALL   = GetConvar("qu_oncall_webhook", "")
local ROLE_ID  = GetConvar("qu_oncall_role_id", "")

local function page(severity, summary, details)
  local target = severity >= CRIT and ONCALL or CHANNEL
  if target == "" then return end
  local mention = severity >= CRIT and ("<@&" .. ROLE_ID .. "> ") or ""
  local body = json.encode({
    content = mention .. ("**[%s]** %s"):format(({"OK","WARN","CRIT"})[severity], summary),
    embeds  = { { description = details, color = severity >= CRIT and 15158332 or 15844367 } }
  })
  PerformHttpRequest(target, function() end, "POST", body,
    { ["Content-Type"] = "application/json" })
end

-- example: payroll job stalls
AddEventHandler("qu_jobs:payrollFailed", function(err, n)
  page(CRIT, "Payroll failed", ("error: %s\non-duty at fail: %d"):format(err, n))
end)
exports("page", page)

Three rules keep this honest: (1) boot header in the first 20 lines, the responder never has to ask “what version is running?”. (2) severity gates the channel, CRIT pages the on-call role, WARN posts to the team channel, OK never pages. (3) runbook URL is a convar, you can update the doc without redeploying. Pair this with a one-paragraph incident template in the runbook (timeline, blast radius, mitigation, rollback decision) and the responder is reading, not improvising.

Recap 3

Senior depth: production readiness

Shipping is not a release; it is a rollback rehearsal. You can swap the active build with one atomic mv -Tf, you have a backup that has been restored into staging within the last seven days, and your boot logs print release / artifact / dashboard / runbook URLs in the first 20 lines. When the page fires at 03:00, the on-call has links, not riddles.

Common mistakes

Deploying without a backup window.

Always snapshot DB first. A failed migration with no snapshot is the worst night of your life.

RCON password committed to git.

Use exec server-secrets.cfg with that file gitignored, or env vars resolved at boot.

No sv_enforceGameBuild.

Players crash on Rockstar updates with no warning. Pin and bump deliberately.

Single point of failure on saves.

Have both scheduled flushes and shutdown flushes. Either alone leaves a tail of lost work.

No version in the console.

During an incident you need to see the running version in the first 10 lines of logs. Print it.

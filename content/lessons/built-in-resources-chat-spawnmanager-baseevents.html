Built-in resources live inside the server binaries, not in your resources/ folder.

How to use it
You do not ensure them. FiveM starts them before any of your scripts run.
Example: spawnmanager is always running; you just call its exports.

Outcome

You will write a small resource that listens for onPlayerDied (from baseevents), chatMessage (from chat), and playerSpawned (from spawnmanager) — and prints each one to the console as they happen.

Why it matters

Every day, a beginner posts "my death script stopped working" because they accidentally stopped or replaced baseevents. Knowing what the built-ins do and which events they own means you debug server-wide problems in minutes instead of hours.

Vocabulary

Built-in resource
A resource embedded in the FiveM server executable, started automatically before any user resources.
Example: chat, spawnmanager, baseevents, sessionmanager, hardcap, rpc, webadmin, monitor.
chat:addMessage
The export provided by the chat resource to show text in the chat box.
Example: exports['chat']:addMessage({ args = { 'Hello', 'world' } }) — no script needed.
spawnmanager
The resource that decides where and how players appear when they first join.
Example: by default it spawns you at the airport. You override it with setAutoSpawnCallback.

Exercise

Common failures

  • chatMessage handler never fires — you registered it on the client but it fires on the server. Move the handler to server.lua.
  • playerSpawned never fires — you registered it on the server but it fires on the client. Move to client.lua.
  • Death events fire twice — both onPlayerDied and onPlayerKilled fire on death. They are different events with different parameters. Pick the one you need.
  • Spawnmanager override does not work — you must call setAutoSpawnCallback before the player finishes loading. Use it in a resource that starts early, or add it to a shared script.

Refactor

Create a custom spawn point using exports.spawnmanager:setAutoSpawnCallback that reads coordinates from your config.lua, so every new player spawns at your custom location instead of the airport.

Cheat sheet

All built-in resources at a glance

  • chat — Captures chat input, provides chat:addMessage export and chatMessage event.
  • spawnmanager — Controls player spawns. Provides spawnPlayer, setAutoSpawn, forceRespawn exports.
  • baseevents — Fires onPlayerDied, onPlayerKilled, onPlayerWasted, onPlayerActivatedLocally.
  • sessionmanager — Manages player slots, deferrals, and the connecting screen. Do not touch.
  • hardcap — Enforces player slot limits. Do not disable unless you know why.
  • webadmin — The built-in web admin panel at port 30120. Can be disabled.

The big idea, in one minute

When you type /heal and it works, you thank your script. But your script only runs because chat captured the keystrokes, turned them into a chatMessage event, and passed them to RegisterCommand. When a player dies and your death log writes to the database, it works because baseevents fired onPlayerDied. These built-in resources are the platform underneath your platform. They are invisible until they break — and then nothing works.

Think of a house's plumbing.

You turn on the tap and water comes out. You do not think about the pipes in the walls. Chat, spawnmanager, and baseevents are the pipes. Your heal command is the tap. If you rip out the pipes, the tap runs dry. Knowing where the pipes run means you know where to look when the water stops.

Step 1

Create the listener resource

This resource does nothing but listen. It is a stethoscope on the built-in event system.

  1. Create resources/qu_builtins_test/ with fxmanifest.lua.
  2. The manifest needs fx_version 'cerulean', game 'gta5', server_script 'server.lua', client_script 'client.lua'.
  3. Create both script files. Leave them empty for now.
  4. Ensure the resource: ensure qu_builtins_test.
Step 2

Listen for chat and death events on the server

These events fire on the server. Registering them on the client does nothing.

  1. Paste the server.lua code from the Minimal Example.
  2. chatMessage receives the player's server ID, their name, and the raw message string.
  3. onPlayerDied receives the killer entity (or -1 for environmental), the death reason hash, and sets source to the victim.
  4. Save. Type something in chat in-game. The console prints it.
Step 3

Listen for spawn events on the client

The spawn event fires on the client, because the player's character exists on their machine.

  1. Paste the client.lua code from the Minimal Example.
  2. playerSpawned fires after the player model is loaded and placed in the world.
  3. It is the safest moment to run native calls like GetEntityCoords.
  4. Save. Rejoin the server. The console prints your spawn coordinates.
Recap

Three observations, in order

Run each test and watch the console. Every line you see was fired by code you did not write.

  1. Observation 1: chat.
    • Action: type "hello world" in chat.
    • Expected: console prints [qu_builtins] chatMessage from YourName: hello world.
    • What it means: the chat resource intercepted your keystroke and broadcast it as an event.
  2. Observation 2: death.
    • Action: die from fall damage or a weapon.
    • Expected: console prints [qu_builtins] playerDied source=1 killer=-1 reason=....
    • What it means: baseevents detected the death and fired the event for any script to use.
  3. Observation 3: spawn.
    • Action: rejoin the server.
    • Expected: console prints [qu_builtins] playerSpawned at x=....
    • What it means: spawnmanager placed your character and announced it.
Can I disable a built-in resource like chat?

You can, but then chat:addMessage and RegisterCommand stop working. Most servers replace chat with a custom one (like chat-theme resources) but keep the underlying resource running. Disabling it completely breaks command registration. Only disable built-ins if you have a replacement that provides the same events and exports.

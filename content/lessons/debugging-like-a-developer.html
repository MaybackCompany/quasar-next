Debugging like a developer

Something will break. That is not a maybe, it is a certainty. The difference between a beginner and a developer is not that the developer writes perfect code. The difference is that the developer knows how to read the error, find the line, and fix it calmly. This lesson teaches you that skill.

Step 1

Logging with tags: stop printing mystery messages

Outcome: every print in your code has a tag that tells you which resource and action produced it.

The print() function sends text to the console. But when you have 20 resources running, a bare print('hello') is useless. You have no idea which script said "hello" or why.

Use a consistent tag format: [resource:action] what happened.

The tag format is simple:

  • [resource_name:action] at the start of every print
  • Include the player ID (source on server, or the player name) so you can trace who did what
  • Include the actual data: what amount, what item, what result
Why this matters at 3 AM

When your server is live and something breaks at 3 AM, you will scroll through hundreds of console lines looking for what happened. Tags let you search (Ctrl+F) for [qu_heal] and find every line from that resource instantly. Without tags, you are reading every single line.

Step 2

F8 console vs server console: what shows where

Outcome: you know which console to look at depending on where the code runs.

FiveM has two consoles. Which one you see depends on where the code ran:

F8 console (client-side)
Press F8 in-game. This shows output from client scripts only. Any print() in a client.lua file appears here. The server never sees these messages. Think of it as your personal view of what is happening on your machine.
Server console (server-side)
This is the terminal window (or txAdmin live console) where your server runs. Any print() in a server.lua file appears here. All players' server events show up in this one place. This is where you look for security logs, cooldown checks, and database operations.
Quick rule

If the file is called client.lua (or is listed as client_script), the prints go to F8. If the file is server.lua (or server_script), the prints go to the server console. Always check the right console for the right side.

Step 3

pcall: catching errors instead of crashing

Outcome: you can wrap risky code in pcall so your script logs the error instead of dying.

pcall stands for "protected call." It runs a function, and if that function crashes, pcall catches the crash and gives you the error message instead of killing your entire script.

How pcall works, step by step
  • pcall(function() ... end): runs the function you pass to it. If the function completes without errors, ok is true and err is nil.
  • If the function crashes (for example, trying to read a key on a nil table), ok is false and err is a string describing what went wrong.
  • tostring(err): converts the error to a string so you can print it safely. Sometimes the error is already a string, but tostring is a safe habit.
  • Your script keeps running after the pcall. Without pcall, the same error would kill the entire script.

Do not use pcall to hide bugs

pcall catches the error so your script survives. But you must log the error. If you write pcall without printing the error, you are hiding bugs from yourself. The problem still exists; you just can't see it anymore. Always print inside the if not ok block.

Step 4

debug.traceback(): reading the call stack

Outcome: you can print a stack trace that shows exactly which functions called which.

Sometimes knowing what crashed is not enough. You need to know the chain of calls that led to the crash. That is what a stack trace (also called a traceback) gives you.

When to use traceback

Use debug.traceback() when you have multiple functions calling each other and you need to know which path led to the crash. For simple scripts with one or two functions, the error message alone is usually enough.

Step 5

The "stop and collect data" checklist

Outcome: you have a repeatable process for diagnosing any bug, not just staring at the screen.

When something goes wrong, beginners often freeze or start changing random code. Developers follow a checklist. Before you change anything, answer these five questions:

  1. What event fired? Which event name triggered the code that broke? Check your tagged logs.
  2. What was the payload? What data was sent with the event? Print the arguments before the risky code runs.
  3. What was source? On the server, which player triggered this? Print source at the top of every handler.
  4. What was the expected result? What should have happened? Write it down in one sentence.
  5. What was the actual result? What did happen? An error? Wrong data? Nothing at all?

Once you have answers to all five questions, the fix is usually obvious. The hard part is not the fix. The hard part is collecting the right information first.

Remove debug prints before shipping

These DEBUG prints are for you during development. Remove them (or comment them out) before your script goes live. A production server does not need to print every player's deposit amount to the console on every transaction.

Step 6

Performance basics: stop wasting ticks

Outcome: you understand why some code is slow and how to avoid the worst performance mistakes.

FiveM runs your Lua code inside a game that also renders 3D graphics, plays audio, handles physics, and manages network traffic. If your script does too much work every frame (every "tick"), the game stutters.

Three rules to start with:

Rule 1: Avoid every-tick logic

A Citizen.CreateThread with Wait(0) runs your code every single frame. That is 60 times per second (or more). Most code does not need to run that often.

A cheater can fire events as fast as they want. If your server handler does database work, one cheater spamming events can slow down everyone. The debounce above also serves as rate limiting. Always have one.

Wait(0) is not free

Wait(0) means "run this code every frame." If you have 10 scripts each running a Wait(0) loop, that is 10 chunks of code running 60 times per second. Players will feel the lag. Only use Wait(0) when you genuinely need frame-perfect timing (drawing UI, detecting nearby objects for interaction).

Step 7

Common error messages decoded

Outcome: you can read the three most common Lua errors in FiveM and know exactly what to fix.

These three errors account for the majority of FiveM script crashes. Learn to read them and you will fix most bugs in under a minute.

"attempt to index a nil value"

"attempt to call a nil value"

"SCRIPT ERROR" with a file path and line number

The error always tells you where

Every FiveM script error includes the resource name, file name, and line number. @qu_bank/server.lua:47 means "open the qu_bank resource, open server.lua, go to line 47." The error message after the colon tells you what went wrong. Read the whole message. Do not guess.

Common mistakes

Using print() without tags (impossible to filter)

A bare print('something happened') is useless when 15 resources are running. You will scroll through hundreds of lines with no way to tell which resource produced which message. Always use the [resource:action] tag format. It takes 5 extra seconds to type and saves hours of confusion.

Catching errors with pcall but not logging them

Writing local ok, err = pcall(fn) and then not checking ok or not printing err is worse than not using pcall at all. Without pcall, at least the error shows in the console. With a silent pcall, the error disappears completely. Your script silently does the wrong thing and you have no idea why. Always print the error inside if not ok.

pcall hiding real bugs that should crash

Do not wrap your entire script in one giant pcall. If a critical function fails (like loading player data), you want to know immediately. Use pcall around specific operations that might legitimately fail (database queries, external API calls, optional features), not around everything. A crash that you see is better than a silent failure that corrupts data.

Leaving debug prints in production code

Debug prints that dump player data, amounts, and event payloads are great during development. But on a live server with 64 players, they flood the console and can even leak sensitive data. Clean them up before committing. A good habit: prefix debug prints with DEBUG so you can search and remove them easily.

Cheat sheet

print('[res:action] ...')
Tagged logging. Always include resource name, action, and relevant data.
F8 console
Shows client-side prints. Press F8 in-game.
Server console
Shows server-side prints. The terminal or txAdmin live console.
pcall(fn)
Protected call. Returns true, result on success or false, error on failure. Script keeps running.
debug.traceback()
Returns a string showing the chain of function calls. Read bottom to top.
"attempt to index a nil value"
You tried to read a property on something that does not exist. Check the variable is not nil.
"attempt to call a nil value"
You tried to call a function that does not exist. Check the spelling (Lua is case-sensitive).
Wait(0) vs Wait(n)
Wait(0) = every frame. Use higher values when frame-perfect timing is not needed.
Data-collection checklist
Event? Payload? Source? Expected? Actual? Answer these before changing code.

What you learned

  • Tagged logging with [resource:action] format makes your console searchable and useful
  • Client prints go to F8, server prints go to the server console (never mix them up)
  • pcall catches errors so your script survives, but you must always log the error
  • debug.traceback() shows the chain of function calls that led to a crash
  • The 5-question checklist (event, payload, source, expected, actual) is your debugging process
  • Avoid Wait(0) loops unless you need frame-perfect timing; use distance-based wait scaling
  • The three most common errors all tell you the file and line number; read the whole message

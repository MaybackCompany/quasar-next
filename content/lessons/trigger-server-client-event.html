Before you start

How to read this lesson

The page looks bigger than the idea. You only need to learn four verbs: TriggerServerEvent, TriggerClientEvent, source, and validation. Everything else is just examples of that same loop.

Boxed rule: copy source before async work

source exists only inside server event handlers. Treat it like a hot value: first line inside the handler is local src = source, then use src everywhere. If the handler does any Wait, SQL await, callback, or delayed work, never read source again after that point.

The mental model: two radios, one channel

When an event crosses the wall, you use different verbs. The client uses TriggerServerEvent to talk to the server. The server uses TriggerClientEvent to talk to a client. Same event idea, different direction.

There is one extra rule: the receiving side must declare that it is ready to accept that event name. That declaration is RegisterNetEvent. Without it, the message will not reach the handler.

Why RegisterNetEvent exists

Think of it as a safety switch. RegisterNetEvent is you saying: "yes, this exact event name is allowed to cross the wall." On the server, treat every RegisterNetEvent like an open door and decide carefully if it should exist.

Four verbs: know which one goes which direction

Write these four lines on a sticky note. You'll use this table more than any other in the course.

Three things matter here. First, -1 means "send to every connected client." Second, source is the trusted server-side id of the player who fired the event. Third, the direction is built into the function names, so choose the one that matches where the message is going.

Step 1

Client → Server: the ask

Outcome: a client command that tells the server "I want to heal" and the server acknowledges it.

We'll build qu_netevents. The client registers /healme and fires a server event. The server registers the event (with RegisterNetEvent) and prints a line when it arrives.

The source line looks a bit unusual. Inside a server net-event handler, FiveM automatically fills in source with the id of the player who fired the event. A good habit is to copy it immediately with local src = source.

Why copy source immediately

source is not something you should keep reading later in the handler after waits or async work. Copy it once at the top and use your local variable after that.

Step 2

Server → Client: the answer

Outcome: the server tells the requesting client to play the heal effect, completing a round trip.

The server shouldn't just log; it should act. Let's close the loop. Server decides yes, server credits whatever it needs to credit, then server tells the one client to do the visible part.

Run /healme. The server console logs the request, and the F8 console logs the heal. That is the full loop: client asks -> server decides -> client performs.

Two separate RegisterNetEvents

Notice both sides now have their own. The server registers qu:server:heal because the client will fire it across. The client registers qu:client:doHeal because the server will fire it across. Any event that crosses the wall needs a RegisterNetEvent on the receiving side, full stop.

The choice between RegisterNetEvent and AddEventHandler is a security decision

These two are not interchangeable, and the difference is your first line of defense. RegisterNetEvent opens an event to the network, so any player can fire it from their console. AddEventHandler on its own does not, it only hears events fired by your own Lua on the same side. FiveM flatly refuses to route a network message to an event you never registered.

That refusal is a feature, not a limitation. It means the rule is simple: only call RegisterNetEvent for events that genuinely need to be fired from the other side. If an event is just server code talking to server code, keep it as a plain AddEventHandler with no RegisterNetEvent, and you have made it impossible for a player to reach. Every needless RegisterNetEvent is one more door a cheater can knock on.

Step 3

Broadcast: one server, many clients

Outcome: the server sends a message to every connected player at once.

The -1 target means "all clients." It is useful for announcements or anything everyone should see at once.

From the server console (not F8), type sayall dinner is ready. Every connected client sees the line in their F8 console. Swap -1 for a specific server id to hit one player instead. That's the whole targeting model: one id, or minus one.

Broadcasts are expensive, use them deliberately

A broadcast with 100 players pushes the message 100 times. For anything that isn't genuinely for everyone, send to specific clients or use a state bag (Module 05 territory). Resist the temptation to -1 every effect, good servers reserve broadcasts for real "everybody" events.

Step 4

Validate arguments: because they came from a client

Outcome: your server handler refuses to act on malformed or hostile input.

The moment a net event carries arguments, those values came from the player's machine. That means they can be wrong, weird, or malicious. Validate them before you use them.

Use the same four checks every time: type, range, authority, and context. Lesson 12 introduced them. This is where they become real.

Never trust types

Lua is dynamically typed. A cheater can send TriggerServerEvent('qu:server:giveItem', {}, 'lol') and your server will crash the moment it tries #itemName on a table. type() guards aren't pedantic, they're the difference between a live server and a 2 a.m. restart.

Step 4+

The full security checklist: every gate, in order

Outcome: one mental checklist you run top to bottom on every server net-event handler that does anything valuable.

The four checks above (type, range, authority, context) are the foundation. But a real money-or-items handler needs more gates than that, and the order they run in matters. Think of it like airport security: cheap checks first (is this even the right shape?), expensive checks last (charge the card). The moment any gate says "no", you return and nothing happens.

Here is the whole checklist as one labeled artifact. You do not need every line on every event, a handler that just prints a message needs almost none of it, but a handler that touches money or inventory should walk all eleven. Read the comments, they are the lesson.

Two gates trip people up the most, so let's name them plainly.

GateWhat it stopsThe tell-tale exploit
NaN check (qty ~= qty)Garbage math from 0/0Cheater sends 0/0 as the amount and your price math returns nonsense
Whitelist over blacklistNames you never thought of'Admin_Weapon' or 'admin_weapon ' with a trailing space sails past a blacklist
Location check"Use the shop from anywhere"Cheater fires qu:server:buy while standing across the map
Rate limitEvent spam1,000 fires per second to brute-force a timing bug
Lock + atomic deductThe duplication raceTen parallel fires all pass the "has money" check before any one deducts
Whitelist, never blacklist

Listing the bad values (if itemId == 'admin_weapon' then return end) is a losing game. Attackers have infinite spellings, casings, and trailing spaces. Flip it: build a table of the values you allow, look the input up in it, and bail if it is not there. Only known-good passes, everything else is rejected by default.

The duplication race is the one that ruins economies

Imagine ten copies of your handler starting at the same millisecond. Each one checks "does this player have $100?" Yes, yes, yes, ten times, because none of them has deducted yet. Then all ten deduct and all ten hand out an item. The player paid once and got ten. The fix is two layers: a per-player lock so only one copy runs at a time, and an atomic deduction so the database itself refuses to go negative. You will see the database version of this in Module 06.

Attack your own handler before a cheater does

After you write a handler, open the F8 console and try to break it like a player would. Throw the ugliest inputs you can think of at it and watch for any money change, item, or red console error:

If nothing happens on every single line, your gates are doing their job. If anything moves, you just found the bug before your players did.

Recap 1

Trace the bug

A developer reports: "I run /healme and nothing happens. No errors either." You open their code. Find the three problems.

The three fixes
  1. Client uses TriggerEvent instead of TriggerServerEvent. Same-side verb, message never crosses the wall. Fix: change to TriggerServerEvent('qu:server:heal').
  2. Server uses AddEventHandler instead of RegisterNetEvent. Without RegisterNetEvent, the server refuses to deliver a net-fired event to the handler. Fix: replace with RegisterNetEvent('qu:server:heal', function() ... end).
  3. Client uses AddEventHandler for an event coming from the server. Same issue in reverse. Fix: RegisterNetEvent('qu:client:doHeal', function() ... end).

Two of those three bugs were about the direction verb, one about the declaration. Those two categories cover roughly 90% of "events mysteriously not firing" reports in real life.

Step 5

The full loop: a secure mini-feature

Outcome: a complete request/validate/decide/write/tell/perform cycle in working code.

Here's the shape stripped to essentials, a /ping command that round-trips through the server with a tiny state check. No Quasar-specific tooling, no framework. Just the six beats from Lesson 12.

Walk the beats: client asks with TriggerServerEvent. Server checks: the src is trusted (source is set by FiveM, not the player). Server decides: in this toy, always yes. Server writes the streak to its own table. Server tells the client with TriggerClientEvent. Client performs: prints the streak. Every beat is in exactly the right process.

Use this skeleton every time

Before you write any network feature, sketch those six beats. If any one of them lives on the wrong side, the feature is cheatable. If a beat is missing, the feature is broken. This mental checklist prevents the most expensive category of bugs in FiveM.

Recap 2

Name the verb

For each scenario, name the single function call that starts the cross-wall travel, and whether any RegisterNetEvent needs to be added and where.

  1. The player presses E; the server needs to spend $100 from their balance.
  2. The server has just set the in-game weather; every client needs to play the rain VFX.
  3. The server wants to tell player id 42 they've been promoted to bartender.
  4. A client-side HUD resource wants to flash a damage indicator locally when the player takes damage; no server involvement.
Answers
  1. TriggerServerEvent('qu:server:buyThing') on the client. Server registers RegisterNetEvent('qu:server:buyThing', ...).
  2. TriggerClientEvent('qu:client:weatherFx', -1, 'rain') on the server. Every client registers RegisterNetEvent('qu:client:weatherFx', ...).
  3. TriggerClientEvent('qu:client:promoted', 42, 'bartender') on the server. Every client registers RegisterNetEvent('qu:client:promoted', ...) (each client's handler only runs if that client is id 42).
  4. TriggerEvent('qu:client:damageFx'): no wall crossing, no RegisterNetEvent needed. Lesson 13 territory.

Common mistakes: the ones that bite everyone

Passing source as an argument from the client

Never do TriggerServerEvent('qu:server:giveMoney', source, 500) and have the server use that id. A cheater will pass someone else's id. source on the server is already correct, already trusted, already tied to the real sender. Use the server's source, ignore anything the client sends.

Forgetting RegisterNetEvent

AddEventHandler alone does not receive net-fired events. The handler is silent; nothing errors. If an event refuses to fire across the wall, this is the first thing to check, 99% of the time, that's it.

Using source after a Wait or a DB query

source can change under you in async handlers. Copy it to a local first thing: local src = source. This is not a style preference, it's a correctness rule. Every tutorial you'll read has this; now you know why.

Trusting client-sent numbers

"The client tells me how much money to deduct." No. The server looks up the real amount from its own state. The client only sends intent, never values the server could determine itself.

Using the wrong direction verb

TriggerServerEvent fired from the server throws a clear error, that one you'll notice. TriggerEvent fired on the client when you meant TriggerServerEvent silently goes nowhere. When an event doesn't fire, check the verb before anything else.

What you learned

  • TriggerServerEvent sends a message from the client to the server; TriggerClientEvent sends from server to one client (by id) or all clients (-1)
  • RegisterNetEvent must be declared on the receiving side before a cross-wall event can be handled
  • source is the trusted player id on the server -- copy it immediately with local src = source
  • Every argument that crosses the wall came from a player's machine and must be validated: type, range, authority, context
  • The full secure loop: client asks, server checks, server decides, server writes, server tells, client performs

Level up: what's next

Events are a one-way shout. Perfect for "I did a thing" or "please do a thing", not perfect when the client needs to ask the server a question and wait for the answer. "Do I have enough money?" "What's this vehicle's owner?" For those, events are awkward, two events, one for the ask and one for the reply, plus tracking which ask matches which reply. There's a cleaner tool for it. It's called a callback, and it's the next lesson.

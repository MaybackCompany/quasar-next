What natives let you do

Natives are the bridge between your Lua code and the GTA V world. Use them when a script needs to read the player, move an entity, spawn something, draw UI, play an animation, manage a vehicle, or check game state.

Player tools
Read position, heading, health, armor, sprinting, vehicle state, and current weapon.
Example: GetEntityCoords(PlayerPedId()) powers location logs, marker checks, and zone scripts.
World control
Move entities, freeze them, delete them, spawn models, change weather, and create blips.
Example: a garage script uses vehicle natives to spawn, place, repair, and lock cars.
Interaction polish
Draw text, start animations, play sounds, set cameras, show help text, and create clean client feedback.
Example: a job script uses animation and UI natives before it pays the player on the server.
Safe reference habit
Search cfxnatives.dev, an excellent native reference by UzStore, then confirm side and signature on the official docs.
Example: never paste a native until you know if it belongs in client.lua or server.lua.

Most-used natives to recognize first

You do not need to memorize every native. Start with the names below because they appear constantly in real FiveM resources.

Client. Reads or sets facing direction. You use it for parking, ped placement, and scene alignment.
GetEntityHealth(entity) / SetEntityHealth(entity, value)
Client. Reads or changes health for peds and entities. Good for health checks and scripted recovery.

The mental model: a native is the handle on a GTA V lever

Think of GTA V like a giant machine with lots of buttons. One button moves a ped. One button changes weather. One button draws text. Rockstar wrote those buttons in C++. You cannot call that C++ directly from Lua. FiveM gives you a bridge -> a Lua function that presses the button for you. That function is a native.

Every native has a hidden hash under the hood, like 0x06843DA7060A026B, but you never need to type that. You use the readable name like SetEntityCoords, GetPlayerPed, or DrawText. FiveM does the lookup for you.

There are a lot of natives. You are not supposed to memorize them. The real skill is this -> know how to find the right one when you need it.

Two references, one habit

The official reference is docs.fivem.net/natives. This is the official source, but it can feel heavy for beginners. The community alternative is cfxnatives.dev, an excellent native reference by UzStore. It is faster to browse and easier to filter by side, framework, and namespace. Best habit: use cfxnatives.dev to find a native -> then use docs.fivem.net to confirm it.

If this feels slow, good

This lesson is not about memorising names. It is about building a calm habit: search -> confirm the side -> copy the Lua-shaped call -> test it.

Native drills

5 searches to build the muscle

Open cfxnatives.dev. For each question, search the reference and answer out loud. The goal is not to memorize every native. The goal is to build the habit of finding the right one in under 30 seconds.

Drill 1: Find a native that freezes an entity in place. What's the name, and which side does it run on?

FreezeEntityPosition(entity, toggle). Client-side. Namespace ENTITY. Useful for cutscenes, menus, and "don't let the player move during this animation" moments.

Drill 2: Server-only -> find the native that kicks a player with a reason string.

DropPlayer(playerSrc, reason). Server-side. Namespace PLAYER. This is an example of why filtering by side matters: try to call DropPlayer from a client file and Lua throws attempt to call a nil value.

Drill 3: You want to force sunny weather instantly. Find the native and note its namespace.

SetWeatherTypeNow('EXTRASUNNY'). Client-side. Namespace MISC. The MISC namespace is the catch-all area for things that do not fit neatly anywhere else -> weather, time, hash functions, and script globals. If you cannot find something obvious, check MISC.

Drill 4: Spot the return-type difference between GetEntityCoords and GetEntityHeading.

GetEntityCoords(entity, alive) returns a Vector3 -> you read .x .y .z. GetEntityHeading(entity) returns a float -> one single number from 0 to 360. Mixing them up, like using pos.x on a float, is a classic beginner mistake. The return-type column on cfxnatives helps you catch that early.

Drill 5: GetPlayerName exists on both the Client and Server filter. Why is that, and which one should your ban/allow list use?

They are two separate natives with the same name. The client one takes a player index on your machine. The server one takes a source. For identity-sensitive work like bans or logs, always use the server version -> the client can lie. For permanent identity, pair it with the player's license identifier, not the display name. That comes back in Lesson 12.

Notice three things:

  • The name flips from SET_ENTITY_COORDS (SCREAMING_SNAKE_CASE) to SetEntityCoords (PascalCase). That's the Lua convention in FiveM. Always.
  • The void at the front means it returns nothing. If you see Entity or int or float there, the native returns a value you can capture.
  • The argument types are C-style. You translate them to Lua.
Translation table, C type to Lua value
  • Entity, Player, Ped, Vehicle: a handle, i.e. an integer. You get one from another native, like PlayerPedId() or GetPlayerPed(source).
  • float: a Lua number, always written with a decimal point. Use 100.0, not 100.
  • BOOL: Lua true or false.
  • const char*: a Lua string, like 'a_m_y_beach_01'.
  • Hash: an integer hash. Two ways to produce one: `some_name` with backticks (compile-time, when the name is hard-coded), or GetHashKey(name) (runtime, when the name lives in a variable). Step 6 shows both.
  • int: a plain Lua integer.
The float trap, 100.0 vs 100

GTA natives are picky about floats. A native that expects a float and receives an integer-looking number can silently do nothing, misbehave, or error with a cryptic message. Train your fingers now: whenever a native argument is a coordinate, health, speed, heading, or any float, write it with a decimal, 0.0, 1.0, 100.0, -1336.0. Every time.

Step 2

Check the side before you copy-paste

Outcome: you never waste 20 minutes debugging a native that was on the wrong side of the wall.

Lesson 10 drilled in the client/server wall. Natives live on one side or both. Every docs page shows a badge: Client, Server, or Shared. Read it before you paste the native into a file.

  • PlayerPedId(): client only. The server has no concept of "your" ped.
  • GetPlayers(): server only. The client only knows about players near it.
  • GetHashKey('model'): shared. Works on both sides.

If you call a server-only native from a client file, Lua tells you attempt to call a nil value. That error is not random, it means the function you named doesn't exist on the side you're running on.

A 10-second habit

Before you paste a native into your file, glance at the side badge and at your filename. Is this client/main.lua or server/main.lua? Does the native's side match? Ten seconds. Saves hours.

Step 3

Meet the 10 natives you'll use every day

Outcome: you recognize the core toolbox and know which side each native lives on.

You do not memorize thousands. You learn these ten first. The rest you look up later. Open the docs page for each one as you read. The goal here is familiarity, not perfect recall.

PlayerPedId()
Client. Returns a handle to the local player's ped. Your starting point for almost any client script that touches the player.
GetEntityCoords(entity)
Client. Returns the entity's position as a vector3: you can read .x, .y, .z.
SetEntityCoords(entity, x, y, z, ...)
Client. Moves the entity. The four trailing BOOLs are almost always false.
GetEntityHealth(entity) / SetEntityHealth(entity, value)
Client. Read or set HP. Peds default to 200 max.
GetPlayers()
Server. Returns a table of all connected player source IDs. Your entry point for "loop over everyone" logic.
GetPlayerPed(source)
Server. Given a player's server ID, returns their ped handle, the server-side equivalent of PlayerPedId().
GetHashKey(name)
Shared. Converts a string (model name, animation dict, etc.) into the integer hash natives want. Shortcut: if the name is known at write time, write `a_m_y_beach_01` with backticks instead. The compiler turns it into the same integer at parse time, no runtime call.
RequestModel(hash) / HasModelLoaded(hash) / SetModelAsNoLongerNeeded(hash)
Client. The three natives behind every ped and vehicle spawn. You'll see the pattern in Step 6.
DrawText (family)
Client. Draws text on screen. Requires a render loop, covered in a later module, but now you know it exists.
Wait(ms)
Both. Yields the current thread for ms milliseconds. Only works inside a thread, see Step 4.

That's your starter kit. Every other native you meet is a variant of these themes: get something, set something, check something, spawn something.

Native families: your beginner vocabulary

The docs group natives into families (the docs call them namespaces) based on what they touch. You do not need to learn these by heart. The point is to recognize the family a native belongs to, so when you need to do something, you already know roughly where to look. Skim this once, then come back to it like a phone book.

FamilyWhat it touchesNatives you'll meet first
Player The person behind the character: their index, their name, the list of everyone connected. PlayerId(), PlayerPedId(), GetPlayerName(p), GetPlayerServerId(p), GetPlayers() (server)
Entity The shared bucket. Anything in the world with a handle: peds, vehicles, and props all answer to these. GetEntityCoords(e), SetEntityCoords(e, ...), GetEntityHeading(e), DoesEntityExist(e), DeleteEntity(e)
Ped Character-specific reads and changes: health, armor, weapons, and "what is this character doing right now?" checks. GetEntityHealth(ped), SetPedArmour(ped, n), IsPedInAnyVehicle(ped, false), GiveWeaponToPed(ped, hash, ...)
Vehicle Cars, bikes, boats, planes: spawning them, locking doors, reading the plate, checking the class or speed. CreateVehicle(model, ...), SetVehicleEngineOn(v, ...), SetVehicleDoorsLocked(v, 2), GetVehicleNumberPlateText(v)
Coords / Math Positions and distances. FiveM's Lua adds vector3, so you can do real geometry without a math library. GetEntityCoords(e), #(coordsA - coordsB) (distance), Vdist(x1, y1, z1, x2, y2, z2)
Blips Those icons on the pause-menu map and minimap: create one at a point, give it a sprite, color, and name. AddBlipForCoord(x, y, z), SetBlipSprite(b, n), SetBlipColour(b, n), RemoveBlip(b)
The distance trick worth memorizing

FiveM gives Lua a real vector3 type, so you can subtract two positions and measure the gap in one tiny line: local dist = #(GetEntityCoords(ped) - shopCoords). The # sign means "length of this vector", which here is the distance in meters. You will use this constantly for "is the player close enough to the counter?" checks. Plain Lua has no vectors, so this only works inside FiveM.

Server IDs versus client indexes

This catches every beginner once. On the client, PlayerId() hands you a small local index, useful only with client natives. On the server, you work with a player's server ID (the number admins see, the one you target with TriggerClientEvent). They are not the same number, so never mix them. When you need to cross over on the client, GetPlayerServerId(PlayerId()) gives you the server ID.

Recap 1

Which side, which native?

Answer in your head before you peek.

You're in client/main.lua and you want the player's current position as a vector3. Which two natives do you chain?

PlayerPedId() to get your ped handle, then GetEntityCoords(ped) to get the position. Both client-side. One-liner: local pos = GetEntityCoords(PlayerPedId()).

You're in server/main.lua and you want to get the ped of the player whose server ID is in src. What do you call?

GetPlayerPed(src). The server version of PlayerPedId. PlayerPedId() itself doesn't exist on the server, it has no concept of a "local" player.

You wrote SetEntityCoords(ped, 100, 200, 30, false, false, false, false) and nothing visible happens. Why?

Integer coordinates. The native expects floats. Change to 100.0, 200.0, 30.0. This is the single most common beginner mistake and you just inoculated yourself against it.

What every line is doing
  1. CreateThread(function() ... end): spins up a cooperative thread. Wait only works inside a thread; outside one it errors. (You will also see Citizen.CreateThread in older code, it is the legacy alias for the same function. New code should use the short CreateThread form.)

  3. PlayerPedId(): grab your ped handle.
  4. GetEntityCoords(ped): returns a vector3. You read pos.x, pos.y, pos.z.
  5. ('...'):format(...): Lua's string.format. %.2f means "a float rounded to 2 decimals". You'll use this constantly to print readable numbers.
Vector3 is a first-class citizen here

FiveM's Lua understands vector3(x, y, z) natively. You can subtract two vectors and take the length, #(v1 - v2) gives you distance in meters. You'll lean on this for "is the player near the shop counter?" checks in later modules.

Step 6

The model loading pattern

Outcome: you recognize the four-line boilerplate that precedes every ped or vehicle spawn.

Most "spawn something" natives take a model hash. The engine streams models on demand, you have to ask for one, wait for it, then release it when you're done. The pattern is everywhere:

Reading line by line:

  • `a_m_y_beach_01` (backticks, not single quotes): a compile-time hash literal. The CfxLua parser sees the backticks and replaces the whole thing with the integer hash before your script ever runs. Zero runtime cost. Use this whenever the model name is known at write time.
  • RequestModel(hash): tell the engine "start streaming this in".
  • while not HasModelLoaded(hash) do Wait(10) end: poll every 10ms until the model is ready. Yes, this is a busy-wait. Yes, it feels clunky. No, there is no cleaner way, this is the FiveM idiom.
  • Between the loop and SetModelAsNoLongerNeeded, you spawn whatever needs the model (a ped, a vehicle, a prop).
  • SetModelAsNoLongerNeeded(hash): tell the engine you're done. The engine can now unload the asset when it needs the memory.

This is your first taste of an async pattern in FiveM. RequestModel starts a fetch, HasModelLoaded checks progress, the loop holds the thread until it's ready. Every engine has trade-offs; this is one of GTA V's.

Backticks vs GetHashKey

Both produce the same integer hash for the same string. Pick by where the name lives:

  • Backticks `a_m_y_beach_01` when the name is hard-coded in your file. Free at runtime, no function call.
  • GetHashKey(name) when the name is in a variable, a config table, a database row, or anything decided at runtime. local hash = GetHashKey(model).

Older lessons and old resources use GetHashKey for everything. It still works fine, but new code should use backticks for compile-time-known names. It is shorter and faster.

Why the while loop matters

Skip the loop and call the spawn native immediately, the model isn't loaded yet, so nothing spawns. No error. No warning. Just silence. The polite wait is not optional; it's the reason your ped exists.

What are entities, objects, peds, and weapons?

Beginners often see these words everywhere and treat them like the same thing. They are related, but not the same. Think of this section like your tiny field guide.

Entity
The broad bucket. In FiveM and GTA V, an entity is a thing that exists in the game world and has a handle. Peds, vehicles, and objects are all entities.
Ped
A character model. The player is a ped. NPCs are peds. Police officers, shopkeepers, and random civilians are peds.
Object / prop
A world item like a chair, ATM, crate, cone, cash pile, or vending machine. In community talk, object and prop are often used almost interchangeably.
Vehicle
A car, bike, truck, plane, boat, or helicopter. It is its own entity type with vehicle-specific natives.
Weapon
A weapon is usually identified by a weapon name or weapon hash. It is not the same thing as a ped or prop. You give a weapon to a ped, or check what weapon a ped is holding.
The simple mental model

If it exists in the world, start by asking: is it a ped, a vehicle, or an object? If yes, it is also an entity. That helps you search the right native family faster.

The beginner story version

You are not "coding GTA." You are usually doing one of four things: reading something from a ped, moving a ped or vehicle, spawning a prop, or giving a weapon. Once you name the bucket correctly, the docs become much less scary.

Where do you find ped names, object names, and weapon names?

There are two separate things you usually need to find:

  • The native you want to call, like CreatePed or GiveWeaponToPed.
  • The model or weapon name you want to pass into it, like a shopkeeper ped name, a cone prop name, or a pistol weapon name.

For the native itself, use cfxnatives.dev first and confirm on docs.fivem.net.

For the model or weapon name, the practical beginner workflow is:

  1. Search the web for the thing in plain English, like fivem shopkeeper ped name or fivem cone prop name.
  2. Check a trusted model list or code example.
  3. Use GetHashKey('name_here') if the native wants a hash.
  4. Test one small spawn or one small print before building a whole system around it.
Do not memorize names

You are not expected to remember every ped name, prop name, or weapon name. The real skill is finding the right name quickly, then testing it in a tiny script.

Starter guides: peds, cars, props, and weapons

If you are a total beginner, build in this order. Each guide is one small kind of script, not a giant system.

Find one ped model -> load it -> spawn it -> clean up the model. First goal: one shopkeeper or guard standing where you expect.
Vehicle guide

Common mistakes & what the console tells you

Bug: teleport line runs, nothing moves

You passed integers where floats belong. SetEntityCoords(ped, 100, 200, 30, ...) fails silently. Change every coordinate to have a decimal: 100.0, 200.0, 30.0. This is the single most common FiveM-beginner bug. Train your fingers to always type the .0.

Error: attempt to call a nil value (global 'GetPlayers')

You called a server-only native (like GetPlayers) from a client file, or a client-only native (like PlayerPedId) from a server file. The native doesn't exist on that side. Check the docs page's side badge, then move the code to the matching file.

Bug: ped or vehicle just doesn't spawn

You skipped the RequestModel / HasModelLoaded loop. The engine needs time to stream the model in. Without the polite while Wait(10) wait, the spawn native runs before the model is ready and returns a useless handle. Always load before you spawn.

Error: attempt to yield from outside a coroutine

Wait only works inside a CreateThread (or an event handler, later module). Wrap the code that uses Wait in CreateThread(function() ... end) and the error disappears. (Older code uses Citizen.CreateThread: same function, legacy alias.)

Native exercise ladder for total beginners

If you are starting from zero, do not jump straight from "I found a native" to "I built a whole feature". Build up in tiny wins. Each drill below is deliberately small enough to verify with your own eyes.

  1. Coords drill. Write one thread that prints your current coordinates. Success means you can explain why GetEntityCoords returns a vector3.
  2. Heading drill. Add GetEntityHeading(ped) and print it beside the coords. Success means you understand the difference between a vector return and a single float return.
  3. Teleport drill. Move your ped to one safe location with SetEntityCoords. Success means you remembered the side badge and the .0 habit.
  4. Health drill. Search for GetEntityHealth and wire a command called /myhp that prints your current health. Success means you can find and use a native without the lesson handing you the answer.
  5. Model-loading drill. Write the full GetHashKey → RequestModel → HasModelLoaded loop → spawn → SetModelAsNoLongerNeeded pattern. Success means you understand why "requested" and "loaded" are two different states.

Workflow every time: search on cfxnatives.dev, check the side badge, confirm the signature on docs.fivem.net, then write the Lua call yourself.

  • How to read a C-style signature and translate it into a working Lua call (PascalCase, floats with .0, booleans)
  • Always check the side badge (Client / Server / Shared) before pasting a native into your file
  • The model-loading pattern: RequestModel then HasModelLoaded loop then spawn then SetModelAsNoLongerNeeded
  • How to use cfxnatives.dev to find a native quickly, then confirm on docs.fivem.net

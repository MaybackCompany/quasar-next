Customize and reskin script UIs with color themes

By the end of this lesson you will change the color scheme of a script's NUI interface — buttons, backgrounds, text, and borders — by editing CSS variables, and test your reskin in real time with hot reload.

You'll ship
A reskinned inventory or HUD interface with a custom color theme — dark purple backgrounds, gold accents, and white text — applied by changing exactly five CSS variables.
Time
About 25 minutes if you go slow.
You'll learn

Prereqs
You can open a script's html/ folder and edit a CSS file. Basic CSS knowledge helps but is not required.
In 3 bullets

You will reskin a script UI by doing this:

  • Open a script's main CSS file and find the :root { } block with CSS variables like --primary-color and --bg-color.
  • Change exactly five color values: primary, background, text, border, and accent.
  • Restart the resource (or use NUI hot reload) and see the new theme applied to every button, panel, and text element.

What this is
Reskinning means changing the colors, fonts, and visual style of an existing script's UI without touching its JavaScript logic.
Example: this lesson turns a blue UI into a purple-and-gold theme.
Where to find it
Look in html/style.css or html/css/main.css for a :root block with CSS custom properties.
Example: scan for --primary: or --background: declarations.
How to use it
Change the hex color values in the :root block. Save the file. Restart the resource. The entire UI picks up the new colors.
Example: change one variable and watch 50 elements recolor instantly.
What to do now
Change five variables, save, and verify every panel changed.
Example: match the expected visual result in the screenshot before moving on.

Outcome

You will open a script UI, see new colors applied to every element, and confirm that all buttons and inputs still work.

Why it matters

Every server has a brand — colors, fonts, logo. If your scripts look like stock downloads, your server looks amateur. Reskinning is the fastest way to make a server feel custom.

Vocabulary

CSS custom properties
Variables in CSS that start with --. Define a color once, use it everywhere.
Example: --primary: #e94560; then used as color: var(--primary);.
:root selector
A CSS pseudo-class that targets the top-level HTML element. Variables defined here are available to every child element.
Example: :root { --bg: #1a1a2e; } makes --bg global.
Hex color code
A 6-character code representing red, green, and blue values. #e94560 is a pinkish red.
Example: #ffffff is white, #000000 is black.

Minimal example

Expected output: All buttons change from blue to purple. All borders change from dark gray to gold. Text stays white. The UI layout and functionality are unchanged.

Checklist

  1. Open the script's html/ folder and locate the main CSS file.
  2. Find the :root { } block (or add one at the top of the file if missing).
  3. Change exactly five color values: --primary-color, --background, --text-color, --border-color, --accent-color.
  4. Save the file and restart the resource.
  5. Open the UI in-game. Confirm all elements use the new colors.
  6. Test three interactions (click buttons, type input, open/close panels) to verify functionality is intact.

Exercise

Create a "dark mode" theme using --background: #0a0a0a;, --text-color: #cccccc;, --primary-color: #444444;, --border-color: #666666;, and --accent-color: #ff4444;. Apply it to the same script and verify all elements update. Then switch back to your purple/gold theme.

Expected output: UI appears in dark grays with red accents. Switching back restores the purple/gold theme. Both transitions complete within 15 minutes.

Common failures

  • Colors do not change after editing and restarting: the NUI browser caches CSS aggressively. Use Ctrl+Shift+R in the NUI devtools, or add a cache-busting query string to the CSS URL in the HTML file.
  • Only some elements changed: the script may use hardcoded colors (background: #ff0000;) instead of variables. Search for the old hex color values and replace them with var(--your-variable).
  • UI layout breaks after editing: do not change width, height, display, position, or flex properties. Only change color values.
  • No :root block exists: add one at the very top of the CSS file, before any other rules. Define your variables there, then replace hardcoded colors with var() calls.

Refactor

Create a theme.css file that imports the original CSS and overrides only the :root variables. Link theme.css in the HTML instead of editing the original file directly.

Cheat sheet

Five variables to change for a full reskin

  • --primary-color. Buttons, headers, active tabs. Your main brand color.
  • --background. Page or panel background. Usually dark for FiveM.
  • --text-color. Body text. Must contrast with background.
  • --border-color. Dividers, input borders, card edges.
  • --accent-color. Highlights, notifications, hover states. Should pop.

The big idea, in one minute

Most modern FiveM UIs use CSS custom properties — variables that hold colors. Instead of writing background: #2980b9 on every button, the developer writes background: var(--primary-color) once. When you change --primary-color in the :root block, every element that uses that variable recalculates its color instantly. You edit one line and the entire UI shifts theme.

Think of a theater lighting console.

The console has sliders labeled "Main Spot", "Background Wash", and "Accent". Moving the "Main Spot" slider changes the color of every front-facing light in the theater at once — without touching 50 individual fixtures. CSS custom properties are the sliders. :root is the console. Every var(--primary-color) in the CSS is a light fixture plugged into that slider.

Your reskin is a lighting change. Move five sliders and the entire stage changes color.

Bad way and good way, side by side

Most beginners edit every CSS rule individually, missing some and breaking others. The good way uses variables so one change hits everything.

Don't do this

Hunting through 2000 lines of CSS to change every blue hex code by hand.

You miss the progress bar and the notification. Half the UI is purple, half is blue. Players see a broken, inconsistent theme. Finding every usage of one color in 2000 lines is nearly impossible by hand.

Do this

Use CSS variables. Change one line, hit 47 elements.

Changing --primary-color in :root updates all 47 rules simultaneously. No missed elements. No inconsistent colors. A complete reskin in one edit.

Step 1

Find the main CSS file and the :root block

Almost every NUI script has one main CSS file with a :root block at the top.

  1. Navigate to the script's html/ folder. Look for style.css, main.css, or app.css.
  2. Open the file. Search (Ctrl+F) for :root. If found, this is where the color variables live.
  3. If no :root block exists, search for -- (two dashes). Variables may be defined elsewhere.
  4. If no variables exist at all, the script uses hardcoded colors. You will need to create a :root block and refactor the colors (covered in the advanced exercise).
  5. Note the current color values. Take a screenshot of the original UI for comparison.
Step 2

Choose your new color palette

Pick five colors that work together. Use a palette tool if you are unsure.

  1. Visit ColorHunt or Coolors to browse palettes.
  2. Pick a dark background (#0d0d1a, #1a1a2e) — FiveM UIs look best on dark backgrounds.
  3. Pick a primary color for buttons and headers. Purple (#6c3caa), teal (#0d7377), or orange (#e67e22).
  4. Pick an accent color for highlights. Gold (#d4af37), cyan (#00d4ff), or pink (#e94560).
  5. Pick a text color: white (#ffffff) or light gray (#cccccc). Must be readable on your background.
Step 3

Edit the :root variables

Change exactly five values. Do not touch layout properties.

  1. In the :root { } block, find the variable that controls the main button/header color. Change its hex value to your primary color.
  2. Find the background variable. Change it to your dark background color.
  3. Find the text color variable. Change it to white or light gray.
  4. Find the border/accent variables. Change them to your accent color.
  5. Do NOT change width, height, padding, margin, display, flex, or position. Color only.
Step 4

Save, restart, and verify

The NUI browser caches CSS. A restart forces it to reload.

  1. Save the CSS file.
  2. In the server console, restart the resource: ensure ox_inventory (or your script's name).
  3. Join the server and open the UI. All buttons should be your new primary color.
  4. If colors did not change, the NUI browser cache is holding the old CSS. Close FiveM, clear your client cache, and reconnect.
  5. Take a screenshot. Compare with your "before" screenshot. Everything should match your new palette.
Step 5

Test functionality to ensure nothing broke

Color changes should never break functionality, but always verify.

  1. Click every button in the UI. Confirm each one triggers its action.
  2. Type text into any input fields. Confirm typing works.
  3. Open and close the UI three times. Confirm the open/close animation still plays.
  4. Drag any draggable panels. Confirm drag works.
  5. If anything is broken, revert to your backup and check for accidental edits to layout properties.
Recap

Three tests, in order

Run each test and confirm the expected behavior. If something does not match, fix it before moving on.

  1. Test 1, visual change.
    • Action: open the UI after restarting the resource.
    • Expected: all buttons use the new primary color, backgrounds use the new background color, borders use the new accent color.
    • How to verify: compare with your "before" screenshot. Every colored element should be different.
  2. Test 2, no broken elements.
    • Action: scan the entire UI for elements that still show the old colors.
    • Expected: zero elements with the old color scheme.
    • How to verify: if any element still shows old colors, it uses a hardcoded hex value. Search for that hex value in the CSS and replace with the variable.
  3. Test 3, functionality intact.
    • Action: click buttons, type inputs, open/close panels.
    • Expected: everything works exactly as before the reskin.
    • How to verify: perform three actions that change state (e.g., move an item, toggle a setting, submit a form).
Why do some elements ignore the CSS variable change?

Those elements are likely using hardcoded hex values instead of var(). For example, background: #2980b9; will not respond to a variable change. You need to replace the hardcoded value with background: var(--primary-color);. Search-and-replace in your editor can do this in seconds.

The rules you just followed

  • Only change color values inside CSS variables. Never touch layout properties during a reskin.
  • Change colors in the :root block so one edit affects every element that uses that variable.
  • Always restart the resource after editing CSS. The NUI browser caches stylesheets aggressively.
  • Test all interactions after reskinning. Color changes should never break functionality, but verification is mandatory.
  • Keep a backup of the original CSS file. If anything breaks, you can restore it in seconds.

OOP in Lua: classes via metatables

Lua does not have classes. It has tables and a one-line trick called setmetatable that lets you fake them well enough to ship. This lesson teaches the canonical class pattern, single-level inheritance, and, just as important, when to skip OOP entirely and just use a plain table.

Mental model: a metatable is a fallback list (a phone tree)

You phone the front desk. Front desk does not have the answer. They check a sticky note that says "if asked X, redirect to office 12". The sticky note is the metatable. The redirection is __index. The thing it redirects to is the class.

Every "instance" is a tiny table with its own data. When you read a missing key, Lua checks the metatable's __index, that is where the methods live. Method body runs, self is the instance, you mutate the instance, repeat.

FiveM-specific note

OOP is optional and easy to over-apply. The frameworks you use daily (ox_inventory, oxmysql) lean on plain tables and modules; classes appear only when there is real polymorphism. Reach for OOP for entities (Vehicle, Player, Item with subtypes), skip it for config and one-shot helpers.

Step 1

The canonical class pattern

Outcome: you can write a class in five lines without thinking.

Why __index = Vehicle

When you call v:burn(15), Lua looks up burn on v (not there), then on its metatable's __index (which is Vehicle). It finds Vehicle.burn and calls it with self = v. The instance stays small; methods live on the class.

Step 2

: vs .: the one-character trap

Outcome: you stop confusing static functions with methods.

Common confusion

Constructors are static (Vehicle.new) because there is no instance yet. Everything that operates on an existing instance is a method (Vehicle:burn). If you ever see Vehicle:new or v.burn(15), look twice, one is wrong.

Recap 1

Three vehicles, separate state

Create three Vehicle instances. Burn fuel on the first. Print all three. The second and third should still be at 100. If they all dropped, you mutated the class instead of the instance, check that self in burn is the instance.

Step 3

Single-level inheritance

Outcome: a subclass reuses parent methods and adds its own.

Two metatable layers: instance → PoliceVehicle → Vehicle. Lua walks the chain. Stop after one level, deeper hierarchies in Lua become hard to debug fast.

Step 4

When NOT to use OOP

Outcome: you reject OOP for cases where it would just slow you down.

Pro tip

Start every new module with plain tables and functions. Promote to a class only when you find yourself passing the same state argument to half the functions in the file. That is the smell that tells you state and behaviour belong in the same shape.

Recap 2

Polymorphic :describe()

Override describe() on PoliceVehicle to also show the callsign. Confirm that p:describe() returns the new format and v:describe() still returns the original. That is polymorphism working through the metatable chain.

Step 5

Profile metatable cost & memory shape

Outcome: you know the price of a method call vs a plain function, and how 5,000 entity instances actually look in resmon.

5.1: Method dispatch is cheap, but not free

One __index hop costs ~3–5ns over a direct local call on LuaJIT 2.1. That is invisible in normal use and matters only inside hot inner loops (per-frame, every entity). Run profiler record 60, open the trace, and look for the dispatch site:

Per million calls on a 7950X: direct ~6ms, dot ~9ms, colon ~12ms. So colon is roughly twice the cost of a local, still <1us per call. If you ever see a method dispatch in profiler taking >100us, the problem is not OOP, it is what inside the method (a SQL call, an HTTP request, a 10k-row table walk).

5.2: Per-instance memory in resmon 1

Each instance is one Lua table. Empty table = ~56 bytes on LuaJIT. Add four fields and you are at ~120 bytes. With a metatable already shared on the class, instances cost roughly:

  • Vehicle (4 fields): ~120 bytes × 5,000 instances = ~600 KB. Visible in resmon 1 as a flat baseline.
  • Player (12 fields + nested inventory table): ~400–1,200 bytes × 64 = ~25–80 KB. Negligible.
  • String fields are shared. Lua interns strings, 5,000 instances all storing state = 'idle' share one underlying string. Numeric and boolean fields cost the inline 16 bytes per slot.

Antipattern: putting methods on the instance instead of the class. self.burn = function(...) in the constructor allocates a new closure per instance, blowing memory by 5–10× and defeating the whole point of __index. Always put functions on the class table.

5.3: Class registry & live introspection

For long-running servers, register every class so you can query population from the console:

Pair with a :destroy() method that decrements the counter so memory leaks show up as monotonically rising counts. Real servers ship with this: production-grade frameworks like ox_target expose similar instance counts via exports.

Recap 3

Senior depth: object discipline

You can defend three claims now: method dispatch is one extra hash lookup (cheap, but cache the bound function inside hot loops), per-instance memory is observable in resmon 1 (kill closure inflation), and a class registry plus :toJSON() stops metatables from leaking into the database. If a teammate asks “why is this OOP layer adding 200 microseconds per tick?”, you can answer with numbers, not a shrug.

Common mistakes

Forgetting __index

Without it, your instances find no methods. v:burn(15) errors with "attempt to call a nil value". Always set Class.__index = Class after creating the class table.

Sharing state on the class itself

function Vehicle:setFuel(n) Vehicle.fuel = n end looks like it works on one instance, it actually mutates the class, so every instance sees the new value. Always write to self.x, never to Class.x from a method.

Calling the constructor with a colon

Vehicle:new(plate) implicitly passes Vehicle as self, which then becomes plate. Use a dot: Vehicle.new(plate).

Deep inheritance hierarchies

Three or more levels of inheritance and you cannot tell where a method comes from. Prefer composition: stick a smaller object inside the bigger one (vehicle.engine) instead of subclassing.

OOP for stateless helpers

If a module has no state, it does not need a class. local M = {}; M.fmt = function(s) return s end; return M is enough.

Cheat sheet

Class skeleton
local C = {}; C.__index = C at the top of the file.
Constructor
function C.new(...) local s = setmetatable({}, C); ...; return s end.
Method
function C:method(x) self.field = x end.
Inheritance
local Sub = setmetatable({}, { __index = Parent }); Sub.__index = Sub.
Subclass constructor
Call Parent.new(...), then setmetatable(self, Sub).
Colon vs dot
Method = colon. Static = dot. Constructor = dot.
When to skip
Stateless helpers, plain config: just use a table.

What you learned

  • A class is a table; instances are tables; the link between them is a metatable.
  • __index = Class is the line that turns a metatable into a method-lookup chain.
  • Constructors are static (dot). Methods are colon. Mixing them up is the most common bug.
  • Inheritance is two metatable layers. Stop at one level unless polymorphism really earns it.
  • Plain tables beat OOP for config and stateless helpers. Promote only when state and behaviour cluster.

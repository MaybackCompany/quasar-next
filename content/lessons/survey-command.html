/survey — build an in-game poll players can vote on

By the end of this lesson you will have a working /survey command that asks players a question, collects votes with ox_lib dialogs, and shows live results.

You'll ship
A survey system with question, options, voting, and live result display.
Time
About 35 minutes.
You'll learn
How to use ox_lib input dialogs, collect server-side votes, and prevent double-voting.
Prereqs
You have ox_lib installed and a basic understanding of server events.
In 3 bullets

You will build a survey command:

  • Admin types /survey "Best car?" "Adder" "Zentorno" "T20" to start a poll.
  • Players see the question and options via ox_lib dialog, pick one, and the vote is recorded.
  • Results display in chat showing how many votes each option received.

Start here

What this is
A survey lets one person ask a question and many people answer with one click.
Example: ask your community which car to add next and get real votes.
Where to find it
Votes are stored in a table on the server. The client only displays the dialog.
Example: the server table prevents double-voting by checking player IDs.
How to use it
Admin creates a survey. Players join and vote. Results are shown to everyone.
Example: /survey "Event time?" "8 PM" "9 PM" "10 PM" runs a real-time poll.
What to do now
Build the survey command and test it with at least two players voting.
Example: verify that voting twice is blocked and results are accurate.

Outcome

Run a survey, collect votes from multiple players, and display accurate results.

Why it matters

Community input makes players feel heard. A working poll system takes 50 lines of code and builds loyalty faster than any announcement.

Vocabulary

Survey state
The question, options, and votes stored on the server.
Example: {question='Best car?', options={'Adder','Zentorno'}, votes={}}
Double-vote prevention
Checking whether a player already voted before accepting their new vote.
Example: if votes[src] then return end stops repeat voters.
ox_lib dialog
A UI component that shows options and returns the player's choice.
Example: lib.alertDialog with radio-style selection.

Checklist

  1. Create fxmanifest.lua with ox_lib dependency.
  2. Create server.lua with the survey command and vote handler.
  3. Create client.lua that shows the ox_lib dialog when qu_survey:show fires.
  4. Type /survey "Test?" "Yes" "No" and verify the dialog appears.
  5. Vote twice and verify the second vote is blocked.
  6. Extend server.lua to show results with /survey_results.

Exercise

Add a /survey_close command that ends the survey and prints the final results to all players in chat.

Expected output: Chat shows [SURVEY] Results: Best car? — Adder: 5 votes, Zentorno: 8 votes

Common failures

  • attempt to index a nil value (field 'exports') means ox_lib is not started before qu_survey. Check your ensure order.
  • Dialog does not appear means the client event is not registered or the resource is not ensured on the client side.
  • Votes not recorded means the server event handler is missing or the event name does not match client and server.
  • Double voting still works means the votes table is being reset somewhere. Check that activeSurvey is not recreated on every vote.

Refactor

Replace the in-memory activeSurvey table with KVS storage so surveys survive a server restart.

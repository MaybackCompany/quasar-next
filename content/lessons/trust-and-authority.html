Before you start

How to read this lesson

This page teaches one rule, not ten. If you remember only one sentence, make it this: the client asks, the server decides. Every story and every checklist below is there to make that sentence stick.

The mental model: the one-sentence rule

Almost every trust decision on your server comes back to one sentence. Read it and say it out loud:

The rule

The client is the player. The player can lie. The server is you. The server cannot lie.

The client's Lua runs on the player's own PC. That means the player can read it, edit it, remove checks, and send fake values if they want to cheat. The server runs on your machine. That is where the real rules must live.

So when a script breaks trust, the bug is usually simple: the server believed something the client said without checking it for itself.

Two stories: so the rule sticks

Story A, the cheater's dream

A beginner writes a bank. The client checks the balance, does the math, and tells the server "deposit $500". A cheater opens the client file, removes the balance check, and still sends "deposit $500" with $0. The server accepts it because it never checked for itself. Result -> free money and a broken economy.

Story B, the wrong cooldown

A /heal command has a 60-second cooldown, but the cooldown only exists on the client with os.time(). A cheater deletes that check and spams /heal. The server never stopped them, because the rule was on the wrong side.

Same mistake in both: the server let a rule live on the client. The client is the player; the player rewrote the rule.

Step 1

The four-bullet rubric

Outcome: a screenshot-worthy checklist for deciding who owns every piece of code you write.

Before you build any feature, ask these four questions:

  • Is it money, items, or persistent data? → server decides, period.
  • Is it a permission or role check? → server verifies, not client.
  • Is it a cooldown or rate limit? → server timestamps, not client.
  • Is it visual or cosmetic (particles, hud, camera, animations)? → client performs, on the server's say-so.

Three out of those four usually belong on the server. That is normal. Clients do visuals. Servers do rules.

Step 2

The five-step request/response pattern

Outcome: the shape every trust-critical feature will take from here on.

Every feature that touches money, items, or permissions follows the same shape:

  1. Client asks. "I want to heal." "I want to deposit 500."
  2. Server checks. Cooldown, permission, current state, input validity.
  3. Server decides. Yes or no.
  4. Server writes. The authoritative state, save the new cooldown, deduct the money.
  5. Server tells. The client(s) what visible effect to perform.
  6. Client performs. Plays the animation, updates the HUD, shows a notification.
Why this shape

Notice the split: the server handles the trusted parts, and the client handles the visual or input parts. That wall is not a problem. It is where your security lives.

In Module 04 you'll write this pattern as working code. Today you're learning the shape. The /heal build project (Lesson 19) is this pattern, every step accounted for.

Step 3

Validate everything from the client

Outcome: a four-point sanity pass the server runs on every incoming request.

When the client sends data to the server, like an amount, a target player, or an item name, the server should check four things:

  • Type: is amount a number? type(amount) == 'number'.
  • Range: is it positive? less than the player's real balance? under a sane maximum?
  • Authority: does this player have permission to do this action (ACE, role, job)?
  • Context: is the player near the NPC? enough time since the last request? in the right job?
Pro move

Treat client data like form input on a website. The server is your backend. The client is just a request. You do not trust the request. You validate it.

Recap 1

The trust audit

Read this deliberately broken script and find every trust violation. Do not scroll past the answer box until you've written your list.

The list of violations
  1. Balance lives on client. A cheater sets balance = 999999999 and the math passes.
  2. Cooldown lives on client. A cheater deletes the cooldown check.
  3. No permission check. Anyone can deposit.
  4. No server validation at all. The real balance (in the database) is never consulted.
  5. No source on the server. The server never sees which player did what.

Every single line of this script is cheater-friendly. The fix isn't to patch individual issues, it's to move the entire feature to the server and let the client do nothing except ask.

Step 4

source: the one thing clients can't fake

Outcome: you know where the server's trusted player identifier comes from.

When a client triggers a server event, FiveM's network layer attaches the player's server ID to that message. The server sees it as a magic variable named source. The client cannot change it mid-flight. It's assigned by the infrastructure, not passed by the script.

The trust anchor

source is the trusted identifier on the server. Use it, and only it, to look up the player's state, money, permissions, position. If you catch yourself passing player ID as an argument from the client, stop and rewrite.

Full mechanics and code examples live in Lesson 14. Today you know it exists and you trust it.

Step 4b

Cache source before any async work

Outcome: you never give the wrong player money, items, or punishment because source changed under your feet.

source is trustworthy, but it is also volatile. It is a global that FiveM refreshes at the entry of every event handler. The instant your code yields, with Wait, with an SQL await, with any callback, the server is free to run a different handler for a different player. When control returns to your function, source may now point to that player, not yours. Your "give 500 to source" line just paid the wrong person.

The async gotcha

Read this slowly. source is not a parameter that travels with your function. It is a shared variable the runtime overwrites between yields. If your handler crosses any await, Wait, or callback boundary, you must snapshot source into a local first.

The whole rule fits on a sticky note: first line of every server event handler, local src = source. From that point on, you use src and forget source exists. It is muscle memory you will be glad you trained early.

Why a local fixes it

A local variable lives in this function's stack frame. When the function yields, that local is preserved alongside it; when the function resumes, the local is exactly what you set it to. The global source belongs to the runtime, the local belongs to you. Same trick works for any other ambient global you do not want shifting under you (GetInvokingResource(), for example).

Step 4c

Player lifecycle: the cached src outlives the player

Outcome: you write async server code that does not error out when a player disconnects mid-flight.

Caching source solves the "wrong player" bug, but it opens a second one: the cached src is just a number. It does not know whether that player is still on the server. If the request started a long SQL write and the player rage-quit during it, your code might try to give items to a player who no longer exists.

Add a presence check after every yield where the player needs to still be online for the result to make sense:

Two presence checks worth knowing
  • GetPlayerName(src) returns nil if the player is gone. Cheap and good enough for most flows.
  • DoesPlayerExist(src) in newer builds. Same idea, slightly more explicit name.

Pick one and stay consistent. The lesson 24 cleanup pattern uses GetPlayerName so the rest of this course follows it.

The mirror of this rule is the cleanup side: when a player drops, your playerDropped handler also receives a source. Cache it the same way and use it to wipe their state, save their character, and delete their owned entities. We dig into the full cleanup chain in Lesson 24.

Step 5

What clients do own

Outcome: a clear mental line between "server's job" and "client's job", without the paranoia blurring your view of legitimate client work.

The division isn't "server is good, client is bad". Clients own plenty, all of it worth owning:

  • Visual feedback: particles, HUD, screen effects, cameras.
  • Input: key presses, chat, cursor position, NUI clicks.
  • Animation and movement of the local ped and owned vehicles.
  • Audio and sound: engine noise, music, notification sounds.
  • Local effects: the heal flash, the teleport fade, the job alert popup.

Clients run natives the server can't, PlayerPedId, DrawText, StartScreenEffect. This work has to live on the client because only the client can see the game. The wall isn't a cage; it's a division of labor.

Recap 2

The authority reflex

For each feature below, name the side that owns the logic. No peeking.

  1. Playing a medkit animation when a player heals.
  2. Deciding whether the player has enough cash to buy a vehicle.
  3. Showing a "+$500" floating text after a sale.
  4. Enforcing a 10-minute cooldown on a rob-store action.
  5. Drawing a red outline around a target NPC.
Answers
  1. Client: cosmetic, uses client natives.
  2. Server: money is persistent data; never trust the client's idea of the balance.
  3. Client: visual, after the server has already credited the money.
  4. Server: cooldowns must live server-side or they're theatre.
  5. Client: rendering only; the server doesn't even see the NPC.

If you got 5/5, the authority reflex is forming. If you missed one, re-read the rubric in Step 1.

Anti-patterns: shame them, name them, remember them

Client-side permission check

if IsPlayerAceAllowed(source, 'command.admin') then doAdminThing() end on the client. A cheater removes the if. Always ACE-check on the server.

Client-side cooldown

if lastUsed + 60 > os.time() then return end on the client. A cheater removes the line and spams the action. Always timestamp server-side, keyed by source.

Trusting client-reported numbers

The client says "I have $5000, let me transfer $500", server just does the math? No. Server queries the database or its in-memory state for the real balance, ignores the client's claim.

Using GetPlayerName as identity

Names aren't identity, they're display strings. Two players can have the same name; a player can change theirs. For permissions and persistence, use source plus the player's license identifier. Full treatment in Module 05.

Using source after an await or Wait

The handler called SQL, the SQL came back two ticks later, and the code wrote money to source without thinking. By then source is whoever fired the next event. First line of every server handler: local src = source. Then forget the global.

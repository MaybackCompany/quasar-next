Version control with GitHub Desktop

This is the lesson that saves your afternoon. You've got a working resource from Module 02. One careless edit, one corrupted file, one "I'll just try this", and it's gone. Git is the safety net. GitHub Desktop is the button-based way to start using git right now. This version of the lesson avoids fragile menu paths and sticks to the workflow that still works even when GitHub Desktop moves buttons around.

You'll build
A git repository for qu_hello, a .gitignore tuned for FiveM, two commits, a feature branch, and a published private repo on github.com.
Time
~35 minutes
You need
Your qu_hello resource from Module 02 and a free github.com account.
You'll learn
Install & sign-in · init repo · .gitignore for FiveM · first commit · branches · revert · publish · the daily rhythm

Before you start

Recommended placement

Take this lesson right after the daily dev loop. Git is the safety step before risky edits like handling files, SQL migrations, framework configs, and shipping changes to a live server.

How to read this lesson

You do not need to become a git expert here. Learn just four things: make a repo, make a commit, make a branch, and undo a bad change. That is enough to stop losing work.

The mental model: git is a camera for your folder

Forget every intimidating thing you've read about git. The mental model is trivial: git takes snapshots of your folder. Each snapshot is called a commit. You can jump back to any snapshot at any time. That's it. That's the whole thing.

Everything else git does, branches, merges, pushes, pulls, is plumbing on top of "snapshot your folder". The good news: GitHub Desktop hides the plumbing. You see the snapshots, you see what changed, you click a button to take a new one. The intimidating CLI is still there underneath, but you don't have to touch it until you want to.

Why every FiveM dev should use git

Scripting in FiveM is a lot of "change one line, restart the server, see if it still works." When it doesn't work, you want to know what you changed since the last time it did. That question is unanswerable without git. With git, it's one click. The safety isn't academic, it's the literal reason you'll stop being afraid to experiment.

Step 1

Install GitHub Desktop and sign in

Outcome: GitHub Desktop open, signed in, ready to receive your first repository.

  1. Go to desktop.github.com and download the installer for your OS (Windows or macOS).
  2. Run the installer. Accept the defaults.
  3. On first launch, it asks you to sign in with your GitHub account. Click Sign in to GitHub.com, your browser opens, authorise it, come back to the app.
  4. It asks for your name and email. Use whatever you want, these become the "author" on your commits.

You're in. The home screen shows three big buttons: Clone a repository, Create a new repository, Add an existing repository. The third is the one you want.

Step 2

Turn qu_hello into a repository

Outcome: GitHub Desktop tracking every file inside qu_hello, showing the initial set of changes ready to commit.

Use one of these two safe paths:

  1. Home screen path: click Add an existing repository from your hard drive.
  2. Menu path: click File -> Add local repository.

Browse to your resources/qu_hello folder. GitHub Desktop says something like "This directory does not appear to be a Git repository. Would you like to create one?" Click the create a repository link.

A dialog opens. Fill in:

  • Name: qu_hello (must match the folder name, and this is the convention for FiveM resources)
  • Description: anything, e.g. "my first FiveM resource"
  • Local path: the parent of qu_hello, already filled in
  • Initialize with README: tick it (auto-creates README.md)
  • Git Ignore: leave as None, we'll write a custom one in Step 3
  • License: leave as None for now

Click Create repository. You land on the main screen. On the left: a list of every file inside qu_hello, all with checkboxes, all ticked. These are the files about to be "snapshotted" into your first commit.

Community convention: one resource, one repo

The FiveM community convention is that each resource is its own git repository. Don't put every resource you write inside one mega-repo. One resource, one repo, one commit history. This is how you install community scripts too (clone the repo into resources/), so you're adopting the shape the whole ecosystem uses.

Step 3

Write a .gitignore file in the repo itself

Outcome: git ignores the files that should never be committed (editor junk, logs, secrets).

A .gitignore file lists patterns git will not track. It's the difference between a clean repo and one full of .DS_Store, editor config, and large streaming assets nobody wants in version control.

Do this in a way that always works:

  1. In GitHub Desktop, click Repository -> Open in Visual Studio Code or open the folder in your editor manually.
  2. Create a new file in the root of qu_hello called .gitignore.
  3. Paste this inside it and save.

Save. Back in GitHub Desktop, the changes list updates: the .gitignore file is now tracked, and anything matching its patterns disappears from the list.

Never commit an .env

Any file that holds secrets (API keys, database passwords, license tokens) must be in .gitignore before you ever commit. If you accidentally commit a secret, rotate it, even after you delete it from the repo, because the history still contains it. Cheap lesson to learn now, expensive to learn later.

Step 4

Your first commit

Outcome: a snapshot of qu_hello captured, with a readable commit message.

Look at the bottom-left of GitHub Desktop. Two fields:

  • Summary (required): a one-line description of the change. Keep it under 70 characters, in the imperative tense. Examples: "add /hello command", "fix runway 24 coordinates", "ignore node_modules".
  • Description (optional): longer prose if you want. For your first commit, skip it.

Write initial commit: /hello command and manifest. Click the big blue Commit to main button. That's a snapshot.

Now go to History (top-left tab). You see one entry: your commit, with your name, the timestamp, and every file that was added. Click any file to see what was captured. This view, "what did I change, in which commit" is the single most-used feature in git.

Quasar commit-message style

Use a type prefix: feat: new feature, fix: bug fix, refactor: code tidy-up, docs: docs only, chore: housekeeping (gitignore, configs). Example: feat: add cooldown to /heal command. One line, present tense, under 70 characters. Future-you reading the history will thank you.

Recap 1

Make a second commit the right way

Open server/main.lua. Add a single line: print('qu_hello loaded') at the top. Save.

Back in GitHub Desktop, the Changes tab shows server/main.lua with a green + next to the new line. On the right, a diff view: red lines for removed, green for added, white for unchanged. Exactly one green line should be visible.

Commit with the message chore: log when resource starts. Check History, two commits now. Click the new one, confirm only that one line was captured.

The diff shows more than one line changed, what's wrong?

Your editor probably normalised line endings or added a trailing newline on save. This is harmless but noisy. Long-term fix: add a .editorconfig to the repo. Short-term fix: just commit as-is, a trailing-newline change is not going to ruin anyone's day.

I don't see the file in Changes, what did I miss?

Either the file isn't saved (check your editor's dirty-dot indicator), or the file is outside the repository folder. The qu_hello folder is the boundary, anything you edit inside it shows up, anything outside doesn't.

Step 5

Branches: experiment without fear

Outcome: a separate branch called feature/cooldown where you can break things freely, then either merge it in or throw it away.

Here's the situation you'll hit in Module 06: you're about to add a cooldown to /heal, but you're not 100% sure your plan works. You don't want to mess up the working main branch while you experiment. The answer is a branch: a parallel line of commits that doesn't affect main until you explicitly merge it.

In GitHub Desktop: top bar, click Current branch: main. A dropdown, a text field. Type feature/cooldown, click Create new branch. You're now on the new branch. Make a change, any change, inside server/main.lua: add local lastHealAt = 0 at the top. Commit it with message feat: scaffold heal cooldown.

Now switch back to main (click the branch dropdown, pick main). Open server/main.lua. The line you just added is gone. That's not a bug, that's branches working. Your main is untouched. Switch back to feature/cooldown, the line returns.

The daily workflow

For every non-trivial change, the pattern is: (1) branch off main, (2) commit as you go, (3) when the change is proven, merge back to main, (4) delete the branch. In GitHub Desktop, merge is Branch → Merge into current branch, pick the source, click merge. Delete with Branch → Delete. Six clicks total. Once this is muscle memory, you'll branch for every experiment.

Step 6

The revert button: the real reason you need git

Outcome: you can restore any previous state of any file in one click.

Scenario: you've been coding for two hours, your /heal command broke somewhere, you can't figure out what. Without git, you sit there, add print statements everywhere, wonder which of the last six edits introduced the bug. With git, you ask one question: which commit was the last one where it worked?

Go to History. Find the commit where /heal last worked. Right-click it. Two useful options:

  • Revert changes in commit: undoes the changes that commit introduced (creating a new commit that does the opposite). Main history intact, bad change reversed.
  • View the diff first: click the commit and inspect the changed lines before you revert anything. This is the safest first move for beginners.

For the heal bug, you'd inspect the last commit, then revert it, test, and see if the bug is gone. If yes, you know the culprit was that commit's changes, and you have a clean diff of exactly which lines caused it. If not, move one commit older and inspect again.

For this course, use revert

Revert is the beginner-safe button. It keeps history clean and undoable. Destructive history tools exist, but you do not need them in this course. If something broke, inspect the diff, then revert.

Step 7

Publish to GitHub (private repo)

Outcome: your repo lives on github.com, syncs between machines, and is ready to share with the Quasar team when you want a review.

Right now your repo is local only. If your laptop's hard drive dies, your commits die with it. Publishing to GitHub takes 10 seconds.

In GitHub Desktop, top bar: Publish repository. A dialog opens:

  • Name: pre-filled, leave as qu_hello
  • Keep this code private: tick it (you can un-privatise later; don't publicise in-progress learning code)
  • Organisation: None for personal, or pick one if you have it

Click Publish Repository. A few seconds later it's up at github.com/<your-username>/qu_hello. Private, only you can see it. Any future commit you make, click Push origin to sync it up. Any commit you make on another machine, click Fetch origin then Pull origin to pull it down.

Why private for now

Your first FiveM code will have bugs, security holes, and embarrassing comments. That's fine, that's learning. Keep it private until the resource is actually useful. When you're ready, flip to public via Settings → Change repository visibility. Community conventions: open-source finished resources, keep works-in-progress private, never commit anything with secrets in the history.

The daily rhythm from here on

Every lesson in Modules 03 through 06 asks you to change the same resources. Your new workflow:

Before a big change
Create a branch. Branch → New branch → feature/<what-you're-adding>.
Every 10-15 minutes of working code
Commit with a clear message. Small commits are cheap; untangling a giant one is expensive.
When a change is done and tested
Merge the branch into main. Delete the feature branch.
When something breaks mysteriously
Open History. Revert the most recent commit. Test. Binary-search until the bug's culprit is identified.
End of every session
Push to GitHub. Your work is on a second machine now.

In Lesson 19, when you build the /heal command: branch, commit after the spec, commit after the server handler, commit after the client side, commit after tests pass. If the test breaks, you revert to the last green commit and you know exactly what you changed. In Lesson 20, your teleport menu's Config.lua grows one location at a time, each one a commit. In Lesson 21, your banking capstone's ten-item test matrix is ten commits. Beginners don't work this way; professionals can't imagine not working this way.

Cheat sheet: GitHub Desktop, the buttons you'll actually press

New repo from existing folder
File → Add local repository, then create a repository if the folder is not already a git repo.
Commit
Bottom-left pane. Summary + optional description. Commit to <branch>.
Branch
Top bar → Current branch → type name → Create new branch.
Switch branch
Top bar → Current branch → pick from list.
Merge
Branch → Merge into current branch, pick source branch, Merge.
Revert a commit
History tab -> inspect the diff -> right-click commit -> Revert changes in commit.
Publish to GitHub
Top bar → Publish repository. Tick Keep this code private for learning repos.
Sync
Top bar → Push origin (after local commit) or Fetch origin + Pull origin (before starting a session).
.gitignore patterns for FiveM
Editor cruft (.DS_Store, .vscode/), secrets (*.env), build output (node_modules/, dist/), large stream assets.
Commit message style
feat: / fix: / refactor: / docs: / chore:, under 70 chars, imperative tense.

What you learned

  • Git takes snapshots (commits) of your folder -- you can jump back to any snapshot at any time.
  • A .gitignore keeps secrets, editor junk, and large assets out of your repository.
  • Branches let you experiment without touching your working main branch.
  • The revert button undoes a bad commit safely, without destroying history.
  • Publishing to GitHub gives you a backup and lets you sync between machines.

Five-minute exercise

Open GitHub Desktop with your qu_hello repo. Create a new branch called feature/test-exercise. Add a comment to the top of server/main.lua: -- exercise branch test. Commit it. Switch back to main and confirm the comment is gone. Switch back to feature/test-exercise and confirm it reappears. Then delete the branch.

Expected output

Hint

Use the branch dropdown at the top of GitHub Desktop to create and switch branches. After committing on the feature branch, switching to main will make the file revert to the version without the comment. That is branches working as intended.

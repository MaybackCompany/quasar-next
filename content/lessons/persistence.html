Persistence: cache + write-back

Reading from MySQL on every action is slow. Writing on every action floods your DB and locks tables. The fix is the cache + write-back pattern: keep player state in RAM, mark it dirty when it changes, flush on a timer and on disconnect. This is how every well-built FiveM economy survives 64-player nights.

Before you start

Mental model: scratchpad on the desk, filing cabinet behind

You take notes on a scratchpad all day. At the end of the day you copy the totals into the filing cabinet. The scratchpad is fast (RAM). The filing cabinet is durable (DB). You never throw away the cabinet, but you also never run to it for every penny.

If the building catches fire (server crash) before you copy, you lose what is on the scratchpad. So you also flush periodically (every 30 seconds) and always on player disconnect, the equivalent of "copy when you stand up to leave the desk".

Why not write every change?

A player might earn cash 30 times a minute (each NPC kill, each delivery, each tip). 30 writes/minute × 64 players = 1920 writes/min on a single table. MySQL can handle it but your other resources will queue. Cache + write-back collapses this to 64 writes per 30 seconds at most.

Step 5

Persistence at scale: batching, WAL, sharded flush

Outcome: 64-player flush completes in <40 ms p95, no flush spike at the 30-second boundary, and a kill -9 mid-flush replays cleanly on next boot.

The Step 1–4 cache works on a quiet 32-slot server. It breaks at 64+ slots when the flush thread fires 64 separate UPDATE statements in the same tick: oxmysql queues them, the connection pool saturates, and player events stall for 200–400 ms while the queue drains. Step 5 turns that single-shot fire into a durable, smoothed pipeline.

5.1 · Batched upsert vs N updates

One INSERT ... ON DUPLICATE KEY UPDATE with a multi-row VALUES list replaces N updates. MySQL parses one statement, walks the index once per row, commits once. On a 64-row dirty set this drops total flush time from ~140 ms to ~22 ms.

The dirty flag resets only after the batched await succeeds. In production, collect the exact licenses you flushed, wrap the await with pcall, and leave those rows dirty with a retry timestamp if the write fails.

5.2 · Sharded flush window: smooth the spike

One flush every 30 seconds creates a 22 ms tick spike every 30 seconds. Better: split the cache into 6 shards by license hash, flush one shard every 5 seconds. The DB sees the same total throughput; the player sees zero detectable spike.

Worst case latency to disk for any mutation is still 30 seconds (six shards × 5 s) but the median is 15 s and the tick cost is split across six events instead of one.

5.3 · Write-ahead log for crash recovery

Sharded flush still loses up to 30 s of mutations on kill -9. A write-ahead log appends every mutation to a flat file before touching RAM; on boot, replay the file, then truncate it after a clean flush. This costs ~30 us per mutation (one fsync-batched append) and turns a 30-second loss window into a 0-second loss window.

The 200 ms append window batches WAL writes, a player earning $5 thirty times in 30 seconds appends one disk write, not thirty. The replay path applies as deltas (cash = cash + ?) so it survives a partial in-flight flush without double-counting.

Persistence bugs surface as either "I lost my money" or "the server stalled". Both are loud. Each row below is one outage shape.

Recap 3

Senior depth: durability

You know why N synchronous updates per tick destroy a server (lock contention, network round trips, log flushes) and why a batched upsert on a sharded flush window is the answer. You can sketch a write-ahead log that survives a crash mid-flush, and you know that an unrecoverable cache is a bug pretending to be an optimization. Persistence is not a feature; it is the contract.

Common mistakes

Writing on every mutation anyway

The whole point of the pattern is to coalesce writes. If AddCash calls MySQL.update.await internally, you have no cache; you have a slow proxy.

Forgetting onResourceStop

Without it, a graceful resource restart loses up to 30 seconds of unsaved data. Always handle it and use .await so the resource cannot exit until writes complete.

Two sources of truth

If another script reads cash directly from DB, you serve stale data. Expose GetCash via exports and force everyone to use it. The DB is write-only from outside the cache module.

Flushing the whole table every 30 seconds

Skip rows where dirty == false. Otherwise you write 64 rows every 30s even when nothing changed. With the dirty flag you write 0–3 rows on a quiet minute.

Not loading on playerJoining

If you load on playerSpawned, the player is already mid-spawn before their data exists. playerJoining fires before character selection, load there.

What you learned

  • RAM is the truth during a session; DB is the truth between sessions.
  • Dirty flags collapse hundreds of writes into one.
  • Three save points: periodic, on drop, on shutdown.
  • License is the only safe primary key for player data.
  • Other resources read via your exports, never via direct DB access.

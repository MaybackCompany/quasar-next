Player permissions in QBCore and ESX

By the end of this lesson you will understand how both major frameworks decide who is an admin, who is a regular player, and how to add or remove permissions without breaking anything.

You'll ship
A working permissions check that blocks a chat command unless the player has the right group in either QBCore or ESX.
Time
About 25 minutes if you focus.
You'll learn
How groups and permissions work in both frameworks, where they are stored, and how to add a new admin.
Prereqs
You have a server running QBCore or ESX. You know how to open the database or run a server command.
In 3 bullets

You will understand permissions in both frameworks like this:

  • QBCore uses the permissions table in the database. A player has a group like admin or god. Permissions are inherited: god has every power admin has plus more.
  • ESX uses Ace permissions built into FiveM itself. Groups map to Ace principals like group.admin. ESX functions check these aces behind xPlayer.getGroup().
  • Both frameworks expose a simple API: get the player, get their group, compare. The storage is different but the idea is the same.

Step by step

Lesson 06: Player permissions in QBCore and ESX

Start here

What this is
Permission systems decide who can run admin commands, open restricted menus, or spawn items.
Example: an admin can kick a player; a regular user cannot even see the command.
Where to find it
QBCore stores groups in the permissions database table. ESX uses FiveM's built-in Ace system.
Example: in QBCore, query SELECT * FROM permissions WHERE license = '...'.
How to use it
Get the player object, check their group, gate your command behind a group check.
Example: if xPlayer.group ~= 'admin' then return an error message end.
What to do now
Write one command that checks permissions in whichever framework your server uses.
Example: a /setjob command that only admins can use.

Outcome

You will write a server command protected by a permission check and confirm that non-admin players are blocked.

Why it matters

Every admin command you write must be gated. A /kick command without a permission check lets anyone kick anyone. Permissions are your first and most important security layer on the server side.

Vocabulary

Group
A named set of permissions. 'admin' is one group. 'user' is another. A player belongs to exactly one group.
Example: a player with group 'admin' can run admin commands. A 'user' cannot.
Ace
FiveM's built-in access control entry. A string like command.kick that can be allowed or denied.
Example: add_ace group.admin command.kick allow gives admins kick permission.
Principal
A target for Ace permissions. Can be a player identifier, a group, or a resource.
Example: player.license:abc123, group.admin, resource.qu_admin.

Minimal example

Expected output: An admin using /setjob 1 police sees a confirmation. A regular player sees "You do not have permission."

Checklist

  1. Create resources/[system]/qu_perms/ with fxmanifest.lua and server.lua.
  2. Paste the server.lua code above into your resource.
  3. Add ensure qu_perms after your framework's ensure line in server.cfg.
  4. Restart and join as a regular player. Try /setjob 1 police. Confirm it is blocked.
  5. Promote yourself to admin (see your framework's docs). Rejoin and try the command again.
  6. Confirm the command now works and prints the GRANTED log line.

Exercise

Add a second command /refreshperms that is gated to god group only. It should print a message to the console but otherwise do nothing. Confirm that an admin cannot use it, but god can.

Expected output: Admin player sees "You do not have permission." God player sees "Permissions refreshed." in chat.

Common failures

  • attempt to index a nil value (global 'QBCore') means qb-core is not started before qu_perms. Move ensure qu_perms after ensure qb-core in server.cfg.
  • xPlayer.getGroup() returns nil means the player was not loaded yet. ESX may need a Citizen.Wait(100) after the player spawns before their group is available.
  • All players are blocked, including admins means your group check logic is wrong. Print group to console to see what value it actually returns.
  • setJob is not a function means you called a framework function that does not exist in your version. Check your framework's documentation for the correct function name.

Refactor

Create a hasPermission(src, minGroup) function that compares a player's group against a hierarchy: user < admin < superadmin < god. Now every command only calls hasPermission(src, 'admin') and you change the hierarchy in one place.

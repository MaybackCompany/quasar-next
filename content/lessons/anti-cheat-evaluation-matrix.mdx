## The layers that actually stop cheaters

The order here is on purpose. Each layer catches a kind of attack the one above it cannot. A paid anti-cheat only earns its money once the first four are in place.

- **Server authority.** The client may ask. Only the server grants, and only after a check. This is the floor. A client that asks for money gets recomputed by the server, never trusted.
- **Rate limits.** Cap how often an action can happen. This blocks spam and automation. A real player robs a store every few minutes. A bot tries every frame.
- **Server-side invariants.** Reject impossible state changes, bad event arguments, out-of-range actions, and any request that does not follow the order the server expects. (An invariant is a rule that must always hold true. For example, you cannot collect a reward you were never given.)
- **Logging and review.** Catch patterns after the fact and ban with evidence, not vibes. In practice you record suspect requests on the server side: to the server console, the txAdmin Live Console, or a logging resource. Then a ban can point to a logged event instead of a hunch.
- **Anti-cheat product.** A layer for injected menus and known cheat signatures. It sits on top of all of the above. Never under it.

**Server authority is the floor. The product is the ceiling. Build from the bottom up.**

## How it works

The words above stay abstract until you know what each layer is fighting. Two threats get mixed up all the time, and the difference decides which layer handles them.

### Injected menus versus resource tampering

An **injected menu** (also called an external menu or mod menu) is a separate program the cheater runs next to the game. It hooks into the game client's memory and calls game functions directly, or fires fake network events at your server. Nothing on your server gave it permission. It lives entirely inside the player's machine, which the cheater controls completely.

**Resource tampering** is different. Here the cheater changes a resource your server actually sent them, or runs an extra client resource that talks to your scripts. The attack rides on top of real resources instead of bypassing them. An integrity layer is aimed at this case. It looks for client state that does not match what the server shipped. (As the next section notes, plain Lua cannot reliably hash every client file, so treat any product's integrity claims as something to verify, not assume.)

Knowing which is which tells you which layer does the work. Server authority and rate limits defend against both, because they live on the server and judge requests no matter where they came from. Integrity checks and a product anti-cheat are aimed mostly at the injected and tampered client.

### Why client-side detection can be faked

This is the mechanism behind never trust the client. The cheater owns the client process. Anything that runs there runs inside a sandbox the cheater controls. That includes a detection script that is supposed to report cheating. The cheater can patch it, feed it false data, or stop it from ever sending its report.

So a check that runs on the client and sends a "clean, nothing wrong" message to the server is worthless on its own. A cheater makes that message always say clean. The same check, run on the server against a request the client cannot forge, holds. The server decides the outcome, and the client cannot edit the server. That is the whole reason server authority sits at the bottom of the stack. It is the only place a check cannot be tampered out of existence.

### What the server can actually inspect

Do not promise that plain Lua can list a cheater's private injected code or prove every client file hash. Focus first on behavior and state the server can see and own:

- **Bad event arguments.** Reject wrong types, impossible quantities, entities the player does not own, and coordinates outside the allowed area.
- **Rate and sequence violations.** A purchase that arrives before the server opened a transaction, or hundreds of requests inside a real cooldown, is evidence the server can see.
- **Impossible state changes.** A reward request without the prerequisite the server recorded, or a value or position change outside the allowed bounds, should be rejected and logged.

None of these trust a clean-or-dirty self-report from the client. They check the request against state the server owns. A paid anti-cheat may add platform-specific detections on top. Evaluate those claims with evidence, and do not let them replace these controls.

### A worked example: scoring two products

Say you are choosing between two anti-cheats for an ESX server. Run both through the six scoring points and write down what you find.

| Scoring point | Product A | Product B |
|---|---|---|
| Server-side detection | Yes, checks events on the server | "Client sends a report" only |
| Active maintenance | Updated this month | Last update 14 months ago |
| Performance cost (resmon) | 0.2 ms idle | 1.8 ms idle, spikes higher |
| Transparency | Source is readable | Sealed blob, full access |
| Support and community | Active docs and Discord | One pinned sales post |
| Bypass history | Public bypass patched in days | Open bypass, unpatched |

Product A wins on every line that matters. Product B trusts the client, is stale, is heavy, is opaque, and stays bypassed. The price does not enter the table until two options are roughly tied on these six. **Score first, compare price second.**

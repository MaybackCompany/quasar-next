Inventory migration: QBCore, ESX, and ox_inventory

You want to switch your inventory system. Maybe from QBCore's default to ox_inventory, maybe from ESX's to a shared one. This lesson maps the data, writes the migration script, and catches every edge case so no player loses their items.

You'll ship
A migration script that moves every player's inventory from one system to another, preserving all items, metadata, and durability, with a rollback plan.
Time
About 60 minutes to write, test, and run the migration. 10 minutes to roll back if needed.
You'll learn
How each inventory system stores items in the database, how to map one schema to another, and how to run a migration without downtime.
Prereqs
You have a running server with an inventory system. You know how to run a SQL query.
In 3 bullets

Your migration script will do this:

  • Read every item from the old inventory table (qb-inventory, esx_inventory, or custom).
  • Transform each item into the ox_inventory schema: name, count, metadata as a JSON object, slot number.
  • Insert into ox_inventory's items table, preserving the original citizen ID and stash references.

Checklist

  1. Back up your database with mysqldump. Do not skip this.
  2. Install ox_inventory alongside your old inventory. Do not remove the old one yet.
  3. Write the migration script to read the old schema and insert into the new schema.
  4. Run the migration on a test database first, not production.
  5. Spot-check 5 random players — count items in old and new tables.
  6. Only after confirming counts match, swap the inventory resources.

Exercise

Run the migration, then write a verify command /countitems that prints the item count for the calling player from both the old and new tables. The numbers must match.

Expected output: [verify] old_table=34 items, new_table=34 items — MATCH for every player who runs it.

Common failures

  • json.decode failed — the old inventory column has a different JSON structure than you expected. Print one row first.
  • Duplicate entry for key 'PRIMARY' — the migration ran twice. Truncate ox_inventory and run once.
  • Metadata column is NULL — you passed nil to json.encode. Use item.info or {} as a default.
  • Weapons lost their ammo — your old system stored ammo in a separate column. You must merge it into metadata before inserting.

Refactor

Extract the item transform logic into transformItem(oldFormat) so you can add format-specific transforms (ESX JSON wrapper, QBCore metadata) without touching the insert loop.

Cheat sheet

Inventory schema map

  • qb-inventory. Items stored as JSON in players.inventory column. Each key is a slot number, value is {name, amount, info}.
  • ESX inventory. Separate user_inventory table with identifier, item, count. Metadata in a separate column or absent.
  • ox_inventory. One items table. Columns: owner, name, count, metadata (JSON), slot. Stashes use owner like stash_house_12.
  • Always back up first. mysqldump -u root -p your_database > backup_before_migration.sql

The big idea, in one minute

An inventory migration is a data shape change. The old system stores items as a big JSON blob in one column. The new system stores items as individual rows in a dedicated table. Your migration script is the bridge: it reads the blob, breaks it into pieces, and writes each piece as a new row. If you lose a piece, a player loses an item. Precision matters.

Think of moving from a filing cabinet to a digital database.

The filing cabinet (old inventory) has one folder per player, with all their items stuffed inside. The digital database (ox_inventory) has one row per item. Your migration script is the intern who opens every folder, takes out each item, photographs it, and creates a new database entry. If the intern skips a folder or mislabels a photo, that item is lost.

Run the script once. Verify the counts. Then throw away the filing cabinet.

Risky migration and safe migration, side by side

Most people run the migration on the live database, with players online, without a backup. Do not be most people.

Risky migration

Players are online. No backup. Script runs and crashes halfway through. Half the items are gone.

If the loop crashes on item #500 out of 10,000, items 1-499 are in the new table, items 500-10,000 are lost. The old table was never backed up. Players are furious.

Safe migration

Server offline. Backup made. Migration runs in a transaction. Verification query confirms counts before going live.

If the transaction fails, MySQL rolls back the DELETE. The old table is untouched. The backup exists as a second safety net. Players never see a missing item.

Before you start

If any box is unchecked, you are not ready to migrate.

Step 1

Inspect the old data

You cannot write a migration until you see the exact shape of the old data.

  1. Open your database and find the old inventory table.
  2. For qb-inventory: SELECT citizenid, LEFT(inventory, 200) FROM players LIMIT 3;.
  3. For ESX: SELECT identifier, item, count FROM user_inventory LIMIT 5;.
  4. Note the JSON structure. Does it use amount or count? Does metadata live under info or metadata?
  5. Write down every field name exactly as it appears. One typo in your transform function and zero items migrate.
Step 2

Write the migration script

Test on a copy of the database. Never run a migration script for the first time on production.

  1. Create resource qu_migrate with a manifest and server.lua.
  2. Write a SELECT that pulls all rows from the old table.
  3. Write a loop that transforms each old item into the new schema.
  4. Wrap the entire write in a MySQL transaction.
  5. After the INSERT loop, SELECT COUNT(*) from both tables and print them. They must match.
Step 3

Run and verify

The migration runs once. You verify twice. Then you switch.

  1. Stop your server or kick all players.
  2. Run ensure qu_migrate in the console. Then run migrate_inv.
  3. Watch the output. If you see any SQL errors, stop. Fix them. Restore the backup. Try again.
  4. Run SELECT COUNT(*) FROM old_inventory; and SELECT COUNT(*) FROM ox_inventory;. They must match.
  5. Spot-check 5 players. Pick random citizen IDs. Check their items manually in both tables.
  6. If everything matches, stop the old inventory resource. Start ox_inventory. Start the server.
Recap

Three verifications, in order

Do not skip any of these. Each one catches a different class of data loss.

  1. Verify 1, row counts match.
    • Action: SELECT COUNT(*) FROM old_table; and SELECT COUNT(*) FROM ox_inventory;.
    • Expected: the numbers are identical.
    • How to verify: if old has 12,345 rows and new has 12,344, one item was lost. Find it before going live.
  2. Verify 2, player spot-check.
    • Action: pick 5 random citizen IDs. Check their items in both tables.
    • Expected: same items, same counts, same metadata.
    • How to verify: old item "water" count 5 → new row name="water" count=5 metadata="{}".
  3. Verify 3, stash integrity.
    • Action: check one vehicle trunk and one house stash in both tables.
    • Expected: trunk contents and house contents are preserved exactly.
    • How to verify: old stash name "trunk_plateXYZ" → new owner "trunk_plateXYZ" with same items.
What if the counts do not match?

Restore the backup. Print the first 10 items that failed to insert. The most common cause is a field name mismatch. Your old JSON uses "amount" but your script reads "count". Fix the field name in the transform function and run the migration again.

The rules you just followed

  • Always back up the database before running a migration. No exceptions.
  • Take the server offline during the migration. Live data changes will corrupt the counts.
  • Run the entire migration inside a MySQL transaction. If it fails, nothing changes.
  • Verify row counts between old and new tables. They must match exactly.
  • Spot-check individual players' inventories. Automation misses edge cases; human eyes catch them.

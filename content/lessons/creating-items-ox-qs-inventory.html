Vocabulary

Items data file
The Lua table where all item definitions live. Every item in the server must be here.
Example: ox_inventory's data/items.lua contains hundreds of items.
Item use callback
A function that runs when the player clicks "Use" on the item in their inventory.
Example: a bandage item calls a heal function when used.
Metadata
Extra data attached to a specific instance of an item, like durability or a serial number.
Example: a weapon instance has a serial: "SN-0042".

Common failures

  • Item does not appear after adding: you must restart ox_inventory. Item data is loaded at resource start and cached in memory.
  • attempt to index a nil value (field 'golden_key'): the item name in the use callback must exactly match the key in the Items table, including case and underscores.
  • Item shows as "undefined" or blank label: the label field is missing or misspelled. ox_inventory requires label, not name.
  • Stack does not work: stack = false means one per slot. If you want multiple, set stack = true and check weight is reasonable.

Cheat sheet

Required item fields (ox_inventory)

  • label. Display name in the inventory UI. Human-readable.
  • weight. In grams. 1000 = 1 kg. Affects carry capacity.
  • stack. true = items of the same type stack in one slot. false = one per slot.
  • description. Shown in the item tooltip. Optional but recommended.

The big idea, in one minute

An inventory item is a definition plus a callback. The definition (in items.lua) tells the inventory UI what the item looks like, how much it weighs, and whether it stacks. The callback (in the server module) tells the server what happens when the player clicks "Use". The definition is loaded once when the resource starts. The callback runs every time a player uses the item. You cannot skip either one.

Think of a restaurant menu.

The menu (items.lua) lists every dish with its name, description, and price. The kitchen (use callback) is the actual recipe that runs when a customer orders. If a dish is on the menu but there is no recipe in the kitchen, the order fails. If the recipe exists but is not on the menu, nobody can order it.

Your item definition is the menu entry. Your use callback is the kitchen recipe. Both must exist for the item to work.

Bad way and good way, side by side

Most beginners add the item definition but forget the use callback, then wonder why "Use" does nothing. The good version adds both.

Don't do this

Adding only the item definition. The "Use" button appears but does nothing.

The inventory UI shows the item. The player can move it around. But the "Use" button exists and does nothing. The player clicks it and nothing happens — no feedback, no action, no error. Silent failure.

Do this

Add both the definition and the server-side use callback.

The item appears in inventory. The "Use" button triggers the server callback. The console prints a confirmation. The item is removed. The player sees the item disappear from their inventory — clear feedback.

Step 1

Locate the items data file

Every inventory system has one file that defines all items. Find it first.

  1. Open your server's resources/ folder.
  2. Navigate to ox_inventory/data/ (or qs-inventory/config/ for QS).
  3. Open items.lua. You will see a large Lua table starting with Items = { or Config.Items = {.
  4. Scroll through to see existing items. Notice the pattern: each item is a key-value pair inside the table.
  5. Make a backup copy of this file. If you break the table syntax, the entire inventory will fail to load.
Step 2

Add the item definition

Copy the structure of an existing item, paste it, and change the values.

  1. Find a simple existing item like ['water'] or ['bandage'].
  2. Copy its entire block, from the opening [ to the closing },.
  3. Paste it at the end of the Items table, just before the closing }.
  4. Change the key to ['golden_key'], the label to 'Golden Key', weight to 500.
  5. Add description = 'A mysterious golden key that glows faintly',. Save the file.
Step 3

Add the use callback (what happens on "Use")

Without a callback, the "Use" button does nothing. This step makes it work.

  1. Navigate to ox_inventory/modules/items/server.lua.
  2. Search for Item( to find existing use callbacks. You will see patterns like Item('water', function(source, ...).
  3. Copy an existing callback block and paste it at the end of the callbacks section.
  4. Change the item name to 'golden_key' and the body to print the expected line.
  5. Optionally add RemoveItem(source, 'golden_key', 1, false, slot) if the item should be consumed on use.
Step 4

Restart and give yourself the item

Item definitions only load on resource start. A restart is required.

  1. In the server console, type ensure ox_inventory (or restart via txAdmin).
  2. Watch the console for any Lua syntax errors. If you see unexpected symbol near ',', you have a missing or extra comma in items.lua.
  3. Once the resource loads without errors, join the server.
  4. Open chat and type /giveitem golden_key 1 (or use your framework's give command).
  5. Open your inventory. You should see the "Golden Key" item with its weight and description.
Step 5

Test the "Use" action

Click "Use" and watch the server console for the print line.

  1. With the golden key in your inventory, click on it to select it.
  2. Click the "Use" button in the inventory UI.
  3. Check the server console. You should see [QUIET][item] golden_key used by source=X slot=Y.
  4. If you added RemoveItem, confirm the item is gone from your inventory.
  5. If nothing happens, check that the item name in the callback matches exactly the key in items.lua.
Recap

Three tests, in order

Run each test and confirm the expected behavior. If something does not match, fix it before moving on.

  1. Test 1, item appears in inventory.
    • Action: type /giveitem golden_key 5, open inventory.
    • Expected: five golden keys visible with weight 0.5 each, description showing on hover.
    • How to verify: total weight increases by 2.5 kg. Items stack if stack=true, or occupy 5 slots if stack=false.
  2. Test 2, "Use" triggers the callback.
    • Action: click "Use" on one golden key.
    • Expected: server console prints [QUIET][item] golden_key used by source=1 slot=3.
    • How to verify: watch the server console. If RemoveItem is in the callback, the count drops from 5 to 4.
  3. Test 3, item data persists after restart.
    • Action: restart the server, join, check inventory.
    • Expected: the golden key items (if not consumed) are still in your inventory.
    • How to verify: the item data is stored in the database, not in items.lua. The definition file is reference only.
Why does the item survive a restart if it is removed from items.lua?

The items.lua file is a reference catalog — it tells the inventory what the item looks like and how it behaves. The actual item instances are stored in the database (in the ox_inventory table). If you remove the definition, the inventory will not know how to display existing items, but they will still exist in the database.

The rules you just followed

  • Every item needs a definition in items.lua. Without it, the inventory does not know the item exists.
  • Every usable item needs a server callback registered with Item('name', function(...) end).
  • The item name string must match exactly in both places — case, underscores, everything.
  • Weight is in grams. 500 = 0.5 kg. Stack controls whether multiples share a slot.
  • Item definitions are loaded at resource start. Restart the inventory resource after any change to items.lua.

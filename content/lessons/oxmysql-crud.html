Database with oxmysql: CRUD without the footguns

Almost every server you build will save state. Bans, money, garages, jobs, all of it lives in MySQL. oxmysql is the modern, async-friendly driver everyone uses. This lesson teaches the four query helpers, why prepared statements stop attackers cold, and how to size the connection pool so you do not deadlock at 32 players.

Mental model: oxmysql is a kitchen line cook with prepped ingredients

You walk up to the pass and call out an order. The cook has already chopped onions, cubed beef, and warmed the pan, that is a prepared statement. You hand over the protein (your parameter) and the dish lands on the pass. Fast, safe, repeatable.

String concatenation is the opposite. You hand the cook a sticky note saying "do whatever this paper says", and a customer sneaks '; DROP TABLE users; -- onto the note. That is SQL injection. Prepared statements never read the note as instructions; they treat it as an ingredient.

FiveM-specific note

oxmysql ships with both promise-style (:await()) and callback APIs. Pick promises and stay there. Mixing styles in the same file is the #1 source of "why is my query running twice".

Step 2

The four helpers you actually use

Outcome: you pick the right helper without thinking.

Each helper exists for a reason. Use the narrowest one that fits.

Why :await() instead of callbacks

Callbacks pyramid quickly. Three queries deep and you are reading sideways. :await() yields the current thread until the query resolves, so your code reads top-to-bottom. Just remember: :await() only works inside a thread (handler, CreateThread, etc.), never at file top-level.

Recap 1

You can read your own row

Insert one user with your license, then MySQL.single.await back by license. Print the row. If it comes back as nil, your WHERE clause does not match what you inserted.

Step 3b

Race conditions: the buy-gun double-spend

Outcome: you can write money and inventory writes that survive a player spamming the button.

Injection is the attack everyone knows. Race conditions are the bug that actually drains economies, and almost nobody writes their first money handler safely. Here is the classic. A gun costs $80. The player has $100. Your handler reads the cash, checks it, deducts, and gives the gun:

The player clicks twice in the same network frame, or a cheater fires the event ten times in a loop. Because :await() yields, a second copy of the handler starts running while the first is still waiting on the database. Both read cash = 100. Both pass the check. Both deduct. The account lands at -60 and the player owns two guns. There are three ways to make this safe; in production you stack them.

Fix 1: atomic conditional UPDATE (do this always)

Stop checking in Lua. Push the check into the WHERE clause so the deduction only fires when the money exists. The database serializes writes to a row, so exactly one of the two racing updates can match.

The first update drops cash to $20. The second update finds no row where cash >= 80, touches zero rows, returns affected = 0, and bails before handing out a second gun. This single change kills the dupe.

Fix 2: per-player Lua lock (belt and suspenders)

The atomic UPDATE protects the money column, but your handler might also touch inventory, logs, and webhooks, things the database cannot gate for you. A cheap in-memory lock stops a second copy of the handler from starting at all while the first is mid-flight.

Always release the lock

Every early return must clear busy[src] first, and you must clear it on playerDropped. A leaked lock means that player can never buy anything again until a restart. Wrap risky bodies in pcall if a query could error mid-handler.

Fix 3: framework money functions

If you are on Qbox or ESX, the framework already ships an atomic spend. Prefer it over hand-rolling the UPDATE, it logs the transaction and fires the right events for you.

Step 3c

Transactions: all-or-nothing across queries

Outcome: a multi-step write like a money transfer can never leave half-finished state.

The atomic UPDATE handles one row. But some operations need several writes that must all land or all fail. A player-to-player transfer is the textbook case: take money from A, give it to B, log it. If the server crashes between the take and the give, money vanishes into thin air.

Wrap the writes in MySQL.transaction.await. You hand it an array of { query, values } pairs. The database runs them as one unit: if any statement fails, every one of them is rolled back, as if none had run.

Why this beats three separate updates

Run those three queries one at a time and a crash between them leaves the books wrong: sender charged, receiver never paid. Inside a transaction the database guarantees the books always balance. Notice the sender update still carries AND cash >= ?, transactions give you atomicity, but you still want the conditional check so an overdraft rolls the whole transfer back.

Step 4

Pool sizing and the deadlock at 32 players

Outcome: your server does not freeze under load because every connection is in use.

Each :await() holds one MySQL connection while it runs. The default pool is small (often 5). At 32 concurrent players each running 3 queries on a hot tick, you run out of connections and every query queues. Symptom: server hitches every time you save.

Rule of thumb: connectionLimit = max(10, ceil(maxPlayers / 8)). Watch the slow-query log; if you see queries over 100ms, that query needs an index, not a bigger pool.

Common confusion

A bigger pool is not free. MariaDB has its own max_connections. If you raise connectionLimit on FiveM but leave MariaDB at 100 and you run three resources each at 50, the third resource cannot connect.

Recap 2

Slow query log is empty

Set mysql_slow_query_warning to 50, restart, play for two minutes, and check the console. Any query over 50ms gets logged. Add an index for the column it queries on, restart, confirm the warning is gone.

Step 5

Profile under real load

Outcome: you can prove a query is the bottleneck instead of guessing.

Three tools, each answers a different question. Run all three before you blame the database.

1. profiler record: what is the resource doing per tick?

Open the flamegraph and look for orange bars under onResourceStart or your event handlers. A blocking :await() shows up as a single tall bar, that is the tick the server stopped processing players.

2. resmon: who is hogging the tick budget?

In-game, type resmon 1. The server-side column shows ms-per-tick per resource. oxmysql itself should sit under 0.05ms. Your resource using it should sit under 0.30ms. Anything higher means a query is running on every tick instead of in response to events.

3. MariaDB EXPLAIN: will the query scale?

The two numbers that matter: type (want ref or const, never ALL) and rows (want a single digit, never the table size).

Step 6

JSON columns: how frameworks store player data

Outcome: you can read and write JSON columns, and query inside them without a full table scan.

Open any Qbox players table and you will find columns like charinfo, job, gang, and metadata that hold JSON, a whole nested structure crammed into one text cell. MySQL 5.7+ understands these natively. From Lua, you cross the boundary with the FiveM globals json.encode (Lua table to string) and json.decode (string back to Lua table).

You can also filter on a field inside the JSON with JSON_EXTRACT (or the ->> shorthand), which is how frameworks answer "give me every online cop":

JSON lookups do not use a normal index

A plain INDEX on the job column does nothing for WHERE JSON_EXTRACT(job, '$.name') = 'police', so EXPLAIN shows type=ALL and the query scans every row. If you filter on a JSON field often, either add a MariaDB generated column that pulls the value out and index that, or store the hot field (like job_name) in its own real column. Do not run JSON-extract filters on a hot path over a large table.

Common mistakes

Calling :await() at file top level

oxmysql may not be ready when your resource starts. Wrap the first query in AddEventHandler('onResourceStart', ...) or inside a CreateThread. Top-level await crashes the resource silently.

String concatenating a "trusted" value

The word trusted is a trap. Tomorrow another dev exposes the same code path to user input. Use placeholders for every value, every time. There is zero performance cost.

Using query when you wanted single or scalar

query always returns an array, even with one row. You will write rows[1].money when you could have written money. single for one row, scalar for one cell, query for a list.

Forgetting charset=utf8mb4

Without it, player names with emoji or non-Latin characters (Korean, Arabic, Russian) corrupt to ???? on insert. Always include the charset in your connection string and on the table itself.

Letting one heavy query block the tick

SELECT * FROM transactions ORDER BY id DESC on a 4M-row table holds a connection for seconds. Add LIMIT 100, paginate, and put an index on id. The pool is shared; one bad query starves everyone.

Cheat sheet

MySQL.query.await(sql, params)
Returns an array of rows. Default for SELECT.
MySQL.single.await(sql, params)
Returns the first row or nil.
MySQL.scalar.await(sql, params)
Returns the first column of the first row.
MySQL.insert.await(sql, params)
Returns the new auto-increment id.
MySQL.update.await(sql, params)
Returns affected row count.
MySQL.prepare.await
Hot-path helper; reuses the parsed plan.
Placeholders
Always ? in SQL, values in the second-arg array. Never ...
Pool size
connectionLimit in the connection string; rule of thumb ceil(maxPlayers/8).

What you learned

  • Pick the narrowest helper: scalar for one cell, single for one row, query for a list.
  • Every value goes through the parameter array. .. in SQL is a security hole.
  • :await() reads top-to-bottom but must run inside a thread or handler.
  • The connection pool is shared; size it for concurrent players, watch the slow-query log.
  • charset=utf8mb4 in the connection string saves your player names.

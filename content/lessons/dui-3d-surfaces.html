Why it matters

DUI surfaces power every in-game computer, phone screen, billboard, and interactive panel in FiveM. Without DUI, your server screens are static textures that cannot show live data.

Vocabulary

DUI
Dynamic UI. An off-screen Chromium browser instance that renders HTML, CSS, and JavaScript.
Example: a web page that increments a counter when you click it.
Render target
A texture created at runtime that the GPU can sample. DUI pixels go into a render target, then the render target is drawn in 3D.
Example: CreateRuntimeTextureFromDuiHandle makes this connection.
NUI callback
A message sent from JavaScript inside the DUI browser to Lua. Used to communicate clicks and data.
Example: SendNUIMessage in Lua sends a JSON object to the browser.

Common failures

  • attempt to call a nil value (global 'CreateDui') means the script is on the server. DUI is client-only.
  • Screen does not appear: check that ui_page 'html/screen.html' is in fxmanifest.lua. The DUI loads this file, and without it the browser has no content.
  • Screen is black/blank: the HTML file path in CreateDui must use the full https://cfx-nui-RESOURCENAME/path format. Relative paths do not work inside DUIs.
  • Clicks do nothing: SendDuiMouseDown requires a delay (Wait(50)) before SendDuiMouseUp or the click is too fast for Chromium to register.

Cheat sheet

DUI surface pipeline

  • CreateDui. Opens an invisible browser. Give it an HTML URL.
  • CreateRuntimeTextureFromDuiHandle. Copies browser pixels into a GPU texture.
  • DrawSpritePoly. Draws that texture at 3D coordinates every frame.
  • SendDuiMouseDown/Up. Sends clicks into the browser for interactivity.

The big idea, in one minute

A DUI surface is a three-step pipeline: browser → texture → 3D draw. First, FiveM opens an invisible Chromium browser (CreateDui) and loads your HTML into it. Second, the browser's pixels are copied into a runtime texture (CreateRuntimeTextureFromDuiHandle). Third, that texture is drawn as a 3D sprite every frame (DrawSpritePoly). The browser is invisible to the player, but the 3D sprite is not. Clicks go backward through the pipeline: player clicks the 3D sprite, Lua sends SendDuiMouseDown to the browser, and the browser's JavaScript responds.

Think of a projector and a screen.

The projector (DUI browser) plays a movie (HTML page) in a back room. A camera (render target) films the movie and sends it to a screen on stage (3D sprite). The audience (players) sees the screen, not the projector. When an audience member walks up and touches the screen, a sensor (mouse event) tells the projector to change the movie.

Your DUI surface is the screen on stage. The browser is the projector in the back room. The render target is the camera between them.

Bad way and good way, side by side

Most beginners create a new DUI every frame. The good version creates it once and reuses the texture.

Don't do this

Creating a DUI inside the draw loop leaks browser instances and crashes the client.

Each frame creates a new browser process. After 10 seconds (600 frames), 600 Chromium instances are running. The client crashes with an out-of-memory error.

Do this

Create the DUI and texture once, outside the loop. Reuse them every frame.

The DUI browser is created once and lives for the entire session. The draw loop only reads the existing texture. No memory leaks, no crashes.

Step 1

Create the resource with HTML

DUI needs an HTML file to display. Start there.

  1. Create resources/qu_dui_surface/ and inside it create fxmanifest.lua.
  2. In the manifest, add client_scripts { 'client.lua' } and ui_page 'html/screen.html'.
  3. Create the html/ subfolder and inside it create screen.html.
  4. Paste the HTML from the minimal example. Save. The HTML is ready.
  5. Add ensure qu_dui_surface to server.cfg.
Step 2

Create the DUI and runtime texture

The DUI browser and texture must be created at the top of the file, outside any loop.

  1. In client.lua, define screenPos, screenWidth, and screenHeight.
  2. Call CreateDui with the full https://cfx-nui-qu_dui_surface/html/screen.html URL.
  3. Call CreateRuntimeTxd('qu_dui_tex') to create a texture dictionary.
  4. Call CreateRuntimeTextureFromDuiHandle(txd, 'dui_tex', GetDuiHandle(dui)).
  5. These three lines run once when the resource starts. Do not put them in a loop.
Step 3

Draw the 3D surface every frame

Use DrawSpritePoly to place the texture at world coordinates.

  1. Create a Citizen.CreateThread with while true do ... Wait(0) end.
  2. Add a distance guard: if dist < 30.0 then so the screen only draws when nearby.
  3. Call DrawSpritePoly with the four corners of the rectangle: top-left, bottom-left, top-right, bottom-right.
  4. The texture dictionary name ('qu_dui_tex') and texture name ('dui_tex') must match what you created in Step 2.
  5. Join the server. Travel to LSIA. The HTML screen should be visible on the wall.
Step 4

Add mouse interaction

Send mouse events from Lua into the DUI browser so HTML buttons respond.

  1. Create a second thread that checks for IsControlJustPressed(0, 25) (right mouse button).
  2. Check if the player is within 5 meters of the screen using #(playerCoords - screenPos).
  3. If close enough, call SendDuiMouseDown(dui, 'left') followed by Wait(50) then SendDuiMouseUp(dui, 'left').
  4. Print a confirmation line to F8 so you know the click was sent.
  5. Right-click near the screen. The counter should increment on the HTML display.
Step 5

Verify everything works end-to-end

Walk up, look at the screen, click, and confirm the counter changes.

  1. Travel to LSIA airport. Look at the terminal wall for the dark screen.
  2. Confirm the counter starts at 0 and the red "Click Me" button is visible.
  3. Right-click while facing the screen. The counter should change to 1.
  4. Right-click three more times. The counter should reach 4.
  5. Walk 40 meters away. The screen should disappear (distance guard). Walk back. The screen reappears and the counter still shows 4.

    • Action: walk 40 meters away from the screen location.
    • Expected: the screen is no longer drawn. Walk back within 30 meters and it reappears.
    • How to verify: the counter value is preserved because the DUI browser keeps running even when the sprite is not drawn.
Why does the counter survive the distance guard?

The DUI browser is a separate process that keeps running as long as the resource is active. The distance guard only controls whether DrawSpritePoly is called — it does not pause or destroy the browser. When you walk back, you see the browser's current state, including any clicks you made earlier.

The rules you just followed

  • Create the DUI browser once. Never inside a loop. Each CreateDui spawns a Chromium process.
  • Create the runtime texture once, linked to the DUI handle. The texture updates automatically when the browser renders.
  • Draw the texture every frame with DrawSpritePoly using a distance guard.
  • Send mouse events with a delay (Wait(50)) between down and up so Chromium registers the click.
  • The DUI URL must use the https://cfx-nui-RESOURCENAME/ format. Relative paths fail silently.

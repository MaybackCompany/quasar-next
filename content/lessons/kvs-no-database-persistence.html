Start here

What this is
A built-in FiveM storage system that saves strings under named keys, surviving restarts.
Example: KVS stores "player:1:logins" = "42" on disk.
Where to find it
Available via GetResourceKvpString and SetResourceKvp on the server.
Example: these functions exist only in server-side Lua and C#.
How to use it
Set a value under a unique key, get it back by that key later.
Example: SetResourceKvp('my_key', 'hello world') then GetResourceKvpString('my_key').
What to do now
Save one value, restart the server, and confirm it survived.
Example: set a key, restart, read the key, print it.

Outcome

You will restart your server and watch a player's login count increase from where it left off, proving data persisted without a database.

Why it matters

Setting up MySQL for a tiny script is overkill. KVS handles small persistence needs — player settings, server configuration, cooldowns that survive crashes — with zero setup. Knowing the limit (64 KB per key, strings only) keeps you from building a banking system on it.

Vocabulary

KVP / KVS
Key-Value Pair / Key-Value Store. A simple storage where every piece of data is looked up by its unique name (the key).
Example: 'player:5:money' is the key, '1500' is the value.
Resource KVP
The KVS namespace tied to your resource. Keys are scoped to the resource that set them.
Example: a key set by qu_kvs_demo is not readable by qu_heal.
Flush
Writing cached KVS data to disk. FiveM does this automatically but you can trigger it manually.
Example: savings happen on a timer; crash before flush means data loss.

Common failures

  • attempt to call a nil value (global 'GetResourceKvpString') — you are running this on the client. KVS functions are server-only. Move to server.lua.
  • KVS returns empty string but I expected nil — GetResourceKvpString returns '' for missing keys, never nil. Use (val ~= '' and val) or defaultValue.
  • Data lost after crash — KVS flushes to disk periodically. If the server crashes within 60 seconds of a write, the data may be lost. Call SetResourceKvpNoSync for less critical data.
  • Key collision between resources — KVS keys are scoped per resource, but if you use generic keys like 'money', two resources can clash. Always prefix with your resource name.

Refactor

Cheat sheet

KVS limits and rules

  • 64 KB per key. Do not store large JSON blobs. Use MySQL for anything bigger.
  • Strings only. Convert numbers with tostring() and tonumber().
  • Server-only. The client cannot read or write KVS. Send data via events.
  • Resource-scoped. Keys set by qu_kvs_demo cannot be read by qu_other.

The big idea, in one minute

Not every script needs a database. A /heal cooldown that resets on restart is fine. But a player's lifetime login count? That should survive a restart. KVS fills the gap between "lost on restart" (in-memory variables) and "needs MySQL setup" (oxmysql). It is a text file managed by FiveM. You write a string under a key. You read it back later. It is that simple.

Think of a sticky note on the fridge.

In-memory variables are like telling someone a number — gone when they leave the room. MySQL is a filing cabinet with labelled folders. KVS is a sticky note. Quick to write, survives a power outage, but you would not store a novel on it. Use it for the right size of data.

The whole point of KVS is surviving a restart. Test it.

Recap

Two tests, in order

Run each test and confirm before trusting KVS for real data.

  1. Test 1, basic read/write.
    • Action: join, disconnect, rejoin.
    • Expected: login count increments each time.
    • How to verify: console shows increasing numbers, not resetting to 1.
  2. Test 2, restart survival.
    • Action: note the current count, restart the server, join again.
    • Expected: count is one higher than before the restart, not back to 1.
    • How to verify: the number after restart is higher than before restart.
What happens if two players join at the exact same time?

Each player has a unique source value, so they write to different keys (player:1:logins vs player:2:logins). No conflict. If two scripts write to the same key simultaneously, the last write wins. KVS is not transactional.

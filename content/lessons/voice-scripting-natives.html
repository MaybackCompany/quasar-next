What this is
The set of FiveM natives that control proximity voice chat — who hears whom and at what distance.
Example: NetworkSetTalkerProximity(15.0) makes your voice audible up to 15 meters.
Where to find it
Voice natives are client-side. Server commands use client-triggered events.
Example: MumbleSetPlayerMuted runs on the client, triggered by a server event.
How to use it
Set proximity on the client. Mute on the client. The server says who and when.
Example: server decides "player 3 is muted", sends event, client calls the mute native.
What to do now
Build a proximity switcher with three ranges and one mute command.
Example: press F1 to cycle whisper → normal → shout, see range in a notification.

Outcome

You will press a key to change your voice range and see a notification confirming the new range. An admin will type /voicemute [id] and the target player will be silenced for everyone immediately.

Why it matters

Voice is the most-used feature on roleplay servers. A broken voice system kills immersion faster than any other bug. Knowing the natives means you can debug voice issues, add custom ranges, and understand why your pma-voice errors say "proximity" in the stack trace.

Vocabulary

Proximity
The distance in meters at which your voice is audible to other players.
Example: NetworkSetTalkerProximity(5.0) = whisper range.
Mumble
The open-source voice protocol FiveM uses under the hood. Natives that start with "Mumble" control its settings.
Example: MumbleIsPlayerMuted(playerServerId) returns true or false.
Voice channel
A numeric channel (0-65535) that groups players. By default everyone is on channel 0.
Example: NetworkSetVoiceChannel(1) puts you in a private channel.

Common failures

  • attempt to call a nil value (global 'NetworkSetTalkerProximity') — this is a client native. You are calling it from server.lua. Move to client.lua.
  • Voice range does not change — you have a third-party voice resource like pma-voice or mumble-voip installed. These override the native proximity settings. Disable them or use their exports instead.
  • Mute command seems to work but player still hears — MumbleSetPlayerMuted must run on the listener's client, not the speaker's client. If A mutes B, the mute command must run on A's machine.
  • Voice does not work at all — check voice_enable 1 in server.cfg and ensure no firewall blocks UDP port 30120 (or your configured voice port).

Refactor

Replace the static RANGES table with a KVS-backed per-player preference, so each player's last-used range is restored when they reconnect.

Cheat sheet

Voice natives at a glance

  • NetworkSetTalkerProximity(value) — how far your voice carries (in meters). Client-side.
  • NetworkSetVoiceChannel(channel) — put players on a private channel (0-65535). Client-side.
  • MumbleSetPlayerMuted(serverId, toggle) — mute or unmute a specific player for the caller. Client-side.
  • MumbleIsPlayerMuted(serverId) — check if a player is currently muted. Client-side.

The big idea, in one minute

Voice in FiveM works through Mumble, an open-source VoIP protocol. Every player has a proximity radius. When two players' radii overlap, they hear each other. The server does not process voice audio — it only tells each client who is in range. The audio travels peer-to-peer. This is why voice commands (mute, proximity) must run on the client: the client is the one doing the listening. The server acts as a coordinator, not an audio mixer.

Think of a crowded room.

Each person has a "hearing bubble" around them. You can hear everyone inside your bubble. If you put on earplugs (mute), you hear no one. If someone shouts (megaphone range), their bubble is bigger. The room itself does not process sound — it just tells you who is nearby. FiveM voice works the same way: the server says "player 2 is 10 meters away", and your client decides if that is close enough to hear.

Step 1

Create the resource and proximity cycle

The client is where proximity lives. Start with the F1 cycle.

  1. Create resources/qu_voice/ with fxmanifest.lua, client.lua, and server.lua.
  2. The manifest needs client_script 'client.lua' and server_script 'server.lua'.
  3. Paste the proximity cycle code from the Minimal Example into client.lua.
  4. Ensure the resource. Join the server. Press F1. You should see "Voice range: normal (15m)".
  5. Press F1 again. It cycles to "shout (30m)". Press again. "whisper (5m)".
Step 2

Test proximity with a friend

The proof is walking away until the voice cuts out.

  1. Have a friend join the server and stand next to you.
  2. Both set range to "normal (15m)".
  3. Talk. You hear each other. Now walk away from each other.
  4. At around 15 meters, the voice fades and then cuts out completely.
  5. Both switch to "shout (30m)". You can now hear each other from much farther.
Step 3

Build the server mute command

The server decides who gets muted. The client executes the mute.

  1. In server.lua, add: 
    RegisterCommand('voicemute', function(source, args)
    local targetId = tonumber(args[1])
    if not targetId then return end
    -- Send mute to ALL clients so everyone mutes the target
    TriggerClientEvent('qu:voice:mute', -1, targetId)
    print(('[qu_voice] server: muted %d (broadcast to all clients)'):format(targetId))
end, true) -- restricted = true, admin only

RegisterCommand('voiceunmute', function(source, args)
    local targetId = tonumber(args[1])
    if not targetId then return end
    TriggerClientEvent('qu:voice:unmute', -1, targetId)
    print(('[qu_voice] server: unmuted %d'):format(targetId))
end, true)
  2. -1 as the target sends the event to all clients, so everyone mutes the player.
  3. Add the qu:voice:mute and qu:voice:unmute handlers to client.lua as shown in the Minimal Example.
Step 4

Test the mute

Two players, one admin command, immediate silence.

  1. Have two players in the server: player A (admin) and player B (target).
  2. Player B talks. Player A can hear them.
  3. Player A types /voicemute B'sServerId in console or chat.
  4. Player B's voice immediately goes silent for Player A.
  5. Type /voiceunmute B'sServerId. Voice returns.
Recap

Two tests, in order

Run each test and confirm before adding voice features to a production server.

  1. Test 1, proximity ranges.
    • Action: set to whisper (5m), have a friend stand at 3m, then 6m, then 12m.
    • Expected: audible at 3m, very quiet or silent at 6m, silent at 12m.
    • How to verify: the cutoff point matches the configured distance.
  2. Test 2, mute broadcast.
    • Action: admin mutes player B. Player C (third player) and admin both listen to player B.
    • Expected: neither admin nor player C can hear player B.
    • How to verify: player B's mic indicator still moves, but no one hears them.
Why use -1 (all clients) for the mute event instead of just the admin's client?

If you only mute player B on the admin's client, the admin stops hearing B. But player C still hears B. That is a personal mute, not a server mute. Sending the event to all clients (-1) ensures everyone stops hearing the muted player. For a personal mute (one player mutes another), you would send to just the requesting player's source.

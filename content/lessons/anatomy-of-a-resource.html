Anatomy of a resource

A FiveM resource is just a folder with a label card. That label card is fxmanifest.lua. It tells FiveM what this folder is, which files belong to it, and which side should run them. This lesson teaches you to read that card line by line without guessing.

The mental model: what a resource is

Strip away every buzzword and a FiveM resource is a folder with a short instruction card inside it. The folder can live anywhere under resources/. What makes it a real resource is one file named exactly fxmanifest.lua.

That manifest is not your gameplay logic. It is your description page. It says: this is my resource, this is the game, these are my files, and this is which side should run them. FiveM reads that description first, then loads your real scripts.

That is why this lesson matters. If you can read a manifest calmly, the rest of a resource stops feeling like magic and starts feeling like named pieces in the right boxes.

FiveM note

Two consoles exist in FiveM: the server console (the window where you type ensure) and the client F8 console (inside the game, per player). Server scripts log to the server console. Client scripts log to F8. Keep both visible this lesson, you'll use them to prove what ran where.

Step 1

Understand the six core fields

Outcome: you can read a manifest and name every line without guessing.

Here's a manifest with every field you'll meet in this lesson. Read it once, then we'll walk each line.

What every line does
  • fx_version 'cerulean': the manifest format version. Think of it like "Lua for manifests, v3". cerulean is current (2024+). Its predecessors were adamant and bodacious; you'll only see those in old resources. Use cerulean on everything new.
  • game 'gta5': which CitizenFX game this resource targets. 'gta5' for FiveM, 'rdr3' for RedM, or a table like { 'gta5', 'rdr3' } for both. We're on gta5 for the whole course.
  • author, description, version: metadata. Shown in txAdmin and the server console. Harmless to skip, but please don't, future-you will thank present-you.
  • shared_script / client_script / server_script: the three script entries. Each one points to a .lua file (or a list or a glob) and says "run this on this side". Details in the next step.
  • files { ... }: non-script assets: HTML, CSS, JS, images, JSON configs. Anything that isn't a .lua you want executed goes here.
  • dependency: "don't start me unless that resource is already up". The server enforces the order.
About lua54 'yes' and __resource.lua (legacy)

You will see lua54 'yes' in nearly every old fxmanifest. As of June 2025 the Lua 5.3 runtime was removed; every resource now runs on Lua 5.4 by default. The lua54 directive still parses but is officially marked deprecated, the official docs no longer ask you to add it. Don't add it to new resources. If you inherit a codebase full of it, leave it for now, it's harmless, just stop typing it on new files.

You may also see a file named __resource.lua in very old resources. That was the manifest format before fxmanifest.lua existed. It is fully deprecated, current FiveM still loads it but every new resource uses fxmanifest.lua. If you ever migrate an old resource: rename the file, swap the manifest header to fx_version 'cerulean' + game 'gta5', and you're done.

Tip

The order of fields doesn't matter to FiveM, but it matters to humans. Put the version/game/metadata at the top, scripts in the middle, files and dependencies at the bottom. Every manifest you read from a serious studio follows roughly that shape.

Step 2

The three script entries

Outcome: you can decide, for any file you write, which side it belongs on.

FiveM runs your code in two places: the server process (the machine hosting the game) and the client process (each connected player's PC). The three script entries tell FiveM which side to load a file on.

  • server_script 'server.lua': runs on the server process. Database queries, money, bans, authority checks, server.
  • client_script 'client.lua': runs on every connected player. UI, animations, the world around the player, keypresses, client.
  • shared_script 'shared.lua': runs on both. Use it for pure Lua definitions with no FiveM-specific natives: shared constants, config tables, helper functions that work the same on either side.

Each entry accepts three shapes. Learn all three, you'll see every one of them in the wild:

A single string for a single file. A table for a fixed list. A glob (notice the plural client_scripts) to sweep in every .lua under a folder. The glob form is lovely for large resources where files come and go.

Load order matters

Files inside a script entry load in the order they're listed. If menu.lua calls a function defined in main.lua, then main.lua has to be listed first. Globs sort alphabetically, so don't rely on them when order is load-bearing.

One load-order rule worth memorising: shared goes first

There is a second kind of order on top of file order. Across the three entries, FiveM always loads shared_scripts before client_scripts and server_scripts. That ordering is the whole reason the shared layer is useful: by the time your client or server file runs, anything defined in shared (a Config table, a constant, a helper) already exists. So a good habit is to put your shared and config entries near the top of the manifest, then client and server below them. You will lean on this exact guarantee in the next lesson when both sides read one shared Config table.

Recap 1

Read the manifest out loud

Before you move on, answer these without scrolling up:

What's the difference between server_script and client_script?

server_script runs on the server process (the host machine). client_script runs on each connected player's PC. Same manifest, two different places, two different consoles.

What runtime does my code run on if I don't add lua54 'yes'?

Lua 5.4. As of June 2025 the old Lua 5.3 runtime was removed and every resource runs on 5.4 by default. The lua54 directive is officially deprecated, leave it off new resources.

If a manifest has no fx_version, what happens?

The resource refuses to load. The server console will print an error asking for a manifest version. FiveM needs to know which format rules to apply before it reads anything else.

SHARED_VERSION has no local in front of it, which makes it a global: reachable from any file on the same side. Module 01 told you to always prefer local, and that's still the right default. For one constant shared across a tiny resource a global is fine; a later Module 02 lesson shows you how to return a table or expose a proper pattern for bigger projects.

Files, dependencies, and the @ import

You won't use these today, but you'll meet them fast. A preview so the syntax doesn't surprise you.

files: non-script assets

Anything that isn't a .lua you want executed goes in files: HTML pages for a NUI menu, CSS, JavaScript, images, audio, JSON configs. Without this entry, FiveM won't even serve the file to clients.

dependency: load order

A dependency is another resource your code needs in order to work. If your script calls exports.oxmysql:execute(...), then the resource oxmysql must already be running by the time your first line executes. If oxmysql isn't up yet, that export doesn't exist and your script crashes on the very first query.

Two places control this, and you need both:

  • fxmanifest.lua declares the dependency. The server reads this and refuses to start your resource if the named one is missing or has failed.
  • server.cfg controls the boot order. FiveM starts resources top to bottom. The dependency must be ensured before the dependant, otherwise yours boots first, hits the missing export, and dies.

If you get the order wrong, the console tells you exactly what happened:

Both errors mean the same thing: the resource you're calling into isn't ready. Either it isn't ensured at all, it's listed below yours, or it failed to start (check its own console output for a red error and fix that one first).

Manifest vs cfg, what each one does

The manifest says I refuse to run without this. The cfg says start this one first. You need both. Declaring the dependency without ordering the cfg leaves a race; ordering the cfg without declaring the dependency means a future dev removes the ensure line and your resource breaks silently.

ui_page: the entry HTML for a NUI menu

If your resource has an in-game browser UI (a phone, an inventory, a shop), the manifest has to point FiveM at the entry HTML file. ui_page does that. The page itself, plus any CSS, JS, images, and fonts it loads, must also be listed in files, otherwise the client can't fetch them and the menu shows up blank.

Two rules: ui_page takes one string (one entry point per resource), and every asset the page references has to appear in files. We rebuild a NUI menu from scratch in Module 06, this lesson just teaches you to spot the line.

exports: functions you let other resources call

An export is a function your resource publishes. Other resources call it with exports.your_resource:functionName(args). The manifest declares which functions are exported on which side: exports for client-side, server_exports for server-side.

Module 03 covers exports in depth (when to pick an export over an event, the cost of the call, the trust rule). For today: recognize exports and server_exports in the manifest, and know they're how resources hand each other named functions.

The @ prefix: importing from another resource

Prefix a path with @name/ to reference a file that lives inside another resource. The other resource has to be started (and typically listed as a dependency), and then its file loads into yours as if it were your own.

You'll use this exact pattern in Module 05 when we pull in a real library, and again in Module 04 when we wire oxmysql. For now, just recognize the syntax when you see it.

Two of these you will type by hand on almost every real resource, so it helps to know what they hand you:

'@oxmysql/lib/MySQL.lua'
Loads the database wrapper. After this line you can call MySQL.query, MySQL.update, and friends.
Goes in server_scripts, because the database is server-only.
'@ox_lib/init.lua'
Loads the ox_lib toolkit so you can call lib.notify, lib.callback, and the rest.
Goes in shared_scripts so both sides can use it.
Forget the @ line and you get a confusing error

The import line is what actually creates those globals. Skip it and call MySQL.query(...) anyway, and the console throws attempt to index a nil value (global 'MySQL'). That error reads like your code is broken, but it almost always means one thing: you forgot the matching '@oxmysql/lib/MySQL.lua' line in your manifest. Same story for lib and the '@ox_lib/init.lua' import. When a library global is suddenly nil, check the manifest before you touch the code.

Recap 2

Decide the side

For each file below, answer in your head: does it belong in server_script, client_script, shared_script, or files?

A config.lua defining a list of vehicle prices used by both sides.

shared_script. It's pure data with no natives, and both sides need it.

A database.lua that runs SQL queries against your MariaDB.

server_script. Databases never touch clients, clients can't be trusted with credentials or direct queries.

A hud.lua drawing speed and fuel on the player's screen.

client_script. Drawing only makes sense on the player's PC.

A ui.html file for a NUI menu.

files. It isn't Lua; it's a browser asset. The manifest ships it to the client's game window.

Common mistakes & what the console tells you

Filename typo, FXManifest.lua / fx_manifest.lua / fxmanifest.Lua

The filename is fxmanifest.lua. One word. All lowercase. No underscore, no capitals, nothing creative. If the name is wrong, FiveM doesn't even recognize the folder as a resource and refuses to load it, silently. Worst trap in the course because there's no error.

Error: could not find a fx_version in manifest (or similar)

You forgot the fx_version 'cerulean' line. Every manifest needs it, it tells FiveM which format rules to apply. Add it as the first line and restart.

Error: could not load script client.lua

You listed a file in the manifest that doesn't exist on disk. Check the filename, check the folder, check for a typo. The path in the manifest is relative to the resource folder, 'client/main.lua' means qu_hello/client/main.lua.

Things run, but natives fail or log to the wrong console

You put a server-only file in client_script (or vice versa). Drawing on screen from a server_script does nothing. Running a database query from a client_script either errors out or, worse, exposes logic the client shouldn't see. When something "kind of works but weird", always ask: am I on the right side?

What you learned

  • fxmanifest.lua is the label card every resource needs -- without it, FiveM ignores the folder entirely.
  • fx_version 'cerulean' and game 'gta5' are the two required header fields. The old lua54 'yes' directive is deprecated, every resource runs Lua 5.4 by default since June 2025.
  • server_script, client_script, and shared_script tell FiveM which side should run each file.
  • files ships non-Lua assets (HTML, CSS, JSON) to the client, and dependency enforces load order between resources.
  • The @name/file.lua prefix lets you import a file from another resource into yours.

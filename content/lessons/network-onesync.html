OneSync & routing buckets

OneSync is the system that lets a server hold authority over entities across many clients. Routing buckets are the partition system that lets two players stand on the same coordinates without seeing each other. Together they unlock interiors, instances, and minigame arenas without trickery.

Before you start

Mental model: routing buckets are floors of a building

Imagine a 30-storey hotel. Floor 7 has the heist crew. Floor 8 has another crew running their own heist. Both crews stand at "coordinates 0,0" of their floor. They never bump into each other because the floor (the bucket) keeps them separate.

OneSync is the building's central control. It tracks every guest, decides who hears the elevator chime, and lets the server move people between floors without telling anyone else.

OneSync is mandatory

OneSync is enabled by default on Cfx.re. Without it you cannot see more than 32 other players, you cannot create server-side entities, and routing buckets do nothing. Always set onesync on in server.cfg.

Step 1

Enable OneSync (server.cfg)

Outcome: onesync shows infinity in status.

Restart. In the server console run onesync, it should print OneSync is enabled (infinity). If it prints off, your provider's panel may override the cfx flag; ask support.

Step 2

Server-side entities

Outcome: a vehicle that exists before any client sees it.

Server entity = scoped by distance

An entity created with CreateVehicleServerSetter has no owner until a client comes near. Once a client is in scope, OneSync assigns them as owner. They control physics, anim, AI, the server still has authority over creation/deletion.

Recap 1

Spawn a server vehicle, walk to it

Add a /qu_loot command (server side) that spawns a server vehicle 30 m in front of the calling player. Drive towards it. Confirm via console that NetworkGetEntityOwner returns your source ID once you're in scope.

Step 3

Routing buckets: instance the world

Outcome: a 4-player party in bucket 7 that no one else can see.

Bucket 0 is the public server. Anything 1+ is private. Players in different buckets exist on different "channels", entities, voice (with pma-voice integration), and chat respect the bucket.

Outcome: you pick the right mode for your use case.

Step 5

Profile OneSync at scale: ownership, scope, and the bucket lifecycle

Outcome: you can spawn 200 server entities across 16 buckets at 64 players without ownership thrash, replication storms, or stuck players.

5.1 · Ownership migration is the silent cost

OneSync gives every entity an owner, the client that runs its physics, animation, and AI. Default entity scope distance is ~424 m. When two clients are within that radius of an entity, OneSync picks one as owner. Move a third client closer and ownership migrates. Each migration is a network round-trip: tell old owner to release, tell new owner to claim, broadcast the new owner to everyone in scope.

If the same entity migrates more than 2-3 times per minute under normal load, owners are crossing the scope edge constantly. Either widen scope (set onesync_distanceCullVehicles false for vehicles you care about) or pin the owner with NetworkSetNetworkIdCanMigrate(netId, false).

5.2 · Server-spawn vs client-spawn: pick by intent

The two options are not equivalent. CreateVehicleServerSetter creates an entity that exists in the server's authority space first; ownership is assigned to a client only when one comes into scope. CreateVehicle on the client creates a local entity, then asks the server to register a network ID. The cost shape is different.

The "owner = -1" trap

Right after CreateVehicleServerSetter returns, NetworkGetEntityOwner(v) often returns -1. That is not a bug, no client is in scope yet. Wait one tick, or wait for a client to come within 424 m, before assuming ownership exists.

5.3 · Bucket session lifecycle: registry, disconnect, restart

The single most common production bug with routing buckets is stuck players, someone disconnects from bucket 7 mid-heist, reconnects, lands in bucket 7 (which is now empty), and reports the server is "broken". Solve it with a registry that owns the player→bucket map and clears on every exit path.

This pattern guarantees: every exit path (clean end, disconnect, resource restart) returns the player to bucket 0. Combine it with a small DB write on bucket entry/exit to recover state across full server crashes.

Pro tip: sweep at startup

Add an onResourceStart handler that iterates every connected player and forces them to bucket 0. Catches every "stuck after restart" bug at the cost of one bucket reset per player on resource reload, cheap insurance.

Recap 3

Senior depth: bucket lifecycle

Routing buckets are cheap to create and brutal when leaked: ownership migrates, but the bucket id is sticky to the entity owner. You know when to spawn server-side (shared world, deferred ownership) vs client-side (one-player instance), and you know how to rebucket on disconnect, resource restart, and onResourceStart sweep. If a player reports “I see no one,” you can answer in three lines of GetPlayerRoutingBucket.

Common mistakes

Calling SetPlayerRoutingBucket on the client

It is server-side only. The client cannot move itself between buckets, that would defeat the purpose. Always trigger from server.

Forgetting to send the player back to bucket 0

If a player disconnects from bucket 7 and reconnects without your code resetting them, they reload into 7 (now empty). Always reset on heist end and on playerDropped.

Spawning entities before moving the player

If you create an entity in bucket 0 and then move the player to bucket 7, the player will not see it. Move first, then spawn.

Treating client entity handle as portable

The number you get on the client is local. Use NetworkGetNetworkIdFromEntity to get the netId, send that to the server, then NetworkGetEntityFromNetworkId on the other side.

Forgetting onesync_population

If population is disabled globally, server-side AI behaves oddly. Default to true unless you specifically want a barren server.

What you learned

  • OneSync is mandatory and unlocks server-authoritative entities and large player counts.
  • Server entities exist before any client owns them; ownership transfers as players come into scope.
  • Routing buckets partition the world; same coordinates, different floors.
  • Strict lockdown keeps entities scoped to a bucket; relaxed lets them bleed across.
  • Always reset bucket on disconnect, end of session, and ad-hoc admin commands.

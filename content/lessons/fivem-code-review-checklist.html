Code review checklist before shipping FiveM resources

By the end of this lesson you will have a code review checklist that catches the 12 most common FiveM scripting mistakes before they reach your players, saving hours of emergency patches and lost trust.

You'll ship
A 12-point code review checklist in a REVIEW.md file, plus a Lua script that runs automated checks against your resource before every release.
Time
About 30 minutes if you go slow.
You'll learn

You have written at least one Lua script for FiveM. That is enough.
In 3 bullets

You will build a review system that does this:

  • Run 12 checks covering events, performance, security, error handling, and resource hygiene — the categories where most bugs ship.
  • Catch silent failures like nil-safe checks on server events and client-only natives called on the wrong side.
  • Automate 8 of the 12 checks with a Lua lint script so you run them in 10 seconds before every commit.

Vocabulary

Event Hygiene
Ensuring events are named consistently, registered on the correct side, and validated.
Example: a server event that does not check source is a security hole.
Nil Safety
Checking that variables exist before using them, especially player objects.
Example: GetPlayerName(src) returns nil if src is invalid.
Side Discipline
Confirming client-only natives stay in client files and server-only functions stay in server files.
Example: SetEntityHealth is client-only and will nil-error on the server.

Common failures

  • client-only native in server.lua means SetEntityHealth or similar is called on the wrong side. Move it to a client file.
  • unvalidated player ID means source is used without checking GetPlayerName(src) first. Add a nil guard.
  • event name mismatch means the TriggerServerEvent name does not match any RegisterNetEvent. Check for typos.
  • missing config validation means a convar or config value is read without checking it exists. Add a default fallback.

Refactor

Package the lint script as a standalone FiveM resource that runs on server start and checks all other resources, printing a summary to console.

Cheat sheet

The 12-point checklist

  • 1. Manifest: fxmanifest.lua exists and lists all scripts.
  • 2. Events: Names match between trigger and register. Correct side.
  • 3. Source check: Every server event handler validates source.
  • 4. Nil guards: Player objects and config values checked before use.
  • 5. Side discipline: Client natives only in client files, server functions only in server files.
  • 6. Error handling: pcall or xpcall around external calls.
  • 7. Config defaults: Every config value has a fallback.
  • 8. Logging: Key actions print to console with a module prefix.
  • 9. Performance: No infinite loops, no blocking HTTP calls on client.
  • 10. Cleanup: Resource stop removes blips, markers, and threads.
  • 11. Version: version.lua matches CHANGELOG.md.
  • 12. Dependencies: ensure order is correct in server.cfg.

The big idea, in one minute

Your brain cannot hold 12 checks at once while also writing code. That is why checklists exist. Pilots use checklists before takeoff — not because they are inexperienced, but because a missed step at 30,000 feet cannot be undone. Your server is at 30,000 feet the moment players join. A missed nil-check crashes the resource for everyone. A checklist catches it on the ground.

Think of a pre-flight checklist.

The pilot does not memorize every bolt on the wing. The checklist lists them. The pilot runs down the list, ticking boxes. If one box cannot be ticked, the plane does not take off. Your code review is the pre-flight. Your players are the passengers.

Bad way and good way, side by side

Most developers review by glancing at the diff and saying "looks fine." That is not a review — it is a prayer.

Don't do this

Ship without review. Hope nothing breaks.

Testing live is not testing — it is gambling. A missing nil check here crashes the event for everyone. A typo in the event name means the command silently does nothing.

Do this

Run the checklist. Fix every finding. Ship with confidence.

Every line has a purpose tied to a checklist item. The review caught 4 issues before a single player connected. This is what shipping safely looks like.

Before you start

Step 1

Create the REVIEW.md checklist

One file. Twelve checks. Every release.

  1. Create REVIEW.md in your project root (not inside one resource — this is for your whole script store).
  2. Copy the 12-point checklist from the cheat sheet above into the file.
  3. Add a checkbox [ ] next to each item. Mark them as you review.
  4. Save the file. This is now your pre-release gate. No unchecked boxes before shipping.
Step 2

Automate what you can

Eight checks are pattern-matches a script can run faster than a human.

  1. Create lint.lua from the minimal example above.
  2. Add checks for: missing fxmanifest, event name mismatches, client natives in server files, and missing nil guards.
  3. Run it with lua lint.lua ../my-resource. Every FAIL must be fixed.
  4. The automated checks cover points 1, 2, 5, 6, 8, 10, 11, and 12 from the checklist.
Step 3

Manual review: the four that need eyes

No script can check intent. A human must review logic, security, config, and performance.

  1. Logic (check 4): Read every event handler. Does every variable get validated before use?
  2. Security (check 3): Is source checked? Are admin commands gated behind permission checks?
  3. Config (check 7): Does every config value have a default? If the customer deletes a line, does the script still start?
  4. Performance (check 9): Are there loops with no wait? Are large files loaded synchronously? Does the resource clean up on stop?
Recap

Three tests, in order

Run each test on a resource you plan to ship.

  1. Test 1, automated checks.
    • Action: run lua lint.lua against your resource.
    • Expected: all automated checks return PASS.
    • How to verify: no FAIL lines in the output. Fix any that appear.
  2. Test 2, manual review.
    • Action: go through checks 3, 4, 7, and 9 by reading the code.
    • Expected: you find at least one thing you would improve.
    • How to verify: you add at least one nil guard or config default that was missing.
  3. Test 3, checklist complete.
    • Action: open REVIEW.md and confirm all 12 boxes are checked.
    • Expected: every box has an [x].
    • How to verify: you would bet your reputation on this release. If not, uncheck and fix.
How long should a code review take?

For a typical 200-line resource: 5 minutes for automated checks, 10 minutes for manual review. Longer means the code is too complex — refactor first.

The rules you just followed

  • Every resource goes through a 12-point checklist before shipping.
  • Automated checks catch the 8 pattern-based failures that humans miss.
  • Manual review covers the 4 judgment-based checks that scripts cannot evaluate.
  • No unchecked box means the release goes out. One unchecked box means it does not.
  • The checklist lives in the repo as REVIEW.md, versioned alongside the code.

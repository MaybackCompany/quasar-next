IP protection, licensing, and FiveM asset escrow

By the end of this lesson you will protect your Lua source code with FiveM asset escrow, build a license key system that stops leaks, and understand what IP protection actually means for script developers.

You'll ship
An escrowed resource that cannot be read without your private key, plus a license validator that checks keys against your server before loading.
Time
About 35 minutes if you go slow.
You'll learn
How escrow obfuscates your code, how license keys tie a purchase to one server, and why both layers together form real IP protection.
Prereqs
You have sold or plan to sell a FiveM script. A working Tebex store helps (Lesson 01).
In 3 bullets

You will build IP protection that does this:

  • Run your Lua files through FiveM asset escrow so the source code is replaced with encrypted bytecode only your key can read.
  • Generate unique license keys per purchase that bind to the customer's server IP, preventing one key from powering five servers.
  • Add a server-side license check that refuses to load if the key is invalid, expired, or already in use on another machine.

Why it matters

Without escrow, anyone who buys your script can copy, modify, resell, or leak it. One leak can kill your revenue overnight.

Vocabulary

Asset Escrow
FiveM's built-in encryption that replaces Lua source with protected bytecode.
Example: server.lua becomes an encrypted blob only your key unlocks.
License Key
A unique string that authorizes one server to run your script.
Example: license:abc123def456 is validated on startup.
IP Binding
Tying a license to a specific server IP address or identifier.
Example: the license only works on 123.45.67.89:30120.

Common failures

  • Failed to verify escrow signature means the encrypted file was modified after escrow. Re-download from Keymaster.
  • license key already in use means the same key is running on two servers. Revoke and issue a new one.
  • resource loads but does nothing means the escrowed files are in the wrong folder or the manifest does not reference them.
  • IP mismatch on every restart means the server uses a dynamic IP. Bind to a static identifier like Steam Web API key instead.

Refactor

Extract the license check into validateLicense(key, ip) so new resources can call the same validator without duplicating code.

Cheat sheet

IP protection layers

  • Layer 1: Escrow. Encrypt the code. No one reads your source without the key.
  • Layer 2: License key. One purchase equals one key. No key, no load.
  • Layer 3: IP binding. The key only works on the server it was registered for.

The big idea, in one minute

Your script is a product. Every product needs a lock. Escrow is the lock on the box — it stops people from opening the package and copying what is inside. A license key is the receipt — it proves the buyer paid. IP binding is the name on the receipt — it proves the buyer is the one using it. Without all three, your script is a file on a public forum by tomorrow morning.

Think of a software CD from the 2000s.

The CD has the program (your code). The CD key on the back of the case is the license. The installer checks the CD key against the internet before it runs. If the key is already registered to another computer, the installer says no. Your FiveM resource works the same way: the escrowed files are the CD, the license key is the CD key, and your validation server is the online check.

Bad way and good way, side by side

Many developers ship plain-text Lua files and wonder why they appear on leak sites the next day.

Don't do this

Ship raw Lua with no escrow and no license check. Anyone can copy it.

The entire script is readable. Anyone can remove your credits, change the name, and sell it as their own. Nothing stops them.

Do this

Escrow the code, validate the license on startup, and refuse to load if invalid.

The source is encrypted. The license check blocks unauthorized servers. The IP binding prevents key sharing. Three layers, no easy way around them.

Before you start

Step 1

Register your resource on Keymaster

The Keymaster is where FiveM manages escrow keys.

  1. Go to keymaster.fivem.net and log in with your Cfx.re account.
  2. Click "Register New Resource" and give it a name matching your resource folder.
  3. Enable "Asset Escrow" for this resource. You will get a public/private key pair.
  4. Download your private key and store it somewhere safe — never share it.
Step 2

Escrow your files

Upload your Lua files and download the encrypted replacements.

  1. In the Keymaster dashboard for your resource, find the Escrow Files section.
  2. Upload server.lua, client.lua, and any shared files.
  3. Download the escrowed (encrypted) versions. They will have the same filenames but different contents.
  4. Replace the original files in your resource folder with the escrowed versions.
  5. Keep your original source files in a private repository — escrowed files are one-way.
Step 3

Add the license check

The license check runs before any game logic. If it fails, the resource stops.

  1. Add qu_license_key as a convar in server.cfg: set qu_license_key "abc123def".
  2. In server.lua, at the very top, read the convar with GetConvar.
  3. On onResourceStart, send an HTTP request to your license server with the key.
  4. If the server returns invalid, call StopResource and print the failure reason.
  5. Save and test. The resource must not load without a valid key.
Recap

Three tests, in order

Run each test and verify the output before moving on.

  1. Test 1, valid key.
    • Action: set a valid license key and start the resource.
    • Expected: the resource loads and prints the validation success message.
    • How to verify: console shows [QUIET][license][start] followed by normal script output.
  2. Test 2, invalid key.
    • Action: set an invalid or expired key and restart.
    • Expected: the resource stops immediately after startup.
    • How to verify: console shows INVALID KEY — stopping resource and the resource status is stopped.
  3. Test 3, escrow verification.
    • Action: try to open an escrowed file in a text editor.
    • Expected: the file contents are not readable plain text.
    • How to verify: the file starts with binary or non-ASCII characters, not local MODULE =.
Why escrow the server files if the buyer controls the server?

Because escrow stops casual copying and modification. A determined attacker with server access can still extract bytecode, but 99% of leaks are casual buyers dragging files into public Discord servers. Escrow blocks the 99%.

The rules you just followed

  • Every script file is escrowed before distribution. Plain-text Lua is a leak waiting to happen.
  • The license check runs on resource start, before any game logic initializes.
  • Invalid licenses stop the resource immediately — no partial loading, no grace period.
  • License keys are stored as server convars, never in client-accessible files.
  • Original source files live in a private repository, separate from the escrowed distribution.

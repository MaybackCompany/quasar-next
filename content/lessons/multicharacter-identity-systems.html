Multi-character and identity management

Players want more than one character. This lesson shows you how multi-character works across QBCore, Qbox, ESX, and ox_core, and how to build a character selection screen that actually saves data.

You'll ship
A working character select screen where players create up to 5 characters, each with independent inventory, job, and bank data.
Time
About 40 minutes to understand the architecture, 20 minutes to configure.
You'll learn
How character data is keyed by citizen ID, why multi-char breaks naive scripts, and how to scope every query to the active character.
Prereqs
You have a framework (QBCore/Qbox/ESX/ox_core) running. You have seen a database table at least once.
In 3 bullets

By the end of this lesson you will understand:

  • A player is a license identifier. A character is a citizen ID. One player can own multiple citizens, each with their own data.
  • When a script runs SELECT * FROM players WHERE license = ? with no citizen ID filter, it pulls the wrong character's data. This is the #1 multi-char bug.
  • The character selection screen is a client-side NUI that sends a citizen ID to the server, which then loads that character's row from the database.

Outcome

You will understand how multi-character data flows and how to write queries that never mix up characters.

Why it matters

A single-character server limits every player to one life. The moment you add a second slot, every SQL query in every resource must be updated. If you miss one, Player A's bank balance leaks into Player A's second character. That breaks your economy.

Vocabulary

License
The unique ID for a player's copy of GTA V. One per Steam/Rockstar account.
Example: license:abc123 is always the same person.
Citizen ID
The unique ID for one character. A player can have many.
Example: citizen ABC0001 and ABC0002 belong to license:abc123.
Active character
The citizen ID the player is currently playing as.
Example: Player.PlayerData.citizenid returns the active one.

Checklist

  1. Open your database and confirm the players table has a citizenid column.
  2. Check your framework config for MaxCharacters (set to at least 3).
  3. Search every SQL query in your custom resources for WHERE license =.
  4. Replace with WHERE citizenid = using the active character ID.
  5. Test with two characters on the same license — bank balances must be independent.
  6. Test with two characters — inventory items must be independent.

Exercise

Write a command /whoami that prints the player's license, citizen ID, and character name to the console. Run it on two different characters and confirm the citizen IDs are different.

Expected output: [whoami] license=abc123 citizenid=XYZ0001 name="John Doe" for character 1. [whoami] license=abc123 citizenid=XYZ0002 name="Jane Doe" for character 2.

Common failures

  • Data leaks between characters — you queried by license instead of citizen ID. Every character sees the first character's data.
  • Max characters reached but no UI shown — the character select NUI is not loading. Check the spawnmanager resource.
  • Character deleted but items remain — your inventory table has no CASCADE delete on the citizen ID foreign key.
  • New character overwrites old character data — your INSERT uses REPLACE with only the license as the unique key.

Refactor

Create a helper getActiveCitizenId(source) that every resource imports, so you never accidentally hardcode a license query again.

Cheat sheet

Multi-character data model

  • Player (license). Maps to one Steam/Rockstar account. Has many characters.
  • Character (citizenid). One row per character. Has inventory, bank, job, appearance.
  • Active character. The citizen ID currently loaded. Stored in Player.PlayerData.citizenid.
  • Every query must filter by citizenid. If you query by license, you get the wrong character's data.

The big idea, in one minute

A player is a person. A character is a role they play. One person can play five different roles. Each role has its own wallet, its own car keys, its own job. The database must reflect this. If your bank_accounts table only stores by license, all five characters share one bank account. That is not multi-character. That is a bug.

Think of a hotel with room keys.

The guest (license) can book multiple rooms (characters). Each room has its own key, its own minibar, its own checkout time. If you hand the guest a master key that opens every room and shares one bill, the hotel collapses. The database is the hotel's ledger. Each room gets its own row. The guest is just a foreign key linking the rooms together.

Your citizen ID is the room number. Your license is the guest name. Every transaction writes to one room, never to the guest directly.

Wrong query and right query, side by side

This is the single most common multi-character bug. One wrong WHERE clause, and every character sees the first character's bank account.

Wrong: query by license

Returns the bank balance of whichever character was created first. All characters see the same number.

MySQL returns the first matching row. Since both characters share the same license, you always get character 1's balance. Character 2's balance is invisible.

Correct: query by citizenid

Returns exactly the bank balance of the active character. Each character sees only their own money.

The citizen ID uniquely identifies the character. Every query scoped to citizen ID returns the correct row. Character 1 and character 2 have independent balances.

Before you start

If any box is unchecked, take five minutes to fix it first.

Step 1

Understand the data model

Before you change any code, open the database and look at the tables.

  1. Open your database with HeidiSQL, DBeaver, or phpMyAdmin.
  2. Find the players table. Look at the columns: license, citizenid, charinfo.
  3. Run SELECT license, citizenid, JSON_EXTRACT(charinfo, '$.firstname') FROM players;.
  4. You should see the same license multiple times, each with a different citizen ID and first name.
  5. That is the proof. Same player, multiple characters. Each citizen ID is a separate identity.
Step 2

Audit your custom resources

Every SQL query you wrote must use citizen ID, not license.

  1. Open every custom resource you have written.
  2. Search for the string WHERE license = in all .lua files.
  3. Every hit is a potential data leak between characters.
  4. Change each one to WHERE citizenid = and get the citizen ID from the framework's player object.
  5. In QBCore/Qbox: Player.PlayerData.citizenid. In ESX: xPlayer.getIdentifier() returns citizenid if multi-char is on. In ox_core: player.charid.
Step 3

Test with two characters

The only way to know you fixed it is to test it.

  1. Create two characters on your account. Give them different names.
  2. Log in as character 1. Deposit $100 in the bank. Note the balance.
  3. Log out, switch to character 2. Check the bank balance. It should be $0 (or whatever you started with).
  4. If character 2 sees $100, you missed a query. Go back to Step 2.
  5. Repeat for inventory. Give character 1 a weapon. Switch to character 2. Character 2 should not have that weapon.
Recap

Three tests, in order

Run each test with two characters. The first character should never affect the second.

  1. Test 1, bank independence.
    • Action: character 1 deposits $500. Switch to character 2.
    • Expected: character 2's balance is unchanged.
    • How to verify: run SELECT money FROM bank_accounts WHERE citizenid = 'CHAR2_ID';.
  2. Test 2, inventory independence.
    • Action: character 1 receives a pistol via command. Switch to character 2.
    • Expected: character 2 has no pistol.
    • How to verify: open character 2's inventory UI. The pistol is not there.
  3. Test 3, job independence.
    • Action: character 1 sets job to police. Switch to character 2.
    • Expected: character 2 is not police.
    • How to verify: character 2's job is whatever was set for that character (usually unemployed).
Why not just use a separate table per character?

Because that would create hundreds of tables as your player base grows. One table with a citizenid column scales better, makes queries simpler, and lets you join data across characters when you need to (e.g., "find all characters who have ever been banned").

The rules you just followed

  • A license identifies a player. A citizen ID identifies a character. Never confuse them.
  • Every SQL query that reads or writes player data must scope to citizen ID.
  • The active citizen ID comes from the framework's player object, never from a client event parameter (which a cheater can spoof).
  • Test every resource with two characters before considering it production-ready.
  • Foreign keys from inventory/job/bank tables should reference citizenid, not license.

What you learned

  • Multi-character means one license (player) owns multiple citizen IDs (characters).
  • Every database table that stores player data must key on citizenid, not license.
  • The active citizen ID is stored in the framework's server-side player object.
  • Querying by license on a multi-character server returns the wrong character's data.
  • Test with two characters on the same account to confirm full independence.

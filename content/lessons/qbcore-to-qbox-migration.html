QBCore to Qbox migration, step by step

QBCore was the dominant FiveM framework for years. Qbox is its actively maintained successor. This lesson walks you through the full migration: what changes, what breaks, and how to fix every error along the way.

You'll ship
A Qbox server running your existing QBCore resources, with a migration checklist that catches every rename and API change.
Time
About 2-4 hours depending on how many custom resources you have.
You'll learn
The exact renames from QBCore to Qbox, how to update your fxmanifest dependencies, and which QBCore patterns are deprecated.
Prereqs
You have a running QBCore server. You know how to edit fxmanifest.lua and search across files.
In 3 bullets

Your migration will follow this pattern:

  • Replace the core resource: qb-core becomes qbx_core. The database stays. Player data is untouched.
  • Update every fxmanifest dependency: 'qb-core' becomes 'qbx_core'. Shared exports like 'qb-target' become 'ox_target'.
  • Replace renamed functions: QBCore.Functions becomes exports.qbx_core. Event names change from QBCore:Server: to qbx_core:server:.

Checklist

  1. Clone qbx_core into your resources folder.
  2. Update server.cfg: change ensure qb-core to ensure qbx_core.
  3. In every fxmanifest, change 'qb-core' to 'qbx_core'.
  4. Search all files for exports['qb- and replace with exports.qbx_.
  5. Search all files for QBCore: event names and replace with qbx_core:.
  6. Start the server. Fix errors one resource at a time until every resource loads.

Common failures

  • No such export GetCoreObject in resource qbx_core — you wrote exports['qbx_core'] with brackets. Qbox exports do not use bracket syntax. Write exports.qbx_core.
  • attempt to call a nil value (field 'CreateUseableItem') — you called it on the QBCore table instead of as an export. Use exports.qbx_core:CreateUseableItem(...).
  • Event QBCore:Server:OnPlayerLoaded not found — the event name changed. Use 'qbx_core:server:onPlayerLoaded'.
  • Resource qb-xxx not found — a dependency in fxmanifest still references an old qb- resource. Update it to the qbx_ equivalent.

Refactor

After migration, remove the local QBCore = exports.qbx_core:GetCoreObject() line entirely. Qbox resources should call exports directly instead of caching a reference that might become stale on restart.

Cheat sheet

QBCore → Qbox rename table

  • Core resource: qb-core → qbx_core. Export: exports['qb-core'] → exports.qbx_core.
  • Player events: QBCore:Server:OnPlayerLoaded → qbx_core:server:onPlayerLoaded. Note the lowercase.
  • GetPlayer: QBCore.Functions.GetPlayer(src) → exports.qbx_core:GetPlayer(src).
  • CreateUseableItem: QBCore.Functions.CreateUseableItem(...) → exports.qbx_core:CreateUseableItem(...).
  • GetCoreObject: still exists as exports.qbx_core:GetCoreObject() for backwards compat, but prefer direct exports.

The big idea, in one minute

Qbox is not a rewrite. It is a rename with cleanup. The database schema is the same. Player data is the same. Event flow is the same. The only thing that changed is what things are called. Your migration is a giant find-and-replace operation, verified resource by resource.

Think of moving to a new apartment in the same building.

Your furniture (your data) stays the same. Your daily routine (your event flow) stays the same. The only thing that changes is your address. QBCore was apartment 4B. Qbox is apartment 5A. Same floor plan, different number on the door. You are updating every envelope (every fxmanifest and export call) to the new address.

Nothing is lost. Nothing is rebuilt. You are just changing the labels.

Scattershot migration and systematic migration, side by side

Most people change a few things, start the server, see 50 errors, and give up. The system works.

Scattershot migration

You change a few files, start the server, see a wall of red, and cannot tell which error is from which resource.

Mixing migrated and unmigrated resources creates cascading errors. An unmigrated resource triggers an old event name, the new core does not recognize it, everything downstream fails.

Systematic migration

You work through the rename table resource by resource. Each resource is fully migrated and tested before you touch the next one.

Each resource is a contained unit. Errors are trapped at the resource level. You never face a wall of red. By resource 10, the pattern is automatic and each one takes 2 minutes.

Before you start

If any box is unchecked, do it before touching a single file.

Step 1

Set up the new core alongside the old one

Do not delete qb-core yet. Run both side by side while you migrate.

  1. Clone https://github.com/Qbox-project/qbx_core.git into resources/[qbx]/qbx_core.
  2. Clone ox_lib if you do not already have it: resources/[standalone]/ox_lib.
  3. Create a test config: comment out all resources in server.cfg except qbx_core, ox_lib, and chat.
  4. Start the server. You should see [qbx_core] Server started successfully and no errors.
  5. If you see errors, fix the qbx_core setup before touching any custom resources.
Step 2

Update all fxmanifest dependencies

Every fxmanifest.lua that says 'qb-core' must say 'qbx_core'.

  1. Run a global search for 'qb-core' in all fxmanifest.lua files.
  2. Replace with 'qbx_core'. Do not change anything else in the manifest yet.
  3. Run a second search for 'qb-target' and replace with 'ox_target'.
  4. Run a third search for 'qb-menu' and replace with 'ox_lib' (ox_lib provides the menu now).
  5. Save all files. Do not start the server yet.
Step 3

Update Lua code in every resource

This is the bulk of the work. Work resource by resource, not file by file.

  1. Search all .lua files for exports['qb-core']. Replace with exports.qbx_core.
  2. Search for QBCore.Functions.GetPlayer(. Replace with exports.qbx_core:GetPlayer(.
  3. Search for QBCore.Functions.CreateUseableItem(. Replace with exports.qbx_core:CreateUseableItem(.
  4. Search for event names: 'QBCore:Server: → 'qbx_core:server:. 'QBCore:Client: → 'qbx_core:client:. Note the lowercase after the colon.
  5. Search for 'qb- exports. Example: exports['qb-inventory'] → exports.ox_inventory.
Step 4

Start resources one at a time

Do not start everything at once. One resource, one test, one pass or fail.

  1. In server.cfg, uncomment one migrated resource. Start the server.
  2. Watch the console. If the resource starts clean, test one of its commands or features.
  3. If an error appears, read it. The error message almost always names the exact file and line.
  4. Fix the error. Restart the resource. Confirm it starts clean. Then move to the next resource.
  5. Repeat for all 47+ resources. By the end, your server.cfg has every resource uncommented and the console is clean.
Recap

Three checkpoints, in order

Each checkpoint proves a layer of the migration is complete. Do not skip any.

  1. Checkpoint 1: bare core starts.
    • Action: start server with only qbx_core, ox_lib, and chat.
    • Expected: three resources start clean. No red lines. Chat works.
    • How to verify: type in chat. If the message appears, the core is healthy.
  2. Checkpoint 2: first custom resource migrates.
    • Action: pick your simplest custom resource. Migrate it. Start it alongside the core.
    • Expected: the resource starts clean. Its commands or events work.
    • How to verify: use the resource's main feature. If it works, the pattern is correct.
  3. Checkpoint 3: full server starts.
    • Action: uncomment all resources. Start the server.
    • Expected: every resource starts. Console has zero SCRIPT ERROR lines.
    • How to verify: count the resources in the console. It should match your old QBCore server's count minus any resources you intentionally removed.
Why do event names switch to lowercase after the colon?

Qbox adopted a consistent naming convention: resource:side:camelCaseEventName. The side is always lowercase (server/client). The event name is always camelCase. This makes events predictable and grep-friendly. QBCore used mixed conventions that varied per resource.

The rules you just followed

  • Never delete the old core until the new core starts clean with all resources.
  • Migrate one resource at a time. Test each one before touching the next.
  • Use global search-and-replace for the structural renames (exports, events, dependencies).
  • Read every error. The error message tells you exactly which rename you missed.
  • Commit or back up before starting. You will make mistakes. That is normal. Revert and retry.

What you learned

  • Qbox is a rename and cleanup of QBCore. The database, player data, and event flow are unchanged.
  • The three things that change: export syntax (dot notation), event names (lowercase side), and dependency names (qbx_ prefix).
  • Migrate resource by resource, starting with the simplest one, testing each before continuing.
  • Use global search-and-replace for structural renames, then fix per-resource edge cases.
  • A clean Qbox console with all resources started is the only proof of a complete migration.

Common failures

  • attempt to call a nil value (global 'AddBlipForCoord') means the script is running on the server side. Blips, markers, and checkpoints are client-only.
  • Marker does not appear: you forgot the infinite loop with Citizen.CreateThread and Wait(0). Markers are drawn per-frame.
  • Checkpoint never fires: the distance check uses #(playerCoords - coords) which returns a float. Make sure checkpointRadius is large enough (at least 3.0).
  • Blip shows at LSIA instead of Sandy Shores: you swapped X and Y. GTA coordinates are (X east, Y north, Z up).

Refactor

Cheat sheet

Blip vs Marker vs Checkpoint

  • Blip. Radar and map only. Use for "go here" targets.
  • Marker. 3D world visual. Use when players need to see a destination from the game world.
  • Checkpoint. Trigger zone with built-in ring. Use for arrival detection.

The big idea, in one minute

GTA V gives you three tools for the same job: guiding a player to a location. A blip works on the 2D radar. A marker works in the 3D world. A checkpoint works as both a visual and a trigger zone. Each one solves a different problem, and you will use all three in nearly every script you write.

Think of a gas station.

The GPS (blip) shows the station on a map. The tall sign (marker) is visible from the highway so you know where to turn. The pressure plate at the pump (checkpoint) detects when your car is in the right spot. Three tools, one location.

Your map elements work the same way. Place the blip for navigation, the marker for world visibility, and the checkpoint for interaction. Each one lives on the client and is drawn every frame.

Bad way and good way, side by side

Most beginners draw markers without cleaning them up or checking distance. The good version is only a few more lines, and it keeps your server fast.

Don't do this

A marker drawn every frame from the other side of the map wastes GPU time.

No distance check. The marker is drawn for every player on the server, all the time, even when it is not visible. On a 64-player server, that is 64 useless draw calls every frame.

Do this

Add a distance guard. Only draw when the player is close enough to see it.

The if dist < drawDistance guard means the DrawMarker call only runs when the player is within 100 meters. That saves 63 draw calls every frame on a full server.

Step 1

Create the resource

One folder, two files. That is all.

  1. Inside resources/, create a folder named qu_map_elements.
  2. Create fxmanifest.lua with the bare minimum manifest.
  3. Create an empty client.lua.
  4. Add ensure qu_map_elements to server.cfg.
  5. Save everything. Step 1 is done.
Step 2

Place the blip

The blip is the easiest of the three. It only runs once.

  1. Open client.lua and define a coords vector3.
  2. Call AddBlipForCoord(coords.x, coords.y, coords.z). Save the return value in a variable.
  3. Set the sprite, color, scale, and name with the four SetBlip* natives.
  4. Join the server and open the map. The blip should be visible.
  5. A blip does not need a loop. It is created once and the game remembers it.
Step 3

Draw the 3D marker

Markers need a loop. They are drawn fresh every frame.

  1. Add a Citizen.CreateThread block with while true do ... Wait(0) end.
  2. Inside the loop, call DrawMarker with type 2 (a floating diamond).
  3. Add a distance guard so it only draws when the player is within 100 meters.
  4. If the distance check is missing, try commenting it out. You will still see the marker. The guard is for performance, not visibility.
  5. Join the server. Fly or drive to the coordinates. The marker should float above the ground.
Step 4

Add the checkpoint ring

A checkpoint is a visual ring plus a distance check in one thread.

  1. Add a second Citizen.CreateThread loop for the checkpoint.
  2. Call CreateCheckpoint with type 47 (a glowing ring). Type 0 is a cylinder.
  3. Use #(playerCoords - coords) to measure distance from the player to the checkpoint center.
  4. When dist < checkpointRadius, print a line to F8 and maybe break out of the loop to avoid repeated triggers.
  5. Walk into the ring. Check F8 for the print line.
Step 5

Clean up and verify all three

Run each test once and confirm the behavior.

  1. Open the map. Confirm the blip exists and is the correct sprite and color.
  2. Drive to the coordinates. Confirm the 3D marker is visible from at least 50 meters away.
  3. Drive through the checkpoint ring. Confirm the F8 console prints the expected line.
  4. Remove the blip with RemoveBlip(blip) in a test command to make sure you can clean up.
  5. All three elements are verified. The lesson is complete.

Why do markers need a loop but blips do not?

Blips are managed by the game's radar system. You create them once and the game keeps them. Markers are raw 3D shapes that must be drawn every frame, like a particle effect. If you stop calling DrawMarker, the marker disappears.

The rules you just followed

  • Blips are created once and persist until removed. No loop needed.
  • Markers must be drawn every frame inside a Citizen.CreateThread loop.
  • Checkpoints combine a visual ring drawn per-frame with a distance check for triggering events.
  • Always add a distance guard to marker and checkpoint loops. Drawing from 10km away is wasted GPU time.
  • All three — blips, markers, checkpoints — are client-only natives. The server cannot draw them.

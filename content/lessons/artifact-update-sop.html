Artifact update SOP, safe upgrades every time

By the end of this lesson you will have a written Standard Operating Procedure for updating your FiveM server artifacts that prevents downtime, catches breakage before it hits players, and always lets you roll back in under two minutes.

You'll ship
A one-page SOP you follow every time you update server artifacts, from backup through verification.
Time
About 15 minutes to write the SOP, then 10 minutes per update forever.
You'll learn
Why skipping the backup step costs you a full reinstall, how to test an update on a staging server, and which files you must never overwrite blindly.
Prereqs
Your server is running and you know where the artifacts folder lives.
In 3 bullets

You will build a repeatable artifact update process that does this:

  • Before touching anything, you back up the current artifacts folder and your server-data into a timestamped tarball.
  • You download the new artifacts to a staging folder, test them with a copy of your configuration, and only then deploy to production.
  • If anything breaks, you restore from the backup and are back online in under two minutes — no panic, no guessing.

Outcome

Why it matters

Artifact updates change the core engine your server runs on. A bad update can break every script, corrupt your database, or prevent players from connecting. Without a rollback plan, you are reinstalling from scratch while your community waits.

Vocabulary

Artifacts
The compiled server binary and its supporting files. This is the FiveM server engine itself.
Example: fxserver-linux.zip contains the fxserver binary, citizen/ directory, and native docs.
Server-data
Your configuration and resources. This is separate from artifacts and should never be overwritten by an update.
Example: the folder containing server.cfg, resources/, and your database files.
Rollback
Restoring the previous artifacts version after a failed update.
Example: if build 12345 breaks your scripts, copy back build 12300 from the backup.

Checklist

  1. Copy the SOP above into a file on your server machine: ~/fxserver/UPDATE-SOP.txt.
  2. Edit the paths to match your actual server layout (change /home/fxserver to your real path).
  3. Run through the backup step (Step 2) right now on your production server to confirm it works.
  4. Verify the backup exists and has the expected files.
  5. Choose a low-traffic time and run the full SOP for your next artifact update.
  6. After the update, update the "Last updated" date at the top of the SOP.

Exercise

Run Step 2 (backup only) on your live server right now. Do not stop the server if players are on — wait for a quiet moment. Create the timestamped backup folders and verify both exist with ls -la ~/backups/.

Expected output: Two folders appear in ~/backups/: artifacts-YYYYMMDD-HHMM/ and server-data-YYYYMMDD-HHMM/.

Common failures

  • tar: Cannot open: No such file or directory means the download URL was wrong or the file did not download. Check the URL on the Cfx.re build page and try again.
  • Server starts but no resources load means you copied artifacts over your server-data, or the +exec path in the startup command is wrong. Restore from backup and double-check paths.
  • Backup folder is empty means you ran cp without -r on a directory. Always use cp -r for folders.
  • Staging server works but production does not means the production startup command or server.cfg is different from staging. Compare the two run.sh commands line by line.

Refactor

Turn this SOP into a shell script: update-artifacts.sh. The script runs each step, asks for confirmation at Step 5 before deploying to production, and logs everything to a file. Now your update is one command: ./update-artifacts.sh.

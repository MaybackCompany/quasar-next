Exports: sharing functions across resources

Events fire and forget. Exports are function calls: synchronous, return-a-value, same-side. They are how every framework you have ever used wires resources together, exports.ox_inventory:GetItemCount, exports.ESX:getPlayerFromId. This lesson teaches you to expose them safely and consume them without surprises.

Mental model: exports are front-desk phone extensions

Picture a hotel. Each department (resource) has a phone. The front desk publishes the extension list: dial 201 for housekeeping, 305 for the kitchen. That is exports. Anyone in the hotel can pick up a phone and dial, the call is direct, the answer is immediate.

Events are the PA system. You announce "checkout in 5 minutes" and everyone who is listening hears it. No reply. Different tool, different job. Use exports when you need an answer; use events when you need to broadcast.

FiveM-specific note

Exports are same-side. A server resource can call another server resource's exports. A client resource can call another client resource's exports. Crossing client-server still requires an event. There is no exports.serverThing from the client.

Declare an export

Outcome: another resource can call your function and get a return value.

The string in exports('Name', fn) is what callers type. Conventional style: PascalCase for the function name, snake_case for the resource name.

Step 2

Call an export from another resource

Outcome: you can read another resource's state from your own.

Why the colon syntax

exports.qu_money:GetBalance(src) is sugar for exports.qu_money.GetBalance(exports.qu_money, src). Just like Lua method calls. Stick to the colon, it is the convention every framework follows and it makes the resource name jump out.

Common confusion

Calling an export before the target resource has started returns nil from a phantom function and silently breaks. Use dependency 'qu_core' in your fxmanifest so FiveM enforces start order, and gate any first-tick call behind onResourceStart.

If you find yourself calling TriggerEvent and then waiting for a callback event to come back with the result, you wanted an export. Export the function, drop both events.

Step 5

Profile the export call path

Outcome: you can prove an export-heavy hot path is not eating your tick budget, and you know when to cache an export reference vs re-resolving it every call.

Every exports.qu_money:GetBalance(src) resolves through three layers: the resource registry lookup, the function table lookup, and the colon-call self-bind. None of it is free. On a tick that runs the call 200 times, the lookup cost matters more than the function body.

5.2 resmon 1 to spot a hot caller

If resmon 1 shows your resource at 0.40–1.00 ms server-side at idle, you are almost certainly burning the tick on a polling loop that exports another resource every frame. Pattern to kill: CreateThread + while true do exports.X:Y() Wait(0) end. Replace with an event push from the source of truth.

5.3 Detect dependency-start races

Add a one-liner that fires on resource start and verifies every export you depend on is reachable. Without this, a missing export looks like a quiet nil for 30 minutes until a player triggers the bug.

Common mistakes

Forgetting the manifest line

Without server_export 'Name' in fxmanifest, the export is unreachable from other resources even though the function works locally. Every export needs both: the runtime registration (exports('Name', fn)) and the manifest declaration.

Calling before the dependency starts

Use dependency 'qu_core' in your manifest. Without it, FiveM may start your resource first; the export call returns nil and your code silently misbehaves.

Trying to call client exports from server

You cannot. Exports are same-side. Send a NetEvent and have the client call its own export, then return via another NetEvent, or rethink the design so the data lives where you need it.

Returning multiple values

Exports return one value reliably. Multi-return works on some builds and not others. Return a table instead: { ok = true, balance = 5000 }.

Naming clashes between resources

Two resources cannot register the same export name to the same table. Use namespaced names: qu_money:GetBalance, not GetBalance. The resource name is the namespace; lean on it.

What you learned

  • Exports are synchronous function calls between resources, same-side only.
  • Both exports('Name', fn) and a manifest server_export 'Name' are required.
  • exports.resource:Name(args) is the canonical call shape.
  • Use dependency in your manifest so the resource you depend on starts first.
  • Reach for events when broadcasting; reach for exports when you need an answer.

Before you start

How to read this lesson

Do not try to learn every ox_lib function today. Learn the five daily ones first: notify, menu, input, progress, and callback. That is enough to make later scripts feel modern instead of homemade.

The mental model: a UI kit plus a standard library

Think of ox_lib as a big ready-made toolbox. Instead of building every menu and pop-up from scratch, you open the toolbox and pick the piece you need.

It gives you:

  • Consistent UI components: notifications, menus, dialogs that look the same in every resource that uses them. No more Frankenstein UIs.
  • Quality-of-life helpers: callbacks (Lesson 15), zones, requests for models, player data caching.
  • Framework-agnostic: works with ESX, Qbox/QBCore, or raw scripts.

Your job this lesson is not to memorize everything ox_lib can do. It is to meet the few pieces you will use again and again, then wire them together in one tiny practice resource.

Where the docs live

ox_lib is open source at overextended/ox_lib. The official docs, searchable, always current, are at overextended.dev/ox_lib. Bookmark that URL. You'll visit it more than you'd expect.

Install check: the 30-second sanity pass

Before you call a single lib.* function, make sure ox_lib is actually running. In the server console, type resource ox_lib. You should see it listed as started. If not, fix that first, every snippet in this lesson assumes it's up.

That one line loads ox_lib's API on both sides. Without it, lib is nil and every call will throw. Put it in the manifest before the script lists, not after.

Step 1

lib.notify: polished toasts in one line

Outcome: a command that pops a branded notification in the corner instead of spamming the F8 console.

Build qu_ui alongside your previous resources. Client file gets a /notify command:

Run /notify. A toast slides in, displays your title and description, and fades out after 4 seconds. No NUI to write, no CSS to author, no HTML, ox_lib handles all of it. The type field changes the colour and icon (green check, red cross, yellow triangle, blue info).

Replace every F8 print the player should see

print is for you, the developer. lib.notify is for the player. Any time your script would have printed "bought a pizza", "not enough money", "cooldown 30s", make it a notify instead. The difference in feel is enormous for one line of code.

Step 2

Server-side notify: sending a toast to a specific player

Outcome: the server can push a notification to one client in a single call.

You can lib.notify from the server too. The first argument becomes the target server id.

From the server console: warn 3 sends the warning to player with server id 3. Under the hood ox_lib is firing a net event to that client, but the API hides that. You just say what you want; the library does the wiring.

Server notify needs the id as the first arg

Client code: lib.notify({ ... }). Server code: lib.notify(targetId, { ... }). Forget the id on the server and you'll get a runtime error. Different shapes for different sides, a recurring ox_lib pattern.

Step 3

lib.registerContext + lib.showContext: a real menu

Outcome: a keystroke opens a branded, multi-option menu with click-to-act entries.

Context menus are the Swiss Army knife of FiveM UIs. Inventory, job menus, admin tools, vehicle options, almost anything with "pick one of these" shape can be a context menu. Ox_lib's version looks clean, handles keyboard navigation, and takes seconds to register.

Run /menu and a menu opens. Each entry has a title, optional description, optional icon (Lucide icon names, you've seen these in the rules section), and an onSelect function. Click one and the function runs.

A menu isn't a new concept, you just couldn't build one before. Now you can. That's ox_lib's job: give you the piece you were about to write by hand.

Submenus, free

A menu option can have a menu = 'other_menu_id' field instead of onSelect. Clicking it opens that menu. Chain as deep as you like; ox_lib handles the back button. This is how you build a whole settings tree in 30 lines.

Step 4

lib.inputDialog: forms that just work

Outcome: a dialog that asks the player a question and returns their answer as a table.

When the player needs to type something, a license plate, a transfer amount, a message, use lib.inputDialog. It opens a form, waits for submit or cancel, and returns the answers.

Three things to notice. First, lib.inputDialog blocks the current coroutine like lib.callback.await did, same coroutine semantics, same safety. Second, it returns nil if the player hits cancel or escape, branch on that first, always. Third, results come back as a 1-indexed table in the same order as the fields; unpack them immediately into named locals for sanity.

Field types worth knowing: input (text), number, textarea, checkbox, select (dropdown), date, color. Each supports required, min, max, placeholder. The docs list the full schema per field.

Always validate input again server-side

The dialog's min and max are client-side. A cheater can bypass them. Every value coming out of an input dialog is still client input. Re-validate on the server before you touch any authoritative state.

Recap 1

Wire the menu to a real feature

Right now the "Heal me" menu option fires qu:server:heal but nothing else changes in the UI. Add two things so the player feels the action:

  1. A success notify on the client once the heal event from the server fires back.
  2. An error notify if the server rejects the heal (imagine a cooldown).

Sketch both handler blocks before opening the solution.

The solution

Notice the shape: action trigger (menu → event), result path (event → notify). That's the standard pattern, UI libraries like ox_lib handle the visual polish; your events are still the backbone.

Step 5

lib.progressCircle: long actions with feedback

Outcome: an action that takes time shows a circular progress indicator with an icon and can't be skipped.

Any action that takes more than a split-second, lockpicking, patching a tire, drinking a beer, should show progress. Ox_lib's progressCircle locks controls, plays an animation, and returns whether the player completed or cancelled.

Run /patch. A progress circle appears, the ped plays a fixing animation, controls are locked for 5 seconds, and the function returns true on completion or false if the player cancelled or took damage. You get to branch on that boolean and decide what happens next.

Pair every progress with a server event

The client completing the progress circle doesn't grant anything, it's still a cosmetic. Fire the qu:server:tirePatched event after the ok check so the server can verify the player is near a broken tire and apply the real effect. Progress UIs are feedback; they're not trust.

Recap 2

Predict the flow

Given these four lines of client code run in order, describe what the player sees and which of them can the player cancel.

The answer
  1. Notify pops immediately and fades, non-blocking.
  2. Input dialog opens; the function pauses until submit or cancel. Cancellable: returns nil.
  3. Context menu opens after the dialog closes. Cancellable: pressing escape closes it and lib.showContext returns.
  4. Progress circle runs for 2 seconds. Cancellable only if canCancel = true is passed (default in 2.x is true). Returns ok = false if cancelled.

Takeaway: three of the four are blocking. The notify is the only fire-and-forget in this batch. Design flows carefully, stacking three blocking UIs in a row makes the player feel held hostage. Interleave them with actions and server round-trips.

Common mistakes: ox_lib edition

attempt to index a nil value (global 'lib')

You forgot shared_script '@ox_lib/init.lua' in fxmanifest.lua. Add it, restart. This is the #1 ox_lib error in history.

Registering a context menu inside a command

lib.registerContext goes at the top level of a client file, once. Putting it inside a command re-registers the menu on every keypress, not fatal, but wasteful and makes debugging harder. Register once, show many times.

Trusting inputDialog values

Client-side min/max can be patched out. Every value coming out of an input dialog is player input, validate on the server.

Calling server-side lib.notify without a target id

On the server, the first argument is the server id. Missing it silently does nothing or throws. Shape differs from the client, memorise both.

Building custom NUI when ox_lib already has the component

Context menus, progress bars, notifications, radial menus, zones, all in ox_lib. Before writing 200 lines of HTML/CSS, skim the docs. 80% of the time the component exists.

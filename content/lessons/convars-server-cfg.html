Convars: server.cfg config you can read from Lua

A convar (configuration variable) is a server-side key/value pair you set in server.cfg or at runtime, then read from any resource. Convars are how you externalize things that change per server, debug flags, database name, max stack size, Discord invite, owner ID, without editing Lua and without rebuilding the resource. Three flavors decide who sees them, the API is four functions, and you already use them every time you write sv_hostname.

Why convars at all

You already know two ways to configure a resource: a config.lua table the script reads at start, and a SQL settings row. Both work. Both have a cost.

  • config.lua requires a code edit + resource restart. Anyone with shell access can change it, but server owners have to know which file to open.
  • SQL settings need a query, a cache, an admin command to update, and a resource that handles all of it.
  • Convars are built into the runtime. You write one line in server.cfg, you read it from Lua. No table, no migration, no cache. The server owner just edits the line everyone already knows about.

The trade-off: convars hold strings (or integers parsed from strings). They are not for arrays, nested tables, or anything that wants structure. For complex config you still want config.lua. For one-line toggles and per-server tuning, convars are the right tool.

This lesson comes before SQL and webhooks

Any lesson that shows mysql_connection_string, API keys, Discord webhooks, or panel tokens depends on this one. Put secrets in a server-only config file first, then continue to SQL from zero, banking with audit logs, or HTTP webhooks.

Fix: keep the real password in server-secrets.cfg, load it with exec server-secrets.cfg, grant the user to the correct database, and restart until the console prints [script:oxmysql] Database server connection established!.

Step 1

The three flavors: set, setr, sets

Outcome: you can pick the right flavor in two seconds based on who needs to read the value.

The decision tree

Ask one question: does the client need to read this value?

  • No, server-only. Use set. Standard convar. Visible on server, invisible on client.
  • Yes, client reads it too. Use setr. Server replicates the value to every client; clients can read but not write.
  • Yes, and outsiders should see it on the server browser / info.json. Use sets. Server-information convar, public, indexed by the master server list.
# server.cfg

# server-only: scripts on this server read it, clients never see it
set qu_debug 1
set qu_starting_balance 5000
set qu_db_name "qu_bank"

# replicated: clients can call GetConvar('qu_max_stack', '20') and get '64'
setr qu_max_stack 64

# server info: shows up in the server browser and at /info.json
sets Discord "https://discord.gg/quasaracademy"
sets sv_projectName "QU Roleplay"
Never put secrets in sets

sets values are public. They appear at http://<server_ip>:<port>/info.json and on the master server list. Use set (server-only) for tokens, API keys, and webhooks.

Step 2

Reading convars: GetConvar and GetConvarInt

Outcome: you read a string convar with a default, and an integer convar with a default, on whichever side is allowed.

The two read functions

FiveM gives you exactly two reads:

  • GetConvar(name, default), returns a string. Always pass a default. If the convar is not set, you get the default back instead of nil.
  • GetConvarInt(name, default), same, but parsed as an integer. Use this for numeric flags. The default must be a number.
-- server/main.lua
local DEBUG = GetConvarInt('qu_debug', 0) == 1
local STARTING_BALANCE = GetConvarInt('qu_starting_balance', 1000)
local DB_NAME = GetConvar('qu_db_name', 'qu_bank')

if DEBUG then
    print(('[qu_bank] starting_balance=%d  db=%s'):format(STARTING_BALANCE, DB_NAME))
end
Read once at start, not on every event

Convars rarely change at runtime. Read them into local variables when the resource starts (or when an event handler is registered) and reuse the locals. Calling GetConvar hundreds of times per minute is wasted work.

Reading a replicated convar from the client

Clients can only read convars that were set with setr. The same GetConvar function works on both sides, the API is identical, the difference is only which convars are visible.

-- client/main.lua
local MAX_STACK = GetConvarInt('qu_max_stack', 20)

RegisterNUICallback('uiReady', function(_, cb)
    cb({ maxStack = MAX_STACK })   -- UI needs to know the per-server cap
end)
Standard set convars are invisible to the client

If you write set qu_max_stack 64 in server.cfg and call GetConvar('qu_max_stack', '20') on the client, you get '20', the default. The client cannot see standard convars. Use setr when the client needs the value.

Step 3

Setting convars from a script

Outcome: you know when (and when not) to write a convar from Lua, and you know which API to call.

The three write functions 
-- server-only
SetConvar('qu_debug', '1')

-- replicated to clients
SetConvarReplicated('qu_max_stack', '64')

-- server info, public
SetConvarServerInfo('Discord', 'https://discord.gg/quasaracademy')

Note that values are always passed as strings. Even if you write SetConvar('qu_debug', '1'), you read it back with GetConvarInt if you want a number.

When to set a convar from script (rarely)

Most of the time, server.cfg is the right place. Setting from a script makes sense for:

  • Computed defaults at boot. Resource A reads its own version and exposes it via SetConvarServerInfo('qu_bank_version', '2.4.1') so the server browser shows it.
  • Cross-resource handshake. A framework resource sets setr qu_framework qbox at start so other resources can detect it.
  • One-shot migration. An admin command toggles a debug flag without editing server.cfg.
Script-set convars do not survive a restart

SetConvar writes the in-memory value. It does not edit server.cfg. If you want the change to persist across restarts, the value belongs in server.cfg in the first place.

Step 4

Patterns that pay off

Outcome: you have three reusable convar patterns you can drop into any resource.

Pattern 1: debug toggle

Replace every print('[qu_bank] ...') with a guarded helper. Server admins flip the flag without editing your code.

-- server/main.lua
local DEBUG = GetConvarInt('qu_debug', 0) == 1

local function log(fmt, ...)
    if not DEBUG then return end
    print(('[qu_bank] '):format() .. fmt:format(...))
end

log('starting up, balance=%d', GetConvarInt('qu_starting_balance', 1000))
# server.cfg
set qu_debug 1   # flip to 0 in production
Pattern 2: feature flag for risky rollouts

Ship a new code path behind a flag. Default off. Flip on per server.

-- server/main.lua
local USE_NEW_AUDIT_FLOW = GetConvarInt('qu_audit_v2', 0) == 1

RegisterNetEvent('qu_bank:withdraw', function(amount)
    if USE_NEW_AUDIT_FLOW then
        return withdrawV2(source, amount)
    end
    return withdrawV1(source, amount)
end)

If v2 breaks, the admin sets qu_audit_v2 0 and restarts the resource. No code revert.

Pattern 3: setr for the UI

The client UI needs to know per-server caps. Don't trigger an event, just publish a replicated convar.

# server.cfg
setr qu_max_stack 64
setr qu_robbery_cooldown 600

-- client/main.lua
local CFG = {
    maxStack       = GetConvarInt('qu_max_stack', 20),
    robberyCool    = GetConvarInt('qu_robbery_cooldown', 300),
}

SendNUIMessage({ type = 'config', payload = CFG })

Step 5

Where convars come from

Outcome: you know the load order and the four ways a convar can get its value.

Sources, in order
  1. server.cfg, loaded once at server boot. Any set/setr/sets line lands here.
  2. exec other.cfg, nested config files. Use this to split secrets into secrets.cfg (gitignored) and keep server.cfg in version control.
  3. Server console, type set qu_debug 1 at the live console. Takes effect immediately for new reads. Lost on restart unless persisted in cfg.
  4. SetConvar* from a resource, same lifetime as the console form, lost on restart.

Read precedence is "last write wins". A console set overrides what was in server.cfg for the rest of the session.

How to inspect a convar at runtime

Type the convar name into the live server console:

cfx> qu_starting_balance
 "qu_starting_balance" is "5000"
 default: ""
 type: string

If the convar is unset, the console returns No such command qu_starting_balance., that means your GetConvarInt call is falling back to the default, which is fine if intentional.

Splitting secrets out of server.cfg

Add exec secrets.cfg to server.cfg, put your set steam_webApiKey ... and set rcon_password ... in secrets.cfg, and add secrets.cfg to .gitignore. Now you can commit your config without leaking keys.

Mistakes & gotchas

1. GetConvar with no default returns the empty string, not nil

Code like if GetConvar('qu_debug') then ... is always true, the empty string "" is truthy in Lua. Always pass a default and compare to it explicitly: GetConvar('qu_debug', '0') == '1' or GetConvarInt('qu_debug', 0) == 1.

2. Reading a server-only convar from the client

Standard set convars are invisible to clients. The client gets the default. If both sides need to read the same value, the server must setr it (replicated), or you trigger an event with the value at client connect.

3. Putting secrets in sets

sets values are public, they show up at http://<server_ip>:<port>/info.json and on the master server list. sets is for Discord, tags, project name. Never for tokens, webhooks, or API keys.

4. Editing server.cfg and expecting a hot reload

server.cfg is parsed once at boot. Edits take effect on next restart. To change a value live, type the set command into the live console, then mirror it in server.cfg so it persists.

5. Using GetConvarInt on a non-numeric value

GetConvarInt('qu_db_name', 0) on a value of "qu_bank" returns 0, the default. There is no error, just silent fallback. Use GetConvar for strings and GetConvarInt for numbers, never mix them.

6. Forgetting that script-set convars do not persist

SetConvar('qu_debug', '1') is in-memory only. If you actually want the change to survive a restart, write it to server.cfg. The script API is for runtime overrides and computed-at-boot values, not durable config.

Recap

  • Convars are FiveM's built-in config, one keyword in server.cfg, one function call in Lua.
  • Three flavors. set = server-only. setr = replicated to clients (read-only on the client). sets = public, indexed by the server browser.
  • Read with GetConvar(name, default) or GetConvarInt(name, default). Always pass a default.
  • Write from script with SetConvar, SetConvarReplicated, SetConvarServerInfo, but the values are in-memory and lost on restart.
  • Use convars for one-line toggles, debug flags, and per-server tuning. Use config.lua for structured config and tables.
  • Never put secrets in sets. Split secrets into secrets.cfg via exec and gitignore that file.

Frameworks: ESX, QBCore, Qbox + ox_lib

Real FiveM servers usually run a framework. A framework answers the same boring questions for every script: who is this player, how much money do they have, what job are they on, what inventory do they carry. This lesson reads like a translation guide. Learn the idea once, then see how ESX, QBCore, and Qbox spell it.

  • Start with the mental model.

  • Read ESX, QBCore, and Qbox as translations of the same idea.

  • Treat export-heavy details as optional depth until you need them.

The mental model: why every framework exists

Forget the branding for a second. A framework exists so every script on your server answers the same three questions in the same way:

  • Who is this player? A unique identifier (license, Steam hex) plus a character id.
  • What's their state? Money, job, gang, permissions, inventory.
  • How do I read or change it? A standard set of functions every resource agrees on.

Without a framework, every script would invent its own player, money, and job system. Then nothing would match. A framework fixes that. It gives you one shared player shape, one shared money shape, and one shared way to read and change them.

The three frameworks, one sentence each

ESX -> older, still common. QBCore -> the default shape many public scripts target. Qbox -> very close to QBCore, but cleaner and more ox_*-focused. Same big idea, different spelling.

First read

What a player object gives you

A framework player object is not magic. It is the server's structured record for one connected character. Your first goal is only to read three fields safely:

Question
Typical field
Who is this character?
identifier or citizenid
How much money do they have?
money, cash, or bank
What job are they on?
job.name and job.grade

If you can read those three without changing anything, you are ready for the ESX, QBCore, and Qbox translations below.

Optional advanced: exports

Exports: the one FiveM concept all three assume

Before the framework-specific details, you need one FiveM idea: exports. A resource can expose a function with exports('name', fn). Another resource can call it with exports.resourceName:name(args). It is a direct function call on the same side, server to server or client to client. Every framework uses exports heavily, so once you understand this pattern, the docs get much easier to read.

If this section slows you down, skip to the first player-object example for your framework. You can use a framework before you fully understand every export pattern.

Framework 1

ESX (es_extended)

ESX is one of the oldest big FiveM roleplay frameworks. It uses a slightly older style, and it thinks about money as accounts. Cash is one thing. Bank is another account. Black money can be another account too. The maintained version people usually mean today is ESX Legacy.

Get the core object

The ESX player object is called xPlayer. Key fields and methods:

  • xPlayer.identifier: the license:xxxx string, stable per Rockstar account.
  • xPlayer.getMoney(): cash on hand. xPlayer.addMoney(n) / removeMoney(n).
  • xPlayer.getAccount('bank').money: bank balance. addAccountMoney('bank', n) / removeAccountMoney('bank', n).
  • xPlayer.job.name / .grade / .grade_label. Change with xPlayer.setJob('police', 1).
  • xPlayer.getInventoryItem('bread'): returns { name, count, label, weight }. (Legacy ESX inventory; modern installs usually swap to ox_inventory, see below.)
Open the ESX client-event reference
ESX's oldest trap: getSharedObject timing

Old tutorials often show a long polling loop to get ESX. That was common years ago. Modern ESX Legacy supports the direct export shown above. So if you see the old loop in a snippet, think "old style" -> not "impossible to use."

Framework 2

QBCore (qb-core)

QBCore became popular as a cleaner alternative to ESX. Instead of an account-style money system, it stores money directly on the player object. Jobs also include an on-duty flag. If you ever open a resource called qb-something, it is almost always written for QBCore.

Get the core object

QBCore's player object has PlayerData (the data) and Functions (the setters). Key paths:

  • Player.PlayerData.citizenid: stable per character.
  • Player.PlayerData.charinfo.firstname / .lastname / .phone / .birthdate.
  • Player.PlayerData.money.cash / .bank / .crypto.
  • Player.PlayerData.job.name / .label / .grade.level / .grade.name / .onduty.
  • Player.PlayerData.metadata: hunger, thirst, stress, whatever you add.

Change state: go through Functions

Open the QBCore client-event reference
Always null-check the player

If a player is still joining, GetPlayer can return nil. So always write if not Player then return end. The same habit applies to ESX and Qbox too.

RemoveMoney hands you back a yes-or-no, so read it

On QBCore and Qbox, RemoveMoney returns a boolean: true if the player had enough and the money came out, false if they were short and nothing happened. If you ignore that return value, you can hand someone an item even when they could not actually pay for it. That is one of the most common ways shops leak money.

ESX does not return a reliable boolean here, so on ESX you check the balance first with xPlayer.getMoney() before calling removeMoney. Either way the rule is the same: confirm the money left the account before you give anything in return.

Framework 3

Qbox (qbx_core)

Qbox is a community-maintained fork of QBCore. The important beginner takeaway is simple: it keeps a very similar player shape, but it swaps much of the older qb-* stack for the newer ox_* stack like ox_lib, ox_inventory, and ox_target. If you are starting fresh, many developers now prefer Qbox for that reason.

No GetCoreObject: direct exports

Player object has the same shape as QBCore: Player.PlayerData.* and Player.Functions.AddMoney / RemoveMoney / SetJob / SetJobDuty all exist and behave identically. What changes:

  • Core object access is direct: exports.qbx_core:GetPlayer(src), no GetCoreObject() round-trip.
  • Client-side too: local PlayerData = exports.qbx_core:GetPlayerData() (after the load event).
  • Jobs and gangs are typed configs in qbx_core/shared/jobs.lua, cleaner than QBCore's.
  • Inventory is always ox_inventory (not qb-inventory).
  • Callbacks, menus, and notifications are always ox_lib (not qb-menu / qb-input).
Open the Qbox client-event reference
Qbox keeps QBCore's events on purpose

Qbox keeps the familiar QBCore client events on purpose, so many QBCore-style scripts still feel familiar. The main thing you usually change is how you get the player or player data -> use exports.qbx_core:GetPlayer or GetPlayerData instead of exports['qb-core']:GetCoreObject().

Utility library

ox_lib (Overextended)

ox_lib is not a framework. It's a utility library that sits on top of whichever framework you chose, giving you one consistent API for notifications, menus, callbacks, input dialogs, progress bars, and cached values. Qbox requires it. QBCore servers usually add it. ESX servers increasingly do too. Learning ox_lib once saves you from learning qb-menu, esx_menu_default, and mythic_notify separately.

QBCore and Qbox are byte-identical below the core-object line. ESX is the odd one out, method syntax on xPlayer plus the accounts model (cash is not an account, bank is). The notify call is identical on all three because ox_lib is framework-agnostic.

Trust rules still apply

The framework handles persistence, the HUD, and the audit trail. It does not handle trust. args[1] came from the client. Validate type, range, positivity. A negative amount with removeMoney is just addMoney in disguise.

ox_inventory: the shared inventory layer

All three frameworks used to ship their own inventory. Today, all three can (and usually do) run ox_inventory instead. Qbox requires it. QBCore servers install it by choice. ESX has first-class ox_inventory integration via the framework's config. One inventory API, regardless of who's underneath.

Items live in a config, not in your code

Items are declared in ox_inventory/data/items.lua with label, weight, stack size, and optional metadata rules. Adding an item name that isn't declared will log a warning and silently fail. When AddItem mysteriously doesn't work, check the item list first.

Choosing a framework in 2026

Honest decision tree for a brand-new server, no existing installation to respect:

Most scripts you want are labelled "QBCore"
Install Qbox. 90%+ of QBCore scripts run unchanged because Qbox keeps the QBCore event names and player-object shape. You get a modern codebase plus the huge ecosystem.
Most scripts you want are labelled "ESX"
Install ESX Legacy. Don't install "ESX 1.2" or any pre-Legacy version, those are abandoned. Legacy is the only maintained line.
You want maximum performance and modern Lua patterns
Qbox. Statebag-driven state, strict typing, best-in-class internals. Short-term cost: smaller script library than QBCore, but growing fast.
You're hiring developers who learned on YouTube in 2021–2023
QBCore. The dominant tutorial target. Developers land running. Swap to Qbox later if performance demands it, most scripts port in an afternoon.
You're joining an existing server
Whatever that server runs. Don't migrate someone else's framework without a very good reason, the migration cost is always higher than you think.
Install ox_lib on every single one

Whichever framework you pick, install ox_lib. It's the one library where the right answer is "yes, add it" regardless of context. Every serious script either uses it or has a flag to use it.

Detecting the framework at runtime

So far every snippet has hardcoded one framework. That's fine for your own server. But the moment you write a script that someone else might run, ESX or QBCore or Qbox, hardcoding becomes a maintenance trap. The fix is one native: GetResourceState(name). It returns a string telling you whether a resource exists and where it is in the lifecycle.

The state you care about almost always is 'started'. 'missing' means the resource isn't installed at all. 'starting' means it's loading but not ready, you'd hit nil exports if you called it now. So the safe check is one line:

Order matters: check Qbox before QBCore

A Qbox server may also have qb-core installed as a stub for compatibility. Always check the more specific one first (Qbox), then fall back. Otherwise Qbox servers get detected as QBCore and your shim calls the wrong export.

When to detect: not at script load

Resources start in dependency order, but your script's top-level code runs the moment the file is parsed. If you call GetResourceState at the top of server.lua, the framework may still be 'starting'. Two solid patterns:

Pattern A is best for scripts where the framework is optional. Pattern B is best when you list the framework in dependencies {}, FiveM guarantees it's started before yours.

Why this matters in practice

Every script that supports both QBCore and Qbox out of the box doubles its reach. Every public release on GitHub gets twice the install rate. Hardcoding one framework is a self-imposed limit, not a technical one.

The compatibility shim

Detection is half the answer. The other half is hiding the framework behind one uniform API so the rest of your code never has to if Framework == 'esx' then .... The pattern is a single shim file that exposes Framework.GetPlayer, Framework.AddMoney, Framework.GetJob, and dispatches internally.

The shim ends at semantic differences

A shim works because all three frameworks model the same idea (player, money, job). It breaks the moment you reach into framework-specific data: ESX accounts beyond cash/bank, QBCore metadata flags, Qbox's typed job configs. For those, expose them as separate optional methods (Framework.GetMetadata) and document which framework supports each.

What about ox_lib's existing shim?

If you're already on ox_lib, check lib.framework in newer versions, it covers a subset of this for you. But hand-rolling your own is cheap and gives you exact control over the API surface. Most public scripts ship their own.

Common mistakes across all three

Writing directly to money fields

Player.PlayerData.money.cash = 1000 doesn't persist, doesn't update the HUD, doesn't notify other resources. Always AddMoney / RemoveMoney (QBCore / Qbox) or addMoney / addAccountMoney (ESX).

Using the player object before the load event

Client-side, GetPlayerData() returns empty until the framework fires esx:playerLoaded / QBCore:Client:OnPlayerLoaded. Wire your init to that event, not to onResourceStart.

Client-side permission checks

if PlayerData.job.name == 'police' then doAdminStuff() end on the client. A cheater overwrites that table in seconds. Permission checks always server-side, always against the framework's authoritative copy.

Confusing QBCore's and Qbox's exports

Snippets that start with QBCore = exports['qb-core']:GetCoreObject() target QBCore. On Qbox, that export doesn't exist, replace with exports.qbx_core:GetPlayer / GetPlayerData. Everything downstream usually works unchanged.

Mixing legacy ESX with modern accessors

Old ESX tutorials use ESX = nil; TriggerEvent('esx:getSharedObject', ...). Modern ESX Legacy supports exports['es_extended']:getSharedObject() synchronously. Either works, don't mix both.

Hardcoding one framework in a public script

Shipping a script that calls exports['qb-core']:GetCoreObject() at the top of server.lua locks every install to QBCore. The same script with GetResourceState detection plus a 60-line shim runs on QBCore and Qbox unchanged, and adds ESX with one extra branch. Detection beats branching at install time, install instructions get shorter, support tickets drop, and the script doesn't bit-rot when the user migrates frameworks.

Calling GetResourceState at the top of the file

Top-level code runs the moment the file parses, before onResourceStart. The framework may still be 'starting', not 'started'. Either detect lazily (on first use) or detect inside onResourceStart for your own resource, with the framework listed in dependencies {}.

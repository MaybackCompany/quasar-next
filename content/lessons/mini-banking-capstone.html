Cheat sheet

Banking safety rules

  • Client requests. It sends an amount request, not the final balance.
  • Server validates. It checks type, range, player, and available funds.
  • SQL stores. Every player value uses a ? placeholder.
In one minute

What you build, in one minute.

  • One MySQL table holds each player's bank balance, keyed by identifier.
  • The client sends a short message saying "I want to deposit 100." The server reads the real balance from MySQL and decides.
  • Every SQL query uses ? placeholders so a cheater cannot inject SQL.

What you will do, in plain English

You are going to write the smallest possible banking script. The player types /bank, picks deposit or withdraw, types an amount, and the server saves the new balance in a database. That is it. No transfers, no fees, no menus inside menus. The point is to learn the safe shape: client asks, server checks, database stores.

Three words you will see a lot in this lesson, defined here once:

  • Source. A number FiveM gives every connected player. The server uses it to know who is sending an event. source = 4 means "player number 4 right now." It changes every reconnect.
  • Identifier. A long string that stays the same forever for one player, for example license:8b1a9.... We use this as the database key so the balance survives reconnects and server restarts.
  • Parameterized query. A SQL command where values are replaced by ? and the real values are passed in a separate list. The database driver inserts them safely. This is the rule that blocks SQL injection.

One more pair you must keep straight: player money is the cash in the player's pocket, handled by the framework (Qbox, ESX, QBCore). Bank balance is the number we keep in our own MySQL table. Deposit moves money from the first into the second. Withdraw goes the other way.

The bank-teller analogy

Imagine a real bank window. You slide a paper through the slot that says "deposit 100." The teller does not believe the number you wrote on the paper. The teller looks at your account file, counts the cash you handed over, and only then writes a new balance. The client is the slip of paper. The server is the teller. The database is the account file.

If the teller trusted whatever number was on the paper, the bank would be empty by lunchtime. That is what happens with bad FiveM code.

The classic mistake, and the safe shape

Below is the single most common mistake a new dev makes when they first wire MySQL to a money script. On the right is the same feature, written safely. Read both sides slowly.

Don't do this

The server trusts the amount the client sent and pastes raw values into the SQL string. Two huge holes at once.

What goes wrong: the client can send any newBal it wants ("set my balance to one billion"). The string-built SQL also lets a crafted value run extra commands, the classic SQL injection.

Do this

The server reads the real balance from MySQL, decides the new number itself, and uses ? placeholders for every value.

What is fixed: the client only says how much, never the new total. Every value goes through a ?, so the database driver escapes it for us. The key is the stable identifier, not source.

Step 1

Create the resource folder

Make a new folder inside resources/[local]/ called qu_bank. Inside it, create three empty files: fxmanifest.lua, server.lua, and client.lua.

The line server_script '@oxmysql/lib/MySQL.lua' loads the helper that gives us MySQL.scalar, MySQL.update, and friends on the server. Without it those names are nil.

Step 2

Create the database table

Run this SQL once against your server's MySQL database. Use HeidiSQL, phpMyAdmin, or your editor of choice. Copy and paste exactly.

Three things to notice. identifier is the primary key, so each player gets one row. balance is UNSIGNED, so it cannot go below zero at the database level, that is a second safety net. updated_at updates itself, useful for spotting suspicious changes later.

Step 3

Server: the balance reader

Open server.lua. Add the helper that gets the player's identifier and fetches their current balance. If the player has never had a row, return zero.

The client will use lib.callback.await('qu:bank:balance') to ask "how much do I have right now?" The server looks it up by identifier, never by anything the client supplied. MySQL.scalar.await returns one cell from the first row, exactly what we want for a single number.

Step 4

Server: deposit, with full validation

This is the heart of the lesson. Read each check before pasting.

Notice the order: validate the amount, find the player, read the cash, take the cash, write the new balance. If any earlier step fails, we return early with a clear message and never touch the database. Both values in the SQL go through ? placeholders, no string concatenation anywhere.

Step 5

Server: withdraw, the mirror image

Same shape as deposit. Validate, find player, read the real balance from MySQL, check the balance is enough, write, then hand over the cash. The amount the client sent never reaches the SQL string raw, it only reaches the parameter list.

Step 7

Client: the menu

The client's only jobs are: ask the server for the latest balance, draw the menu, send the chosen action and amount to the server, and show the result message. No numbers are decided on the client. The menu is a window, not a vault.

ActionExpected resultHow to verify
Open /bank, click Deposit, type 100 while you have at least $100 cash. Green notify "Deposited $100. New balance $100." Cash drops by 100. Reopen /bank, the title shows the new balance. Run SELECT balance FROM qu_bank WHERE identifier = ? in HeidiSQL with your license, the row matches.
Open /bank, click Withdraw, type 50. Green notify "Withdrew $50. New balance $50." Cash rises by 50. Menu title updates on next open. Database row equals 50.
From the F8 console run TriggerServerEvent('qu:bank:deposit', -100), then again with 'free'. Red notify "Invalid amount." both times. No row change. No error in the server console. Database row is still 50. Server console shows no SQL error, the validation rejected before the query ran.

The third test is the one beginners skip. It is the proof that your validation works against an attacker, not just against a polite menu.

Why we never call RemoveMoney with the client's number directly

Player money lives inside the framework. Even though RemoveMoney('cash', amount) looks like a safe call, the amount here came from the network. We validated it first with validAmount, and we read the current cash from Player.PlayerData, never from the client. That two-step is the difference between a safe handler and a duplication exploit.

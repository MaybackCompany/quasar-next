Script config files, structure and patterns

Write a config.lua that every other file in your resource reads from — so you change one number and the entire script updates, without hunting through five files.

You'll ship
A config.lua file with sections for general settings, feature toggles, and per-item configuration that two other files consume correctly.
Time
About 20 minutes.
You'll learn
How to structure a config table, why nested tables are safer than flat variables, and how to validate config values at startup.
Prereqs
You have written at least one Lua file that uses variables.
In 3 bullets

You will build a config system that:

  • Stores all tunable values in one file so users never touch your logic code.
  • Uses a nested Lua table so related settings stay grouped and self-documenting.
  • Validates itself on resource start and prints clear warnings if something is wrong.

Step by step

Lesson 03: Script config files, structure and patterns

Start here

What this is
A Lua file that returns a table of settings other files read at startup.
Example: Config.HealAmount = 200 in config.lua, used by server.lua.
Where to find it
In the root of every well-written resource, named config.lua or shared/config.lua.
Example: open any qb-core resource and look for Config = {}.
How to use it
Load it first in fxmanifest.lua, then reference Config.Foo everywhere else.
Example: shared_script 'config.lua' goes above other scripts.
What to do now
Write one config table with three sections and read it from two files.
Example: general settings, feature toggles, and a list of items.

Outcome

You will change one number in config.lua, restart the resource, and see both the server and client behaviour update without touching any other file.

Why it matters

Without a config file, every setting is hard-coded inside logic files. A server owner who wants to change the heal cooldown has to open server.lua and risk breaking the code. A config file separates tuning from logic — the owner edits config.lua and never sees a function body.

Vocabulary

Config table
A Lua table that holds all tunable values for a resource.
Example: Config = { HealAmount = 200, Cooldown = 60 }.
Shared script
A script declared in fxmanifest.lua that runs on both server and client.
Example: shared_script 'config.lua' makes Config available everywhere.
Startup validation
Code that checks config values when the resource starts and warns about impossible combinations.
Example: assert type(Config.HealAmount) == 'number', 'HealAmount must be a number'.

Exercise

Add a new item to Config.Items called defibrillator with heal = 200 and time = 15000. Then loop through all items in server.lua and print each name and heal value.

Common failures

  • attempt to index a nil value (global 'Config') — config.lua is not loaded before the file that uses it. Move shared_script 'config.lua' above all other scripts in fxmanifest.lua.
  • attempt to index a nil value (field 'Heal') — you referenced Config.Heal but the table does not have that key. Check the spelling and nesting.
  • Config values seem ignored — you are reading a global variable instead of Config.Something. Lua does not warn about this.
  • Assertion fails silently — assert only prints if the condition is false. Add a print before the assert to see which line runs.

Refactor

Wrap the validation in a function ValidateConfig() that loops over expected keys and types, so adding a new setting automatically gets checked without writing a new assert line.

Cheat sheet

Config patterns

  • Use nested tables. Config.Feature.Setting is clearer than Config_Feature_Setting.
  • Validate at startup. Crash early with a clear message, not silently with wrong values.
  • One file, loaded first. Never spread config across multiple files.

The big idea, in one minute

Every FiveM resource has knobs: cooldowns, amounts, feature toggles, locale strings. If those knobs are scattered across ten files, tuning a server means editing logic code — which is how bugs get introduced. A config file collects all the knobs in one place. The server owner turns the knobs. The developer writes the logic. Those two people are often the same person at different times. The config file respects that.

Think of a car dashboard.

The dashboard has knobs and buttons: volume, temperature, fan speed. You do not open the hood to adjust the radio. The config file is the dashboard. The logic files — server.lua, client.lua — are the engine. The owner touches the dashboard. The mechanic touches the engine. The config file draws that line.

Step 1

Create the resource and manifest

The manifest declares what loads and in what order. Config must come first.

  1. Create resources/qu_config_demo/.
  2. Create fxmanifest.lua inside it.
  3. Add the following lines: 
    fx_version 'cerulean'
game 'gta5'

shared_script 'config.lua'
server_script 'server.lua'
client_script 'client.lua'
  4. Save. shared_script means config.lua runs on both sides.
Step 2

Write the config file

The config is a plain Lua table. Groups keep it organised.

  1. Create config.lua in the resource folder.
  2. Paste the Config table from the Minimal Example above.
  3. Add one more setting of your own: a greeting message string.
  4. Save. Your config is ready to be read.
Step 3

Read config from the server

Server code uses Config just like any other global table. No special import needed.

  1. Create server.lua.
  2. Paste: 
    print(('[qu_config_demo] Server sees Heal.Amount = %d')
    :format(Config.Heal.Amount))

for _, item in ipairs(Config.Items) do
    print(('[qu_config_demo] Item: %s heals %d in %d ms')
        :format(item.name, item.heal, item.time))
end
  3. Save. The loop prints every item in the config.
Step 4

Read config from the client

The client sees the same Config table because shared_script runs on both sides.

  1. Create client.lua.
  2. Paste: 
    if Config.General.Debug then
    print('[qu_config_demo] Client: debug mode is ON')
end

print(('[qu_config_demo] Client sees %d items in config')
    :format(#Config.Items))
  3. Save. All four files are ready.
Step 5

Run it and change a value

The real test: change one number and watch every file react.

  1. Add ensure qu_config_demo to server.cfg and start the resource.
  2. Check the console for the config loaded message and the item list.
  3. Change Config.Heal.Amount from 200 to 150 in config.lua. Save.
  4. Restart the resource with ensure qu_config_demo in the console.
  5. The printed value changes to 150. You changed nothing in server.lua or client.lua.
Recap

Two tests, in order

Run each test and confirm before moving on.

  1. Test 1, config loads.
    • Action: start the resource.
    • Expected: console prints [qu_config_demo] Config loaded: HealAmount=200 Cooldown=60.
    • How to verify: both server and client prints appear with no nil errors.
  2. Test 2, broken config fails early.
    • Action: change Config.Heal.Amount to "two hundred" (a string). Restart.
    • Expected: the resource fails to start and prints Heal.Amount must be a number.
    • How to verify: the assert message appears, and the rest of the resource does not load.
Why use shared_script instead of two separate config files?

If you have two config files, one for server and one for client, you will eventually change one and forget the other. Then the server thinks the cooldown is 60 seconds but the client shows a timer counting from 30. One shared file means one source of truth.

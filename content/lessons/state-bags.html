State bags: Entity(x).state, Player(s).state, GlobalState

Half the time you reach for TriggerClientEvent, what you actually want is a state bag: a key/value blob attached to an entity, a player, or the whole server, automatically replicated across the network. Set the value once on the server; every client sees it. Hook a handler; you get notified when it changes.

Why state bags exist

Imagine you want every client to know which players are on duty as police. The naive approach is "loop every player, TriggerClientEvent every other player." That is O(N²) traffic and you have to re-broadcast on every join, leave, and toggle. With state bags you write Player(src).state.duty = 'police' on the server. The engine syncs the value to every interested client, automatically, including new joiners. You read it from any client with Player(target).state.duty.

Mental model

State bags are FiveM's built-in shared key-value store. Three scopes: per-entity (sticks to the vehicle/ped/object), per-player (sticks to the player session), and global (one bag for the whole server). Server can write any of them; clients can write only their own player bag and entities they own.

Step 1

Entity state: data on a specific vehicle, ped, or object

Outcome: tag a vehicle as "rented" on the server; any client can read the tag without an extra event.

What :set(key, value, replicate) does
  • Direct assignment Entity(v).state.k = value works and replicates by default from the server. From a client it does not replicate.
  • :set('k', value, true) forces replication. Use the explicit form on the client when you want a value to reach the server.
  • :set('k', value, false) keeps the value local, useful when the server wants to remember something without telling clients.
Step 2

Player state: data tied to a session

Outcome: track each player's duty status so any other client can render their blip differently.

Notice: from the client we use Player(serverId), not LocalPlayer. LocalPlayer.state reads the local player's bag. Player(N).state reads any player's bag by their server ID.

Recap 1

Entity vs player scope

For each piece of data, write down the right scope: vehicle plate, vehicle owner, weapon currently equipped by the player, current job, current shop's stock, server-wide opening hours.

Step 3

GlobalState: one bag for the whole server

Outcome: server flips a flag; every client reads it without you wiring a single event.

GlobalState is server-write, client-read. Clients trying to write GlobalState are silently ignored.

Step 4

Reacting to changes with AddStateBagChangeHandler

Outcome: react to a state change with one handler instead of polling on a thread.

Signature breakdown
  • AddStateBagChangeHandler(key, bagName, handler)
  • key: the state key to watch. Pass nil to fire on any key.
  • bagName: an exact bag name like 'player:5' to scope to one bag, or nil for all bags of any kind.
  • handler(bagName, key, value, reserved, replicated): fires after the change is committed. Use GetPlayerFromStateBagName or GetEntityFromStateBagName to resolve the owner.
Handlers are global per resource

If you register the same key handler twice in the same resource, it fires twice. Track the return value and unregister with RemoveStateBagChangeHandler(cookie) on resource stop if you need to clean up.

Step 5

The shallow-update trap

Outcome: you avoid the silent bug where a sub-key never replicates.

State bags serialize the entire value at the top-level key. Mutating a nested table does not trigger replication.

The flat-key form is what the official docs recommend. It avoids both the mutation bug and the cost of re-serializing the whole metadata blob on every fuel tick.

Common mistakes

Mutating a nested table and expecting replication

Covered above. The serializer only sees writes at the top-level key. Either reassign the top key, or use flat 'a:b' keys.

Using state bags for high-frequency data

State bag writes serialize and broadcast. For values that change every frame (fuel, speedometer), use a local thread or a NUI message instead. State bags are for "this value changed once and now everyone needs to know."

Trying to write GlobalState from a client

Silently ignored. If a client needs to update server-wide state, fire a regular event and let the server validate, then write GlobalState.

Forgetting to clean up bags on entity delete

Entity bags die with the entity, so this one is safe. But if you keep references to a deleted entity in a registry, the next read returns nil for everything, not a stale value.

Using LocalPlayer.state on the server

LocalPlayer only exists on the client. From the server, use Player(source).state.

Trusting client-set state

Client Player(serverId).state:set('k', v, true) replicates to the server. A modded client can lie about anything. Validate every value the same way you would for a regular net event.

Cheat sheet

Entity(v).state.k = value
Per-entity bag. Server writes replicate by default; client writes are local unless you use :set(k, v, true).
Player(src).state.k = value
Per-player bag. Server can write any. Clients can write only their own (use LocalPlayer.state).
LocalPlayer.state.k
Shorthand for the local player's bag. Client only.
GlobalState.k = value
Server-only write, client-read. One bag for the whole server.
state:set(key, value, replicate)
Explicit form. Pass true to replicate, false to keep local.
AddStateBagChangeHandler(key, bagName, handler)
Fire when the state changes. Pass nil for either argument to widen the match.
GetPlayerFromStateBagName(name) / GetEntityFromStateBagName(name)
Resolve the player ID or entity handle from inside a change handler.
Flat keys
Prefer state['meta:fuel'] over state.meta.fuel: nested mutation does not replicate.

What you learned

  • Three scopes: entity state, player state, GlobalState, and which fits which kind of data
  • The replication rule: server writes broadcast by default; client writes are local unless you pass true to :set
  • How to react to changes with AddStateBagChangeHandler instead of polling on a thread
  • The shallow-update trap: only top-level key writes serialize the bag
  • That state bags still cross the trust boundary, client-set state must be validated server-side

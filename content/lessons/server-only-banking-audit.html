Why audit logs matter

An audit log is a record of every action that changed something important. In a bank, that means every deposit and every withdrawal gets a row in a separate table. Why bother?

  • Debugging. A player says "I had $5000 and now I have $200." Without logs, you have no idea what happened. With logs, you can see every action, timestamp, and amount.
  • Cheater detection. If someone exploits a money duplication bug, the audit log shows it: 50 deposits of $999999 in one second. Without logs, you only find out when the economy is already broken.
  • Rollback. If a bug gives everyone $1M, the audit log lets you find and reverse exactly the broken transactions instead of wiping everyone's balance.

The rule is simple: if money moves, a row gets written. No exceptions.

Step 1

The migration: create both tables on start

Outcome: your resource creates the accounts and audit_log tables automatically the first time it runs.

A migration is a SQL script that sets up or changes your database schema. The cleanest pattern for FiveM: run your CREATE TABLE statements when the resource starts. Because we use IF NOT EXISTS, they are safe to run every time.

Two tables:

  • qu_accounts -- one row per player. UNIQUE on identifier means a player can only have one account. balance is an integer (no floating point, no cents -- keep it simple).
  • qu_audit_log -- one row per action. action is a short string like 'deposit' or 'withdraw'. note is optional context. created_at is auto-stamped.

Why two separate MySQL.query calls?

oxmysql's MySQL.query runs one statement at a time. If you try to put both CREATE TABLE statements in one string separated by a semicolon, it may fail. One query per call is the safe pattern.

Step 2

Ensure the account exists

Outcome: a helper that creates a player's account row if it does not exist yet.

Before you can deposit or check a balance, the player needs a row in qu_accounts. This helper tries to insert one; if the identifier already exists (thanks to UNIQUE), it does nothing.

INSERT IGNORE means "try to insert, but if the row already exists (because of the UNIQUE constraint), silently skip it". No error, no duplicate. This is the safe way to handle first-time players.

Step 3

Server reads the balance from DB

Outcome: the server always reads balance from the database, never from a Lua variable.

MySQL.scalar.await returns a single value: the first column of the first row. Perfect for reading one number. We call ensureAccount first so there is always a row to read.

Never cache the balance in a Lua upvalue

"I'll store it in a local table for speed." The moment any other resource modifies the database, your cached number is wrong. Always read from the database when making money decisions. One extra query is cheap. A wrong balance is a bug report.

Step 4

Deposit with audit log

Outcome: depositing money updates the balance and writes an audit row in a single transaction, both succeed together or neither does.

Three things every money move must guarantee. This deposit covers all three:

  1. Math in the database, not in Lua. balance = balance + ? is a single SQL statement. You never read the old balance into Lua, add, and write back, that would be a classic read-modify-write race when two deposits land in the same millisecond.
  2. Balance change and audit row commit together. MySQL.transaction.await wraps both statements in a single transaction. If the audit INSERT fails (table missing, disk full, restart mid-write) the UPDATE is rolled back. You never end up with money credited but no evidence of where it came from.
  3. The transaction is awaited. The function does not return until MySQL has committed (or rolled back). On a server restart the moment after this call, the on-disk state is consistent, there is no half-finished write floating in oxmysql's queue.
Why the unawaited MySQL.insert from Lesson 21 is a bug here

Fire-and-forget audit logging works for tutorials and breaks the moment a server crash, restart, or oxmysql queue overflow drops the in-flight insert. The balance change has already committed; the audit row never lands. You wake up to a player whose balance jumped with no audit trail. Always await any write that is part of a money decision.

The client NEVER sends the amount

In Lesson 21, the client sent an amount to the server. That was fine for learning events, but in a real system it is a security hole. A cheater can fire TriggerServerEvent('qu:bank:deposit', 999999). In this version, the server decides the amount based on game logic (job payout, item sold, etc.). The client only says "I want to deposit" -- the server figures out how much.

Step 5

Withdraw with balance check

Outcome: withdrawals deny on insufficient funds atomically, the SQL guard rejects two concurrent overdrafts that would both pass a separate balance check.

This is the version that survives concurrent traffic. The naive pattern from Lesson 21, read balance, validate in Lua, then UPDATE, is a textbook time-of-check / time-of-use (TOCTOU) race. Picture two withdrawals arriving at the same instant, both reading a $300 balance, both validating $200 ≤ $300, both running the UPDATE. The player ends up at -$100 and a duplication bug just shipped.

The fix is to push the check into the UPDATE itself with WHERE balance >= ?. MySQL holds a row lock for the duration of the statement, so the second withdrawal sees the post-debit balance and the guard rejects it. affectedRows tells you which side of the guard you landed on: 1 means the row was updated, 0 means the guard rejected it.

The pattern, generalised

Any time you want to mutate a row only if it is in a particular state, do the check in the WHERE clause and trust affectedRows. Reading state into Lua, deciding, then writing back is a race waiting to happen, for money, for inventory, for anything where two clients can fight over the same row.

Step 6

Wire commands and read the audit log

Outcome: players can deposit, withdraw, check balance, and view their transaction history.

DATE_FORMAT in the SELECT

DATE_FORMAT(created_at, "%Y-%m-%d %H:%i") tells MySQL to format the timestamp as a readable string like 2026-05-03 14:30. Without it, you get a raw timestamp object that Lua doesn't know how to print nicely. The AS ts part gives the formatted column a short name you can use in Lua: row.ts.

The key: the client fires TriggerServerEvent('qu:bank:deposit') with no arguments. The server uses its own JOB_PAYOUT constant. A cheater who changes the event cannot manipulate the amount because the amount is never read from the event payload.

Common mistakes

Client sending "new balance" to server

If the client sends TriggerServerEvent('qu:bank:deposit', 999999) and the server trusts it, the player has infinite money. The server must compute the new balance itself using balance = balance + ? in SQL or by reading from the database. The client's only job is to say "I want to do this action." The amount comes from the server's game logic.

Not checking balance before withdraw

If your withdraw handler does UPDATE SET balance = balance - ? without first checking balance >= amount, a player can end up with a negative balance. Negative balances break everything downstream: other scripts that check "can the player afford this?" get confused, and exploiters can game transfer logic. Always check first, then update.

Missing audit log on a transfer

Every money movement needs a log row. If you add a transfer feature later and only log the sender's side, you have an incomplete trail. When debugging, you see money leave player A but no record of it arriving at player B. Log both sides: one transfer_out row for the sender and one transfer_in row for the receiver.

Database connection timeout

If the database goes down or the connection times out, MySQL.query.await can return nil or throw an error. Wrap critical operations in a nil check: local result = MySQL.query.await(...) if not result then -- handle failure end. For production servers, monitor your MariaDB uptime and set mysql_connection_string's connectTimeout parameter if needed.

Cheat sheet: SQL banking patterns

MySQL.insert(sql, params)
Insert a row. Returns the new id. Use for audit log entries.
MySQL.update.await(sql, params)
Update rows and wait for it to finish. Returns number of affected rows.
MySQL.query.await(sql, params)
Run a SELECT and get the rows back as a Lua table.
INSERT IGNORE INTO
Insert if not exists, skip silently if UNIQUE constraint blocks it.
balance = balance + ?
Let the database do the math. Avoids race conditions in Lua.
WHERE identifier = ?
Always filter by player. Never update all rows by accident.
DATE_FORMAT(col, '%Y-%m-%d %H:%i')
Format a timestamp into a readable string inside the query.
Audit pattern
Every money change = one UPDATE to balance + one INSERT to audit log. No exceptions.

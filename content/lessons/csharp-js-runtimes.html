C# and JavaScript runtimes, when to use which

FiveM runs code in three languages: Lua, C#, and JavaScript. Learn what each runtime does, when to pick one over another, and why most scripts you will write stay in Lua.

You'll ship
A resource that uses Lua for game logic and a C# server script for a background number-crunching task, proving you can mix runtimes.
Time
About 35 minutes.
You'll learn
The four FiveM runtimes (Lua server, Lua client, C#, JS), their performance profiles, and the real-world reasons to choose one over another.
Prereqs
You have written Lua scripts that run on a FiveM server. C# knowledge is helpful but not required.
In 3 bullets

You will understand the FiveM runtimes so that:

  • You know Lua handles 95% of game scripts — chat commands, UI, inventory — and is the right default.
  • You know C# is for heavy computation, external APIs, and code you want to protect from casual reading.
  • You know JavaScript exists for NUI (browser-based UI) and nothing else on a FiveM server.

Where to find it
Runtime selection happens in fxmanifest.lua and in resource folder structure.
Example: server_script 'main.lua' vs server_script 'Main.cs'.
How to use it
Pick Lua unless you have a specific reason to use C# or JS.
Example: start with Lua, profile the performance, and only then consider C# for the bottleneck.
What to do now
Write one Lua resource and one C# resource that print from the same event.
Example: both resources listen for playerConnecting and log it.

Outcome

You will start a resource with a C# server script alongside your Lua scripts and see both runtimes produce output in the same server console.

Why it matters

Choosing the wrong runtime wastes hours. C# scripts need compilation and have slower iteration. JavaScript on the server is rarely the right call. Knowing the tradeoffs upfront saves you from rewriting a half-built resource.

Vocabulary

Runtime
The engine that executes your code inside the FiveM process.
Example: the Lua runtime runs RegisterCommand; the C# runtime runs EventHandlers.
Mono
The open-source .NET framework that FiveM's C# runtime is built on. Not the same as modern .NET.
Example: C# scripts target .NET Framework 4.5, not .NET 8.
NUI
Native UI — the Chromium-based browser layer that renders HTML/JS/CSS interfaces in-game.
Example: the phone screen in qb-phone is an NUI page written in JS.

Checklist

  1. Create the Lua resource qu_lua_runtime with a standard manifest and server.lua.
  2. Ensure it starts and prints its message in the console.
  3. Create the C# resource qu_csharp_runtime with the correct folder structure.
  4. Write the C# class, compile it into a .dll, and place it in the resource.
  5. Configure the manifest to load the C# assembly with server_script 'MyResource.net.dll'.
  6. Ensure it and confirm both runtimes print their messages.

Exercise

Add a Tick handler in the C# resource that increments a counter every second, and a Lua command that reads that counter via an export. Prove the two runtimes share memory.

Expected output: Typing /checkcounter in chat prints the C# tick counter value from Lua.

Common failures

  • C# resource does not start — the .dll targets the wrong .NET version. Must target .NET Framework 4.5, not .NET Core or .NET 8.
  • Could not load file or assembly — missing dependencies. All referenced NuGet packages must be bundled in the resource folder.
  • C# debug prints do not appear — use Debug.WriteLine not Console.WriteLine. FiveM captures the debug stream.
  • JavaScript server scripts seem to work but silently fail on native calls — the JS runtime has incomplete native bindings. Stick to Lua or C# for server code.

Refactor

Extract the C# tick counter into a standalone class with dependency injection so other C# resources can consume it without a Lua bridge.

Cheat sheet

Which runtime for what?

  • Lua (server + client). Chat commands, inventory, admin tools, framework glue. 95% of all scripts.
  • C# (server only). Database heavy lifting, external HTTP APIs, Discord bots, obfuscation-sensitive paid scripts.
  • JavaScript (NUI only). Phone screens, HUD overlays, tablet UIs. Runs in a browser, not in the game engine.

The big idea, in one minute

FiveM is a Lua-first platform. The vast majority of community scripts, frameworks, and tutorials are in Lua. The C# runtime exists for developers who need strong typing, compiled performance, or code obfuscation. JavaScript runs inside NUI browsers and has no direct access to game natives. Choosing Lua means choosing the path of least resistance — more examples, more community help, faster iteration. Choosing C# means choosing compile-time safety and raw speed, at the cost of a longer edit-test cycle.

Think of a kitchen.

Lua is a chef's knife. You use it for almost everything. C# is a stand mixer — perfect for heavy dough that would tire your arm, but you do not pull it out to chop an onion. JavaScript is the toaster. It does one thing (NUI) and does it well. Bringing the toaster to the counter to stir soup is not going to work.

Step 1

Create the Lua resource baseline

The Lua resource is the control group. It proves your server can load a resource.

  1. Create resources/qu_lua_runtime/ with fxmanifest.lua and server.lua.
  2. The manifest needs only fx_version 'cerulean', game 'gta5', and server_script 'server.lua'.
  3. Paste the Lua handler from the Minimal Example. Ensure the resource and confirm the print appears.
Step 2

Set up the C# resource folder

A C# resource needs a specific folder layout: the .dll file, a manifest, and any dependencies.

  1. Create resources/qu_csharp_runtime/.
  2. Create fxmanifest.lua with server_script 'QuCSharpRuntime.net.dll'.
  3. The .dll goes in the resource root, next to the manifest.
Step 3

Write and compile the C# script

The C# class inherits from BaseScript. The constructor registers event handlers.

  1. Create a new Class Library project targeting .NET Framework 4.5.
  2. Add the CitizenFX.Core NuGet package (the FiveM server API).
  3. Paste the C# code from the Minimal Example into your class file.
  4. Build the project. Find the output .dll and copy it into resources/qu_csharp_runtime/.
Step 4

Run and compare

Both resources start and print from the same event, proving the runtimes coexist.

  1. Ensure both resources in server.cfg: ensure qu_lua_runtime and ensure qu_csharp_runtime.
  2. Start or restart the server.
  3. Look for both hello messages in the console.
  4. Now change the print message in the Lua file. Save and ensure qu_lua_runtime — it updates instantly.
  5. Change the print in the C# file. You must rebuild the .dll, copy it, and restart the resource. Notice the difference in iteration speed.
Recap

Two tests, in order

Run each test and confirm before deciding which runtime to use for your next project.

  1. Test 1, both runtimes alive.
    • Action: start the server with both resources ensured.
    • Expected: two hello messages appear in the console, one from Lua, one from C#.
    • How to verify: the messages have different resource names in brackets.
  2. Test 2, iteration speed comparison.
    • Action: change the print message in each resource and redeploy.
    • Expected: Lua updates in under 3 seconds. C# requires a rebuild and takes 20+ seconds.
    • How to verify: time both from save to console output. This is why Lua is the default.
Can I call C# functions from Lua?

Yes, through server exports. In C# you mark a method with the [Export] attribute, and in Lua you call exports['resource_name'].MethodName(). The reverse (calling Lua from C#) uses TriggerEvent which both runtimes understand.

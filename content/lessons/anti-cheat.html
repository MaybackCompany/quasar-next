Anti-cheat patterns

No client-side check is real security. Every "give me money" event a cheater finds is one your server should have rejected. This lesson covers the four patterns that stop 95 percent of attacks: server authority, rate limiting, sanity checks, and audit logging.

Before you start

Mental model: the bouncer checks the wristband every time

A bouncer at a club does not say "you came in once, you must be VIP forever". They check your wristband at every door, every drink, every backstage corridor. Anti-cheat is the same: the server re-verifies on every request, no matter how innocent the client looks.

Cheaters use F8 console + tools to fire any net event. If your event blindly trusts what the client sends, money disappears. If your server checks "is this player actually near the ATM, do they actually own this car, did they fire this 5 ms ago", the cheat fails.

The golden rule

Never trust a number that came from the client. Look it up server-side. If the client says "give me $5000 from job X", the server checks: do you have job X, when was your last payout, are you on duty, what is the configured payout amount? Then the server hands the money to the server-side wallet.

Anti-cheat at scale: budget, buckets, durability

Steps 1–4 stop a bored cheater. They do not stop a coordinated attack: 50 alts hammering 8 events each at 200 Hz, while one of them spams a webhook from a different IP to drown your audit channel. Step 5 is the version that runs on a 96-slot server during a stress-test raid.

5.1 · Threat model and the 2 ms validation budget

Set a hard rule: every server net-event handler must complete in under 2 ms p95 even with full validation. If validation takes 8 ms and a cheater fires 200 events/sec, you have used 1600 ms of CPU per second, one core is gone, and that is before the rest of the server runs. Cheap-to-detect checks come first; expensive ones run only after the cheap ones pass.

The ACE cache turns a 50 us native call into a 50 ns table read after the first hit. With 96 players each triggering an admin-gated event 4 times/sec, that is 96×4 = 384 calls/sec; uncached this costs 19 ms/sec, cached it costs 0.02 ms/sec.

5.2 · Token bucket beats last-call map

The lastCall[k] map from Step 2 enforces an average rate but lets a cheater fire 60 events in the same window-edge millisecond and still pass, each call sees an empty lastCall, then writes now faster than the gate can fire. A token bucket gives the burst budget back honestly: capacity tokens, refilled at rate per second.

Capacity 4 with refill 0.5/sec means: a player gets 4 instant calls, then must wait 2 seconds for each next one. A cheater firing 1000 events/sec uses the first 4 tokens in 4 ms and is rejected for the remaining 996 events, honest users with bursty UI never notice.

5.3 · Audit pipeline durability

Discord webhooks rate-limit at 30 requests/minute per webhook. A cheater spamming 50 invalid events triggers 50 audit calls; calls 31–50 are silently dropped by Discord, the cheater wins, and you never see the alert. Batch audits with a 1-second flush window and a ring buffer that survives resource restarts.

The onResourceStop drain prevents losing the last second of violations during a hot-reload. Failed POSTs re-queue and back off 5 seconds, if the webhook is down, audits buffer in memory until it is back.

For the three-strikes system referenced in Level Up, persist strikes by license identifier (not src) with a 24h TTL so a banned account cannot reset by reconnecting:

Recap 3

Senior depth: trust budget

You have a 2 ms validation window and a token bucket that beats last-call gating under burst. You can answer “what stops a hooked client from bypassing this check?” with: server-authoritative state, shape validation at the boundary, throttle per-account not per-socket, and audit pipeline durability so the trail survives the node going away. Rate limiting without an audit log is theater.

Common mistakes

Trusting client coords

Anything TriggerServerEvent('foo', myCoords) sends is editable. Read the coords server-side from the player ped instead.

Letting the client decide whether they are admin

Permissions live in the server's ACE list. IsPlayerAceAllowed(src, 'qu.admin'), never a client-side flag.

Rate limit map that grows forever

Always clean up entries on playerDropped. Otherwise after a week your lastCall table has 50000 stale keys.

Webhook URL in code

Use GetConvar with a setr in server.cfg. Pushing a webhook URL to git is a leak that lets anyone spam your staff channel.

Soft drops

If a player triggers an obvious cheat (negative price, impossible distance), DropPlayer them. They will reconnect; staff have time to ban based on identifiers in the audit log.

What you learned

  • The server is the only authority that matters; the client is a remote rendering layer.
  • Every net event needs type, range, rate, and state checks before doing work.
  • Coords from the server side are honest; client-supplied coords are not.
  • Drop the player on obvious cheats; log the violation with identifiers to Discord.
  • Secrets live in convars, never in code or git history.

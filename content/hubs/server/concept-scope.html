Server Owner Path › Lesson 2

Concept and scope

Write the server before buying the server. A non-technical owner can make good build decisions when the city promise, first-hour loop, must-have systems, budget, and later list are written on one page.

Outcome
A completed one-page city brief that a developer can quote from.
Why
Vague servers become random script piles. Clear scope keeps money, dependencies, and staff work under control.
Artifact
Server concept, audience, first-hour player loop, v1 systems, later systems, budget decisions, and dependency notes.
Duration
30-45 minutes.
Prerequisites
Owner toolkit folder from Lesson 1 and one server idea.
Pass condition
server-brief.md explains the first hour, v1 scope, later list, and budget limits.
Owner rule

If you cannot explain what a new player does in the first hour, you are not ready to install scripts. Plan the first hour first, then buy only the systems that support it.

The sample server concept

Use this worked example as the shape of your own brief. The point is not to copy the theme. The point is to see how a server idea turns into install decisions.

Worked concept
Server name
Harbor County RP
Target player
Adults who want relaxed civilian roleplay, legal work, small business ownership, and police interaction without a heavy gang focus.
Promise
A new player can arrive, get an ID, earn first money, meet one staff-guided contact, and understand the main city loop in under 60 minutes.
Tone
Grounded, helpful, low-drama, economy-first.
First social moment
The player meets a city guide or taxi driver at spawn, then gets sent to a legal starter job.
Step 1

Create the planning file

Outcome: a file named server-brief.md exists before any shopping starts.

  1. Open your project folder. Example: Documents\FiveM\HarborCounty\.
  2. Create a folder named planning.
  3. Open VS Code.
  4. Click File → Open Folder → choose Documents\FiveM\HarborCounty\planning.
  5. Create a new file named server-brief.md.

Expected output

VS Code's left file tree shows planning/server-brief.md. To confirm, the planning folder is expanded with server-brief.md selected.

If the file becomes server-brief.md.txt

Windows is hiding file extensions. Open File Explorer → View → Show → turn on File name extensions, then rename it to server-brief.md.

Step 2

Write the first-hour player loop

Outcome: the server has one clear beginner route.

Do not start with "we need gangs, shops, drugs, apartments, and ten jobs." Start with what a brand-new player sees and does.

Expected proof

A staff member can read the six lines and explain where the player goes without asking you in Discord.

Must-have now vs later

This is where owners save money. A must-have system is required for the first-hour loop. A later system can be exciting and still be wrong for v1.

DecisionHarbor County exampleWhy
Must have nowCharacter creation, identity, spawn, legal job, banking, inventory, vehicle rental, basic policeThese support the first hour and data persistence.
Must have nowtxAdmin roles, test accounts, database backup, rollback notesThese protect the server before public players arrive.
LaterCustom housing, full businesses, complex crime, casino, racing, advanced phone appsThey add dependencies and staff load before the core loop is proven.
Not for this conceptMilitary roleplay, zombie survival, heavy gang warsThey fight the grounded civilian promise.
Step 3

Make dependency and budget decisions

Outcome: every planned paid script has a reason and a risk note.

Before purchase, ask three questions: does it support the first hour, what framework does it require, and who will maintain it when it breaks?

Owner safety rule

Never buy a script only because a showcase video looks good. Require framework support, install docs, dependency list, update history, and a rollback plan.

Completed example

This is the minimum quality bar for your own brief. It is specific enough for a developer to estimate and narrow enough for staff to test.

Questions to ask a developer

  1. Which framework best supports this first-hour loop: QBCore, ESX, or Qbox?
  2. Which systems replace default framework behavior, such as inventory, vehicles, housing, or phone?
  3. Which scripts require SQL imports?
  4. Which scripts must be installed before others?
  5. What can be staged locally before touching production?
  6. What is the rollback command or restore step if the script fails?

Failure recovery

The concept is still too broad

Cut the v1 list until it has one player loop, one economy route, one staff route, and one rollback plan. Anything that does not support those four goes to later.

Staff disagree about the server direction

Use the first-hour loop as the tie-breaker. The system that improves the first hour wins. The system that only sounds exciting waits.

A bought script does not fit the framework

Do not force it into production. Put it in later, ask the seller about framework support, and keep the server on the planned stack.

Pass condition

You are done when server-brief.md names the audience, first-hour loop, framework direction, must-have v1 list, later list, budget cap, dependency risks, and pass condition. To confirm, the completed server-brief.md is open in VS Code with every section filled.

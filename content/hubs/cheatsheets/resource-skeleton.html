Resource skeleton: fxmanifest, client, server

The smallest FiveM resource that runs. Five fields in fxmanifest.lua, three script folders, and one rule about which side gets which file.

fx_version cerulean
Latest stable manifest version. Use it on every new resource.
game gta5
Required field. FiveM refuses to start without it.
Server is the truth
Money, inventory, jobs, and SQL run on the server. Never the client.
Shared is read-only
Config that both sides need. Never put logic that mutates state in shared.
Pin this card

Open the resource skeleton next to your editor every time you start a new script. Five minutes saved per resource adds up.

Folder layout

my_resource/
├── fxmanifest.lua
├── config.lua              -- shared config, read by both sides
├── client/
│   └── main.lua            -- client logic, UI, ped, vehicle, draw
├── server/
│   └── main.lua            -- truth: money, inventory, SQL, validation
├── shared/
│   └── locales.lua         -- read-only data for both sides
└── html/                   -- optional NUI assets
    ├── index.html
    ├── style.css
    └── app.js

fxmanifest.lua minimum

fx_version 'cerulean'
game 'gta5'

author 'You'
description 'Short description of what this resource does'
version '1.0.0'

shared_scripts {
  'config.lua',
  'shared/*.lua',
}

client_scripts {
  'client/*.lua',
}

server_scripts {
  '@oxmysql/lib/MySQL.lua',     -- if you use the database
  'server/*.lua',
}

ui_page 'html/index.html'

files {
  'html/index.html',
  'html/style.css',
  'html/app.js',
}

dependencies {
  'ox_lib',
  'oxmysql',
}

The trust rule

Never trust the client

The client can edit any local file and lie about anything. Money, inventory, jobs, item give, vehicle spawn, all of it must be validated on the server, even if the client UI says otherwise.

-- WRONG: client decides money
RegisterNetEvent('shop:buy', function(item, price)
  exports['ox_inventory']:AddItem(source, item, 1)   -- client said price, no check
end)

-- RIGHT: server decides
RegisterNetEvent('shop:buy', function(itemId)
  local source = source
  local item = Config.Items[itemId]                  -- server-side config
  if not item then return end
  if not removeMoney(source, item.price) then return end
  exports['ox_inventory']:AddItem(source, item.name, 1)
end)

Dependencies

Use the dependencies block in fxmanifest. The server refuses to start your resource until its dependencies are running.

dependencies {
  'ox_lib',
  'oxmysql',
  'ox_inventory',
}
One-line check

If GetResourceState('ox_lib') ~= 'started', your resource is missing a dependency. Check the console for the exact missing name.

Resource lifecycle

AddEventHandler('onResourceStart', function(name)
  if name ~= GetCurrentResourceName() then return end
  -- runs once when this resource starts
  print('my_resource started')
end)

AddEventHandler('onResourceStop', function(name)
  if name ~= GetCurrentResourceName() then return end
  -- cleanup: delete entities, save state, close DB connections
end)
Always cleanup on stop

Spawned peds, vehicles, blips, and target zones leak across restarts if you do not delete them in onResourceStop. The entity cleanup cheatsheet covers this in detail.

Common mistakes

SQL files in client_scripts

oxmysql exposes server-only functions. Loading @oxmysql/lib/MySQL.lua on the client throws attempt to index a nil value (global 'MySQL').

Wrong fx_version

Old resources use 'adamant' or 'bodacious'. New resources should use 'cerulean'.

lua54 'yes' is no longer needed

As of June 2025, Lua 5.3 was removed and every resource runs Lua 5.4. The lua54 'yes' manifest line is harmless but redundant, the official docs now mark it deprecated. You will still see it in older resources; do not flag it as a bug, just stop adding it to new ones.

Missing files block for NUI

If you ship HTML or images, list them in files {}. Otherwise the client cannot fetch them and the NUI page is blank.

Module 06A - Planning

Outcome

You will write a small feature spec before code changes.

Why it matters

A spec makes rollback possible. Logs make broken rules visible.

Vocabulary

Spec
A short contract for a feature.
Example: inputs, rules, outputs, logs.
Rollback
A fast path back to the last safe version.
Example: disable one resource.
Metric
A number that proves behavior.
Example: failedDeposits counts rejects.

Setup

Resource

Minimal example

Expected output: [QUIET][spec][ready] prints the rollback plan.

Production checklist

  1. Confirm the code is on the correct side.
  2. Validate every payload before acting.
  3. Log with [QUIET][module][action].
  4. Add one cleanup or rollback path.
  5. Copy the expected output into your test note.

Things to try

  • Write a spec for a reward command.
  • Add one reject log.
  • Name the fastest rollback step.

Common failure modes

  • attempt to call a nil value means a native, export, or helper is unavailable on this side.
  • No such export GetCoreObject in resource qb-core means the dependency is missing or ensured late.
  • attempt to index a nil value means a table, player object, or entity handle is missing.
  • SQL error: Table 'qu_spec_plan' doesn't exist means the migration did not run.

Module 05A - Frameworks

Outcome

You will read player data on the server and avoid client authority.

Why it matters

Money and jobs must be trusted server data. Client values are only requests.

Vocabulary

Player object
Framework data plus helper functions.
Example: Player.PlayerData.job.name.
Server write
A change made by the server after validation.
Example: AddMoney runs server-side.
Client request
A message asking the server to act.
Example: a client asks to claim a reward.

Setup

Resource
qu_framework_patterns
Side
server
File path
resources/qu_framework_patterns/server.lua
Ensure order
ensure qu_framework_patterns

Minimal example

Expected output: [QUIET][framework][job] prints the server-owned job name.

Production checklist

  1. Confirm the code is on the correct side.
  2. Validate every payload before acting.
  3. Log with [QUIET][module][action].
  4. Add one cleanup or rollback path.
  5. Copy the expected output into your test note.

Things to try

  • Read cash without changing it.
  • Add an ESX comparison note.
  • Reject the event when Player is missing.

Common failure modes

  • attempt to call a nil value means a native, export, or helper is unavailable on this side.
  • No such export GetCoreObject in resource qb-core means the dependency is missing or ensured late.
  • attempt to index a nil value means a table, player object, or entity handle is missing.
  • SQL error: Table 'qu_framework_patterns' doesn't exist means the migration did not run.

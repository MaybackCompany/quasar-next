Day 3 · Events & net messaging · ~30 min

Events and net messaging

This is the day where FiveM stops being "just Lua" and becomes multiplayer scripting. One side talks. The other side answers. That is all an event is. If you are starting from zero, keep one sentence in your head the whole time: the client asks, the server decides.

Day 3 in plain English

Imagine pressing a doorbell.

The button press is the event.

The wire carrying the signal is the net message.

The person inside deciding whether to open the door is the server.

That is the whole mental model. Events are just named signals. Some stay on one machine. Some travel over the wire between client and server.

Concepts

Local vs net events

A plain AddEventHandler is local: it only fires on the machine that raised it. A net event travels between client and server. You mark it with RegisterNetEvent('name') so FiveM knows it's allowed over the wire.

Who talks to whom

From the client, call TriggerServerEvent('name', args...) to send to the server. From the server, call TriggerClientEvent('name', targetId, args...) to reach one player (or -1 for everyone).

Never trust the client

When the client says "give me $1000", assume they're lying. The server is the source of truth. Check permissions, check state, then act. This is the single most-broken rule in beginner code.

The goal of Lab 1 -> in plain English

You are making your own doorbell signal called qu:greet.

  • First you tell FiveM that this event name is allowed over the network.
  • Then you attach a handler, the code that runs when the event fires.
  • When the test fires the event with the name Alex, your handler should print greeted: Alex.

One event name. One handler. One print. Small on purpose.

Lab 1 · Register a net event with a handler

Register a net event called qu:greet. When it fires, print greeted: <name>, where name is the argument.

What to do:

  1. Write RegisterNetEvent('qu:greet').
  2. Below it, write AddEventHandler('qu:greet', function(name) ... end).
  3. Inside the function, print greeted: plus the name value.
  4. Run the test. The harness will fire the event for you.
Pattern. RegisterNetEvent('qu:greet') then AddEventHandler('qu:greet', function(name) print('greeted: ' .. name) end).

If the handler test fails: you may have registered the event but never attached the function, or you attached the function to the wrong event name.

The goal of Lab 2 -> in plain English

Now you flip the direction. Instead of the test firing an event for you, your own command will fire one.

  • The player types /ping.
  • Your command handler runs.
  • Inside the handler, you send a message to the server: qu:ping with the word pong.

This is the pattern behind most real features: command or key press on the client, validation and real action on the server.

Lab 2 · Client-side command that pings the server

Register a /ping command. When the player runs it, fire a server event called qu:ping with the string pong as its one argument, then print sent pong to server so the learner sees a clear success message.

What to do:

  1. Keep the existing RegisterCommand('ping', ...) wrapper.
  2. Inside the function, call TriggerServerEvent('qu:ping', 'pong').
  3. After that, print sent pong to server so the result is visible to the learner.

If the call-count test fails: you probably used TriggerEvent instead of TriggerServerEvent. TriggerEvent stays local. TriggerServerEvent crosses the wall.

If the argument test fails: check the exact event name and exact string. The test wants qu:ping and pong.

Before you move on

Stop and say these out loud:

  • "RegisterNetEvent marks an event name as allowed over the network."
  • "AddEventHandler attaches the code that runs when the event fires."
  • "TriggerServerEvent sends a message from client to server."
  • "The client can ask, but the server is the source of truth."

If those four lines feel natural now, Day 4 will make much more sense.

What you learned today

  • RegisterNetEvent declares an event name so FiveM allows it to travel over the network between client and server.
  • AddEventHandler attaches a function that runs when that event fires. No handler means the event does nothing.
  • TriggerServerEvent sends a message from the client to the server. This is how the client asks the server to do something.
  • The server is the source of truth. Always validate data on the server side before acting on it.

Common mistakes

Handler never fires, no error in console

You forgot to call RegisterNetEvent('yourEventName') before AddEventHandler. Without registering the event first, FiveM silently blocks it from arriving over the network. Always register, then attach.

Event fires but the wrong handler runs (or nothing happens)

The event name string does not match. 'qu:greet' and 'qu:Greet' are two different events. Copy-paste the exact same string into both RegisterNetEvent and AddEventHandler. One typo and the wires are disconnected.

Server event handler runs but anyone can trigger it from the client

You are not checking source on the server. A cheater can fire any registered net event from their client. On the server side, always verify that source is a valid player and that the request makes sense before changing any state.

Day 5 · ESX · ~35 min

ESX fundamentals

ESX (es_extended) is one of the big FiveM frameworks. Framework code can look heavy at first, but today's idea is still small: get the player, check the cash, move money between two buckets. That is all your /deposit command is doing.

Day 5 in plain English

Framework code can look scary because the object names are longer, but the idea is still tiny.

Get the player object. Check their cash. If they have enough, move money from one bucket to another. If they do not, stop.

That is the whole day.

How to use Day 5

Read the plain-English section first. Then do Lab 1 until the guard rule makes sense. Only after that should you do the callback lab. The balance check matters more than the framework name.

Concepts

Server side: ESX.GetPlayerFromId(src)

Takes a server ID, returns an xPlayer object. That object has methods like xPlayer.addMoney(amount), xPlayer.removeMoney(amount), xPlayer.getMoney(), xPlayer.addAccountMoney(account, amount). All money mutations happen through these.

Accounts are just named pots

An xPlayer has several accounts: money (on-hand cash), bank, and black_money. To move cash to bank: xPlayer.removeMoney(100) then xPlayer.addAccountMoney('bank', 100). No magic, just arithmetic on two accounts.

Callbacks: client asks, server answers

When the client needs a value the server owns (like "how much money do I have?"), the server exposes a callback: ESX.RegisterServerCallback('name', function(source, cb) cb(value) end). The client calls it and gets the value back.

Guard the balance before you move it

Day 3 said "never trust the client". Here's where that rule earns its keep. A cheater can spam /deposit with 0 cash; a beginner's script happily calls removeMoney(100) and drives cash negative, then adds 100 to the bank from thin air. Always check first: if xPlayer.getMoney() < 100 then return end. No balance, no deposit. The lab tests this, you can't skip it.

Lab 1 · /deposit command (ESX)

Write a server-side /deposit command. Step one: guard with xPlayer.getMoney() >= 100, otherwise return. Step two: remove 100 cash, add 100 to bank. The sandbox's mock xPlayer starts with 500 cash and 1000 bank. Valid run: 400 cash, 1100 bank. If cash is below 100 the deposit must not happen, bank must not grow.

What to do: fetch xPlayer, write the guard first, then write the two money-moving lines. Guard first is non-negotiable.

Read this like a story: start at cash = 500 and bank = 1000. A valid deposit ends at cash = 400 and bank = 1100. An invalid deposit ends with both numbers unchanged.

If the guard test fails (Guard: with 50 cash, deposit is refused): you are moving money before checking the balance. Add if xPlayer.getMoney() < 100 then return end above the removeMoney line. Do not delete the guard just to make the test green -> the whole reason for this lesson is to stop players from using commands their cash cannot cover.

Lab 2 · Server callback for current cash

Register a server callback called qu:getCash. When a client invokes it, return the current cash of the invoking player. Signature: ESX.RegisterServerCallback(name, function(source, cb) cb(value) end).

What to do: register the callback, get the player from source, then hand the cash value into cb(...).

The test will pretend to be the client, call the callback, and then print cash=500. That print line is your learner-facing proof that the answer came back from the server correctly.

What you learned today

  • ESX.GetPlayerFromId(source) gets a player object on the server. That object holds all the methods you need to read and change money.
  • xPlayer methods like getMoney(), removeMoney(), and addAccountMoney() handle cash and bank. All money changes go through these.
  • Server callbacks let the client ask the server a question and get an answer back. ESX.RegisterServerCallback on the server, ESX.TriggerServerCallback on the client.
  • Always check the balance before removing money. If you skip the guard, a player with zero cash can generate money from nothing.

Common mistakes

Script error: attempt to call method 'getMoney' (a nil value)

You are calling ESX player methods on the client side. xPlayer objects only exist on the server. If your code is in a client script, you cannot use xPlayer.getMoney(). Use a server callback to ask the server for the value instead.

Player's cash goes negative after running the command

You called removeMoney without checking the balance first. Add if xPlayer.getMoney() < 100 then return end before any money removal. No balance check means a player with $0 can deposit $100 they do not have.

Server event handler works but any random client can trigger it

You forgot to use source in the server event. Always pass source to ESX.GetPlayerFromId so the server knows which player is making the request. Never let the client send their own player ID as an argument -- a cheater will send someone else's ID.

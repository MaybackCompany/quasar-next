Qbox and one script for all three

This is the hardest day in the starter kit, so slow down. You are not learning a brand-new money idea now. You are learning how one script can survive in three framework worlds. Keep repeating the same sentence: same script idea, different framework vocabulary.

Day 7 in plain English

You already know the real rule from Days 5 and 6: no balance check, no deposit.

Day 7 only adds one extra idea: a server might run ESX, QBCore, or Qbox, so your script may need to detect which world it is in before it chooses the right code branch.

Think of it like this:

The question is always the same: "which framework am I talking to?"

The answer is one of three names: qbox, qbcore, or esx.

The behavior is still the same: guard first, move money second.

How to use Day 7

This is the hardest starter day, so do it in two passes. First make the Qbox-only version work. Then do the runtime detection. Only after that should you attempt the one-command-for-all-frameworks lab.

Concepts

Side-by-side cheat sheet: ESX uses xPlayer.getMoney(). QBCore uses Player.Functions.GetMoney('cash'). Qbox usually starts with exports.qbx_core:GetPlayer(src), then reads cash from Player.PlayerData.money.cash. Different names, same idea: look up the player, read the cash, guard first.

Qbox is export-first

No more GetCoreObject() dance. You use a direct export to get the player object: exports.qbx_core:GetPlayer(src). After that, the returned player looks very close to QBCore: read data from Player.PlayerData and mutate money with Player.Functions.AddMoney / RemoveMoney. Cleaner entry point, familiar player shape.

Inventory lives in ox_inventory

Qbox delegates inventory to ox_inventory. Adding an item: exports.ox_inventory:AddItem(src, 'water', 1). Same API on ESX or QBCore if you switch them to ox_inventory as well.

Framework-agnostic code

You can ship one resource that works on all three. At top of file, detect: if exports.qbx_core exists, prefer Qbox. Else if QBCore is set, use QB. Else if ESX is set, use ESX. Each branch has its own money API, and each one still needs the balance check.

Third framework, same rule

Qbox still lets you write bad money logic if you skip the guard. Check before you move: if Player.PlayerData.money.cash < 100 then return end. Then use Player.Functions.RemoveMoney and Player.Functions.AddMoney. Three frameworks, three APIs, one rule: never trust the client, never skip the balance check.

Lab 1 · /deposit, the Qbox way

Step one: fetch the player with exports.qbx_core:GetPlayer(source) and guard on Player.PlayerData.money.cash >= 100. Step two: use Player.Functions.RemoveMoney('cash', 100, ...) and Player.Functions.AddMoney('bank', 100, ...).

What to do:

  1. Get the player once and store it in a local variable.
  2. Check Player.PlayerData.money.cash before you touch anything else.
  3. If cash is below 100, use return immediately.
  4. Only after the guard passes, remove 100 from cash and add 100 to bank.

If the guard test fails (Guard: with 50 cash, deposit is refused): read Player.PlayerData.money.cash and place return above the money calls. If you remove money first and check later, the test is right to fail.

Lab 2 · Detect the framework at runtime

Write a function detectFramework() that returns one of the strings 'qbox', 'qbcore', or 'esx', picking in that priority order. Then print framework: <name>.

In the sandbox, all three frameworks are defined at once at first, so your detection needs priority. Then the tests flip the sandbox into Qbox-only, QBCore-only, and ESX-only modes one at a time, so every branch gets verified for real.

What to do:

  1. Check Qbox first: if exports and exports.qbx_core exist, return 'qbox'.
  2. If not, check QBCore.
  3. If not, check ESX.
  4. Print the result so the test can read it.

Lab 3 · One /deposit, all three frameworks

This is the payoff. Write one /deposit command that works on Qbox, QBCore, or ESX, whichever the server is running. Reuse your detectFramework() from Lab 2. Each branch fetches the right player, checks the balance, and moves 100 cash to bank. Same rule every time: if the player has less than 100 cash, return without touching money.

When the file first loads, all three frameworks are present, so your detection priority still matters: qbox → qbcore → esx. After that, the tests switch the sandbox into one framework at a time and check both the happy path and the guard path on every branch.

Build order:

  1. Write detectFramework() first and keep it tiny.
  2. Inside the command, store the framework name in fw.
  3. Write the Qbox branch fully before touching the others.
  4. Copy the shape of the branch, not the exact API names, into QBCore and ESX.
  5. End every branch the same way: if not enough cash, return.

The sandbox now checks all three branches, not just Qbox. If QBCore or ESX fail, compare your branch against the earlier day pages. Same rule, different property names. Fix one branch at a time until every happy-path and guard test is green.

What you learned today

  • exports.qbx_core:GetPlayer(source) is how Qbox gives you the player object. No GetCoreObject() dance needed -- just a direct export call.
  • Runtime detection checks which framework is loaded by testing if exports.qbx_core, QBCore, or ESX exist, in that priority order.
  • One script can support all three frameworks by branching on the detected framework name. Same logic, different API spellings in each branch.
  • Shared patterns (guard the balance, get the player, move money) reduce duplicate code. The rule never changes across frameworks: check first, act second.

Common mistakes

Detection always returns the wrong framework

You are checking frameworks in the wrong order. If you check ESX before QBCore, a server running both will always pick ESX. Check Qbox first, then QBCore, then ESX. Priority order matters because some servers load multiple framework globals.

Script crashes with "attempt to index a nil value" on a server with no framework

You forgot to handle the "no framework found" case. If none of the three globals exist, your detection returns nil and the next line that tries to use it crashes. Always add a fallback branch (like return 'none') and handle it gracefully in your command.

Deposit works on QBCore but crashes on ESX (or vice versa)

You mixed up method names between frameworks. ESX uses xPlayer.removeMoney(100) (lowercase, no account name for cash). QBCore uses Player.Functions.RemoveMoney('cash', 100) (uppercase, with account string). Each branch must use its own framework's exact API. Do not copy-paste one branch into another without changing the method names.

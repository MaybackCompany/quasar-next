Day 6 · QBCore · ~35 min

QBCore fundamentals

QBCore (qb-core) is the other big framework. The names are different from ESX, but the idea is the same: get the player, check the cash, move money safely. Today you write the same /deposit command again, this time in QBCore language.

Day 6 in plain English

Day 6 is Day 5 with different method names.

You still get the player. You still check the cash first. You still move 100 from cash to bank. You still stop immediately if the player cannot afford it.

The beginner trap here is thinking the longer names mean a different idea. They do not. Only the API spelling changed.

How to use Day 6

Think of this as the QBCore translation day. Reuse the same mental model from Day 5: get player, check cash, move money. If the names change but the logic stays the same, you are understanding it correctly.

Concepts

Get the core, then the player

At the top of a server script you usually write local QBCore = exports['qb-core']:GetCoreObject(). Then QBCore.Functions.GetPlayer(source) gives you the player object.

PlayerData and Functions

The QB player has two halves. Player.PlayerData is read-only data (citizenid, money, job, items). Player.Functions has the methods that change that data: AddMoney(account, amount, reason), RemoveMoney(...), GetMoney(account), AddItem(...).

Callbacks: CreateCallback

Same pattern as ESX, different name. QBCore.Functions.CreateCallback('name', function(source, cb, ...) cb(value) end).

Same rule, different API: check the balance first

QBCore's Player.Functions.RemoveMoney is happy to drive cash negative if you let it. Always check before you remove: if Player.Functions.GetMoney('cash') < 100 then return end. Money moves live on the server; the server is the one thing cheaters can't patch. Your /deposit must refuse when the player can't afford it.

Lab 1 · /deposit, the QB way

Register a /deposit. Step one: guard with GetMoney('cash') >= 100, otherwise return. Step two: RemoveMoney('cash', 100) and AddMoney('bank', 100). Starting state: 500 cash, 1000 bank. Guard must refuse when cash is below 100.

What to do: get Player, guard on GetMoney('cash'), then move the money with RemoveMoney and AddMoney.

The before-and-after story is the same as ESX: start at 500 / 1000, end at 400 / 1100 if the guard passes, or leave both numbers alone if the guard fails.

If the guard test fails (Guard: with 50 cash, deposit is refused): you are calling RemoveMoney before checking the balance. Read Player.PlayerData.money.cash first, or call Player.Functions.GetMoney('cash'). If it is below 100, use return. If you skip that check, RemoveMoney can push the balance negative in this sandbox -> and sometimes in real setups too.

Lab 2 · QBCore callback for current cash

Create a QB callback called qu:getCash that returns the current cash for the invoking player.

What to do: register the callback, fetch the player, then send Player.Functions.GetMoney('cash') into cb(...).

The test will call your callback and then print cash=500. That visible line is the proof that the callback returned a real answer instead of just registering silently.

What you learned today

  • QBCore.Functions.GetPlayer(source) gets the player object on the server. You need the core object first, then use its Functions table.
  • Player.Functions methods like GetMoney('cash'), RemoveMoney('cash', amount), and AddMoney('bank', amount) handle all money changes.
  • QBCore.Functions.CreateCallback registers a server callback. The client calls it, the server answers with a value through cb(...).
  • Same rule as ESX: never trust the client, always check the balance on the server before moving money.

Common mistakes

GetMoney returns nil or zero when it should not

You passed the wrong money type string. QBCore uses 'cash' and 'bank' as account names. Writing 'money' or 'Cash' (capital C) returns nothing because the account name does not match. Use the exact lowercase string the framework expects.

Script error: attempt to index a nil value (Player)

You called QBCore.Functions.GetPlayer with the wrong source value. If source does not match a connected player, the function returns nil and the next line crashes when you try to use Player.Functions. Make sure you are passing the real source from the command handler, not a hardcoded number.

Callback registers but the client gets nil back

You forgot to call cb(value) inside the callback function. CreateCallback expects you to pass your answer into cb. If you just return the value instead of calling cb(value), the client never receives the answer.

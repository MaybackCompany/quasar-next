What this template builds

ox_lib ships a ready-made input dialog: a clean NUI panel with labeled fields, validation, and a submit and cancel flow. This template registers a /startsurvey command that opens that dialog with three questions, then reads the returned answers as an ordered table. You write zero HTML, JS, or CSS for the interface. Drop it in, ensure ox_lib first, and the form works.

Read the field types from the docs

The input dialog supports more than text. You get number fields with min and max, checkboxes, selects, sliders, color pickers, and more. The full list lives in the ox_lib documentation. This template uses input, number, and checkbox to keep the example clear.

Introduction

ox_lib is a versatile library for FiveM that provides reusable modules and UI elements, streamlining the development of custom server features. One handy use is building dynamic in-game surveys to gather player feedback or preferences in real time, without writing your own NUI from scratch.

Setup

Make sure ox_lib is installed and configured in your server. Follow the official documentation for installation, then ensure it before any resource that calls it.

Registering the survey command

Start with the command that opens the survey. Everything else hangs off this.

Defining the survey questions

Inside the command, call lib.inputDialog with a title and a list of fields. Each field is a table that sets its type, label, and rules. The dialog returns an ordered table of answers, or nil if the player cancels, so check for that before you read any answer.

Always check for cancel first

When the player closes the dialog without submitting, lib.inputDialog returns nil. Read an index off a nil value and the script errors. The if not survey then return end line is not optional. Keep it before you touch any answer.

Full example

Here is the complete command with the cancel check, named locals for each answer, and a clear log of every response. The checkbox value comes back as a boolean, so it is converted to a readable Yes or No before printing.

This dialog runs on the client

The answers come from the player's machine, so treat them as untrusted. If you forward survey results to the server to save them, validate types and ranges there before you write anything to the database. Never trust the client number field to stay inside min and max: re-check on the server.

Install

  1. Create resources/[local]/qu_survey/.
  2. Copy the command into a client.lua and add a manifest with client_script 'client.lua' and shared_script '@ox_lib/init.lua'.
  3. Add ensure ox_lib before ensure qu_survey in server.cfg.
  4. Restart the server and run /startsurvey in game.
  5. Fill in the form, submit, and watch the answers print in the server console.

Conclusion

By leaning on ox_lib, you get a dynamic, interactive in-game survey with no custom NUI to maintain. It is a fast way to collect player feedback and gives server admins real signal to improve the experience. Once the answers land, the natural next step is to send them to the server and store them.

Make it yours

Change the dialog title and the questions to match what you actually want to learn from players. Swap the print lines for a server event that saves the answers, and add an ACE permission check if the survey should be staff-triggered. Keep the cancel check, and re-validate every value on the server before you store it.

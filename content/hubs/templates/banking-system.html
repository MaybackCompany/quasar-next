Banking system

A drop-in bank menu your players open with /bank. They can deposit cash, withdraw from the bank, and transfer money to another player, all through a small NUI window. One config switch flips it between ESX and QBCore.

Purpose
Give players a working bank: deposit, withdraw, transfer.
Dependencies
ESX (es_extended) or QBCore (qb-core). No oxmysql needed: money lives in the framework.
Files
fxmanifest.lua, config/config.lua, locales/*.lua, server/server.lua, client/client.lua, and the html/ NUI.
Open it
Type /bank in chat to bring up the menu.
How it fits together

The client asks the server for the player's balance with a callback, then opens the NUI menu. Every button in the menu calls back to the client over fetch, the client validates the number and fires a server event, and the server moves the money using the framework's account functions. The server is the only place that touches money, which is exactly where that logic belongs.

Files

  • fxmanifest.lua, declares the UI page, scripts, and the NUI files the client is allowed to load.
  • config/config.lua, picks the framework and language, and gives you the _U locale helper.
  • locales/en.lua and locales/es.lua, the notification text players see.
  • server/server.lua, the money logic: balance callback, deposit, withdraw, transfer, plus framework adapter functions.
  • client/client.lua, the /bank command, the NUI open event, and the NUI callbacks.
  • html/index.html, html/css/style.css, html/js/script.js, the menu the player sees and clicks.

fxmanifest.lua

This is the resource's table of contents. The ui_page line tells FiveM which HTML to render, and the files block lists every NUI asset the client is allowed to fetch. If a CSS or JS file is missing from files, the menu will load blank, so keep that list in sync with your html/ folder.

config/config.lua

One place to choose your framework and language. The _U function looks up a key in the active locale table, so the rest of the resource sends short keys like deposit_success instead of hardcoded sentences. Set Config.Framework to 'esx' or 'qbcore' and you are done.

Tip

Because _U falls back to the raw key when a translation is missing, a typo shows up in game as the key itself instead of a crash. That makes missing locale entries easy to spot.

locales/en.lua and locales/es.lua

These tables hold the short notification strings the player sees after each action. Add a new language by copying one of these files, changing the table name (the part in brackets), and translating the values. Then point Config.Locale at it.

server/server.lua

This is the heart of the resource. It detects the framework, exposes a balance callback, and handles the three money events. Notice that every deposit, withdraw, and transfer re-checks the amount and the balance on the server before moving anything. That check is what keeps a player from sending a fake amount and printing money.

Framework detection

At the top, the server grabs the right core object based on your config. If the name is wrong, it prints an error instead of silently doing nothing.

Get balance callback

When the player runs /bank, the client asks for these two numbers. The server reads the bank account and the cash on hand, then sends both back so the menu can show them.

Deposit, withdraw, and transfer events

Three net events, one per action. Each one looks up the player, reads the relevant balance, and only moves money when the amount is positive and the player actually has it. On success or failure it fires a notify back to the client with a locale key.

Framework adapter functions

The events above never call ESX or QBCore directly. Instead they call these small helpers, and the helper decides which framework function to use. This is the trick that lets one resource serve both frameworks: to add support for a third, you only edit these functions.

Never trust the amount the client sends

The client can send any number it wants. The reason this template is safe is that the server reads the player's real balance with GetCash or GetBank and refuses the action unless amount > 0 and the player has the funds. If you ever move that check to the client, you hand players an exploit. Keep all money math on the server.

client/client.lua

The client is the bridge between the player and the server. It registers the /bank command, opens the NUI menu when the balance comes back, and turns NUI button presses into server events. It never moves money itself: it only forwards validated requests.

Framework init and the /bank command

On resource start the client grabs the same core object. The /bank command asks the server for the balance and cash, and only opens the menu once both values arrive.

Opening the NUI menu

This event hands control to the mouse. SetNuiFocus(true, true) lets the player click the menu, and SendNUIMessage pushes the balance and cash into the page so the JavaScript can display them.

NUI callbacks

Each button in the menu posts back to one of these callbacks. The client does a first sanity check with tonumber, fires the matching server event, and answers the UI with ok or error. The close callback releases focus so the player can move again.

The notify handler

When the server reports back, this handler runs the locale key through _U and shows it as a native GTA notification. That is why the server only ever sends short keys: the client decides the language.

Focus is a two-step dance

Whenever you call SetNuiFocus(true, true) to open a menu, you must call SetNuiFocus(false, false) to close it. If the close callback never fires, the player is stuck with a cursor and frozen controls. Always give the player a way out.

html/index.html

The NUI markup is small on purpose. There is the main menu with the balance, the cash, and four buttons, plus a hidden modal that doubles as the input box for all three actions. The target ID field only shows up for transfers.

html/js/script.js

This script wires up the menu. It listens for the openMenu message from the client to fill in the numbers, opens the modal with the right title per action, and on confirm it posts the amount (and target ID for transfers) back to the client over fetch. GetParentResourceName() is the FiveM helper that gives the script its own resource name so the URL resolves.

Tip

The alert() on bad input is fine for a first pass, but it pauses the whole UI thread. When you polish this menu, swap it for an inline error message under the input so the player stays in flow.

html/css/style.css

The styling keeps the menu centered and readable, with a simple modal that fades and slides in. Nothing here is load-bearing for the logic, so this is the first file to restyle when you want the bank to match your server's look.

Install

  1. Create resources/[local]/qu_bank/ and drop the file tree above inside it.
  2. Open config/config.lua and set Config.Framework to 'esx' or 'qbcore'.
  3. Make sure es_extended or qb-core starts before this resource in your server.cfg.
  4. Add ensure qu_bank after your framework line.
  5. Restart the server, join, and type /bank to open the menu.

Testing

  • Run /bank and confirm the balance and cash match your character.
  • Deposit a small amount, then withdraw it, and watch the numbers move the right way.
  • Get a second player nearby, note their server ID, and transfer them a few dollars.
  • Try an empty or negative amount and confirm you get the invalid_amount notification, not a balance change.
  • Flip Config.Framework and re-run the same checks to confirm both frameworks behave the same.

Make it yours

  • Framework and language: set Config.Framework and Config.Locale in config/config.lua.
  • Wording: edit the strings in locales/en.lua, or add a new locale file and point the config at it.
  • Look: restyle html/css/style.css to match your server. The logic does not depend on the colors.
  • Limits: if you want a max transfer or a fee, add that check in the server events in server/server.lua, never in the NUI.

Framework matrix: QBCore vs ESX vs ox_lib

Same task, three frameworks. Pick the one your server uses, then read the others to understand tutorials, snippets, and Discord answers written for a different stack. Examples are deliberately short. Real code adds checks, error handling, and validation.

Why a matrix
Most FiveM tutorials assume one framework. When yours is different, the names change but the shape is the same.
How to read a row
Title is the task. Three cells show how each framework expresses it. n/a means that framework does not provide it directly.
ox_lib note
ox_lib is framework-neutral. It works alongside QBCore or ESX. Some rows show ox_lib helpers; others show ox_inventory or ox_target where they replace the base framework feature.
Always check docs
API names change across forks (qb-core vs qbox, ESX Legacy vs ESX 1.2). Use this matrix as a map, then confirm with the official docs for your version.

  • Get the player object on the server

    Player

    Use the FiveM source (player id) to load the framework's player record. Always check it is not nil before reading fields.

    QBCore 
    local QBCore = exports['qb-core']:GetCoreObject()
local Player = QBCore.Functions.GetPlayer(source)
if not Player then return end
local cid = Player.PlayerData.citizenid
    ESX 
    ESX = exports['es_extended']:getSharedObject()
local xPlayer = ESX.GetPlayerFromId(source)
if not xPlayer then return end
local id = xPlayer.identifier
    ox_lib 
    -- ox_core (optional). With QBCore/ESX,
-- use the framework's player object.
local player = exports.ox_core:GetPlayer(source)

  • Get the player object on the client

    Player

    Client-side player data mirrors a subset of server data. Treat it as read-only.

    QBCore 
    local QBCore = exports['qb-core']:GetCoreObject()
local PlayerData = QBCore.Functions.GetPlayerData()
local cid = PlayerData.citizenid
    ESX 
    ESX = exports['es_extended']:getSharedObject()
local xPlayer = ESX.GetPlayerData()
local job = xPlayer.job and xPlayer.job.name
    ox_lib 
    n/a. Use the underlying framework
or ox_core's client API.

  • Stable player identifier

    Identity

    The id you save in the database. Different frameworks use different shapes, so saved data does not auto-migrate between them.

    QBCore 
    -- citizenid: 8-char string, e.g. "ABCD1234"
local cid = Player.PlayerData.citizenid
    ESX 
    -- identifier: license, steam, or fivem hex
local id = xPlayer.identifier
    ox_lib 
    -- ox_core uses charId (number).
-- For raw FiveM, GetPlayerIdentifierByType
local lic = GetPlayerIdentifierByType(source, 'license')

  • Add money to a player

    Money

    Always run money changes on the server. Never trust the client to ask for money.

    QBCore 
    Player.Functions.AddMoney('cash', 250, 'reward-quest-1')
    ESX 
    xPlayer.addMoney(250)
-- accounts: xPlayer.addAccountMoney('bank', 250)
    ox_lib 
    n/a. Money is a framework concern.
Use QBCore/ESX/ox_core money API.

  • Remove money from a player

    Money

    Check the balance first. The framework will not stop you spending below zero in older forks.

    QBCore 
    if Player.PlayerData.money.cash >= 100 then
  Player.Functions.RemoveMoney('cash', 100, 'shop-buy')
end
    ESX 
    if xPlayer.getMoney() >= 100 then
  xPlayer.removeMoney(100)
end
    ox_lib 
    n/a. Use the framework money API.

  • Read a player's job

    Jobs

    Both frameworks expose name and grade. Names differ slightly, so always read with a guard.

    QBCore 
    local job = Player.PlayerData.job
local name = job and job.name        -- "police"
local grade = job and job.grade.level -- 0..N
    ESX 
    local job = xPlayer.job
local name = job and job.name         -- "police"
local grade = job and job.grade        -- number
    ox_lib 
    n/a. Read from QBCore/ESX/ox_core.

  • Set a player's job

    Jobs

    Server-side. Make sure the job exists in shared/jobs.lua before calling this.

    QBCore 
    Player.Functions.SetJob('police', 0)
    ESX 
    xPlayer.setJob('police', 0)
    ox_lib 
    n/a. ox_core has its own group/job system.

  • Show a quick notification

    Notify

    Each framework ships a default notify. ox_lib's is the most flexible. Pick one and use it everywhere for visual consistency.

    QBCore (client) 
    QBCore.Functions.Notify('Saved.', 'success')
-- types: 'success' | 'error' | 'primary'
    ESX (client) 
    ESX.ShowNotification('Saved.')
-- styled by ~g~ ~r~ etc. in legacy
    ox_lib (client) 
    lib.notify({
  title = 'Saved',
  description = 'Your changes are live.',
  type = 'success',
})

  • Notify from the server to a client

    Notify

    From the server, send a TriggerClientEvent to one source. Do not loop the whole player list unless you mean to broadcast.

    QBCore 
    TriggerClientEvent('QBCore:Notify', source,
  'Saved.', 'success')
    ESX 
    TriggerClientEvent('esx:showNotification', source,
  'Saved.')
    ox_lib 
    TriggerClientEvent('ox_lib:notify', source, {
  title = 'Saved', type = 'success'
})

  • Open a simple input dialog

    UI

    For text/number entry. ox_lib's input is the cleanest cross-framework choice.

    QBCore (qb-input) 
    local res = exports['qb-input']:ShowInput({
  header = 'Name plate',
  submitText = 'Save',
  inputs = {{ text='Plate', name='plate', type='text' }},
})
    ESX (esx_textui or addon) 
    -- ESX has no native input modal.
-- Use ox_lib.inputDialog or a UI addon.
    ox_lib 
    local input = lib.inputDialog('Name plate', {
  { type = 'input', label = 'Plate', required = true },
})
if not input then return end
local plate = input[1]

  • Show a context menu

    UI

    For action lists like a phone menu, garage menu, or admin tool.

    QBCore (qb-menu) 
    exports['qb-menu']:openMenu({
  { header = 'Garage' },
  { header = 'Cars', txt = 'Pick one', params = {
      event = 'garage:list'
  }},
})
    ESX (esx_menu_default) 
    ESX.UI.Menu.Open('default', GetCurrentResourceName(),
  'garage', { title = 'Garage', elements = {
    { label = 'Cars', value = 'cars' }
  }}, function(data, menu)
    -- handle data.current.value
  end)
    ox_lib 
    lib.registerContext({
  id = 'garage_menu',
  title = 'Garage',
  options = {
    { title = 'Cars', event = 'garage:list' }
  }
})
lib.showContext('garage_menu')

  • Add an item to a player

    Inventory

    Server-side. The item must exist in the inventory's items table. ox_inventory is recommended for new servers.

    QBCore (qb-inventory) 
    Player.Functions.AddItem('bread', 1)
TriggerClientEvent('inventory:client:ItemBox',
  source, QBCore.Shared.Items['bread'], 'add')
    ESX (legacy) 
    xPlayer.addInventoryItem('bread', 1)
    ox_inventory 
    exports.ox_inventory:AddItem(source, 'bread', 1)

  • Check if a player has an item

    Inventory

    Always check on the server. Clients can lie about their inventory.

    QBCore 
    local item = Player.Functions.GetItemByName('bread')
if item and item.amount > 0 then
  -- has bread
end
    ESX 
    local item = xPlayer.getInventoryItem('bread')
if item and item.count > 0 then
  -- has bread
end
    ox_inventory 
    local count = exports.ox_inventory:GetItemCount(
  source, 'bread')
if count > 0 then
  -- has bread
end

  • Use-item callback (server)

    Inventory

    Run code when a player uses an item from their inventory. Each system registers callbacks differently.

    QBCore 
    QBCore.Functions.CreateUseableItem('bread',
function(source, item)
  -- consume + apply effect
end)
    ESX 
    ESX.RegisterUsableItem('bread', function(source)
  local xPlayer = ESX.GetPlayerFromId(source)
  xPlayer.removeInventoryItem('bread', 1)
  -- apply effect
end)
    ox_inventory 
    -- Define in items.lua with:
-- ['bread'] = { server = { export = 'myres.useBread' } }
-- then:
exports('useBread', function(event, item, inventory, slot, data)
  if event == 'usingItem' then
    -- effect
  end
end)

  • Add an interaction zone on an entity

    Target

    Each target system has slightly different option shapes. ox_target is the cross-framework standard.

    qb-target 
    exports['qb-target']:AddTargetEntity(entity, {
  options = {{
    type = 'client',
    event = 'shop:open',
    icon  = 'fas fa-store',
    label = 'Open shop',
  }},
  distance = 2.0,
})
    ESX 
    n/a. ESX does not ship a target system.
Most servers run ox_target or qb-target.
    ox_target 
    exports.ox_target:addLocalEntity(entity, {{
  name   = 'shop_open',
  event  = 'shop:open',
  icon   = 'fa-solid fa-store',
  label  = 'Open shop',
  distance = 2.0,
}})

  • Server callback (client asks server, gets answer)

    Callbacks

    For things like "do I have permission" or "give me my saved data". Always validate the request on the server.

    QBCore 
    -- server
QBCore.Functions.CreateCallback('shop:can-buy',
function(source, cb, item)
  cb(Player.PlayerData.money.cash >= Items[item].price)
end)
-- client
QBCore.Functions.TriggerCallback('shop:can-buy',
function(ok) if ok then ... end, 'bread')
    ESX 
    -- server
ESX.RegisterServerCallback('shop:can-buy',
function(source, cb, item)
  local xPlayer = ESX.GetPlayerFromId(source)
  cb(xPlayer.getMoney() >= Items[item].price)
end)
-- client
ESX.TriggerServerCallback('shop:can-buy',
function(ok) ... end, 'bread')
    ox_lib 
    -- server
lib.callback.register('shop:can-buy',
function(source, item)
  return hasMoney(source, Items[item].price)
end)
-- client (await)
local ok = lib.callback.await('shop:can-buy',
  false, 'bread')

  • Progress bar / circle while doing an action

    UI

    Lock the player into a short action so they cannot move or shoot during it. ox_lib's progressCircle is the most polished.

    QBCore (qb-core helper) 
    QBCore.Functions.Progressbar('lockpick',
  'Lockpicking...', 5000, false, true, {
    disableMovement = true, disableCarMovement = true,
    disableMouse = false, disableCombat = true,
  }, {}, {}, {}, function() ... end)
    ESX 
    n/a. ESX has no built-in progress bar.
Use ox_lib.progressCircle or progress addon.
    ox_lib 
    if lib.progressCircle({
  duration = 5000,
  label = 'Lockpicking...',
  position = 'bottom',
  disable = { move = true, combat = true },
}) then
  -- success
end

  • Listen for player loaded (after spawn)

    Player

    Run setup once after the player is fully in the world. Use this for spawning gear, restoring state, or sending welcome data.

    QBCore 
    RegisterNetEvent('QBCore:Client:OnPlayerLoaded',
function()
  -- ready
end)
    ESX 
    RegisterNetEvent('esx:playerLoaded')
AddEventHandler('esx:playerLoaded', function(xPlayer)
  -- ready
end)
    ox_core / ox_lib 
    -- raw FiveM works for any framework:
AddEventHandler('playerSpawned', function()
  -- ready
end)

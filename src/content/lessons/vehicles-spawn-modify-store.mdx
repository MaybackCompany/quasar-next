    Open `client.lua` and paste this. The lines are spread out on purpose so the next section can walk through them one at a time:

    ```lua
local testVehicle

local function loadModel(model)
    if not IsModelInCdimage(model) or not IsModelAVehicle(model) then
        return false
    end

    RequestModel(model)
    local deadline = GetGameTimer() + 5000
    while not HasModelLoaded(model) do
        if GetGameTimer() > deadline then return false end
        Wait(0)
    end
    return true
end

RegisterCommand('testvehicle', function()
    if testVehicle and DoesEntityExist(testVehicle) then
        DeleteEntity(testVehicle)
    end

    local model = joaat('sultan')
    if not loadModel(model) then
        print('[qu_vehicles_spawn_modify_store] invalid vehicle model or load timeout')
        return
    end

    local ped = PlayerPedId()
    local c = GetEntityCoords(ped)
    testVehicle = CreateVehicle(model, c.x + 3.0, c.y, c.z, GetEntityHeading(ped), true, false)

    SetVehicleNumberPlateText(testVehicle, 'QUCAR')
    SetVehicleColours(testVehicle, 12, 12)
    TaskWarpPedIntoVehicle(ped, testVehicle, -1)

    SetModelAsNoLongerNeeded(model)
    print('[qu_vehicles_spawn_modify_store] spawned sultan')

    local spawned = testVehicle
    SetTimeout(60000, function()
        if spawned and DoesEntityExist(spawned) then
            DeleteEntity(spawned)
            print('[qu_vehicles_spawn_modify_store] auto-deleted')
        end
    end)
end, false)
    ```
  </Step>

## How it works

The car spawned. Now take the code apart so the next time you write a garage or a dealership you are deciding, not copying. The whole command is one story in five beats: clear the old car, load the model, spawn it, dress it up, and arm a timer to delete it. We will follow that order.

### Why a model must be loaded before you spawn it

```lua
local model = joaat('sultan')
if not loadModel(model) then return end
```

A vehicle in GTA is built from a model: a bundle of meshes, textures, and physics data that the game keeps on disk, not in memory. `CreateVehicle` cannot build a car from a model that is not in memory yet, so you have to ask the engine to stream it in first. That is exactly what these three pieces do.

`RequestModel(model)` tells the engine to start loading the Sultan. It does not wait. The load happens in the background over the next few frames, so the line after `RequestModel` runs long before the model is actually ready. If you called `CreateVehicle` immediately you would get a null entity, because the data simply is not there yet.

The helper also checks `IsModelInCdimage` and `IsModelAVehicle` before requesting, then gives streaming five seconds. `Wait(0)` yields each pass so the game keeps running. The timeout matters when a model is unavailable: without it, a yielding loop can wait forever even though it does not freeze the whole client.

### What joaat actually returns

```lua
local model = joaat('sultan')
```

`joaat` is the Jenkins one-at-a-time hash function, and it turns the text `'sultan'` into a number. Internally the game does not identify models by their string name; it identifies them by a 32-bit hash of that name. `CreateVehicle` and `RequestModel` both expect that hash, not the word. `joaat('sultan')` and `GetHashKey('sultan')` compute the exact same number, so they are interchangeable. `joaat` is just the shorter, native Lua-side spelling. Either way, the rule is the same: hand the spawn natives the hash, never the raw string.

### CreateVehicle and the two flags that confuse everyone

```lua
testVehicle = CreateVehicle(model, c.x + 3.0, c.y, c.z, GetEntityHeading(ped), true, false)
```

This is the line that builds the car. The first five arguments are the obvious ones: the model hash, then x, y, z, then the heading. We read the player's position with `GetEntityCoords(ped)` and add `3.0` to x so the car spawns a few metres to the side instead of inside the player. `GetEntityHeading(ped)` points the car the same way the player is facing.

The last two booleans are where most beginners are silently wrong, and they are the network ownership basics this lesson is really about.

- The first boolean, `true`, is `isNetwork`. When it is `true`, the entity is registered with FiveM's networking, gets a network ID, and is replicated to other players. The car you spawn is a real shared object everyone on the server can see. Set this to `false` and the car would exist only on your own machine; no one else would see it. For a car a player sits in, you want `true`.
- The second boolean, `false`, is `netMissionEntity` (sometimes called `bScriptHostObject`). Leave it `false` for ordinary spawns. You set it `true` only for entities a single script wants to keep authority over across clients, which is an advanced mission-script concern, not something you need for a garage car.

The client that calls `CreateVehicle` becomes the initial network owner of the car. Ownership is the right to control the entity's physics and state, and it can migrate to another player as they drive near it. The takeaway: `isNetwork = true` is what makes a spawned vehicle a proper networked entity instead of a ghost only you can see.

<Checkpoint question="A teammate spawns a car with isNetwork set to false and asks why nobody else can see it. What do you tell them?">
  The `false` first boolean means the car was never registered with FiveM's networking, so it exists only on the machine that spawned it. Other players have no network ID for it and the server never replicates it, so to them the spot is empty. The fix is to spawn with `isNetwork = true` (the first boolean), which gives the car a network ID and makes it a shared entity. The second boolean, `netMissionEntity`, stays `false` for an ordinary car and is unrelated to whether others can see it.
</Checkpoint>

### Dressing the car: plate and color

```lua
SetVehicleNumberPlateText(testVehicle, 'QUCAR')
SetVehicleColours(testVehicle, 12, 12)
```

`SetVehicleNumberPlateText` takes the vehicle handle and a string, and stamps that string onto the plate. Plates are capped at eight characters, so `QUCAR` is fine.

`SetVehicleColours(vehicle, primary, secondary)` is the line with the magic numbers. Those two numbers are not RGB values. They are indexes into GTA's built-in paint table, where each number maps to a specific named color. `12` is matte black, which is why the body comes out a flat black. The full table lives in the FiveM and GTA modding references under vehicle colors; common anchors are `0` (metallic black), `12` (matte black), `27` (metallic red), `64` (metallic blue). Passing the same index for primary and secondary, as we do here, paints the whole car one solid color. The reason to use an index instead of raw RGB is that these are the factory paints the game already ships, so they render correctly under the game's lighting without you defining a custom color.

### Putting the player in the seat instantly

```lua
TaskWarpPedIntoVehicle(ped, testVehicle, -1)
```

`TaskWarpPedIntoVehicle` teleports a ped straight into a seat with no walking animation. The third argument is the seat index, and the seat numbering is the part to memorise: `-1` is the driver seat, `0` is the front passenger, `1` and `2` are the rear seats, and so on. So `-1` here drops the player behind the wheel the instant the car exists.

The word "warp" is the contrast to learn. `TaskWarpPedIntoVehicle` is instant and unconditional, which is exactly what you want for a test command or an admin spawn. The alternative, `TaskEnterVehicle`, makes the ped physically walk over, open the door, and climb in, which looks natural but takes seconds and can fail if the path is blocked. For a spawn-and-go command, warp wins.

### Releasing the model

```lua
SetModelAsNoLongerNeeded(model)
```

Back when you called `RequestModel`, you told the engine to pin the Sultan model in memory. Once the car is built, the spawned entity holds its own reference to that data, so the pin you placed is no longer needed. `SetModelAsNoLongerNeeded` lifts your hold and lets the streamer free the model when nothing else is using it. Skipping this will not crash anything, but pile up enough un-released models across a session and you leak memory. Release what you requested: that is the discipline.

### The timer, the closure, and the existence guard

```lua
local spawned = testVehicle
SetTimeout(60000, function()
    if spawned and DoesEntityExist(spawned) then
        DeleteEntity(spawned)
        print('[qu_vehicles_spawn_modify_store] auto-deleted')
    end
end)
```

`SetTimeout(60000, fn)` schedules the function `fn` to run once, 60000 milliseconds (60 seconds) later. The function you hand it is a closure: it captures the variables in scope when it was created and remembers them until it fires. Here it captures `spawned`, which we set to the current `testVehicle` on the line above.

That extra `local spawned = testVehicle` is deliberate, and it is the named takeaway of this lesson. `testVehicle` is the shared variable at the top of the file, and it changes every time the command runs. If the timer closed over `testVehicle` directly, then running `/testvehicle` a second time would overwrite `testVehicle` with the new car, and the first car's 60-second timer would fire and delete the new car by mistake. By snapshotting the handle into a fresh `spawned` for each run, every timer deletes exactly the car it was created for.

`DoesEntityExist(spawned)` is the guard. By the time the timer fires, the car might already be gone: the player could have left and rejoined, another script could have cleaned it up, or the entity could have been streamed out. Calling `DeleteEntity` on a handle that no longer points to a live entity is exactly the kind of careless call that throws errors in the F8 console. The guard says "only delete this if it is still really here," which is the cleanup discipline from Lesson 25 applied to a real entity.

### When you reach for this whole pattern

Stream a model, spawn a networked entity, customize it, seat the player, and arm a guarded cleanup: that five-beat shape is the skeleton under almost every vehicle feature in FiveM.

- **A garage or dealership.** Spawning the car the player selected, painting it to their saved color, and dropping them into the driver seat is this exact code with the model and colors read from a database instead of hard coded.
- **A test or admin command.** Spawn any vehicle on demand to check handling or a livery, then let the timer clean it up so you do not litter the map.
- **Temporary mission vehicles.** A getaway car or a delivery truck that should disappear when the job ends uses the same snapshot-and-guard cleanup so the right entity gets removed at the right time.

<Exercise>
  Make the command accept a model name and a color so it is a real spawn tool instead of a fixed demo. FiveM passes the typed words into the handler as the `args` table:

  ```lua
  RegisterCommand('testvehicle', function(source, args)
      local name = args[1] or 'sultan'
      local color = tonumber(args[2]) or 12
      local model = joaat(name)
      if not IsModelInCdimage(model) or not IsModelAVehicle(model) then
          print('[qu_vehicles_spawn_modify_store] not a valid vehicle: ' .. name)
          return
      end
      RequestModel(model)
      while not HasModelLoaded(model) do
          Wait(0)
      end
      -- spawn, then: SetVehicleColours(testVehicle, color, color)
  end, false)
  ```

  Run `/testvehicle adder 27` and confirm a red Adder spawns. Then run `/testvehicle notacar` and confirm the `IsModelInCdimage` and `IsModelAVehicle` guard prints the rejection line instead of trying to spawn a broken entity. You just turned a hard-coded demo into a command that validates user input before touching the spawn natives, which is the difference between a toy and a tool.

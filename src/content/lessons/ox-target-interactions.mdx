>
  Most FiveM servers have things players can interact with: NPCs, doors, trunks, registers, ATMs. Before `ox_target`, building those interactions meant lots of repeated distance checks and prompt code. With `ox_target`, most of that becomes one small options table. `ox_target` is an Overextended community resource, the successor to qtarget and bt-target, not an official Cfx.re resource. This lesson teaches its one core pattern slowly, then uses it three different ways.

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_ox_target_interactions
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout. This lesson has a client file (the interactions live where the player is) and a server file (giving cash is a server decision):

    ```text
resources/qu_ox_target_interactions/
  fxmanifest.lua
  client.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load and what to wait for.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

client_script 'client.lua'
server_script 'server.lua'

dependencies {
    'ox_lib',
    'ox_target',
    '/onesync'
}
    ```

    The canonical manifest key is the plural `dependencies { ... }` with a table. The singular `dependency 'ox_target'` is tolerated by FiveM, but the plural form is the form you should write and the form you will see in real resources. We list both `ox_lib` and `ox_target` because `ox_target` itself depends on `ox_lib` being running first. Older tutorials add a `lua54 'yes'` line here. As of June 2025 that setting is deprecated and ignored, since Lua 5.4 is now the only Lua runtime, so you leave it out.
  </Step>

  <Step num={4} title="Write the client code" outcome="Three interactions register the moment the resource starts.">
    Open `client.lua` and paste this:

    ```lua
-- 1. A box zone over the ATM spot at Legion Square.
exports.ox_target:addBoxZone({
    coords = vec3(215.76, -810.12, 30.73),
    size = vec3(2.0, 2.0, 2.0),
    rotation = 0.0,
    debug = true,
    options = {
        {
            name = 'qu_atm_use',
            icon = 'fa-solid fa-money-bill',
            label = 'Use ATM',
            distance = 2.0,
            canInteract = function(entity, distance)
                return distance <= 2.0
            end,
            onSelect = function()
                print('[qu_ox_target_interactions] ATM used')
            end
        }
    }
})

-- 2. A merchant ped: any ped using this model gets a Talk option.
exports.ox_target:addModel('a_m_y_business_03', {
    {
        name = 'qu_merchant_talk',
        icon = 'fa-solid fa-comment',
        label = 'Talk to merchant',
        distance = 2.5,
        onSelect = function()
            print('[qu_ox_target_interactions] merchant talked to')
        end
    }
})

-- 3. A give cash option on every OTHER player ped.
exports.ox_target:addGlobalPlayer({
    {
        name = 'qu_give_cash',
        icon = 'fa-solid fa-hand-holding-dollar',
        label = 'Give $100',
        distance = 2.0,
        onSelect = function(data)
            local serverId = GetPlayerServerId(NetworkGetPlayerIndexFromPed(data.entity))
            TriggerServerEvent('qu_ox_target_interactions:giveCash', serverId)
        end
    }
})
    ```
  </Step>

  <Step num={5} title="Write the server code" outcome="The server fixes the amount, validates the target and distance, and rate-limits the request.">
    Open `server.lua` and paste this:

    ```lua
local TRANSFER_AMOUNT = 100
local MAX_DISTANCE = 3.0
local lastRequestAt = {}

RegisterNetEvent('qu_ox_target_interactions:giveCash', function(targetId)
    local fromId = source
    targetId = tonumber(targetId)

    if not targetId or targetId == fromId or not GetPlayerName(targetId) then
        print(('[qu_ox_target_interactions] rejected invalid target from %s'):format(fromId))
        return
    end

    local now = os.time()
    if now - (lastRequestAt[fromId] or 0) < 1 then
        print(('[qu_ox_target_interactions] rate-limited %s'):format(fromId))
        return
    end
    lastRequestAt[fromId] = now

    local fromPed = GetPlayerPed(fromId)
    local targetPed = GetPlayerPed(targetId)
    if fromPed == 0 or targetPed == 0 then return end

    local distance = #(GetEntityCoords(fromPed) - GetEntityCoords(targetPed))
    if distance > MAX_DISTANCE then
        print(('[qu_ox_target_interactions] rejected distant target from %s'):format(fromId))
        return
    end

    -- Proof only. A real transfer must also read the sender's balance on the
    -- server and use the installed framework's server-side money functions.
    print(('[qu_ox_target_interactions] validated %s -> %s for $%s'):format(
        fromId, targetId, TRANSFER_AMOUNT
    ))
end)

AddEventHandler('playerDropped', function()
    lastRequestAt[source] = nil
end)
    ```
  </Step>

  <Step num={6} title="Start and test it" outcome="The expected proof appears in the correct consoles.">
    Open `server.cfg` and add this line. It must come after the lines that ensure `ox_lib` and `ox_target`:

    ```text
    ensure qu_ox_target_interactions
    ```

    Save, then run:

    ```text
    restart qu_ox_target_interactions
    ```

    Join the server. You will see a translucent debug box floating at the ATM spot. Walk into it, aim at it, and select Use ATM. Then aim at any other player nearby and select Give $100.

    <ExpectedOutput>
      F8: [qu_ox_target_interactions] ATM used
      txAdmin: [qu_ox_target_interactions] validated 1 -> 2 for $100
    </ExpectedOutput>

    The ATM line prints in F8 because `onSelect` runs on the client. The give cash line prints in txAdmin Live Console because that work was sent to the server. The numbers `1` and `2` are server ids, so they will match whoever is actually connected.
  </Step>
</StepGroup>

## How it works

You selected an option and the right line appeared in the right console. Now take the code apart so the next time you wire an interaction you are deciding, not copying. The whole system rests on one idea: every `add*` function takes the same shape of option, and the option describes what the player sees and what happens when they pick it.

### The shared option schema

This is the table you will write hundreds of times. Every entry inside `options` supports the same fields, no matter which `add*` function you handed it to:

- `name` is a unique id string for this option. It must be unique across your whole server, which is why we prefix ours with the resource name (`qu_atm_use`, not `use`). If two resources register the same `name`, the second one silently overwrites the first and you lose an interaction with no error.
- `icon` is a Font Awesome class string, like `fa-solid fa-money-bill`. ox_target ships Font Awesome, so you pick from their free icon set. A wrong or misspelled class shows a blank square instead of an icon, which is the number one reason a new option looks broken.
- `label` is the text the player reads in the eye menu, like `Use ATM`. This is the prompt, the human-facing half of the option.
- `distance` is the maximum range in metres at which the option can be selected. ox_target uses it as a first cheap filter before it ever calls your `canInteract`.
- `canInteract(entity, distance)` is an optional function that returns a boolean. ox_target calls it to decide whether to show the option at all. Return `true` to show it, `false` to hide it. `entity` is the thing being aimed at and `distance` is how far the player is from it. This is where gating lives: only show Open till if the player is a cashier, only show Lockpick if the door is locked.
- `onSelect` is the function that runs when the player actually clicks the option. For a model or a player option it receives a `data` table whose `data.entity` is the entity that was targeted. This is the payload half of the option.

<Checkpoint question="The same options table works for a box zone, a model, and every other player. What is the one field that behaves differently depending on which add* function you used?">
  `onSelect` (and `canInteract`). For `addBoxZone` there is no specific entity, so `data.entity` is not meaningful and you usually ignore the argument. For `addModel` and `addGlobalPlayer` the player is aiming at a real entity, so ox_target hands you a `data` table and `data.entity` is the ped or prop they targeted. That entity is how you know which player to give cash to in option three. The fields are the same shape everywhere, but what the entity means depends on what you attached the option to.
</Checkpoint>

### addBoxZone: an interaction tied to a place

```lua
exports.ox_target:addBoxZone({
    coords = vec3(215.76, -810.12, 30.73),
    size = vec3(2.0, 2.0, 2.0),
    rotation = 0.0,
    debug = true,
    options = { ... }
})
```

A box zone is an invisible 3D box sitting in the world. When the player aims inside it, the options appear. The box is defined by three things:

- `coords` is the centre point of the box, a `vec3` of world coordinates `(x, y, z)`. `vec3` is the ox_lib short form; FiveM also exposes the same value as `vector3`. They are interchangeable here, and `vec3` is what Overextended resources use.
- `size` is a `vec3` of the box dimensions in metres along each axis: width, depth, height. `vec3(2.0, 2.0, 2.0)` is a two metre cube. A flat counter would be wide and shallow, like `vec3(2.0, 0.6, 1.2)`.
- `rotation` spins the box around its vertical axis in degrees. `0.0` leaves it aligned to the world. You set this when the thing you are wrapping, a counter or a desk, sits at an angle.

The field that makes placement possible is `debug = true`. With debug on, ox_target draws the box as a translucent shape you can see in game, so you can stand at the spot, read whether the box actually covers the ATM, and nudge the coords. Once it lines up, you set `debug = false` (or delete the line) for production, and the box becomes invisible while still working. Shipping with debug on is a common mistake that leaves coloured boxes all over a live server.

### addModel: an interaction tied to a kind of thing

```lua
exports.ox_target:addModel('a_m_y_business_03', {
    { name = 'qu_merchant_talk', ... }
})
```

`addModel` attaches options to a model rather than a place. Every entity that uses model `a_m_y_business_03`, anywhere on the map, gets the Talk to merchant option. This is how you make a merchant ped: spawn that ped model somewhere, and because it shares the model, it inherits the interaction automatically. Note the argument order is different from `addBoxZone`: the model name comes first, then the options table comes second as its own argument, not wrapped inside a single options key. You can pass one model string or a table of model names to cover several at once.

### addGlobalPlayer: an interaction tied to every other player

```lua
exports.ox_target:addGlobalPlayer({
    {
        name = 'qu_give_cash',
        onSelect = function(data)
            local serverId = GetPlayerServerId(NetworkGetPlayerIndexFromPed(data.entity))
            TriggerServerEvent('qu_ox_target_interactions:giveCash', serverId, 100)
        end
    }
})
```

`addGlobalPlayer` attaches options to the ped of every other player, with no entity or model to name. This is the line that delivers the give cash outcome. The interesting work is in `onSelect`, and it is where the lesson becomes correct rather than convenient.

When you aim at a player, `data.entity` is their ped, which is a local game handle that means nothing to the server. The server thinks in server ids, not ped handles. So you translate: `NetworkGetPlayerIndexFromPed` turns the ped into a local player index, and `GetPlayerServerId` turns that index into the server id the server understands. Giving someone cash is a server decision. The client therefore sends only the selected target id. It does not choose the amount, because a modified client could send any number it wanted.

On the server, `RegisterNetEvent` receives that untrusted request. The `source` global is the player who sent it. The handler rejects a missing, disconnected, or self target, limits each sender to one request per second, and uses server-side coordinates to reject players more than three metres apart. The amount is the server constant `TRANSFER_AMOUNT`, never client input. This lesson deliberately stops at a validated print: it does not pretend money moved. A real implementation must next read the sender's balance on the server and call the installed framework's server-side debit and credit functions as one controlled operation.

<Checkpoint question="Why does the give cash logic run on the server instead of finishing inside the client onSelect?">
  Because money is authoritative state, and the client cannot be trusted with it. A client `onSelect` runs on the player's machine, which can be modified. The client may identify who was selected, but the server owns the amount, checks the target and distance, rate-limits the request, verifies the sender's balance, and applies the transfer. Moving code into a server event is not sufficient by itself; the checks inside that handler are what make the boundary useful.
</Checkpoint>

### When you reach for this pattern

ox_target is the default answer any time a player should walk up to something and act on it. Three real cases:

- **A fixed station.** Registers, ATMs, clothing stores, garage spawn points. These live at known coordinates, so `addBoxZone` is the tool, and `debug = true` is how you place the box exactly over the counter.
- **A class of object or NPC.** Every shop clerk, every dumpster, every harvestable plant. Attach the option once with `addModel` and every entity of that model inherits it, however many you spawn.
- **Player to player actions.** Give cash, show id, cuff, revive, search. These hang off `addGlobalPlayer`, and they almost always end in a `TriggerServerEvent`, because the interesting outcome is a change to shared state the server must own.

## Try it yourself

<Exercise>
  Make the merchant interaction conditional and two-way. Right now any player can Talk to the merchant. Change it so the Talk option only appears when the player is on foot, and add a second option, Rob register, that only appears when the player is crouched. Use `canInteract` to gate each one:

  ```lua
  exports.ox_target:addModel('a_m_y_business_03', {
      {
          name = 'qu_merchant_talk',
          icon = 'fa-solid fa-comment',
          label = 'Talk to merchant',
          distance = 2.5,
          canInteract = function()
              return not IsPedInAnyVehicle(PlayerPedId(), false)
          end,
          onSelect = function()
              print('[qu_ox_target_interactions] merchant talked to')
          end
      },
      {
          name = 'qu_merchant_rob',
          icon = 'fa-solid fa-mask',
          label = 'Rob register',
          distance = 2.5,
          canInteract = function()
              return IsPedDuckingOrCrouching(PlayerPedId())
          end,
          onSelect = function()
              TriggerServerEvent('qu_ox_target_interactions:rob')
          end
      }
  })
  ```

  Add a matching `RegisterNetEvent('qu_ox_target_interactions:rob', ...)` on the server that prints the robber's source. Restart, then confirm Talk shows while standing and Rob register shows only while crouched. You just used `canInteract` to drive which options a single entity exposes based on player state, the exact mechanism real jobs use to show the right actions to the right people.
</Exercise>

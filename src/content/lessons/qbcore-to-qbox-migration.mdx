## A staged migration

<StepGroup>
  <Step num={1} title="Back up and stand up a test server" outcome="You can experiment without risking the live server.">
    Full database backup, then clone the server onto a test environment. Match the live server's artifact (FXServer) version and `sv_enforceGameBuild` on the test box so what you prove there holds in production. Note: sv_enforceGameBuild is a startup-only convar (set it in server.cfg before the server boots; it cannot be changed while running). Its value is a GTA content game build number such as 3095 and up -- a SEPARATE numbering scheme from the FXServer artifact number (the 31000+ range). Copy whatever value your live server already uses so the test and live builds match exactly. Every step below happens on the test server first. Never migrate live.
  </Step>

  <Step num={2} title="Install the ox dependency stack" outcome="oxmysql, ox_lib, and ox_inventory are running.">
    Qbox is ox-native, so its required dependencies must be in place before qbx_core. qbx_core declares two hard dependencies in its fxmanifest: ox_lib and oxmysql. Install and start oxmysql and ox_lib first, with no missing-dependency errors in the console. ox_inventory is the inventory Qbox expects, but it is configured separately and is not required for qbx_core to boot, so install it as part of the same stack but know that a missing ox_inventory does not stop qbx_core from starting. These are Overextended (CommunityOx) resources, so pull them from the official Overextended GitHub releases, not from a QBCore bundle.

    Where this happens: each resource is a folder you place inside your server's resources directory (for example resources/[ox]/oxmysql) and then add to server.cfg with an ensure line (ensure oxmysql). After editing server.cfg, start or restart resources from the txAdmin Live Console or the FXServer console with commands like 'ensure oxmysql' or 'restart qbx_core'. If you do not yet have a server folder and server.cfg, complete the server-setup lesson first.
  </Step>

  <Step num={3} title="Install qbx_core and turn on the bridge" outcome="QBCore scripts run against Qbox where compatibility allows.">
    Install `qbx_core`. The compatibility layer is not a separate download: `qbx_core` ships a QBCore shim that rebuilds the old `QBCore` object on top of the Qbox API. `qbx_core` ships with the QBCore bridge enabled by default. You can set it explicitly by adding this line to your `server.cfg` (open the file in a text editor such as VS Code, add the line above your `ensure qbx_core` line, save, then restart the server). `setr` means a replicated convar (its value is sent to clients too); the capital B in enableBridge matters because convar names are case-sensitive:

    ```cfg
    setr qbx:enableBridge "true"
    ```

    With the bridge on, many QBCore exports and events are translated to Qbox, so a lot of scripts work as-is. Expect a list of scripts that still need edits; that is normal, not failure. The next section shows you exactly where the line falls.
  </Step>

  <Step num={4} title="Test script by script, then cut over" outcome="You know exactly what works before going live.">
    Go through your resources: core (banking, jobs, inventory) first, then the rest. Note what works, what needs an edit, and what to drop. Confirm the production artifact version and game build match what you tested. Only cut production over once the core loop is solid on the test server.

    <ExpectedOutput>
      Illustrative (exact wording varies by version): after qbx_core starts you should see it load without dependency errors, and in-game players spawn with money and jobs working and your core scripts loading with no red console errors. There is no single guaranteed 'bridge enabled' line to look for; the real proof is behavioural -- the QBCore scripts you bridged still function in-game.
    </ExpectedOutput>
  </Step>
</StepGroup>

## How it works

The bridge translates the QBCore surface that scripts call into Qbox calls behind the scenes. That is why "best-effort, not total" is the right frame: anything that maps cleanly to a Qbox export gets translated for you, and anything that reached into QBCore internals or assumed qb-inventory has to be rewritten. Two concrete cases make the line obvious.

### A call the bridge handles for you

The most common QBCore call in any script is fetching a player object server-side:

```lua
-- QBCore original, still works through the bridge
local QBCore = exports['qb-core']:GetCoreObject()
local Player = QBCore.Functions.GetPlayer(source)
print(Player.PlayerData.money.cash)
```

With `qbx:enableBridge` on, `qbx_core` answers that `qb-core` export and rebuilds the `QBCore` object, so `QBCore.Functions.GetPlayer(source)` returns a player you can read. The script does not know it is talking to Qbox. The Qbox-native way to write the same thing is `exports.qbx_core:GetPlayer(source)`, and that is what you migrate toward over time, but you do not have to rewrite every script on day one.

### A call you have to edit yourself

Inventory is where the bridge runs out. QBCore had its own `qb-inventory`, and Qbox uses `ox_inventory` instead. A direct QBCore inventory call has no clean translation:

```lua
-- QBCore inventory call: no equivalent on Qbox, must be rewritten
Player.Functions.AddItem('water', 1)

-- Qbox / ox_inventory replacement
exports.ox_inventory:AddItem(source, 'water', 1)
```

The shapes differ (ox_inventory takes the player `source`, not a player-bound method), so this is a manual edit, not something the bridge papers over. The rule of thumb: player, job, and money reads usually survive the bridge; inventory, custom metadata, and anything that poked QBCore internals usually needs a Qbox-native rewrite. That is the list you build in step 4.

  items={[
    { symptom: "Migrated live and broke the server", fix: "Always migrate on a test server first; cut production over only after the core loop is proven." },

    { symptom: "Old QBCore scripts behave as if Qbox is not there", fix: "The bridge was probably set to false.     { symptom: "Inventory exports fail after switching", fix: "qb-inventory is gone; Qbox uses ox_inventory. Rewrite AddItem and RemoveItem calls to the ox_inventory exports; the bridge does not cover these." },
    { symptom: "Player data missing after switch", fix: "Restore the backup; confirm the database schema and character ids map correctly before retrying.

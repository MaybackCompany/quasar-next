<Callout variant="fivem" title="Why Qbox, honestly">
  Qbox (qbx_core) is the actively maintained, ox-native successor to QBCore, built to keep QBCore scripts working where it can. It depends on oxmysql, ox_lib, and ox_inventory. You migrate for maintenance and modern dependencies, not because QBCore stops running overnight.
</Callout>

## A staged migration

<StepGroup>
  <Step num={1} title="Back up and stand up a test server" outcome="You can experiment without risking the live server.">
    Full database backup, then clone the server onto a test environment. Every step below happens on the test server first. Never migrate live.
  </Step>

  <Step num={2} title="Install the ox dependency stack" outcome="oxmysql, ox_lib, and ox_inventory are running.">
    Qbox is ox-native, so the modern dependency spine must be in place before qbx_core. Install and start oxmysql, ox_lib, and ox_inventory, in order, with no missing-dependency errors.
  </Step>

  <Step num={3} title="Install qbx_core and use the bridge" outcome="QBCore scripts run against Qbox where compatibility allows.">
    Install qbx_core and its compatibility layer. Many QBCore exports and events are bridged, so a lot of scripts work as-is. Expect a list of scripts that still need edits; that is normal, not failure.
  </Step>

  <Step num={4} title="Test script by script, then cut over" outcome="You know exactly what works before going live.">
    Go through your resources: core (banking, jobs, inventory) first, then the rest. Note what works, what needs an edit, and what to drop. Only cut production over once the core loop is solid on the test server.

    <ExpectedOutput>
      On the test server: players spawn, money/jobs/inventory work, and your core scripts run without console errors.
    </ExpectedOutput>
  </Step>
</StepGroup>

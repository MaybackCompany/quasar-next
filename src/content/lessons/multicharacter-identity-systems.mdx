<Callout variant="fivem" title="The identifier that everything hangs on">
  A player connects with a `license:` identifier (a Rockstar Online Services hash) that names the account. It is the conventional account anchor, though it can change if the player switches Rockstar accounts. A multi-character system maps that one license to several characters, each with its own generated `citizenid`. Every other resource must store data against the `citizenid`, not the license, or two characters on the same account will share money, items, and jobs.
</Callout>

## Build it

This is a server-only resource. There is no client file and no F8 output: identity lives entirely on the server, against the database, which is the whole point. You will run one command in the server console and read the proof there.

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_identity
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">

resources/qu_identity/

  </Step>

  <Step num={3} title="Write fxmanifest.lua and the tables" outcome="FiveM loads oxmysql first, and the database has the account/character shape.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script '@oxmysql/lib/MySQL.lua'
server_script 'server.lua'

dependencies {
    'oxmysql'
}
    ```

    Then run this SQL against your MariaDB before starting the resource. These two tables are the entire identity model in miniature:

    ```sql
CREATE TABLE IF NOT EXISTS qu_accounts (
  license VARCHAR(64) NOT NULL,
  PRIMARY KEY (license)
);

CREATE TABLE IF NOT EXISTS qu_characters (
  citizenid  VARCHAR(16) NOT NULL,
  license    VARCHAR(64) NOT NULL,
  first_name VARCHAR(40) NOT NULL,
  money      INT NOT NULL DEFAULT 0,
  PRIMARY KEY (citizenid),
  UNIQUE KEY uniq_license_name (license, first_name),
  FOREIGN KEY (license) REFERENCES qu_accounts (license)
);
    ```
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The license-to-character mapping and the per-character money rule are runnable.">
    Open `server.lua` and paste this:

    ```lua
local function makeCitizenId()
    return ('QU%07d'):format(math.random(0, 9999999))
end

local function ensureCharacter(license, firstName)
    local existing = MySQL.scalar.await(
        'SELECT citizenid FROM qu_characters WHERE license = ? AND first_name = ?',
        { license, firstName }
    )
    if existing then return existing end

    local citizenid = makeCitizenId()
    MySQL.insert.await('INSERT INTO qu_characters (citizenid, license, first_name) VALUES (?, ?, ?)',
        { citizenid, license, firstName })
    return citizenid
end

RegisterCommand('identitytest', function()
    local license = 'license:qu000000000000000000000000000000demo'

    MySQL.insert.await('INSERT IGNORE INTO qu_accounts (license) VALUES (?)', { license })

    local charA = ensureCharacter(license, 'Avery')
    local charB = ensureCharacter(license, 'Blake')

    MySQL.update.await('UPDATE qu_characters SET money = ? WHERE citizenid = ?', { 5000, charA })

    local moneyA = MySQL.scalar.await('SELECT money FROM qu_characters WHERE citizenid = ?', { charA })
    local moneyB = MySQL.scalar.await('SELECT money FROM qu_characters WHERE citizenid = ?', { charB })

    print(('[qu_identity] one license: %s'):format(license))
    print(('[qu_identity] character A %s money %d'):format(charA, moneyA))
    print(('[qu_identity] character B %s money %d'):format(charB, moneyB))
end, true)
    ```
  </Step>

  <Step num={5} title="Start and test it" outcome="Two characters share one license, but only one has the money.">
    Open `server.cfg` and add this line:

    ```text
    ensure qu_identity
    ```

    Save, then run this in the server console (txAdmin Live Console):

    ```text
    restart qu_identity
    ```

    Now run the test command in the same console:

    ```text
    identitytest
    ```

    <ExpectedOutput>
      [qu_identity] one license: license:qu000000000000000000000000000000demo
[qu_identity] character A QU0431785 money 5000
[qu_identity] character B QU2980014 money 0
    </ExpectedOutput>

    The two `QU` ids are random, so yours will differ. What must match is the shape: both characters hang off the same license line, character A holds 5000, and character B holds 0. You just stored money against a character, and the other character on the same account never saw it.

    <Callout variant="warning" title="Security: this command is restricted, and that is on purpose">
      The `true` at the end of `RegisterCommand('identitytest', ..., true)` marks the command as **restricted**: only a player or console with the matching ACE permission can run it. Any command that creates accounts, moves money, or touches identity must be permission-gated this way — never leave such a command runnable by every player. In production you would also derive the license from the connected player with `GetPlayerIdentifierByType(src, 'license')` and validate every value server-side, never from a hardcoded or client-sent string. The fixed demo license here exists only so the console (`source` 0) can run the example.
    </Callout>

## How it works

You ran one command and watched two characters on a single account hold different balances. That is the entire job of an identity system, and it lives in three places: the identifier you anchor on, the two tables that map one account to many characters, and the column you key downstream data against. We will go through each, then connect it back to what your real framework does on character selection.

### The identifier you anchor on

A connecting player does not arrive with one id. FiveM hands you a list of identifiers, and you can read them on the server with `GetPlayerIdentifiers(src)` or pull a single type with `GetPlayerIdentifierByType(src, 'license')`. A typical list looks like this:

```text
license:qu1a2b3c4d5e6f7a8b...
license2:1a2b3c...
steam:110000112345678
discord:281234567890123456
fivem:1234567
ip:203.0.113.42
```

Each one names the player from a different angle. `steam:` only exists if they launched through Steam. `discord:` only exists if they linked Discord. `ip:` changes the moment they move house. By convention frameworks anchor the account on `license:`, the Rockstar Online Services hash, because it is present for normal players and tied to their Rockstar account rather than a launcher or network. Be aware of its real limits: the `license:` value CHANGES if the player switches to a different Rockstar/Social Club account, and FiveM's docs name no identifier as always-present or impossible to spoof. For high-stakes use such as bans, Cfx.re recommends the `fivem:` (Cfx.re user) identifier instead. In production you would write `local license = GetPlayerIdentifierByType(src, 'license')`; the build above uses a fixed demo license so the command runs deterministically from the console, where `source` is `0` and there is no real player to read identifiers from.

<Checkpoint question="Why anchor the account on license: instead of steam: or ip:?">
  Because `license:` is normally present and is tied to the player's Rockstar account rather than a launcher or a network connection. `steam:` is missing for anyone who did not launch through Steam, `discord:` is missing unless they linked it, and `ip:` changes whenever the player's network changes, which would orphan their account and characters. `license:` is the conventional anchor because it has fewer of those failure modes for everyday players — but it is not bulletproof: the value is derived from the Rockstar account, so it CHANGES if the player switches Rockstar/Social Club accounts, and FiveM does not treat any identifier as unspoofable. For bans specifically, Cfx.re points to the `fivem:` identifier rather than `license:`.
</Checkpoint>

### The two-table model

The schema is the lesson in miniature. There are two tables and the relationship between them is the whole idea.

```sql
CREATE TABLE IF NOT EXISTS qu_accounts (
  license VARCHAR(64) NOT NULL,
  PRIMARY KEY (license)
);
```

`qu_accounts` is one row per human. Its primary key is the `license`, so the database guarantees a license appears at most once. This is the account layer: it knows the person exists, nothing more.

```sql
CREATE TABLE IF NOT EXISTS qu_characters (
  citizenid  VARCHAR(16) NOT NULL,
  license    VARCHAR(64) NOT NULL,
  first_name VARCHAR(40) NOT NULL,
  money      INT NOT NULL DEFAULT 0,
  PRIMARY KEY (citizenid),
  UNIQUE KEY uniq_license_name (license, first_name),
  FOREIGN KEY (license) REFERENCES qu_accounts (license)
);
```

`qu_characters` is one row per character, and a single license can own many rows here. Read the keys in order:

- `citizenid` is the **primary key**: a generated id, unique per character, that everything downstream will reference. It is deliberately separate from the license, because one license owns several citizenids.
- `license` is a plain column plus a **foreign key** back to `qu_accounts`. The foreign key means you cannot create a character for an account that does not exist, which keeps orphan characters out of the table.
- `UNIQUE KEY uniq_license_name (license, first_name)` stops the same account creating two characters with the same name, and it is what makes the build idempotent: re-running the test cannot duplicate Avery. Without a unique key, an upsert has nothing to collide on, so every run inserts again.

The `citizenid` is generated, never the license. That separation is the point: the account is one thing, a character is another, and the link is the `license` column on the character row.

### Generating and reusing the character id

```lua
local function makeCitizenId()
    return ('QU%07d'):format(math.random(0, 9999999))
end
```

A character id has to be unique and stable. Real frameworks generate a longer alphanumeric id; this is a teaching-sized version that pads a random number to seven digits, like `QU0431785`. The exact format does not matter. What matters is that it is generated once and then stored, so the character keeps the same id forever. (This demo does not seed `math.random`, so collisions are possible in theory; a real generator would seed it and check for an existing id before inserting, which is why the `(license, first_name)` unique key acts as the backstop here.)

```lua
local function ensureCharacter(license, firstName)
    local existing = MySQL.scalar.await(
        'SELECT citizenid FROM qu_characters WHERE license = ? AND first_name = ?',
        { license, firstName }
    )
    if existing then return existing end

    local citizenid = makeCitizenId()
    MySQL.insert.await('INSERT INTO qu_characters (citizenid, license, first_name) VALUES (?, ?, ?)',
        { citizenid, license, firstName })
    return citizenid
end
```

This is read-then-create. First it asks the database whether this account already has a character with that name, using `MySQL.scalar` to pull back a single value. If a `citizenid` comes back, the character exists and you reuse it. If not, you generate a fresh id and insert the row. That guard is why running `identitytest` twice does not create four characters: the second run finds Avery and Blake already there and returns their existing ids. The `(license, first_name)` unique key is the database-level backstop for the same rule.

### The column that decides everything

```lua
MySQL.update.await('UPDATE qu_characters SET money = ? WHERE citizenid = ?', { 5000, charA })
```

Here is the single most important line in the lesson. The money is set `WHERE citizenid = ?`, keyed to character A's id. Character B has a different `citizenid`, so this update does not touch B's row. When you read both back, A has 5000 and B has 0. That is correct multi-character behavior, and it falls out of one decision: keying money on `citizenid`.

Now picture the bug the audit and every shared-money report come from. Suppose a banking script stored balances against the account instead:

```lua
-- The bug: money keyed on the license (the account), not the character
MySQL.update.await('UPDATE qu_balances SET money = ? WHERE license = ?', { 5000, license })
```

Both of your characters share that one license. A row keyed on `license` is shared by every character on the account, so depositing 5000 as Avery means Blake logs in to find 5000 too. Switch characters, the money follows, because there was only ever one balance row for the whole account. The fix is never some clever sync; it is to key the balance on `citizenid` so each character owns its own row. One wrong column in one query is the difference between a working economy and "all my characters share money" in your bug channel.

<Checkpoint question="A player reports that money on one character shows up on all their characters. Where do you look first?">
  The query that writes money. Find the `UPDATE` or `INSERT` that stores the balance and look at its `WHERE` clause and the column it keys on. If it keys on `license`, `identifier`, or any account-level id instead of the character's `citizenid`, that is the bug: one row is shared by every character on the account. Change the key to `citizenid` (and migrate the existing rows so each character gets its own). The same audit applies to inventory and jobs: anything that should be per-character must key on the character id, not the account.
</Checkpoint>

### What your framework does on selection

The build is a hand-cranked version of what a real multichar resource does for you. When a player picks a slot in a character menu, the framework runs the same shape: it has the player's `license`, it loads or creates the matching character row, and then it sets that `citizenid` as the active character for the session so every other resource reads the right one.

- **Qbox** uses `qbx_multicharacter`, generates a `citizenid`, and on selection you read the active character server-side with the core export `exports.qbx_core:GetPlayer(src)`, whose `PlayerData.citizenid` is the active character. Qbox is a community-maintained fork of QBCore built on the ox stack (`ox_lib` and `oxmysql` are hard dependencies).
- **QBCore** uses `qb-multicharacter` with the same `citizenid` concept and `QBCore.Functions.GetPlayer(src)`.
- **ESX Legacy** uses `esx_multicharacter` and keys characters by the `identifier` (the license) plus a character slot, exposed through `ESX.GetPlayerFromId(src)`. ESX does not generate a separate citizenid the way QBCore and Qbox do; it leans on the identifier plus slot.

The practical rule when you add any new script: find where it stores per-player data and confirm it keys on the framework's active character id, not the raw license. If it keys on the license, you have just reintroduced the shared-money bug for every player with more than one character.

## Try it yourself

<Exercise>
  Add a third character and a per-character inventory so you prove the same rule on a second data type. Extend `server.lua`: after the existing characters, create a third with `ensureCharacter(license, 'Casey')`, then give character A a single item keyed on its `citizenid` and read it back for both A and Casey.

  ```lua
  local charC = ensureCharacter(license, 'Casey')

  MySQL.insert.await(
      'INSERT INTO qu_inventory (citizenid, item, count) VALUES (?, ?, ?) ' ..
      'ON DUPLICATE KEY UPDATE count = VALUES(count)',
      { charA, 'water', 3 }
  )

  local itemsA = MySQL.scalar.await('SELECT count FROM qu_inventory WHERE citizenid = ? AND item = ?', { charA, 'water' })
  local itemsC = MySQL.scalar.await('SELECT count FROM qu_inventory WHERE citizenid = ? AND item = ?', { charC, 'water' })
  print(('[qu_identity] A water %d, C water %s'):format(itemsA or 0, tostring(itemsC)))
  ```

  First create the table with a `UNIQUE KEY (citizenid, item)` so the upsert has a key to collide on. Restart `qu_identity`, run `identitytest`, and confirm A shows `3` while C shows `nil`. You have now proven the citizenid rule on inventory as well as money, which is exactly the audit you run on every new script before you trust it.
</Exercise>

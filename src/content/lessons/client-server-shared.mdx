## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_client_server_shared
    ```
  </Step>

  <Step num={2} title="Create the three folders and their files" outcome="The three-folder convention exists on disk.">
    Create this exact layout. The folder names are the convention you will see everywhere:

    ```text
resources/qu_client_server_shared/
  fxmanifest.lua
  shared/
    config.lua
  server/
    main.lua
  client/
    main.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM loads each folder into the right runtime.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

shared_scripts {
    'shared/*.lua'
}

server_scripts {
    'server/*.lua'
}

client_scripts {
    'client/*.lua'
}
    ```

    Older tutorials add a `-- Lua 5.4 is the default runtime in 2026; the lua54 directive is no longer needed` line here. As of June 2025 that setting is deprecated and ignored: Lua 5.4 is the only Lua runtime now, so you leave it out.
  </Step>

  <Step num={4} title="Write the shared config" outcome="One config table that both sides will read.">
    Open `shared/config.lua` and paste this:

    ```lua
Config = {}
Config.Greeting = 'three folders, one resource'
Config.MaxDistance = 25.0
    ```
  </Step>

  <Step num={5} title="Write the server code" outcome="The server reads shared config and fires an event across the network.">
    Open `server/main.lua` and paste this:

    ```lua
print('[qu_client_server_shared] server read Config.Greeting: ' .. Config.Greeting)

RegisterCommand('splitproof', function(src)
    print('[qu_client_server_shared] server fired splitproof for src ' .. src)
    TriggerClientEvent('qu_client_server_shared:proof', src, Config.MaxDistance)
end, false)
    ```
  </Step>

  <Step num={6} title="Write the client code" outcome="The client reads the same shared config and receives the server's event.">
    Open `client/main.lua` and paste this:

    ```lua
print('[qu_client_server_shared] client read Config.Greeting: ' .. Config.Greeting)

RegisterNetEvent('qu_client_server_shared:proof', function(maxDistance)
    print('[qu_client_server_shared] client received event, Config.MaxDistance = ' .. maxDistance)
end)
    ```
  </Step>

  <Step num={7} title="Start and test it" outcome="The shared value appears on both sides, and the event crosses the network.">
    Open `server.cfg` and add this line:

    ```text
    ensure qu_client_server_shared
    ```

    Save. Because this resource is brand new and has never started, do NOT use `restart` yet — `restart` only works on a resource the server is already running. In the server console (the txAdmin Live Console you opened in the prerequisites), first make the server notice the new files, then start the resource:

    ```text
    refresh
    ensure qu_client_server_shared
    ```

    `refresh` rescans the `resources` folder so FiveM sees the files you just created; `ensure` then starts the resource (and from now on, after you edit a file, `restart qu_client_server_shared` is the command you reuse because the resource is already running).

    Join the server, open the in-game chat box with T, and run:

    ```text
    /splitproof
    ```

    The command uses `false` because this proof action only prints and sends a number back to the player who ran it. Running it as a connected player gives the handler a real player source such as `1`. The server console has source `0`, which is not a connected client and therefore cannot receive `TriggerClientEvent`.

    <ExpectedOutput>
      Server: [qu_client_server_shared] server read Config.Greeting: three folders, one resource
Server: [qu_client_server_shared] server fired splitproof for src 1   (your number may differ — it is your player id, not always 1)
F8: [qu_client_server_shared] client read Config.Greeting: three folders, one resource
F8: [qu_client_server_shared] client received event, Config.MaxDistance = 25.0
    </ExpectedOutput>
  </Step>
</StepGroup>

## How it works

It worked, so now take it apart. The whole point of this lesson is one idea that beginners get wrong constantly: a FiveM resource is not one program. It is two programs that happen to ship in the same folder, plus a shared layer that gets copied into both. Once you see it that way, the folder split stops being a style choice and becomes the map of where your code actually runs.

### Two runtimes, not one

When the resource starts, FiveM spins up two separate Lua virtual machines. One runs on the server machine. A separate copy runs inside every connected player's game. They do not share memory. A variable you set on the server is invisible to the client unless you deliberately send it across the network.

That is why the proof printed in two different places. The server's print landed in txAdmin Live Console because `server/main.lua` only ever runs on the server. The client's print landed in F8 because `client/main.lua` runs inside your game. Same resource, two consoles, because two runtimes.

### What shared_script actually does

Here is the part that trips people up. `Config` is not a synced object. The `shared/config.lua` file is loaded into BOTH Lua VMs, so the code runs twice, once per side, and `Config` becomes a plain global on each side independently.

Read that again, because it is the core mental model. There is no magic link between the server's `Config` and the client's `Config`. They are two separate tables that happen to start identical because they were built from the same source file. If the server later does `Config.MaxDistance = 99.0`, the client's copy is still `25.0`. Shared means "the same starting code runs on both sides," not "the same live value lives on both sides."

That is exactly why config belongs in shared. Both sides need to agree on constants like a max interaction distance or a label. Putting those constants in one shared file means both VMs read the same numbers without an event round-trip on startup. One source of truth, zero network traffic to read it.

### The decision rule for where a line goes

Stop guessing per file. Use this rule for every line you write:

- **client/** if the line touches the local player: the ped, the camera, drawing on screen, NUI, key presses, anything that only makes sense for one person's game. Natives like `PlayerPedId()`, `DrawMarker`, `SetNuiFocus` live here.
- **server/** if the line touches authority: the `source` of an event, money, inventory, the database, bans, or anything a cheating client must not be allowed to decide. This side is the referee. It is the only side a player cannot edit.
- **shared/** only for data and helpers that are safe on both sides: a number, a label, a list of job names, or a function that turns cents into a money string. No secrets, no authoritative mutable state, and no native unless the official reference marks it available on both client and server.

When you are unsure, ask one question: does this line decide something a player could cheat? If yes, it goes in server. The server is the only side you control; the client runs on the player's machine and they can read and change it.

### What must never go in shared

Two things will burn you if they end up in `shared/`.

The first is secrets. Database connection strings, API keys, admin passwords. Anything in `shared/` is shipped to every client and a determined player can dump it out of their game memory. A secret in shared is a secret you have handed to the public. Secrets belong in `server/` or, better, in convars set in `server.cfg`.

The second is side-specific natives. Natives like `GetPlayerIdentifiers` or `DropPlayer` only exist in the server runtime. If a shared file calls one, it loads fine on the server but throws `attempt to call a nil value` when the client runs that same file. Some Cfx natives are available on both sides, so check the API-set badge in the official native reference rather than applying a blanket "no natives" rule.

### Why the glob import works and why load order matters

Look back at the manifest. You wrote `shared_scripts { 'shared/*.lua' }`, not a single filename. The `*` is a glob: FiveM expands it to every matching `.lua` file in that folder. Add `shared/jobs.lua` tomorrow and the manifest already includes it. The official manifest reference does not promise an alphabetical order for glob matches, so list files explicitly whenever one shared file depends on another.

The curly-brace list also matters for a second reason: order. FiveM loads `shared_scripts` before `client_scripts` and `server_scripts`. That ordering is the reason your prints worked. By the time `server/main.lua` and `client/main.lua` run, `shared/config.lua` has already run on that side, so `Config` exists. Flip the order, load a script before the shared file that defines `Config`, and you get `attempt to index a nil value (global 'Config')`. Shared first is not a suggestion; it is what makes the config available when the rest of your code reads it.

One more detail on the manifest key for the future. When this resource depends on another (say `oxmysql` or `ox_lib`, the community Overextended libraries you will meet later), the canonical key is the plural `dependencies { 'oxmysql' }`. The singular `dependency 'oxmysql'` is tolerated but the plural braced form is the one to learn.

<Checkpoint question="You set Config.MaxDistance = 25.0 in shared. On the server you later write Config.MaxDistance = 99.0. What does the client print when the event fires, and why?">
  The client still prints `25.0`. The server and the client each loaded their own copy of `shared/config.lua`, so they hold two independent `Config` tables. Editing the server's copy does nothing to the client's copy. Shared means the same starting code ran on both sides, not that the live value is synced. If you genuinely needed the new value on the client, you would have to send it, which is exactly what the `TriggerClientEvent` line does: it pushes `Config.MaxDistance` across the network as the event payload, and the client reads what arrived in its handler, not its own table.
</Checkpoint>

### When you reach for this split

You reach for it on the very first file of every resource, not as cleanup later. The moment a resource has both a server rule and a client display, you split it. A shop: the client draws the marker and menu, the server validates the purchase and moves the money, and shared holds the item list and prices both sides reference. A job: client handles the on-screen prompts, server tracks who is clocked in, shared lists the job names. Set the three folders up first and the decision rule answers "where does this go" for you every time you add a line.

## If something went wrong

<FailureList
  items={[
    { symptom: "attempt to index a nil value (global 'Config')", fix: "A client or server file ran before shared loaded, or the file is in the wrong folder. Confirm config.lua is inside shared/ and the manifest lists shared_scripts before client_scripts and server_scripts." },
    { symptom: "Could not find dependency / no script files matched 'shared/*.lua'", fix: "The folder name or glob is wrong. The path is case sensitive and relative to the resource root. Confirm the folder is named shared (not Shared) and the file ends in .lua." },
    { symptom: "Server console prints, but F8 shows nothing", fix: "The client print and event live in client/main.lua, which only runs in-game. You must join the server, not just restart it from the console. Read F8 while connected." },
    { symptom: "attempt to call a nil value when the resource starts", fix: "A shared file is calling a native that is unavailable on one side. Check the native's API-set badge and move the call into server/ or client/ unless it is documented for both." },
    { symptom: "splitproof prints source 0 but nothing reaches F8", fix: "You ran the command from txAdmin. Source 0 is the server console, not a player. Join the server and run /splitproof in chat, or type splitproof without a slash in F8." },

## Try it yourself

<Exercise>
  Prove the "two independent copies" claim with your own eyes. In `server/main.lua`, set `Config.Greeting = 'server edited this'` on the line right after the existing server print, then add a second command:

  ```lua
  RegisterCommand('checkconfig', function(src)
      print('[qu_client_server_shared] server Config.Greeting is now: ' .. Config.Greeting)
      TriggerClientEvent('qu_client_server_shared:checkconfig', src)
  end, false)
  ```

  In `client/main.lua`, add the matching handler:

  ```lua
  RegisterNetEvent('qu_client_server_shared:checkconfig', function()
      print('[qu_client_server_shared] client Config.Greeting is still: ' .. Config.Greeting)
  end)
  ```

  Restart `qu_client_server_shared`, join, and run `/checkconfig` in chat. The server prints `server edited this` while the client still prints `three folders, one resource`. You just watched the two VMs disagree, which is the whole lesson. Now move that `Config.Greeting = ...` edit into `shared/config.lua` instead and confirm both sides agree again, because now both copies start from the changed source.
</Exercise>

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_network_onesync
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_network_onesync/
  fxmanifest.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'
    ```

    Older tutorials add a `lua54 'yes'` line here. As of June 2025 that setting is deprecated and ignored: Lua 5.4 is the only Lua runtime now, so you leave it out.
  </Step>

  <Step num={4} title="Write the lesson code" outcome="A real instance manager that partitions players exists.">
    Open `server.lua` and paste this:

    ```lua
-- Move yourself into a heist instance (its own routing bucket).
RegisterCommand('heist', function(src)
    local bucket = 10

    -- 1. Make this bucket a clean, locked-down instance.
    SetRoutingBucketPopulationEnabled(bucket, false)
    SetRoutingBucketEntityLockdownMode(bucket, 'strict')

    -- 2. Move the player into it. They now share a world only with
    --    other players in bucket 10.
    SetPlayerRoutingBucket(src, bucket)

    print(('[qu_network_onesync] moved %s (id %s) into heist bucket %d'):format(GetPlayerName(src), src, bucket))
end, false)

-- Send yourself back to the shared world (bucket 0).
RegisterCommand('leaveheist', function(src)
    SetPlayerRoutingBucket(src, 0)
    print(('[qu_network_onesync] returned %s (id %s) to bucket 0'):format(GetPlayerName(src), src))
end, false)

-- Read which bucket a player is currently in.
RegisterCommand('mybucket', function(src)
    print(('[qu_network_onesync] %s is in bucket %d'):format(GetPlayerName(src), GetPlayerRoutingBucket(src)))
end, false)
    ```

    <Callout variant="warning" title="These commands are unrestricted on purpose, for the lesson only">
      The `false` at the end of each `RegisterCommand` is the `restricted` flag set to off, so ANY connected player can run `/heist`, `/leaveheist`, and the `/heistwith` from the exercise. That is fine for a local test, but on a live server it means anyone can self-instance or pull another player into a bucket. Before shipping anything like this, set `restricted` to `true` and grant the command through an ACE permission (for example `add_ace group.admin command.heist allow` in `server.cfg`). Routing-bucket control should never be a public command.
    </Callout>
  </Step>

  <Step num={5} title="Start and test it" outcome="The expected proof appears in the txAdmin Live Console.">
    Open `server.cfg` and add this line:

    ```text
    ensure qu_network_onesync
    ```

    Save `server.cfg`. Then type this into the server console (the txAdmin Live Console, or the black FXServer console window if you run it directly), not PowerShell and not the in-game chat:

    ```text
    restart qu_network_onesync
    ```

    Join the server. In the in-game chat (press T) type `/heist`, press Enter, then `/mybucket`. (In the F8 console you would type the same commands without the leading slash.) The printed lines appear in the server console, not on your screen:

    <ExpectedOutput>
      [qu_network_onesync] moved Alex (id 3) into heist bucket 10
[qu_network_onesync] Alex is in bucket 10
    </ExpectedOutput>

    `Alex` and `id 3` are illustrative: you will see your own in-game name and the server id you were assigned on join. The lines that must match exactly are the `[qu_network_onesync]` prefix and `bucket 10`.
  </Step>

  <Step num={6} title="Prove the invisibility" outcome="Two players confirm a routing bucket really partitions the world.">
    A bucket number printing is not proof of anything. The whole point of this feature is that players in different buckets cannot see each other. So test that directly.

    Get two players on the server (a friend, or a second FiveM client on another machine). Have both stand next to each other so you can see each other's character. Then on player one only, run:

    ```text
    /heist
    ```

    Player one is now in bucket 10. Player two is still in bucket 0. Watch what happens: player one's character vanishes from player two's screen, and player two vanishes from player one's. You are standing on identical coordinates and you cannot see each other. Run `/leaveheist` on player one and you reappear to each other instantly. That disappearance is the routing bucket doing its only job.
  </Step>
</StepGroup>

## How it works

You moved a player into a private slice of the world and proved no one outside it could see them. Now take the code apart so the next time you build an instance, an interior, or a minigame arena, you are choosing each native on purpose rather than copying. There are three natives doing the real work, and one prerequisite that has to be true for any of them to fire.

### The mental model: a bucket is a parallel world

A routing bucket is a server-managed dimension. Picture the server holding several stacked copies of Los Santos. Bucket 0 is the default, shared world where everyone spawns. Bucket 10 is a separate copy sitting on top of it at the same coordinates. The rule that makes buckets useful is simple: a player in bucket A receives network traffic only for players and entities that are also in bucket A. Bucket 10 and bucket 0 share the same map geometry, but they do not share each other's players, vehicles, or spawned objects. Two players at coordinate (0, 0, 72), one in bucket 0 and one in bucket 10, are invisible to each other because the server never tells either client the other exists.

Buckets are useful for separate sessions, racing lobbies, and match instances. Cfx.re's routing-bucket guidance explicitly says not to use them for interiors; interior concealment has different requirements and uses conceal natives or an interior system designed for that job.

### `SetPlayerRoutingBucket`: the one line that moves a player

```lua
SetPlayerRoutingBucket(src, bucket)
```

This is the native that actually relocates a player between worlds. `src` is the server ID of the player (the same value `source` gives you inside a command or net event), and `bucket` is the integer dimension you want them in. The instant this runs, the server stops sending that player updates about bucket 0 and starts sending them updates about bucket 10 instead. To everyone left in bucket 0, the player simply disappears. `GetPlayerRoutingBucket(src)` is the read-only counterpart that returns the player's current bucket, which is what `/mybucket` uses.

On its own, this native is enough to hide players from each other. But an empty bucket is not yet a usable instance, which is what the other two natives fix.

### `SetRoutingBucketPopulationEnabled`: kill the ambient world

```lua
SetRoutingBucketPopulationEnabled(bucket, false)
```

By default a fresh bucket still grows GTA's ambient population: pedestrians, traffic, parked cars. For a heist room or a clean arena that is noise you do not want. Passing `false` stops the engine spawning ambient peds and vehicles inside that bucket, so the instance holds only the players and entities you put there. Leave it `true` (the default) when you want an instance that still feels like a living city, such as a parallel free-roam lobby.

### `SetRoutingBucketEntityLockdownMode`: decide who may spawn things

```lua
SetRoutingBucketEntityLockdownMode(bucket, 'strict')
```

This native controls who is allowed to create networked entities inside the bucket. It takes one of three string modes:

- `'inactive'` is the default. Clients can create any entity they want, which is normal free-roam behavior.
- `'relaxed'` blocks only script-owned entities created by clients.
- `'strict'` blocks all client-created entities. Only the server may spawn entities into the bucket.

For a controlled mission instance, `'strict'` is the one you want: the server creates the entities the mission needs, and clients create none. Creation authority and simulation ownership are different: a server-created entity can still migrate to a client for simulation, so strict mode does not mean every entity permanently reports the server as its network owner.

### Where OneSync and entity ownership tie in

Every one of these natives depends on OneSync being enabled, because OneSync is what moves entity authority from the clients to the server. On a legacy non-OneSync server there is no server-side dimension to assign players to, so the calls do nothing. That is why the prereq and the second callout insist on `set onesync on` before you write a line.

Once OneSync is on, the server owns the networked world. You can ask who currently owns a given entity with `NetworkGetEntityOwner(entity)`, which returns the player index currently responsible for simulating it (or -1 when the server itself owns it, as happens for server-created entities). You will not call it in this lesson's three commands, but it is the read side of the same authority model: routing buckets decide which world an entity lives in, lockdown mode decides who may create entities there, and `NetworkGetEntityOwner` tells you who is simulating one right now.

<Checkpoint question="Two players stand at the same coordinates. One runs /heist, the other does nothing. Why does each player vanish from the other's screen?">
  Because `/heist` calls `SetPlayerRoutingBucket(src, 10)`, which moves that player into bucket 10 while the other stays in bucket 0. A routing bucket is a server-managed dimension, and the server only sends a client network updates for players and entities in the same bucket. With the two players now in different buckets, the server stops telling either client that the other exists, so each disappears even though their world coordinates are identical. Run `/leaveheist` to call `SetPlayerRoutingBucket(src, 0)` and they share bucket 0 again, so they reappear. None of this would work without OneSync enabled, because OneSync is what gives the server the authority to partition the world in the first place.
</Checkpoint>

### When you reach for this pattern

Routing buckets are the answer almost every time a server needs more than one copy of the same place. Three concrete cases:

- **Instanced missions and heists.** Four players enter a vault. They should share a world with each other and no one else. Put the four of them in the same bucket, set it strict, disable population, and the vault is theirs alone.
- **Separate match or mission sessions.** Two parties can run the same mission coordinates without sharing players, vehicles, or props. Do not copy this pattern for ordinary interiors; follow Cfx.re's conceal/interior guidance instead.
- **Lobbies and arenas.** Race grids, deathmatch rooms, and waiting lobbies each get a bucket so countdowns, spawns, and props in one match never leak into another.

## Try it yourself

<Exercise>
  Turn the single hard-coded heist bucket into a small party system that puts a group of players into one shared instance. Add a `/heistwith` command that takes a target player ID, picks a fresh bucket number, and moves both the caller and the target into it:

  ```lua
  local nextBucket = 100

  RegisterCommand('heistwith', function(src, args)
      local target = tonumber(args[1])
      if not target then
          print('[qu_network_onesync] usage: /heistwith <playerId>')
          return
      end

      local bucket = nextBucket
      nextBucket = nextBucket + 1

      SetRoutingBucketPopulationEnabled(bucket, false)
      SetRoutingBucketEntityLockdownMode(bucket, 'strict')

      SetPlayerRoutingBucket(src, bucket)
      SetPlayerRoutingBucket(target, bucket)

      print(('[qu_network_onesync] put %s and %s into shared bucket %d'):format(src, target, bucket))
  end, false)
  ```

  Restart `qu_network_onesync`, then with three players connected run `/heistwith <id of player two>` as player one. Confirm players one and two can see each other but player three (still in bucket 0) sees neither of them. The `&lt;` and `&gt;` in the usage text are placeholders for a real player ID. You have just built the core of every instanced mission system: pick a free bucket, lock it down, and drop your party in together.
</Exercise>

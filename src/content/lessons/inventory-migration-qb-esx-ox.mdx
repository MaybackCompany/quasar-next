<Callout variant="danger" title="Back up before you migrate">
  Inventory data is player property. Take a full database backup before you touch anything. Run this in your server's terminal (Windows PowerShell, or the Linux shell on a VPS) from the folder you want the file saved in — NOT in the FXServer console: `mysqldump -u root -p your_db > backup_pre_ox.sql` (replace `your_db` with your database name; `-p` makes it prompt for the MySQL/MariaDB password). On newer MariaDB the binary may be `mariadb-dump`. A failed inventory migration without a backup means wiped player items and a community revolt.
</Callout>

<Callout variant="fivem" title="A note on frameworks in 2026">
  When older guides say QB, they usually mean QBCore. QBCore and ESX (specifically ESX Legacy) are the two established, most widely used frameworks — pick the one your scripts and community already run on. Qbox is a newer, ox-native fork of QBCore that keeps QBCore compatibility; it is worth knowing and it is what this migration lesson targets, but it is not a default you must switch to. ox_inventory, oxmysql, and ox_lib are community resources (originally Overextended, now maintained under the CommunityOx org), not official Cfx.re ones, so you install them from their GitHub releases and read their docs (overextended.dev still resolves; the current docs also live at coxdocs.dev), not the FiveM docs.
</Callout>

## Plan the migration

<StepGroup>
  <Step num={1} title="Map what touches the inventory" outcome="You know every script that adds, removes, or checks items.">
    List the resources that call inventory functions: shops, jobs, crafting, drugs, anything that grants or takes items. These are what will break, and what you must update or bridge.
  </Step>

  <Step num={2} title="Understand the compatibility bridge" outcome="You know whether old scripts can keep working during the move.">
    ox_inventory ships compatibility shims that re-expose the common framework inventory exports (the QB `AddItem`/`RemoveItem`/`GetItemByName` calls and the ESX `addInventoryItem`/`removeInventoryItem`/`getInventoryItem` calls) so many scripts work with minimal change. The docs are explicit that these are best-effort: anything that read the framework's raw inventory table or used a niche export still needs real edits. Plan for both.
  </Step>

  <Step num={3} title="Migrate the items definitions and the data" outcome="Existing items map into the ox_inventory format and player data carries over.">
    Convert your framework `items` (QB `shared/items.lua` or the ESX `items` table) into ox_inventory's `data/items.lua` format, then migrate the existing player inventory rows. For ESX, ox_inventory ships an official migration command: with the server running, type `convertinventory esx` into the FXServer console (the same window where you see server logs, or the txAdmin Live Console), then `convertinventory esxproperty` if you use property storage, and restart the server when it finishes. For QBCore, ox_inventory ships `convertinventory qb`, which migrates both player and vehicle inventories (and maps `QBShared.Items` into ox_inventory's item data); run it the same way and restart the resource when it finishes. Either way, do it on a copy of the database first, against your backup. The next section shows the exact shape difference so you understand what the converter (or your script) is doing.
  </Step>

  <Step num={4} title="Test on a copy before going live" outcome="Items survive and scripts still grant/take correctly.">
    On a test server with the backup restored: load a character, confirm their items are intact, then run each item-touching script (buy, sell, job reward) and confirm it works.

    <ExpectedOutput label="Observable proof (illustrative — verify in-game, not from a console line)">
      Log the test character into the test server and open the inventory (default key F2). The items you expect are present with the right counts and the bag is not empty. ox_inventory does not print a per-character "loaded N items" line, so the proof is the in-game inventory itself, not console text.
    </ExpectedOutput>
  </Step>
</StepGroup>

## How it works

The plan above hides one genuinely hard step inside Step 3. The reason inventory migrations wipe items is that QB, ESX, and ox_inventory do not store items the same way. If you copy rows across blind, the new system reads garbage and shows an empty bag. So before you map anything, you need to see the shapes.

### The data-shape difference

Older ESX setups typically stored each character's items in the `users` row (the `inventory` column, a JSON string) as an object keyed by item name, like `{"bread":2,"water":1}`; exact shape varies by ESX version and inventory resource, so always inspect YOUR own `users.inventory` value before transforming. QBCore stored a JSON array of slot objects on the player row, like `[{"name":"bread","amount":2,"slot":1}]`. ox_inventory standardises on a single JSON column (`inventory`) on the character row, holding an array of slot objects with `name`, `count`, `slot`, and an optional `metadata` table:

```json
[
  { "name": "bread", "count": 2, "slot": 1 },
  { "name": "water", "count": 1, "slot": 2, "metadata": {} }
]
```

The migration is therefore a transform, not a copy. The two fields that trip people up: ESX/QB call the quantity `amount`, ox_inventory calls it `count`; and the ESX object form has no slots at all, so you have to assign `slot` numbers as you convert. Metadata (durability, serials, registration) only exists if the old system tracked it, so most items map with an empty or absent `metadata`.

### What an items definition looks like

ox_inventory will not show an item it has never heard of, so every item name in the data above must exist in `data/items.lua`. A single definition looks like this:

```lua
['bread'] = {
    label = 'Bread',
    weight = 100,
    stack = true,
    close = true,
    description = 'Fills you up a little'
},
```

The field that actually matters for a migration is the key (`['bread']`): it must match the `name` you wrote into the inventory JSON exactly, lowercase and identical. `label` is the display name, `weight` drives the new weight-based limit (ox_inventory uses weight, not slot count, as the real cap), and `stack` decides whether two of the same item share a slot. Any item that exists in player data but is missing from `items.lua` silently disappears on load, which is the most common cause of "some items came back, some did not".

### What the bridge actually covers

The compatibility bridge from Step 2 intercepts the framework's inventory exports and reroutes them into ox_inventory. In practice that means the QB `exports.ox_inventory:` shims for `AddItem`, `RemoveItem`, `GetItemByName`, and `GetItemCount`, and the ESX equivalents `addInventoryItem`, `removeInventoryItem`, and `getInventoryItem`. A script that called those exports keeps working. A script that reached past the export and read `Player.PlayerData.items` directly, or queried the old inventory column in SQL, is not covered by best-effort and has to be edited to ask ox_inventory instead. That is the line between "the bridge handled it" and "I have to touch this script".

<Exercise>
  Before you migrate anything, write the transform by hand on one fake character so the shape sticks. Take this old ESX-style row and convert it into the ox_inventory array:

  ```json
  { "bread": 2, "water_bottle": 1, "phone": 1 }
  ```

  The answer is an array of slot objects where `amount` becomes `count`, you assign `slot` 1, 2, 3 in order, and every name must already exist as a key in `data/items.lua`:

  ```json
  [
    { "name": "bread", "count": 2, "slot": 1 },
    { "name": "water_bottle", "count": 1, "slot": 2 },
    { "name": "phone", "count": 1, "slot": 3 }
  ]
  ```

  If you can do this for one character on paper, you can write the SQL or Lua that does it for ten thousand. Two safety rules when you do: (1) never run the transform against the live database first — run it only against a restored copy until items survive; (2) if you build SQL in Lua via oxmysql, always use `?` placeholders with a params table, never string-concatenate item names or values into the query (that both breaks on apostrophes and is an injection risk). If you cannot do the by-hand transform, you are not ready to run it against real data yet — and for ESX, prefer the built-in `convertinventory esx` command over a hand-written script.
</Exercise>

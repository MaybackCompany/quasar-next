## The frameworks at a glance

Each framework has its own docs and its own way of spelling the same ideas. Bookmark the one your server actually runs. Qbox describes itself as a QBCore successor and keeps a QB compatibility bridge; ESX Legacy and QBCore remain separate ecosystems. `ox_lib` is optional utility code, not a requirement of ESX or QBCore. The maintained 2026 `ox_lib`, `oxmysql`, `ox_target`, and `ox_inventory` repositories live under **Community Ox**. They are community resources, not Cfx.re platform features.

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_frameworks
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_frameworks/
  fxmanifest.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

shared_script '@ox_lib/init.lua'
server_script 'server.lua'

dependencies {
    'ox_lib'
}
    ```

    The canonical manifest key is the plural `dependencies { 'ox_lib' }`. The singular `dependency 'ox_lib'` is still tolerated, but show the plural so it reads the same as every modern resource. Older tutorials add a `lua54 'yes'` line; as of June 2025 that is deprecated and ignored, because Lua 5.4 is now the only Lua runtime. Leave it out.
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `server.lua` and paste this:

    ```lua
local function detectFramework()
    if GetResourceState('qbx_core') == 'started' then return 'qbox' end
    if GetResourceState('qb-core') == 'started' then return 'qbcore' end
    if GetResourceState('es_extended') == 'started' then return 'esx' end
    return 'none'
end

RegisterCommand('fwprobe', function()
    local fw = detectFramework()
    lib.print.info(('[qu_frameworks] detected framework: %s'):format(fw))
end, true)
    ```
  </Step>

  <Step num={5} title="Start and test it" outcome="The expected proof appears in the correct console.">
    Open `server.cfg` and add this line, after the line that ensures `ox_lib`:

    ```text
    ensure qu_frameworks
    ```

    Save, then run:

    ```text
    restart qu_frameworks
    ```

    Run this test from the txAdmin Live Console:

    ```text
    fwprobe
    ```

    <ExpectedOutput>
      [qu_frameworks] detected framework: none
    </ExpectedOutput>

    On a bare practice server with no framework installed, the answer is `none`, and that is the correct, passing result. It means your detection works and simply found nothing running. Install ESX, QBCore, or Qbox and ensure it before `qu_frameworks`, restart, and the same command prints `esx`, `qbcore`, or `qbox` instead. You will see why below.
  </Step>
</StepGroup>

## How it works

You ran it and a line appeared. The detection is the easy half. The valuable half is what the framework name unlocks: once you know which one is running, you know how to ask it who the player is, how much money they have, and what job they are on. That is the translation guide this whole lesson exists for. Take the code apart first, then learn the three idioms you will read every day.

### What GetResourceState actually returns

```lua
if GetResourceState('es_extended') == 'started' then return 'esx' end
```

`GetResourceState` is a FiveM *native* — a built-in function the game server itself provides (as opposed to a framework export or an ox_lib call), so it works the same on every server regardless of which framework is installed. You hand it a resource name as a string and it tells you the lifecycle state of that resource. The documented states are `missing`, `started`, `starting`, `stopped`, `stopping`, `uninitialized`, and `unknown`. The only state that means *this framework is live and its exports are callable* is `started`. So the probe is really asking, in order, is `qbx_core` started, then `qb-core`, then `es_extended`, and it returns the first match. The order matters: it checks Qbox first, then QBCore, then ESX, and returns the moment it finds one. On a server with none of them, every check fails and the function falls through to `return 'none'`. That is why a bare practice server prints `none`. Nothing is wrong; there is simply no framework resource in the `started` state to find.

The resource names are the real folder names each framework ships under, and they are easy to mix up:

- **ESX Legacy** runs under `es_extended`.
- **QBCore** runs under `qb-core` (with a hyphen).
- **Qbox** runs under `qbx_core` (with an underscore and an x).

If you check the wrong string, detection silently returns `none` even though the framework is running. Match the folder name exactly.

### Why the ox_lib dependency earns its place

In the old version of this lesson, `ox_lib` was declared as a dependency but never called, which is dead weight. Here the print goes through it on purpose:

```lua
lib.print.info(('[qu_frameworks] detected framework: %s'):format(fw))
```

`lib` is the global that `@ox_lib/init.lua` injects into your resource. The `shared_script '@ox_lib/init.lua'` line in the manifest is what loads it, and the `dependencies { 'ox_lib' }` line is what guarantees `ox_lib` starts first so `lib` exists by the time your code runs. `lib.print.info` is `ox_lib`'s leveled logger: it prints with an `[info]` tag and respects a per-resource log-level convar, so you can silence debug noise in production without deleting print lines. The `('...'):format(fw)` part is Lua string formatting, where `%s` is replaced by the value of `fw`. That is the same as writing `'... ' .. fw`, just cleaner when you have several values.

That single call is your first real `ox_lib` usage. The library also gives you the things you will reach for constantly: `lib.notify` for on-screen toasts, `lib.callback` for clean client to server request and response, `lib.registerContext` and `lib.showContext` for menus, `lib.progressBar` for timed actions, and `lib.callback.register` on the server. Every modern framework script you read leans on these instead of reinventing them.

### The translation guide: player, money, job

This is the part the title promised. The framework name you just detected tells you which dialect to speak. Every other line in this section depends on one move first: turning a server id into a player object. Get that one call right and the rest is just reading fields.

#### First, get the player object

You cannot read money or a job until you hold the player object. Each framework hands it to you a little differently, but the shape is identical: feed it the player's server id, get back a table you then read from. Here are the three calls side by side, nothing else.

    ["QBCore", "QBCore.Functions.GetPlayer(src)", "Player (nil if not loaded)"],

  ]}

Read it as one idea in three spellings. ESX grabs the shared object first, then calls `GetPlayerFromId`. QBCore grabs the core object, then calls `Functions.GetPlayer`. Qbox exposes the native `exports.qbx_core:GetPlayer(src)` getter and separately provides a QB compatibility surface for older resources. In every case `src` is the server-side `source`, and the return value can be `nil` until the character has loaded. Always keep a nil guard before reading the object.

Here is each call on its own, with the line that proves you got a real object. These three blocks are illustrative: they only run on a server that actually has that framework installed and a player loaded, so do not expect them to print on the bare practice server from the Build it section.

```lua
-- ESX Legacy (server side)
local ESX = exports['es_extended']:getSharedObject()
local xPlayer = ESX.GetPlayerFromId(src)
print(xPlayer.getName())                  -- proves the object exists
```

```lua
-- QBCore (server side)
local QBCore = exports['qb-core']:GetCoreObject()
local Player = QBCore.Functions.GetPlayer(src)
print(Player.PlayerData.charinfo.firstname) -- the character's first name; proves the object loaded
```

```lua
-- Qbox (server side)
local player = exports.qbx_core:GetPlayer(src)
print(player.PlayerData.charinfo.firstname) -- same field in Qbox; proves the object loaded
```

That getter is the spine of the next lesson, [How a script talks to a framework](/lessons/how-scripts-talk-to-frameworks), which turns this one call into a full framework-agnostic resource that gets the player, gives them money, and reads their job on all three. Here you just need to recognize the call. There you build the whole thing.

Once you hold the object, reading money and job is the same job in each one. Here is the full pattern.

<Callout variant="warn" title="Security: never ship an unvalidated giveCash event">
  Any client can trigger a `RegisterNetEvent` handler. These giveCash snippets are illustrative of the money API only — never ship a net event that hands out money without server-side checks (who is allowed, how much, how often). An unvalidated giveCash event is a classic money-duplication exploit.
</Callout>

**ESX Legacy.** You grab the shared object, then ask it for a player by server id. The player object is usually called `xPlayer`.

```lua
local ESX = exports['es_extended']:getSharedObject()

RegisterNetEvent('myscript:giveCash', function()
    local src = source
    local xPlayer = ESX.GetPlayerFromId(src)
    local cash = xPlayer.getAccount('money').money -- cash on hand (getMoney() also works but ESX docs mark it backwards-compat only)
    xPlayer.addMoney(500)                     -- add 500 cash
    local job = xPlayer.job.name              -- e.g. 'police'
    print(('%s has $%d, job %s'):format(GetPlayerName(src), cash, job))
end)
```

**QBCore.** You grab the core object, get the player, then read a `PlayerData` table. Money lives in `PlayerData.money`, a table keyed by account.

```lua
local QBCore = exports['qb-core']:GetCoreObject()

RegisterNetEvent('myscript:giveCash', function()
    local src = source
    local Player = QBCore.Functions.GetPlayer(src)
    local cash = Player.PlayerData.money.cash   -- cash account
    Player.Functions.AddMoney('cash', 500)
    local job = Player.PlayerData.job.name       -- e.g. 'police'
    print(('%s has $%d, job %s'):format(GetPlayerName(src), cash, job))
end)
```

**Qbox.** Qbox is the QBCore successor, so the shape is nearly identical to QBCore on purpose. The modern entry point is the `exports.qbx_core` getter, and the same `PlayerData` layout applies.

```lua
RegisterNetEvent('myscript:giveCash', function()
    local src = source
    local player = exports.qbx_core:GetPlayer(src)
    local cash = player.PlayerData.money.cash
    exports.qbx_core:AddMoney(src, 'cash', 500, 'qu_frameworks example')
    local job = player.PlayerData.job.name
    print(('%s has $%d, job %s'):format(GetPlayerName(src), cash, job))
end)
```

Read those three side by side and the pattern is clear. Every framework gives you a way to turn a server id into a player object, a way to read and change money, and a `job` field. The verbs differ (`getMoney` vs `PlayerData.money.cash`), but the questions are the same four every time: who, how much money, what job, what inventory. Once you internalize the questions, learning a new framework is just looking up its spelling.

### Load events: why you wait

There is a trap here that bites every beginner. The player object does not exist the instant a player connects. The framework has to pull their character from the database, build the object, and hand it to the client. If your client code asks for the player object too early, it gets `nil`. So each framework fires a load event the moment the player is ready, and you do your setup inside it.

- **ESX Legacy:** the client event `esx:playerLoaded` (it hands you the `xPlayer` data), plus the server-side `esx:playerLoaded`.
- **QBCore:** the client event `QBCore:Client:OnPlayerLoaded`, and server-side `QBCore:Server:OnPlayerLoaded`.
- **Qbox:** keeps QBCore's `QBCore:Client:OnPlayerLoaded` (and server-side `QBCore:Server:OnPlayerLoaded`) for compatibility, and reuses much of the same event surface; its own events use a lowercase `qbx_core:` prefix, and it also exposes ox-style state for player data.

```lua
-- ESX client: only safe to read the player after this fires
RegisterNetEvent('esx:playerLoaded', function(xPlayer)
    print('Player is ready, job is ' .. xPlayer.job.name)
end)
```

The mental model: connecting and being loaded are two different moments, exactly like registering a command and running it are two different moments. You register a handler for the load event up front, and the framework runs it later, once the character is actually in the world. Reading money or job before that event is the number one cause of `attempt to index a nil value` in framework scripts.

<Checkpoint question="A teammate says their ESX script keeps erroring with 'attempt to index a nil value (local xPlayer)' right when a player joins. What is almost certainly wrong, and what is the fix?">
  They are calling `ESX.GetPlayerFromId(src)` (or reading `xPlayer` on the client) before the framework has built the player object, so it returns `nil` and indexing it throws. The fix is to move the code inside the load event. On the client, wrap it in `RegisterNetEvent('esx:playerLoaded', function(xPlayer) ... end)` and use the `xPlayer` the event hands you. On the server, only call `ESX.GetPlayerFromId(src)` after the player is loaded (inside the server `esx:playerLoaded`, or in response to an event the client only triggers after it is loaded). Connecting is not the same moment as being loaded; you must wait for the load event before the player object exists.
</Checkpoint>

### When you reach for this pattern

Framework detection plus the translation idioms is the foundation under almost every script you will write or read on a real server:

- **Writing a framework-agnostic script.** You detect the framework once at startup, then branch your money and job calls so the same resource runs on ESX, QBCore, and Qbox. This is how published scripts support every framework from one download.
- **Reading someone else's resource.** When you open a script and see `exports['qb-core']:GetCoreObject()` you instantly know it is QBCore and you know where money and job live. The translation guide is a reading skill as much as a writing one.
- **Debugging on a live server.** A quick `fwprobe`-style command tells you what framework an unfamiliar server is actually running before you start guessing.

## Try it yourself

<Exercise>
  Turn the probe into a real framework-agnostic money reader. Add a `mymoney` command that detects the framework, gets the player who ran it, reads their cash, and prints it. Branch on the detected framework so the same command works on ESX and QBCore or Qbox:

  ```lua
  RegisterCommand('mymoney', function(src)
      if src == 0 then
          lib.print.warn('[qu_frameworks] run this in-game, not from the console')
          return
      end
      local fw = detectFramework()
      local cash
      if fw == 'esx' then
          local ESX = exports['es_extended']:getSharedObject()
          cash = ESX.GetPlayerFromId(src).getMoney()
      elseif fw == 'qbcore' then
          cash = exports['qb-core']:GetCoreObject().Functions.GetPlayer(src).PlayerData.money.cash
      elseif fw == 'qbox' then
          cash = exports.qbx_core:GetPlayer(src).PlayerData.money.cash
      else
          lib.print.warn('[qu_frameworks] no framework detected')
          return
      end
      lib.print.info(('[qu_frameworks] %s has $%d'):format(GetPlayerName(src), cash))
  end, false)
  ```

  Restart `qu_frameworks`, join the server, and run `/mymoney` from the chat box (press T) (this one is unrestricted because a real player triggers it). Confirm it prints your name and your cash on hand. You just wrote one command that speaks all three dialects, which is exactly how published scripts stay framework-agnostic.
</Exercise>

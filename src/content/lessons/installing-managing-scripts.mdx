<Callout variant="fivem" title="The whole install in one line">
  A script is a folder. Put the folder in `resources`, add `ensure folder-name` to `server.cfg`, restart. `ensure` is the FXServer command that starts a resource (or restarts it if it is already running). Ninety percent of "it doesn't work" is one of four things: a wrong folder name, a missing dependency, the wrong load order, or an escrowed asset whose key is not linked to your server.
</Callout>

## Build it

<StepGroup>
  <Step num={1} title="Unzip and check the folder" outcome="You have a clean resource folder that contains an fxmanifest.lua.">
    Unzip the download. Open the folder and confirm it contains an **`fxmanifest.lua`**. The `fxmanifest.lua` is the manifest file that names the resource and lists what it needs. Older scripts call it `__resource.lua`. That file is what makes the folder a resource.

    Keep the official folder name from the README or the release archive. A GitHub source ZIP often adds `-main` to the name. Use the author's release asset when one exists. Rename a folder only when the official install instructions tell you to. Manifests, dependencies, escrow checks, and other resources all reference the exact name, so a changed name breaks them.

    ```text
    server-data/resources/qs-vehiclekeys/
      fxmanifest.lua
      ...
    ```
  </Step>

  <Step num={2} title="Install dependencies first" outcome="Everything the script needs is already running before it loads.">
    Open the script's README or its `fxmanifest.lua` and read its **dependencies**. A dependency is another resource this script needs in order to run. Most modern scripts need `oxmysql`, `ox_lib`, and a framework. `oxmysql` and `ox_lib` are Overextended community resources, not official Cfx.re built-ins, so you install them yourself like any other script.

    **Install and start every dependency before the script that needs them.** If a dependency loads after the script, the script errors on boot.

    <TechLinks ids={["oxmysql", "ox_lib", "esx", "qbox"]} />
  </Step>

  <Step num={3} title="Drop it into resources" outcome="The resource sits where the server can find it.">
    Move the folder into your server's `resources` directory. If your server runs on a VPS, upload it with WinSCP or your panel's file manager. A VPS is a rented remote server you connect to over the internet, the alternative to running the server on your own PC.

    ```text
    server-data/resources/qs-vehiclekeys
    ```

    <TechLink id="winscp" />
  </Step>

  <Step num={4} title="Ensure it in the right order" outcome="server.cfg starts dependencies, then the framework, then the script.">
    Load order runs top to bottom. Dependencies and the framework come first. Your script comes after. Add this to `server.cfg`:

    ```text
    ensure oxmysql
    ensure ox_lib
    ensure es_extended      # or your framework
    ensure qs-vehiclekeys
    ```

    If the script needs a database table, import its `.sql` file into your MariaDB database with HeidiSQL or DBeaver before you start the script.
  </Step>

  <Step num={5} title="Refresh, start, and read the console" outcome="The resource starts with no red errors and you see the started line.">
    You do not need a full restart to add one resource. In the txAdmin Live Console, run:

    ```text
    refresh
    ensure qs-vehiclekeys
    ```

    `refresh` re-scans the resources folder so the server notices the new one. `ensure` starts it. Then read the console.

    <ExpectedOutput>
      Started resource qs-vehiclekeys
    </ExpectedOutput>
  </Step>
</StepGroup>

## Organize with categories

Once you run more than a handful of scripts, group them with bracketed category folders. Then you can load a whole group at once:

```text
resources/
  [framework]
  [jobs]
  [vehicles]
  [custom]
```

One line loads everything inside a category:

```text
ensure [vehicles]
```

The brackets are a FiveM convention. A folder named `[something]` is a category, and `ensure [something]` starts every resource inside it. **Keep each category after its dependencies in `server.cfg`.**

## A worked example: getting the order right

Order is the one thing beginners get wrong most often. So here is one real install traced end to end. You are installing `ox_inventory`, the Overextended inventory. Read its README and you find a dependency chain. It needs `oxmysql` for the database, `ox_lib` for shared functions, and a framework (`es_extended` or `qb-core`) to know who the player is. Each of those is itself a resource you `ensure`. Each must already be running before the thing that calls it loads.

The rule is one sentence. **A resource must be `ensure`d after everything it depends on.** Draw the chain and the load order falls out of it. The arrows point from a resource to what it needs:

```text
ox_inventory                 # the script you actually want
   |
   +-- needs --> es_extended    (the framework: identifies the player)
   |                 |
   |                 +-- needs --> oxmysql   (database layer)
   |                 +-- needs --> ox_lib    (shared library)
   |
   +-- needs --> oxmysql        (writes inventory rows to the DB)
   +-- needs --> ox_lib         (UI + helper functions)
```

Everything at the bottom of the arrows has to start first. Flatten the tree from the leaves upward and you get the exact `ensure` order for `server.cfg`:

```cfg
# server.cfg - load order is top to bottom.
# Leaves of the dependency tree first, the script that needs them last.
ensure oxmysql        # 1. database layer, depends on nothing else here
ensure ox_lib         # 2. shared library, depends on nothing else here
ensure es_extended    # 3. framework, needs oxmysql + ox_lib (both already up)
ensure ox_inventory   # 4. the script, needs all three above (all already up)
```

Flip two lines and the console tells you at once. Put `ensure ox_inventory` above `ensure oxmysql` and on boot you get a red line like `attempt to index a nil value`. The script reached for the database exports before `oxmysql` had registered them. An export is a function one resource publishes for other resources to call. If `ox_inventory` declares `oxmysql` as a manifest dependency, you instead see a line that names the missing one, like `Could not find dependency oxmysql for resource ox_inventory` followed by `Couldn't start resource ox_inventory`.

<Checkpoint question="ox_inventory failed on boot with attempt to index a nil value, but the folder name and spelling are correct. What is the most likely cause?">
  A dependency loads too late, or not at all. `ox_inventory` calls into `oxmysql` and `ox_lib` the instant it starts. So if either of them is `ensure`d below `ox_inventory` in `server.cfg`, or is missing entirely, those exports do not exist yet and indexing them returns nil. Fix the order. `ensure oxmysql` and `ensure ox_lib` must sit above `ensure ox_inventory`. The same logic applies to the framework. Reorder the lines, then `restart ox_inventory`.
</Checkpoint>

The manifest can declare this for you, so you do not have to remember it. A script that lists its needs uses the `dependencies` key. Note the plural and the table, not a single string:

```lua
-- fxmanifest.lua inside ox_inventory
dependencies {
    'oxmysql',
    'ox_lib'
}
```

The `dependencies` key tells the server those resources must load before this one. The docs describe it as "Requires the specified resource to load before the current resource." If a listed dependency is missing or fails to start, the console usually names it, like `Could not find dependency oxmysql for resource ox_inventory`, instead of leaving you with a silent nil. It still does not reorder `server.cfg` for you. The top-to-bottom rule above is what actually controls startup.

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_dev_loop
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_dev_loop/
  fxmanifest.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'
    ```

    Older tutorials add a `lua54 'yes'` line here. As of June 2025 that setting is deprecated and ignored: Lua 5.4 is now the only Lua runtime, so you leave it out.
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `server.lua` and paste this:

    ```lua
print('[qu_dev_loop] resource started')

RegisterCommand('dev_loop', function()
    print('[qu_dev_loop] command fired')
end, false)
    ```

    There are two prints on purpose. The first one is at the top level of the file, so it runs the moment the resource starts. That is what proves your restart actually reloaded the code. The second one lives inside a command handler, so it only runs when you type the command.
  </Step>

  <Step num={5} title="Start and test it" outcome="The expected proof appears in the server console.">
    Open `server.cfg` and add this line:

    ```text
    ensure qu_dev_loop
    ```

    Save the file. Because this folder is brand new, the running server has not scanned it yet, so first tell it to look, in the txAdmin Live Console (the server console):

    ```text
    refresh
    ```

    Then start the resource, also from the server console:

    ```text
    ensure qu_dev_loop
    ```

    Use `ensure`, not `restart`, the first time: the resource has not started yet this session, and `restart` only reloads a resource that is already running, so it would do nothing here. (Once the resource is already up, `restart` is the command you use after each edit.) You should see the start line print immediately. Now run the command, also from the server console:

    ```text
    dev_loop
    ```

    <ExpectedOutput>
      [qu_dev_loop] resource started
      [qu_dev_loop] command fired
    </ExpectedOutput>
  </Step>
</StepGroup>

## How it works

You restarted the resource and saw the start line. You typed the command and saw the second line. That is the entire dev loop in two prints. Now take it apart so you stop guessing which server command to run and start choosing on purpose.

### The two prints, and why one fires on restart

```lua
print('[qu_dev_loop] resource started')

RegisterCommand('dev_loop', function()
    print('[qu_dev_loop] command fired')
end, false)
```

When you run `restart qu_dev_loop`, FiveM reloads the resource and reads `server.lua` from the top, once. The first `print` is at the top level of the file, so it runs right then. That single line is your loop confirmation: if you see it, the new code is live. If you change the file, save, restart, and the start line does not change, the server is still running the old code and your edit never reached it.

The second `print` is inside the `RegisterCommand` handler, so reading the file does not run it. FiveM only stores the handler and waits. When you type `dev_loop`, the handler runs and the second line prints. Registering and running are two separate moments. That gap is why you need both a restart step (reload the code) and a test step (fire the code) in every loop.

### The four commands, and when each one is used

A FiveM server controls resources with a small set of console commands. The whole dev loop is choosing the right one. Here is what each does and when you reach for it.

<Cheatsheet
  title="Resource control commands"
  rows={[
    { cmd: "ensure <res>", desc: "Start the resource if it is stopped, or restart it if it is already running. The safe default. This is the line you put in server.cfg." },
    { cmd: "start <res>", desc: "Start the resource if it is stopped. It does not reload a resource that is already running." },
    { cmd: "restart <res>", desc: "Stop then start a resource that is already running. This is what you type after each edit to reload your changed code." },
    { cmd: "stop <res>", desc: "Stop a running resource. Used to take a script offline without removing it from server.cfg." },
    { cmd: "refresh", desc: "Rescan the resources folder for new or changed manifests. Run it once after you create a brand new resource folder, so the server knows the resource exists." },
  ]}
/>

The trap beginners fall into is creating a new folder and immediately typing `start qu_dev_loop`, getting `Resource qu_dev_loop does not exist`, and assuming their code is broken. It is not. The server has not scanned the folder yet. The fix is `refresh` first, then `ensure`. Once the resource is known and listed in `server.cfg`, your day-to-day loop never touches `refresh` again. You just edit, save, and `restart`.

### Why ensure is the one you put in server.cfg

`ensure` is the useful idempotent default: if the resource is stopped it starts it, and if it is already running it restarts it. `start` only starts a stopped resource and does not reload one that is already running. Both commands are valid in `server.cfg`; modern templates usually use `ensure` because the same line also works during manual recovery when the resource's state is uncertain.

`restart` versus `refresh` is the other distinction worth locking in. `restart qu_dev_loop` reloads the code of one resource that already exists. `refresh` does not restart anything. It only rescans the folder so the server learns about new resources and manifest changes. New folder: `refresh`, then `ensure`. Changed code in an existing resource: just `restart`.

### The restricted flag on RegisterCommand

```lua
RegisterCommand('dev_loop', function()
    print('[qu_dev_loop] command fired')
end, false)
```

The third argument is the restricted flag, and it is the reason your command either works or silently does nothing.

- With `false`, the command is open. Anyone can run it, including you from the server console (type the bare name `dev_loop`) and any connected player from their chat box (press T, then type it with a leading slash: `/dev_loop`). For a teaching command you want to fire freely while testing, `false` is correct. That is what this lesson uses.
- With `true`, the command is restricted. It requires an ace permission, which is a rule in the server config that grants a specific identifier the right to run it. The server console always has every ace, so a restricted command still works from the console. But a beginner who sets `true`, then tries to run the command from anywhere without an ace, sees nothing print and assumes the code is broken.

The old version of this lesson used `true` and never explained aces, so the proof line could silently fail to appear. You use `false` here on purpose: the command runs cleanly whether you type it in the server console or test it later in chat, and there is no hidden permission gate to debug. You will meet aces and restricted commands properly when you build admin actions, where keeping players out is the entire point.

<Checkpoint question="You created a brand new resource folder, added `ensure my_thing` to server.cfg, and typed `start my_thing` in the console. It says the resource does not exist. What is wrong, and what two commands fix it?">
  The running server has not scanned the new folder, so it does not know the resource exists yet. Run `refresh` to load new resource manifests, then `ensure my_thing` to start it. `start my_thing` would also start it after the refresh; `ensure` is preferred here because it works whether the resource is stopped or already running.
</Checkpoint>

### Reading the server console

The last step of the loop is read the console, and that only helps if you can tell a good line from a bad one. After `ensure qu_dev_loop` or `restart qu_dev_loop`, the server prints a status line. A clean start looks like this:

```text
Started resource qu_dev_loop
[qu_dev_loop] resource started
```

The first line is the server telling you it brought the resource up. The second is your own top-level `print` confirming your code ran. Seeing both is the signal that the loop closed: new code is live.

A failed start looks different. The server prints an error instead of `Started resource`, and your own start line never appears:

```text
Error parsing script @qu_dev_loop/server.lua in resource qu_dev_loop: server.lua:4: '<eof>' expected near 'end'
Failed to load script server.lua.
```

When you see that, do not retype commands hoping it fixes itself. Read the line. It names the file, the line number, and what the parser expected. Open that file, go to that line, fix the syntax, save, and `restart` again. The console is not noise to scroll past. It is the feedback half of your loop, and learning to read it is the whole skill.

This is also why `qu_dev_loop` is a server-only resource. There is no `client_script`, so nothing runs on a player's game and there is no F8 client output to check. Every line for this lesson lands in the server console. When a resource does have client code, the split matters: server `print` goes to the server console, client `print` goes to F8 in the game.

### When you reach for this loop

This is not a beginner ritual you outgrow. It is the loop you run thousands of times.

- **After every code change.** Edit, save, `restart qu_dev_loop`, read the console, test. The tighter you make this loop, the faster you build.
- **When a resource misbehaves on a live server.** `restart` it to reload it clean, watch the console for the start line and any errors, then reproduce the problem.
- **When you add a new resource to a running server.** `refresh` once so the server sees it, then `ensure` it, then add the `ensure` line to server.cfg so it survives the next reboot.

## Try it yourself

<Exercise>
  Turn the proof into a restart counter that survives restarts, so the start line itself shows your loop working. FiveM gives every resource a tiny key-value store on disk. Read a number, add one, save it back, and print it on start:

  ```lua
  local count = GetResourceKvpInt('restart_count') + 1
  SetResourceKvpInt('restart_count', count)
  print('[qu_dev_loop] restart number ' .. count)

  RegisterCommand('dev_loop', function()
      print('[qu_dev_loop] command fired')
  end, false)
  ```

  Save, then run `restart qu_dev_loop` three times in the server console. Confirm the printed number climbs to 1, then 2, then 3. Each climb is undeniable proof the new code reloaded, because the old code could not have known the new count. The `..` operator joins the text and the number into one string. You just built a loop you can trust at a glance instead of squinting at whether anything changed.
</Exercise>

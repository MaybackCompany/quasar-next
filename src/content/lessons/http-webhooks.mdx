## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Using Windows File Explorer (or `mkdir` in a Linux shell, or the VS Code Explorer), go to your server-data folder's `resources` directory — the same folder your other resources live in — and create a new folder. The full path is your server-data path plus:

    ```text
    resources/qu_http_webhooks
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_http_webhooks/
  fxmanifest.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'
    ```

    Older tutorials add a `-- Lua 5.4 is the default runtime in 2026; the lua54 directive is no longer needed` line here. As of June 2025 that setting is deprecated and ignored: Lua 5.4 is now the only Lua runtime, so you leave it out. Notice there is no `client_script` line. Everything in this lesson runs on the server only, which is exactly what keeps the webhook URL off players' machines.
  </Step>

  <Step num={4} title="Set the webhook in server.cfg" outcome="The secret URL lives in config, not in code.">
    The webhook URL is a secret. Anyone who has it can post to your Discord channel forever. Never paste the real URL into chat, a screenshot, a public repo, or a screen-share — and if it leaks, delete that webhook in Discord and create a new one (this rotates the token). It does not go in the script. It goes in `server.cfg` as a convar, above the `ensure qu_http_webhooks` line you will add in Step 6:

    ```text
set qu_http_webhooks_webhook "https://discord.com/api/webhooks/123456789/your-token-here"
    ```

    Create the webhook in Discord under Server Settings, then Integrations, then Webhooks. Copy the URL it gives you and paste it in place of the example above.
  </Step>

  <Step num={5} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `server.lua` and paste this. It reads the URL from the convar, builds a JSON body, POSTs it, and retries once if Discord rate-limits you:

    ```lua
local webhook = GetConvar('qu_http_webhooks_webhook', '')

local function postToDiscord(message, attempt)
    attempt = attempt or 1
    local body = json.encode({ username = 'qu_http_webhooks', content = message })
    PerformHttpRequest(webhook, function(code, response, headers)
        if code == 429 and attempt == 1 then
            local data = response and json.decode(response) or nil
            local retryAfter = tonumber(data and data.retry_after)
                or tonumber(headers['retry-after'] or headers['Retry-After'])
                or 1
            print('[qu_http_webhooks] rate-limited, retrying in ' .. retryAfter .. 's')
            SetTimeout(retryAfter * 1000, function()
                postToDiscord(message, 2)
            end)
            return
        end
        print('[qu_http_webhooks] Discord status ' .. code)
    end, 'POST', body, { ['Content-Type'] = 'application/json' })
end

RegisterCommand('webhooktest', function(src)
    if webhook == '' then
        print('[qu_http_webhooks] missing webhook, set qu_http_webhooks_webhook in server.cfg')
        return
    end
    local name = src == 0 and 'console' or GetPlayerName(src)
    postToDiscord('test from ' .. name, 1)
end, true)
    ```

    That final `true` marks the command **restricted**: it needs the `command.webhooktest` ACE permission. The server/txAdmin console always has it, which is why you run the test there. A normal in-game player would be silently denied unless you grant them with an `add_ace` line in `server.cfg`.
  </Step>

  <Step num={6} title="Start and test it" outcome="The expected proof appears in the server console.">
    Open `server.cfg` and add this line below the `set` line from Step 4:

    ```text
    ensure qu_http_webhooks
    ```

    Save, then run:

    ```text
    restart qu_http_webhooks
    ```

    Open the txAdmin web panel (by default `http://localhost:40120`, or your panel URL), click **Live Console** in the left sidebar, then type this command into the input box at the bottom and press Enter:

    ```text
    webhooktest
    ```

    A message appears in your Discord channel, and the console prints:

      [qu_http_webhooks] Discord status 204
    </ExpectedOutput>
  </Step>

## How it works

You posted to Discord and read back a status code. Now take the code apart so the next time you call an API you are deciding, not copying. The whole lesson lives inside one native, `PerformHttpRequest`, plus the JSON you feed it and the convar that keeps the URL safe.

### The five arguments of PerformHttpRequest

Look at the call, ignoring the retry logic for a moment:

```lua
PerformHttpRequest(url, callback, method, body, headers)
```

That is the entire native. Five arguments, in this exact order:

- **`url`** is the address you are talking to. Here it is the Discord webhook string read from the convar.
- **`callback`** is a function FiveM runs **later**, when the response comes back. This is the part that surprises everyone. More on it in the next subsection.
- **`method`** is the HTTP verb as a string: `'POST'` to send data, `'GET'` to fetch it, `'PUT'` and `'DELETE'` for the rest. Discord webhooks want `'POST'`.
- **`body`** is the data you send, as a single string. For a JSON API that string has to be JSON, which is why you run `json.encode` first.
- **`headers`** is a Lua table of extra information about the request. Here it carries `Content-Type` set to `application/json`, which tells Discord to read the body as JSON instead of plain text.

Get the order wrong and the request fails silently, because nothing in Lua forces you to pass the right type in each slot. Memorize the order: url, callback, method, body, headers.

### Why it is callback-based, and why you cannot store the result

Here is the line beginners write and cannot understand why it breaks:

```lua
local status = PerformHttpRequest(url, ...) -- this does NOT work
```

`PerformHttpRequest` does not return the status code. It returns nothing useful. The reason is that an HTTP request travels across the internet to Discord and back, which takes time, maybe 50 milliseconds, maybe two seconds. FiveM will not freeze your whole server waiting for that. Instead it fires the request, moves on immediately, and runs your callback **whenever** the response finally arrives.

So the answer does not come back on the same line. It comes back inside the function you handed as the second argument:

```lua
function(code, resultData, headers, errorData)
    -- code     = the HTTP status number, e.g. 204
    -- resultData = the response body as a string, often empty for webhooks
    -- headers  = a table of response headers from Discord
    -- errorData = transport-level error text (usually empty); useful when the request never reached Discord
end
```

This is the same register-now, run-later idea behind command handlers and events. You hand FiveM a block of code and say, hold this, run it when the response lands. That is why the status is printed *inside* the callback, not after the call. If you tried to print it after the `PerformHttpRequest` line, it would not exist yet.

<Checkpoint question="Why can't you write local status = PerformHttpRequest(...) and then print status on the next line?">
  Because `PerformHttpRequest` is asynchronous. It sends the request and returns straight away, before Discord has answered, so there is no status to assign yet. The status arrives later, in the callback function you pass as the second argument. The response body and status code only exist inside that callback, which is why every line that needs the result has to live there. This is the same register-now, run-later pattern you saw with `RegisterCommand`: you describe what to do with the answer, and FiveM runs it when the answer is ready.
</Checkpoint>

### json.encode: turning a Lua table into the body Discord expects

Discord does not understand Lua tables. It speaks JSON, a text format. So before you can send your message you translate:

```lua
local body = json.encode({ username = 'qu_http_webhooks', content = message })
```

`json.encode` is a function built into the FiveM Lua runtime. You hand it a Lua table and it returns a JSON string. The table above becomes the text `{"username":"qu_http_webhooks","content":"test from console"}`. That string is what goes in the `body` slot of `PerformHttpRequest`.

Two pieces have to agree for Discord to accept it. The body must be valid JSON, which `json.encode` guarantees. And the `Content-Type` header must say `application/json`, which is the label that tells Discord to parse the body as JSON rather than treat it as raw text. Send JSON without that header and Discord will reject it. The encode and the header are a pair; you always set both.

When a response comes back as JSON, the mirror function `json.decode` turns the string back into a Lua table you can read. Webhooks return an empty body on success so you do not need it here, but the moment you call an API that returns data, like a license-check service, `json.decode(resultData)` is how you read the answer.

### Why reading the URL from a convar hides it from clients

This is the security point of the lesson, and it rests on one fact about how FiveM ships files:

```lua
local webhook = GetConvar('qu_http_webhooks_webhook', '')
```

A file listed under `server_script` is **never sent to a player's game**. The server reads it and runs it; the client never receives a copy. A file under `client_script`, by contrast, is downloaded to every player who joins. So a secret is safe in a server script and exposed in a client script.

The webhook URL is the secret. If you hardcoded it into a client file, or pulled it down to the client over an event, any player could open their FiveM cache, read the URL, and spam your Discord channel. By keeping the URL in `server.cfg` and reading it with `GetConvar` inside a server script, the URL exists only on the machine you control. The second argument to `GetConvar`, the empty string `''`, is the fallback returned when the convar is not set, which is what the missing-webhook guard in the command checks for.

The rule generalizes: any secret, a database password, an API key, a webhook, lives server-side and is read with `GetConvar`. It never touches a client file.

### The retry on 429

Discord limits how fast you can post. Hit it too hard and it answers with HTTP status `429 Too Many Requests` instead of accepting your message. A naive script drops the message on the floor. A correct one waits and tries again:

```lua
if code == 429 and attempt == 1 then
    local data = response and json.decode(response) or nil
    local retryAfter = tonumber(data and data.retry_after)
        or tonumber(headers['retry-after'] or headers['Retry-After'])
        or 1
    print('[qu_http_webhooks] rate-limited, retrying in ' .. retryAfter .. 's')
    SetTimeout(retryAfter * 1000, function()
        postToDiscord(message, 2)
    end)
    return
end
```

Discord's 429 response includes a JSON `retry_after` value in seconds (it can be a decimal, e.g. `1.5`), and can also send a `Retry-After` header. On FiveM the JSON body is the reliable source — response headers are sometimes not delivered to the callback on non-200 responses — so the code reads the JSON value first, falls back to the header if present, and finally uses one second if neither parses. `SetTimeout` accepts milliseconds, so the code multiplies by 1000. The attempt counter allows one retry and then stops.

This is the smallest honest version of retry-on-rate-limit. A busy logger needs a bounded queue and backoff; it must not create one unbounded timer per failed message.

### What the status codes mean

The number printed by the callback tells you what happened:

- **`204 No Content`** is success. Discord accepted the post and has nothing to send back, hence an empty body. This is the expected output for this lesson.
- **`401 Unauthorized`** means the webhook token is wrong.
- **`404 Not Found`** means the webhook was deleted, or the URL is mistyped.
- **`429 Too Many Requests`** is the rate limit you just handled.

Printing the status is not optional polish. It is the only way you find out a webhook quietly stopped working before your ban-logs vanish for a week.

## The two folders that are not the same

Before you touch anything, lock in the single most important split in a FiveM server. There are two top-level folders and they do completely different jobs.

```text
C:\FXServer\
├── server/ # the ARTIFACTS: FXServer.exe / run.sh, the engine. Update freely.
└── server-data/ # YOUR content: resources/ + server.cfg. This is what you back up.
```

The `server/` folder holds the **artifacts**: the FiveM server binaries you downloaded from the artifacts page. Think of it as the engine. You replace its contents every time you update the server, and you never edit anything inside it by hand.

The `server-data/` folder holds **your** content: the `resources/` folder full of scripts, and the `server.cfg` that decides what runs. This is the folder you version-control and back up. They are kept separate on purpose, so updating the engine never risks your scripts.

For the full launch and boot instructions, see [A complete server.cfg, annotated](/lessons/server-cfg-annotated).

## Build it

<StepGroup>
 <Step num={1} title="Open server-data and find resources" outcome="You are standing in the folder that holds your content.">
 Open your `server-data/` folder in VS Code. You should see exactly two things at the root: the `resources/` folder and your `server.cfg` file.

 ```text
server-data/
├── resources/
└── server.cfg
 ```

 The `server.cfg` lives at the root of `server-data/`, right next to `resources/`, not inside it. Everything you build next happens inside `resources/`.
 </Step>

 <Step num={2} title="Create the four category folders" outcome="resources/ now has four bracketed groups.">
 Inside `resources/`, create four folders whose names are wrapped in square brackets:

 ```text
resources/
├── [framework]/ # the framework + its hard dependencies
├── [standalone]/ # shared deps that any script can use
├── [scripts]/ # your gameplay scripts
└── [maps]/ # MLOs, map streams, YMAPs
 ```

 A folder whose name is wrapped in `[brackets]` follows a community convention: FiveM sees the brackets and treats the folder as a **category group** instead of a resource. The brackets are part of the real folder name on disk - the folder is literally called `[framework]`, square brackets and all. You can still put resources directly in `resources/` without brackets; the brackets are for organizing, not a requirement.
 </Step>

 <Step num={3} title="Drop resources into the right group" outcome="Each resource sits in the category that matches its job.">
 Now place a few real resources so the layout has shape. Note the load-sensitive trio inside `[framework]`: the database layer, the shared library, then the framework itself.

 ```text
resources/
├── [framework]/
│ ├── oxmysql/ # database layer (CommunityOx)
│ ├── ox_lib/ # shared library (CommunityOx)
│ └── es_extended/ # one example framework (or qb-core / qbx_core)
├── [standalone]/
│ ├── ox_target/ # 3D interaction (CommunityOx)
│ └── ox_inventory/ # inventory (CommunityOx)
├── [scripts]/
│ └── qu_first_line_of_lua/
└── [maps]/
 └── my_custom_mlo/
 ```

 `oxmysql`, `ox_lib`, `ox_target`, and `ox_inventory` are community resources (originally Overextended, now maintained by **CommunityOx** at [github.com/CommunityOx](https://github.com/CommunityOx)), not part of QBCore or ESX. They are standalone building blocks you add yourself. Putting the database and library inside `[framework]` keeps everything the framework depends on in one place.
 </Step>

 <Step num={4} title="Wire it up in server.cfg" outcome="The cfg starts each resource by name, in dependency order.">
 Open `server.cfg` and add your resource section. Order is everything here, so read the comments as you type:

 ```cfg
## --- Built-in resources first ---
ensure mapmanager
ensure chat
ensure spawnmanager
ensure sessionmanager

## --- Framework group: dependencies BEFORE the framework ---
ensure oxmysql # database layer, must load first
ensure ox_lib # shared library for resources whose manifests require it
ensure es_extended # framework itself; follow its official recipe dependencies

## --- Standalone deps, then your gameplay scripts ---
ensure ox_target
ensure ox_inventory
ensure qu_first_line_of_lua

## --- Maps last ---
ensure my_custom_mlo
 ```

 `ensure` starts a resource if it is stopped and restarts it if it is already running. It is the safe, idempotent directive. Prefer it over `start`, which only acts on a stopped resource and does nothing if it is already running, so it never picks up your latest changes; `ensure` restarts it.
 </Step>

 <Step num={5} title="Start your scripts by name" outcome="The cfg starts each resource, and the console confirms it.">
 Here is where the brackets earn their keep, and where a common myth gets cleared up. The brackets organize your folders for discovery, but `ensure` always acts on **one resource at a time**, by its own name. There is no resource called `[scripts]`, so you start each script by name:

 ```cfg
## Start each resource inside [scripts] by its own name:
ensure qu_first_line_of_lua
 ```

 Save `server.cfg`, then from the txAdmin Live Console run:

 ```text
 refresh
 ensure qu_first_line_of_lua
 ```

 <ExpectedOutput>
 Started resource qu_first_line_of_lua
 </ExpectedOutput>

 `refresh` rescans the `resources/` folder so the server notices any folders you added, including ones nested inside `[scripts]`. Then `ensure qu_first_line_of_lua` starts that one resource. Add a new script to `[scripts]` later, and you `refresh` then `ensure` it by its own name too.
 </Step>
</StepGroup>

## How it works

The server booted clean and every resource started by name in the right order. Now take the layout apart so the next time you add a resource you know exactly where it goes and why the order matters.

### Artifacts vs data: why the split exists

The reason `server/` and `server-data/` are separate folders comes down to one word: **updates**. The artifacts in `server/` change constantly. The FiveM team ships new server builds often, and updating is supposed to be as simple as deleting the contents of `server/` and dropping in the new build. If your scripts and your config lived in that same folder, every update would put your work at risk of being wiped.

By keeping your content in `server-data/`, the two lifecycles never touch. You update the engine without fear, because your data is somewhere else entirely. You back up your data without bloat, because the binaries are somewhere else entirely. This is also why `server-data/` is the folder you put under version control with Git: it is the part that is actually yours.

### What a [bracket] folder really is

A folder named `[scripts]` is not a resource. FiveM never tries to start it as a script, because it has no `fxmanifest.lua` of its own. Instead, when the server scans `resources/`, it sees the brackets and treats that folder as a **container**: a labelled box whose only job is to hold other resources so the server can find them during discovery.

Here is the part people get wrong, so be clear about it. The brackets help the server **discover** resources; they do not let you **start** a whole folder. There is no resource called `[scripts]`, and `ensure` acts on exactly one resource by its own name. `ensure [scripts]` does not start everything inside the box. You still `ensure qu_first_line_of_lua`, and any other resource in there, by name. What the bracket buys you is discovery and tidiness: every resource inside `[scripts]` is found in the scan and sits in one obvious place, so you always know what to `ensure` and where it lives.

The names are a community convention, not a fixed list. `[framework]`, `[standalone]`, `[scripts]`, and `[maps]` are the common ones, but the server does not care what the brackets say. You could make `[jobs]` or `[economy]` tomorrow. What matters is only that the name is wrapped in brackets, which marks the folder as a "category group" the scan walks into rather than a resource it tries to start.

<Checkpoint question="You drop a brand-new script folder inside [scripts] while the server is running and then type ensure your_new_script, but it does not start. Why?">
 Because the server has not noticed the new folder yet. The resource list is built when the server scans `resources/`, and a folder added while the server is live is not in that scan. Run `refresh` first: it rescans `resources/` and loads the new manifest. Then `ensure your_new_script` (the resource's own name) finds it and starts it. The fix is always the same pair when adding resources live: `refresh`, then `ensure <resource_name>`.
</Checkpoint>

### Why folder order becomes load order

This is the part that bites everyone once. Resources do not load in alphabetical order, and they do not load all at the same instant. They load in the **order their ensure lines appear in server.cfg**, top to bottom. Whichever line is higher in the file loads first.

That ordering is not cosmetic. Many resources depend on others being ready first. A framework like `es_extended` calls into the database layer the moment it starts, so `oxmysql` has to already be running. The same framework reads from the shared library `ox_lib`, so that has to be running too. Get the order right and everything finds what it needs:

```cfg
# CORRECT: dependency first, dependent second
ensure oxmysql # registers the database exports
ensure es_extended # calls those exports on start -> they exist -> OK
```

Get it backwards and the dependent resource reaches for exports that do not exist yet, and it typically errors out during boot (it can also fail silently if it touches the export later instead of at start):

```cfg
# WRONG: dependent before dependency -> startup error
ensure es_extended # tries to use oxmysql exports that are not registered yet -> error
ensure oxmysql
```

The reliable rule is **dependency before dependent**, using each resource's README and `fxmanifest.lua` as the source of truth. Database-backed frameworks commonly need `oxmysql`; a script that imports `@ox_lib/init.lua` needs `ox_lib`. Do not assume every framework needs every ox resource because an example config includes them.

<Checkpoint question="Everything in [framework] needs to load before everything in [scripts]. Can you just write ensure [framework] and ensure [scripts] and trust the order?">
 No, and the reason matters. You cannot `ensure` a bracket folder at all: there is no resource named `[framework]` or `[scripts]`, so those lines start nothing. `ensure` acts on one resource by its own name. You control load order by the order you write the individual `ensure` lines in `server.cfg`, top to bottom: list `oxmysql`, then `ox_lib`, then `es_extended`, then your scripts, in exactly that sequence. Brackets group folders for discovery so you know what lives where; they do not start resources and they do not sort load order. The `ensure` lines do.
</Checkpoint>

### When you reach for this layout

This structure is not academic tidiness. It saves you real time the moment a server has more than a handful of resources:

- **Adding a script.** You downloaded a new job script. It drops into `[scripts]`, you `refresh` and `ensure` it by its own name, and you are done. No hunting for where it belongs: the bracket folder told you exactly where it goes.
- **Updating a dependency.** A new `ox_lib` release comes out. You know exactly where the old one lives, inside `[framework]`, because every shared dependency lives there. Swap the folder, restart, done.
- **Handing the server to someone else.** A teammate opens `server-data/` and instantly reads the shape of the server from the folder names alone. The structure is the documentation.

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| Resources in the wrong folder - "Could not find resource" | The folder is outside `resources/` or misspelled | Every resource folder must be directly inside `resources/` or `resources/[category]/`. Check spelling exactly |
| `[category]` folders don't affect load order | Folder names like `[gameplay]` or `[esx]` are only for organization | Load order is controlled by `ensure` lines in server.cfg, not by folder names |
| Two resources with the same name in different folders | FiveM only loads the first one it finds | Never have duplicate resource names. Rename one or delete the extra copy |
| Deleted a resource but it still shows in console | txAdmin cached the folder list | Restart txAdmin or the full server (not the resource) |

## Try it yourself

<Exercise>
 Add a new category group of your own and prove the layout makes every resource easy to find and start. Create a folder named `[jobs]` inside `resources/`, then put two empty resource folders inside it, each with a minimal manifest:

 ```text
resources/
└── [jobs]/
 ├── qu_taxi_job/
 │ └── fxmanifest.lua
 └── qu_courier_job/
 └── fxmanifest.lua
 ```

 Give each `fxmanifest.lua` the two required lines so the server recognizes it as a real resource:

 ```lua
fx_version 'cerulean'
game 'gta5'
 ```

 Then add one `ensure` line per resource to `server.cfg`. Remember: there is no resource called `[jobs]`, so you `ensure` each one by its own name:

 ```cfg
ensure qu_taxi_job
ensure qu_courier_job
 ```

 From the console run `refresh`, then `ensure qu_taxi_job` and `ensure qu_courier_job`, and confirm the console reports both starting. You invented your own category group: the `[jobs]` brackets kept both resources in one obvious place for the scan to discover, and you started each one by name. The brackets organize and help discovery; the `ensure` lines, one per resource, are what actually start them.
</Exercise>

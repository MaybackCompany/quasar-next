<Callout variant="warning" title="Do not create empty .ymap or .ytyp placeholders">
  GTA map files contain structured game data. Renaming an empty text file to <code>.ymap</code> or <code>.ytyp</code> does not create a valid asset and proves nothing about streaming. Use a real test package from its author, keep its original manifest, and test away from production players.
</Callout>

## The file names, in plain language

Packages differ, so do not expect every map to contain the same four files. These are the common roles:

- **`.ymap`** places entities at world coordinates and stores map relationships.
- **`.ytyp`** defines custom archetypes, including models or interior data that a `.ymap` can place.
- **`.ydr`** contains a drawable model.
- **`.ytd`** contains textures used by models.
- **`.ybn`** contains collision bounds, such as floors and walls.
- **`_manifest.ymf`** can describe map and LOD relationships generated by authoring tools. It is common in authored map packages, but its presence alone does not prove the map is valid.

An MLO can use more files than this, and a simple map edit can use fewer. The package author's manifest and documentation are the source of truth for that package.

## Install and validate it

<StepGroup>
  <Step num={1} title="Record a clean baseline" outcome="You can compare before and after and return to the exact starting state.">
    Use a local or staging server, not the live server. Record:

    ```text
    Map package:
    Version/date:
    Source or purchase record:
    Documented coordinates:
    Current server artifact:
    Existing map resources near those coordinates:
    ```

    Make a copy of `server.cfg` and the current map-resource folder. If the package includes Lua, JavaScript, DLL, or executable files, inspect them before starting the resource. A map-only package normally does not need arbitrary server scripts.
  </Step>

  <Step num={2} title="Check the folder boundary" outcome="FiveM sees one resource, not an accidental folder inside a folder.">
    The resource root must contain `fxmanifest.lua`. This is correct:

    ```text
resources/[maps]/qu_test_map/
  fxmanifest.lua
  stream/
    example.ymap
    example.ytyp
    example.ybn
    example.ydr
    example.ytd
    ```

    If extraction produced `qu_test_map/qu_test_map/fxmanifest.lua`, move the inner folder up one level. Do not rename the asset files unless the author says to; references inside map data may depend on those names.
  </Step>

  <Step num={3} title="Read the manifest before changing it" outcome="You know what FiveM will mount and what code, if any, will execute.">
    A small current map manifest often has this shape:

    ```lua
fx_version 'cerulean'
game 'gta5'

this_is_a_map 'yes'
    ```

    `this_is_a_map 'yes'` marks the resource as a GTA map and tells FiveM to reload map storage when it starts. Files inside `stream/` are streamed to clients. Some packages also declare `files {}` or `data_file` entries for their specific content. Preserve documented entries instead of adding a copied `DLC_ITYP_REQUEST` line to every map.

    Remove an old `lua54 'yes'` line only if you are maintaining the package yourself. It has been deprecated and ignored since Lua 5.4 became the only Lua runtime in June 2025. Do not make unrelated edits to a paid or escrowed resource.
  </Step>

  <Step num={4} title="Load only this map on staging" outcome="The resource starts with no manifest or streaming errors.">
    Add the resource after its required dependencies:

    ```text
ensure qu_test_map
    ```

    For a newly copied folder, run:

    ```text
refresh
ensure qu_test_map
    ```

    Read the full txAdmin console output. “Started resource” proves the manifest was accepted. It does **not** prove the interior, collision, doors, or LODs work. Any missing-file, duplicate-archetype, or data-file error is a failed test that you resolve before production.
  </Step>

  <Step num={5} title="Test at the documented coordinates" outcome="The map works as a player experience, not just as a started resource.">
    Join the staging server and check each item:

    1. Approach from far away and watch when the exterior and interior appear.
    2. Walk every floor, stair, doorway, and edge to test collision.
    3. Check doors, portals, lighting, reflections, and weather transitions.
    4. Drive around the block and approach from several directions.
    5. Test with the other map resources enabled, looking for duplicate walls, flicker, holes, or overlapping collision.
    6. Ask a second player to join. Confirm both clients see the same result and neither reports a crash.

    Take screenshots and write the coordinates of every failure. “It loaded for me once” is not a release test.
  </Step>

  <Step num={6} title="Check cost with the right evidence" outcome="You have before/after evidence without pretending one number measures an entire map.">
    Record the package's compressed and extracted size, join/download time on a clean client, and client behavior while approaching the location. Open `resmon` in F8 if the resource includes client scripts; it measures script CPU and memory, not whether collision and LODs are correct. An asset-only map can show almost no script CPU while still having oversized textures or poor LODs.

    Test on representative hardware and watch for stutter, texture loss, late pop-in, and crashes. For server scripts shipped with the package, use the documented server profiler. Keep the before/after notes with the package version so a later update can be compared to the same baseline.
  </Step>

  <Step num={7} title="Pass it or roll it back" outcome="A failed map leaves no production residue.">
    A pass requires clean startup, correct visuals, solid collision, acceptable approach behavior, no known overlap, and an acceptable download/runtime result. If any blocking check fails:

    ```text
stop qu_test_map
    ```

    Remove or comment its `ensure` line, restore the saved configuration if needed, and reconnect to confirm the old map state is back. Report the package version, coordinates, screenshots, and exact console error to the author.
  </Step>
</StepGroup>

## What the manifest can and cannot prove

`fxmanifest.lua` tells FiveM how to treat the resource. It can mark a map with `this_is_a_map`, list files, register supported data-file types, and declare dependencies. It cannot inspect the contents of a `.ytyp`, confirm a `.ymap` points to the intended coordinates, or verify a `.ybn` has a usable floor. Those are runtime and asset-authoring checks.

FiveM supports raw `.ymap` and `.ytyp` files in a resource. That does not mean every downloaded package is valid. The game parser still has to understand the data, and map relationships still have to be authored correctly.

## Overlap and LODs

Two maps at the same coordinates do not automatically crash every client. The real outcomes depend on their contents: z-fighting, duplicated props, conflicting doors, overlapping collision, visual holes, performance loss, or client instability. Test the exact combination you plan to ship. Do not state “same address always crashes” as a rule.

LOD means **level of detail**. Authors use simpler distant models and link them to higher-detail nearby models through map hierarchy. Missing or incorrect LOD setup can cause obvious pop-in, disappearing geometry, or excessive rendering cost. Server owners diagnose this by approaching from several distances and returning evidence to the asset author; fixing LOD hierarchy is an asset-authoring task, not a `server.cfg` tweak.

<Exercise>
  Create a one-page QA record for one real test map.

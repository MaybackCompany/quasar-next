<Callout variant="fivem" title="What an MLO actually is">
  An MLO (Map Loader Object) is a custom interior built from a handful of binary files: an archetype definition, a collision file, a placement file, and a manifest that ties them together. The client downloads every byte on join. A missing collision file drops players through the floor, a missing detail level (LOD) causes pop-in, and stacked interiors over the same coordinates fight and crash. The fix is always knowing which file is which.
</Callout>

## Build it

You are not going to hand-author a binary interior in this lesson. Those files come out of CodeWalker or 3ds Max. What you will do is build the resource that *registers* an interior the way every MLO must be registered, then prove with a real number that FiveM accepted your registration. That manifest line is the single thing beginners get wrong, and it is the difference between an interior that loads and one the server silently ignores.

<StepGroup>
  <Step num={1} title="Make the resource and stream folder" outcome="You have an empty qu_mlo resource with a stream/ folder ready for interior files.">
    Interior assets live inside a `stream/` folder. The `stream/` folder is special: FiveM picks up the asset files inside it automatically and ships them to clients. Inside your server's `resources` folder, create this layout:

    ```text
resources/qu_mlo/
  fxmanifest.lua
  proof.lua
  stream/
    ```

    The four files a real interior drops into `stream/` are described in the next section. For this proof you only need to add one placeholder so the count is non-zero. Create an empty file named `stream/qu_interior.ytyp` (right now it is just a stand-in so you can see the wiring work; a real `.ytyp` comes from your map artist).
  </Step>

  <Step num={2} title="Write fxmanifest.lua" outcome="FiveM knows this resource ships a custom interior archetype.">
    A file in `stream/` is loaded as a streamed asset on its own, but a `.ytyp` interior definition needs an explicit `data_file` line so the game registers its archetypes. This is the canonical entry for any custom interior. Open `fxmanifest.lua` and paste this exactly:

    ```lua
fx_version 'cerulean'
game 'gta5'

author 'Quasar Academy'
description 'MLO registration proof'

-- Registers the custom interior's archetype definition.
-- Every MLO needs this exact data_file line, one per .ytyp it ships.
data_file 'DLC_ITYP_REQUEST' 'stream/qu_interior.ytyp'

-- Our server-side proof that the registration was accepted.
server_script 'proof.lua'
    ```

    There is no `lua54 'yes'` line. As of June 2025 that directive is deprecated and ignored: Lua 5.4 is the only Lua runtime now, so you leave it out.
  </Step>

  <Step num={3} title="Write the proof" outcome="The resource prints, on start, exactly how many interior archetypes it registered.">
    Open `proof.lua` and paste this. It asks FiveM, when the resource starts, how many `data_file 'DLC_ITYP_REQUEST'` entries this manifest declared, and prints the count:

    ```lua
AddEventHandler('onResourceStart', function(resourceName)
    if resourceName ~= GetCurrentResourceName() then return end

    local count = GetNumResourceMetadata(resourceName, 'data_file')
    print(('[qu_mlo] registered interior archetypes: %d'):format(count))
end)
    ```
  </Step>

  <Step num={4} title="Start it and read the number" outcome="The expected count appears in the server console.">
    Open `server.cfg` and add this line:

    ```text
    ensure qu_mlo
    ```

    Save, then run:

    ```text
    restart qu_mlo
    ```

    <ExpectedOutput>
      [qu_mlo] registered interior archetypes: 1
    </ExpectedOutput>

    A `1` means FiveM read your `data_file 'DLC_ITYP_REQUEST'` line and accepted the archetype registration. A `0` means the line is missing or misspelled and a real interior would never load no matter how good the assets are. You now have a yes-or-no probe for the most common MLO failure, before you ever travel to the location.
  </Step>
</StepGroup>

## How it works

You restarted the resource and read a `1`. That number is the whole point: it is FiveM confirming it parsed the one manifest line that makes an interior real. Now take the pieces apart so the next interior you add is a decision, not a copy-paste prayer.

### The four files an MLO is actually made of

An MLO is not one file. It is a small set of binary files that each own a different job, and almost every MLO problem maps cleanly to one of them being missing or wrong.

- **`.ytyp` (the archetype definition).** This is the catalogue of every room, portal, and prop the interior contains. It is the file you register with `data_file 'DLC_ITYP_REQUEST'`. If the `.ytyp` is not registered, the game has no idea the interior exists and you walk through an empty lot. A wrong or missing `.ytyp` is the "no interior" failure.
- **`.ybn` (the collision/bounds).** This is the physical shape players and vehicles bump into: floors, walls, stairs. The game streams it automatically from `stream/`. If the `.ybn` is missing or fails to stream, the meshes still render but there is nothing solid behind them, so you fall through the floor.
- **`.ymap` (the placement).** This says where in the world the interior sits and which props spawn inside it. Two MLOs that ship a `.ymap` pinned to the same coordinates are the classic crash: both try to own the same patch of Los Santos and the game chokes.
- **`_manifest.ymf` (the manifest).** This ties the archetypes and the map data together and declares the interior's bounds to the streamer. A broken `_manifest.ymf` shows up as props that flicker, refuse to load, or load in the wrong place.

The mental model: the `.ytyp` says *what exists*, the `.ybn` says *what is solid*, the `.ymap` says *where it goes*, and the `_manifest.ymf` says *how it all fits together*. When something breaks, name the symptom and you have named the file.

### The manifest line that registers it

```lua
data_file 'DLC_ITYP_REQUEST' 'stream/qu_interior.ytyp'
```

`data_file` is a manifest directive, not Lua code that runs. When the resource starts, FiveM reads this line and registers the named file as a specific kind of game data. The first argument, `'DLC_ITYP_REQUEST'`, is the data type: it tells the game "this is an interior archetype definition, load it the way you load DLC interior types." The second argument is the path to the file inside the resource. You write one `data_file 'DLC_ITYP_REQUEST'` line per `.ytyp` the MLO ships. A two-room interior split across two `.ytyp` files needs two of these lines. Forget the line and the `.ytyp` just sits in `stream/` as inert bytes, which is exactly the silent failure your proof catches.

Notice what the proof reads back:

```lua
local count = GetNumResourceMetadata(resourceName, 'data_file')
```

`GetNumResourceMetadata` is a server native that counts how many times a given manifest key appears. We ask for `'data_file'`, and because our manifest has exactly one `data_file` line, it returns `1`. This is why the proof is honest: it is not checking that the interior looks good, it is checking that FiveM parsed the registration you wrote. That separation matters. Plenty of MLOs ship gorgeous meshes and still load as nothing because this one line was wrong.

### Why stacking two interiors over the same spot crashes

The `.ymap` placement is anchored to fixed world coordinates. If you ensure two MLO resources that both place an interior at, say, the same Mirror Park house, both `.ymap` files claim the same volume of the map. The streamer cannot resolve which interior owns the space, props from both spawn into each other, collision overlaps, and clients near that coordinate crash. This is not a size problem you can optimize away. It is a conflict, and the only fix is to remove one of the interiors or relocate it. Before you ensure a new MLO, check whether anything you already run sits at the same address.

### What an LOD actually is

LOD stands for Level Of Detail, and it is the reason a distant building is not rendering every door handle. The game ships an asset at several detail tiers and swaps between them by distance:

- **HD**: full detail, used when you are close.
- **LOD**: a lighter version shown at medium distance.
- **SLOD** (super-LOD): the cheapest silhouette, shown far away.

A well-built MLO includes these tiers so detail fades in smoothly as you approach. "Missing LODs" is now a concrete, diagnosable thing: if the HD model has no LOD or SLOD behind it, the interior either pops in abruptly at close range or shows as invisible geometry from a distance, because the game has nothing to draw in the gap. You cannot fix this from the manifest. It is baked into the asset, so the fix is a properly built MLO from a competent artist.

### Measuring the cost

Loading is half the job. The other half is knowing what the interior costs once it is in. Open the in-game profiler with the console command:

```text
resmon
```

Press it again to expand to detailed view (or use `resmon 1`). You are watching two numbers for your interior's resource:

- **CPU time in milliseconds.** A streamed MLO with no scripts should sit near `0.00ms` to `0.01ms` server-side, because it ships assets, not logic. If a map resource is burning real CPU time, something is running a thread it should not.
- **Streaming memory.** This is the weight every joining player downloads and holds. Watch it climb as you add interiors. There is no single hard cap, but a single interior eating hundreds of megabytes is a red flag, and total streaming memory is what blows up join times and crashes weaker clients.

The discipline is simple: add the interior, open `resmon`, read the streaming memory before and after, and decide whether the look is worth the bytes. A number you measured beats "keep textures sane" every time.

<Checkpoint question="Your proof prints 1, but when you travel to the location there is no interior. Which file is the suspect, and why does the proof not catch it?">
  The suspect is the `.ytyp` archetype content (or the `.ymap` placement), not the registration. Your proof only confirms FiveM parsed the `data_file 'DLC_ITYP_REQUEST'` line, that is why it printed `1`. It does not validate that the `.ytyp` actually describes a real interior or that a `.ymap` places it in the world. So a `1` plus an empty lot means the wiring is correct but the asset content is wrong or the placement file is missing. Registration and content are two separate things, and the proof deliberately only tests the first one. To go further you walk to the coordinates and check whether anything streams in.
</Checkpoint>

## Try it yourself

<Exercise>
  Extend the proof so it reports the actual `.ytyp` path it registered, not just the count. `GetResourceMetadata` reads the value at a given index, so you can pull back the path string itself:

  ```lua
  AddEventHandler('onResourceStart', function(resourceName)
      if resourceName ~= GetCurrentResourceName() then return end

      local count = GetNumResourceMetadata(resourceName, 'data_file')
      for i = 0, count - 1 do
          local path = GetResourceMetadata(resourceName, 'data_file', i)
          print(('[qu_mlo] data_file[%d] = %s'):format(i, path))
      end
  end)
  ```

  Restart `qu_mlo` and confirm it prints `[qu_mlo] data_file[0] = stream/qu_interior.ytyp`. Then add a second `data_file 'DLC_ITYP_REQUEST' 'stream/qu_interior2.ytyp'` line to the manifest, create that placeholder file, restart, and watch both paths print. You have just built a tool that lists exactly which interiors a resource claims to register, which is how you audit a downloaded MLO before you trust it on a live server.
</Exercise>

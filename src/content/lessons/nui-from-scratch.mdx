RegisterCommand('openui', function()

</html>

const menu = document.getElementById('menu');
const title = document.getElementById('title');
const button = document.getElementById('action');

// Lua to JS: reveal the menu only when Lua pushes the open message.
window.addEventListener('message', (event) => {
    if (event.data.action === 'open') {
        title.textContent = event.data.title;
        menu.classList.remove('hidden');
    }
});

button.addEventListener('click', () => {
    fetch(`https://${GetParentResourceName()}/buttonClicked`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },

});

## How it works

You typed a command, a panel appeared, you clicked once, and a chat line came back carrying text the button sent. Four moving parts made that happen, and the order they fire in is the whole mental model. Take them in the order the data flows: declare the page, capture focus, push state Lua to JS, then send a payload JS to Lua.

### The manifest: ui_page and files

```lua
ui_page 'web/index.html'

files {
    'web/index.html',
    'web/app.js',
    'web/style.css'
}
```

`ui_page` does one job: it names the single HTML file FiveM mounts as this resource's NUI layer. That page renders fullscreen on top of the game, transparent where you do not draw anything. There is exactly one `ui_page` per resource.

`files` is a separate, mandatory list. NUI runs inside a sandboxed browser, and that browser may only load assets you explicitly whitelist here. The HTML page itself is loaded because it is the `ui_page`, but every asset the page then asks for, the stylesheet, the script, an image, a font, must also appear in `files`. Leave `app.js` out of the `files` block and the browser's request for it returns a 404, the script never runs, and your button does nothing, with no Lua error to explain why. That silent 404 is the single most common reason a NUI page looks dead. When a NUI feature half-works, check `files` first.

Note the dependency-style detail while you are here. If this resource needed another resource loaded first, the canonical key is the plural block:

```lua
dependencies {
    'some_resource'
}
```

The singular `dependency 'some_resource'` is tolerated but the plural block is the form to learn. We do not need it for this lesson, since NUI is built into the client runtime.

### SetNuiFocus: who is the keyboard and mouse talking to

```lua
SetNuiFocus(true, true)
```

By default the game owns your mouse and keyboard, and the NUI layer is click-through: input passes straight to GTA. `SetNuiFocus` is how you hand input to the browser instead. It takes two booleans, and they are independent.

- The first boolean is **keyboard/input focus** (`hasFocus`). `true` routes key presses into the page and captures input (it also blocks WASD movement) — what you want for a text input or search box.
- The second boolean is **mouse-cursor focus** (`hasCursor`). `true` shows the cursor and lets the browser receive clicks. Without it, your button is visible but unclickable because every click still goes to the game underneath.

The rule that saves you hours: whatever you capture, you must release. When the menu closes you call `SetNuiFocus(false, false)`. Forget it and the player is stuck staring at a cursor, unable to move, walk, or open the chat, with the game apparently frozen. It is not frozen. Focus never went back to the game. In this lesson the release happens on the first line of the callback, the moment the click comes in, so focus returns before anything else runs.

### SendNUIMessage: pushing state from Lua to JS

```lua
SendNUIMessage({ action = 'open', title = 'NUI from scratch' })
```

```js
window.addEventListener('message', (event) => {
    if (event.data.action === 'open') {
        title.textContent = event.data.title;
        menu.classList.remove('hidden');
    }
});
```

`SendNUIMessage` is the one-way channel from Lua into the page. You hand it a Lua table; FiveM serializes that table to JSON and dispatches it to the browser as a DOM `message` event. On the JS side you listen with `window.addEventListener('message', ...)` and read the payload off `event.data`. The keys you set in Lua are the keys you read in JS: `action` and `title` here.

This is the piece the old version of this lesson was missing, and it is why the page now starts hidden. The menu carries the `hidden` class until a message with `action` equal to `open` arrives, and only then does it reveal itself and stamp in the title Lua sent. So the UI is not just sitting there waiting; Lua is driving it. That is the real pattern: you almost never want a NUI page that shows itself. You want Lua to decide when, and to ship the data the page should display in the same message. An `action` field is the convention for telling the page which of its many states to switch into.

`SendNUIMessage` is fire-and-forget. It does not return a value and the page cannot reply through it. For a reply you need the other direction.

### RegisterNUICallback: sending a payload from JS back to Lua

```js
fetch(`https://${GetParentResourceName()}/buttonClicked`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ label: 'Send to Lua and close' })
});
```

```lua
RegisterNUICallback('buttonClicked', function(data, cb)
    SetNuiFocus(false, false)
    local label = data.label or 'unknown'
    -- ...
    cb({ ok = true })
end)
```

The page talks back to Lua by making an HTTP request. The current documented form is `https://${GetParentResourceName()}/&lt;endpoint&gt;`. `GetParentResourceName()` is provided inside NUI and returns the resource that owns the page, so the code keeps working if the folder is renamed. The endpoint after the slash must exactly match the string passed to `RegisterNUICallback`. Do not hardcode `cfx-nui-&lt;resource&gt;` for callbacks: that prefix is the resource-file URL scope, while the official callback API uses the parent resource name directly.

On the Lua side, `RegisterNUICallback('buttonClicked', function(data, cb) ... end)` is the listener for that endpoint. The handler is handed two arguments:

- `data` is the request body, already parsed from JSON into a Lua table. The `{ label: '...' }` object the `fetch` sent arrives as `data.label`. That is how you read the value the button carried. Defaulting with `data.label or 'unknown'` guards against a malformed body.

  Security note: a NUI callback body is client input, so treat it as untrusted. `data.label or 'unknown'` only guards against a *missing* value, not a malicious one. The moment a callback does anything that matters (gives an item, moves money, fires a server event), validate every field server-side and never trust a value just because your own page sent it.
- `cb` is the response function. The browser's `fetch` is a promise waiting on a reply, and `cb({ ok = true })` is that reply. Until you call `cb`, the `fetch` promise never resolves, so the `.then` that hides the menu never runs. You must call `cb` exactly once in every callback, even if you only send `cb({})`. A callback that forgets `cb` leaves the page hanging on every click.

The chat line is the visible proof the data crossed: `TriggerEvent('chat:addMessage', ...)` fires the built-in `chat` resource's message event on the client and prints the `label` the JS sent. If that text shows up in chat, the round-trip is real. An empty request body that the callback ignored, which is what the old lesson shipped, could never produce it.

### Attaching Chrome devtools to a live NUI page

NUI is a real Chromium instance, so it has the same devtools you use in a browser. With the game running, open `http://localhost:13172` in Chrome on the same machine. (You can also open the inspector from inside the game by typing `nui_devTools` in the F8 console, once FiveM's developer mode is on.) You get the standard inspector pointed at every live NUI page: the Console tab shows your JS errors and `console.log` output, the Network tab shows each callback request and its response, and Elements lets you tweak CSS live. The moment a NUI feature misbehaves, this is the first place to look, because a JS error that the game would otherwise swallow shows up here in red.

<Checkpoint question="The button click fires the fetch, but the menu never closes and the chat line never appears, with no error in F8. Where do you look first, and what is the most likely cause?">
  Open the NUI devtools at `http://localhost:13172` and check the Network tab. The most likely cause is a mismatch between the `fetch` URL and the registered callback: the URL should use `GetParentResourceName()` and the endpoint string after the slash must match `RegisterNUICallback` (`buttonClicked` in both places). A failed or pending request confirms it. If the request succeeds but the menu still does not close, the Lua callback ran but never called `cb`, so the `fetch` promise that hides the menu in `.then` never resolves.
</Checkpoint>

### When you reach for this pattern

These four pieces are the entire foundation of every NUI resource, framework or not. You reach for them whenever a feature needs a screen the player interacts with, rather than a simple key prompt. Three real cases:

- **An interaction menu.** Lua detects the player is near an NPC, calls `SetNuiFocus` and `SendNUIMessage` with the menu options, and each click comes back through a callback. Garages, shops, and job menus are all this shape.
- **A minigame.** A lockpick, a hacking puzzle, a skill-check. Lua starts it with a message carrying the difficulty, the page runs the game loop in JS, and a single callback reports success or failure back. The video at the top of this lesson is exactly this.
- **A hud.** Here you usually skip `SetNuiFocus` entirely, because a hud should never steal the cursor. Lua just streams state with `SendNUIMessage` on a timer and the page redraws. Same message channel, no focus, no callback.

## Try it yourself

<Exercise>
  Turn the one button into a two-button menu that tells Lua which one was pressed. Add a second button to `index.html` and send the button's own label in the `fetch` body:

  ```js
  document.querySelectorAll('button').forEach((btn) => {
      btn.addEventListener('click', () => {
          fetch(`https://${GetParentResourceName()}/buttonClicked`, {
              method: 'POST',
              headers: { 'Content-Type': 'application/json' },
              body: JSON.stringify({ label: btn.textContent })
          }).then(() => menu.classList.add('hidden'));
      });
  });
  ```

  The Lua callback already reads `data.label`, so it needs no change. Restart `qu_nui_from_scratch`, open the menu, and click each button. Confirm the chat line names the exact button you pressed. You just used one callback to handle many UI actions by letting the page say which one fired, which is how real menus route every option through a single endpoint instead of one callback per button.
</Exercise>

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_audit_bank
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_audit_bank/
  fxmanifest.lua
  server.lua
    ```

    This is a server-only resource. There is no client file and no NUI, so there is no F8 output to read. You will drive it with commands that you run from your own in-game chat, because a command typed in chat by a connected player carries a real player `source`, and the `source` is the heart of this lesson.
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script '@oxmysql/lib/MySQL.lua'
server_script 'server.lua'

dependencies {
    'oxmysql'
}
    ```

    There is no `lua54 'yes'` line. As of June 2025 that directive is deprecated and ignored, since Lua 5.4 is the only Lua runtime. The `dependencies { 'oxmysql' }` block is the canonical plural form and tells the server to start oxmysql before `qu_audit_bank`, so the `@oxmysql/lib/MySQL.lua` import resolves.
  </Step>

  <Step num={4} title="Create the two tables" outcome="The database has one balance per player and one append-only audit log.">
    A bank with a record needs two tables, not one. Run this SQL once against your database before starting the resource. oxmysql speaks to MariaDB, the recommended engine for FiveM in 2026:

    ```sql
CREATE TABLE IF NOT EXISTS qu_audit_balances (
  citizenid VARCHAR(64) NOT NULL,
  balance INT NOT NULL DEFAULT 0,
  PRIMARY KEY (citizenid)
);

CREATE TABLE IF NOT EXISTS qu_audit_log (
  id INT AUTO_INCREMENT PRIMARY KEY,
  citizenid VARCHAR(64) NOT NULL,
  source INT NOT NULL,
  action VARCHAR(16) NOT NULL,
  amount INT NOT NULL,
  balance_before INT NOT NULL,
  balance_after INT NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
    ```

    The balances table keys on `citizenid`: one row per player, and the primary key forbids a duplicate. The audit log is the opposite shape on purpose. It keys on an `AUTO_INCREMENT` id and is append-only, so every transaction adds a new row and nothing is ever overwritten. The `created_at` column fills itself in with the server time on every insert, so you never pass a timestamp by hand.
  </Step>

  <Step num={5} title="Write the lesson code" outcome="A real player's intent flows through a server event into a balance change plus an audit row.">
    Open `server.lua` and paste this:

    ```lua
-- In a real server the citizenid comes from the player's framework identity
-- (QBCore, Qbox, ESX). For the lesson we derive a stable id from the source so
-- you can test with a real connected player and still get a consistent account.
local function citizenIdFor(source)
    return 'player_' .. source
end

-- Read the balance, creating the account at 0 on first touch.
local function getBalance(citizenid)
    local row = MySQL.single.await(
        'SELECT balance FROM qu_audit_balances WHERE citizenid = ?',
        { citizenid }
    )
    if not row then
        MySQL.insert.await(
            'INSERT INTO qu_audit_balances (citizenid, balance) VALUES (?, ?)',
            { citizenid, 0 }
        )
        return 0
    end
    return row.balance
end

-- The single trusted path. Every money move on the server goes through here.
-- This is a same-runtime handler fired by AddEventHandler below, so the player
-- is passed in explicitly as `src` (the magic `source` is only populated for a
-- real client -> server net event, which this is not).
AddEventHandler('qu_audit_bank:request', function(src, action, amount)
    local citizenid = citizenIdFor(src)

    -- Validate the intent. The client may only ask to deposit or withdraw a
    -- positive whole number. It never sends a balance.
    amount = tonumber(amount)
    if action ~= 'deposit' and action ~= 'withdraw' then return end
    if not amount or amount <= 0 or amount ~= math.floor(amount) then
        print('[qu_audit_bank] REJECTED bad amount from source ' .. src)
        return
    end

    local balanceBefore = getBalance(citizenid)
    local balanceAfter

    if action == 'deposit' then
        balanceAfter = balanceBefore + amount
    else -- withdraw
        if amount > balanceBefore then
            print('[qu_audit_bank] REJECTED withdraw of ' .. amount .. ', balance is only ' .. balanceBefore)
            return
        end
        balanceAfter = balanceBefore - amount
    end

    -- One safe UPDATE applies the change the server computed.
    MySQL.update.await(
        'UPDATE qu_audit_balances SET balance = ? WHERE citizenid = ?',
        { balanceAfter, citizenid }
    )

    -- One INSERT records the transaction. This runs for every action.
    MySQL.insert.await(
        'INSERT INTO qu_audit_log (citizenid, source, action, amount, balance_before, balance_after) VALUES (?, ?, ?, ?, ?, ?)',
        { citizenid, src, action, amount, balanceBefore, balanceAfter }
    )

    print('[qu_audit_bank] ' .. action .. ' ' .. amount ..
        ' for ' .. citizenid .. ': ' .. balanceBefore .. ' -> ' .. balanceAfter)
end)

-- Thin commands so you can fire the request as a real player from chat.
-- These commands are registered in a server_script, so their handlers run on
-- the server, where RegisterCommand hands you the player's server id as the
-- first argument. We pass that trusted source into the same-runtime event with
-- TriggerEvent (the correct same-runtime call; TriggerServerEvent is client-only).
RegisterCommand('deposit', function(source, args)
    TriggerEvent('qu_audit_bank:request', source, 'deposit', args[1])
end, false)

RegisterCommand('withdraw', function(source, args)
    TriggerEvent('qu_audit_bank:request', source, 'withdraw', args[1])
end, false)

-- Print the last few audit rows so you can read the trail.
RegisterCommand('auditlog', function()
    local rows = MySQL.query.await(
        'SELECT id, citizenid, source, action, amount, balance_before, balance_after FROM qu_audit_log ORDER BY id DESC LIMIT 5',
        {}
    )
    for i = #rows, 1, -1 do
        local r = rows[i]
        print('[qu_audit_bank] #' .. r.id .. ' src=' .. r.source ..
            ' ' .. r.action .. ' ' .. r.amount ..
            ' (' .. r.balance_before .. ' -> ' .. r.balance_after .. ')')
    end
end, true)
    ```
  </Step>

## How it works

You ran it as a real player, the server moved your money, and a permanent record appeared. Now take the code apart so you understand the three things that make this different from a number on a screen: where `source` comes from, why the client can only send an intent, and why the audit row is written on the same trip as the balance change.

### source is the server's proof of who asked

```lua
RegisterCommand('deposit', function(source, args)
    TriggerEvent('qu_audit_bank:request', source, 'deposit', args[1])
end, false)

AddEventHandler('qu_audit_bank:request', function(src, action, amount)
    local citizenid = citizenIdFor(src)
```

The mini bank ran from the console, where there is no real player. This lesson runs from a connected player, but it stays entirely on the server: the `deposit` command is registered inside a `server_script`, so its handler runs on the server, and there `RegisterCommand` hands you the player's server slot as the first callback argument, `source`. That is the one piece of identity the player cannot fake, because the server assigns the slot, not the client. There is no client file and no round-trip across the wire for the identity; the trusted `source` comes straight from the command callback.

Because everything happens in one runtime, the command passes that trusted `source` explicitly into the handler with `TriggerEvent`, which fires a same-runtime event. The handler receives it as its first parameter, `src`, and uses that local everywhere after. This is the key reason for the explicit pass: the magic `source` global is only populated for a real client -> server net event, which this is not, so a plain same-runtime handler must be told who the player is. From `src` you derive the `citizenid`, and that is the account the rest of the handler reads and writes. The client typed `action` and `amount` into chat. It did not, and could not, supply the identity.

### The client sends an intent, never a balance

```lua
amount = tonumber(amount)
if action ~= 'deposit' and action ~= 'withdraw' then return end
if not amount or amount <= 0 or amount ~= math.floor(amount) then
    print('[qu_audit_bank] REJECTED bad amount from source ' .. src)
    return
end
```

Look at what crosses the wire from the client: an `action` string and an `amount`. That is the entire intent. The client is saying "I would like to deposit 100." It is not saying "my balance is now 100," and it has no way to. The server never reads a balance from the request, so there is nothing the client can lie about that touches the truth.

Because the intent is untrusted, the server validates it at the boundary before acting. The `action` must be exactly `deposit` or `withdraw`. The `amount` must be a number, must be positive, and must be a whole number, so `withdraw -500` (a deposit in disguise), `deposit 0`, and `deposit 10.5` are all refused before any query runs. This is the same edge-validation rule you follow with `?` placeholders: distrust input the moment it arrives, and only let clean values past the gate.

### Read, decide, write, in that order

```lua
local balanceBefore = getBalance(citizenid)
local balanceAfter

if action == 'deposit' then
    balanceAfter = balanceBefore + amount
else -- withdraw
    if amount > balanceBefore then
        print('[qu_audit_bank] REJECTED withdraw of ' .. amount .. ', balance is only ' .. balanceBefore)
        return
    end
    balanceAfter = balanceBefore - amount
end

MySQL.update.await(
    'UPDATE qu_audit_balances SET balance = ? WHERE citizenid = ?',
    { balanceAfter, citizenid }
)
```

The server reads the real `balanceBefore` from its own table, decides the `balanceAfter` itself, then writes it in a single UPDATE. The reject-if-larger check sits between the read and the write, comparing the requested `amount` against the balance the server just read, never against anything the client sent. A withdraw bigger than the balance prints a rejection and returns before the UPDATE, so the balance never goes negative.

Notice the deposit and the withdraw both end at the same single UPDATE. There is exactly one place in this handler where the stored balance changes, and it writes the value the server computed. Keeping the write to one statement is what makes the audit row trustworthy: the number you log as `balance_after` is the number you actually stored, computed in the same handler, not a second guess.

### The audit row is written on every action, on the same trip

```lua
MySQL.insert.await(
    'INSERT INTO qu_audit_log (citizenid, source, action, amount, balance_before, balance_after) VALUES (?, ?, ?, ?, ?, ?)',
    { citizenid, src, action, amount, balanceBefore, balanceAfter }
)
```

This INSERT is the lesson. Immediately after the balance changes, the server appends one row to the append-only log: who (`citizenid` and the player's `src`, stored in the `source` column), what (`action`), how much (`amount`), and the balance on both sides of the move (`balance_before`, `balance_after`). The `created_at` timestamp fills itself in from `DEFAULT CURRENT_TIMESTAMP`, so you do not pass it.

Two details matter. First, the INSERT runs for every successful action, not just some, so the log is complete: any balance the table holds can be reconstructed by replaying the rows. Second, it sits below both rejection `return`s on purpose. A rejected withdraw never moved money, so it never writes an audit row. The log records transactions, not attempts. If you also wanted to record refused attempts for fraud review, you would add a separate INSERT before each `return`, with its own action like `withdraw_rejected`.

<Checkpoint question="A player runs /withdraw 1000000 while holding 50. What does the audit log show afterward, and why?">
  Nothing new. The amount is the only thing the player typed; the identity, the balance, and the decision all live on the server. Because the command is registered in a server_script, `RegisterCommand` hands the handler the player's real server slot as `source`, which it passes into the request. The handler derives the `citizenid` from that trusted `src`, reads the true balance of 50 from its own table, sees the requested 1,000,000 is larger, prints a rejection, and `return`s before either the UPDATE or the audit INSERT runs. No money moved, so no row is logged. The player can type any amount, but they cannot touch the balance the server reads, cannot forge the slot the server assigns, and cannot reach the INSERT. The only thing they control is the request, never the record.
</Checkpoint>

### When you reach for this pattern

A server-owned balance plus an append-only audit log is the shape every economy a server admin will ever have to answer for. Three real cases:

- **Money disputes.** A player claims a shop charged them twice. With an audit log you read the two rows, see the two `balance_before -> balance_after` steps, and settle it in seconds instead of guessing. Without it, you have only the current number and no history.
- **Catching exploits.** A balance that jumps from 200 to nine million between two rows is a hole in some other resource. The audit log is where that jump becomes visible, because every legitimate change leaves a small, explainable step.
- **Reversals and refunds.** Because the log records `balance_before` for every move, you can undo a bad transaction by writing the inverse, and that reversal is itself a logged row. The trail stays honest even when you correct it.

  items={[
    { symptom: "source is nil or attempt to concatenate a nil value (source)", fix: "You ran deposit or withdraw from the server console instead of in-game.

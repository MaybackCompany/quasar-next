## Find a native

A **native** is a built-in game or Cfx function exposed to your script. There are thousands. Nobody memorizes them; look them up in the official reference, then use the Lua manual for the surrounding language behavior.

<TechLinks ids={["natives", "fivem-docs", "lua"]} />

When you open a native, read three things before you use it:

- **The side badge.** Each native is marked **CLIENT** or **SERVER**. A client native like `GetEntityCoords` will not run in a `server_script`, and the reverse is also true. Match the native's side to the file it lives in.
- **The arguments.** The signature shows what to pass in and in what order, for example `GetEntityCoords(entity)`. Pass the wrong type and you get `nil` or a crash.
- **The return value.** Some natives hand you something back (coords, health, a handle); some just do a thing and return nothing.

<Checkpoint question="A native is marked CLIENT. Can you call it from your server.lua?">
  No. Client natives only run in client scripts (loaded with `client_script`), and server natives only run in server scripts. The side badge in the docs tells you which file the call belongs in. Calling across the line is the single most common beginner native error. Every native you use in this lesson is a CLIENT native, which is exactly why the file you build is a `client_script` and not a `server_script`.
</Checkpoint>

    Open `client.lua` and paste this:

    ```lua
RegisterCommand('whereami', function()
    local ped = PlayerPedId()
    local coords = GetEntityCoords(ped)
    local health = GetEntityHealth(ped)
    print(('[qu_natives_toolbox] x %.2f y %.2f z %.2f health %d'):format(coords.x, coords.y, coords.z, health))
end, false)

RegisterCommand('tpsafe', function()
    local ped = PlayerPedId()
    local x, y = 215.0, -810.0

    -- Stream in collision at the destination so we have ground to stand on.
    RequestCollisionAtCoord(x, y, 100.0)

    -- Drop the ped at a high z first so the world below has a chance to load.
    SetEntityCoords(ped, x, y, 100.0, false, false, false, false)

    -- Wait until the ground under that x,y is known, then snap down onto it.
    local groundZ = 0.0
    local found = false
    local tries = 0
    while not found and tries < 20 do
        Wait(50)
        found, groundZ = GetGroundZFor_3dCoord(x, y, 100.0, false)
        tries = tries + 1
    end

    if found then
        SetEntityCoords(ped, x, y, groundZ, false, false, false, false)
        print(('[qu_natives_toolbox] teleported to %.2f %.2f %.2f'):format(x, y, groundZ))
    else
        print('[qu_natives_toolbox] ground not found, staying at safe height')
    end
end, false)
    ```
  </Step>

## How it works

You ran it and the lines appeared. Now take the code apart so the next time you open the native docs you are deciding, not copying. Every line below is a native you looked up, used, and can now read on your own.

### Reading your ped: PlayerPedId and GetEntityCoords

```lua
local ped = PlayerPedId()
local coords = GetEntityCoords(ped)
local health = GetEntityHealth(ped)
```

`PlayerPedId()` is a CLIENT native that returns a **handle** to your own character, the ped you are currently controlling. A handle is just a number the game uses internally to point at one entity. You almost never read the number itself; you pass it to other natives that need to know which entity you mean. That is why the very first line of nearly every client script that touches your character starts with this call. It takes no arguments and hands you back one value.

`GetEntityCoords(ped)` takes that handle and returns the entity's position. Look at the docs and the return type is a **vector3**: a single value bundling three numbers, an x, a y, and a z. You pull each one out with `coords.x`, `coords.y`, and `coords.z`. In GTA V, x and y are the flat map position and z is height above sea level. So `coords.z` going up means you are climbing a hill or standing on a roof.

`GetEntityHealth(ped)` returns a plain number, not a vector. The exact maximum depends on the ped and server setup; do not hard-code `200` as a universal full-health rule. Notice the shape of the pattern: one native produces a handle and other natives consume it.

### Building the output line: the :format method

```lua
print(('[qu_natives_toolbox] x %.2f y %.2f z %.2f health %d'):format(coords.x, coords.y, coords.z, health))
```

The string here has placeholders in it. `%.2f` means insert a floating point number rounded to two decimal places. `%d` means insert a whole integer. `:format(...)` is a **method** you call directly on the string: it walks left to right, replacing the first `%.2f` with `coords.x`, the second with `coords.y`, the third with `coords.z`, and the `%d` with `health`. The result is one clean line instead of a string glued together with lots of `..` joins.

Why bother rounding? `GetEntityCoords` returns coordinates with many decimal places, which is noise when you are reading the console. `%.2f` trims each one to two places so the line stays scannable. The leading `[qu_natives_toolbox]` tag is text you typed on purpose, so you can pick your own output out of a console where twenty resources are all printing at once.

### A real safe teleport, not a nudge

This is the part most beginner tutorials get wrong. Moving a ped is not as simple as setting its coordinates, because the world around those coordinates may not be loaded yet. Set the position blindly and you fall through an unloaded map into the water, or land inside a hill. A *safe* teleport loads the ground first, then places you on it.

```lua
local x, y = 215.0, -810.0
RequestCollisionAtCoord(x, y, 100.0)
SetEntityCoords(ped, x, y, 100.0, false, false, false, false)
```

`RequestCollisionAtCoord(x, y, z)` is a CLIENT native that tells the game to stream in the physical collision mesh at that map position. Without collision there is literally no floor at the destination. Then `SetEntityCoords` drops you in at a deliberately high z of `100.0`, well above any building, so you are hanging in the air over the spot while the world below finishes loading.

Now look at `SetEntityCoords`' arguments. The first is the ped handle and the next three are the x, y, z you are moving it to. The four `false` values at the end are boolean flags the docs spell out:

- **alive, deadFlag, ragdollFlag** (the first three booleans): low-level physics flags the engine uses for dead and ragdolling peds. `alive` is effectively unused by the game; for a normal living ped you leave all three `false`. (Older guides mislabel these as `xAxis/yAxis/zAxis` "keep alignment" flags, which is wrong.)
- **clearArea** (the last boolean): whether to clear nearby objects and peds around the destination. `false` leaves the area as it is.

You do not need to memorize those four; you need to know they exist and where to read them, because a native with trailing booleans is exactly the kind of signature you must look up rather than guess.

```lua
local groundZ = 0.0
local found = false
local tries = 0
while not found and tries < 20 do
    Wait(50)
    found, groundZ = GetGroundZFor_3dCoord(x, y, 100.0, false)
    tries = tries + 1
end
```

`GetGroundZFor_3dCoord(x, y, z, includeWater)` returns **two values**: a boolean saying whether the ground was found, and the z height of that ground. Passing `false` for `includeWater` means the water surface is NOT treated as ground, so the ped lands on solid terrain. Right after a teleport the answer is often `false` for a frame or two, because the collision you requested has not finished streaming. (`GetGroundZFor_3dCoord` also only returns a real height when the target x,y is within your streaming distance, which is why this pattern works for moving yourself but not for probing the ground far across the map.) So you loop: `Wait(50)` yields for fifty milliseconds, you ask again, and you stop the moment `found` becomes `true`. The `tries < 20` guard caps the wait at one second so a bad coordinate never hangs the loop forever.

```lua
if found then
    SetEntityCoords(ped, x, y, groundZ, false, false, false, false)
    print(('[qu_natives_toolbox] teleported to %.2f %.2f %.2f'):format(x, y, groundZ))
else
    print('[qu_natives_toolbox] ground not found, staying at safe height')
end
```

Once you know the real ground height, you call `SetEntityCoords` one final time to snap the ped down onto it. That is the whole trick: drop high, wait for the ground to be known, then place onto the ground. Compare this with a naive `SetEntityCoords(ped, x, y, z, ...)` to a guessed z. A guessed z that is too low buries you in terrain; too high drops you from the sky. The two-step pattern is why this teleport is called *safe*.

<Checkpoint question="Why teleport to z 100.0 first and only then snap to the ground, instead of going straight to the final z?">
  Because the destination's collision mesh has to stream in before the game can tell you where the ground actually is. `RequestCollisionAtCoord` starts that load, the high drop at z 100.0 parks you safely above everything while it finishes, and `GetGroundZFor_3dCoord` keeps returning `false` until the ground exists. Once it returns `true` with a real height, you snap down onto it. Skip the wait and `GetGroundZFor_3dCoord` hands you a stale or zero height, so you teleport into terrain or under the map. The loop is not busywork; it is you waiting for the world to be real before you stand on it.
</Checkpoint>

### When you reach for this pattern

Reading a ped and moving it safely are the two most common things a client script does, and the workflow you just used is the one you repeat for every new native:

- **Reading state for a feature.** Any job, robbery, or interaction script starts by reading where the player is and what condition they are in. `PlayerPedId` plus `GetEntityCoords` is the opening move every single time.
- **Admin and respawn teleports.** A `/tp` command, a "return to last position" feature, or a respawn point all need the safe-teleport pattern, or players fall through the map the first time the destination has not loaded.
- **Capturing coordinates while building.** Run `/whereami`, stand somewhere, read the line, and you have the exact x, y, z to hard code into a spawn point or a blip. That is how developers harvest the coordinates you see baked into scripts.

## Try it yourself

<Exercise>
  Turn `tpsafe` into a `/tpto` command that teleports you to coordinates you type, instead of the hard coded Legion Square spot. FiveM passes the typed words into the handler as the second argument, a table called `args`, where each word arrives as a string.

  ```lua
  RegisterCommand('tpto', function(source, args)
      local x = tonumber(args[1])
      local y = tonumber(args[2])
      if not x or not y then
          print('[qu_natives_toolbox] usage: /tpto x y')
          return
      end
      -- Reuse the same safe-teleport steps from tpsafe, but with this x and y.
  end, false)
  ```

  Fill in the body with the request-collision, drop-high, wait-for-ground, snap-down pattern you just learned. `tonumber` converts the typed strings into numbers, and the `if not x or not y` guard prints a usage line and bails when someone forgets an argument. Test it with `/tpto 215.0 -810.0` and confirm you land on the ground at the same spot the original command used. Then try a coordinate you captured earlier with `/whereami`.
</Exercise>

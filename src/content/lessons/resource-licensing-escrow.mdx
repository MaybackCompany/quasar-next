## What to understand before you transact

<StepGroup>
  <Step num={1} title="Know open vs escrowed" outcome="You can tell what a buyer will and will not be able to edit.">
    **Open** resources ship as readable Lua: fully editable, fully copyable. **Escrowed** resources protect the core; the buyer edits the exposed config and any files the author left open, but not the protected logic. Neither is "better"; they fit different products.
  </Step>

  <Step num={2} title="As a buyer, check editing rights before you buy" outcome="No surprise that you cannot make the change you needed.">
    If you bought a script specifically to modify its core behavior, an escrowed version will block you. Read the listing: what is configurable, what is open, what is locked. Ask before buying, not after.
  </Step>

  <Step num={3} title="As a seller, decide what to protect" outcome="You expose a clean config and protect what is yours.">
    Expose a well-documented config so buyers rarely need the core. Protect the logic that is your actual product. Leaving everything open invites resale of your work; locking everything frustrates legitimate customization.
  </Step>

  <Step num={4} title="Respect licenses both ways" outcome="You only ship and run what you are licensed to.">
    Do not resell, re-leak, or strip another author's escrow. On the buy side, a license is per the terms you agreed to (often per-server). Reselling someone else's work is the fast path to a marketplace ban.
  </Step>
</StepGroup>

## How to recognize escrow in the wild

You will not always be told a script is escrowed. You have to read the signs, both in the files you receive and in the listing before you buy.

### What an escrowed resource looks like on disk

When you unzip an escrowed resource, most of it is unreadable. The core Lua files are encrypted, and opening one in a text editor shows a protected blob, not code you can follow or edit. What stays readable is whatever the author deliberately left out of protection: usually a `config.lua` (or a `locales` folder) so you can change settings without touching the core.

The author controls that line with one manifest directive. In `fxmanifest.lua` the seller lists the files to keep open with `escrow_ignore`:

```lua
fx_version 'cerulean'
game 'gta5'

shared_script 'config.lua'
client_script 'client/main.lua'
server_script 'server/main.lua'

-- Files the seller leaves OPEN and editable. Everything else is encrypted.
escrow_ignore {
    'config.lua',
    'locales/*.lua'
}

dependencies {
    'ox_lib'
}
```

When that resource is packaged for sale, FiveM's Asset Escrow system encrypts the protectable files *except* the ones named in `escrow_ignore`. You may also notice a small `.fxap` file inside an escrowed resource: that is the escrow manifest file the system writes in, and the resource needs it (plus a valid grant) to load. At runtime the protected core executes normally on the server, so the script works as intended, but the buyer never sees that code on disk in readable form. If you open `config.lua` and it reads like plain settings, while `server/main.lua` is an unreadable blob, that is escrow doing exactly its job.

<Callout variant="fivem" title="What escrow does not do">
  Escrow protects the source from being read and copied; it does not give the seller remote control of your server. But it is not purely offline: an escrowed resource only runs when your server's Cfx.re license key holds a valid grant for that asset, and the runtime checks that entitlement when the resource loads. If the key has no grant you get a "You lack the required entitlement" error and the resource will not start. So it is code protection plus an ownership check, not a remote kill switch the seller can flip at will.
</Callout>

<Callout variant="warning" title="Escrow protects the seller, not you">
  Escrow hides the core code from YOU as much as from anyone else. Because you cannot read an escrowed server file, you cannot verify what it does — a malicious author could hide a backdoor or an external HTTP call inside the protected core. Escrow is not a safety guarantee for the buyer. Only buy escrowed scripts from reputable authors through a legitimate marketplace (Tebex), and watch your server console and outbound connections after installing anything new.
</Callout>

### How to read a listing before you buy

A good Tebex listing tells you where the author drew the escrow line before you pay. Look for the section that splits the resource three ways:

- **Configurable** means exposed in the open `config.lua`: prices, coordinates, labels, item names. You can change these freely.
- **Open** means a full file left out of escrow that you can read and edit, sometimes a client UI file or a hook the author wants you to extend.
- **Locked** means encrypted core: the logic that is the actual product. You cannot read it, edit it, or copy it.

Match that split against the change you actually need. If the one thing you came to modify sits in the locked column, an escrowed copy will not let you do it, no matter how good the config is. When a listing does not spell this out, ask the seller which files are open before you buy.

<Callout variant="fivem" title="Escrow in one sentence">
  Escrow is a lock on a script's core code. FiveM's asset escrow lets a seller encrypt the core of a resource while leaving a config (and sometimes a few open files) for the buyer to change. You run it and you configure it, but you can't read or copy the locked core.
</Callout>

## What to understand before you transact

<StepGroup>
  <Step num={1} title="Know open vs escrowed" outcome="You can tell what a buyer will and will not be able to edit.">
    Open means readable Lua you can edit and copy in full. Escrowed means the core is locked: the buyer edits the exposed config and whatever files the author left open, but not the protected logic. Neither one is "better." They fit different products.
  </Step>

  <Step num={2} title="As a buyer, check editing rights before you buy" outcome="No surprise that you cannot make the change you needed.">
    If you bought a script specifically to modify its core behavior, an escrowed version will block you. Read the listing: what is configurable, what is open, what is locked. Ask before buying, not after.
  </Step>

  <Step num={3} title="As a seller, decide what to protect" outcome="You expose a clean config and protect what is yours.">
    Expose a well-documented config so buyers rarely need the core. Protect the logic that is your actual product. Leaving everything open invites resale of your work; locking everything frustrates legitimate customization.
  </Step>

  <Step num={4} title="Respect licenses both ways" outcome="You only ship and run what you are licensed to.">
    Do not resell, re-leak, or strip another author's escrow. On the buy side, a license is per the terms you agreed to (often per-server). Reselling someone else's work is the fast path to a marketplace ban.
  </Step>
</StepGroup>

## How to recognize escrow in the wild

You will not always be told a script is escrowed. You have to read the signs, both in the files you receive and in the listing before you buy.

### What an escrowed resource looks like on disk

Unzip an escrowed resource and most of it is gibberish. The core Lua files are encrypted, so opening one in a text editor shows a scrambled blob, not code you can read or edit. The only readable parts are the files the author chose to leave out: usually a `config.lua` (or a `locales` folder) so you can change settings without touching the core.

The author draws that open-vs-locked line with one manifest directive. In `fxmanifest.lua` (the file that tells the server what a resource is and what it loads) the seller lists the files to keep open with `escrow_ignore`:

```lua
fx_version 'cerulean'
game 'gta5'

shared_script 'config.lua'
client_script 'client/main.lua'
server_script 'server/main.lua'

-- Files the seller leaves OPEN and editable. Everything else is encrypted.
escrow_ignore {
    'config.lua',
    'locales/*.lua'
}

dependencies {
    'ox_lib'
}
```

When the resource is packaged for sale, FiveM's Asset Escrow system encrypts every protectable file *except* the ones named in `escrow_ignore`. You may also spot a small `.fxap` file inside: that is the escrow manifest the system adds, and the resource needs it (plus a valid grant) to load. At runtime the locked core runs normally on the server, so the script works exactly as intended, but you never see that code on disk in readable form. So if `config.lua` reads like plain settings while `server/main.lua` is a scrambled blob, that is escrow doing its job, not a broken download.

<Callout variant="fivem" title="What escrow does not do">
  Escrow is not a remote kill switch. It hides the source from being read and copied; it does not hand the seller control of your server. But it isn't fully offline either: an escrowed resource only runs when your server's Cfx.re license key holds a valid grant (proof you own the asset), and the runtime checks that grant every time the resource loads. No grant means a "You lack the required entitlement" error and the resource won't start. So think of it as code protection plus an ownership check, not a switch the seller can flip whenever they like.
</Callout>

<Callout variant="warning" title="Escrow protects the seller, not you">
  Escrow hides the core from YOU just as much as from everyone else. You can't read an escrowed server file, so you can't verify what it does. A shady author could bury a backdoor or a secret HTTP call in the locked core and you'd never see it. Escrow is not a safety guarantee for the buyer. Only buy escrowed scripts from reputable authors through a legitimate marketplace (Tebex), and watch your server console and outbound connections after you install anything new.
</Callout>

### How to read a listing before you buy

A good Tebex listing tells you where the escrow line sits before you pay a cent. Look for the section that splits the resource three ways:

- **Configurable** means exposed in the open `config.lua`: prices, coordinates, labels, item names. You can change these freely.
- **Open** means a full file left out of escrow that you can read and edit, sometimes a client UI file or a hook the author wants you to extend.
- **Locked** means encrypted core: the logic that is the actual product. You cannot read it, edit it, or copy it.

Match that split against the change you actually need. If the one thing you came to modify sits in the locked column, an escrowed copy will not let you do it, no matter how good the config is. When a listing does not spell this out, ask the seller which files are open before you buy.

  Spawning is easy. Cleaning up is the part that separates scripts that work from scripts that destroy servers. If you spawn entities and never delete them, the world fills with ghost objects, entity counts climb, and the server lags or crashes. This lesson teaches the three moments cleanup has to happen, and why one of them cannot be done from the client at all.

    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

client_script 'client.lua'

dependencies {
    '/onesync'
}
    ```

    The `dependencies { '/onesync' }` line tells the server to refuse to start this resource unless OneSync (state awareness) is on.   </Step>

  <Step num={4} title="Write the client code" outcome="The client spawns a networked box and reports it to the server.">
    Open `client.lua` and paste this:

    ```lua
local objects = {}

RegisterCommand('spawnbox', function()
    local model = joaat('prop_boxpile_07d')
    if not IsModelInCdimage(model) or not IsModelValid(model) then return end

    RequestModel(model)
    local deadline = GetGameTimer() + 5000
    while not HasModelLoaded(model) do
        if GetGameTimer() > deadline then
            print('[qu_cleanup_discipline] model load timed out')
            return
        end
        Wait(0)
    end

    local coords = GetEntityCoords(PlayerPedId())
    local object = CreateObject(model, coords.x + 2.0, coords.y, coords.z, true, true, false)
    objects[object] = true
    SetModelAsNoLongerNeeded(model)

    local netId = NetworkGetNetworkIdFromEntity(object)
    TriggerServerEvent('qu_cleanup:registerBox', netId)
    print('[qu_cleanup_discipline] spawned object ' .. object .. ' netId ' .. netId)

    SetTimeout(30000, function()
        if DoesEntityExist(object) then
            DeleteEntity(object)
            objects[object] = nil
            print('[qu_cleanup_discipline] timeout deleted object ' .. object)
        end
    end)
end, false)

RegisterCommand('clearboxes', function()
    for object in pairs(objects) do
        if DoesEntityExist(object) then DeleteEntity(object) end
        objects[object] = nil
    end
    TriggerServerEvent('qu_cleanup:clearMine')
    print('[qu_cleanup_discipline] registry cleared')
end, false)

AddEventHandler('onResourceStop', function(name)
    if name == GetCurrentResourceName() then
        for object in pairs(objects) do
            if DoesEntityExist(object) then DeleteEntity(object) end
        end
    end
end)
    ```
  </Step>

  <Step num={5} title="Write the server code" outcome="The server tracks each player's boxes and deletes them on drop.">
    Open `server.lua` and paste this:

    ```lua
local BOX_MODEL = joaat('prop_boxpile_07d')
local MAX_BOXES_PER_PLAYER = 10
local boxesByPlayer = {}

local function countBoxes(owned)
    local count = 0
    for _ in pairs(owned or {}) do count = count + 1 end
    return count
end

local function deleteTracked(src, reason)
    local owned = boxesByPlayer[src]
    if not owned then return 0 end

    local deleted = 0
    for netId in pairs(owned) do
        local entity = NetworkGetEntityFromNetworkId(netId)
        if entity ~= 0 and DoesEntityExist(entity) then
            DeleteEntity(entity)
            deleted = deleted + 1
        end
    end

    boxesByPlayer[src] = nil
    print(('[qu_cleanup_discipline] %s player %s, deleted %s boxes'):format(reason, src, deleted))
    return deleted
end

RegisterNetEvent('qu_cleanup:registerBox', function(netId)
    local src = source
    netId = tonumber(netId)
    if not netId then return end

    local entity = NetworkGetEntityFromNetworkId(netId)
    if entity == 0 or not DoesEntityExist(entity) then return end
    if NetworkGetEntityOwner(entity) ~= src then return end
    if GetEntityModel(entity) ~= BOX_MODEL then return end

    boxesByPlayer[src] = boxesByPlayer[src] or {}
    if countBoxes(boxesByPlayer[src]) >= MAX_BOXES_PER_PLAYER then
        DeleteEntity(entity)
        return
    end

    boxesByPlayer[src][netId] = true
    print('[qu_cleanup_discipline] tracking netId ' .. netId .. ' for player ' .. src)

    SetTimeout(30000, function()
        local owned = boxesByPlayer[src]
        if not owned or not owned[netId] then return end

        local trackedEntity = NetworkGetEntityFromNetworkId(netId)
        if trackedEntity ~= 0 and DoesEntityExist(trackedEntity) then
            DeleteEntity(trackedEntity)
        end
        owned[netId] = nil
        if next(owned) == nil then boxesByPlayer[src] = nil end
    end)
end)

RegisterNetEvent('qu_cleanup:clearMine', function()
    deleteTracked(source, 'cleared')
end)

AddEventHandler('playerDropped', function()
    local src = source
    deleteTracked(src, 'dropped')
end)
    ```
  </Step>

## How it works

You spawned a box and cleared it, and a third path is waiting on a timer. Now take the code apart so cleanup becomes a habit you reach for without thinking. The whole lesson is one idea: every entity you create is a debt, and you pay it back in three places. Miss any one of them and the debt piles up as ghost objects.

### Why a leak is dangerous, and how to see it

An entity leak is not abstract. Every networked box you leave alive consumes an entity slot and may need to be synchronized to players for as long as it exists. Repeating `/spawnbox` without cleanup steadily adds work and memory. On a busy server, enough leaked entities contribute to rising frame time, streaming problems, and pool exhaustion.

You can prove the lesson without relying on an undocumented console command. The server prints once when it accepts a box and again when it clears or drops a player's tracked set. Spawn several boxes, run `/clearboxes`, and confirm the objects disappear and the server reports the cleanup. For production profiling, use the documented client resource monitor and server profiler, then correlate any growth with your resource's own registry and logs.

### The registry is the only memory you have

```lua
local objects = {}
-- ...
objects[object] = true
```

You cannot delete what you cannot name. `CreateObject` hands you back one number, the entity handle, and that number is the only way to refer to the box later. The registry is a table whose keys are those handles. Storing `objects[object] = true` means the value does not matter, only the key, so the table behaves like a set of handles. When it is time to clean up you walk the table with `pairs` and you have every box you ever made in one place. Without this table, the moment the spawn function returns and the handle goes out of scope, the box is unreachable. It still exists in the world. You just lost the receipt.

### DoesEntityExist guards every delete

```lua
if DoesEntityExist(object) then DeleteEntity(object) end
```

A handle in your registry is a claim that a box existed at spawn time, not a promise it still exists now. Another resource or the timeout may already have deleted it. `DoesEntityExist` asks whether the handle still points at a real entity before you act on it. A stale handle makes cleanup fail or produce misleading results. The guard is cheap and makes the operation explicit: verify immediately before deleting.

### Path one: cleanup on resource stop

```lua
AddEventHandler('onResourceStop', function(name)
    if name == GetCurrentResourceName() then
        for object in pairs(objects) do
            if DoesEntityExist(object) then DeleteEntity(object) end
        end
    end
end)
```

`onResourceStop` fires for every resource that stops anywhere on the server, not just yours. That is why the `name` argument and the `GetCurrentResourceName()` comparison are not optional. Without that filter, your handler would run its cleanup loop every time any unrelated resource on the box restarts, which is constant on a live server. The guard narrows it to the one moment you care about: your own resource shutting down. When you `restart qu_cleanup_discipline`, the stop half fires first, this handler runs, and every box you spawned is deleted before the new copy loads. That is what stops ghost objects surviving a restart, the single most common cleanup failure in the wild.

### Path two: timed auto-delete

```lua
SetTimeout(30000, function()
    if DoesEntityExist(object) then
        DeleteEntity(object)
        objects[object] = nil
        print('[qu_cleanup_discipline] timeout deleted object ' .. object)
    end
end)
```

`SetTimeout` schedules a function to run once after a delay in milliseconds, so `30000` is thirty seconds. The client deletes quickly when it is still alive. The server schedules its own matching timeout after it validates the entity, so cleanup does not depend on the client staying connected or behaving honestly. Both paths check existence first, making a second cleanup harmless when the first one already succeeded.

### Path three: cleanup on player drop, and why it lives on the server

This is the path the client physically cannot do, and the reason this lesson has a `server.lua` at all. Think it through. The cleanup handlers above all live in `client.lua`. They run inside the player's own game. If that player crashes to desktop, loses internet, or force-quits, their game stops executing instantly. No `onResourceStop`, no timeout callback, nothing client-side runs, because the thing that would run it is dead. The boxes they spawned are networked, so they stay alive on the server with no owner to clean them. That is a guaranteed leak that grows every time someone rage-quits.

The server is the one party still alive when a client disappears, so the drop cleanup has to live there. The client reports each box upward as it spawns:

```lua
local netId = NetworkGetNetworkIdFromEntity(object)
TriggerServerEvent('qu_cleanup:registerBox', netId)
```

`NetworkGetNetworkIdFromEntity` converts the local entity handle into a network id that refers to the same entity across machines during that entity's lifetime. Network ids can be reused after an entity is removed, which is another reason to resolve and validate immediately. A raw entity handle is local to one runtime, so you send the net id, not the handle. `TriggerServerEvent` crosses from client to server, and inside the matching handler `source` is the player who sent it:

```lua
RegisterNetEvent('qu_cleanup:registerBox', function(netId)
    local src = source
    netId = tonumber(netId)
    if not netId then return end

    local entity = NetworkGetEntityFromNetworkId(netId)
    if entity == 0 or not DoesEntityExist(entity) then return end
    if NetworkGetEntityOwner(entity) ~= src then return end
    if GetEntityModel(entity) ~= BOX_MODEL then return end

    boxesByPlayer[src] = boxesByPlayer[src] or {}
    boxesByPlayer[src][netId] = true
end)
```

The server must not store a net id merely because a client sent it. A modified client could submit another player's entity id and trick a naive cleanup handler into deleting it later. This handler resolves the id, confirms the entity exists, checks that the sender currently owns it, checks the expected box model, and caps each player at ten tracked boxes. Cfx.re's event-security guidance is the rule behind those checks: client event data is an untrusted request.

`boxesByPlayer` is a registry of registries: one inner table of validated net ids per player server id. `boxesByPlayer[src] = boxesByPlayer[src] or {}` creates the player's bucket on the first accepted box. When `playerDropped` fires, the server has a checked set to delete:

```lua
for netId in pairs(owned) do
    local entity = NetworkGetEntityFromNetworkId(netId)
    if entity ~= 0 and DoesEntityExist(entity) then
        DeleteEntity(entity)
    end
end
```

`NetworkGetEntityFromNetworkId` is the reverse of what the client did: it turns the net id back into a server-side entity handle. This resolution only works because OneSync (state awareness) is on, which is why the manifest depends on `/onesync`. It returns `0` when no entity matches, for example because the box was already cleaned by the timeout, so you check `entity ~= 0` before the existence guard. Delete each one, drop the player's whole bucket, and the server returns to baseline. A crashed player no longer leaks.

<Checkpoint question="Why can the player-drop cleanup not just live in client.lua next to the other two handlers?">
  Because client handlers only run while that player's game is running. A crash or hard disconnect stops its code immediately. The server remains alive, so it validates each reported net id while the player is connected, stores only entities matching the expected owner and model, and deletes that set when `playerDropped` fires. Its timeout is also the fallback when the client never runs local cleanup.
</Checkpoint>

### When you reach for this pattern

Any time a resource spawns world entities, all three paths apply, and you wire them once at the start rather than bolting them on after the first leak report:

- **Resource stop** is non-negotiable for anything you develop iteratively. You restart the resource dozens of times a day; without it every restart leaves a fresh pile of ghosts.
- **A timeout or interaction end** fits anything temporary: a dropped item that should despawn, a job prop that vanishes when the job ends, an effect entity tied to an animation.
- **Player drop** is the one beginners forget and the one that hurts most on a busy server, because disconnects are constant. If your entities are networked and tied to a player, the server must clean them on drop, full stop.

<Callout variant="fivem" title="How FiveM voice works">
  FiveM includes an internal Mumble voice system. The community resource `pma-voice`, maintained in the `AvarianKnight/pma-voice` repository, configures proximity, radio, and call channels on top of it. A normal setup does not require a separate TeamSpeak or Mumble server, though pma-voice also supports an explicitly configured external Mumble address. Network loss can hurt voice, but so can the wrong input device, conflicting voice resources, or incorrect convars, so diagnose in order rather than assuming one cause.
</Callout>

## Get it working

<StepGroup>
  <Step num={1} title="Install pma-voice by itself" outcome="One voice resource starts with no errors.">
    Download the release from the official `AvarianKnight/pma-voice` repository and place it in `resources` with the folder name `pma-voice`. It uses FiveM's internal Mumble support; **do not install or run `mumble-voip` beside it**. The pma-voice README says it is incompatible with other voice systems, including vMenu voice unless that feature is disabled.

    ```cfg
    ensure pma-voice
    ```

    Restart and watch the console. If another voice resource is running, stop it and remove its `ensure` line before troubleshooting pma-voice.
  </Step>

  <Step num={2} title="Set proximity and mode in config" outcome="Talk, whisper, and shout ranges behave as intended.">
    pma-voice's supported configuration is through convars in `server.cfg`. Start with the defaults and override only values you intentionally want to change:

    ```cfg
    # 1 = Whisper, 2 = Normal, 3 = Shouting
    setr voice_defaultVoiceMode 2
    setr voice_useNativeAudio false
    setr voice_enableRadioAnim false
    ```

    These lines do not set three distance values. `voice_defaultVoiceMode` chooses the initial mode. `voice_useNativeAudio` changes the audio path and is required for submixes. `voice_enableRadioAnim` only controls the radio animation. The project's current ranges live in its shared code; changing them is a maintained fork, not a normal convar tweak. Follow the README for the release you installed.
  </Step>

  <Step num={3} title="Confirm mic permission and device" outcome="Your voice is captured and transmitted.">
    In FiveM settings, set the correct input device and push-to-talk key, and allow microphone access at the OS level. A muted OS mic looks exactly like a broken server.
  </Step>

  <Step num={4} title="Test with a second player" outcome="Two players hear each other by proximity.">
    Voice is hard to test alone. Bring a second client (or a friend), stand close, and confirm proximity falloff as you walk apart.
  </Step>
</StepGroup>

## How it works

Two facts explain almost every voice problem you will hit, and both are worth holding in your head before you start tuning anything.

**Voice uses FiveM's Mumble integration.** In the default setup, pma-voice uses the internal voice server. It can also be pointed at an external address and port with documented convars, so it is inaccurate to assume every installation uses one identical network path. If audio cuts out, check the client device and push-to-talk, confirm only one voice resource controls the natives, inspect pma-voice's debug output, and then check latency, packet loss, and server health.

**Proximity has three modes.** The default mode is selected by `voice_defaultVoiceMode`, and players cycle modes with the configured key (F11 by default for new players). First confirm the player's current Whisper/Normal/Shouting mode. Do not tell operators to edit a nonexistent `config.lua`; use the README's convars, and treat changes to the resource's shared code as a fork you must maintain.

## Troubleshooting order

  items={[
    { symptom: "Nobody can hear anyone", fix: "Confirm pma-voice started, disable every other voice system, then check OS microphone permission and the FiveM input/output devices." },

<Callout variant="fivem" title="How FiveM voice works">
  FiveM has a built-in voice system that rides the same UDP connection as the game itself. A resource like `pma-voice` (a community resource, now maintained under the Overextended org) configures proximity, radio, and phone channels on top of that built-in system. You do not need a separate TeamSpeak or Mumble server for proximity voice in 2026; the in-game system handles it. Because voice and game traffic share one path, anything that hurts the connection (packet loss, an overloaded server thread) hurts voice too, which is why robotic or cutting voice is almost always a network symptom rather than a config one.
</Callout>

## Get it working

<StepGroup>
  <Step num={1} title="Install pma-voice and its dependency" outcome="The voice resource starts with no missing-dependency errors.">
    Download `pma-voice` and drop it in your `resources` folder. It depends on `mumble-voip`, which ships with the server artifacts, so make sure that resource exists and is not deleted. Add the start line to `server.cfg`:

    ```cfg
    ensure pma-voice
    ```

    Restart and watch the console. A missing-dependency error here almost always means `mumble-voip` was removed from your artifacts or renamed.
  </Step>

  <Step num={2} title="Set proximity and mode in config" outcome="Talk, whisper, and shout ranges behave as intended.">
    pma-voice reads its settings from convars (set in `server.cfg`) and from `config.lua`. The three you touch first set the proximity ranges and the audio mode. Defaults work; tune ranges only after voice works:

    ```cfg
    # Proximity ranges in metres: whisper / normal / shout
    setr voice_defaultVoiceMode 2
    setr voice_useNativeAudio false
    setr voice_enableRadioAnim false
    ```

    The talk, whisper, and shout distances themselves live in `config.lua` as `Cfg.voiceModes` (three entries: a range and a label each). Leave `voice_useNativeAudio` at `false` for standard mumble positional audio; flip it to `true` only if you specifically want OneSync native audio and have tested it.
  </Step>

  <Step num={3} title="Confirm mic permission and device" outcome="Your voice is captured and transmitted.">
    In FiveM settings, set the correct input device and push-to-talk key, and allow microphone access at the OS level. A muted OS mic looks exactly like a broken server.
  </Step>

  <Step num={4} title="Test with a second player" outcome="Two players hear each other by proximity.">
    Voice is hard to test alone. Bring a second client (or a friend), stand close, and confirm proximity falloff as you walk apart.
  </Step>
</StepGroup>

## How it works

Two facts explain almost every voice problem you will hit, and both are worth holding in your head before you start tuning anything.

**Voice rides the game connection.** FiveM does not open a separate voice socket. Speech travels over the same UDP path as position updates, entity sync, and everything else the game sends. That single fact is why voice quality tracks connection quality so tightly. When a player's voice goes robotic, stutters, or drops mid-sentence, the cause is rarely the voice config; it is packet loss on that player's link or a server hitch (a resource hogging a thread, visible in resmon as a spike). Fixing it means looking at ping and server load, not at proximity ranges. Routing buckets, used to split players into separate game instances, also split voice, and buckets require OneSync to be enabled, so if voice works in the main world but not in an instanced job, that is the bucket boundary doing its job.

**The proximity ranges are just numbers in config.** pma-voice does not invent a voice mode per player; it reads a fixed list of modes from `Cfg.voiceModes` in `config.lua` and lets the player cycle between them. Each entry is a distance and a label, for example a short whisper range, a default talk range, and a longer shout range. When a player complains that nobody can hear them from across the room, they are simply on the whisper mode, not on a broken setup. Teaching players that the modes are a deliberate cycle (the keybind switches whisper to talk to shout) prevents most of the "voice is broken" reports that are really just mode confusion. The convars in `server.cfg` set the defaults and the audio mode; the per-mode distances live in `config.lua`. Change the numbers there, restart, and the ranges change for everyone.

## Build it

<StepGroup>
  <Step num={1} title="Confirm ox_lib is installed and started" outcome="The lib global will exist when your resource loads.">
    Your resource borrows the `lib` global from ox_lib, so ox_lib has to be running before your resource starts. In `server.cfg`, ox_lib must be ensured above your lesson resource:

    ```text
    ensure ox_lib
    ensure qu_ox_lib_notifications_ui
    ```

    If you do not have ox_lib yet, download the latest release from the CommunityOx GitHub (github.com/CommunityOx/ox_lib/releases) and drop the `ox_lib` folder into `resources`. Do not rename it: the `@ox_lib/init.lua` path in the next step depends on the folder being called exactly `ox_lib`.
  </Step>

  <Step num={2} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_ox_lib_notifications_ui
    ```
  </Step>

  <Step num={3} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_ox_lib_notifications_ui/
  fxmanifest.lua
  client.lua
    ```
  </Step>

  <Step num={4} title="Write fxmanifest.lua" outcome="FiveM knows which files to load, and in what order.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

shared_script '@ox_lib/init.lua'
client_script 'client.lua'

dependencies {
    'ox_lib'
}
    ```

    Older guides write the dependency as `dependency 'ox_lib'` (singular). That form is still tolerated, but the documented modern form is the plural table `dependencies { 'ox_lib' }`, so that is what you use here.
  </Step>

  <Step num={5} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `client.lua` and paste this:

    ```lua
lib.registerContext({
    id = 'qu_ox_lib_notifications_ui_menu',
    title = 'qu_ox_lib_notifications_ui',
    options = {
        { title = 'Ask name', event = 'qu_ox_lib_notifications_ui:ask' }
    }
})

RegisterCommand('oxdemo', function()
    lib.notify({
        title = 'qu_ox_lib_notifications_ui',
        description = 'Notification works',
        type = 'success'
    })

    lib.showContext('qu_ox_lib_notifications_ui_menu')
end, false)

AddEventHandler('qu_ox_lib_notifications_ui:ask', function()
    local input = lib.inputDialog('Save a name', { 'Name' })
    if not input then return end

    lib.progressCircle({ duration = 1000, label = 'Saving' })

    print('[qu_ox_lib_notifications_ui] saved ' .. input[1])
end)
    ```
  </Step>

  <Step num={6} title="Start and test it" outcome="The expected proof appears in the correct console.">
    With the two `ensure` lines from Step 1 already in `server.cfg`, save the file, then run:

    ```text
    restart qu_ox_lib_notifications_ui
    ```

    Join the server, then type this in the chat box (press T):

    ```text
    /oxdemo
    ```

    A green notification appears, a menu opens with one option, choosing it asks for a name, a progress circle runs for one second, and then the line below prints to your F8 client console. Type `Mike` when the dialog asks.

    <ExpectedOutput>
      F8: [qu_ox_lib_notifications_ui] saved Mike
    </ExpectedOutput>
  </Step>
</StepGroup>

## How it works

You ran it and the four widgets fired in sequence. Now take the code apart so the next time you want a notification or a menu you are deciding which ox_lib call to use, not copying a block you do not understand. The whole resource is two pieces: a manifest that loads the library, and a client file that calls into it.

### Why the manifest line `@ox_lib/init.lua` comes first

```lua
shared_script '@ox_lib/init.lua'
client_script 'client.lua'
```

The `@` prefix means "reach into another resource and load this file from it." So `@ox_lib/init.lua` runs ox_lib's own init file inside your resource. That init file is what creates the `lib` global you call everywhere. Without this line, `lib` does not exist in your resource and the first `lib.notify` blows up with `attempt to index a nil value (global 'lib')`.

Order matters here. Manifest entries load top to bottom, so `@ox_lib/init.lua` is listed before `client.lua` on purpose: the library has to be in place before your code tries to use it. It is declared as a `shared_script` rather than a `client_script` because ox_lib runs on both the client and the server, and loading its init on both sides is what lets the same `lib` table work wherever you need it.

The `dependencies { 'ox_lib' }` line is a second, coarser guarantee. It tells the server not to start `qu_ox_lib_notifications_ui` at all until the `ox_lib` resource is already started. The `@` path handles loading the file; the dependency handles resource start order. You want both.

<Checkpoint question="The console says attempt to index a nil value (global 'lib') the moment you run /oxdemo. What is wrong?">
  The `lib` global was never created in your resource, which almost always means the `shared_script '@ox_lib/init.lua'` line is missing, misspelled, or placed below `client.lua` in the manifest. Check that the line is present, that the folder is named exactly `ox_lib` (the `@` path is case and name sensitive), and that ox_lib is ensured above your resource in `server.cfg`. Fix the manifest, save, then run `restart qu_ox_lib_notifications_ui`.
</Checkpoint>

### lib.notify and the type field

```lua
lib.notify({
    title = 'qu_ox_lib_notifications_ui',
    description = 'Notification works',
    type = 'success'
})
```

`lib.notify` takes one table of options, not a list of separate arguments. That table style is how almost every ox_lib call works: you pass a single `{ }` and set named keys inside it. Here `title` is the bold top line, `description` is the body text, and `type` controls the colour and icon. The common values are `'success'` (green), `'error'` (red), and `'inform'` (neutral blue). You chose `'success'` so the proof reads as a green toast. There are more keys available, such as `duration`, `position`, and `icon`. The full list lives in the ox_lib docs linked at the bottom of this lesson, and that is the page you will return to most.

### Why the context option uses `event=` and not the handler directly

```lua
lib.registerContext({
    id = 'qu_ox_lib_notifications_ui_menu',
    title = 'qu_ox_lib_notifications_ui',
    options = {
        { title = 'Ask name', event = 'qu_ox_lib_notifications_ui:ask' }
    }
})

lib.showContext('qu_ox_lib_notifications_ui_menu')
```

A context menu is built in two moves. `lib.registerContext` defines the menu and stores it under an `id`, and in this resource it runs once at the top of `client.lua` when the resource loads. `lib.showContext` then opens the menu by that same `id`, and in this resource it lives inside the `/oxdemo` command handler so the menu opens each time you run the command. Splitting definition from display means you register the menu once on load and reopen it as many times as you like.

Each entry in `options` is one row in the menu. Our row has a `title` and an `event`. When the player clicks it, ox_lib fires a local client event, which is why the code listens with `AddEventHandler`. Nothing crosses the network, so `RegisterNetEvent` would unnecessarily expose a local UI action as network-callable. The static context is registered once when the resource loads and the command only opens it, matching the ox_lib guidance not to re-register static menus repeatedly.

`event=` is one of two ways to wire a menu row. The other is `onSelect`, where you hand the option an inline function instead of an event name:

```lua
{ title = 'Ask name', onSelect = function() askForName() end }
```

`onSelect` is convenient when the action is small and local to this file. `event=` is the better choice when the same action needs to be triggerable from somewhere else too, since any code can `TriggerEvent('qu_ox_lib_notifications_ui:ask')` to reuse it. This lesson uses `event=` so you see the wiring explicitly.

### Why inputDialog returns a table and how the cancel case is handled

```lua
local input = lib.inputDialog('Save a name', { 'Name' })
if not input then return end
```

`lib.inputDialog` opens a form with a title and a list of fields. The second argument `{ 'Name' }` is a list of one field, so the dialog shows one text box labelled `Name`. Because a dialog can have many fields, ox_lib gives you the answers back as a table, indexed in the order the fields were declared. With one field, your answer is `input[1]`. If you had asked for three things, you would read `input[1]`, `input[2]`, and `input[3]`. That is why the print joins `input[1]` and not `input`: `input` is the whole table, `input[1]` is the first answer inside it.

The `if not input then return end` line handles the player pressing Escape or closing the dialog without submitting. In that case `lib.inputDialog` returns `nil` instead of a table. If you skipped this guard and went straight to `input[1]`, indexing `nil` would throw `attempt to index a nil value (local 'input')` every time someone cancels. Reading the cancel path first, before you touch the data, is the habit to keep: treat a cancelled dialog as a normal outcome, not an error.

### Why the progress circle runs after input is collected

```lua
lib.progressCircle({ duration = 1000, label = 'Saving' })
print('[qu_ox_lib_notifications_ui] saved ' .. input[1])
```

`lib.progressCircle` shows a circular timer and, importantly, it blocks until the timer finishes. `duration = 1000` means it waits one thousand milliseconds, one second, before the code continues. So the `print` on the next line does not run until the circle completes. This is why the order on screen is exactly: dialog closes, circle spins for a second, line prints. If you put the progress circle before `lib.inputDialog`, the player would watch a bar fill before you even asked them anything, which makes no sense. The pattern is collect the input first, then show progress while you pretend to do the work, then report the result. In a real script the gap during the progress bar is where you would actually save to a database or call the server.

<Callout variant="warning" title="Anything typed into inputDialog is untrusted client input">
  The value you read from `input[1]` comes from the player's own machine, so a player can send anything at all, not just what your dialog seems to allow. In this lesson it only goes to a local `print`, which is safe. The moment a real script sends that value to the server, the server must treat it as untrusted: check the type and length, reject what does not fit, and when it reaches a database use a parameterized query (the `?` placeholder form) rather than building the SQL string by gluing the value in. Trusting raw client input server-side is how servers get money duped, items spawned, and databases broken.
</Callout>

<Checkpoint question="A teammate moves the lib.progressCircle call above the lib.inputDialog call. What does the player experience, and why?">
  The progress circle runs first, blocking for a full second while the screen shows a spinner, and only after it finishes does the input dialog appear asking for the name. The behaviour is backwards because `lib.progressCircle` blocks until its `duration` elapses, so whatever you place after it waits. You want the bar to represent work done on data you already have, so the input has to be collected before the circle, not after.
</Checkpoint>

### When you reach for this pattern

These four calls cover most of the UI a gameplay script needs, and you will combine them constantly:

- **lib.notify** for any quick confirmation or rejection. Bought an item, action failed, cooldown active.
- **lib.registerContext plus lib.showContext** whenever a player needs to pick from a short list. Shop categories, vehicle options, a job menu.
- **lib.inputDialog** when you need typed or selected values back. Naming something, entering an amount, choosing from a dropdown.
- **lib.progressCircle** to gate an action behind a visible timer. Lockpicking, crafting, repairing, anything that should not be instant.

The reason to use ox_lib instead of building these by hand, the way earlier lessons built things from scratch, is consistency and time. Every script on your server using the same library means one visual style and one set of behaviours your players already understand, and you write logic instead of UI plumbing.

## Try it yourself

<Exercise>
  Extend the dialog to collect two fields and gate the save behind a longer timer. Ask for both a name and an amount, then report both:

  ```lua
  AddEventHandler('qu_ox_lib_notifications_ui:ask', function()
      local input = lib.inputDialog('Save a record', { 'Name', 'Amount' })
      if not input then return end

      lib.progressCircle({ duration = 2500, label = 'Saving record' })

      lib.notify({
          title = 'Saved',
          description = input[1] .. ' set to ' .. input[2],
          type = 'success'
      })
      print('[qu_ox_lib_notifications_ui] saved ' .. input[1] .. ' = ' .. input[2])
  end)
  ```

  Restart `qu_ox_lib_notifications_ui`, run `/oxdemo`, choose the menu row, and enter a name and an amount. Confirm the progress circle now runs for 2.5 seconds, a second notification reports both values, and F8 prints a line like `[qu_ox_lib_notifications_ui] saved Mike = 500`. You just read a second field as `input[2]`, proving the dialog hands answers back in field order.
</Exercise>

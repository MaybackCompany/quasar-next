## How it works

You typed one command and got back one number that the server owns. That is the whole point of a callback: the client asks a question, the request travels to the server, the server computes an answer, and that exact answer comes back to the same line of client code that asked. Now take the round trip apart so you can build your own.

### The server registers an answer

```lua
lib.callback.register('qu_callbacks:getBalance', function(src)
    print('[qu_callbacks] server answering ' .. GetPlayerName(src))
    return 500
end)
```

`lib.callback.register` does on the server what `RegisterCommand` did for commands: it stores a named handler now and runs it later when something asks for it. The first argument, `'qu_callbacks:getBalance'`, is the name of the question. The `resource:thing` shape is a convention, not a rule, but it keeps names from colliding when twenty resources all register callbacks.

The second argument is the function that produces the answer. Two things make it different from a normal event handler:

- `src` is passed in for you automatically. `ox_lib` puts the server id of the player who asked as the first parameter, so you always know who is asking without reaching for the global `source`. Here you use it to look up that specific player's name.
- Whatever you `return` becomes the answer. A plain `TriggerClientEvent` sends a fire-and-forget message and returns nothing meaningful. A callback handler's `return` value is shipped back to the client as the result of its question. Return `500` here and the client receives `500`. In a real script you would replace `return 500` with a database read or a framework call like `return Player(src).money`.

This is the server side of trust. The balance is computed on the server, where the player cannot touch it. The client never gets to decide its own balance, it only gets to ask.

### The client asks and waits

```lua
RegisterCommand('mybalance', function()
    local balance = lib.callback.await('qu_callbacks:getBalance', false)
    print('[qu_callbacks] my balance is ' .. balance)
end, false)
```

`lib.callback.await` is where the magic lives. Read the middle line as a single sentence: "ask the server the `qu_callbacks:getBalance` question, then pause right here until the answer comes back, and put that answer in `balance`."

The function blocks. The line that prints does not run until the server's `return` has travelled all the way back across the network. This is why callbacks read like normal top-to-bottom code even though a network round trip is happening in the middle. You write it as if the answer is just sitting there, and `ox_lib` handles the waiting for you.

That pause is only possible because a command handler runs inside a coroutine, a FiveM thread that is allowed to yield and resume. `await` yields the thread while it waits and resumes it when the answer arrives. If you ever call `lib.callback.await` from code that is not inside a thread, you will get a "cannot resume" style error. Inside a `RegisterCommand` handler, a `CreateThread`, or an event handler, you are safe.

#### What the second argument actually does

```lua
lib.callback.await('qu_callbacks:getBalance', false)
```

That `false` is the most copied and least understood value in the whole lesson. It is the `delay`, in milliseconds: a re-trigger cooldown, not a response timeout. It controls the minimum time that must pass before the same callback name can be triggered again from this client.

- Pass a number, like `5000`, and you set a five second cooldown on this callback name. If you try to trigger `qu_callbacks:getBalance` again within that window, the new request is silently dropped, no answer comes back, and you do not even get a `nil`. Use this to rate-limit a callback a player could spam.
- Pass `false` and you set no cooldown. Every call is allowed through immediately. That is the right choice here, and the common one, because most callbacks do not need to be rate-limited.

Note what `delay` is not: it is not how long `await` waits for the answer. The response timeout is a separate global convar, covered below. If you want to throttle a spammable callback, pick a cooldown like `5000`; otherwise pass `false` and move on.

You should still guard the result before you trust it, because the server handler can return `nil`:

```lua
local balance = lib.callback.await('qu_callbacks:getBalance', false)
if balance == nil then
    print('[qu_callbacks] server returned no balance')
    return
end
print('[qu_callbacks] my balance is ' .. balance)
```

The real response timeout lives somewhere else entirely. It is a global convar named `ox:callbackTimeout`, set once for the whole server, not per call. Its default is `300000` (five minutes). You change it in `server.cfg` with a single line:

```text
setr ox:callbackTimeout 10000
```

When that limit is hit, `await` does not hand you `nil`, it raises an error like `callback event 'qu_callbacks:getBalance' timed out`. So a timeout surfaces as a thrown error you can see in the console, while a `nil` means the server handler ran but returned nothing useful.

<Checkpoint question="Why does the print line wait for the server instead of running immediately, and what would make balance come back as nil?">
  `lib.callback.await` blocks the handler's coroutine at that line. It yields the thread, sends the request to the server, and only resumes, letting the next line run, once the server's `return` value has arrived back. So the print cannot run with an empty `balance`, it runs with the real answer. `balance` comes back `nil` when the server handler returns `nil` or returns no value at all. The second argument, `false`, is a re-trigger cooldown, not a timeout, so it has nothing to do with `nil`. The actual response timeout is the global convar `ox:callbackTimeout`, and when it fires it raises an error rather than handing you `nil`.
</Checkpoint>

### Why this beats two raw events

You could fake a callback with the tools from the previous lessons: the client fires `TriggerServerEvent('askBalance')`, the server fires back `TriggerClientEvent('balanceAnswer', src, 500)`, and a separate `RegisterNetEvent('balanceAnswer', ...)` on the client catches it. That works, but look at what you lost.

The answer arrives in a different place from where you asked. Your "what do I do with the balance" logic now lives in the event handler, far from the command that started everything. If you fire two balance requests close together, nothing tells you which answer belongs to which request, you have to hand-match them yourself with request ids. And there is no safety net for a dropped response, so a lost answer leaves the client waiting on an event that never comes, with no way to notice.

`lib.callback.await` solves all three for free. The answer comes back to the exact line that asked, so the request and its handling stay together. Each request is correlated to its own response under the hood, so two questions never cross wires. And the global `ox:callbackTimeout` convar means a request that is never answered eventually errors out instead of hanging silently forever. That is why frameworks moved to callbacks for any client-to-server question.

### The async form, when you do not want to block

`await` is the blocking, synchronous form. There is a second form that does not freeze the thread. Instead of returning the value, you hand `lib.callback` a function that runs when the answer arrives:

```lua
RegisterCommand('mybalance_async', function()
    lib.callback('qu_callbacks:getBalance', false, function(balance)
        print('[qu_callbacks] async balance is ' .. balance)
    end)
    print('[qu_callbacks] this line runs before the answer arrives')
end, false)
```

Notice the call is `lib.callback` with no `.await`, and the last argument is a function. The request fires, the handler does not pause, the second print runs immediately, and the supplied function runs later when the server answers. Reach for the async form when you have other work to do while you wait, or when blocking the thread would freeze something time sensitive like a render loop. Reach for `await` when the next line genuinely needs the answer before it can do anything, which is most of the time. Both forms hit the same registered handler on the server, the only difference is whether the client thread waits.

## Try it yourself

<Exercise>
  Make the callback answer a real yes/no question instead of a fixed number, and pass an argument from the client. Change the server to take a price and decide whether the player can afford it:

  ```lua
  lib.callback.register('qu_callbacks:canAfford', function(src, price)
      local balance = 500
      return balance >= price
  end)
  ```

  Change the client to send a price and branch on the boolean answer. Here `5000` is the `delay`: a 5 second re-trigger cooldown on `qu_callbacks:canAfford`, not a 5 second budget for the answer. Any value after that second argument is forwarded to the server handler as an extra argument, which is how `price` reaches the server:

  ```lua
  RegisterCommand('canibuy', function(_, args)
      local price = tonumber(args[1]) or 0
      local ok = lib.callback.await('qu_callbacks:canAfford', 5000, price)
      if ok then
          print('[qu_callbacks] you can afford ' .. price)
      else
          print('[qu_callbacks] not enough money for ' .. price)
      end
  end, false)
  ```

  Restart `qu_callbacks`, then run `/canibuy 300` and `/canibuy 900` from F8. Confirm the first prints that you can afford it and the second does not. You just sent data up with the request, made the server decide, and branched on the answer, which is the real shape of a "can I buy this?" check.
</Exercise>

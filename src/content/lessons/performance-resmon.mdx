## How it works

You ran it, toggled the marker, and watched a number move in resmon. Now take the loop apart so you can tune any thread on your server, not just this one. The whole lesson lives in one idea: a thread costs CPU only during the frames it is actually awake, and `Wait` is the dial that controls how often it wakes up.

### Reading resmon

`resmon` is the client-side resource monitor. You opened it with `resmon 1` in the F8 console, and it draws a live table: one row per running resource, sorted by cost. The column you care about is the millisecond cost, the average time that resource's threads spend running per frame.

Use the colors as hints from the current client build, not universal pass/fail thresholds. Compare the same resource on the same machine at idle and while doing its intended work. If it is unexpectedly expensive, capture a Cfx profiler trace to locate the exact function instead of guessing from one average.

The critical mental model: resmon measures the **client** frame, because `DrawMarker` is a client native that runs on each player's machine. This is not the server tick. The server has its own budget and its own profiler. When you read resmon you are watching what your code costs the player's game, frame by frame, which is exactly where a runaway `Wait(0)` loop does its damage.

To watch a single resource closely, type its name into the resmon filter, or just read its row. With the marker off you will see `qu_performance_resmon` resting in green near 0.00 ms, because the thread sleeps a full second between wakeups and does nothing in between.

### Why Wait(0) is expensive and Wait(500) is nearly free

Look at the two values `sleep` can take:

```lua
local sleep = 1000
-- ...
sleep = 0
```

A FiveM thread is a loop that yields control back to the engine every time it hits `Wait`. The number you pass is how long the engine is allowed to leave the thread asleep before waking it again.

- `Wait(0)` means "wake me next frame." At 60 fps that is about 60 wakeups per second. Every wakeup runs the body, so reserve it for work such as `DrawMarker` that genuinely must happen every rendered frame.
- `Wait(1000)` means "leave me asleep for a second." The body runs about once per second. A thread that is asleep costs essentially nothing, which is why an idle resource shows 0.00 ms in resmon.

So the cost of a thread is not about how much code it contains. It is about how often that code is allowed to run. `Wait(0)` is not wrong. Drawing a marker smoothly genuinely needs every frame. The mistake is running `Wait(0)` when nothing on screen needs updating that often.

### The distance-check pattern

This is the real fix, and it is the heart of the loop:

```lua
local sleep = 1000

if markerOn then
    local coords = GetEntityCoords(PlayerPedId())
    local dist = #(coords - markerCoords)

    if dist < 20.0 then
        sleep = 0
        DrawMarker(...)
    end
end

Wait(sleep)
```

Read it as a decision the loop makes on every wakeup. It starts each pass assuming the cheapest answer: `sleep = 1000`. Only if the marker is on **and** the player is within 20 metres does it switch to `sleep = 0` and actually draw. The `#(coords - markerCoords)` expression is a vector length: it measures the straight-line distance between two points, so `dist < 20.0` is "within 20 metres."

The payoff is the cost curve you watch in resmon. Far away, the loop wakes once a second and goes back to sleep. Inside 20 metres it flips to `Wait(0)` and draws every frame. The amount saved is whatever your own before/after measurement shows.

<Checkpoint question="The marker is on, but you are standing 200 metres away. How many times per second does the loop run DrawMarker, and what does resmon show?">
  Zero times per second. Each wakeup the loop computes `dist`, finds it is far larger than 20.0, never enters the inner `if`, and leaves `sleep` at 1000. So the thread wakes about once per second, runs one subtraction and one comparison, and sleeps again. `DrawMarker` is never reached. resmon shows the resource resting in green near 0.00 ms, even though the marker is logically "on." The cost only appears once you are inside the 20 metre radius and the loop flips to `Wait(0)`.
</Checkpoint>

### CreateThread vs CreateThreadNow

You started the loop with `CreateThread`. The Cfx Lua runtime also has an immediate scheduler helper, `Citizen.CreateThreadNow`; it appears in runtime and framework internals but is not the default beginner API.

- `CreateThread(fn)` schedules the function to begin on the **next** game tick. The line returns immediately and the rest of your file keeps loading; the thread starts a frame later. This is the right default for a background loop like ours, because nothing depends on it running this instant.
- `CreateThreadNow(fn)` runs the function **immediately and synchronously**, right where you call it. It does not return until the function either finishes or hits its first `Wait`. You reach for this when later code in the same file depends on setup the thread performs, and you cannot afford to lose a frame waiting for the scheduler.

For a marker loop, `CreateThread` is correct. If you ever find yourself adding artificial delays to make one thread "win the race" against another at resource start, that is the signal that the must-run-first one should be `CreateThreadNow` instead.

### Throttling inside a Wait(0) loop with GetGameTimer

Sometimes you genuinely need `Wait(0)` because part of the loop must run every frame, but another part inside it is expensive and does not need to. Sleeping the whole thread with a longer `Wait` is too blunt; it would slow down the frame-critical part too. The tool for that is `GetGameTimer`, which returns the number of milliseconds since the game started as an ever-increasing integer.

The pattern is to remember the last time you did the expensive work and only repeat it once enough time has passed:

```lua
local lastHeavy = 0

CreateThread(function()
    while true do
        Wait(0) -- this part must be smooth every frame

        local now = GetGameTimer()
        if now - lastHeavy > 500 then
            lastHeavy = now
            -- expensive work, at most twice a second
        end
    end
end)
```

The outer loop still wakes every frame, so anything frame-critical stays smooth. But the block guarded by `GetGameTimer` runs at most about twice a second, because `now - lastHeavy > 500` is only true once 500 ms have elapsed since the last run. You get per-frame responsiveness for the cheap part and a throttle on the expensive part, inside one thread. In our marker script we did not need this, because the whole loop can sleep when you are far away. You reach for `GetGameTimer` when the loop cannot sleep but one chunk of it should still be rate-limited.

### When you reach for each tool

- **Distance-check / Wait tuning** is your default for any world-drawing or proximity loop: markers, blips, zone checks, target prompts. Sleep far, `Wait(0)` only when close.
- **CreateThreadNow** is for resource-start setup that other code reads immediately.
- **GetGameTimer throttling** is for the rare loop that must stay on `Wait(0)` for one job while rate-limiting a heavier job beside it.

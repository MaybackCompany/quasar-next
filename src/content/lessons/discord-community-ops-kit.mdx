## Roles and escalation

<StepGroup>
  <Step num={1} title="Define role tiers" outcome="Every member has exactly one clear standing.">
    Keep tiers few and obvious: Member, Supporter/Donator, Staff, Admin, Owner. More tiers means more confusion. Map perks and access to these, not to one-off custom roles.
  </Step>

  <Step num={2} title="Set up a ticket flow" outcome="A player issue becomes a tracked, assignable item.">
    Use a ticket bot so opening a ticket creates a private thread or channel only the player and authorized staff can see. Configure the panel to ask what broke, when it happened, the in-game/character name, and any console error. Ask for a raw player identifier only when staff genuinely need it, keep it in the private ticket, and define retention/deletion rules; identifiers and logs are user data, not decoration. Treat ticket transcripts and screenshots the same way: redact identifiers before pasting them anywhere outside the private ticket.
  </Step>

  <Step num={3} title="Write the escalation path" outcome="Staff know who handles what without asking.">
    Document it: support staff handle player issues, admins handle bans and server changes, owner handles money and policy. Put it in staff-only so nobody freezes during an incident.
  </Step>

  <Step num={4} title="Establish an announcement rhythm" outcome="Players trust one channel for real news.">
    Post launches, updates, and events in announcements only. A predictable rhythm (weekly update, event teasers) keeps players returning.
  </Step>

## How it works

Two pieces of this kit look simple but trip up real servers: the logs webhook and the ticket thread. Both are about treating community ops like infrastructure, not chat.

### Why the logs webhook URL is a secret

To get server events into your private logs channel, open that channel's settings (gear icon, or right-click the channel) -> Integrations -> Webhooks -> New Webhook -> Copy Webhook URL. Discord gives you a URL that looks like this (the same Webhooks screen is where you later Delete or Regenerate it if it leaks). A server-side resource, built in a later lesson, reads the convar below and POSTs to this URL; this lesson only sets up the channel and stores the secret, so the logs channel stays empty until that resource exists. That empty channel is expected, not a mistake. If you want to confirm the webhook itself works before that resource exists, send one optional test message from PowerShell, replacing the URL with your own:

```powershell
curl.exe -H "Content-Type: application/json" -d '{"content":"webhook test"}' "https://discord.com/api/webhooks/123.../AbCd..."
```

A single line saying "webhook test" should appear in your logs channel. Delete that test message afterward.

```text
https://discord.com/api/webhooks/123456789012345678/AbCdEf...your-token...
```

That URL is a credential, not a link. Anyone who has it can POST a message to your logs channel as your server, with no login and no permission check. There is no separate password. Possession of the URL is the permission. If it leaks, someone can spam your staff channel, impersonate your bot, or bury a real incident under noise, and your only fix is to delete the webhook and issue a new URL. You do this in the same place you created it: Channel settings -> Integrations -> Webhooks -> select the webhook -> Delete Webhook, then create a fresh one and update the convar value in server.cfg. The old URL stops working the moment you delete it.

So treat it exactly like any other secret. Do not paste it in a public channel, a screenshot, or a commit. It belongs in your `server.cfg` as a convar, read by your resources, never hard coded into a script you might share:

```cfg
set discord_log_webhook "https://discord.com/api/webhooks/.../..."
# Use `set` (server-side only). Do NOT use `setr` (it replicates the value to every
# connected client) or `sets` (it publishes the value on the public server list and the
# open http://<server_ip>:<server_port>/info.json file). Either one leaks the webhook.
```

This is the same rule code review enforces on every other secret: keys and tokens live in config the server reads, not in the source. A webhook URL is no different.

### How a ticket maps to the incident runbook

A ticket bot that opens a private thread is doing more than starting a chat. The thread is a per-incident container: one issue, one place, only the player and staff inside. That maps directly onto the incident runbook, where each incident is its own tracked item with a clear owner.

The fields you make the panel collect (what broke, when, in-game name or license ID, the error text) are the inputs the runbook needs first: who and where to reproduce it, what the symptom was, and the raw error to search the logs for. Capturing them up front in the thread means staff are not chasing the player for basics before they can even start. The thread title becomes the incident, the fields become the report, and your escalation path decides who picks it up. That is why the ticket flow and the runbook share the same shape: the ticket is where an incident is born.

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| Welcome messages not sending to new members | Bot doesn't have permission or the welcome channel doesn't exist | Give the bot "Send Messages" and "Read Message History" in the welcome channel. Check the channel ID is correct |
| @everyone or @here being abused by members | Permission to mention large groups was given to regular roles | Remove "Mention @everyone" from member roles. Only admins should have it |
| Discord channels are a mess - no one knows where to post | Too many channels or no clear organization | Keep it simple: #announcements, #general, #support, #staff. Add more only when a channel gets too noisy |
| Members leaving because Discord is "dead" | No regular activity or engagement | Post updates about the server (events, changes, new content). A weekly #server-update post keeps people checking back |

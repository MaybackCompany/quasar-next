  Most beginners think the only way to make data survive a restart is a database. It is not. FiveM ships a per-resource **key-value store**, the KVS, that writes small values straight to disk for you. No schema, no connection string, no SQL. In this lesson you build a counter that climbs every time you run a command, then survives a full server restart untouched. Once it works, you will know exactly when the KVS is the right tool and when it is the wrong one.

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_kvs_counter
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_kvs_counter/
  fxmanifest.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'
    ```

    The KVS natives are available in Cfx scripting contexts, but this lesson deliberately uses them in a `server_script` so the saved counter is controlled by the server and not by a player's local client store. There is no `lua54 'yes'` line; Lua 5.4 is the only Lua runtime.
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The counter reads from disk, increments, and writes back.">
    Open `server.lua` and paste this:

    ```lua
local KEY = 'visit_count'

-- Read the saved number on startup. An unset integer key reads as 0.
local count = GetResourceKvpInt(KEY)
print(('[qu_kvs_counter] loaded count from KVS: %d'):format(count))

RegisterCommand('countup', function()
    count = count + 1
    SetResourceKvpInt(KEY, count)
    print(('[qu_kvs_counter] count is now %d (saved to disk)'):format(count))
end, true)

RegisterCommand('countshow', function()
    print(('[qu_kvs_counter] current count: %d'):format(GetResourceKvpInt(KEY)))
end, true)

RegisterCommand('countreset', function()
    DeleteResourceKvp(KEY)
    count = 0
    print('[qu_kvs_counter] count deleted from KVS, back to 0')
end, true)
    ```
  </Step>

  <Step num={5} title="Start and test it" outcome="The number climbs, then survives a full restart.">
    Open `server.cfg` and add this line:

    ```text
    ensure qu_kvs_counter
    ```

    Save, then run this in the txAdmin Live Console:

    ```text
    restart qu_kvs_counter
    ```

    Run the counter up three times, then check it:

    ```text
    countup
    countup
    countup
    countshow
    ```

    <ExpectedOutput>
      [qu_kvs_counter] loaded count from KVS: 0
[qu_kvs_counter] count is now 1 (saved to disk)
[qu_kvs_counter] count is now 2 (saved to disk)
[qu_kvs_counter] count is now 3 (saved to disk)
[qu_kvs_counter] current count: 3
    </ExpectedOutput>

    Now the real proof. Fully stop the server process and start it again. This is not a `restart qu_kvs_counter`, it is a cold boot of the whole server, the thing a database is supposed to survive. When the resource loads, watch the startup line:

    <ExpectedOutput>
      [qu_kvs_counter] loaded count from KVS: 3
    </ExpectedOutput>

    The number came back as `3` with no database anywhere. That is the KVS doing its job.
  </Step>
</StepGroup>

## How it works

You ran it, the count climbed, and it survived a cold restart. Now take the code apart so you understand where the data actually lives and why no `console.log` style guesswork was needed to confirm it. There are three natives doing the real work and one habit that keeps you out of trouble.

### Reading the saved value on startup

```lua
local count = GetResourceKvpInt(KEY)
print(('[qu_kvs_counter] loaded count from KVS: %d'):format(count))
```

`GetResourceKvpInt(key)` is a SERVER native that reads an integer you previously stored under that key. The key is just a string you choose, here `'visit_count'`. The store is **scoped to this one resource**, so `qu_kvs_counter` cannot see, read, or stomp on a key written by `qu_persistence` or any other resource. Each resource gets its own private drawer.

The most important detail is what happens the very first time, before you have ever written the key. `GetResourceKvpInt` returns `0` for a key that has never been set, so your `count` starts at a clean zero with no extra guard. If you used `GetResourceKvpString` instead, an unset key comes back as `nil`, which is why string reads almost always need an `or` fallback. The integer and float variants are friendlier here because they hand you a usable number on a fresh server.

This line runs once, at the top level of the file, the moment the resource starts. That timing is the whole reason persistence works: on every boot, the first thing the resource does is rehydrate its state from disk before any command can touch it.

### Incrementing and writing back

```lua
RegisterCommand('countup', function()
    count = count + 1
    SetResourceKvpInt(KEY, count)
    print(('[qu_kvs_counter] count is now %d (saved to disk)'):format(count))
end, true)
```

`SetResourceKvpInt(key, value)` writes an integer to the store under that key. There is no application-level save button or transaction to commit. Treat the call as a persistence write, not as free in-memory work: do not run it every frame or use KVS for high-frequency economy state.

Notice the two-step rhythm: bump the in-memory `count`, then push it to the KVS. The local variable is your fast working copy for this session; the server-side KVS is the durable copy. The restricted flag is `true`, so the command requires the `command.countup` ACE; the server console can run it. A remote client cannot call this server-side KVS code directly, but any network event you expose around it would still need normal server validation.

FiveM gives you three typed pairs, and you pick by the shape of your value:

- `SetResourceKvpInt` / `GetResourceKvpInt` for whole numbers like this counter.
- `SetResourceKvpFloat` / `GetResourceKvpFloat` for decimals like `1000.50`.
- `SetResourceKvp` / `GetResourceKvpString` for text, and for anything structured you serialize with `json.encode` into a string first.

Match the setter to the getter. Write with `SetResourceKvpInt` and read with `GetResourceKvpInt`. Mixing types, for example writing an int and reading it as a string, gives you garbage or `nil`.

### Deleting a key

```lua
RegisterCommand('countreset', function()
    DeleteResourceKvp(KEY)
    count = 0
    print('[qu_kvs_counter] count deleted from KVS, back to 0')
end, true)
```

`DeleteResourceKvp(key)` removes the key from the store entirely. After this, `GetResourceKvpInt(KEY)` reads `0` again because the key no longer exists. Setting the value to `0` and deleting the key are not the same thing: a deleted key is gone, while a key set to `0` still occupies a slot you can find when you list. We reset the local `count` to `0` in the same handler so the working copy matches the now-empty store.

<Checkpoint question="After countreset, you restart the whole server. What does the startup line print, and why?">
  It prints `loaded count from KVS: 0`. `DeleteResourceKvp` removed the `visit_count` key, so on the next boot `GetResourceKvpInt(KEY)` finds nothing under that key and returns its default of `0`. The number did not come back as `3` because there is no longer anything saved to come back. That is the difference between deleting a key and merely zeroing it: deleting wipes the durable copy, so the restart starts from the same blank slate a brand-new resource would.
</Checkpoint>

### Listing every key your resource owns

The three commands above each work on one known key. Sometimes you do not know the keys ahead of time, for example you stored one entry per player under keys like `player_money_42` and `player_money_57` and now you want to walk all of them. That is what the find iterator is for:

```lua
RegisterCommand('countkeys', function()
    local handle = StartFindKvp('')
    local found = 0
    while true do
        local key = FindKvp(handle)
        if not key then break end
        found = found + 1
        print(('[qu_kvs_counter] key %d: %s'):format(found, key))
    end
    EndFindKvp(handle)
    print(('[qu_kvs_counter] total keys: %d'):format(found))
end, true)
```

`StartFindKvp(prefix)` opens a search over every key in this resource's store whose name begins with the prefix string. An empty `''` prefix matches every key. It hands back a **handle**, a number that points at the search, the same idea as the ped handle from the natives lesson. `FindKvp(handle)` returns the next matching key name each time you call it, and returns `nil` once there are no more, which is why the `if not key then break end` line ends the loop. `EndFindKvp(handle)` closes the search and frees it. Always pair a `StartFindKvp` with an `EndFindKvp`, the same way you pair `function` with `end`, or you leak the handle.

The find natives return key **names**, not their values. To read each one you take the returned key and call the matching getter, for example `GetResourceKvpInt(key)`. The iterator finds the drawers; you still open each drawer yourself.

### When you reach for the KVS, and when you do not

The KVS is the right tool when all of these are true:

- The data is **small** and **per-resource**: settings, a counter, a feature toggle, the last value a script saw. No other resource needs to read it.
- There is **no schema and no relationships**. You are storing loose keys, not rows that join to other rows.
- You will **never query it**. You can fetch a key by name or list keys by prefix, and that is the entire query language. There is no "find every player with more than 5000 dollars".

You reach for **MariaDB plus oxmysql** the moment any of those breaks. oxmysql is the Overextended community resource that connects your server to a MariaDB database, and MariaDB is the database engine to prefer in 2026 over plain MySQL. You need it when the data is **relational** (players own vehicles, vehicles belong to garages), when you must **query and filter** (leaderboards, "all characters for this license"), or when the data is **shared across many resources** that each read and write the same tables. A whole-server economy with 64 players hammering wallets is squarely database territory, not KVS territory.

This is the line that separates this lesson from **Persistence: cache plus write-back**. KVS is for small, low-frequency, resource-private state. It still performs persistent storage work, so cache or debounce frequent changes. MariaDB plus oxmysql is for relational, shared, queryable data and supports transactions and indexing. Pick by the data model and durability requirements, not by which API is shorter.

## Common mistakes

<FailureList
  items={[
    { symptom: "Writing with SetResourceKvpInt but reading back nil or a wrong number", fix: "You read with the wrong typed getter. An int written with SetResourceKvpInt must be read with GetResourceKvpInt, a float with GetResourceKvpFloat, a string with GetResourceKvpString. Match the getter to the setter exactly." },
    { symptom: "attempt to perform arithmetic on a nil value (local 'count')", fix: "You used GetResourceKvpString to read a numeric key. String reads return nil for an unset key, and nil cannot be added to. Use GetResourceKvpInt for whole numbers, which returns 0 for an unset key, or add an 'or 0' fallback." },
    { symptom: "The counter resets to 0 every time you restart the whole server", fix: "You are reading the key into count but never writing it with a Set call, or you are writing to a different key string than you read from. Confirm the same KEY value is used in both the GetResourceKvpInt and the SetResourceKvpInt calls." },
    { symptom: "countkeys prints nothing or hangs", fix: "Either the store is genuinely empty (run countup first), or you forgot the 'if not key then break end' guard so the while loop never ends. FindKvp returns nil when there are no more keys, and that nil is your only signal to stop." },
    { symptom: "Other resources cannot read your visit_count key", fix: "That is by design, not a bug. The KVS is scoped per resource. If two resources must share the same data, the KVS is the wrong tool and you want a database via oxmysql instead." },

## Try it yourself

<Exercise>
  Turn the single counter into a per-player visit tracker that survives restarts, then list the leaderboard. The key for each player is built from their identifier, and the find iterator reads them all back. Build a server command that increments the caller's own count and one that prints every stored player count:

  ```lua
  -- Server side. Use a stable identifier for the durable key.
  RegisterCommand('myvisits', function(source)
      local license = GetPlayerIdentifierByType(source, 'license')
      if not license then
          print('[qu_kvs_counter] no license identifier for source ' .. source)
          return
      end
      local key = ('visits_%s'):format(license)
      local n = GetResourceKvpInt(key) + 1
      SetResourceKvpInt(key, n)
      print(('[qu_kvs_counter] player %s now has %d visits'):format(source, n))
  end, false)

  RegisterCommand('allvisits', function()
      local handle = StartFindKvp('visits_')
      while true do
          local key = FindKvp(handle)
          if not key then break end
          print(('[qu_kvs_counter] %s = %d'):format(key, GetResourceKvpInt(key)))
      end
      EndFindKvp(handle)
  end, true)
  ```

  The command receives a temporary server source, then converts it to a stable `license:` identifier before building the persistent key. Never use `source` itself as a durable key: sources are connection slots and are reused. The `'visits_'` prefix makes `allvisits` ignore the main `visit_count` key. Because this dump prints player identifiers, keep `allvisits` ACE-restricted and never expose its output to clients. Once you need sorting, filtering, cross-resource access, or many players, move this data to MariaDB.
</Exercise>

<Callout variant="danger" title="The IP line you do not cross">
 Do not stream ripped real-world brands, leaked paid packs, or assets you have no license to use. It is the fastest way to a takedown, a marketplace ban, or worse. Use assets you created, bought with a server license, or that are explicitly free for FiveM use. Keep a written note of where each pack came from. If you cannot point to a license, do not ship it.
</Callout>

## Build it

<StepGroup>
 <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
 Inside your server's `resources` folder, create this layout. The `stream/` folder name is not optional. FiveM scans a folder with exactly that name and auto-registers whatever asset files it finds inside.

 ```text
resources/qu_stream_proof/
 fxmanifest.lua
 server.lua
 stream/
 ```

 Leave `stream/` empty for the structural check. When you validate a real pack later, its valid `.ydd`, `.ytd`, `.ymt`, `.ytyp`, or other documented files go here. Renaming an empty text file to `.ytyp` does not create a valid archetype and can produce a misleading or broken test.
 </Step>

 <Step num={2} title="Write fxmanifest.lua" outcome="FiveM knows the resource and registers the streamed type.">
 Open `fxmanifest.lua` and paste this:

 ```lua
fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'
 ```

 No `-- Lua 5.4 is the default runtime in 2026; the lua54 directive is no longer needed` line is needed. When a real prop pack includes a `.ytyp`, copy the pack author's manifest entry, commonly `data_file 'DLC_ITYP_REQUEST' 'stream/name.ytyp'`. Do not add a `data_file` entry for a file that does not exist or invent one for a clothing pack that does not ship it.
 </Step>

 <Step num={3} title="Write the proof line" outcome="The resource reports the moment it registers its stream.">
 Open `server.lua` and paste this. It listens for the resource starting and prints one line confirming the `stream/` folder went live:

 ```lua
AddEventHandler('onResourceStart', function(resourceName)
 if resourceName ~= GetCurrentResourceName() then return end

 print('[qu_stream_proof] resource started; validate real assets in the client')
end)
 ```

 This print proves only that the server resource loaded. Server-side `onResourceStart` cannot prove that a client downloaded or rendered an asset. We guard it with the `resourceName` check so it reacts only to this resource.
 </Step>

 <Step num={4} title="Start it" outcome="The proof line appears in the server console.">
 Open `server.cfg` and add this line:

 ```text
 ensure qu_stream_proof
 ```

 Save, then run this in the txAdmin Live Console:

 ```text
 restart qu_stream_proof
 ```

 <ExpectedOutput>
 [qu_stream_proof] resource started; validate real assets in the client
 </ExpectedOutput>
 </Step>

 <Step num={5} title="Measure it in resmon" outcome="Streaming memory becomes a number, not a vibe.">
 In your own FiveM client connected to the server, open the F8 console and run:

 ```text
 strmem 1
 ```

 Use the streaming diagnostics only after adding a **valid, licensed asset**. An empty folder has nothing meaningful to measure. `strmem` is a developer command, blocked on a normal client with `Access denied / disabled in production mode`. Relaunch FiveM with launch arg `set moo 31337` or a non-production update channel (same for `resmon`/`strdbg`/`strlist`), then reconnect and run `strmem 1`. Use the official client-console reference linked below and confirm the asset visually in game as well as in diagnostics.
 </Step>
</StepGroup>

## How it works

You restarted the resource and proved its manifest and server script load. You have **not** proved asset streaming yet because the folder is empty. A real validation needs a valid licensed asset, a clean client download, and an in-game visual or streaming diagnostic. Keep that boundary clear: a server start line is not a client-render proof.

### The stream/ folder is the magic, not a config

Most FiveM features need you to wire something up: register an event, call a native, list a file. Streaming is different. There is one rule: any asset file inside a folder named `stream/` is auto-registered when the resource starts. You do not list each `.ydd` or `.ytd` in the manifest. FiveM walks the folder for you.

```text
stream/
 qu_proof.ytyp
```

When a client connects, the server sends it the manifest of streamed files for every started resource. The client downloads them on first join and caches them. This is why a 500 MB clothing pack is felt as a slow first join: every player pulls every byte before they can play. The pack author does not control that cost. You do, by what you choose to put in `stream/`.

There are exactly two kinds of file in that folder. The meshes and textures (`.ydd` drawables, `.ytd` texture dictionaries) auto-register with no help. The type definitions (`.ytyp`, used by prop packs to declare new object types) are the exception that needs the manifest line you wrote.

### Why the data_file line is required for props

```lua
data_file 'DLC_ITYP_REQUEST' 'stream/qu_proof.ytyp'
```

A `.ytyp` is a type archetype: it tells the game a new prop or object class exists, its bounds, its LOD distances. Auto-registration handles the drawables and textures, but the engine will not load a `.ytyp` as an active type request on its own. `data_file 'DLC_ITYP_REQUEST'` is the explicit instruction to do so. The first argument names the data file kind the game expects; the second is the path inside your resource. Leave this line out of a prop pack and you get the classic symptom: the resource starts with no error, but the props are invisible or fall through the world because their type was never requested.

Pure clothing packs (drawables and textures only, no new prop types) often do not need this line. The moment a pack includes accessory props with their own `.ytyp`, it does.

### How EUP differs from a normal clothing swap

"EUP" is commonly used for emergency-service uniform packs, but it is not one universal resource layout. A pack may be a replacement or an add-on clothing collection and may include drawable/texture files plus metadata such as `.ymt` files. Install the exact manifest and metadata shipped by the licensed pack; do not assume every pack must be renamed `eup-stream` or requires a `.ytyp`. First prove the resource starts, then use the game's drawable/texture counts or the appearance resource's documented refresh path to prove the new variations exist.

### How a clothing menu maps slots to your drawables

Your players never type file names. They open a menu (illenium-appearance, qb-clothing, or similar) and pick a number. That menu speaks the game's component model, which you have to understand to wire a pack correctly. Every ped has a fixed set of component slots, each an integer:

<Cheatsheet
 title="GTA ped component IDs"
 rows={[
 { cmd: "0", desc: "Head / face" },
 { cmd: "1", desc: "Mask" },
 { cmd: "3", desc: "Torso / upper body and arms" },
 { cmd: "4", desc: "Legs / pants" },
 { cmd: "6", desc: "Shoes / feet" },
 { cmd: "8", desc: "Undershirt / accessory base" },
 { cmd: "11", desc: "Jacket / top (the slot most clothing packs target)" },
 ]}
/>

Within a slot, a **drawable** is which model variation (the jacket shape), and a **texture** is which skin on that drawable (the color or pattern). So a menu selection is three numbers: component ID, drawable index, texture index. When you add a clothing pack, you are appending new drawable indexes onto one of these slots. The menu's config is where you tell it the new indexes exist, which is the concrete version of the old lesson's hand-wavy "wire it to your menu." You are mapping slot 11, drawable 42, texture 0 to a label your players can click.

<Checkpoint question="A new EUP pack downloads fine (you see it in resmon) but the uniforms never appear in your clothing menu. The files are streaming. What is the most likely cause?">
 Streaming bytes is only one stage. Confirm the pack's shipped metadata loaded, the pack targets the ped model you are testing, and the appearance resource supports or refreshes add-on collections. Follow the pack and menu's official install instructions rather than assuming every menu needs a hand-written index list.
</Checkpoint>

### Measuring instead of guessing

"Keep textures at sane resolutions" is useless advice because it has no number attached. Here is the number. With `strmem 1` running, the streaming overlay shows each resource's memory in MB and the total against your streaming budget. A healthy clothing pack is tens of MB, not hundreds. Most bloat is `.ytd` texture dictionaries saved at 2048x2048 or 4096x4096 when 1024x1024 would look identical at gameplay distance. Open a fat pack's textures, downscale the oversized ones, repack, and watch the resmon number drop. That is the whole optimization loop: change something, restart, read the overlay, compare. You ship the version with the smaller number that still looks right.

### When you reach for this pattern

The `stream/` folder plus an optional `data_file` line is not for clothing. It is how you add every custom visual asset to FiveM: vehicles, weapons, map props, MLO interiors, and clothing all ride the same mechanism. Once you can build, start, and measure `qu_stream_proof`, you can reason about any of them, because they are all the same folder doing the same job. The only variation is which file types are inside and whether any of them need a `data_file` line to register their type.

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| Custom clothes don't appear on character | Files not in `files {}` or wrong folder structure | Check every .ytd/.yft is listed in `files {}`. Verify the stream folder path matches |
| Clothes appear but are solid white (no textures) | .ytd texture file missing or not loaded | The texture dictionary (.ytd) must be in `stream/` and listed in `files {}` |
| "Could not load resource" after adding EUP pack | Too many files or broken manifest syntax | Check fxmanifest for missing commas. Large packs may need splitting into multiple resources |
| Clothes override default GTA clothes | The addon is replacing instead of adding | Check vehicles.meta (for clothes: peds.meta) has a unique `<modelName>` that doesn't match a GTA original |
| EUP/clothes pack crashes the server | Missing dependency or wrong game build | Check the pack requires a specific game build. Update `sv_enforceGameBuild` if needed |

## Try it yourself

<Exercise>
 Turn the structure into a real size test. Find a clothing or prop pack you are licensed to use, preserve its documented folder layout, and copy its valid `stream/` contents into `qu_stream_proof/stream/`. If it ships `.ytyp` files, preserve the author's corresponding manifest entries.

 ```lua
 data_file 'DLC_ITYP_REQUEST' 'stream/pack_props_a.ytyp'
 data_file 'DLC_ITYP_REQUEST' 'stream/pack_props_b.ytyp'
 ```

 Restart the resource, then run `strmem 1` in F8 and write down the MB the resource reports. Now open the pack's largest `.ytd` in OpenIV or CodeWalker, find any texture saved above 1024x1024, downscale it, repack, and restart again. Read the overlay a second time. Confirm the MB dropped and the asset still looks right in game. You ran the optimization loop on real numbers: change, restart, measure, compare. That is the difference between shipping a pack and shipping a pack you can defend.
</Exercise>

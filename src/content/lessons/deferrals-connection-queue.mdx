## The handshake before the player loads

When someone clicks Connect, the server fires a `playerConnecting` event **before** the player has a ped, a server id slot they can act from, or any resource loaded on their screen. This is the handshake window. It runs entirely on the server, and it is the one moment where you hold all the control and the player holds none.

The platform passes your handler three things: the connecting player's `name`, a `setKickReason` function you can mostly ignore in favor of deferrals, and a `deferrals` object. That last one is the whole point of this lesson. Calling `deferrals.defer()` tells the server "do not finish this join yet, I am going to take my time and decide." Until you call `deferrals.done()`, the player sits on the connecting screen and waits for you.

Why does this matter so much? Because **this is the only place you can block a join**. Once a player is fully connected, they already have a slot and their client is loading resources. Kicking them then is a cleanup, not a gate. Blocking them here, during `playerConnecting`, means they never get in at all. And because the check runs on the server, the player cannot fake their way past it. The client may *request* a connection, but only the server *grants* it. That is what server-authoritative means: the decision lives on the machine the player does not control.

    Open `server.lua` and paste this:

    ```lua
-- Hardcoded allowlist of permitted license identifiers.
-- Replace these with your own. Format is 'license:' followed by a long hex string.
local allowedLicenses = {
    ['license:0000000000000000000000000000000000000000'] = true,
}

AddEventHandler('playerConnecting', function(name, setKickReason, deferrals)
    local source = source

    -- Pause the join. Nothing finishes until we call deferrals.done().
    deferrals.defer()

    -- Give the player something to read while we work. A short yield first
    -- lets the deferral register cleanly before the first update.
    Wait(0)
    deferrals.update(('Hello %s, checking your access...'):format(name))

    -- Read this player's license identifier on the server.
    local license = GetPlayerIdentifierByType(source, 'license')

    Wait(500)

    if license and allowedLicenses[license] then
        print(('[qu_connection_gate] ALLOW %s (%s)'):format(name, license))
        deferrals.done()
    else
        print(('[qu_connection_gate] REJECT %s (%s)'):format(name, tostring(license)))
        deferrals.done('Not whitelisted')
    end
end)
    ```
  </Step>

## How it works

You connected and the gate decided. Now take the handshake apart so the next time you need to control who gets in, you are deciding, not copying. The whole flow is one event handler and four calls on the `deferrals` object.

### The event and its three arguments

```lua
AddEventHandler('playerConnecting', function(name, setKickReason, deferrals)
    local source = source
```

`playerConnecting` is a built-in server event. You attach to it with `AddEventHandler`, not `RegisterNetEvent`, because the **server itself** raises it, not a client. The platform calls your function once per connecting player with three arguments in this order:

- `name` is the display name the player is connecting with, a string. Use it for friendly messages and log lines, never as a security check, because a name is trivial to fake.
- `setKickReason` is a function you could call to set a plain kick message. In modern code you let `deferrals.done('reason')` do that job instead, so this argument usually goes unused.
- `deferrals` is the object that controls the join. Everything that follows hangs off it.

The first line of the body, `local source = source`, captures the connecting player's source into a local. In a server event handler, `source` is the special value that identifies which player triggered the event. Copying it into a local right away is a habit worth keeping, because any `Wait` you call later can let other code run, and reading the global `source` after a yield is a classic way to read the wrong player. You read it once, up front, and use the local from then on.

### Pausing the join: deferrals.defer

```lua
deferrals.defer()
```

This is the call that buys you time. The instant you call `deferrals.defer()`, the server stops trying to finish this join and waits for you to make a decision. The player stays on the connecting screen. If you never call `deferrals.done()`, they sit there forever, which is itself a bug to avoid. Think of `defer` as raising your hand and saying "wait, I am not done deciding." Everything after it runs while the player is held.

### Talking to the player: deferrals.update

```lua
Wait(0)
deferrals.update(('Hello %s, checking your access...'):format(name))
```

`deferrals.update(message)` replaces the text on the player's connecting screen with whatever string you pass. This is your one channel to tell a held player what is happening, for example "checking your access," "you are number 3 in queue," or "verifying your account." The `:format(name)` method drops the player's name into the `%s` placeholder so the message reads personally.

The `Wait(0)` before it yields for a single frame. Calling `defer` and then immediately calling `update` in the same tick can race, so a one-frame yield lets the deferral register cleanly first. It is a small piece of defensive timing, not decoration.

### Reading who is connecting: GetPlayerIdentifierByType

```lua
local license = GetPlayerIdentifierByType(source, 'license')
```

This is the heart of the gate. `GetPlayerIdentifierByType(source, kind)` is a server native that returns one of the player's identifiers as a string. Ask for `'license'` and you get back something like `license:1a2b3c...`, a stable id tied to the player's Rockstar/Cfx account. A license does not change when the player reboots or moves house, which is exactly why it is the standard handle for an allowlist. A player has several identifier types (`steam`, `discord`, `ip`, and more), but `ip` changes and `steam` can be absent, so `license` is the dependable one to gate on.

Because you are reading this server side during `playerConnecting`, the value is trustworthy. The player cannot hand you a fake license here. Compare that to anything the client sends you over a net event, which you must always treat as a claim, not a fact.

### Deciding: deferrals.done and deferrals.done(reason)

```lua
if license and allowedLicenses[license] then
    print(('[qu_connection_gate] ALLOW %s (%s)'):format(name, license))
    deferrals.done()
else
    print(('[qu_connection_gate] REJECT %s (%s)'):format(name, tostring(license)))
    deferrals.done('Not whitelisted')
end
```

`allowedLicenses[license]` is the allowlist check. The table uses each permitted license string as a key set to `true`, so looking up a key that exists returns `true` and any other key returns `nil`. The `license and` guard in front handles the rare case where the identifier comes back `nil`, so you never index the table with a missing value.

Then comes the decision, and it is the same function both ways:

- `deferrals.done()` with **no argument** ends the deferral and **allows** the join. The player leaves the connecting screen and starts loading in.
- `deferrals.done('Not whitelisted')` with a **string argument** ends the deferral and **rejects** the join. The player is dropped, and the string you passed shows up as the reason on their disconnect screen.

That single function, called with or without a reason, is the entire allow/reject switch. You must call `done` exactly once on every path through the handler. Miss it on one branch and that player hangs on the connecting screen.

<Checkpoint question="Why is playerConnecting the only place you can truly block a join, and why can the player not fake their way past it?">
  Because `playerConnecting` fires on the server before the player is fully connected. At that point you hold the join open with `deferrals.defer()` and the player has no slot, no ped, and no loaded resources, so calling `deferrals.done('reason')` stops them from ever getting in. Anywhere later is a kick after the fact, not a gate. They cannot fake it because the check runs entirely on the server: you read the `license` with `GetPlayerIdentifierByType` server side, and the player has no way to forge that value or to grant their own connection. The client requests, the server grants. That is what server-authoritative means.
</Checkpoint>

### When you reach for this pattern

The deferrals handshake is the backbone of nearly every access feature a serious server runs:

- **A whitelist.** Exactly what you built. Swap the hardcoded table for a database lookup against MariaDB and you have a real whitelist that staff can manage without editing code.
- **A connection queue.** While a player is deferred you can hold them and stream `deferrals.update` messages like "you are number 4 in queue" until a slot opens, then call `deferrals.done()`. The held state is what makes a queue possible at all.
- **An adaptive card prompt.** `deferrals.presentCard(card, cb)` renders a richer Adaptive Card on the connecting screen with buttons and inputs, for example a rules screen with an Accept button or a code entry. You build the card as a Lua table or JSON, and the callback fires when the player submits it. You still finish with `deferrals.done()` or `deferrals.done('reason')` based on what they chose.

All three are the same shape: defer, communicate, decide. You just built the smallest complete version of it.

## Common mistakes

<FailureList
  items={[
    { symptom: "Players get stuck on the connecting screen forever", fix: "A code path reached the end without calling deferrals.done(). Make sure every branch, both allow and reject, calls done() exactly once. An error thrown after defer() also leaves the player hanging, so wrap risky lookups carefully." },
    { symptom: "attempt to call a nil value (field 'defer')", fix: "The deferrals argument was not received, usually because the handler signature is wrong. The order is function(name, setKickReason, deferrals) and you must use AddEventHandler('playerConnecting', ...), not RegisterNetEvent." },
    { symptom: "Everyone is rejected even after you add a real license", fix: "The license string in the table does not match exactly. Copy the full value from the console log, keep the license: prefix, and watch for trailing spaces. Table keys are compared character for character." },
    { symptom: "license comes back nil so every player is rejected", fix: "You read source after a Wait without capturing it first, or asked for an identifier the player does not have. Capture local source = source before any Wait, and gate on 'license', which every Cfx account has." },
    { symptom: "The reject reason does not show on the player's screen", fix: "You passed the reason to setKickReason or called deferrals.done() with no argument. The reason must be the string argument to deferrals.done('Not whitelisted')." },
  ]}

## Try it yourself

<Exercise>
  Turn the silent reject into a friendly, informative one, and prove the held state is real. Two changes:

  First, before you decide, hold the player for three seconds while counting down a visible message, so you can see `deferrals.update` working over time:

  ```lua
  for i = 3, 1, -1 do
      deferrals.update(('Verifying access... %d'):format(i))
      Wait(1000)
  end
  ```

  Second, when you reject, tell the player exactly which license was refused so they can send it to staff to be added:

  ```lua
  deferrals.done(('Not whitelisted. Send this to staff: %s'):format(tostring(license)))
  ```

  Restart `qu_connection_gate` and connect with a license that is not in the table. Confirm you watch the countdown tick from 3 to 1 on the connecting screen, then get dropped with a reason that includes your own license string. You have now used every part of the deferrals object: defer to hold, update to communicate over time, and done with a reason to reject. Bonus: move the allowlist into a separate Lua table file and load it, so adding a license never means touching the handler logic.
</Exercise>

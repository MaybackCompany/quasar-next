  Some work is too heavy to run all at once inside the main game tick - database queries, HTTP calls, complex math, large table processing. FiveM lets you spread this work across separate cooperative threads. This is an advanced technique. Use it only when you have already applied distance checks, Wait tuning, and throttling and still need more headroom.

<Callout variant="warn" title="There is no parallelism here">
  FiveM Lua is single-threaded and cooperative. `CreateThread` does NOT start an OS thread or run code in parallel - it starts a coroutine that the scheduler runs one at a time on the same tick. A heavy synchronous pass still blocks everything until it yields. The real fix is to break the work into chunks and call `Wait(0)` inside the loop so the scheduler can run rendering, the game loop, and other resources between chunks. Before reaching for extra threads, try distance checks, increasing Wait intervals, throttling with timers, and caching results.
</Callout>

## Build it

<StepGroup>
  <Step num={1} title="Create the resource" outcome="A basic resource with a light loop and a chunked heavy loop.">
    ```text
resources/qu_multithread_demo/
  fxmanifest.lua
  client.lua
    ```

    ```lua fxmanifest.lua
fx_version 'cerulean'
game 'gta5'
client_script 'client.lua'
    ```

    ```lua client.lua
-- Light loop: keep this fast, yields every frame
CreateThread(function()
    while true do
        -- light work here: UI updates, proximity checks
        Wait(0)
    end
end)

-- Heavy loop: chunk the work and yield WITHIN the loop
CreateThread(function()
    while true do
        -- Process a large table a few items at a time, then yield.
        -- Without the inner Wait(0) this whole pass blocks the tick,
        -- because both loops share the same single thread.
        local allVehicles = GetGamePool('CVehicle')
        for i, v in ipairs(allVehicles) do
            -- Heavy processing for one vehicle here
            if i % 20 == 0 then
                Wait(0)  -- yield to the scheduler between chunks
            end
        end
        Wait(1000)  -- re-run the bulk pass once per second
    end
end)
    ```
  </Step>

  <Step num={2} title="Understand thread scheduling" outcome="You know why yielding inside the loop is what keeps the game smooth.">
    FiveM runs all threads cooperatively on a single thread. Nothing runs in parallel. Each `Wait()` yields control back to the scheduler, which then resumes the next ready coroutine. A loop that does heavy work without yielding blocks EVERYTHING until it finally hits a `Wait()`. Splitting work across two `CreateThread` loops does not give you two CPUs - it only helps if each loop yields often. So:

    - **Light loop**: yields every frame (Wait(0)), stays responsive
    - **Heavy loop**: yields mid-pass (Wait(0) every chunk), then waits a second before the next bulk pass

    The game loop, rendering, and other resources all share the same single thread with your code. The smoothness comes from yielding frequently, not from running anywhere "in parallel".

    <ExpectedOutput>
      The game stays smooth because the heavy pass yields between chunks instead of hogging the tick.
    </ExpectedOutput>
  </Step>

  <Step num={3} title="Use SetTimeout for deferred one-shot work" outcome="Heavy work runs once, not in a loop.">
    Sometimes you don't need a loop - you need to defer a single heavy operation:

    ```lua client.lua
-- Defer heavy work without blocking the current tick
SetTimeout(0, function()
    -- This runs on the NEXT tick, not this one
    local result = ProcessHeavyData()
    SendNUIMessage({type = 'dataReady', data = result})
end)
    ```

    `SetTimeout(0, fn)` is the FiveM equivalent of `setTimeout(fn, 0)` in JavaScript. It defers execution to the next available tick without blocking the current one.

    <ExpectedOutput>
      The current tick finishes instantly. The heavy work runs on the next tick without users noticing a stutter.
    </ExpectedOutput>
  </Step>
</StepGroup>

## Common failures

| Exact error | Cause | Fix |
|---|---|---|
| Two loops read/write the same data and interleave unexpectedly | A yield (Wait) lets the other loop run mid-update, so shared state is seen half-finished | Update shared state in one loop, or use a queue pattern - do not split a single update across a Wait |
| Thread never yields (no Wait call) | Infinite loop with no Wait() | Every loop MUST call Wait() - even Wait(0) is enough |
| resmon shows high tick time despite two loops | A loop does heavy work without yielding, so it blocks the tick before its Wait | Chunk the work and call Wait(0) inside the loop; stagger bulk passes so they do not run on the same tick |

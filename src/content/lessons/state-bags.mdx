RegisterCommand('setmood', function(src, args)

    print('[qu_state_bags] set mood ' .. mood .. ' on player ' .. src)

local myBagName = ('player:%s'):format(GetPlayerServerId(PlayerId()))

## How it works

You set one value on the server, and a totally different machine, the client, printed it back. You sent zero events to make that happen. That is the whole promise of a state bag. Here is the deal: understand how that value crossed the network once, and the exact same three lines unlock entity state and global state too.

### What a state bag actually is

A state bag is a **key/value store bolted onto something the engine already syncs across the network**: a player, an entity, or the server itself. You write `Player(src).state:set('mood', 'focused', true)`, FiveM stores `mood = "focused"` against that player, and it copies the value out to whoever can see it.

The big difference from an event: an event fires once and is gone. A state bag holds the current value for as long as its owner exists. But this is live network state, not saved-to-disk memory. When the owner leaves, the state goes with it. So it does not replace MariaDB or the KVS for anything that must survive a restart.

There are three scopes, and picking the right one is the real skill here. Scope just means where the bag lives and who gets the copy:

- **`Entity(handle).state`** belongs to a specific entity: a vehicle, a ped, a prop. The value travels with that entity. A common use is an engine-on flag or a locked flag on a car. When the entity is deleted, its bag goes with it.
- **`Player(serverId).state`** belongs to a player slot. Use it for player-level facts that should follow the person rather than their current body: a job, a stress level, an admin duty flag. This is what you built.
- **`GlobalState`** belongs to the whole server. There is exactly one, shared by everyone. Use it for world facts: the current weather, whether a heist is active, a server-wide event timer.

The rule of thumb: scope the data to the thing it describes. A car door does not belong on a player, and the weather does not belong on a car.

### The server line, argument by argument

```lua
Player(src).state:set('mood', mood, true)
```

`Player(src)` finds the player by their **server id**, the slot number you saw printed as `1`. `src` comes from the command handler, just like in the events lessons: it is the player who typed the command. `.state` is that player's bag. `:set(...)` writes one key into it. Read the three arguments left to right:

- `'mood'` is the **key**, the name of the slot you are writing. You read it back later by this exact name.
- `mood` is the **value**, here whatever word the player typed. Values can be strings, numbers, booleans, or whole tables, so `{ level = 3, color = 'red' }` works just as well. One catch: `args[1]` is raw player input, and players type whatever they want. So for any state that hands out power (a job, an admin flag) you validate it or lock down who can set it on the server. Never trust the argument as-is.
- `true` is the **replicate flag**. Server-set state already replicates by default, so leaving the third argument off would still send it to clients. Passing `true` just makes the intent obvious. Pass `false` to keep that write server-only.

### The client handler, argument by argument

```lua
local myBagName = ('player:%s'):format(GetPlayerServerId(PlayerId()))

AddStateBagChangeHandler('mood', myBagName, function(bagName, key, value)
    print('[qu_state_bags] my mood is now ' .. tostring(value))
end)
```

`AddStateBagChangeHandler` says "run this function whenever a matching bag changes." It takes three arguments:

- First, `'mood'`, is the **key filter**: only changes to a key named `mood` wake this handler up. Pass `nil` and you would hear about every key on every matching bag instead.
- Second, `myBagName`, is the **bag filter**: which bag to watch. Player bags are named `player:<serverId>`, so building that name from your own server ID points the handler at your bag and nobody else's. Only pass `nil` when you actually want to watch every matching bag.
- Third is the **callback**, the function FiveM runs when a change fires. It hands you `bagName`, `key`, and `value`. `bagName` is a string like `player:1`, where `1` is the server id whose bag changed. `value` is the new value.

Learn the bag name shape, because it repeats across scopes: a player bag is `player:<serverId>`, an entity bag is `entity:<netId>`, and global state is just `global`. Want every player bag? Pass `nil` and sort out each bag yourself inside the callback. Want only your own? Build the exact name like we did here. It is simpler and it never mixes up server IDs, local player indexes, and not-found values.

<Checkpoint question="Why does this handler build `player:<serverId>` instead of comparing a resolved player value with `PlayerId()`?">
  A server ID, a client-local player index, and a state-bag name are three different identifiers. Building the exact bag name from `GetPlayerServerId(PlayerId())` lets FiveM filter before the callback runs and avoids ambiguous conversions. The result is one handler for exactly this client's player bag.
</Checkpoint>

### When you reach for a state bag instead of an event

Use a state bag when "what is the value right now?" has to be answerable by anyone who asks, including the player who joined ten seconds ago. Three concrete cases:

- **A flag that must survive.** A player goes on admin duty. With an event you would have to re-broadcast the flag every time someone new connects. With `Player(src).state:set('onDuty', true, true)` you set it once and every present and future client can read it.
- **A property of a thing, synced for free.** A vehicle's locked status. Put it on `Entity(veh).state` and every client that can see the car knows whether it is locked, without you tracking which clients to message.
- **A world fact.** A server-wide heist is active. `GlobalState:set('heistActive', true, true)` flips one switch and every script, on every client, can branch on `GlobalState.heistActive`.

If instead you only need a one-time "do this now" with no value worth remembering, an event is still the right tool. State bags are for state. Events are for moments.

## Try it yourself

<Exercise>
  Promote the mood from a player fact to a world fact. Add a second command, `/serverstatus`, that writes to GlobalState instead of a player, and a second handler that reads it on every client:

  ```lua
  -- server.lua
  RegisterCommand('serverstatus', function(_, args)
      local status = args[1] or 'calm'
      GlobalState.serverStatus = status
      print('[qu_state_bags] server status is now ' .. status)
  end, true)
  ```

  ```lua
  -- client.lua
  AddStateBagChangeHandler('serverStatus', 'global', function(bagName, key, value)
      print('[qu_state_bags] world status changed to ' .. tostring(value))
  end)
  ```

  Restart `qu_state_bags`, run `serverstatus busy` from txAdmin Live Console, and confirm every connected client prints the new status, with no per-player guard needed. The command is ACE-restricted because a world-wide status is server authority, not something every player should set. Notice you passed `'global'` as the bag filter instead of `nil`, which narrows the handler to the server-wide bag.
</Exercise>

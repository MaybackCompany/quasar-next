## How it works

You set one value on the server and a different machine, the client, printed it back without you sending a single event. That is the whole promise of a state bag, and it is worth understanding exactly how the value crossed the network, because the same three lines unlock entity state and global state too.

### What a state bag actually is

A state bag is a **key/value store bolted onto something the engine already networks**: a player, an entity, or the server itself. You write `Player(src).state:set('mood', 'focused', true)` and FiveM stores `mood = "focused"` against that player and ships the change to every client that needs it. It is the opposite of an event in one important way. An event is fire-and-forget: you call `TriggerClientEvent`, it runs once on whoever is listening right now, and then it is gone. A state bag is **a value that stays set**. Anyone who reads the bag later, including a player who connects an hour from now, sees the current value. That persistence is the entire reason to choose a state bag over a broadcast event.

Three scopes exist, and picking the right one is the skill the lesson title promises:

- **`Entity(handle).state`** belongs to a specific entity: a vehicle, a ped, a prop. The value travels with that entity. A common use is an engine-on flag or a locked flag on a car. When the entity is deleted, its bag goes with it.
- **`Player(serverId).state`** belongs to a player slot. Use it for player-level facts that should follow the person rather than their current body: a job, a stress level, an admin duty flag. This is what you built.
- **`GlobalState`** belongs to the whole server. There is exactly one, shared by everyone. Use it for world facts: the current weather, whether a heist is active, a server-wide event timer.

The rule of thumb: scope the data to the thing it describes. A car door does not belong on a player, and the weather does not belong on a car.

### The server line, argument by argument

```lua
Player(src).state:set('mood', mood, true)
```

`Player(src)` looks up the player by their **server id**, the slot number you saw printed as `1`. The `src` comes from the command handler, exactly as it did in the events lessons: it is the player who typed the command. `.state` is that player's state bag, the key/value store. `:set(...)` writes one key. Read its three arguments left to right:

- `'mood'` is the **key**, the name of the slot you are writing. You read it back later by this same name.
- `mood` is the **value**, here whatever word the player typed after the command. State bag values can be strings, numbers, booleans, or tables, so you could store `{ level = 3, color = 'red' }` just as easily.
- `true` is the **replicated flag**, and it is the argument everyone skips and then wonders why nothing reached the client. State set from the server replicates to clients by default, but you should pass `true` explicitly so your intent is unmistakable and the value is guaranteed to cross the network. Pass `false` and the value stays server-only: it is set, the server can read it, but no client is ever told. A server-only flag is occasionally what you want, but for anything a client must react to, `true` is the line between working and silently doing nothing.

### The client handler, argument by argument

```lua
AddStateBagChangeHandler('mood', nil, function(bagName, key, value)
    local player = GetPlayerFromStateBagName(bagName)
    if player == 0 then return end
    if player ~= PlayerId() then return end
    print('[qu_state_bags] my mood is now ' .. tostring(value))
end)
```

`AddStateBagChangeHandler` registers a function that fires whenever a matching bag changes. It takes three arguments:

- The first, `'mood'`, is the **key filter**: only changes to a key named `mood` wake this handler. Pass `nil` here and you would hear about every key on every matching bag.
- The second, `nil`, is the **bag filter**: which bag to watch. `nil` means all of them, so this handler hears the `mood` change on every player, not just yours. That is exactly why the guard below matters.
- The third is the **callback**, and FiveM hands it `bagName`, `key`, and `value` when a change fires. `bagName` is a string like `player:1`, where `1` is the server id whose bag changed. `value` is the new value.

Now the two guard lines, which are the part the old version of this lesson got subtly wrong:

`GetPlayerFromStateBagName(bagName)` parses that `player:1` string and converts the server id into a **local player index**, the kind of number `PlayerId()` returns. It returns `0` when no matching player exists on this client. Here is the trap: `0` is also a valid local player index, the local player, so a naive `if player == PlayerId()` check passes when the function actually failed to find anyone. That is why the first guard is `if player == 0 then return end`: bail out before the not-found value can masquerade as you. Only after that do we ask `if player ~= PlayerId() then return end`, which drops every other player's mood change and keeps only your own. The handler runs on every client for every player, so without these two lines you would print other people's moods as if they were yours.

The bag name format is worth memorizing because the same shape shows up for the other scopes. A player bag is `player:<serverId>`. An entity bag is `entity:<netId>`, and you parse it with `GetEntityFromStateBagName` the same way. Global state changes report a bag name of `global`.

<Checkpoint question="Why does the client handler need both the `player == 0` guard and the `player ~= PlayerId()` guard, when one comparison looks like it should be enough?">
  Because `GetPlayerFromStateBagName` returns `0` both for the local player and as its not-found sentinel. The bag filter is `nil`, so the handler fires for every player's `mood` change on the server, not just yours. When a change arrives for a player your client cannot resolve, the function returns `0`. If you only wrote `if player == PlayerId()`, that `0` would equal your local index of `0` and you would wrongly print a stranger's mood as your own. The `if player == 0 then return end` line throws out the not-found case first, and only then does `player ~= PlayerId()` cleanly reject every player who is not you. Together they guarantee the print fires for exactly one player: the one running this client.
</Checkpoint>

### When you reach for a state bag instead of an event

Reach for a state bag when the answer to "what is the value right now?" needs to be true for anyone who asks, including late joiners. Three concrete cases:

- **A flag that must survive.** A player goes on admin duty. With an event you would have to re-broadcast the flag every time someone new connects. With `Player(src).state:set('onDuty', true, true)` you set it once and every present and future client can read it.
- **A property of a thing, synced for free.** A vehicle's locked status. Put it on `Entity(veh).state` and every client that can see the car knows whether it is locked, without you tracking which clients to message.
- **A world fact.** A server-wide heist is active. `GlobalState:set('heistActive', true, true)` flips one switch and every script, on every client, can branch on `GlobalState.heistActive`.

If instead you only need a one-time "do this now" with no value worth remembering, an event is still the right tool. State bags are for state. Events are for moments.

## Try it yourself

<Exercise>
  Promote the mood from a player fact to a world fact. Add a second command, `/serverstatus`, that writes to GlobalState instead of a player, and a second handler that reads it on every client:

  ```lua
  -- server.lua
  RegisterCommand('serverstatus', function(src, args)
      local status = args[1] or 'calm'
      GlobalState:set('serverStatus', status, true)
      print('[qu_state_bags] server status is now ' .. status)
  end, false)
  ```

  ```lua
  -- client.lua
  AddStateBagChangeHandler('serverStatus', 'global', function(bagName, key, value)
      print('[qu_state_bags] world status changed to ' .. tostring(value))
  end)
  ```

  Restart `qu_state_bags`, run `/serverstatus busy`, and confirm every connected client prints the new status, with no per-player guard needed. Then explain to yourself why the global handler has no `GetPlayerFromStateBagName` check: a `GlobalState` change is not tied to any one player, so there is no owner to filter by. Notice you passed `'global'` as the bag filter this time instead of `nil`, which narrows the handler to only the server-wide bag.
</Exercise>

<Callout variant="fivem" title="The chain, in one picture">
  Your Lua script → oxmysql (the connector) → MariaDB (the database) → HeidiSQL (your window into it). Each piece does one job. Your script asks oxmysql to save data. oxmysql translates that into SQL. MariaDB stores it. HeidiSQL lets you see it.

## Build it

<StepGroup>
  <Step num={1} title="Create a database" outcome="An empty database is ready to receive tables.">
    Open HeidiSQL, connect to MariaDB (Host: `127.0.0.1`, User: `root`, Port: `3306`). Right-click the server → **Create new** → **Database**. Name it `fivem_db`.

    A database is a container for tables. It starts empty. Scripts create tables inside it when you import their SQL files.

    <ExpectedOutput>
      `fivem_db` appears in HeidiSQL's database list. It has no tables yet.
    </ExpectedOutput>
  </Step>

  <Step num={2} title="Install oxmysql and connect" outcome="The server console says 'Connection Successful'.">
    Place oxmysql in your resources folder, then add to server.cfg:

    ```bash server.cfg
ensure oxmysql

set mysql_connection_string "mysql://root:YOUR_PASSWORD@127.0.0.1/fivem_db?charset=utf8mb4"
set mysql_slow_query_warning 150
    ```

    Breaking down the connection string:
    - `root` — your MariaDB username
    - `YOUR_PASSWORD` — your MariaDB password (leave empty if you didn't set one)
    - `127.0.0.1` — where MariaDB lives (`localhost` = same machine)
    - `fivem_db` — the database name you created in Step 1
    - `?charset=utf8mb4` — character encoding (always include this)

    Restart the server. Watch the console for the magic line:
    ```text
    [oxmysql] Connection Successful
    ```

    <ExpectedOutput>
      The console prints "Connection Successful." If it prints an error, check your password, database name, and that MariaDB is running.
    </ExpectedOutput>
  </Step>

  <Step num={3} title="Understand SQL files" outcome="You know what a .sql file does and why scripts ship them.">
    Most FiveM scripts include a `.sql` file. This file creates the tables the script needs.

    Example `users.sql`:
    ```sql
CREATE TABLE IF NOT EXISTS users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    money INT DEFAULT 500,
    job VARCHAR(50) DEFAULT 'unemployed'
);
    ```

    This creates a table called `users` with four columns: `id`, `name`, `money`, and `job`. The table is the skeleton — rows of data fill it later. Without importing this file, the script's database queries will fail because the table doesn't exist.
  </Step>

  <Step num={4} title="Import an SQL file" outcome="New tables appear in HeidiSQL.">
    In HeidiSQL:
    1. Select your database (`fivem_db`) on the left
    2. Click **File → Load SQL file** (or drag the `.sql` file into the query window)
    3. Review the SQL — understand what tables it creates
    4. Click **Execute** (the blue triangle)
    5. Right-click the database → **Refresh**

    New tables should appear. Common table names: `users`, `players`, `owned_vehicles`, `jobs`, `houses`, `ox_inventory`.

    <ExpectedOutput>
      After refresh, the database shows the new tables with their columns.
    </ExpectedOutput>
  </Step>

  <Step num={5} title="Verify the connection from a script" outcome="A test query proves everything works.">
    Create a test resource. The `MySQL` global is provided by oxmysql — you need it in your fxmanifest:

    ```lua fxmanifest.lua
fx_version 'cerulean'
game 'gta5'
server_script 'server.lua'
shared_script '@oxmysql/lib/MySQL.lua'  -- this makes MySQL.query available
    ```

    Now create the server script:

    ```lua server.lua
-- Try to fetch all users from the database
local result = MySQL.query.await('SELECT * FROM users')
print('Database query returned: ' .. json.encode(result))
    ```

    If the table is empty, you'll see `[]`. That's fine — it means the connection works. If the table has data, you'll see the rows.

    <ExpectedOutput>
      The console prints the query result without error. No "connection failed" or "table doesn't exist" messages.
    </ExpectedOutput>
  </Step>
</StepGroup>

## The database install workflow

Every time you install a script that needs a database:

1. **Read the README** — what framework? What dependencies?
2. **Find the .sql file** — usually in the script's root folder or a `sql/` subfolder
3. **Import it** — open HeidiSQL, select your database, load and execute the .sql
4. **Check the tables appeared** — refresh and verify
5. **Add ensure lines** — in server.cfg, AFTER the dependency's ensure line
6. **Restart the server** — and watch the console for errors

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| `[oxmysql] Connection Failed` | Wrong password, database name, or MariaDB not running | Check MariaDB is started. Verify the connection string matches your setup exactly |
| `Unknown database 'fivem_db'` | The database doesn't exist | Create it in HeidiSQL first: right-click → Create new → Database |
| `Table 'users' doesn't exist` | SQL file wasn't imported | Import the script's .sql file in HeidiSQL |
| `Access denied for user 'root'` | Wrong password or MariaDB authentication settings | Reset MariaDB root password or check auth plugin settings |
| SQL import fails: `Table already exists` | You already imported this file before | That's fine — the data is already there. You can skip this or drop and re-import |
| `mysql_connection_string` has no effect | oxmysql not in server.cfg before scripts that need it | `ensure oxmysql` must come BEFORE any script that queries the database |

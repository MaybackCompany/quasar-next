    ```text
  server.lua
    ```

  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load and what must load first.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'

dependencies {
    'ox_inventory'
}
    ```

    `ox_inventory` is a community resource from the Overextended team, not a built-in Cfx.re resource, so it must already be installed and started on your server for this lesson to work. The `dependencies { }` block is the canonical FiveM form. You will see the singular `dependency 'ox_inventory'` in older code; it still works, but prefer the plural block because it scales to a list without rewriting the line.
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `server.lua` and paste this:

    ```lua
RegisterCommand('burger', function(source)
    local src = source

    -- Add 5 burgers. AddItem returns success + a response string.
    local success, response = exports.ox_inventory:AddItem(src, 'burger', 5)
    if not success then
        print('[qu_inventories] add failed: ' .. tostring(response))
        return
    end

    -- Read the TRUE count back from the inventory, do not trust the count we asked for.
    local count = exports.ox_inventory:Search(src, 'count', 'burger')
    print('[qu_inventories] burger count ' .. count)
end, true)
    ```
  </Step>

  <Step num={5} title="Start and test it" outcome="The expected proof appears in the server console.">

    ```text
    ensure qu_inventories
    ```

    Save, then run:

    ```text
    restart qu_inventories
    ```

    Join the server so you have a player and an inventory, then run this from the txAdmin Live Console, passing your own server id (the example uses `1`):

    ```text
    burger 1
    ```

    </ExpectedOutput>

## How it works

You ran it and the count appeared. The reason the count is the proof, and not the return value of `AddItem`, is the whole point of this lesson. Take it apart so that next time you reach for an inventory you are deciding, not copying.

### The two models, and why the count is the truth

Before the code, the mental model. There are two ways to limit what a player can carry.

- **Slot-based.** The inventory is a fixed grid of numbered pockets, say 50 slots. Each kind of item occupies a slot, and a slot holds up to a stack limit of that item. Run out of free slots and you cannot carry more, even if each item is feather-light.
- **Weight-based.** The inventory has a single kilogram cap, say 60 kg. Nothing else is counted. Carry one heavy item or a thousand light ones, the only question is the total weight.

`ox_inventory` enforces **both at once**: every player has a slot count and a weight cap, and an add can fail because either limit is hit. That is exactly why you cannot trust the number you asked for. You requested 5 burgers, but if the player had room for only 3 by weight or slots, the inventory holds 3, not 5. So the lesson reads the real number back instead of assuming the request was honored in full.

### Adding the item

```lua
local success, response = exports.ox_inventory:AddItem(src, 'burger', 5)
```

`exports.ox_inventory:AddItem` is how one resource calls a function another resource has published. `ox_inventory` exports `AddItem`; you call it with `exports.ox_inventory:` followed by the function name. The arguments here are the inventory to add to (`src`, the player's server id), the item name (`'burger'`), and the count (`5`).

It returns two values: `success`, a boolean, and `response`, a string. On a clean success, `success` is `true`. On a hard failure, such as a completely full inventory, `success` is `false` and `response` is a reason like `inventory_full`. That is why the code captures both and prints `response` when the add fails, instead of printing a bare "add failed" that tells you nothing.

Here is the trap the old version of this lesson fell into. `success` is not a clean "all five landed" flag. It tells you the add was accepted, not how many units actually fit once weight and slot caps were applied. Gating only on the boolean and assuming you got all five is how phantom items and desync bugs are born. The honest move is the next line.

### Reading the truth back

```lua
local count = exports.ox_inventory:Search(src, 'count', 'burger')
```

`Search` asks the inventory what it actually holds. The second argument is the **mode**, and the mode decides what you get back:

- `'count'` returns a single number: the total quantity of that item across every slot it occupies. That is what we print, because a number is the cleanest proof that the item is really there.
- `'slots'` returns an array of tables, one per slot the item lives in, each with its `slot` number, its `count`, and its `metadata`. You reach for `'slots'` when you need to know *where* the item sits or inspect per-stack metadata, not just the grand total.

So the flow is: ask the inventory to add, then ask the inventory what it now holds. The printed `5` is not the `5` you passed to `AddItem`. It is the inventory's own answer, read fresh. If a weight cap had clipped the add to 3, this line would print `3`, and you would see the truth instead of a lie.

### Stacking and metadata

```lua
exports.ox_inventory:Search(src, 'slots', 'burger')
```

Why did five burgers print as one count and not clutter five slots? Because identical items **stack**. Two units of an item are identical when they share the same name and the same metadata, so they merge into one slot and the slot's count goes up. That is what keeps a slot-based inventory usable: 64 ammo is one slot, not 64.

Metadata is the per-item data attached to a single stack: durability, a weapon serial, a registered owner, a crafting quality. The rule that follows is the one to memorize. Items only stack when their metadata matches. Give two burgers different metadata and they can no longer share a slot, because they are no longer identical, so they occupy two slots with a count of 1 each. This is exactly how unique items work. A serialized pistol cannot stack with another pistol, because each carries a different serial in its metadata. You attach metadata as a fourth argument: `AddItem(src, 'burger', 1, { quality = 100 })`. Switch to `'slots'` mode and you will watch a plain burger and a metadata burger split into two separate slots.

### Where the item itself is defined

`AddItem(src, 'burger', 5)` only works if `'burger'` already exists. The name is not magic. It is a key in `ox_inventory`'s `data/items.lua`, a single Lua table where each key is the canonical item name and each value is that item's config. If you call `AddItem` with a name that is not in this table, it fails with `invalid_item`, which is exactly the failure mode listed below.

Here is a real entry from `ox_inventory`'s own `data/items.lua`, kept verbatim. Note `stack` is absent because it defaults to `true`, `weight` is in grams not kilograms, and `consume = 0.3` drains durability per use instead of removing a whole item:

```lua
['testburger'] = {
    label = 'Test Burger',
    weight = 220,
    degrade = 60,
    client = {
        image = 'burger_chicken.png',
        status = { hunger = 200000 },
        anim = 'eating',
        prop = 'burger',
        usetime = 2500,
        export = 'ox_inventory_examples.testburger'
    },
    server = {
        export = 'ox_inventory_examples.testburger',
        test = 'what an amazingly delicious burger, amirite?'
    },
    buttons = {
        { label = 'Lick it', action = function(slot) print('You licked the burger') end },
        { label = 'Squeeze it', action = function(slot) print('You squeezed the burger :(') end }
    },
    consume = 0.3
}
```

The key (`'testburger'`) is the string you pass everywhere: in `AddItem`, in shops, in the database. The icon is a PNG in `web/images/` named after the key (or after `client.image` when set, here `burger_chicken.png`). After you edit `data/items.lua` you must `restart ox_inventory` so it reloads the definitions, otherwise the running server keeps rejecting the new name with `invalid_item`.

A one-line note on the paid alternative: **qs-inventory** (Quasar Store) defines items in `shared/items.lua` with a flatter QBCore-style shape and uses `unique = true` where ox uses `stack = false`, so an ox item config will not stack correctly if you paste it straight into qs without translating those fields. Building a full item in both, icon and capacity check included, is its own walkthrough: [Create a usable item in ox_inventory (and qs-inventory)](/lessons/creating-items-ox-qs-inventory).

<Checkpoint question="You call AddItem(src, 'burger', 5) and it returns success = true, but Search(src, 'count', 'burger') returns 3. What happened, and why is the lesson right to print the Search result instead of 5?">
  The player only had room for 3 burgers before hitting a slot or weight cap, so ox_inventory added 3 and stopped. `success = true` means the add was accepted, not that all five units fit, which is the trap. `Search` with `'count'` mode reads the real total back from the inventory, so it returns the honest `3`. Printing the value you passed to `AddItem` would claim five items that do not exist, and any code that trusted that number would drift out of sync with reality. Always confirm against what the inventory actually holds.
</Checkpoint>

### Why integrate with ox_inventory instead of rolling your own

You could store items in a plain Lua table and a database column. People do, and then they spend months rebuilding what already exists. A real inventory has to enforce weight and slot limits, stack identical items, keep unique items unstacked by metadata, sync state to the client UI, survive a server restart, and prevent duplication exploits when two actions touch the same stack at once. `ox_inventory` solves all of that and is battle-tested across thousands of servers. Your job is to call `AddItem`, `RemoveItem`, and `Search`, and to trust the inventory as the single source of truth. Reach for it any time a player needs to carry, drop, trade, or store items. Hand-roll an item table only for a throwaway prototype, never for anything players will exploit.

## Try it yourself

<Exercise>
  Prove the metadata stacking rule with your own eyes. Extend the command so it adds one plain burger and one burger with metadata, then prints the slot layout instead of just the count:

  ```lua
  RegisterCommand('burger', function(source)
      local src = source
      exports.ox_inventory:AddItem(src, 'burger', 1)
      exports.ox_inventory:AddItem(src, 'burger', 1, { quality = 100 })

      local slots = exports.ox_inventory:Search(src, 'slots', 'burger')
      print('[qu_inventories] burger occupies ' .. #slots .. ' slot(s)')
      for _, item in ipairs(slots) do
          print('[qu_inventories] slot ' .. item.slot .. ' count ' .. item.count)
      end
  end, true)
  ```

  Restart `qu_inventories`, join the server, and run `burger 1`. Confirm the plain burger and the metadata burger refuse to stack: the print reports two slots, each with a count of 1, because their metadata differs. Now change the second add to plain `AddItem(src, 'burger', 1)` with no metadata, restart, and confirm they merge back into one slot with a count of 2. You just watched the stacking rule decide the slot layout, which is the core mechanic every inventory in FiveM is built on.
</Exercise>

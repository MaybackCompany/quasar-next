  A restaurant does not open for guests without prepped stations, a backup line, and a rollback plan if a dish fails. Same here: convars pinned, ACE locked, the database backed up, monitoring on, and a known-good build to revert to. You will build one small resource named `qu_release` that prints a **boot header** on start, then rehearse a backup and a one-command rollback before you ever touch production.

  Treat go-live as a readiness checklist. Each item flips green or it does not ship: pin the build, lock permissions, back up the database, watch for trouble, and keep one command that rolls you back to the last known-good build. Do one step, read the proof, then move to the next step.

## Build it

<StepGroup>
  <Step num={1} title="Make the release resource" outcome="One folder holds your boot-header proof.">
    txAdmin ships bundled inside FXServer, so you already have an admin panel at `http://localhost:40120` the first time you boot. You do not install it separately. Inside your server's `resources` folder, create one folder for this lesson:

    ```text
    resources/qu_release/
      fxmanifest.lua
      server.lua
      VERSION
    ```

    Put your release tag in `VERSION` (no quotes, no extension content beyond the tag):

    ```text
    v1.0.0
    ```
  </Step>

  <Step num={2} title="Write fxmanifest.lua" outcome="FXServer knows which file to load.">
    Open `fxmanifest.lua` and paste this. Note there is **no** `lua54 'yes'` line. Lua 5.4 is the only runtime now, so that line is deprecated and ignored. Leave it out.

    ```lua
    fx_version 'cerulean'
    game 'gta5'

    server_script 'server.lua'

    files {
        'VERSION'
    }
    ```
  </Step>

  <Step num={3} title="Print a boot header" outcome="Every start logs the release tag in the first lines.">
    During an incident you should never have to ask "what version is running?". Open `server.lua` and paste this. It reads `VERSION` from the resource and prints a short header when the resource starts.

    ```lua
    AddEventHandler('onResourceStart', function(resource)
        if resource ~= GetCurrentResourceName() then return end

        local version = LoadResourceFile(GetCurrentResourceName(), 'VERSION') or 'dev'
        version = version:gsub('%s+', '')

        print('[qu_release] release  : ' .. version)
        print('[qu_release] artifact : ' .. GetConvar('sv_enforceGameBuild', 'unset'))
        print('[qu_release] boot ok')
    end)
    ```
  </Step>

  <Step num={4} title="Pin the build and lock permissions" outcome="Clients cannot drift; admins cannot kill the server by accident.">
    Open `server.cfg`. Pin the game build so a Rockstar update cannot crash every player, then load and lock the resource. Use a current build number from the live artifacts list (linked at the bottom) and never pin a stale build you copied from an old guide.

    ```text
    # license: free from the Cfx.re Portal (portal.cfx.re). Required even locally.
    sv_licenseKey "cfxk_yourkeyhere"

    # pin to a CURRENT build from the live list, then bump it on purpose after staging
    sv_enforceGameBuild 2802

    set onesync on
    sv_maxClients 48

    # lock commands: allow admins, but deny quit so a misclick cannot stop the server
    add_ace group.admin command allow
    add_ace group.admin command.quit deny
    add_principal identifier.discord:000000000000000000 group.admin

    # keep secrets out of git: rotate this every deploy, load via a gitignored file
    exec server-secrets.cfg

    ensure qu_release
    ```
  </Step>

  <Step num={5} title="Restart on staging and read the header" outcome="The boot header appears with no errors below it.">
    Open the txAdmin Live Console in your browser at `http://localhost:40120`. Restart only this resource:

    ```text
    restart qu_release
    ```

    Read the first lines. You should see the header, and crucially nothing red underneath it.

    <ExpectedOutput>
      [qu_release] release  : v1.0.0
      [qu_release] artifact : 2802
      [qu_release] boot ok
    </ExpectedOutput>
  </Step>

  <Step num={6} title="Back up the database before you deploy" outcome="A restorable snapshot exists on disk.">
    A backup you have never made is just hope. Take a snapshot before any production deploy. On a hosted panel this is the database backup button. On a machine with `mysqldump` you can run:

    ```bash
    mysqldump --single-transaction --quick --routines fivem > backup-before-release-v1.sql
    ```

    Swap `fivem` for your own database name, the one in your `mysql_connection_string` convar in `server.cfg`, not the literal word `fivem`. `--single-transaction` takes a consistent snapshot of InnoDB tables on MariaDB without locking out players. Keep this file off the game host so a dead host does not take the backup with it.

    <ExpectedOutput>
      backup-before-release-v1.sql written, size greater than 0 bytes
    </ExpectedOutput>
  </Step>

  <Step num={7} title="Rehearse the rollback path" outcome="One pair of commands returns you to the last known-good build.">
    Before you ever need it, prove the rollback works on staging. Keep the previous version's folder on disk as `qu_release_v0`. To roll back, stop the new one and start the old one from the txAdmin Live Console:

    ```text
    stop qu_release
    ensure qu_release_v0
    ```

    The old boot header should print, telling you the previous version is live again. This is muscle memory you want before launch night, not during it.

    <ExpectedOutput>
      [qu_release_v0] release  : v0.9.0
      [qu_release_v0] boot ok
    </ExpectedOutput>
  </Step>
</StepGroup>

## If something went wrong

  items={[
    { symptom: "Server refuses to start: invalid license key.", fix: "Get a free key from the Cfx.re Portal at portal.cfx.re and paste it into sv_licenseKey. It is required even for a local test server." },

    { symptom: "Boot header reads release : dev.", fix: "The VERSION file is missing or not in the files block. Add files { 'VERSION' } to fxmanifest.lua and put the tag in VERSION." },
    { symptom: "Players crash the morning after a Rockstar update.", fix: "sv_enforceGameBuild was unpinned or stale. Pin a current build from the live artifacts list and bump it on purpose only after staging tests." },

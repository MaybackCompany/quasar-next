## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_teleport_menu
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_teleport_menu/
  fxmanifest.lua
  server.lua
  client.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load, and loads ox_lib first.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

shared_script '@ox_lib/init.lua'
client_script 'client.lua'
server_script 'server.lua'

dependencies {
    'ox_lib'
}
    ```

    Two things to know about this manifest. There is no `lua54 'yes'` line: as of June 2025 that directive is deprecated and ignored, because Lua 5.4 is now the only Lua runtime. And `ox_lib` is a Community Ox library, not a built-in Cfx.re resource. You install it yourself and it must be running before this resource, which is what the `dependencies` block enforces.
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The menu, the server check, and the fade all exist as runnable code.">
    Open `server.lua` and paste this:

    ```lua
local allowed = { legion = true, airport = true }

RegisterNetEvent('qu_teleport_menu:request', function(destination)
    local src = source
    if not allowed[destination] then
        print('[qu_teleport_menu] rejected source ' .. src .. ' for ' .. tostring(destination))
        return
    end
    print('[qu_teleport_menu] approved source ' .. src .. ' for ' .. destination)
    TriggerClientEvent('qu_teleport_menu:approved', src, destination)
end)
    ```

    Open `client.lua` and paste this:

    ```lua
local spots = {
    legion = vec3(215.76, -810.12, 30.73),
    airport = vec3(-1037.62, -2737.86, 20.17),
}

RegisterCommand('tpmenu', function()
    lib.registerContext({
        id = 'qu_teleport_menu_menu',
        title = 'Teleport',
        options = {
            { title = 'Legion Square', onSelect = function() TriggerServerEvent('qu_teleport_menu:request', 'legion') end },
            { title = 'Airport', onSelect = function() TriggerServerEvent('qu_teleport_menu:request', 'airport') end },
        },
    })
    lib.showContext('qu_teleport_menu_menu')
end, false)

RegisterNetEvent('qu_teleport_menu:approved', function(destination)
    local c = spots[destination]
    if not c then return end
    DoScreenFadeOut(500)
    Wait(600)
    SetEntityCoords(PlayerPedId(), c.x, c.y, c.z, false, false, false, false)
    DoScreenFadeIn(500)
    print('[qu_teleport_menu] moved to ' .. destination)
end)
    ```
  </Step>

## How it works

You ran it and you teleported. Now take the code apart so the next time you build a menu-driven action you are deciding, not copying. Three ideas carry this lesson: the ox_lib context menu, the client to server to client round-trip that lets the server approve the move, and the fade timing. We will walk each one.

### The ox_lib context menu

ox_lib is a Community Ox utility library. It is not shipped by Cfx.re, so you install it yourself and start it before any resource that uses it. The `shared_script '@ox_lib/init.lua'` line in the manifest pulls ox_lib's helper table into your resource. From that point on, `lib` is a global you can call. If ox_lib is not started, `lib` is `nil` and every line that touches it errors. That is the most common failure in this lesson.

```lua
lib.registerContext({
    id = 'qu_teleport_menu_menu',
    title = 'Teleport',
    options = {
        { title = 'Legion Square', onSelect = function() TriggerServerEvent('qu_teleport_menu:request', 'legion') end },
        { title = 'Airport', onSelect = function() TriggerServerEvent('qu_teleport_menu:request', 'airport') end },
    },
})
lib.showContext('qu_teleport_menu_menu')
```

A **context menu** is a list of options the player clicks through. `lib.registerContext` describes the menu but does not display it. You hand it a table with an `id` (a unique name you choose so you can refer to it later), a `title` shown at the top, and an `options` array. Each option is its own table with a `title` the player reads and an `onSelect` function that runs when the player clicks that row. `lib.showContext` takes the `id` you registered and actually opens the menu on screen. Registering and showing are two separate calls on purpose: you can register a menu once at startup and show it many times.

Look closely at what `onSelect` does. It does **not** teleport. It calls `TriggerServerEvent('qu_teleport_menu:request', 'legion')`, which sends a message to the server saying "this player wants to go to legion." That word, `legion`, is the only thing that crosses the network. The client never decides the coordinates here. That sets up the whole point of the lesson.

### Why the server checks the destination

This round-trip controls what the **resource** will approve. It is useful when a destination requires an ACE, job, item, cooldown, or audit entry. It does not prove that the player's whole client is trustworthy: a modified client can call `SetEntityCoords` or locally trigger a client handler without asking this server event. Protect the valuable server-side result, such as entering a heist state or opening a stash, with its own server check.

```lua
local allowed = { legion = true, airport = true }

RegisterNetEvent('qu_teleport_menu:request', function(destination)
    local src = source
    if not allowed[destination] then
        print('[qu_teleport_menu] rejected source ' .. src .. ' for ' .. tostring(destination))
        return
    end
    print('[qu_teleport_menu] approved source ' .. src .. ' for ' .. destination)
    TriggerClientEvent('qu_teleport_menu:approved', src, destination)
end)
```

`allowed` is an **allow-list**: a table of the only destination names the server will accept. `RegisterNetEvent` registers a handler that runs on the server when a client fires `qu_teleport_menu:request`. Inside a server net event, the global `source` holds the server ID of the player who sent it, so the first line copies it into `src` before anything else can overwrite it. The guard `if not allowed[destination] then ... return end` is the gate: if the player sent a name that is not in the allow-list, the server logs a rejection and returns, and nothing happens. Only when the name passes does the server fire `TriggerClientEvent('qu_teleport_menu:approved', src, destination)`, which sends an approval back to that one player by their `src`.

Notice the shape of the trip. Client to server with `TriggerServerEvent`, then server back to one client with `TriggerClientEvent` plus the target `src`. That out-and-back is the round-trip. A normal client cannot make this resource approve a name outside `allowed`. A modified client can still teleport locally, so never use this table as the only protection for money, items, doors, stashes, or mission state.

<Checkpoint question="A player edits their client.lua to add a third menu option that sends 'bank_vault' to the server. Do they teleport into the vault?">
  This resource will not approve `bank_vault`: the server checks `allowed['bank_vault']`, gets `nil`, and returns. That does not stop an exploited client from moving its own ped by another route. Any reward, stash, mission state, or permission at the destination still needs a separate server-side check.
</Checkpoint>

### The fade and the teleport

The approval comes back to the client, and only now does the actual move happen.

```lua
RegisterNetEvent('qu_teleport_menu:approved', function(destination)
    local c = spots[destination]
    if not c then return end
    DoScreenFadeOut(500)
    Wait(600)
    SetEntityCoords(PlayerPedId(), c.x, c.y, c.z, false, false, false, false)
    DoScreenFadeIn(500)
    print('[qu_teleport_menu] moved to ' .. destination)
end)
```

`spots` is the client-side table that maps each approved name to a `vec3`, a value holding x, y, and z coordinates. The server sent the name; the client looks up the coordinates. The guard `if not c then return end` protects against a name with no matching spot.

The four lines after that are a small choreography, and the order matters:

- `DoScreenFadeOut(500)` fades the screen to black over 500 milliseconds. The number is the duration in ms.
- `Wait(600)` pauses this handler for 600 ms. Net event handlers run in a thread that can yield, so `Wait` is allowed here. This number is deliberately larger than the fade. You want the screen fully black **before** you move the player, so they never see the world snap. If you teleport at 400 ms while the fade is still finishing, the player catches a glimpse of the jump. Wait longer than the fade, always.
- `SetEntityCoords(PlayerPedId(), c.x, c.y, c.z, false, false, false, false)` does the move. `PlayerPedId()` returns the handle of the player's own character on this client. The three coordinates are the destination. The four `false` flags are, in order, `alive`, `deadFlag`, `ragdollFlag`, and `clearArea`. For a plain on-foot teleport, all false is fine.
- `DoScreenFadeIn(500)` fades back from black now that the player is standing in the new spot.

So the sequence reads: black out, wait until fully black, move, fade back in. The teleport hides inside the black frame, which is what makes it feel smooth instead of jarring.

### When you reach for this pattern

The client-asks, server-approves, server-answers round-trip is the backbone of almost every action a player can trigger that the server must police. You just built it once; here is where it shows up again:

- **Job and shop menus.** A player clicks "buy the rifle." The client sends the item id, the server checks the player can afford it and is allowed to buy it, then the server grants it. The client never sets its own money.
- **Admin teleports and noclip.** Same menu shape, but the server's allow-list is "is this player an admin" instead of a list of place names.
- **Any menu that grants something.** Garages, stashes, faction doors. The menu is just the UI. The decision always belongs on the server.

When the action is purely cosmetic and local, like opening a settings panel, you can skip the server trip. When it changes authoritative game state, route the request through the server and validate again at the protected action. A server-approved client teleport alone is not a security boundary.

## Try it yourself

<Exercise>
  Add an admin-only destination that the menu offers to everyone but the server refuses unless the player is an admin. Add a third option `Police Armory` to the client menu that sends `armory`, and add `armory` to the client `spots` table with any coordinates you like. On the server, do not put `armory` in the plain `allowed` table. Instead, inside the request handler, check the player's permission before approving it:

  ```lua
  if destination == 'armory' then
      if not IsPlayerAceAllowed(src, 'qu_teleport_menu.armory') then
          print('[qu_teleport_menu] denied armory to source ' .. src)
          return
      end
      TriggerClientEvent('qu_teleport_menu:approved', src, destination)
      return
  end
  ```

  Grant yourself the ace in server.cfg with `add_ace identifier.fivem:YOUR_ID qu_teleport_menu.armory allow`, restart, and confirm you reach the armory while a second account without the ace gets the denied line in the server console. You have now built a menu where the same UI gives different power to different players, with the server as the only judge.
</Exercise>

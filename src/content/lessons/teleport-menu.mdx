## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_teleport_menu
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_teleport_menu/
  fxmanifest.lua
  server.lua
  client.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load, and loads ox_lib first.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

shared_script '@ox_lib/init.lua'
client_script 'client.lua'
server_script 'server.lua'

dependencies {
    'ox_lib'
}
    ```

    Two things to know about this manifest. There is no `-- Lua 5.4 is the default runtime in 2026; the lua54 directive is no longer needed` line: as of June 2025 that directive is deprecated and ignored, because Lua 5.4 is now the only Lua runtime. And `ox_lib` is a Community Ox library, not a built-in Cfx.re resource. You install it yourself and it must be running before this resource, which is what the `dependencies` block enforces.
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The menu, the server check, and the fade all exist as runnable code.">
    Open `server.lua` and paste this:

    ```lua
local allowed = { legion = true, airport = true }

RegisterNetEvent('qu_teleport_menu:request', function(destination)
    local src = source
    if not allowed[destination] then
        print('[qu_teleport_menu] rejected source ' .. src .. ' for ' .. tostring(destination))
        return
    end
    print('[qu_teleport_menu] approved source ' .. src .. ' for ' .. destination)
    TriggerClientEvent('qu_teleport_menu:approved', src, destination)
end)
    ```

    Open `client.lua` and paste this:

    ```lua
local spots = {
    legion = vec3(215.76, -810.12, 30.73),
    airport = vec3(-1037.62, -2737.86, 20.17),
}

RegisterCommand('tpmenu', function()
    lib.registerContext({
        id = 'qu_teleport_menu_menu',
        title = 'Teleport',
        options = {
            { title = 'Legion Square', onSelect = function() TriggerServerEvent('qu_teleport_menu:request', 'legion') end },
            { title = 'Airport', onSelect = function() TriggerServerEvent('qu_teleport_menu:request', 'airport') end },
        },
    })
    lib.showContext('qu_teleport_menu_menu')
end, false)

RegisterNetEvent('qu_teleport_menu:approved', function(destination)
    local c = spots[destination]
    if not c then return end
    DoScreenFadeOut(500)
    Wait(600)
    SetEntityCoords(PlayerPedId(), c.x, c.y, c.z, false, false, false, false)
    DoScreenFadeIn(500)
    print('[qu_teleport_menu] moved to ' .. destination)
end)
    ```
  </Step>

  <Step num={5} title="Start and test it" outcome="The expected proof appears in the correct console.">
    Open `server.cfg` and add this line (after the line that ensures ox_lib):

    ```text
    ensure qu_teleport_menu
    ```

    Save, then run this in the server console (txAdmin Live Console or the FXServer console). Because this resource has never started yet, use `ensure`, which starts it if it is stopped (`restart` only works once a resource is already running):

    ```text
    refresh
    ensure qu_teleport_menu
    ```

    After this first start, you can use `restart qu_teleport_menu` whenever you edit the code and want to reload it.

    Join the server, then run this test in the chat box (press T):

    ```text
    /tpmenu
    ```

    Choose `Legion Square` from the menu. Your screen fades out, you arrive at Legion, and the screen fades back in.

    <ExpectedOutput>
      F8: [qu_teleport_menu] moved to legion
      Server console: [qu_teleport_menu] approved source 1 for legion
    </ExpectedOutput>
  </Step>
</StepGroup>

## How it works

You ran it and you teleported. Now take the code apart so the next time you build a menu-driven action you are deciding, not copying. Three ideas carry this lesson: the ox_lib context menu, the client to server to client round-trip that lets the server approve the move, and the fade timing. We will walk each one.

### The ox_lib context menu

ox_lib is a Community Ox utility library. It is not shipped by Cfx.re, so you install it yourself and start it before any resource that uses it. The `shared_script '@ox_lib/init.lua'` line in the manifest pulls ox_lib's helper table into your resource. From that point on, `lib` is a global you can call. If ox_lib is not started, `lib` is `nil` and every line that touches it errors. That is the most common failure in this lesson.

```lua
lib.registerContext({
    id = 'qu_teleport_menu_menu',
    title = 'Teleport',
    options = {
        { title = 'Legion Square', onSelect = function() TriggerServerEvent('qu_teleport_menu:request', 'legion') end },
        { title = 'Airport', onSelect = function() TriggerServerEvent('qu_teleport_menu:request', 'airport') end },
    },
})
lib.showContext('qu_teleport_menu_menu')
```

A **context menu** is a list of options the player clicks through. `lib.registerContext` describes the menu but does not display it. You hand it a table with an `id` (a unique name you choose so you can refer to it later), a `title` shown at the top, and an `options` array. Each option is its own table with a `title` the player reads and an `onSelect` function that runs when the player clicks that row. `lib.showContext` takes the `id` you registered and actually opens the menu on screen. Registering and showing are two separate calls on purpose: you can register a menu once at startup and show it many times.

Look closely at what `onSelect` does. It does **not** teleport. It calls `TriggerServerEvent('qu_teleport_menu:request', 'legion')`, which sends a message to the server saying "this player wants to go to legion." That word, `legion`, is the only thing that crosses the network. The client never decides the coordinates here. That sets up the whole point of the lesson.

### Why the server checks the destination

This round-trip controls what the **resource** will approve. It is useful when a destination requires an ACE, job, item, cooldown, or audit entry. It does not prove that the player's whole client is trustworthy: a modified client can call `SetEntityCoords` or locally trigger a client handler without asking this server event. Protect the valuable server-side result, such as entering a heist state or opening a stash, with its own server check.

```lua
local allowed = { legion = true, airport = true }

RegisterNetEvent('qu_teleport_menu:request', function(destination)
    local src = source
    if not allowed[destination] then
        print('[qu_teleport_menu] rejected source ' .. src .. ' for ' .. tostring(destination))
        return
    end
    print('[qu_teleport_menu] approved source ' .. src .. ' for ' .. destination)
    TriggerClientEvent('qu_teleport_menu:approved', src, destination)
end)
```

`allowed` is an **allow-list**: a table of the only destination names the server will accept. `RegisterNetEvent` registers a handler that runs on the server when a client fires `qu_teleport_menu:request`. Inside a server net event, the global `source` holds the server ID of the player who sent it, so the first line copies it into `src` before anything else can overwrite it. The guard `if not allowed[destination] then ... return end` is the gate: if the player sent a name that is not in the allow-list, the server logs a rejection and returns, and nothing happens. Only when the name passes does the server fire `TriggerClientEvent('qu_teleport_menu:approved', src, destination)`, which sends an approval back to that one player by their `src`.

Notice the shape of the trip. Client to server with `TriggerServerEvent`, then server back to one client with `TriggerClientEvent` plus the target `src`. That out-and-back is the round-trip. A normal client cannot make this resource approve a name outside `allowed`. A modified client can still teleport locally, so never use this table as the only protection for money, items, doors, stashes, or mission state.

<Checkpoint question="A player edits their client.lua to add a third menu option that sends 'bank_vault' to the server. Do they teleport into the vault?">
  This resource will not approve `bank_vault`: the server checks `allowed['bank_vault']`, gets `nil`, and returns. That does not stop an exploited client from moving its own ped by another route. Any reward, stash, mission state, or permission at the destination still needs a separate server-side check.
</Checkpoint>

### The fade and the teleport

The approval comes back to the client, and only now does the actual move happen.

```lua
RegisterNetEvent('qu_teleport_menu:approved', function(destination)
    local c = spots[destination]
    if not c then return end
    DoScreenFadeOut(500)
    Wait(600)
    SetEntityCoords(PlayerPedId(), c.x, c.y, c.z, false, false, false, false)
    DoScreenFadeIn(500)
    print('[qu_teleport_menu] moved to ' .. destination)
end)
```

`spots` is the client-side table that maps each approved name to a `vec3`, a value holding x, y, and z coordinates. The server sent the name; the client looks up the coordinates. The guard `if not c then return end` protects against a name with no matching spot.

The four lines after that are a small choreography, and the order matters:

- `DoScreenFadeOut(500)` fades the screen to black over 500 milliseconds. The number is the duration in ms.
- `Wait(600)` pauses this handler for 600 ms. Net event handlers run in a thread that can yield, so `Wait` is allowed here. This number is deliberately larger than the fade. You want the screen fully black **before** you move the player, so they never see the world snap. If you teleport at 400 ms while the fade is still finishing, the player catches a glimpse of the jump. Wait longer than the fade, always.
- `SetEntityCoords(PlayerPedId(), c.x, c.y, c.z, false, false, false, false)` does the move. `PlayerPedId()` returns the handle of the player's own character on this client. The three coordinates are the destination. The four `false` flags are, in order, `alive`, `deadFlag`, `ragdollFlag` (low-level physics flags the engine uses for dead and ragdolling peds; `alive` is effectively unused), and `clearArea` (when true, clears nearby entities at the destination). For a plain on-foot teleport, all four false is fine. (Some third-party docs mislabel the first three as `xAxis/yAxis/zAxis`; FiveM's native reference names them `alive/deadFlag/ragdollFlag`.)
- `DoScreenFadeIn(500)` fades back from black now that the player is standing in the new spot.

So the sequence reads: black out, wait until fully black, move, fade back in. The teleport hides inside the black frame, which is what makes it feel smooth instead of jarring.

### When you reach for this pattern

The client-asks, server-approves, server-answers round-trip is the backbone of almost every action a player can trigger that the server must police. You just built it once; here is where it shows up again:

- **Job and shop menus.** A player clicks "buy the rifle." The client sends the item id, the server checks the player can afford it and is allowed to buy it, then the server grants it. The client never sets its own money.
- **Admin teleports and noclip.** Same menu shape, but the server's allow-list is "is this player an admin" instead of a list of place names.
- **Any menu that grants something.** Garages, stashes, faction doors. The menu is just the UI. The decision always belongs on the server.

When the action is purely cosmetic and local, like opening a settings panel, you can skip the server trip. When it changes authoritative game state, route the request through the server and validate again at the protected action. A server-approved client teleport alone is not a security boundary.

## A second shape: the /tp [number] command

The menu you built constrains the player to options you defined, so the only thing that crosses the network is a name you chose. There is a second teleport shape you will see constantly, especially for quick solo testing and admin tools: a command that takes a typed argument, like `/tp 2`. Here the player types a value you did not pre-approve, so the lesson shifts from "click a safe option" to "parse and validate whatever they typed."

This variant is client-only on purpose. It is a builder/admin convenience, not a player-facing feature, so it skips the server round-trip. That trade is fine for a tool you gate behind a permission, and dangerous if you ever expose it to everyone. We will come back to that.

<Callout variant="fivem" title="What changes versus the menu">
  The menu's `onSelect` already knew exactly which destination it meant, so there was nothing to parse. A command argument is raw text the player typed. Before you act on it you must turn it into a number and prove it points at a real destination. That parse-and-check step is the whole new idea in this section.
</Callout>

```lua
local teleportLocations = {
    { x = -75.0, y = -819.0, z = 326.0, name = 'Main Rooftop' },
    { x = -428.0, y = 1111.0, z = 327.0, name = 'Vinewood Hills' },
    { x = 402.0, y = -996.0, z = -99.0, name = 'Hidden Bunker' },
}

RegisterCommand('tp', function(source, args)
    local index = tonumber(args[1])
    if not index or not teleportLocations[index] then
        print('[qu_teleport_menu] /tp needs a number 1 to ' .. #teleportLocations)
        return
    end

    local spot = teleportLocations[index]
    DoScreenFadeOut(500)
    Wait(600)
    SetEntityCoords(PlayerPedId(), spot.x, spot.y, spot.z, false, false, false, true)
    DoScreenFadeIn(500)
    print('[qu_teleport_menu] moved to ' .. spot.name)
end, false)
```

Read it argument first. `RegisterCommand` hands your function two things: `source` (the server ID of whoever ran it; on the client this is your own player) and `args`, an array of the words typed after the command. When the player types `/tp 2`, `args[1]` is the **string** `"2"`, not the number `2`. That distinction is the trap. `tonumber(args[1])` converts that string to a number, and returns `nil` if the player typed something that is not a number, like `/tp rooftop`.

The guard `if not index or not teleportLocations[index] then` does two checks in one line. `not index` catches the case where `tonumber` returned `nil` (the player typed nonsense or nothing at all). `not teleportLocations[index]` catches a number that is out of range, like `/tp 9` when there are only three spots: indexing past the end of the array returns `nil`. Either failure prints a usage hint and returns, so an empty or garbage argument never crashes the handler. `#teleportLocations` is the length of the array, so the hint always names the real upper bound even after you add spots.

Notice the last `SetEntityCoords` flag here is `true`, not `false` like in the menu build. That final flag is `clearArea`: with `true` the engine clears nearby entities at the destination, which is handy when you teleport onto a rooftop or into a tight interior and do not want to land inside a parked car. The three flags before it are still `alive`, `deadFlag`, and `ragdollFlag`, unchanged from the menu build.

To try it, add this code to a client file in `qu_teleport_menu`, run `restart qu_teleport_menu` in the **server console** (txAdmin Live Console or the FXServer console), then in the **chat box** in game (press T) type `/tp 1`. Try `/tp 9` and `/tp banana` too, and watch the usage hint print to **F8** instead of the handler breaking.

<ExpectedOutput>
  F8 (after /tp 1): [qu_teleport_menu] moved to Main Rooftop
  F8 (after /tp 9): [qu_teleport_menu] /tp needs a number 1 to 3
</ExpectedOutput>

<Callout variant="warning" title="This shape has no server check">
  Because there is no `TriggerServerEvent` here, the server never sees or approves the move. That is acceptable for a builder tool you gate with a restricted command or an ACE, and unacceptable as a player feature: a player who can run `/tp` can reach any coordinate in your array, and an exploited client can reach any coordinate at all. If you want players to use numbered destinations, send `index` to the server and re-run the allow-list check there, exactly like the menu build did with a name. Convenience commands stay on the client only when you also restrict who can run them.
</Callout>

## If something went wrong

<FailureList
  items={[
    { symptom: "attempt to index a nil value (global 'lib')", fix: "ox_lib is not started. Add ensure ox_lib to server.cfg before the line that ensures qu_teleport_menu, then restart. The @ox_lib/init.lua import only works when ox_lib is running first." },
    { symptom: "The menu opens but choosing a place does nothing", fix: "The destination name in onSelect does not match a key in the server allowed table or the client spots table. Check that 'legion' and 'airport' are spelled the same in all three places." },
    { symptom: "Server console shows rejected source N for ...", fix: "That destination is not in the server allowed table. This is the server doing its job. Add the name to allowed on the server and to spots on the client if it is a real destination." },
    { symptom: "Screen fades but you do not move, or the world snaps without a clean fade", fix: "Confirm the Wait after DoScreenFadeOut is larger than the fade duration. With DoScreenFadeOut(500) the wait should be 600 or more so the screen is black before SetEntityCoords runs." },
    { symptom: "You teleport but fall through the map, land in the air, or spawn under the ground at a far-away or high destination", fix: "The destination's collision and ground have not streamed in yet, so the engine has nothing solid to put you on. Before the final move, request the collision and wait for the real ground height. Replace the bare SetEntityCoords with: first place the ped roughly with SetEntityCoords(ped, x, y, z, false, false, false, true), then RequestCollisionAtCoord(x, y, z), then loop a few frames calling local found, groundZ = GetGroundZFor_3dCoord(x, y, z + 5.0, false) with a Wait(0) each loop until found is true, then SetEntityCoords(ped, x, y, groundZ, false, false, false, true). The +5.0 starts the ground probe slightly above your target so it casts downward onto the surface. Note GetGroundZFor_3dCoord returns two values, the found boolean first and the height second." },
  ]}

## Try it yourself

<Exercise>
  Add an admin-only destination that the menu offers to everyone but the server refuses unless the player is an admin. Add a third option `Police Armory` to the client menu that sends `armory`, and add `armory` to the client `spots` table with any coordinates you like. On the server, do not put `armory` in the plain `allowed` table. Instead, inside the request handler, check the player's permission before approving it:

  ```lua
  if destination == 'armory' then
      if not IsPlayerAceAllowed(src, 'qu_teleport_menu.armory') then
          print('[qu_teleport_menu] denied armory to source ' .. src)
          return
      end
      TriggerClientEvent('qu_teleport_menu:approved', src, destination)
      return
  end
  ```

  Grant yourself the ace in server.cfg with `add_ace identifier.fivem:YOUR_ID qu_teleport_menu.armory allow`, restart, and confirm you reach the armory while a second account without the ace gets the denied line in the server console. You have now built a menu where the same UI gives different power to different players, with the server as the only judge.
</Exercise>

  Reading and writing the database on every high-frequency action adds avoidable load. The cache plus write-back pattern keeps working state in RAM, marks it dirty when it changes, and flushes snapshots on a timer and at graceful lifecycle points. It reduces writes in exchange for a bounded crash-loss window, so it is a deliberate tradeoff rather than a guarantee that no data can ever be lost.

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_persistence
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_persistence/
  fxmanifest.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script '@oxmysql/lib/MySQL.lua'
server_script 'server.lua'

dependencies {
    'oxmysql'
}
    ```

    `oxmysql` is the database library this lesson talks to. It is an Overextended (community) resource, not something Cfx.re ships in the box, so you install and start it yourself. The `dependencies { 'oxmysql' }` block tells FiveM to load oxmysql before this resource, so the `MySQL` global exists by the time your `server.lua` runs. You will see the singular `dependency 'oxmysql'` in older tutorials. It still works; the docs list both forms, and the plural block is the one to reach for when a resource has more than one dependency.

    Now create the table. This lesson assumes MariaDB, which is the 2026-preferred database for FiveM servers. oxmysql speaks to MariaDB and MySQL identically, so the SQL below is the same either way. Run it once in a database client that is connected to the same database your `mysql_connection_string` points at (for example HeidiSQL, DBeaver, or phpMyAdmin), before you start the resource:

    ```sql
CREATE TABLE IF NOT EXISTS qu_persistence_rows (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(80) NOT NULL,
  amount INT NOT NULL DEFAULT 0,
  UNIQUE KEY uniq_name (name)
);
    ```

    The `UNIQUE KEY uniq_name (name)` line is load-bearing and the next section explains why. Without it, the write-back silently breaks.
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `server.lua` and paste this:

    ```lua
local cache = {
    cash = 0,
    dirty = false,
    version = 0,
    loaded = false,
    flushing = false,
}

local function flush()
    if not cache.loaded or not cache.dirty or cache.flushing then return false end

    cache.flushing = true
    local amount = cache.cash
    local version = cache.version
    local ok, err = pcall(MySQL.query.await,
        'INSERT INTO qu_persistence_rows (name, amount) VALUES (?, ?) ON DUPLICATE KEY UPDATE amount = VALUES(amount)',
        { 'wallet', amount }
    )
    cache.flushing = false

    if not ok then
        print('[qu_persistence] flush failed: ' .. tostring(err))
        return false
    end

    -- Do not clear a newer change that happened while the query was awaiting.
    if cache.version == version then cache.dirty = false end
    print('[qu_persistence] flushed ' .. amount)
    return true
end

RegisterCommand('cashadd', function(_, args)
    if not cache.loaded then
        print('[qu_persistence] wallet is still loading, try again')
        return
    end

    local amount = tonumber(args[1])
    if not amount then
        print('[qu_persistence] usage: cashadd <number>')
        return
    end

    cache.cash = cache.cash + amount
    cache.version = cache.version + 1
    cache.dirty = true
    print('[qu_persistence] cached ' .. cache.cash)
end, true)

CreateThread(function()
    local saved = MySQL.scalar.await(
        'SELECT amount FROM qu_persistence_rows WHERE name = ?',
        { 'wallet' }
    )
    cache.cash = tonumber(saved) or 0
    cache.loaded = true
    print('[qu_persistence] loaded ' .. cache.cash)

    while true do
        Wait(30000)
        flush()
    end
end)

AddEventHandler('playerDropped', function()
    flush()
end)

AddEventHandler('onResourceStop', function(resource)
    if resource ~= GetCurrentResourceName() then return end
    flush()
end)
    ```
  </Step>

Run `SELECT amount FROM qu_persistence_rows WHERE name = 'wallet'` in your database tool and confirm there is exactly one row, not a new one for every flush.
  </Step>

## How it works

You ran it and the wallet flushed once, into one row. Now take the code apart so you can build a real economy on top of it instead of copying a snippet. There are four moving parts: the cache, the dirty flag, the flush function, and the three things that call it.

### The cache and the dirty flag

```lua
local cache = { cash = 0, dirty = false, version = 0, loaded = false, flushing = false }
```

`cache.cash` is the live wallet value held in RAM. `loaded` prevents a command from changing the default zero before the database read finishes. `flushing` prevents two lifecycle paths from writing concurrently. `version` increments on every mutation so a slow write cannot clear a newer dirty change.

`cache.dirty` answers one question: has anything changed since the last successful save? It starts `false`; after the startup query loads the saved amount, RAM and the database match. Every mutation increments `version` and sets `dirty = true`. If nothing changed, the timer performs no query.

```lua
RegisterCommand('cashadd', function(_, args)
    if not cache.loaded then return end
    local amount = tonumber(args[1])
    if not amount then return end
    cache.cash = cache.cash + amount
    cache.version = cache.version + 1
    cache.dirty = true
    print('[qu_persistence] cached ' .. cache.cash)
end, true)
```

`cashadd` stands in for a mutation. It first refuses work until hydration is complete, validates the number, changes the value, increments the version, and marks the cache dirty. The `true` flag makes the command require the `command.cashadd` ACE; the server console can run it. The command never writes directly to the database.

### The flush function

```lua
local function flush()
    if not cache.loaded or not cache.dirty or cache.flushing then return false end
    cache.flushing = true
    local amount = cache.cash
    local version = cache.version
    local ok, err = pcall(MySQL.query.await,
        'INSERT INTO qu_persistence_rows (name, amount) VALUES (?, ?) ON DUPLICATE KEY UPDATE amount = VALUES(amount)',
        { 'wallet', amount }
    )
    cache.flushing = false
    if not ok then return false end
    if cache.version == version then cache.dirty = false end
    print('[qu_persistence] flushed ' .. amount)
    return true
end
```

Pulling the write into its own function matters, because three different events all need to save and you do not want three copies of the query drifting apart. Read it top to bottom:

- The first guard skips work until loading finishes, when nothing changed, or while another flush is already in flight.
- `amount` and `version` snapshot exactly what this query is saving. `pcall` keeps `dirty` set when the query errors so a later timer can retry.
- `MySQL.query.await(...)` runs the SQL and blocks this coroutine until the database answers. We use `MySQL.query`, not `MySQL.insert`. `MySQL.insert` is built to return the new row's `insertId`, which is meaningful only when you are actually inserting. Our statement updates an existing row most of the time, so that id would be meaningless and would only reinforce the duplicate-row bug. `MySQL.query` is the right helper for an upsert.
- The `?` placeholders are parameterized query slots. The values come from the second argument, the table `{ 'wallet', cache.cash }`, in order. Never build SQL by concatenating strings. Parameterized values are how you stay safe from injection, and they are the only correct way to pass data into a query.
- `ON DUPLICATE KEY UPDATE amount = VALUES(amount)` is the upsert. It tells the database: try to insert this row, but if a row with the same unique key already exists, update its `amount` instead. `VALUES(amount)` means use the amount I just tried to insert.

Here is the trap the original version of this lesson fell into. `ON DUPLICATE KEY UPDATE` only fires when the insert collides with a `PRIMARY` or `UNIQUE` key. The first schema had only `id INT AUTO_INCREMENT PRIMARY KEY` and a plain, non-unique `name` column. Nothing ever collided, so every single flush inserted a brand new row and the update branch never ran. You would end up with hundreds of `wallet` rows and a wallet that never actually saved. Adding `UNIQUE KEY uniq_name (name)` to the table is what makes the collision happen, which is what makes the update happen, which is what makes write-back work at all. The unique key is not decoration. It is the mechanism.

After the write succeeds, the function clears `dirty` only if `version` is unchanged. If another command changed cash while the query yielded, the newer version remains dirty and the next flush saves it. Clearing `dirty` unconditionally after an awaited query creates a lost-update race.

### Loading before accepting changes

```lua
local saved = MySQL.scalar.await(
    'SELECT amount FROM qu_persistence_rows WHERE name = ?',
    { 'wallet' }
)
cache.cash = tonumber(saved) or 0
cache.loaded = true
```

Persistence has two halves: save and load. The startup thread reads the existing amount before enabling `cashadd`. Without this hydration step, the database could contain `25` while every restart rebuilt the live cache as `0`; the next change would overwrite good data from the wrong starting point.

<Checkpoint question="Why does the wallet end up as a single row that updates, instead of a new row every 30 seconds?">
  Because the table has a `UNIQUE KEY` on `name` and the query uses `ON DUPLICATE KEY UPDATE`. The first flush inserts the `wallet` row. Every flush after that tries to insert a second `wallet` row, collides with the unique key, and the database runs the `UPDATE amount = VALUES(amount)` branch instead, overwriting the existing row in place. Remove the `UNIQUE KEY` and the collision never happens, so the database falls back to a plain insert every time and you get a fresh duplicate row on every flush. The unique constraint is the thing that turns INSERT into upsert.
</Checkpoint>

### The three things that call flush

A timer alone leaves a loss window. A 30-second interval means an ungraceful crash can lose up to roughly 30 seconds of changes. Disconnect and stop handlers reduce that window during orderly lifecycle events, but they cannot make process crashes or database outages lossless.

```lua
CreateThread(function()
    while true do
        Wait(30000)
        flush()
    end
end)
```

This is the periodic flush. A thread loops forever, sleeps 30 seconds with `Wait(30000)`, then calls `flush`. This is the steady-state save that keeps the database roughly current while the server runs. The 30-second number is a deliberate tradeoff: longer means fewer writes but a bigger loss window, shorter means more writes but a smaller window. You tune it to how much state you can afford to lose.

```lua
AddEventHandler('playerDropped', function()
    flush()
end)
```

`playerDropped` is a server event that fires the moment a player disconnects. This is the per-player save: catch the player on the way out and write their state immediately, instead of waiting up to 30 seconds for the timer and possibly losing their last actions. In a real economy you would flush only that one player's data here, keyed by their identifier, rather than a single global wallet.

```lua
AddEventHandler('onResourceStop', function(resource)
    if resource ~= GetCurrentResourceName() then return end
    flush()
end)
```

`onResourceStop` fires when a resource is stopping, so the first thing you do is check the name against `GetCurrentResourceName()`. The flush is best effort during an orderly resource restart. It is not a durability guarantee for a process kill, host crash, or database outage. For scheduled maintenance, expose a save command or maintenance flow that flushes and confirms success before txAdmin stops the server.

### When you reach for this pattern

Reach for cache plus write-back when a value changes more often than it needs durable writes. If some loss after an ungraceful crash is acceptable, hydrate into RAM, mark mutations dirty, and flush snapshots on a timer plus graceful lifecycle points. If a write must be durable before the action is acknowledged, such as a purchase receipt or money transfer, write through immediately and use a database transaction when multiple rows must change together.

## Try it yourself

<Exercise>
  Turn the single global wallet into a per-player wallet. Keep the live cache indexed by player source only while the player is connected, but persist with a stable identifier such as `GetPlayerIdentifierByType(src, 'license')` or your framework's character ID. Never store a server source as the durable key: sources are temporary slots and can be reused after reconnects.

  Then make the save points per-player. In the `playerDropped` handler, read `local src = source` and flush only that player's entry, then drop it from the cache so the table does not grow forever. In the timer and in `onResourceStop`, loop over every cached player and flush each dirty one.

  Hydrate each player from their stable key before accepting changes. When you finish, test two players, disconnect one, and confirm the database holds two distinct rows that update in place rather than duplicating. Also repeat the version check from the main lesson per player so a write in flight cannot clear a newer mutation.
</Exercise>

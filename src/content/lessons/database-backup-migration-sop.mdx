## The routine

<StepGroup>
 <Step num={1} title="Take a full dump on a schedule" outcome="You have a recent, complete copy of the database.">
 Export the whole database to a `.sql` file on a schedule (daily for an active server). In HeidiSQL: right-click the database, Export database as SQL, to a dated file. From the command line, the dump tool does the same.

 Run the command below in a terminal **on the machine where MariaDB is installed** (a Linux shell, or on Windows the PowerShell/Command Prompt window where the MariaDB `bin` folder is on your PATH - that is where `mariadb-dump` lives):

 ```bash
 mariadb-dump -u root -p fivem > fivem-2026-06-12.sql
 ```

 What each part means: `-u root` is the database username (use your DB user, not necessarily `root`); `-p` makes it prompt you for that user's password (type it when asked - nothing shows as you type, that is normal); `fivem` is the database name to back up (use yours); and `> fivem-2026-06-12.sql` writes the dump into a file with today's date in the current folder.

 On a pure MariaDB box the dump tool is `mariadb-dump`. The old `mysqldump` name still works because MariaDB ships it as a compatibility symlink, but if `mysqldump` returns `command not found`, use `mariadb-dump` instead. They take the same flags.

 To make daily real instead of aspirational, run that command on a schedule. On Linux, a cron line (added with `crontab -e`) that dumps every night at 4am:

 ```bash
 0 4 * * * /usr/bin/mariadb-dump --defaults-extra-file=/root/.mariadb-backup.cnf --single-transaction --routines fivem > /backups/fivem-$(date +\%F).sql
 ```

 Reading it left to right: `0 4 * * *` is the cron schedule (minute 0 of hour 4, every day); `--defaults-extra-file=...` points at a credentials file so no password is typed on the line; `--single-transaction` takes a consistent snapshot of InnoDB tables without blocking your players' writes; `--routines` includes stored procedures/functions; `fivem` is the database; and `> /backups/fivem-$(date +\%F).sql` writes to a dated file. The backslash in `\%F` is required because cron treats a bare `%` as a newline - escaping it lets `date +%F` insert today's date (YYYY-MM-DD).

 Put the backup user's credentials in `/root/.mariadb-backup.cnf` so the password never appears in the command. That file is plain text in MariaDB option-file format, for example:

 ```ini
 [mariadb-dump]
 user=backup_user
 password=your-backup-user-password
 ```

 Then lock it down so only its owner can read it (`chmod 600 /root/.mariadb-backup.cnf`), and never place a password directly in a cron line or command history. On Windows, use Task Scheduler with a protected option file. txAdmin documents scheduled **restarts and events**, not a general operating-system task runner, so do not rely on it to execute a backup script.
 </Step>

 <Step num={2} title="Store backups off the server box" outcome="A server failure does not take the backups with it.">
 Copy dumps somewhere other than the server machine (another disk, a private bucket). A backup that dies with the server is not a backup.
 </Step>

 <Step num={3} title="Migrate schema safely" outcome="Adding columns/tables never destroys existing data.">
 Before any schema change: take a fresh dump, run the change on a throwaway copy first, and prefer additive changes (`ADD COLUMN`) over destructive ones. Never run an untested `ALTER`/`DROP` against live data.

 First build the copy from your latest dump and run the migration there:

 ```bash
 mariadb -u root -p -e "CREATE DATABASE fivem_migrate_test;"
 mariadb -u root -p fivem_migrate_test < fivem-2026-06-12.sql
 mariadb -u root -p fivem_migrate_test -e "ALTER TABLE players ADD COLUMN bank INT NOT NULL DEFAULT 0;"
 ```

 Only once that runs clean on the copy do you touch production. Put the server in maintenance and stop the resources that write to the affected tables so application code cannot race the migration. Then run the change and bring the resources back only after validation:

 ```sql
 ALTER TABLE players ADD COLUMN bank INT NOT NULL DEFAULT 0;
 ```
 </Step>

 <Step num={4} title="Run a restore test" outcome="You have proven you can bring data back.">
 Create a throwaway database and import a dump into it. This is the step everyone skips and everyone regrets skipping.

 ```bash
 mariadb -u root -p -e "CREATE DATABASE fivem_restore_test;"
 mariadb -u root -p fivem_restore_test < fivem-2026-06-12.sql
 ```

 Do not eyeball it. Prove the restore by comparing a real row count per table between production and the restore. Open an interactive client session by running `mariadb -u root -p` (it prompts for your password, then shows a `MariaDB [(none)]>` prompt), then type each query and press Enter:

 ```sql
 SELECT COUNT(*) FROM fivem.players;
 SELECT COUNT(*) FROM fivem_restore_test.players;
 ```

 The interactive client prints a boxed table like this (the number 1487 is an example - yours will be your own row count):

 <ExpectedOutput>
 +----------+
 | COUNT(*) |
 +----------+
 | 1487 |
 +----------+
 +----------+
 | COUNT(*) |
 +----------+
 | 1487 |
 +----------+
 </ExpectedOutput>

 Note: if you instead run these with the `-e "..."` flag from the shell (as in the exercise below), the client prints plain tab-separated output (a `COUNT(*)` header line, then the number) rather than the boxed table - the value is the same, only the formatting differs.

 The two counts match. If they differ, the dump is incomplete or the import errored, and that backup is not safe to rely on.
 </Step>
</StepGroup>

## Common mistakes

 items={[

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| Backup file is 0 bytes or corrupted | mysqldump failed silently | Run the dump command manually and check for errors. Common causes: wrong password, disk full, or MySQL not running |
| Restore fails with "table already exists" | The backup includes CREATE TABLE statements that conflict with existing tables | Drop the existing tables first, or use `--add-drop-table` in your mysqldump command |
| Backup takes too long and times out | The database is too large for a single dump | Back up individual tables: `mysqldump database_name table_name > backup.sql` |
| Can't find the backup when you need it | You didn't name it clearly or saved it somewhere random | Always name backups with the date: `backup-2026-06-19.sql`. Keep them in ONE folder: `C:ackups\` |

## Try it yourself

<Exercise>
 Run one full backup-and-restore-test cycle on your own database, end to end, and report the numbers.

 1. Take a dated dump of your live database with `mariadb-dump`.
 2. Create `fivem_restore_test`, import the dump into it.
 3. Pick one table that actually has rows on your server, for example `players` or `users`, and run `SELECT COUNT(*)` against both the live database and the restore.

 ```bash
 mariadb-dump -u root -p fivem > fivem-test.sql
 mariadb -u root -p -e "CREATE DATABASE fivem_restore_test;"
 mariadb -u root -p fivem_restore_test < fivem-test.sql
 mariadb -u root -p -e "SELECT COUNT(*) FROM fivem.players;"
 mariadb -u root -p -e "SELECT COUNT(*) FROM fivem_restore_test.players;"
 ```

 The two counts must match exactly. Write down both numbers. If they match, you have a backup you have actually restored, which is the only kind that counts. When you are done, drop the test database with `DROP DATABASE fivem_restore_test;` so it does not sit around getting stale.
</Exercise>

## The routine

<StepGroup>
  <Step num={1} title="Take a full dump on a schedule" outcome="You have a recent, complete copy of the database.">
    Export the whole database to a `.sql` file on a schedule (daily for an active server). In HeidiSQL: right-click the database, Export database as SQL, to a dated file. From the command line, the dump tool does the same.

    ```bash
    mariadb-dump -u root -p fivem > fivem-2026-06-12.sql
    ```

    On a pure MariaDB box the dump tool is `mariadb-dump`. The old `mysqldump` name still works because MariaDB ships it as a compatibility symlink, but if `mysqldump` returns `command not found`, use `mariadb-dump` instead. They take the same flags.

    To make daily real instead of aspirational, run that command on a schedule. On Linux, a cron line that dumps every night at 4am:

    ```bash
    0 4 * * * /usr/bin/mariadb-dump -u root -pYOURPASS fivem > /backups/fivem-$(date +\%F).sql
    ```

    On Windows, create a Task Scheduler task that runs the same `mariadb-dump` line in a `.bat` file. If you run txAdmin, its scheduled-tasks feature can fire a backup script on the same cadence as your restarts.
  </Step>

  <Step num={2} title="Store backups off the server box" outcome="A server failure does not take the backups with it.">
    Copy dumps somewhere other than the server machine (another disk, a private bucket). A backup that dies with the server is not a backup.
  </Step>

  <Step num={3} title="Migrate schema safely" outcome="Adding columns/tables never destroys existing data.">
    Before any schema change: take a fresh dump, run the change on a throwaway copy first, and prefer additive changes (`ADD COLUMN`) over destructive ones. Never run an untested `ALTER`/`DROP` against live data.

    First build the copy from your latest dump and run the migration there:

    ```bash
    mariadb -u root -p -e "CREATE DATABASE fivem_migrate_test;"
    mariadb -u root -p fivem_migrate_test < fivem-2026-06-12.sql
    mariadb -u root -p fivem_migrate_test -e "ALTER TABLE players ADD COLUMN bank INT NOT NULL DEFAULT 0;"
    ```

    Only once that runs clean on the copy do you touch production. Stop writes first: a live `ALTER` while oxmysql is still writing can leave half-applied rows. Stop the resources that write to this database (or `stop`/`ensure` the framework), run the change, then bring them back:

    ```sql
    ALTER TABLE players ADD COLUMN bank INT NOT NULL DEFAULT 0;
    ```
  </Step>

  <Step num={4} title="Run a restore test" outcome="You have proven you can bring data back.">
    Create a throwaway database and import a dump into it. This is the step everyone skips and everyone regrets skipping.

    ```bash
    mariadb -u root -p -e "CREATE DATABASE fivem_restore_test;"
    mariadb -u root -p fivem_restore_test < fivem-2026-06-12.sql
    ```

    Do not eyeball it. Prove the restore by comparing a real row count per table between production and the restore. Run the same `SELECT COUNT(*)` against both:

    ```sql
    SELECT COUNT(*) FROM fivem.players;
    SELECT COUNT(*) FROM fivem_restore_test.players;
    ```

    <ExpectedOutput>
      +----------+
      | COUNT(*) |
      +----------+
      |     1487 |
      +----------+
      +----------+
      | COUNT(*) |
      +----------+
      |     1487 |
      +----------+
    </ExpectedOutput>

    The two counts match. If they differ, the dump is incomplete or the import errored, and that backup is not safe to rely on.
  </Step>
</StepGroup>

## Common mistakes

## Try it yourself

<Exercise>
  Run one full backup-and-restore-test cycle on your own database, end to end, and report the numbers.

  1. Take a dated dump of your live database with `mariadb-dump`.
  2. Create `fivem_restore_test`, import the dump into it.
  3. Pick one table that actually has rows on your server, for example `players` or `users`, and run `SELECT COUNT(*)` against both the live database and the restore.

  ```bash
  mariadb-dump -u root -p fivem > fivem-test.sql
  mariadb -u root -p -e "CREATE DATABASE fivem_restore_test;"
  mariadb -u root -p fivem_restore_test < fivem-test.sql
  mariadb -u root -p -e "SELECT COUNT(*) FROM fivem.players;"
  mariadb -u root -p -e "SELECT COUNT(*) FROM fivem_restore_test.players;"
  ```

  The two counts must match exactly. Write down both numbers. If they match, you have a backup you have actually restored, which is the only kind that counts. When you are done, drop the test database with `DROP DATABASE fivem_restore_test;` so it does not sit around getting stale.
</Exercise>

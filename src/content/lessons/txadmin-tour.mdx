>
 txAdmin is the cockpit for your server. It ships **bundled inside FXServer**, so there is nothing extra to install. Six panels do 95% of the work: **Live Console**, **Players**, **Resources**, **Recipes**, **Settings**, and the in-game **txAdmin menu** (opened with `/tx` or your assigned txAdmin menu keybind). We'll click through them all so nothing feels mysterious, and you'll send a live `status` command before the end.
</LessonHero>

<Callout variant="fivem" title="What txAdmin actually is, in one minute">
 txAdmin is a web app that wraps your FXServer process. The server runs as a child of txAdmin: it watches the server, restarts it on crash, and gives you a dashboard so you never have to dig through a raw console window. Think of it as the nurse station outside the kitchen. The kitchen (FXServer) does the cooking. txAdmin tells you who is at the table, what is burning, and lets you send orders without walking in. Do one step, read the proof, then move on.
</Callout>

## Build it

<StepGroup>
 <Step num={1} title="Open the dashboard and read the heartbeat" outcome="You can send a command to your running server and read its reply.">
 Open your browser to the dashboard:

 ```text
 http://localhost:40120
 ```

 Confirm the top-left badge is green and says **Server Online**. In the left sidebar, click **Live Console**. You'll see a black terminal scrolling text: every line FXServer prints, you read here. At the bottom is an input box. Type this and press Enter:

 ```bash
 status
 ```

 The `rconlog` resource provides `status`. It prints the connected-player list (primary identifier, server ID, name, endpoint, and ping). With no players connected, the list is empty. That is your server answering live.

 <ExpectedOutput>
 # Illustrative - exact columns and spacing vary by build. With no players connected you see no player rows.
 # Each connected player prints one line: server ID, primary identifier, name, endpoint, ping.
 </ExpectedOutput>

 Color codes matter: white is info, yellow is a warning, red is an error. If you see red, copy the line and search it. About 90% of red lines are typos in `server.cfg` or a bad `fxmanifest.lua`.
 </Step>

 <Step num={2} title="Find a player in the Players panel" outcome="You can locate any connected player and read their identifiers.">
 In the left sidebar, click **Players**, then the **Online** tab. It lists everyone connected right now (including you, once you join in Step 06). Click any name to open the **Player Modal**, where day-to-day moderation lives:

 ```text
 Info -> identifiers: fivem:, steam:, discord:, license:
 History -> past warnings, kicks, bans
 IDs -> hardware IDs, for ban-evasion checks
 Actions -> Warn / Kick / Ban / DM, each with a reason field
 ```

 The **Info** tab is the one you'll use most. When you connect to your own server later, click your own name and copy the `fivem:` value. That is what you paste into the admin line in `server.cfg`:

 ```text
 add_principal identifier.fivem:1234567 group.admin
 ```
 </Step>

 <Step num={3} title="Restart one resource from the Resources tab" outcome="You can flip a single resource without restarting the whole server.">
 In the left sidebar, click **Resources**. Each row is one resource folder. A green dot means running, grey means stopped, red means crashed. Hover a row to reveal **Start**, **Stop**, and **Restart**. The button at the top of the page, **Refresh List**, re-scans the `resources/` folder for new files you dropped in.

 The core developer loop is: edit a script, save, click **Restart** on that one resource, then alt-tab back into the game. No full server restart needed. You can also do it from the Live Console. Restart the built-in `chat` resource to see it:

 ```bash
 restart chat
 ```

 <ExpectedOutput>
 # Illustrative - you'll see lines confirming chat stopped then started again.
 # Exact wording varies by FXServer build; the point is the console echoes the restart.
 </ExpectedOutput>
 </Step>

 <Step num={4} title="Understand Recipes (and when to use one)" outcome="You know what a recipe is and why you almost never run a second one.">
 Click **Recipes** in the sidebar. A recipe is a file that tells txAdmin: download this list of resources, drop them in `resources/`, and add the matching `ensure` lines to `server.cfg`. One click builds a working server preset. You saw a recipe run in Step 03.

 Recipe availability changes. ESX Legacy, QBCore, and Qbox each publish official sources; Qbox describes itself as a QBCore successor and publishes an official txAdmin recipe. Choose from the current recipe list by checking the framework docs and the compatibility requirements of the resources you intend to install.

 Recipes are one-time scaffolds. Use one to **create a new server**, not to update an existing one. Running a second recipe on a working server overwrites your `server.cfg` and you lose your customizations.
 </Step>

 <Step num={5} title="Edit server.cfg from the browser with the CFG Editor" outcome="You can edit server.cfg in-browser, with highlighting and a save button.">
 Click **Settings**, then the **CFG Editor** tab. Your full `server.cfg` opens in a Monaco editor, the same engine VS Code uses. Scroll to your admin line and confirm your resources are ensured:

 ```text
 ensure chat
 ensure qu_hello
 ```

 Edit, then click **Save**. If a change needs a full restart, txAdmin tells you and you click the **Server Restart** button at the top-right. This editor is the cleanest way to edit a server that runs on a remote box: no SSH, no WinSCP, a password-protected web tab.

 <ExpectedOutput>
 Saved server.cfg. Some changes may require a server restart.
 </ExpectedOutput>
 </Step>

 <Step num={6} title="Set Privacy to Hidden, then open the in-game txAdmin menu" outcome="You hide your dev server from the public list and know how to open the txAdmin menu in-game.">
 Still in **Settings**, open the **Game Mode & Privacy** area. While you're learning, set **Server Privacy** to **Hidden**. A hidden server is not listed in the FiveM browser, so players reach it only through **Direct Connect**. Flip to **Public** when you're ready to advertise. The **Game Mode** label (Roleplay, Drift, Sandbox) is the tag shown next to your server in the browser. Click **Save**.

 Once you connect (Step 06), you can open the in-game txAdmin menu two ways. Either type `/tx` or `/txadmin` in the chat box, or bind a key for it: in the FiveM client open **Settings > Key Bindings > FiveM** and set **(txAdmin) Menu: Open Main Page** to a key of your choice. The txAdmin menu has **no default key**, so it is command-only (`/tx`) until you bind one. The menu gives you teleport, a vehicle spawner, weather and time, player actions, and a Live Editor for placing objects, and it is enabled by default. Access is gated by txAdmin's own account system, **not** by ACE: only someone who has a txAdmin admin account with their **Discord or Cfx.re identifier linked** (Settings > Admin Manager) can open it. If neither identifier is attached to your txAdmin account, the menu will not open for you. This account gate is separate from the `server.cfg` `add_principal`/`group.admin` ACE, which controls console and chat-command permissions, not this menu.

 <ExpectedOutput>
 Privacy saved. Your server is now hidden from the public list.
 </ExpectedOutput>
 </Step>
</StepGroup>

## If something went wrong

<FailureList
 items={[
 { symptom: "Browser shows ERR_CONNECTION_REFUSED at localhost:40120", fix: "FXServer is not running. Re-launch it (run.bat or FXServer), watch its console for crash lines, and reload the page." },
 { symptom: "The Live Console input box is greyed out", fix: "Your txAdmin account lacks command rights. Log in as the master admin you created in Step 03, or promote your account in Settings -> Admin Manager." },
 { symptom: "status replies with red License key error", fix: "Add a free license key from portal.cfx.re to server.cfg as sv_licenseKey, then restart. The key is required even for a local test server." },
 { symptom: "Players panel is empty but I'm in the server", fix: "Refresh the page. The panel polls every few seconds and the first load can miss your join." },
 { symptom: "Typing /tx does nothing in-game", fix: "Two common causes. (1) The menu is disabled: txAdmin's menu is on by default (txAdmin-menuEnabled), and changing that setting needs a server restart. (2) Your account is not linked: you need a txAdmin admin account with your Discord or Cfx.re identifier attached in Settings > Admin Manager. If you see a red authentication message and you ARE registered, type /txAdmin-reauth in chat to retry authentication." }
 ]}
/>

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| Can't log into txAdmin | Forgot the username/password you set on first boot | Delete `txData/<profile>/admins.json` and restart - txAdmin asks you to create a new admin |
| "Server Offline" even though FXServer.exe is running | The server crashed but the process didn't exit | Check the Live Console for red error lines. Fix the error, then restart |
| Resources tab shows nothing | No resources are in the resources folder or server.cfg | Check your resources folder exists and has subfolders. Check server.cfg has `ensure` lines |
| CFG Editor changes don't save | You didn't click "Save" or the server rejected the change | Click the Save button (floppy disk icon). If it fails, the console will say why |
| Live Console is blank | The console buffer was cleared or server just started | Restart the server or wait for it to print new messages. Old messages are in `txData/logs/` |

>
  FiveM artifacts are not forever. Cfx.re supports them on a clock, and an outdated artifact eventually makes your server unjoinable from the browser. But blindly grabbing the newest build the night before a launch is how servers break. This SOP makes updating boring: pin what works, back it up, update, test, and have a one-step rollback.
</LessonHero>

<Callout variant="fivem" title="The support clock">
  Cfx.re support states recommended artifacts are supported until roughly six weeks after the next release, latest artifacts until about two weeks after, and artifacts older than three months become unsupported and unjoinable from the server browser. Translation: update on your schedule, not in a panic, but do not let an artifact rot past three months.
</Callout>

## The update routine

<StepGroup>
  <Step num={1} title="Record the build you run today" outcome="You know exactly what to roll back to.">
    Keep a tiny note next to the server:

    ```text
    Current build: <number>
    Channel: Recommended
    Known-good: yes
    Updated: <date>
    Rollback build: <previous number>
    ```
  </Step>

  <Step num={2} title="Back up before you touch anything" outcome="The current working server folder is safe.">
    Copy the current artifact folder to a dated backup (for example `server-2026-06-12`). Updating is just swapping the artifact binaries; the backup is your undo button if the new build misbehaves.
  </Step>

  <Step num={3} title="Update on the Recommended channel" outcome="The server runs a current, stable build.">
    Download the latest **Recommended** (green) artifact, extract it over a clean copy of the server folder, and start the server. Do not grab bleeding-edge "Latest" for a production server unless you are chasing a specific fix.
  </Step>

  <Step num={4} title="Test, then mark known-good or roll back" outcome="You either confirm the build or revert in one step.">
    Join the server, run a quick smoke test (connect, spawn, use one core feature, check the console for new errors). If clean, update the note to the new build and mark known-good. If broken, restore the dated backup and start it again.

    <ExpectedOutput>
      Console shows the new server version, players connect, and no new resource load errors appear.
    </ExpectedOutput>
  </Step>
</StepGroup>

## Common mistakes

<FailureList
  items={[
    { symptom: "Players kicked: artifact outdated / End of Support", fix: "Your build is past the support window. Update to a current Recommended artifact following this SOP." },
    { symptom: "New build breaks a resource", fix: "Restore the dated backup folder and start it. Then test the update on a copy before retrying." },
    { symptom: "No idea which build was working", fix: "Keep the artifact-pin note updated every time. Without it you cannot roll back cleanly." },
    { symptom: "Game build mismatch after update", fix: "Set sv_enforceGameBuild to a current build the Recommended channel targets, then restart." },
  ]}
/>

>
 FiveM artifacts are not forever. Cfx.re supports them on a clock, and an outdated artifact eventually makes your server unjoinable from the browser. But blindly grabbing the newest build the night before a launch is how servers break. This SOP makes updating boring: pin what works, back it up, update, test, and have a one-step rollback.
</LessonHero>

<Callout variant="fivem" title="The support clock">
 Cfx.re supports artifacts on a clock, and the exact windows change over time, so do not memorize numbers. As a rough mental model, Recommended builds stay supported for a while after the next release, Latest builds drop off support sooner, and any artifact that gets old enough becomes unsupported and unjoinable from the server browser. Before you plan an update, read the current windows from the support article linked at the bottom of this lesson. The point stands either way: update on your schedule, not in a panic, and do not let an artifact rot until the browser kicks your players.
</Callout>

## The update routine

<StepGroup>
 <Step num={1} title="Record the build you run today" outcome="You know exactly what to roll back to.">
 Keep a tiny note next to the server:

 ```text
 Current build: <number>
 Channel: Recommended
 Known-good: yes
 Updated: <date>
 Rollback build: <previous number>
 ```
 </Step>

 <Step num={2} title="Back up the artifact binaries before you touch anything" outcome="The exact build you run today is saved off to one side.">
 Know the two halves of your server folder. The artifact binaries are the FXServer files you downloaded and extracted: `FXServer.exe`, `run.cmd` (or `run.sh` on Linux), the `citizen` folder, and the rest of the runtime. Your server-data is everything you own: the `resources` folder, `server.cfg`, your database, your `.env`. An update only swaps the binaries. It must never touch server-data.

 Copy the current binaries to a dated backup, for example `fxserver-2026-06-12`. That copy is your undo button if the new build misbehaves. Leave `resources`, `server.cfg`, and your database exactly where they are.
 </Step>

 <Step num={3} title="Swap in the latest build" outcome="The server runs a current, stable build with your config untouched.">
 Download the **latest** artifact. Stop FXServer, extract the new archive into a fresh artifact directory, and point your existing txAdmin/server-data setup at that directory. Replacing the whole artifact directory avoids a mixed build containing stale files. Never extract over `resources`, `server.cfg`, `txData`, or the database. Then start the server. The latest build ages out of support sooner than an older pinned one, so update on a schedule and keep your last known-good build to roll back to.
 </Step>

 <Step num={4} title="Smoke-test, then mark known-good or roll back" outcome="You either confirm the build or revert in one step.">
 Run the same checklist every time so a regression cannot hide:

 - **Console clean.** Watch the console as resources load. No new red errors, no resource that failed to start.
 - **Connect.** Join your own server from the FiveM client. If it is running on the same PC, open the F8 console in FiveM and type `connect 127.0.0.1:30120` (30120 is the default FXServer game port). You should reach character select or spawn, not a load loop.
 - **Spawn.** Confirm your ped actually spawns in the world.
 - **Framework health check.** Use one documented, non-destructive action that you already verified on the previous build, such as opening character selection or reading player/job data. Do not assume `/car`, `/setjob`, or `/giveitem` exists or is safe on every framework configuration.
 - **resmon (needs developer mode).** `resmon` is a developer-only client command, so it only works if you launched the FiveM client in developer mode (add the launch argument `+set moo 31337`). On a normal client, `resmon 1` returns `Access denied for command resmon`. With developer mode on, open the F8 console, type `resmon 1`, and confirm no resource is suddenly burning CPU it did not burn before (`resmon 0` closes it). A build change can shift timing. If you cannot run developer mode, watch txAdmin's performance panel instead.

 If every line passes, update the note to the new build and mark it known-good. If any line fails, do not debug live: stop the server, restore the dated artifact backup from Step 2, start it again, and you are back on the build you trusted this morning.

 <ExpectedOutput>
 The startup banner reads `FXServer-master v1.0.0.XXXX windows` (or `linux`), where `XXXX` is your artifact build number; there are no new resource load errors, your join succeeds, and (if you have developer mode) resmon shows normal timings.
 </ExpectedOutput>
 </Step>
</StepGroup>

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| Server won't start after updating artifacts | New artifact requires newer VC++ Redistributable or breaks an old resource | Restore from your backup artifact. Check the release notes at runtime.fivem.net for breaking changes |
| Backup artifact is missing or corrupted | You didn't save the old server.7z before updating | From now on, always save the current artifact before replacing it. Name it `server-backup-YYYYMMDD.7z` |
| "Which build should I update to?" | Unsure which artifact to pick | Use the latest build (the newest one), not the green "Recommended" pin. Check the Cfx.re changelog first, and keep your last known-good build to roll back to |
| Resources stopped working after update | New artifact changed a default or deprecated a feature | Check the release notes. Common causes: new default ConVars, removed natives, or stricter OneSync scoping |

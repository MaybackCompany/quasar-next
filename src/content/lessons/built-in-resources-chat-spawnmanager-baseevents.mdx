The point of this lesson is a habit, not a single script. When you need a chat message, a spawn point, or a notification that a player got in a car, your first move is to ask whether a stock resource already does it. It almost always does.

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_builtins_tour
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    This resource talks to chat from the client, but it reacts to the `baseevents:enteredVehicle` event, which the stock `baseevents` resource fires on the *server*. So the chat suggestion and message live in a `client_script`, and the vehicle handler lives in a `server_script`. Create this exact layout:

    ```text
resources/qu_builtins_tour/
  fxmanifest.lua
  client.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load and which resources must load first.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

dependencies { 'chat', 'baseevents' }

client_script 'client.lua'
server_script 'server.lua'
    ```

    The `dependencies` block names the stock resources you call into. It tells the server to load `chat` and `baseevents` before `qu_builtins_tour`, so their events exist by the time your code runs. The `client_script` holds the chat suggestion and message; the `server_script` holds the vehicle handler, because `baseevents:enteredVehicle` is a server event. Note there is no `lua54 'yes'` line: that directive was deprecated in June 2025 and is ignored, since Lua 5.4 is now the only runtime.
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `client.lua` and paste this. These two calls are genuinely client-side: the chat box lives on each player's own game instance.

    ```lua
-- 1. Tell the chat box about a command we want to hint at.
TriggerEvent('chat:addSuggestion', '/wave', 'Wave at nearby players', {
    { name = 'style', help = 'Optional: short or long' }
})

-- 2. Post one styled system message into the chat box.
TriggerEvent('chat:addMessage', {
    color = { 0, 200, 120 },
    multiline = true,
    args = { 'Quasar', 'Builtins tour loaded. Get in a vehicle to test baseevents.' }
})
    ```

    Now open `server.lua` and paste this. `baseevents:enteredVehicle` is a *server* event: the stock `baseevents` resource detects the vehicle on the client and fires it across to the server with `TriggerServerEvent`, so the handler must live here and read which player it was about from `source`.

    ```lua
-- 3. React to the baseevents:enteredVehicle SERVER event.
-- baseevents fires this with TriggerServerEvent, so we register a net event.
RegisterNetEvent('baseevents:enteredVehicle')
AddEventHandler('baseevents:enteredVehicle', function(currentVehicle, currentSeat, vehicleDisplayName, vehicleNetId)
    local src = source

    -- Send the confirmation line back to just that player's chat box.
    TriggerClientEvent('chat:addMessage', src, {
        color = { 0, 200, 120 },
        multiline = true,
        args = { 'Quasar', ('You entered a %s (seat %d).'):format(vehicleDisplayName, currentSeat) }
    })

    print(('[qu_builtins_tour] player %d enteredVehicle: %s seat %d netId %s'):format(src, vehicleDisplayName, currentSeat, tostring(vehicleNetId)))
end)
    ```
  </Step>

  <Step num={5} title="Start and test it" outcome="The expected proof appears in the chat box and the server console.">
    Open `server.cfg` and add this line below the stock resources:

    ```text
    ensure qu_builtins_tour
    ```

    Save, then run:

    ```text
    restart qu_builtins_tour
    ```

    Join the server so your ped exists. As you load in, watch the chat box for the system message. Then type `/` and look for the `/wave` hint. Finally, get into any vehicle and watch chat plus the server console (txAdmin Live Console):

    ```text
    open the server console, get into a car
    ```

    <ExpectedOutput>
      [Quasar] Builtins tour loaded. Get in a vehicle to test baseevents.
[Quasar] You entered a Sultan (seat -1).
[qu_builtins_tour] player 1 enteredVehicle: Sultan seat -1 netId 65538
    </ExpectedOutput>

    The vehicle name and netId depend on the car you enter, and seat `-1` is the driver's seat. Both `[Quasar]` lines appear in your in-game chat box: the first from the client message, the second from the server sending `chat:addMessage` back to your player with `TriggerClientEvent`. The `[qu_builtins_tour]` line appears in the server console (not F8) because the `print` now runs server-side, and `player 1` is your server id from `source`.
  </Step>
</StepGroup>

## How it works

You ran it, the message appeared, the hint showed up, and getting in a car triggered a reaction. None of that chat or vehicle plumbing is yours. You sent events to resources that were already running, and listened for one they fire. Now take each call apart so you know exactly which resource you are talking to, on which side, and why.

### The chat suggestion: a hint, not a command

```lua
TriggerEvent('chat:addSuggestion', '/wave', 'Wave at nearby players', {
    { name = 'style', help = 'Optional: short or long' }
})
```

`TriggerEvent` fires a named event on the same side it is called from, here the client. The event name `chat:addSuggestion` belongs to the stock `chat` resource. The colon is just a naming convention: the part before it tells you which resource owns the event. You are not defining this event, you are sending it to chat, which is listening for it.

The arguments are read left to right. `'/wave'` is the command text that shows up in the autocomplete list when a player types `/`. `'Wave at nearby players'` is the help text shown next to it. The fourth argument is a table of parameter hints, where each entry has a `name` and a `help` string, so the player sees what to type after the command.

One thing this does not do: it does not create the `/wave` command. A suggestion is only the hint in the autocomplete dropdown. If you want `/wave` to actually do something you still register it yourself with `RegisterCommand`, exactly as you learned in the first-line and natives lessons. Pairing a real `RegisterCommand` with a matching `chat:addSuggestion` is the polished pattern: the command works, and players can discover it.

### The system message: color, args, and the name tag

```lua
TriggerEvent('chat:addMessage', {
    color = { 0, 200, 120 },
    multiline = true,
    args = { 'Quasar', 'Builtins tour loaded. Get in a vehicle to test baseevents.' }
})
```

`chat:addMessage` is how every server announcement, notification, and bot line you have ever seen in a FiveM chat box gets there. You hand it one table with three fields.

`color` is an RGB triple as a Lua table: red, green, blue, each from 0 to 255. Here `{ 0, 200, 120 }` is a calm green that tints the name tag. `multiline = true` lets a long message wrap onto more than one line instead of being clipped. `args` is the message itself, split into parts. By convention the first entry is the sender name shown in brackets, and the second is the body text. So `{ 'Quasar', 'Builtins tour loaded...' }` renders as a `[Quasar]` tag followed by the message. You can think of the name tag as a label that says who is talking, which is why server messages usually read `[System]` or the server name.

### The baseevent handler: reacting to the game

```lua
RegisterNetEvent('baseevents:enteredVehicle')
AddEventHandler('baseevents:enteredVehicle', function(currentVehicle, currentSeat, vehicleDisplayName, vehicleNetId)
    local src = source
    -- ...
end)
```

`AddEventHandler` registers a listener for an event that some other code fires. This is different from `TriggerEvent`, which sends an event. Sending versus listening is the whole shape of event-driven code: one resource triggers, another handles. Here the `baseevents` resource is doing the triggering. Its client-side code watches the game and, the moment a player's local ped sits in a vehicle, it fires `baseevents:enteredVehicle` across to the *server* with `TriggerServerEvent`. That is why this handler lives in `server.lua` and pairs `RegisterNetEvent` with `AddEventHandler`: `RegisterNetEvent` allows the event to be received over the network, and `AddEventHandler` runs your code when it arrives.

Read the parameters in order, the four values baseevents sends: `currentVehicle` is the entity handle of the car. `currentSeat` is which seat the player took, where `-1` is the driver and `0` is the front passenger. `vehicleDisplayName` is the readable model name like `Sultan`. `vehicleNetId` is the network id, the identifier other clients and the server use to refer to that same vehicle across the network. On top of those, `source` is set on every net event the server receives: it is the server id of the player the event came from. We copy it into `local src = source` right away, because `source` can change if any other event fires before we use it.

Inside the handler we do two things: send a chat message back to that one player and `print` a line to the server console. Because the handler runs on the server, it cannot just `TriggerEvent('chat:addMessage', ...)` and expect it in someone's chat box. Instead it uses `TriggerClientEvent('chat:addMessage', src, ...)`, which sends the event to the chat resource running on that specific player's client. The chat line is the user-facing proof; the `print` is your developer probe, and it lands in the server console rather than F8. The `('...'):format(...)` method fills the placeholders, `%s` for the model name string and `%d` for the integer seat, exactly the formatting you saw in the natives lesson.

Why use the stock baseevent instead of writing your own vehicle detection? Without it you would have to spin up a thread on every client that runs every frame, calls `IsPedInAnyVehicle`, tracks whether the answer changed since last frame, and reports it to the server yourself. That is a real loop you would have to write, tune, and keep from eating CPU. `baseevents` already does that work and simply tells the server the instant it happens. Calling its event is cheaper, cleaner, and the same data every other resource on the server is reading.

<Checkpoint question="Why does the vehicle reaction use AddEventHandler, while the chat message uses TriggerEvent?">
  Because the two calls are on opposite ends of the same event. `TriggerEvent` sends an event out, so you use it to push a message into the `chat` resource that is listening for `chat:addMessage`. `AddEventHandler` listens for an event, so you use it to catch `baseevents:enteredVehicle`, which the stock `baseevents` resource fires when a player gets in a car. The catch: `baseevents:enteredVehicle` is a *server* event, fired with `TriggerServerEvent`, so the listener lives in `server.lua` and you pair `AddEventHandler` with `RegisterNetEvent` so the event can arrive over the network. You are the sender for chat and the listener for baseevents. If you swapped the sender and listener, or put the vehicle handler in a `client_script`, it would never run, because triggering an event you mean to listen for, listening for one you mean to send, or listening on the wrong side, does nothing useful.
</Checkpoint>

### Sides matter: which events live where

This is where most beginners get burned, so read the stock code, not your assumptions. `chat:addMessage` and `chat:addSuggestion` are triggered from the client, because the chat box exists on each player's own game instance, which is why those two calls are in `client.lua`.

But `baseevents:enteredVehicle` and `baseevents:leftVehicle` are *server* events. The stock `baseevents` resource does detect the vehicle on the client, but it does not hand you a client event: its client code fires `TriggerServerEvent('baseevents:enteredVehicle', ...)`, and `server.lua` registers it with `RegisterNetEvent`. So a `client_script` that does `AddEventHandler('baseevents:enteredVehicle', ...)` never fires, no matter what. You listen on the server with `RegisterNetEvent` plus `AddEventHandler`, and `source` tells you which player got in the car.

The death events behave differently again. `baseevents:onPlayerDied` and `baseevents:onPlayerKilled` fire on *both* sides: the stock code triggers each one with a client-local `TriggerEvent` and a `TriggerServerEvent` at the same time. So a `client_script` can listen for them with `AddEventHandler` and read the death data directly (there is no `source` on the client), while a `server_script` listens with `RegisterNetEvent` and reads the player who died from `source`. Pick the side that has the data you need. `sessionmanager` rounds this out: it fires `playerActivated` on the server once a player has fully loaded in, which is the correct moment to set up that player's data, far more reliable than reacting to them merely connecting.

The rule to carry forward: read which side a stock event actually fires on before you write the handler. The client can request things, but only the server grants them, and listening for an event on the wrong side is the most common reason a beginner's handler "never fires."

### When you reach for these

You will touch these stock resources in nearly every script you write:

- **chat:addMessage** for any server notification, command feedback, or bot line. Before you reach for a third-party notification library, remember the chat box is already there and free.
- **chat:addSuggestion** every time you add a `RegisterCommand`, so players can discover it by typing `/`. A command without a suggestion is a command nobody finds.
- **baseevents** when a feature should react to a player getting in a car, leaving it, dying, or being killed. Seatbelt scripts, insurance, and kill feeds all start here.
- **spawnmanager** when you control where players appear. `exports.spawnmanager:addSpawnPoint` registers a location, `setAutoSpawnCallback` lets you decide the spawn point in code, and `forceRespawn` sends the player there on demand.
- **sessionmanager** when setup must wait until a player is genuinely ready. `playerActivated` is that moment.

## Try it yourself

<Exercise>
  Make the resource a real driving companion. First, register an actual `/wave` command so the suggestion you added is not a dead hint. Then add a second handler for `baseevents:leftVehicle` so the chat box reacts when the player gets out, not just in. Remember `leftVehicle` is a server event, the same as `enteredVehicle`, so its handler goes in `server.lua` with the same pattern.

  Add the command to `client.lua`, since the command runs on the player typing it and the chat box is client-side:

  ```lua
  -- A real command to back the suggestion you already added.
  RegisterCommand('wave', function(source, args)
      local style = args[1] or 'short'
      TriggerEvent('chat:addMessage', {
          color = { 0, 200, 120 },
          multiline = true,
          args = { 'Quasar', ('You wave (%s) at the locals.'):format(style) }
      })
  end, false)
  ```

  Add the leftVehicle handler to `server.lua`, next to the enteredVehicle one, and send the line back to that player with `TriggerClientEvent`:

  ```lua
  -- React when the player leaves a vehicle (also a SERVER event).
  RegisterNetEvent('baseevents:leftVehicle')
  AddEventHandler('baseevents:leftVehicle', function(currentVehicle, currentSeat, vehicleDisplayName, vehicleNetId)
      local src = source
      TriggerClientEvent('chat:addMessage', src, {
          color = { 200, 120, 0 },
          multiline = true,
          args = { 'Quasar', ('You stepped out of the %s.'):format(vehicleDisplayName) }
      })
  end)
  ```

  Restart `qu_builtins_tour`, type `/wave long` and confirm the styled line appears, then get into a car and back out and watch the green entered line followed by the orange left line. For a stretch, add a `chat:addSuggestion` for `/wave` so a player typing `/` sees the style argument hint, and notice how the command, the hint, and the events all route through chat without you writing a single line of UI.
</Exercise>

## The three runtimes

FiveM exposes the same scripting platform through three language runtimes. They all see the same natives, the same events, the same resource model. The difference is the language you write and the engine that runs it.

    { cmd: "JavaScript", desc: "Runs on Node inside the server. Good for NUI-heavy resources and for developers who already know JS. No compile step, but you opt into the Node version in the manifest." },

### Lua 5.4: the default this course teaches

Lua is the shortest route through this course. There is no project build step: save the file and restart the resource. ESX, QBCore, Qbox, and the Overextended resources all publish Lua-facing APIs, so Lua lets you read the examples used later without changing languages. No reliable official statistic says what percentage of public resources use each runtime, so choose from the code and dependencies you actually plan to maintain.

One accuracy note for 2026: you do **not** add a `-- Lua 5.4 is the default runtime in 2026; the lua54 directive is no longer needed` line to the manifest anymore. The official docs mark that directive deprecated as of June 2025: Lua 5.3 was removed and every Lua script now runs on 5.4, so the line is a no-op and there is nothing to opt into.

### JavaScript on Node: for NUI-heavy work and JS developers

FiveM ships a Node runtime, so a `.js` file declared as a `client_script` or `server_script` runs with no compile step, the same as Lua. Two honest reasons to reach for it. First, your NUI front end is already HTML, CSS, and JavaScript, so a JS server or client script lets you share types and mental model across the boundary instead of switching languages mid-resource. Second, if your team already lives in JavaScript, the ramp is shorter than learning Lua from zero.

The one manifest detail that trips people up: you pin the Node major version with a `node_version` line, for example `node_version '22'`. Leave it off and the server uses its built-in default, a customized Node.js 16.x at the time of writing; pinning `'22'` opts into the newer runtime and keeps the behaviour you tested stable across server updates.

### C# on .NET: compiled, for large or enterprise systems

C# does not run from source. You write `.cs` files, compile them with the .NET SDK into a `.dll`, and the manifest loads that compiled assembly rather than the source. At a high level the flow is: write C# referencing the CitizenFX runtime, run a build (typically `dotnet build`) that produces a `.net.dll`, and declare that dll in the manifest as a client or server assembly. The payoff is static typing, real IDE refactoring, and structure that holds up across a large codebase. The cost is a build step between every change and a heavier toolchain. For a single feature script this is overkill. For a large team shipping an enterprise-scale system where compile-time safety earns its keep, it is a reasonable choice.

## Build it

You will build one resource, `qu_runtimes`, that declares a Lua server script and a JavaScript server script together. Both print one line on start. This is the proof that the runtimes coexist.

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_runtimes
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_runtimes/
  fxmanifest.lua
  server.lua
  server.js
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows to load both a Lua and a JavaScript script.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

node_version '22'

server_script 'server.lua'
server_script 'server.js'
    ```

    Two `server_script` lines, one ending in `.lua` and one in `.js`. The file extension is how FiveM picks the runtime: `.lua` loads on the Lua runtime, `.js` loads on Node. The `node_version '22'` line pins the Node major version so the JavaScript side behaves the same across server updates. There is no `lua54` line: that directive was deprecated in June 2025 and Lua 5.4 is now the only Lua runtime.
  </Step>

  <Step num={4} title="Write the Lua script" outcome="The Lua runtime has one line to print on start.">
    Open `server.lua` and paste this:

    ```lua
print('[qu_runtimes] hello from Lua 5.4')
    ```

    This is top-level code, not inside a handler, so it runs once the moment the resource starts.
  </Step>

  <Step num={5} title="Write the JavaScript script" outcome="The Node runtime has one line to print on start.">
    Open `server.js` and paste this:

    ```js
console.log('[qu_runtimes] hello from JavaScript on Node');
    ```

    `console.log` is the JavaScript equivalent of Lua's `print`: it writes a line to the same server console. This file also runs its top-level code once on start.
  </Step>

  <Step num={6} title="Start and test it" outcome="Two startup lines appear, one per runtime.">
    Open `server.cfg` (the plain-text server config file in your server-data folder, next to where your `resources` folder is referenced) in your text editor and add this line:

    ```text
    ensure qu_runtimes
    ```

    Save the file. Now open the **txAdmin Live Console** (txAdmin web panel, default at `http://localhost:40120`, then your server's Live Console tab) and type this command into the console input box at the bottom, then press Enter:

    ```text
    restart qu_runtimes
    ```

    The same Live Console window that you just typed into is where the output appears. You do not type any further command for this lesson. Both scripts print on start, right after the restart line.

    <ExpectedOutput>
      [qu_runtimes] hello from Lua 5.4
[qu_runtimes] hello from JavaScript on Node
    </ExpectedOutput>

    These are the two lines your scripts print (illustrative). In the real console each line is wrapped with extra text the server adds automatically, such as a timestamp and a resource tag, so you are looking for your `[qu_runtimes]` lines inside that surrounding output, not two lines alone. The two lines may print in either order. What matters is that both appear from one resource, proving the Lua and Node runtimes ran side by side.
  </Step>
</StepGroup>

## How it works

Two scripts, two languages, one resource, two lines in the console. Take it apart so you understand exactly what made both runtimes start.

### The two server_script lines

```lua
server_script 'server.lua'
server_script 'server.js'
```

A `server_script` directive tells FiveM to load a file on the server side. You can list as many as you want, and FiveM loads each in order. The thing that decides which runtime handles a file is the **file extension**, nothing else. A `.lua` file goes to the Lua runtime. A `.js` file goes to the Node runtime. A `.cs` build output, a `.net.dll`, would go to the .NET runtime. So the manifest is not declaring languages by name; it is listing files, and each file's extension routes it to the right engine. That is why one resource can mix runtimes freely: the manifest treats every script the same way and lets the extension sort it out.

### The node_version pin

```lua
node_version '22'
```

This line only matters because you included a `.js` file. It pins the Node major version the JavaScript runtime uses. Without it, your resource runs on whatever Node the server happens to ship, which can shift when the server updates. Pinning it means the behaviour you tested is the behaviour you get later. There is no Lua equivalent line, because Lua is fixed at 5.4 platform-wide and the old `lua54` opt-in is gone as of June 2025.

### print versus console.log

```lua
print('[qu_runtimes] hello from Lua 5.4')
```

```js
console.log('[qu_runtimes] hello from JavaScript on Node');
```

These two lines do the same job through each language's normal API. Lua writes to the console with `print`. JavaScript writes to the console with `console.log`. Both land in the same place, the server console, because FiveM routes every runtime's standard output there. Notice the JavaScript line ends in a semicolon and the Lua line does not. That is a language convention difference, not a FiveM one. The shared `[qu_runtimes]` tag at the front of each line is text you typed on purpose, so you can pick your own output out of a console where many resources print at once.

The mental model to keep: the resource is the unit, and inside it each file speaks its own language to the same platform. You are not bridging two programs. You are running two scripts that both have the full FiveM surface available, just expressed in different syntax.

<Checkpoint question="The manifest never says the word 'JavaScript' or 'Lua'. So how does FiveM know which runtime to use for each script?">
  By the file extension on each `server_script` (or `client_script`) entry. `.lua` routes to the Lua 5.4 runtime, `.js` routes to the Node runtime, and a compiled `.net.dll` routes to the .NET runtime. The manifest is a list of files, not a list of languages; the extension is the only signal. That is exactly why one resource can run Lua and JavaScript together with no extra wiring: each file is dispatched to its engine independently. The `node_version` line is the one JavaScript-specific knob, and it only takes effect because a `.js` file is present.
</Checkpoint>

### When you reach for each runtime

You now know they coexist. The real skill is choosing, and the choice is usually easy:

- **Reach for Lua by default.** Almost every resource. No build step, instant iterate-and-restart, and the entire ecosystem speaks Lua. The frameworks (ESX, Qbox), and the Overextended community resources (`ox_lib`, `oxmysql`, `ox_target`, `ox_inventory`) are all consumed from Lua first. Writing Lua means you can read the code you depend on and others can read yours.
- **Reach for JavaScript when the UI is the resource.** If a script is mostly NUI, sharing the language with your HTML front end keeps one mental model across the boundary. Same call if your team already writes JavaScript and learning Lua from zero would slow you down. Pin `node_version` when you do.
- **Reach for C# when your team benefits from .NET.** Static types, .NET libraries, and IDE refactoring can justify the build step in codebases of many sizes. Project needs and team experience matter more than an arbitrary scale threshold.

This course stays Lua-first on purpose. Not because the other runtimes are wrong, but because Lua is what you will overwhelmingly read and ship, and fluency in the common language is worth more early on than coverage of all three.

## Try it yourself

<Exercise>
  Prove the runtimes coexist on the **client** side too, and make each one read its own runtime fact rather than printing a hardcoded label. Add a client Lua file and a client JavaScript file to `qu_runtimes`, and have each print which language it is running plus the resource name, read from the platform instead of typed by hand.

  ```lua
  -- client.lua
  print(('[%s] client running on Lua'):format(GetCurrentResourceName()))
  ```

  ```js
  // client.js
  console.log(`[${GetCurrentResourceName()}] client running on JavaScript`);
  ```

  Then add both to the manifest:

  ```lua
  client_script 'client.lua'
  client_script 'client.js'
  ```

  Restart `qu_runtimes`, join the server so the client scripts load, and open the F8 console. Confirm two client lines appear, one per language, each showing `qu_runtimes` pulled from `GetCurrentResourceName()`. You have now run the same native from two runtimes and seen client output land in F8 while the earlier server output landed in txAdmin. That split, server output to the server console and client output to F8, is the same regardless of which language wrote the line.
</Exercise>

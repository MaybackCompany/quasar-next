<Callout variant="fivem" title="MAJOR.MINOR.PATCH">
  Versions are three numbers. **PATCH** (1.2.3 -> 1.2.4): a bug fix, safe to take. **MINOR** (1.2.x -> 1.3.0): new feature, backwards compatible. **MAJOR** (1.x -> 2.0.0): a breaking change that needs config or migration work. The number tells the buyer how nervous to be.
</Callout>

## Version and document a release

<StepGroup>
  <Step num={1} title="Decide the bump" outcome="The version number matches the kind of change.">
    Bug fix only -> bump PATCH. New optional feature, nothing breaks -> bump MINOR. Anything that breaks existing configs, exports, or the database -> bump MAJOR and say so loudly.
  </Step>

  <Step num={2} title="Write a short changelog" outcome="Buyers can read what changed in ten seconds.">
    Group by Added / Changed / Fixed / Breaking. One line each. Lead with anything breaking.

    ```text
    ## 2.0.0
    Breaking: config key `cooldown` renamed to `cooldownMs` — update your config.
    Added: support for Qbox.
    Fixed: dupe when spamming the use key.
    ```
  </Step>

  <Step num={3} title="Tag the release" outcome="The version is a real, downloadable point in history.">
    Create a GitHub release tagged with the version (for example `v2.0.0`) and paste the changelog into it. Now buyers can download an exact version and roll back if needed.
  </Step>

  <Step num={4} title="Announce with the migration note" outcome="Nobody is surprised by a breaking change.">
    When you ship a MAJOR, tell buyers exactly what they must change. A breaking update with no migration note is how you earn refunds and bad reviews.
  </Step>
</StepGroup>

## Common mistakes

<FailureList
  items={[
    { symptom: "Buyers afraid to update", fix: "Use semver and a changelog. A PATCH should feel safe because the number says so." },
    { symptom: "A breaking change shipped as a patch", fix: "Breaking changes are MAJOR bumps, with a migration note. Mislabeling them destroys trust." },
    { symptom: "No way to roll back", fix: "Tag every release on GitHub so buyers can download and revert to an exact version." },
    { symptom: "Support flooded after an update", fix: "You shipped a breaking change quietly. Always lead the changelog and announcement with breaking items." },
  ]}

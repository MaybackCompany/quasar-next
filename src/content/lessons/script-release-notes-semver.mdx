<Callout variant="fivem" title="MAJOR.MINOR.PATCH">
  Versions are three numbers. **PATCH** (1.2.3 -> 1.2.4): a bug fix, safe to take. **MINOR** (1.2.x -> 1.3.0): new feature, backwards compatible. **MAJOR** (1.x -> 2.0.0): a breaking change that needs config or migration work. The number tells the buyer how nervous to be.
</Callout>

## Version and document a release

<StepGroup>
  <Step num={1} title="Decide the bump" outcome="The version number matches the kind of change.">
    For a FiveM resource, the bump is not a feeling, it maps to concrete things a buyer's server depends on:

    - **PATCH** (1.2.3 -> 1.2.4): you fixed a bug and nothing the buyer touches changes. A fixed dupe exploit, a corrected blip colour, a typo in a notification. Drop in the new files, restart, done.
    - **MINOR** (1.2.x -> 1.3.0): you added something optional and the old setup still works untouched. A new optional `Config` block with a sensible default, a brand new export alongside the existing ones, a new command. A buyer who ignores the new feature notices nothing.
    - **MAJOR** (1.x -> 2.0.0): you broke something the buyer has wired into. A renamed or removed `Config` key, a changed export signature (different arguments or return), a required SQL migration, a dependency swap (for example moving from a deprecated MySQL wrapper to oxmysql). Their current config or database no longer works as-is.

    The test for MAJOR: would a buyer who drops in your new files and restarts get errors or silent breakage with their existing config and database? If yes, it is a MAJOR bump and it needs a migration note.
  </Step>

  <Step num={2} title="Write a short changelog" outcome="Buyers can read what changed in ten seconds.">
    Group your lines under headings. The Keep a Changelog standard categories are Added, Changed, Deprecated, Removed, Fixed, and Security. Breaking changes do not get their own standard category, so call them out loudly at the top, a bold **Breaking** line above the rest, because that is the one thing a buyer must read before updating. One line each.

    ```text
    ## 2.0.0
    Breaking: config key `cooldown` renamed to `cooldownMs` - update your config.
    Breaking: requires the new SQL migration in /sql/2.0.0.sql before first start.
    Added: support for Qbox alongside ESX Legacy and QBCore.
    Fixed: dupe when spamming the use key.
    ```
  </Step>

  <Step num={3} title="Tag the release" outcome="The version is a real, downloadable point in history.">
    In your project folder, in a terminal (Windows PowerShell or a Linux/macOS shell), tag the current commit and push the tag with: `git tag -a v2.0.0 -m "v2.0.0"` and then `git push origin v2.0.0`. Here `v2.0.0` is the version with a leading `v` (the common Git tag convention) and `origin` is the default name Git gives your GitHub remote. Then on github.com open your repository, click **Releases** (right sidebar), then **Draft a new release**, pick the `v2.0.0` tag you just pushed under **Choose a tag**, paste your changelog into the description, and click **Publish release**. What you should see: a release page at `github.com/<you>/<repo>/releases/tag/v2.0.0` with your notes and an auto-generated **Source code (zip)** download. The tag pins this exact commit, so a buyer can download `v2.0.0` as a zip, or drop back to `v1.9.4` if your update breaks their live server. That rollback is the safety net: when a server owner takes your update mid-session and it errors, they need to revert to a known-good version in seconds, not dig through commit history while players sit in a broken city.
  </Step>

  <Step num={4} title="Announce with the migration note" outcome="Nobody is surprised by a breaking change.">
    When you ship a MAJOR, tell buyers exactly what they must change: which config keys moved, which SQL file to run, which dependency to install first. A breaking update with no migration note is how you earn refunds and bad reviews. Pin the migration note above the download so nobody misses it.
  </Step>
</StepGroup>

## Common mistakes

<FailureList
  items={[
    { symptom: "Buyers afraid to update", fix: "Use semver and a changelog. A PATCH should feel safe because the number says so." },
    { symptom: "A renamed config key shipped as a patch", fix: "Renaming or removing a Config key breaks every existing buyer's config. That is a MAJOR bump with a migration note, never a quiet PATCH." },
    { symptom: "An export's arguments changed and dependent resources broke", fix: "A changed export signature breaks any resource that calls it. Bump MAJOR and document the new signature so buyers can update their callers." },
    { symptom: "A required SQL migration left out of the notes", fix: "If first start needs a new table or column, the migration file and the run order are the most important line in the changelog. Lead with it." },
    { symptom: "No way to roll back after a bad update", fix: "Tag every release on GitHub so a server owner can revert to an exact version mid-incident instead of waiting on you." },
    { symptom: "The tag shows locally but is not on GitHub", fix: "git tag only creates the tag in your local repo. Push it explicitly with git push origin v2.0.0 (a plain git push does not send tags). If the release page cannot find the tag, it was never pushed." },
    { symptom: "Tagged the wrong commit, or want to redo a tag", fix: "Delete it locally and remotely, then re-tag the right commit: git tag -d v2.0.0 then git push origin :refs/tags/v2.0.0, then tag again. Never silently reuse a published version number for different code, bump instead." },
  ]}
/>

## Try it yourself

<Exercise>
  Classify each of these real resource changes as PATCH, MINOR, or MAJOR before you read the answers:

  1. You fixed a money dupe that triggered when a player spam-clicked the use key.
  2. You added an optional `Config.UseTarget` flag that defaults to `false`, so existing servers behave exactly as before.
  3. You renamed the export `getCash` to `getMoney` to match your other resources.
  4. You added a new column to the `player_garages` table and the resource errors on start without it.

  Answers: (1) PATCH, a bug fix with nothing for the buyer to change. (2) MINOR, a new optional feature that is backwards compatible. (3) MAJOR, a changed export signature breaks every resource that called `getCash`. (4) MAJOR, a required SQL migration; ship the `.sql` file and lead the changelog with the run-it-first instruction. Now do the same for the last change you actually made to one of your scripts, write its one-line changelog entry, and decide the bump.
</Exercise>

## How it works

You have two `while true` loops running side by side and neither one freezes the game. That is the whole trick of this lesson, and it rests on three ideas: what `CreateThread` actually creates, why `Wait` is not optional, and how to pick the Wait value on purpose instead of copying it.

### What CreateThread actually creates

The name lies a little. `CreateThread` does not start an operating-system thread. It starts a **Lua coroutine**, and every resource on your server gets exactly one Lua VM to run all of its coroutines in. A coroutine is a function that can pause itself partway through, hand control back, and then be resumed later from the exact line where it paused.

FiveM runs a **scheduler**. Once per frame the engine walks the list of coroutines that are due to run, resumes each one, lets it work until it yields, then moves to the next. When your resource has two `CreateThread` blocks, you have handed the scheduler two coroutines. It resumes them one after another inside that single VM. They are not truly parallel. They take turns, fast enough that to a player they look simultaneous.

This is why a heavy client loop can delay other client script work and the game frame: the scheduler cannot move on until the current coroutine yields. Server-side loops use the same cooperative idea but affect the server runtime instead of a player's renderer.

### Why Wait is mandatory

Look at the shape of either loop with the `Wait` removed:

```lua
CreateThread(function()
    while true do
        -- no Wait here
        if enabled then print('tick') end
    end
end)
```

`while true` never ends on its own. There is no condition that turns false. Without a `Wait` inside it, this coroutine spins forever and never yields. The scheduler resumed it, and it never gave control back. The engine cannot draw the next frame, cannot resume any other coroutine, cannot read input. The game **hard freezes**. Not slows down, freezes, until the runtime kills the resource with a script timeout.

`Wait(ms)` is the yield. When your coroutine calls `Wait`, it pauses itself, tells the scheduler "do not resume me for at least this many milliseconds," and hands control back. The engine is free to draw frames and run everything else. After the time is up, the scheduler resumes your coroutine on the next available frame, right after the `Wait` call, and the loop goes around again. So the real job of `Wait` is not "pause for a bit." Its job is to give the frame back. The pause is just how long you give it back for.

<Checkpoint question="Why does a while-true loop with no Wait freeze the entire game and not just the one resource?">
  Because all of a resource's coroutines share one Lua VM, and the engine resumes coroutines one at a time on the main thread. A `while true` loop with no `Wait` never yields, so the scheduler that resumed it never gets control back. It cannot advance to the next coroutine, cannot draw the next frame, and cannot read input. The single stuck loop blocks the whole main thread, so the symptom is a total freeze, not a slow resource. The `Wait` is what hands the frame back and lets everything else run.
</Checkpoint>

### What the Wait value actually means

`Wait(0)` and `Wait(5000)` both yield. The number is just how long until the scheduler is allowed to resume you.

- `Wait(0)` means "resume me on the very next frame." At 60 fps that loop runs about 60 times every second. It is the most expensive thing you can do, because the body runs every single frame.
- `Wait(5000)` means "do not resume me for 5000 milliseconds." That loop runs once every five seconds. Between runs it costs nothing. The coroutine is asleep and the scheduler skips right over it.

That is the contrast the two loops in this lesson are built to show. The slow tick uses `Wait(5000)` because a heartbeat that prints "slow tick" has no reason to run more than once every few seconds. Running it every frame would do the same job and burn 300 times the work. The keypress poll uses `Wait(0)` because it has no choice, which is the next idea.

### Why the keypress poll needs Wait(0)

`IsControlJustPressed` returns `true` on exactly one frame: the frame the button goes down. On the very next frame, even if you are still holding E, it returns `false`. "Just pressed" means the transition, not the held state.

That single-frame window is why this check has to live in a `Wait(0)` loop. If you polled it inside `Wait(5000)`, you would only look at the control once every five seconds. The odds that your single glance lands on the exact frame the key went down are almost zero, so nearly every press would be missed. To catch a one-frame event reliably, you have to look every frame, which means `Wait(0)`. The cost of `Wait(0)` is the price of never missing the press.

### The tick budget and resmon

Every `Wait(0)` loop you add has a cost, and the tool that shows it is **resmon**. Open the in-game console with F8 and run:

```text
resmon
```

You get a live client-side table of resources and their script time. At 60 fps the whole client frame is about 16.6 ms, shared by the game and client resources. A resource consistently spending multiple milliseconds is consuming a visible part of that local player's frame budget. `resmon` does not measure server tick time; use the server profiler for server-side performance.

Your two-loop resource here will sit near zero, because the only thing the `Wait(0)` loop does each frame is one cheap boolean check. The cost problem shows up when people put expensive work, native calls, drawing, distance math, inside a `Wait(0)` loop that did not need to run every frame. Which is exactly what the next pattern fixes.

### The pattern you are building toward: adaptive Wait

Most real polling loops do not need a fixed Wait at all. They need a long Wait most of the time and a short Wait only in the moment that matters. The standard FiveM shape for this is the **adaptive wait** (the distance-check optimization):

```lua
CreateThread(function()
    local target = vector3(200.0, -900.0, 30.0)
    while true do
        local wait = 1000 -- default: sleep a full second
        local coords = GetEntityCoords(PlayerPedId())
        if #(coords - target) < 20.0 then
            wait = 0 -- close enough to matter: run every frame
            -- draw a marker, check for E, do the per-frame work here
        end
        Wait(wait)
    end
end)
```

The loop starts each pass assuming it can sleep a full second. Only when the player is within 20 metres does it drop `wait` to `0` and do the expensive per-frame work. Standing across the map, the loop costs almost nothing because it runs once a second. Standing on the marker, it runs every frame, exactly when that is worth paying for. One loop, two speeds, chosen by a cheap distance check. This is the takeaway the whole lesson points at: `Wait(0)` is not the default, it is a tool you reach for only on the frames that need it.

### A loop that ends: bounded loops vs while true

Every loop so far has been `while true` — it never stops on its own, so it needs a `Wait` to stay alive without freezing. But a loop does not have to be infinite. When the work has a clear finish line, you give the loop a real exit condition, and when that condition turns false the loop ends, the function returns, and the coroutine is simply gone. The scheduler never resumes it again because there is nothing left to resume.

A countdown is the cleanest example. Add this third thread to `client.lua`, restart `qu_createthread_and_wait`, and run `/threadtoggle`:

```lua
-- Bounded loop: counts down once, then the coroutine ends.
CreateThread(function()
    local remaining = 5
    while remaining > 0 do
        print('[qu_createthread_and_wait] countdown ' .. remaining)
        Wait(1000)
        remaining = remaining - 1
    end
    print('[qu_createthread_and_wait] countdown done')
end)
```

The condition is `remaining > 0`, not `true`. Each pass prints the number, yields for one second with `Wait(1000)`, then subtracts one. After five passes `remaining` reaches `0`, the `while` condition is false, the loop exits, the final `print` runs once, and the function returns. That return ends the coroutine for good.

Two things are worth saying out loud. First, this loop still needs its `Wait(1000)` — a bounded loop with no `Wait` would still spin to its end with no yield in between and could still hitch the frame, so the rule "every loop yields" holds even when the loop is finite. Second, this thread runs once and is done; if you want the countdown to happen again you have to start a fresh `CreateThread`, because the old coroutine is gone, not paused. That is the difference from the slow-tick and keypress loops above, which use `while true` precisely because they must keep watching forever. Pick `while true` when the job never ends, and a real condition when it does.

### When a loop talks to the server: do not spam the wall

Loops and client-server events meet at one dangerous spot. A network event like `TriggerServerEvent('myresource:server:doThing')` sends a packet from this client up to the server, where server code handles it. That is cheap to call once. It is a disaster to call sixty times a second. Dropping a `TriggerServerEvent` straight into a `Wait(0)` loop floods the server with packets every frame, from every player running your resource at once, and a busy server feels it immediately.

The fix is the same Wait discipline from this whole lesson, plus a guard so each press fires the event exactly once:

```lua
-- Good: the network call only fires on the frame the key goes down.
CreateThread(function()
    while true do
        Wait(0)
        if enabled and IsControlJustPressed(0, 38) then
            TriggerServerEvent('qu_createthread_and_wait:server:ePressed')
        end
    end
end)
```

The loop still polls every frame because it has to catch the one-frame press, but the network call sits behind `IsControlJustPressed`, so it sends once per tap instead of once per frame. The loop is for watching; the event is for the moment something actually changed. Never put a raw, unconditional `TriggerServerEvent` inside `Wait(0)`.

<Callout variant="warning" title="The loop runs on the client, so the server must not trust it">
  Anything inside a client `CreateThread` runs on the player's own machine, and a player can edit or fake what their client sends. A loop that decides "the player earned $500, tell the server to pay them" and fires that as a network event is an open door: a cheater triggers the same event by hand and pays themselves forever. The loop may *notice* the event (the keypress, the proximity), but the server must *re-check the facts and own the outcome* — does this player exist, are they actually where they claim, should this be allowed. Treat every value a loop sends across the wall as a request, never as a result.
</Callout>

This lesson stops at "a loop can fire a network event safely." The full round trip — how the server receives it with `RegisterNetEvent`, who `source` is, and how the server answers the one player who asked versus everyone — is its own build. ## Try it yourself

<Exercise>
  Turn the fixed keypress poll into an adaptive one and prove the cost difference on resmon.

  Replace the `Wait(0)` poll with a loop that defaults to `Wait(500)` and only drops to `Wait(0)` while `enabled` is true:

  ```lua
  CreateThread(function()
      while true do
          local wait = 500
          if enabled then
              wait = 0
              if IsControlJustPressed(0, 38) then
                  print('[qu_createthread_and_wait] E pressed')
              end
          end
          Wait(wait)
      end
  end)
  ```

  Restart `qu_createthread_and_wait`. Open F8, run `resmon`, and watch this resource's ms column. With `enabled` false the loop runs twice a second and the ms reads near zero. Run `/threadtoggle` to flip `enabled` true and watch the same loop jump to running every frame. You have just felt the difference between a loop that sleeps and a loop that polls, on the same code, measured in real numbers instead of guessed at.
</Exercise>

<Exercise>
  Make the countdown announce itself across the wall — once, not sixty times.

  This resource is client-only. To send a network event you need a server file, so first add `server_script 'server.lua'` to `fxmanifest.lua` (below the existing `client_script` line) and create `server.lua` next to `client.lua`. In `server.lua`, register the event and print which player fired it:

  ```lua
  RegisterNetEvent('qu_createthread_and_wait:server:countdownDone', function()
      local src = source
      print('[qu_createthread_and_wait] player ' .. src .. ' finished the countdown')
  end)
  ```

  Now modify the bounded countdown loop in `client.lua` so that on the line where it currently prints `countdown done`, it also fires the event exactly once:

  ```lua
  TriggerServerEvent('qu_createthread_and_wait:server:countdownDone')
  ```

  Restart `qu_createthread_and_wait` and run `/threadtoggle`. Watch the countdown tick down in F8 (the client console), then watch the single `player N finished the countdown` line appear in the txAdmin Live Console (the server console). The lesson here is in the placement: because the event sits after the loop ends — not inside the `Wait(1000)` body — it fires once per countdown, not once per second. Move the `TriggerServerEvent` up into the loop body and restart, and you will see the server console fill with one line every second instead. That is exactly the mistake the warning above tells you never to ship.
</Exercise>

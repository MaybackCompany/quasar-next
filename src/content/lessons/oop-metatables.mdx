local Vehicle = {}

function Vehicle.new(model, plate)
    return setmetatable({ model = model, plate = plate }, Vehicle)

function Vehicle:label()
    return self.model .. ' [' .. self.plate .. ']'
end

## How it works

You ran it and two lines appeared. The first proves a class. The second proves a subclass that reuses the first class and overrides one method. Now take the code apart so the next time you model something in Lua you are deciding, not copying. Everything here rests on one rule that Lua applies when a key is missing from a table, so we start there.

### setmetatable and the constructor

```lua
local Vehicle = {}
Vehicle.__index = Vehicle

function Vehicle.new(model, plate)
    return setmetatable({ model = model, plate = plate }, Vehicle)
end
```

`Vehicle` is just an empty table. There is no `class` keyword in Lua, so a class is only ever a regular table that we agree to treat as a blueprint. The methods will live on this table.

`setmetatable(t, mt)` takes two tables and attaches the second one to the first as its **metatable**. A metatable is a side table that holds rules for how Lua should behave when normal table operations on `t` would otherwise fail or need customizing. It does not change what `t` looks like. It changes what Lua does when something is missing from `t`.

Inside `Vehicle.new` we build a fresh table `{ model = model, plate = plate }`. That table holds the data for one specific vehicle and nothing else: no methods, just two fields. We hand it to `setmetatable` with `Vehicle` as its metatable, and `setmetatable` returns that same table back with the metatable now attached. Returning it from a function named `.new` is the constructor convention in Lua. There is no rule forcing the name `new`; it is a habit the whole ecosystem shares so any reader knows "this builds an instance."

So after `Vehicle.new('sultan', 'QU123')`, the variable `car` is a small table with `model` and `plate`, secretly wired to `Vehicle` through its metatable. That wiring is what the next piece cashes in.

### __index = Vehicle: the lookup fallback

This single line is the whole trick:

```lua
Vehicle.__index = Vehicle
```

Here is the rule Lua follows. When you read a key from a table and the key is **not** present, Lua does not immediately give up and return `nil`. It checks whether the table has a metatable with an `__index` field. If `__index` points to another table, Lua looks the key up in that table instead.

Walk it through with our instance. `car` holds `model` and `plate`. It does not hold `label`. So when you write `car:label()`, Lua first looks for `label` on `car`, does not find it, then follows the metatable's `__index` to `Vehicle`, and finds `label` there. The method lives once on `Vehicle`; every instance borrows it through this fallback. That is why `Vehicle.__index = Vehicle` points the table at itself: missing keys on an instance should fall back to the class table where the methods are defined.

This is also why instances stay tiny. A thousand vehicles do not each carry a copy of `label`. They each carry their own `model` and `plate`, and they all share the one `label` method on `Vehicle`. The metatable is the shared spine; the instance is just its own data.

<Checkpoint question="When you call car:label() and car has no label key, where does Lua find the method, and what exact line makes that work?">
  Lua looks for `label` on `car`, does not find it, then consults `car`'s metatable. The metatable is `Vehicle` (set inside `Vehicle.new` by `setmetatable`), and its `__index` field points back to `Vehicle`. So Lua re-runs the lookup on the `Vehicle` table and finds `label` there. The line that makes it work is `Vehicle.__index = Vehicle`. Remove it and `car:label()` would try to call `nil`, because the fallback would have nowhere to go.
</Checkpoint>

### Colon versus dot: where self comes from

You defined the method with a colon and you called it with a colon. Both matter.

```lua
function Vehicle:label()
    return self.model .. ' [' .. self.plate .. ']'
end
```

The colon in `function Vehicle:label()` is pure sugar. Lua expands it to:

```lua
function Vehicle.label(self)
    return self.model .. ' [' .. self.plate .. ']'
end
```

In other words, defining with `:` silently adds a first parameter named `self`. That is where the `self` inside the body comes from; you never declared it, the colon did. `self` is just the instance the method was called on, so `self.model` reads that specific vehicle's model.

The call side mirrors it. When you write `car:label()`, the colon passes `car` in as that hidden first argument. So `car:label()` is shorthand for `Vehicle.label(car)`, and inside the method `self` is `car`.

This is the part that trips everyone at the start: calling with a dot breaks it. `car.label()` calls the method with **no** first argument, so `self` is `nil`, and `self.model` throws an attempt-to-index-nil error. Rule of thumb: define instance methods with `:` and call them with `:`. Reach for `.` only when a function genuinely does not need an instance, the way `Vehicle.new` does not (it builds the instance, it does not operate on one).

### Single-level inheritance: the PoliceVehicle subclass

The subclass is the same trick applied one level up. The instance falls back to its class; now the class falls back to a parent class.

```lua
local PoliceVehicle = {}
PoliceVehicle.__index = PoliceVehicle
setmetatable(PoliceVehicle, { __index = Vehicle })
```

Read it in two halves. `PoliceVehicle.__index = PoliceVehicle` is the same instance fallback as before: a police instance with a missing key should fall back to the `PoliceVehicle` table. Then `setmetatable(PoliceVehicle, { __index = Vehicle })` gives the `PoliceVehicle` table itself a metatable whose `__index` is `Vehicle`. So when a key is missing on `PoliceVehicle` too, Lua falls back one more step to `Vehicle`. That is the inheritance chain: instance then `PoliceVehicle` then `Vehicle`.

```lua
function PoliceVehicle.new(model, plate, callsign)
    local self = Vehicle.new(model, plate)
    self.callsign = callsign
    return setmetatable(self, PoliceVehicle)
end
```

The subclass constructor reuses the parent. It calls `Vehicle.new` to get a table that already has `model` and `plate`, adds its own `callsign` field, then re-points that table's metatable to `PoliceVehicle` so police instances resolve methods against the subclass first. You build on the parent's data instead of repeating it.

```lua
function PoliceVehicle:label()
    return 'UNIT ' .. self.callsign .. ' - ' .. self.model .. ' [' .. self.plate .. ']'
end
```

This is an **override**. `PoliceVehicle` defines its own `label`, so a police instance finds `label` on `PoliceVehicle` and stops there; it never falls through to `Vehicle.label`. That is why the second printed line is formatted differently from the first. If you deleted this method, `cruiser:label()` would fall back through `PoliceVehicle` to `Vehicle.label` and print in the plain `model [plate]` shape instead. Override only what differs; inherit the rest.

### Why the command uses the restricted flag

```lua
RegisterCommand('cartest', function()
    -- ...
end, true)
```

The third argument to `RegisterCommand` is the restricted flag, and here it is `true`. A restricted command can only be run by a principal that holds the matching `command.cartest` ace permission. The server console holds every ace, which is exactly why `cartest` runs cleanly when you type it in the txAdmin Live Console. An ordinary connected player typing `cartest` in chat would get nothing, because they do not hold that ace.

That is the right default for a server-side proof you do not want players triggering. But you must know the two ways to change it, or the silent-failure will bite you later:

- **Open it to everyone** by passing `false` as the third argument. Then any player can run it from chat. Fine for a harmless demo, wrong for anything privileged.
- **Keep it restricted but grant a player** by adding an ace in `server.cfg`, for example `add_ace identifier.fivem:1 command.cartest allow`. Now that specific player can run the restricted command and everyone else still cannot.

For this lesson, leave it `true` and run the test from the server console. The mental model to keep: `true` does not mean "broken for players," it means "gated behind an ace, and the console always has the ace."

### When to throw this whole pattern away

Metatable OOP earns its keep when you have many instances that share behavior, or a genuine is-a relationship like police-car is-a vehicle. It is the wrong tool more often than beginners think. Reach for a plain table instead when:

- **You have exactly one of the thing.** A single config object or one manager does not need a constructor and a metatable. A plain `{ }` literal with fields is clearer and shorter.
- **The data has no behavior.** If you are only storing values and never attaching methods, a plain table is the data. Wrapping it in a class adds ceremony with no payoff.
- **You would only ever make one instance.** A class you instantiate once is a plain table wearing a costume.

If you find yourself writing `__index` and a `.new` for something you build once and never give methods to, delete the class and use the table directly. Knowing when not to reach for this is as much a part of the skill as the pattern itself.

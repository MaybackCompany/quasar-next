## 1. Plan first or regret it later

The fastest way to waste a weekend is to install resources before you decide what the server is.

Write five things first: the promise, the hosting route, the framework shortlist, the must-have systems, and the dependency order. The plan can change later. Its job today is to stop random installs.

### The map in plain English

A roleplay server is a stack of parts. Each part has one job.

| Part | What it does | Typical 2026 choice |
| --- | --- | --- |
| FiveM server platform | Runs FXServer, networking, resources, and player connections. | Cfx.re FXServer with txAdmin |
| Database | Keeps money, characters, vehicles, jobs, and inventories after a restart. | MariaDB |
| Database connector | Lets server-side resources query the database. | oxmysql |
| Framework | Gives resources a shared model for players, jobs, accounts, and callbacks. | ESX Legacy, QBCore, or Qbox |
| Libraries and systems | Add reusable menus, targeting, inventory, voice, and other shared features. | ox_lib, ox_target, inventory choice, pma-voice |
| Gameplay resources | Add the actual jobs, housing, garages, phones, maps, and other features. | Compatible resources you choose |

The order matters. Upper layers depend on the layers below them. A garage resource cannot save a vehicle if its database tables or connector do not exist. A QBCore job resource cannot run on an ESX server unless the author explicitly supports both.

### Build your server plan

Create a blank file called `server-plan.md`. Then complete each step below. The examples are starting points, not decisions made for you.

<StepGroup>
  <Step num={1} title="Write the server promise" outcome="a stranger can understand the server in one sentence.">
    Write who the server is for, the style of roleplay, and what players should spend most of their time doing.

    ```md
    # My FiveM server plan

    ## Server promise
    A beginner-friendly serious-roleplay city for 32 players, built around police,
    EMS, mechanic work, player-owned businesses, and a controlled economy.
    ```

    Avoid vague promises such as "the best city" or "everything included." A useful promise limits the first version so you can actually test it.
  </Step>

  <Step num={2} title="Choose local or hosted" outcome="you know where the first working server will run.">
    Pick one route for the first setup.

    | Choose | Use it when | First proof |
    | --- | --- | --- |
    | Local Windows PC | You are learning alone and want cheap, safe testing. | FiveM connects to `127.0.0.1:30120`. |
    | Hosted server or VPS | Staff or remote testers need access now. | A second PC connects using the hosted address. |

    A VPS is a virtual private server, a rented Linux or Windows machine you control over the internet.

    Add the decision and reason to your plan.

    ```md
    ## Hosting route
    Route: Local Windows PC
    Why: I am learning alone and do not need public access yet.
    Move to hosting when: two remote testers are ready.
    ```

    Do not mix a Windows-local tutorial with Linux VPS paths. The concepts are the same. The folders, permissions, firewall setup, and service management differ.
  </Step>

  <Step num={3} title="Map the platform and data layers" outcome="you can name the program behind each basic server job.">
    Record these roles in your own words.

    - **FXServer artifact:** the Cfx.re server program. You install it in the next lessons. Always install the latest build at the top of the artifacts list.
    - **txAdmin:** the browser control panel included with FXServer. It starts, stops, monitors, and configures the server.
    - **Cfx.re Portal:** the current home for creating and managing the server license key. Use `https://portal.cfx.re`.
    - **MariaDB:** the database engine that stores persistent server data.
    - **HeidiSQL:** a desktop database client used to inspect MariaDB and import SQL files.
    - **oxmysql:** a FiveM resource that lets server-side scripts query MariaDB.

    <TechLinks ids={["fivem", "txadmin", "cfxportal", "mariadb", "heidisql", "oxmysql"]} />

    Add this section.

    ```md
    ## Foundation
    Platform: FXServer with txAdmin
    License management: Cfx.re Portal
    Database: MariaDB
    Database viewer: HeidiSQL
    Database connector: oxmysql
    ```
  </Step>

  <Step num={4} title="Shortlist a framework" outcome="you compare compatibility instead of choosing from hype.">
    A framework gives resources shared concepts such as players, jobs, money, and callbacks. Use one primary framework.

    - **ESX Legacy:** established ecosystem and broad third-party script availability.
    - **QBCore:** established QB ecosystem with its own conventions and resources.
    - **Qbox:** a QBCore-derived framework built around the ox ecosystem, with an official txAdmin recipe. A recipe is a one-click txAdmin starter that downloads a framework and writes your first `server.cfg`.

    There is no universal winner. Check the official recipe, documentation, maintenance activity, and every must-have resource you plan to use. Do not buy an ESX-only package before choosing Qbox, and do not buy a QB-only package before choosing ESX.

    <TechLinks ids={["esx", "qbcore", "qbox"]} />

    ```md
    ## Framework shortlist
    Candidate 1:
    Candidate 2:
    Must-have resources checked for compatibility:
    Final choice: Not chosen until compatibility checks are complete
    ```
  </Step>

  <Step num={5} title="Separate must-have systems from later ideas" outcome="the first playable version has a controlled scope.">
    Make two lists.

    The first list contains only what is required for a player to join and complete your server's basic roleplay loop. Everything else waits.

    ```md
    ## Must have for the first playable test
    - Framework and character system
    - Database connector
    - Inventory
    - Voice
    - One legal job and one emergency job
    - One vehicle shop and garage path

    ## Add after the first test passes
    - Housing
    - Custom phone
    - Extra jobs
    - Paid maps and vehicle packs
    ```

    This is scope control, not a permanent restriction. A small server that starts cleanly is useful. A huge resource folder with 200 errors is not.
  </Step>

  <Step num={6} title="Build the dependency map" outcome="each resource has an owner, source, and prerequisites before installation.">
    For each must-have system, record the exact resource, official source, supported framework, database needs, and dependencies.

    ```md
    ## Dependency map
    | System | Resource | Framework support | Needs | SQL included | Official source checked |
    | --- | --- | --- | --- | --- | --- |
    | Database connector | oxmysql | Framework independent | MariaDB | No | Yes |
    | Shared UI/library | ox_lib | Check each resource | None | No | Yes |
    | Inventory | To decide | To verify | Framework, ox_lib, oxmysql | To verify | No |
    | Voice | pma-voice | Standalone | OneSync/audio config | No | No |
    ```

    OneSync is the FiveM networking mode that raises the player limit above the vanilla 32-slot cap. You will set it later.

    Never assume a dependency from a folder name. Read the current README, documentation, release notes, and `fxmanifest.lua` for the exact version you will install.
  </Step>

  <Step num={7} title="Turn dependencies into an install order" outcome="you have a safe sequence and a test after every layer.">
    Install the foundation before gameplay resources. The exact names vary, but the reasoning stays stable.

    ```text
    FXServer + txAdmin
      -> database and connection string
      -> database connector
      -> framework recipe and framework dependencies
      -> shared libraries
      -> inventory, target, voice, and other core systems
      -> one gameplay resource
      -> test, back up, then add the next resource
    ```

    Do not copy that diagram directly into `server.cfg`. Official recipes often manage multiple resources and category folders for you. The later `server.cfg` lesson teaches the real `ensure` order for the stack you install. `ensure` is the `server.cfg` command that starts a resource at boot.

    ```md
    ## Install and test order
    1. Start the base recipe and confirm txAdmin has no repeating errors.
    2. Confirm the database connection before installing database-backed scripts.
    3. Join with one clean test account.
    4. Install one resource.
    5. Restart that resource and check txAdmin plus F8.
    6. Back up the working state before adding another resource.
    ```

    F8 is the in-game client console. You open it by pressing F8 while the FiveM client is in focus, and it reports client-side script errors.
  </Step>

  <Step num={8} title="Define the first proof" outcome="you know exactly when planning is finished and setup can begin.">
    Finish your file with a pass condition.

    ```md
    ## First proof
    [ ] Hosting route chosen
    [ ] Framework shortlist checked against must-have resources
    [ ] Foundation tools named
    [ ] Dependencies recorded from official sources
    [ ] Install order written
    [ ] First test defined: one player connects and creates a character
    [ ] Rollback rule: back up before every new resource or update
    ```

    <ExpectedOutput>
      `server-plan.md` explains what you are building, where it will run, which framework candidates fit, what the required systems depend on, and what you will test first.
    </ExpectedOutput>
  </Step>
</StepGroup>

## 2. What you need before you start

You need a legitimate GTA V installation and FiveM client, a Cfx.re account, a local Windows PC or hosted machine, an editor, an archive tool, and a safe place for credentials. Database-backed frameworks also need MariaDB plus a database client such as HeidiSQL or DBeaver.

Create the server license key through [portal.cfx.re](https://portal.cfx.re/). The old `keymaster.fivem.net` address now redirects to the same portal, so either link lands you in the right place.

Treat the license key, database password, Tebex secrets, and webhooks as credentials. Do not place them in screenshots or public repositories.

## 3. Installing server artifacts

An artifact is the FXServer program. It is not a GTA game build, and it is not your resource folder. Download it only from the official Cfx.re artifacts page.

For a Windows server, use this page:

```text
https://runtime.fivem.net/artifacts/fivem/build_server_windows/master/
```

For a Linux server, use this one:

```text
https://runtime.fivem.net/artifacts/fivem/build_proot_linux/master/
```

**Install the latest build at the very top of the list.** That is the current build. Ignore the green "recommended" row further down, because it lags behind.

There is one fallback rule for later. If a brand-new build ever misbehaves after an update, drop to the build directly below it. That is the only reason to run an older one.

You do not pin or freeze a build now. Pinning is a production-ops practice you use once the server is live and public, never at first install.

Keep server binaries separate from `server-data`, resources, configuration, and database backups.

## 4. Setting up with txAdmin

txAdmin ships with FXServer. It is the browser panel for profiles, recipes, console output, resources, players, scheduled restarts, and server settings.

The safe first run is one straight line:

1. Launch FXServer.
2. Open the printed txAdmin URL.
3. Link the correct Cfx.re account.
4. Create a profile.
5. Choose an official recipe or blank template.
6. Verify the data path.
7. Let the recipe finish before editing files.

Run `status` in the Live Console once it is up. Save the recipe name and artifact build in your notes.

## 5. How it all fits together

The request path is easier to debug when every layer has one name.

```text
FiveM client
  -> FXServer and txAdmin
  -> framework resources
  -> oxmysql
  -> MariaDB
```

Gameplay resources sit on the framework and shared libraries. MariaDB stores persistent records. HeidiSQL only lets an operator inspect that database. It is not part of the running FiveM request path.

## 6. server.cfg walkthrough

`server.cfg` is the startup plan. It declares endpoints, server identity, convars, the license key, permissions, and resource order. A convar is a configuration variable you set in `server.cfg` to control a resource or the server itself.

Use the config generated by the current recipe as the starting point.

```cfg
endpoint_add_tcp "0.0.0.0:30120"
endpoint_add_udp "0.0.0.0:30120"
sv_hostname "My RP Server"
sv_maxclients 48
onesync on
sv_licenseKey "cfxk_yourkeyhere"

ensure oxmysql
# framework recipe resources next
# shared libraries and systems next
# gameplay resources last
```

Two lines deserve a note.

`sv_licenseKey` takes the `cfxk_` string from the Cfx.re Portal, pasted verbatim with no prefix in front of it. The value is the whole `cfxk_...` string and nothing else.

`onesync on` is the modern default. The vanilla player cap is 32 slots. Enable `onesync on` to go above that, up to 2048 slots. Only `onesync legacy` and `onesync off` are legacy modes.

That block is a teaching skeleton, not a complete production file. The dedicated `server.cfg` lessons provide the complete version.

## 7. Keeping files organized

Keep disposable server binaries apart from data you own.

Inside `resources`, bracket folders such as `[framework]`, `[standalone]`, `[voice]`, `[maps]`, and `[vehicles]` group related resources. They are category folders, not resources themselves.

Do not rename a downloaded resource until you know its exports, dependencies, or documentation do not expect the original name. Keep one backup outside the live server and record every manual edit.

## 8. Tools you will use

| Tool | Job |
| --- | --- |
| VS Code | Edit configs, Lua, JavaScript, and documentation. |
| 7-Zip or WinRAR | Extract downloaded archives without hiding extensions. |
| WinSCP or another SFTP client | Move files to a hosted Linux server. |
| MariaDB | Store persistent server data. |
| HeidiSQL or DBeaver | Inspect schemas, import SQL, and run controlled queries. |
| Git and GitHub Desktop | Record changes and restore known-good files. |
| txAdmin | Operate FXServer and read server console output. |

## 9. MariaDB, HeidiSQL, and oxmysql

MariaDB is the database engine. HeidiSQL is an operator interface. oxmysql is the server resource that runs parameterized queries from scripts.

A framework may use tables named `users`, `players`, or something else. Never assume one framework's schema applies to another.

Keep the connection string server-side. Confirm the database host accepts connections from the server. Never expose credentials with a replicated convar, because a replicated convar is sent to clients.

## 10. Running SQL safely

Back up before importing vendor SQL.

Select the correct database, open the file, and read it. Stop if it contains destructive statements you do not understand. `CREATE TABLE` is expected during setup. `DROP TABLE`, broad `DELETE`, and schema-changing `ALTER TABLE` statements need a migration and rollback plan.

After running the file, refresh the schema and confirm the exact tables or columns the resource documentation names. Do not ignore every "table already exists" message. Compare the expected and installed schema first.

## 11. ESX vs QBCore vs Qbox

- **ESX Legacy** has an established ecosystem and many third-party resources.
- **QBCore** has its own established conventions, documentation, and package ecosystem.
- **Qbox** is QBCore-derived, leans on the ox ecosystem, and provides an official recipe and migration guidance.

Choose based on maintained recipes, the compatibility of your must-have resources, staff knowledge, and migration cost.

Do not install two primary frameworks together unless you are executing a documented migration or bridge plan.

## 12. Adding resources

Read the README first. Confirm framework, dependencies, database changes, config path, and required artifact or game build.

Then work in order: back up, install dependencies, upload one resource, import its reviewed SQL, add it after its dependencies, and run `refresh` followed by `ensure RESOURCE_NAME`.

Check the server console, F8, and the visible feature. Do not install the next resource until this one passes.

## 13. What is fxmanifest.lua?

Every resource uses `fxmanifest.lua` to declare its metadata, scripts, files, and dependencies.

```lua
fx_version 'cerulean'
game 'gta5'

shared_script 'config.lua'
client_script 'client/main.lua'
server_script 'server/main.lua'

dependencies {
    'ox_lib'
}
```

`fx_version 'cerulean'` is the manifest format version the resource is written for. `cerulean` is the current one.

FiveM uses Lua 5.4 by default in 2026. Do not add the retired `lua54 'yes'` directive.

A dependency declaration tells FiveM what must be started first. Your `server.cfg` should still make the intended order readable.

## 14. Vectors and placing things

`vector3(x, y, z)` stores a position.

`vector4(x, y, z, heading)` stores a position plus facing direction.

Whether a config needs three or four values depends on the resource. Use a heading when the spawned ped, player, or vehicle must face a direction. Do not treat vector4 as required for every spawn.

## 15. Script config patterns

A good config exposes safe choices without forcing buyers to edit core logic.

Expect tables for jobs, item names, prices, coordinates, framework adapters, webhooks, and feature toggles.

Change one group at a time. Preserve data types and commas. Then restart only the affected resource and test the changed path.

## 16. Tracking changes with Git

Track server configuration and editable resources in Git.

Exclude secrets, caches, logs, database dumps, and escrowed or licensed files you are not permitted to publish. Escrowed means the seller encrypts the code so you can run it but not read or edit it.

A private repository does not make a leaked credential safe. If a secret is committed, rotate it.

Commit before each risky install and after each verified result. That turns "what changed?" into a diff instead of a guessing session.

## 17. Performance

Use `resmon` and the profiler before calling a resource slow.

Avoid per-frame work when the player is far away. Do not run database queries inside hot loops. Add timeouts to model or network waits.

Measure idle and active behavior with the same test conditions, then change one cause at a time.

## 18. Security

The client requests. The server decides.

Validate permissions, ownership, item counts, money, distance, state, and rate limits on the server before granting a valuable result.

Never place database credentials, license keys, Tebex secrets, or webhook URLs in client or shared files.

ACE is the FiveM permission system. It grants an object (a permission such as `command.kick`) to a principal (a player or a group). A valid pattern grants an object to a group, then attaches a player identifier to that group. An identifier is the unique string FiveM uses to recognize a player, such as their license or Discord ID.

## 19. Debugging

Read the first real error, not the last symptom.

The server console reports server scripts and startup failures. F8 reports client scripts.

Check the named resource, file, line, missing dependency, wrong runtime side, database connection, and config syntax in that order.

Use `ensure` for the first start, `restart` after a running resource changes, and `refresh` when a new resource folder was added while the server was already running.

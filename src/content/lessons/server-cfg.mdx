>
 `server.cfg` is the startup script for your city. It names the server, sets the port, holds your secrets, and decides the **order** every resource loads. FXServer reads it top to bottom, so a script that needs `ox_lib` or `oxmysql` only works if those are `ensure`d **above** it. In this lesson you read a real config, then build two tiny resources that *prove* order matters without touching a single production script.
</LessonHero>

<Callout variant="fivem" title="Order is a rule, not a suggestion">
 FXServer starts resources in the exact order their `ensure` lines appear in `server.cfg`. A dependency must come first; the script that needs it comes second. You will not memorize this. You will watch one console line flip from `provider found: pong` to `provider MISSING` and never forget it.
</Callout>

## Build it

<StepGroup>
 <Step num={1} title="Open server.cfg and find the ensure block" outcome="You can point at the line where resources start.">
 Open your server folder in VS Code (right-click the folder, **Open with Code**), then open `server.cfg`. Scroll until you find the run of lines that start with `ensure`. That block is the load order. A typical 2026 config looks like this:

 ```text
 endpoint_add_tcp "0.0.0.0:30120"
 endpoint_add_udp "0.0.0.0:30120"

 sv_hostname "My ESX Server"
 sv_maxclients 48

 # Game build (optional — remove or update to match a current supported build)
 # sv_enforceGameBuild <latest-tested-build>   # pin a specific GTA content build, or omit to let FiveM auto-detect

 set mysql_connection_string "mysql://root:PASSWORD@127.0.0.1/fivem?charset=utf8mb4"
 set mysql_slow_query_warning 150

 sv_licenseKey "PASTE_YOUR_CFX_PORTAL_KEY_HERE"

 ensure oxmysql
 ensure ox_lib
 ensure es_extended
 ensure ox_inventory
 ensure ox_target
 ```

 <ExpectedOutput>
 You can put your cursor on the first `ensure` line and say out loud: "everything below here starts in this order."
 </ExpectedOutput>
 </Step>

 <Step num={2} title="Know what the important lines do" outcome="No line in the config is a mystery to you.">
 Three of those lines hold the keys to the whole server:

 - `sv_licenseKey` is your **Cfx.re Portal** key. Get it free at `portal.cfx.re` (this is the current portal; older guides call it Cfx.re Portal). It is required even for a local test server.
 - `mysql_connection_string` is how `oxmysql` reaches your database. `mysql_slow_query_warning 150` makes the console warn you about any query slower than 150 ms.
 - `sv_enforceGameBuild` pins which GTA content build players load. The FXServer artifacts page lists **server software** builds, not GTA content builds. Keep the official recipe's value unless your assets require another build, then use the supported list in the Cfx.re server-command reference.

 <Callout variant="fivem" title="Two of these lines are secrets">
 `sv_licenseKey` and `mysql_connection_string` contain secrets. Never paste their real values into Discord, public GitHub, screenshots, videos, or support tickets. Blur them first. If one leaks, revoke the key in the Cfx.re Portal and rotate the database password immediately.
 </Callout>
 </Step>

 <Step num={3} title="Read the ensure block as a dependency map" outcome="You can say why each line sits where it does.">
 Order is not random. Read it as layers, each one depending on the layer above:

 ```text
 # 1. Database bridge first, anything that saves data needs it
 ensure oxmysql

 # 2. Shared helper library second
 ensure ox_lib

 # 3. Framework core third
 ensure es_extended

 # 4. Core gameplay systems after the framework
 ensure ox_inventory
 ensure ox_target

 # 5. Your scripts last, after everything they depend on
 ensure qs-smartphone-pro
 ```

 A script that calls `lib.` functions needs `ox_lib` above it, because `lib` does not exist until `ox_lib` loads and the script pulls it in (Step 6 shows exactly how). A script that saves to the database needs `oxmysql` above it. A job script needs its framework above it. Get one wrong and the script errors at startup, not when you expect it.

 First, one word you need: an **export** is a function one resource publishes so other resources can call it (the FiveM way resources talk to each other). A resource can only answer an export call after it has started.

 Here is the same idea reduced to its smallest form, so you can see exactly what flips. `es_extended` calls `oxmysql` exports the moment it starts. Put the database layer first and those exports already exist. Put the framework first and it reaches for exports that have not been registered yet:

 ```cfg
 # CORRECT: dependency first, dependent second
 ensure oxmysql # registers the MySQL exports
 ensure es_extended # calls exports.oxmysql:* on start -> they exist, boots clean

 # WRONG: dependent before dependency -> startup error
 ensure es_extended # tries to use oxmysql exports that do not exist yet -> errors
 ensure oxmysql # too late, the framework already failed above
 ```

 The fix is never "restart it again" or "ensure it twice." The fix is to move the dependency line **above** the line that needs it, save, and reload. Order is the whole mechanism.

 Here is an **example** chain for a script whose own README and manifest name all three dependencies. Do not assume every framework needs every ox resource; install only what the official recipe and the resource manifest require.

 ```cfg
 ensure oxmysql # database bridge, registers MySQL exports first
 ensure ox_lib # shared library, defines the global lib used by the rest
 ensure es_extended # framework; its recipe requires oxmysql
 ensure qs-smartphone-pro # example only: place it after every dependency its docs name
 ```

 <Callout variant="fivem" title="Organize it into sections">
 A real `server.cfg` grows fast, so group it with comment headers instead of one long wall of lines: an endpoints section, an identity section, a secrets section, then the `ensure` block as its own marked region. Comments start with `#` and FXServer ignores them. The order of your `ensure` lines is the only thing that controls load order, but a tidy file is what lets you find the one line that is in the wrong place. The companion lesson [server.cfg, annotated line by line](/lessons/server-cfg-annotated) shows the full sectioned layout.
 </Callout>
 </Step>

 <Step num={4} title="Build two throwaway resources to test order safely" outcome="You have a provider and a consumer that report whether the order worked.">
 Never test order by restarting `ox_lib`, a framework, or a paid script on a live server, you can kick players and break everything downstream. Instead build two tiny local resources: no players, no data, no risk.

 Create this layout inside your `resources` folder. The `[local]` bracket folder keeps your own test scripts grouped:

 ```text
 resources/[local]/qu_dep_provider/
 fxmanifest.lua
 server.lua
 resources/[local]/qu_dep_consumer/
 fxmanifest.lua
 server.lua
 ```

 `qu_dep_provider/fxmanifest.lua`:

 ```lua
 fx_version 'cerulean'
 game 'gta5'

 server_script 'server.lua'
 ```

 `qu_dep_provider/server.lua`:

 ```lua
 exports('Ping', function()
 return 'pong'
 end)

 print('[qu_dep_provider] export ready') -- this text appears in the txAdmin Live Console when the resource starts
 ```

 `qu_dep_consumer/fxmanifest.lua`:

 ```lua
 fx_version 'cerulean'
 game 'gta5'

 server_script 'server.lua'
 ```

 `qu_dep_consumer/server.lua`:

 ```lua
 CreateThread(function()
 local ok, result = pcall(function()
 return exports.qu_dep_provider:Ping()
 end)

 if ok and result == 'pong' then
 print('[qu_dep_consumer] provider found: ' .. result)
 else
 print('[qu_dep_consumer] provider MISSING (wrong ensure order)')
 end
 end)
 ```

 Notice neither manifest has a `lua54` line. Lua 5.4 is the only runtime in 2026 (Lua 5.3 was removed in June 2025), so the `lua54` setting is deprecated and you do not need to enable it. Leave it out.
 </Step>

 <Step num={5} title="Run the order test, then break it on purpose" outcome="You watch one console line flip when the order is wrong.">
 Make a backup copy of `server.cfg` first. Then add both resources in dependency-first order, near the bottom, and save:

 ```text
 ensure qu_dep_provider
 ensure qu_dep_consumer
 ```

 Open txAdmin in your browser, click **Live Console** in the left sidebar, then type each command into the input box at the bottom of that page and press Enter (one line at a time). Stop both resources first, then start the provider and consumer in that order:

 ```text
 stop qu_dep_consumer
 stop qu_dep_provider
 start qu_dep_provider
 start qu_dep_consumer
 ```

 The consumer found the provider because it loaded second. Your exact lines may print in a slightly different order, and may carry a resource-name prefix the console adds, but the meaning is what matters (illustrative):

 <ExpectedOutput>
 [qu_dep_provider] export ready
 [qu_dep_consumer] provider found: pong
 </ExpectedOutput>

 Note: a resource's exports only become callable after FXServer's first scheduler tick, which is why the consumer waits inside CreateThread before calling the provider.

 Now break it. Swap the two lines so the consumer is ensured **first**, save, then stop both and start them in the wrong order. Merely restarting the consumer while the provider is still running would not test startup order.

 ```text
 ensure qu_dep_consumer
 ensure qu_dep_provider
 ```

 ```text
 stop qu_dep_consumer
 stop qu_dep_provider
 start qu_dep_consumer
 start qu_dep_provider
 ```

 The consumer ran before the provider's export existed, so it reports the failure, the exact problem a real dependency hits when it loads too early (illustrative; line order and any console prefix may vary):

 <ExpectedOutput>
 [qu_dep_consumer] provider MISSING (wrong ensure order)
 [qu_dep_provider] export ready
 </ExpectedOutput>

 Put the lines back in provider-first order and restore your backup if you changed anything else. You proved the rule from Step 3 with your own eyes.
 </Step>

 <Step num={6} title="Check a real script's dependencies before you install it" outcome="You can predict the right ensure position without restarting anything live.">
 For real paid scripts, inspect, do not restart. Open the resource's `fxmanifest.lua` and read its `dependencies` block and any shared scripts it pulls in:

 ```lua
 fx_version 'cerulean'
 game 'gta5'

 shared_script '@ox_lib/init.lua'

 dependencies {
 'ox_lib',
 'oxmysql',
 'es_extended'
 }
 ```

 That `@ox_lib/init.lua` line is not a hint, it is the mechanism. The `@resource/file.lua` syntax tells FXServer to run another resource's file inside this one before its own scripts. So `shared_script '@ox_lib/init.lua'` runs ox_lib's init file here, and that init is what defines the global `lib` table this script then calls. If `ox_lib` has not started yet, the `@ox_lib/init.lua` include has nothing to pull in, `lib` is nil, and the script errors the instant it touches `lib.`. That is the real reason `ox_lib` must be `ensure`d above any script that uses it.

 The `dependencies` block says the same thing in plain words: this script needs `ox_lib`, `oxmysql`, and the framework `ensure`d above it. Confirm each one already sits higher in `server.cfg`, then add the script below them. No live restart required.

 <Callout variant="fivem" title="The resources folder, at a glance">
 Bracket folders like `[local]`, `[jobs]`, or `[cars]` are organizers, FiveM still sees the resources inside them, and `ensure [cars]` starts every resource in that bracket. Your tree usually looks like `resources/[standalone]/ox_lib`, `resources/[qb]/qb-core`, `resources/[local]/qu_dep_provider`, and so on.
 </Callout>
 </Step>
</StepGroup>

## If something went wrong

<FailureList
 items={[
 { symptom: "Couldn't find resource qu_dep_provider", fix: "The folder name and the ensure name must match exactly. Check for a typo or a wrapping zip folder, the resource folder must hold fxmanifest.lua directly." },
 { symptom: "provider MISSING when you expected found", fix: "The consumer was ensured above the provider. Put ensure qu_dep_provider first, save, then restart the provider before the consumer." },
 { symptom: "No such export Ping", fix: "The provider failed to start or has a typo in exports('Ping', ...). Restart qu_dep_provider and read the line above the error in the Live Console." },
 { symptom: "Access denied for user / Unknown database 'fivem'", fix: "Your mysql_connection_string is wrong or the database does not exist yet. Fix the connection string and create the database before ensuring oxmysql-dependent scripts." },
 { symptom: "The command is not recognized", fix: "Type server commands like restart in the txAdmin Live Console, not in the in-game F8 client console." },
 ]}

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| Resource loads but says "dependency missing" | The dependency is below it in server.cfg, or not installed | Move `ensure your_script` BELOW `ensure the_dependency` in server.cfg |
| "Could not find resource X" | The resource folder name doesn't match | Folder names are case-sensitive. `my_resource` ≠ `My_Resource`. Match exactly |
| Server starts but no resources load | server.cfg has a typo or missing line | Check every `ensure` line matches an actual resource folder name |
| `sv_licenseKey` not recognized | Wrong format or missing quotes | Must be: `sv_licenseKey "licensekey:cfxk_..."` with quotes |
| Two resources both need each other (circular dependency) | One must start first, or they need restructuring | Move shared code to a third resource that both depend on |

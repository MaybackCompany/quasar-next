  <Step num={4} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `client.lua` and paste this:

    ```lua
local objects = {}

RegisterCommand('spawnbox', function()
    local model = joaat('prop_boxpile_07d')
    RequestModel(model)
    while not HasModelLoaded(model) do Wait(0) end

    local coords = GetEntityCoords(PlayerPedId())
    local object = CreateObject(model, coords.x + 2.0, coords.y, coords.z, false, false, false)
    objects[object] = true
    SetModelAsNoLongerNeeded(model)
    print('[qu_spawning_entities_safely] spawned object ' .. object)

    SetTimeout(30000, function()
        if DoesEntityExist(object) then
            DeleteEntity(object)
            objects[object] = nil
            print('[qu_spawning_entities_safely] timeout deleted object ' .. object)
        end
    end)
end, false)

RegisterCommand('clearboxes', function()
    local count = 0
    for object in pairs(objects) do
        if DoesEntityExist(object) then DeleteEntity(object) end
        objects[object] = nil
        count = count + 1
    end
    print('[qu_spawning_entities_safely] registry cleared, removed ' .. count)
end, false)

AddEventHandler('onResourceStop', function(name)
    if name ~= GetCurrentResourceName() then return end
    for object in pairs(objects) do
        if DoesEntityExist(object) then DeleteEntity(object) end
    end
end)
    ```
  </Step>

## How it works

You spawned a box, cleared it, and watched a stray one delete itself. Now take the code apart so the next time you need a prop, an NPC, or a parked car you are deciding, not copying. The whole file is built from three ideas stacked on top of each other: load a model, create an entity from it, and never lose track of what you created.

### Loading the model before you spawn

```lua
local model = joaat('prop_boxpile_07d')
RequestModel(model)
while not HasModelLoaded(model) do Wait(0) end
```

A model in GTA is an asset on disk: a mesh, textures, a collision shape. You cannot place one in the world until the game has it loaded into memory. These three lines are the load lifecycle, and you will repeat this exact shape for every entity you ever spawn.

`joaat('prop_boxpile_07d')` turns the model **name** into a **hash**. The game does not look entities up by readable string names at runtime; it looks them up by a number called a joaat hash (the name comes from Jenkins one-at-a-time, the hashing algorithm Rockstar uses). `joaat` is the FiveM helper that computes that number for you from the string. You will also see `GetHashKey('prop_boxpile_07d')` in older code. It does the exact same thing and returns the exact same number. `joaat` is the newer, shorter name; prefer it, but treat the two as interchangeable.

`RequestModel(model)` asks the engine to stream that model in. It does not block and it does not finish instantly. It just files the request and returns.

`while not HasModelLoaded(model) do Wait(0) end` is the part beginners skip and then wonder why their prop never appears. `RequestModel` is a request, not a guarantee. You have to wait until the model is actually in memory before you spawn from it, or `CreateObject` runs against a model that is not ready and you get nothing. `HasModelLoaded` returns `true` once the load finishes. The loop keeps yielding one frame at a time with `Wait(0)` until that happens. `Wait(0)` is what makes this safe: it hands control back to the game each frame instead of freezing it, so the engine actually gets time to do the loading you asked for. A bare `while not HasModelLoaded(model) do end` with no `Wait` would hang the client.

### Creating the object

```lua
local coords = GetEntityCoords(PlayerPedId())
local object = CreateObject(model, coords.x + 2.0, coords.y, coords.z, false, false, false)
```

`PlayerPedId()` is your character's entity handle, and `GetEntityCoords` returns its position as a vector with `.x`, `.y`, and `.z` fields. You read your own position so you can spawn the box near yourself instead of at some fixed corner of the map.

The `coords.x + 2.0` is deliberate. If you spawned at exactly `coords.x, coords.y, coords.z` the box would appear inside your character, which looks broken and can shove you. Adding 2.0 to the x coordinate offsets the spawn two metres to the side so it lands in front of you where you can see it. Coordinates in GTA are in metres.

`CreateObject` is the native that actually places the prop and returns its **entity handle**, the number you saw printed. Read its arguments in order:

- `model` is the loaded hash from step one.
- `coords.x + 2.0, coords.y, coords.z` is where to place it.
- the first `false` is **isNetwork**. `false` means this prop is local to your client only; other players will not see it. Pass `true` to register it with the network so everyone sees the same object.
- the second `false` is **netMissionEntity** (also called registerAsNetworked or the script-host flag). It only matters when the object is networked; it controls whether the object is treated as a mission entity that survives ownership changes. For a local prop it stays `false`.
- the last `false` is **doorFlag**, a niche flag for door objects. Leave it `false` for ordinary props.

The original draft passed `true, true` for the first two flags, which networks the prop to every player. For a single-player teaching proof that is overkill and makes the entity behave differently than the lesson claims, so this version keeps it local with `false, false`. The line in **When you reach for this pattern** below tells you exactly when to flip them.

### Freeing the model

```lua
SetModelAsNoLongerNeeded(model)
```

Once the object exists in the world, the engine no longer needs your reserved copy of the model in memory. `SetModelAsNoLongerNeeded` tells it you are done with that asset so it can be unloaded when nothing else is using it. This does not delete the object you just spawned; the object holds its own reference. Skipping this line will not break a single spawn, but a spawner that runs hundreds of times and never frees its models slowly leaks memory. Call it every time, right after the spawn succeeds.

### The registry and why it exists

```lua
objects[object] = true
```

This one line is what the lesson title is really about. `objects` is a plain Lua table declared at the top of the file. Every time you spawn, you record the new handle as a key. The value `true` is just a marker; you only care that the key exists. Now `objects` is a live list of everything this resource has created and has not yet deleted.

Why bother? Because a handle is the only way to delete an entity later, and the moment `spawnbox` returns, that local `object` variable is gone. Without a registry you would have spawned a box you can never reference again, which means you can never clean it up. Tracking handles in a table is the difference between a spawner and a litter machine.

```lua
for object in pairs(objects) do
    if DoesEntityExist(object) then DeleteEntity(object) end
    objects[object] = nil
    count = count + 1
end
```

`clearboxes` walks the registry and deletes each tracked entity. Two details carry the safety:

- `DoesEntityExist(object)` guards every `DeleteEntity`. An entity can already be gone for reasons outside your control: the timeout fired, another resource removed it, the area streamed out. Calling `DeleteEntity` on a handle that no longer points at anything is asking for trouble. The guard means you only delete what is genuinely still there.
- `objects[object] = nil` removes the key from the table after handling it. If you deleted the entity but left the handle in the registry, the next clear would try to delete a dead handle, and your registry would grow forever with ghosts.

### Auto-delete and restart cleanup

```lua
SetTimeout(30000, function()
    if DoesEntityExist(object) then
        DeleteEntity(object)
        objects[object] = nil
        print('[qu_spawning_entities_safely] timeout deleted object ' .. object)
    end
end)
```

`SetTimeout(30000, fn)` is a FiveM helper that runs `fn` once after 30000 milliseconds. It is the resource's safety net: even if you never type `clearboxes`, every box removes itself after 30 seconds. Notice the same `DoesEntityExist` guard, because by the time the timeout fires you may have already cleared that box by hand; the guard stops a double delete. This timeout branch is half the point of the lesson, which is why the expected output above shows its `timeout deleted object` line.

```lua
AddEventHandler('onResourceStop', function(name)
    if name ~= GetCurrentResourceName() then return end
    for object in pairs(objects) do
        if DoesEntityExist(object) then DeleteEntity(object) end
    end
end)
```

`onResourceStop` fires for every resource that stops, so the first line checks the stopping resource is this one with `GetCurrentResourceName()` and bails out otherwise. When it is your resource, the handler sweeps the registry and deletes whatever is still alive. This is the line that makes the lesson "safe": without it, every time you restart the resource during development, the boxes you already spawned would be orphaned in the world with no script left tracking them, and only a full server restart would clear them. Cleaning up on stop means a restart is always a clean slate.

<Checkpoint question="Why does the code track handles in the objects table instead of just deleting the prop right after CreateObject?">
  Because spawning and deleting happen at different moments, and the handle is the only reference you have to the entity. The `spawnbox` handler finishes the instant the box appears, and its local `object` variable disappears with it. The box, however, stays in the world. `clearboxes`, the 30-second timeout, and `onResourceStop` all run later and need a way to find every box that is still out there. Storing each handle as a key in `objects` keeps a live list across those separate moments, so any of them can loop the table, guard with `DoesEntityExist`, and delete cleanly. Delete-immediately would defeat the purpose; the whole point is a prop that exists for a while and is still tracked.
</Checkpoint>

### Entity handle versus network id

The number you printed is an **entity handle**: a runtime reference that is only valid on the client that created it, only while the entity exists. It is not a stable id and it is not shared. If you spawn the box as networked (the first `CreateObject` flag set to `true`), the entity also gets a **network id** that every client agrees on, which you read with `NetworkGetNetworkIdFromEntity(object)`. Other clients then turn that network id back into their own local handle. For this lesson the box is local, so there is no network id and no other player sees it. Keep the distinction straight: handles are local and per-client, network ids are the shared identity that crosses the wire.

### The same shape for peds and vehicles

Props are the gentlest entity to start with, but `CreateObject` is one of three siblings. Peds and vehicles load their models the exact same way and only differ in the create call:

```lua
-- A pedestrian: CreatePed(pedType, model, x, y, z, heading, isNetwork, netMissionEntity)
local model = joaat('a_m_y_business_03')
RequestModel(model)
while not HasModelLoaded(model) do Wait(0) end
local ped = CreatePed(4, model, coords.x + 2.0, coords.y, coords.z, 0.0, false, false)
SetModelAsNoLongerNeeded(model)

-- A vehicle: CreateVehicle(model, x, y, z, heading, isNetwork, netMissionEntity)
local model = joaat('adder')
RequestModel(model)
while not HasModelLoaded(model) do Wait(0) end
local veh = CreateVehicle(model, coords.x + 4.0, coords.y, coords.z, 90.0, false, false)
SetModelAsNoLongerNeeded(model)
```

`CreatePed` takes a `pedType` as its first argument; `4` means a civilian network ped and is the safe default. `CreateVehicle` takes a `heading` in degrees so the car faces a direction. Everything else is identical: same load loop, same `SetModelAsNoLongerNeeded`, same idea of tracking the returned handle in a registry and deleting it the same way. Learn the prop and you have learned all three.

### When you reach for this pattern

Registry-tracked spawning is not a beginner toy you leave behind. It is the backbone of real resources:

- **Placeable props.** A furniture system, a sign placer, a crime-scene markers tool. Players spawn objects, the resource tracks them, and a clear or a restart never litters the map.
- **Job and event vehicles.** A mechanic that spawns a tow truck, a heist that spawns getaway cars. You hold the handles so you can despawn them when the job ends. For these you set the first `CreateObject`/`CreateVehicle` flag to `true` so every player sees the same entity, and you usually spawn from the server for shared ownership.
- **NPC encounters.** A shopkeeper, a quest giver, a wave of enemies. Spawn with `CreatePed`, register the handles, and tear them down when the scene is over so peds never pile up.

The discipline is always the same: load, create, register, guard, delete on stop. Once that loop is muscle memory you can spawn anything in GTA without leaking it into the world.

## Try it yourself

<Exercise>
  Extend the spawner so it can place a parked car as well as a box, and prove your cleanup tracks both. Add a `spawncar` command that loads the `adder` model, calls `CreateVehicle` four metres in front of you, and records the returned handle in the **same** `objects` registry:

  ```lua
  RegisterCommand('spawncar', function()
      local model = joaat('adder')
      RequestModel(model)
      while not HasModelLoaded(model) do Wait(0) end

      local coords = GetEntityCoords(PlayerPedId())
      local veh = CreateVehicle(model, coords.x + 4.0, coords.y, coords.z, 90.0, false, false)
      objects[veh] = true
      SetModelAsNoLongerNeeded(model)
      print('[qu_spawning_entities_safely] spawned vehicle ' .. veh)
  end, false)
  ```

  Restart the resource, spawn one box and one car, then run `clearboxes`. Because both handles live in the same registry, the single clear loop deletes both and reports `removed 2`. That is the payoff of the pattern: one registry, one cleanup path, no matter how many entity types you add.
</Exercise>

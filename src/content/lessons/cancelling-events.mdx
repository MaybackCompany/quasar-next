AddEventHandler('qu_cancelling_events:check', function(name)

## How it works

Read the blocked output again, because it contains the single most counter-intuitive fact in the whole event system. You called `CancelEvent()` in the first handler, yet the second handler still ran. It printed `second handler saw cancel, skipping`. If `CancelEvent()` truly stopped the event, that line could never appear. So what does it actually do?

### CancelEvent() raises a flag, it does not stop the chain

When you trigger `qu_cancelling_events:check`, FiveM does not run one handler. It runs **every** handler registered for that event name, one after another, in the order they were registered. That ordered list is the **handler chain**. `CancelEvent()` does not break out of the chain. It sets a single boolean flag on the event that is currently running, and then the chain keeps going to the next handler exactly as if nothing happened.

So in plain terms, this is the sequence for `canceltest blocked`:

- The command prints the `--- firing check ---` line and calls `TriggerEvent`.
- FiveM runs the **first** handler. It prints `first handler checked blocked`, sees `name == 'blocked'`, calls `CancelEvent()` (flag now raised), and prints `event cancelled`.
- The first handler returns. The chain is not done. FiveM runs the **second** handler.
- The second handler asks `WasEventCanceled()`. The flag is raised, so it returns `true`, and the handler chooses to `return` early.

That last point is the part beginners miss: the second handler skipped its work because **you wrote it to check the flag**, not because the engine forced it to. Delete the `if WasEventCanceled()` guard and the second handler will happily print `second handler allowed blocked` even though the event was cancelled. The cancel did nothing on its own. Cancellation in FiveM is cooperative: every downstream handler has to opt in to respecting it.

### WasEventCanceled() is the downstream check, and order decides who sees it

`WasEventCanceled()` only makes sense inside a handler for an event that can still be running. It reads the flag the engine set when somebody upstream called `CancelEvent()`. The word **upstream** is doing real work here: a handler can only see a cancel that was raised **before** it ran. Registration order is therefore the whole game.

In your code, the cancelling handler is registered first and the checking handler is registered second, so the check sees the flag. Swap the two `AddEventHandler` blocks and the result flips: the handler that prints `allowed` now runs first, before the flag exists, so it prints `allowed blocked` regardless. Then the cancelling handler runs and raises a flag nobody downstream is left to read. The cancel becomes useless. When you need a veto to actually protect something, the protecting handler must be registered **after** the deciders, or it will check a flag that has not been set yet.

<Checkpoint question="You add a third handler to qu_cancelling_events:check that prints 'third handler ran'. Where in the chain does it run, and will it run when the event is cancelled?">
  It runs last, because it was registered after the other two, and FiveM runs handlers in registration order. And yes, it runs even when the event is cancelled, because `CancelEvent()` never stops the chain. It only raises a flag. The third handler will print `third handler ran` on both `canceltest blocked` and `canceltest open` unless you add your own `if WasEventCanceled() then return end` guard at the top of it. The engine forces nothing; each handler decides for itself whether to honour the cancel.
</Checkpoint>

### Cancellation is local: it does not cross the network

There is one more boundary that trips people up, and it is the reason this lesson is built server-only. `CancelEvent()` and `WasEventCanceled()` operate on the **local handler chain**, on the side where the event is running. They have no reach across the client and server boundary.

Concretely, you cannot have a server handler `CancelEvent()` a client event, and you cannot stop a `TriggerServerEvent` from the client by cancelling it on the server before the work runs in your own code. When a client does `TriggerServerEvent('something')`, by the time your server handler is executing, the event has already crossed the wire and arrived. Cancelling it only affects the rest of the server-side chain for that event, the same way you saw above. It does not reach back to the client. If you want the server to refuse something a client asked for, you do not cancel; you simply do not perform the action and you keep the authority on the server. Cancellation is for vetoing handlers on the same side, not for blocking a message in flight.

### The pattern that matters: vetoing playerConnecting

The toy chain above teaches the mechanics. Here is the production use the intro promised. `playerConnecting` is a built-in server event that fires while a player is still on the loading screen, before they are in the world. Cancelling it stops the join. This is exactly how ban systems and allowlists kick a player before any other resource even sees them.

```lua
AddEventHandler('playerConnecting', function(name, setKickReason, deferrals)
    local src = source
    local license

    for _, id in ipairs(GetPlayerIdentifiers(src)) do
        if id:find('license:') then
            license = id
            break
        end
    end

    if BANNED_LICENSES[license] then
        CancelEvent()
        setKickReason('You are banned from this server.')
    end
end)
```

Read it against everything above. `playerConnecting` runs a handler chain just like your test event did, and the engine itself checks `WasEventCanceled()` after the chain finishes: if any handler raised the flag, the connection is refused. `CancelEvent()` here raises that flag, and `setKickReason` is the callback FiveM hands you to attach the message the player sees. The `deferrals` argument is the third parameter and lets you hold the player on the loading screen while you run a slow check, for example a database lookup, before deciding; you call `deferrals.defer()` to start, `deferrals.update('message')` to show progress, and `deferrals.done()` or `deferrals.done('reason')` to admit or reject. The mechanics are identical to your two-handler toy: a flag, read downstream. The only difference is that downstream the reader is the engine, and the consequence is a real kick instead of a skipped print.

### When you reach for each tool

You now have three different ways to stop work, and they are not interchangeable:

- **CancelEvent()** when you are one handler in a chain for an event the engine or other resources also listen to, and you want to veto the shared outcome. `playerConnecting`, `populationPedCreating`, and other built-in cancellable events are the real targets.
- **return** when you simply want to exit your own handler early. It stops your code, nothing else. The second handler in your build used `return` to skip its own work after reading the flag.
- **deferrals** when you need to pause `playerConnecting` to run an async check before you decide. Cancel is instant; deferrals buy you time.

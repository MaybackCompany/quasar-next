    ```lua
AddEventHandler('qu_cancelling_events:check', function(name)
    print('[qu_cancelling_events] first handler checked ' .. name)
    if name == 'blocked' then
        CancelEvent()
        print('[qu_cancelling_events] event cancelled')
    end
end)

AddEventHandler('qu_cancelling_events:check', function(name)
    print('[qu_cancelling_events] second handler still ran for ' .. name)
end)

RegisterCommand('canceltest', function(_, args)
    local name = args[1] or 'allowed'
    print('[qu_cancelling_events] --- firing check for ' .. name .. ' ---')
    TriggerEvent('qu_cancelling_events:check', name)
    print('[qu_cancelling_events] caller saw cancelled = ' .. tostring(WasEventCanceled()))
end, true)
-- The final `true` is the "restricted" flag: it locks this command behind the
-- command.canceltest ACE permission. The server console (txAdmin Live Console
-- or the FXServer window) always has permission, so run `canceltest` THERE,
-- not from in-game chat, where a normal player would be silently denied.
    ```
  </Step>

## How it works

Read the blocked output again, because it contains the single most counter-intuitive fact in the whole event system. You called `CancelEvent()` in one handler, yet the other handler still ran. If `CancelEvent()` stopped dispatch, that line could never appear. So what does it actually do?

### CancelEvent() raises a flag, it does not stop the chain

When you trigger `qu_cancelling_events:check`, FiveM dispatches the event to its registered handlers. `CancelEvent()` does not stop those other handlers. It records that the current event was cancelled so the code that triggered it, or the game for a cancellable built-in event, can inspect that outcome. Do not depend on handlers running in registration order; the documented contract does not promise one.

So in plain terms, this is the sequence for `canceltest blocked`:

- The command prints the `--- firing check ---` line and calls `TriggerEvent`.
- The checking handler sees `name == 'blocked'`, calls `CancelEvent()`, and prints `event cancelled`.
- The other handler still runs and prints its line.
- After `TriggerEvent` finishes, the command calls `WasEventCanceled()` and prints `true`.

That last point is the part beginners miss: the second handler runs even though the event was cancelled. For a custom local event, the caller decides what to do with the result. For a built-in cancellable event, such as `playerConnecting`, the game understands the flag and applies its documented effect.

### WasEventCanceled() is the caller's check

For the local event in this lesson, call `WasEventCanceled()` immediately after `TriggerEvent` returns. It reports whether any handler cancelled that dispatch. In Lua, `TriggerEvent` also returns the cancellation result, so this is equivalent:

```lua
local canceled = TriggerEvent('qu_cancelling_events:check', name)
print('cancelled = ' .. tostring(canceled))
```

Do not build a permission or security system around one event handler checking what another handler did. Keep the protected action and its validation together, and treat the event's final cancellation result as a signal to the caller.

<Checkpoint question="You add a third handler to qu_cancelling_events:check that prints 'third handler ran'. Will it run when another handler cancels the event?">
  Yes. `CancelEvent()` does not prevent the remaining handlers from running. Do not predict where its line appears relative to the other handlers, because handler order is not a documented guarantee. After all handlers finish, the caller still sees a final cancellation result of `true`.
</Checkpoint>

### Cancellation is local: it does not cross the network

There is one more boundary that trips people up, and it is the reason this lesson is built server-only. `CancelEvent()` and `WasEventCanceled()` operate on the **local handler chain**, on the side where the event is running. They have no reach across the client and server boundary.

Concretely, you cannot have a server handler `CancelEvent()` a client event, and you cannot stop a `TriggerServerEvent` from the client by cancelling it on the server before the work runs in your own code. When a client does `TriggerServerEvent('something')`, by the time your server handler is executing, the event has already crossed the wire and arrived. Cancelling it only affects the rest of the server-side chain for that event, the same way you saw above. It does not reach back to the client. If you want the server to refuse something a client asked for, you do not cancel; you simply do not perform the action and you keep the authority on the server. Cancellation is for vetoing handlers on the same side, not for blocking a message in flight.

### The pattern that matters: vetoing playerConnecting

The toy chain above teaches the mechanics. Here is the production use the intro promised. `playerConnecting` is a built-in server event that fires while a player is still on the loading screen, before they are in the world. Cancelling it stops the join. This is exactly how ban systems and allowlists kick a player before any other resource even sees them.

```lua
AddEventHandler('playerConnecting', function(name, setKickReason, deferrals)
    local src = source
    local license

    for _, id in ipairs(GetPlayerIdentifiers(src)) do
        if id:find('license:') then
            license = id
            break
        end
    end

    if BANNED_LICENSES[license] then
        CancelEvent()
        setKickReason('You are banned from this server.')
    end
end)
```

<Callout variant="warning" title="Which identifier to ban on">
  This example matches on `license:` because it is platform-independent and easy to read. Be aware the `license` id can change if the player switches their Rockstar / Social Club account, so it is not a perfect ban key. For real ban lists Cfx recommends the `fivem` identifier (the player's Cfx.re user id). No single identifier is guaranteed always-present or unspoofable, so production ban systems usually store several.
</Callout>

Read it against everything above. `playerConnecting` runs a handler chain just like your test event did, and the engine itself checks `WasEventCanceled()` after the chain finishes: if any handler raised the flag, the connection is refused. `CancelEvent()` here raises that flag, and `setKickReason` is the callback FiveM hands you to attach the message the player sees. The `deferrals` argument is the third parameter and lets you hold the player on the loading screen while you run a slow check, for example a database lookup, before deciding; you call `deferrals.defer()` to start, `deferrals.update('message')` to show progress, and `deferrals.done()` or `deferrals.done('reason')` to admit or reject. The mechanics are identical to your two-handler toy: a flag, read downstream. The only difference is that downstream the reader is the engine, and the consequence is a real kick instead of a skipped print.

### When you reach for each tool

You now have three different ways to stop work, and they are not interchangeable:

- **CancelEvent()** when you are one handler in a chain for an event the engine or other resources also listen to, and you want to veto the shared outcome. `playerConnecting`, `populationPedCreating`, and other built-in cancellable events are the real targets.
- **return** when you simply want to exit your own handler early. It stops your code, nothing else. The second handler in your build used `return` to skip its own work after reading the flag.
- **deferrals** when you need to pause `playerConnecting` to run an async check before you decide. Cancel is instant; deferrals buy you time.

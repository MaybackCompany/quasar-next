    ```lua
AddEventHandler('qu_cancelling_events:check', function(name)
    print('[qu_cancelling_events] first handler checked ' .. name)
    if name == 'blocked' then
        CancelEvent()
        print('[qu_cancelling_events] event cancelled')
    end
end)

AddEventHandler('qu_cancelling_events:check', function(name)
    print('[qu_cancelling_events] second handler still ran for ' .. name)
end)

RegisterCommand('canceltest', function(_, args)
    local name = args[1] or 'allowed'
    print('[qu_cancelling_events] --- firing check for ' .. name .. ' ---')
    TriggerEvent('qu_cancelling_events:check', name)
    print('[qu_cancelling_events] caller saw cancelled = ' .. tostring(WasEventCanceled()))
end, true)
-- The final `true` is the "restricted" flag: it locks this command behind the
-- command.canceltest ACE permission. The server console (txAdmin Live Console
-- or the FXServer window) always has permission, so run `canceltest` THERE,
-- not from in-game chat, where a normal player would be silently denied.
    ```
  </Step>

## How it works

Read the blocked output again. You called `CancelEvent()` in one handler, yet the other handler still ran. That is the whole lesson.

### CancelEvent() raises a flag, it does not stop the chain

`CancelEvent()` does not stop other handlers. It records that this event was cancelled so the code that triggered it, or the engine for a built-in event, can read that outcome afterward.

Here is the sequence for `canceltest blocked`:

- The command prints `--- firing check ---` and calls `TriggerEvent`.
- The first handler sees `name == 'blocked'`, calls `CancelEvent()`, and prints `event cancelled`.
- The second handler still runs and prints its line.
- After `TriggerEvent` returns, the command calls `WasEventCanceled()` and prints `true`.

The second handler runs even though the event was cancelled. Do not depend on the order the handlers run in; the docs do not promise one.

### WasEventCanceled() is the caller's check

Call `WasEventCanceled()` right after `TriggerEvent` returns. It tells you whether any handler cancelled that dispatch. In Lua, `TriggerEvent` returns the same result, so this is equivalent:

```lua
local canceled = TriggerEvent('qu_cancelling_events:check', name)
print('cancelled = ' .. tostring(canceled))
```

`WasEventCanceled()` only works for local events. Do not build a security check out of one handler reading what another did; keep the protected action and its validation together.

<Checkpoint question="You add a third handler to qu_cancelling_events:check that prints 'third handler ran'. Will it run when another handler cancels the event?">
  Yes. `CancelEvent()` does not stop the other handlers. After they all finish, the caller still sees `WasEventCanceled()` return `true`.
</Checkpoint>

### Cancellation is local: it does not cross the network

`CancelEvent()` and `WasEventCanceled()` only work on the side where the event is running. They do not cross the client and server boundary. This is why the lesson is built server-only.

You cannot cancel a client event from the server, and you cannot block a `TriggerServerEvent` by cancelling it on the server. By the time your server handler runs, the message has already arrived; cancelling only affects the rest of the server-side chain. To refuse what a client asked for, do not cancel - just keep the action and its authority on the server.

### The pattern that matters: vetoing playerConnecting

The toy chain teaches the mechanics. Here is the real use. `playerConnecting` is a built-in server event that fires while a player is still on the loading screen, before they are in the world. Cancel it and the join is refused. This is how ban systems and allowlists kick a player before any other resource sees them.

```lua
AddEventHandler('playerConnecting', function(name, setKickReason, deferrals)
    local src = source
    local license

    for _, id in ipairs(GetPlayerIdentifiers(src)) do
        if id:find('license:') then
            license = id
            break
        end
    end

    if BANNED_LICENSES[license] then
        CancelEvent()
        setKickReason('You are banned from this server.')
    end
end)
```

<Callout variant="warning" title="Which identifier to ban on">
  This example matches on `license:` because it is platform-independent and easy to read. Be aware the `license` id can change if the player switches their Rockstar / Social Club account, so it is not a perfect ban key. For real ban lists Cfx recommends the `fivem` identifier (the player's Cfx.re user id). No single identifier is guaranteed always-present or unspoofable, so production ban systems usually store several.
</Callout>

This is the same flag you just built, read by the engine instead of by your own code. `playerConnecting` runs a handler chain; after it finishes the engine checks `WasEventCanceled()`, and if any handler raised the flag the connection is refused. `CancelEvent()` raises it, and `setKickReason` is the function FiveM hands you to set the message the player sees.

The third argument, `deferrals`, lets you hold the player on the loading screen while you run a slow check such as a database lookup before deciding:

- `deferrals.defer()` starts it (then wait one tick).
- `deferrals.update('message')` shows progress.
- `deferrals.done()` admits the player, `deferrals.done('reason')` rejects them.

### When you reach for each tool

You now have three different ways to stop work, and they are not interchangeable:

- **CancelEvent()** when you are one handler in a chain for an event the engine or other resources also listen to, and you want to veto the shared outcome. `playerConnecting` and other built-in cancellable events are the real targets.
- **return** when you only want to exit your own handler early. It stops your code and nothing else.
- **deferrals** when you need to pause `playerConnecting` to run an async check before you decide. Cancel is instant; deferrals buy you time.

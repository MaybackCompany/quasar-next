## Build it

<StepGroup>
  <Step num={1} title="Install VS Code and turn on file extensions" outcome="Right-clicking a folder shows Open with Code, and File Explorer shows full file names.">
    Download VS Code from [code.visualstudio.com/download](https://code.visualstudio.com/download), run the **Windows User Installer**, and accept the defaults. On the **Additional Tasks** screen, tick both **Open with Code** context-menu options.

    Then open File Explorer, go to the **View** menu, and turn on **File name extensions** and **Hidden items**.

    Why extensions matter: a broken file often looks like `server.cfg.txt`. FiveM needs `server.cfg`. Hidden extensions hide that mistake from you.

    To prove it, create a file on your Desktop:

    ```text
    Desktop\proof.cfg
    ```

    <ExpectedOutput>
      Right-clicking a folder shows "Open with Code", and File Explorer shows the full name proof.cfg (not proof.cfg.txt and not just proof).
    </ExpectedOutput>
  </Step>

  <Step num={2} title="Install an archive tool" outcome="You can extract the .7z and .zip files FiveM ships artifacts in.">
    FiveM artifacts come as compressed archives. Install **7-Zip** from [7-zip.org](https://www.7-zip.org/) (free) or **WinRAR** from [win-rar.com](https://www.win-rar.com/). Use the 64-bit Windows version and accept the defaults.

    To prove it, right-click any `.7z` or `.zip` file. The menu now offers an **Extract** option from your archive tool.

    <ExpectedOutput>
      A .7z or .zip file shows an Extract option in the right-click menu.
    </ExpectedOutput>
  </Step>

  <Step num={3} title="Install WinSCP for later" outcome="WinSCP is installed and ready for VPS file transfers in a future lesson.">
    You do not need this for a local server, but installing it now saves time later. Download WinSCP from [winscp.net/eng/download.php](https://winscp.net/eng/download.php), run the installer with default settings, and **do not connect to anything yet**.

    <ExpectedOutput>
      WinSCP opens to a login screen. Close it for now.
    </ExpectedOutput>
  </Step>

  <Step num={4} title="Create your Cfx.re account and free license key" outcome="The key is stored in a password manager and ready for txAdmin.">
    Every FiveM server needs a license key, **even a local test server**. The key authorizes your server to connect to the Cfx.re platform and run, and it ties your server to your Cfx.re account for the server list and entitlements. The key is **free** and comes from the Cfx.re **Portal** at portal.cfx.re, not from any old "Keymaster" page.

    1. Create or verify your account at [forum.cfx.re](https://forum.cfx.re/) and confirm the email.
    2. Go to the portal at [portal.cfx.re](https://portal.cfx.re/).
    3. Open **Server Management** and create a new license.
    4. Copy the key into your password manager. If you must use a temporary text file during setup, keep it outside the repository, restrict access to your Windows account, and delete it after txAdmin is configured.

    Treat the key like a password. Never paste it in Discord, GitHub, screenshots, or videos. If it leaks, revoke it on the portal and generate a new one.

    <ExpectedOutput>
      Your password manager contains the new key, ready to paste into txAdmin without showing it in a screenshot or commit.
    </ExpectedOutput>
  </Step>

  <Step num={5} title="Install MariaDB" outcome="MariaDB runs as a Windows service on 127.0.0.1:3306.">
    Download the latest Windows x86_64 MSI from [mariadb.org/download](https://mariadb.org/download/) and run the installer.

    1. Set a strong **root password** and store it in a password manager, not a Desktop text file.
    2. Enable **utf8mb4** as the default character set (not the legacy 3-byte `utf8`). This is the full 4-byte Unicode set that stores any player name or emoji, and it matches the collation you set in Step 6. Modern FiveM schemas (ESX Legacy, Qbox) assume it.
    3. Keep port **3306** and install it as a **Windows service**.

    Confirm the service is running. Open PowerShell and run:

    ```powershell
    Get-Service MariaDB*
    ```

    <ExpectedOutput>
      Status   Name      DisplayName
      ------   ----      -----------
      Running  MariaDB   MariaDB
    </ExpectedOutput>

    If the service has a slightly different name, open Windows **Services** and look for **MariaDB** or **MySQL**. The proof you want is the status **Running**.
  </Step>

  <Step num={6} title="Install HeidiSQL and create the database" outcome="HeidiSQL connects to MariaDB and shows a database named fivem.">
    Download HeidiSQL from [heidisql.com/download.php](https://www.heidisql.com/download.php) and install it. Then create a new session:

    1. Name the session `local FiveM`.
    2. Host `127.0.0.1`, user `root`, password from your password manager, port `3306`.
    3. Open the session, then create a database named `fivem`.
    4. Use collation `utf8mb4_unicode_ci`. This pairs with the utf8mb4 charset from Step 5, so player names with accents or emoji store and sort correctly instead of breaking on insert.

    To prove the connection, click the `fivem` database, open a query tab, and run:

    ```text
    SELECT DATABASE();
    ```

    <ExpectedOutput>
      The result grid shows a single value: fivem
    </ExpectedOutput>

    Here is what your screen looks like once it works. The session is open on the left, the `fivem` database sits in the tree, and the query you ran returns its result in the grid below.

    <Mockup app="HeidiSQL · local FiveM">

  </Step>

## How MariaDB and HeidiSQL relate (and why scripts care)

You just installed two things that sound like one thing. They are not. Knowing which is which is the difference between fixing a database problem in ten seconds and uninstalling the wrong program at midnight.

### MariaDB is the engine, HeidiSQL is the window

**MariaDB** is the database engine. It is a background Windows service with no real screen of its own. It listens on `127.0.0.1:3306`, holds your tables on disk, and answers SQL it is handed. When you ran `Get-Service MariaDB*` in Step 5 and saw **Running**, that service is the engine doing its job whether or not any window is open.

**HeidiSQL** is a separate program that connects to that engine and draws it for you: the session on the left, the database tree, the query editor, the result grid. Close HeidiSQL and your data is untouched, because the data never lived in HeidiSQL. It lives in MariaDB. HeidiSQL is just the window you look through.

This is why the two install steps are separate. Step 5 installs the engine and proves it is running. Step 6 installs the window and proves it can see in. If HeidiSQL says "connection refused," the window is fine; the engine behind it is stopped. If HeidiSQL says "Access denied," the window reached the engine but handed it the wrong password.

### How a script reaches the same database

Your FiveM scripts never open HeidiSQL, and they are not MariaDB either. They reach the engine through a third piece you install later as a resource: **oxmysql**, the community connector built by Overextended. You drop it into `resources`, `ensure` it in `server.cfg`, and point it at the same engine HeidiSQL connects to, using one convar:

```cfg
# server.cfg — oxmysql reads this, then your scripts query through it
set mysql_connection_string "mysql://root:YOUR_PASSWORD@localhost/fivem?charset=utf8mb4"
ensure oxmysql
```

Read that string left to right and it is the same four facts you typed into the HeidiSQL session: user `root`, your password, host `localhost` (the same machine as `127.0.0.1`), and the `fivem` database. HeidiSQL connects so a human can look. oxmysql connects so scripts can read and write. Both knock on the same door at `127.0.0.1:3306`. That shared door is the whole point of getting the database right now, before txAdmin or any framework asks for it.

So the chain a script follows is one hop longer than the one you do by hand:

```text
You          ->  HeidiSQL  ->  MariaDB (127.0.0.1:3306)  ->  fivem database
A FiveM script ->  oxmysql  ->  MariaDB (127.0.0.1:3306)  ->  fivem database
```

When you reach the hands-on lesson [HeidiSQL hands-on](/lessons/heidisql-hands-on), you will use that same window to import a framework's tables and browse player rows. Everything you build there is just the human path into the engine your scripts will hit through oxmysql.

<Checkpoint question="HeidiSQL is closed, but your script still cannot read the database. Where is the problem most likely to be, and where is it definitely not?">
  Not in HeidiSQL. HeidiSQL is only a window for humans; closing it changes nothing for scripts, because scripts go through oxmysql, not HeidiSQL. The problem is in the path your script actually uses: either the MariaDB service is stopped (run `Get-Service MariaDB*` and confirm Running), or the `mysql_connection_string` in `server.cfg` has the wrong user, password, host, or database name. Both HeidiSQL and oxmysql hit the same engine at `127.0.0.1:3306`, so if HeidiSQL can connect with a given user and password, point `mysql_connection_string` at exactly those values.
</Checkpoint>

## If something went wrong

  items={[
    { symptom: "You saved server.cfg.txt instead of server.cfg", fix: "Turn on File name extensions in File Explorer, then rename the file so it ends in .cfg with no .txt." },
    { symptom: "MariaDB installer says the port is already in use", fix: "MySQL, XAMPP, or Laragon is already on 3306. Stop that service or pick one database stack, then reinstall on port 3306." },
    { symptom: "HeidiSQL says Access denied for user 'root'", fix: "The password is wrong. Retrieve it from your password manager. If it is genuinely lost, follow MariaDB's official password-reset procedure rather than reinstalling over data." },
    { symptom: "HeidiSQL says connection refused", fix: "The MariaDB service is stopped. Run Get-Service MariaDB* and start it from Windows Services if it is not Running." },
    { symptom: "HeidiSQL connects fine but a script later cannot reach the database", fix: "Scripts go through oxmysql, not HeidiSQL.

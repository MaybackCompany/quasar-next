  A standard FiveM config binds both TCP and UDP on the same game port (default `30120`), and both protocols must be allowed by the host firewall. Do not reduce the whole connection flow to "TCP is the handshake, UDP is gameplay": FXServer exposes multiple HTTP/TCP and UDP services, and the public-list ingress behavior has changed over time. ## Build it

We are going to make the networking config concrete. This is a server-only resource that reads your live network settings and prints them to the console, so the ports, the OneSync mode, and the player cap stop being words in a guide and become numbers you can see. Follow the names exactly. Once it prints, the next section takes it apart and explains what each number controls.

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_networking_basics
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_networking_basics/
  fxmanifest.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Set the endpoints in server.cfg" outcome="The server binds TCP and UDP on the same port.">
    Open `server.cfg`. Near the top, where your other networking lines live, make sure both endpoints are declared on the same port:

    ```cfg
    endpoint_add_tcp "0.0.0.0:30120"
    endpoint_add_udp "0.0.0.0:30120"
    set qu_game_port 30120
    ```

    `0.0.0.0` means "listen on every network interface on this machine", and `30120` is the port players connect to. The extra `qu_game_port` line is a lesson-owned convar so the probe below can report the number; keep it equal to the endpoint port. Cfx.re's default config binds both protocols, so allow both through the firewall.
  </Step>

  <Step num={4} title="Write fxmanifest.lua" outcome="FiveM knows which file to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
    fx_version 'cerulean'
    game 'gta5'

    server_script 'server.lua'
    ```

    No `lua54` line. As of June 2025 that directive is deprecated and ignored, because Lua 5.4 is now the only Lua runtime.
  </Step>

  <Step num={5} title="Write the lesson code" outcome="The topic is now a runnable probe of your real config.">
    Open `server.lua` and paste this:

    ```lua
    RegisterCommand('netinfo', function()
        local port = GetConvarInt('qu_game_port', 0)
        local maxClients = GetConvarInt('sv_maxClients', 48)
        local oneSync = GetConvar('onesync', 'off')

        print('[qu_networking_basics] game port (TCP+UDP): ' .. port)
        print('[qu_networking_basics] txAdmin panel port:  check txAdmin host settings (default 40120)')
        print('[qu_networking_basics] sv_maxClients:        ' .. maxClients)
        print('[qu_networking_basics] onesync:              ' .. oneSync)
    end, true)
    ```
  </Step>

  <Step num={6} title="Start and test it" outcome="Your real networking numbers appear in the server console.">
    Add this line to `server.cfg`:

    ```text
    ensure qu_networking_basics
    ```

    Save, then run:

    ```text
    restart qu_networking_basics
    ```

    Run this test from the txAdmin Live Console:

    ```text
    netinfo
    ```

    <ExpectedOutput>
      [qu_networking_basics] game port (TCP+UDP): 30120
      [qu_networking_basics] txAdmin panel port:  check txAdmin host settings (default 40120)
      [qu_networking_basics] sv_maxClients:        48
      [qu_networking_basics] onesync:              on
    </ExpectedOutput>

    Your `sv_maxClients` and `onesync` values will be whatever your own `server.cfg` sets. The point is that they are now readable, not memorized.
  </Step>
</StepGroup>

## How it works

You just printed the four numbers that decide whether players can reach you, how many fit, and how much bandwidth each one costs. Take the code apart so each line maps to a real decision instead of a magic value.

### Reading your live config

```lua
local port = GetConvarInt('qu_game_port', 0)
local maxClients = GetConvarInt('sv_maxClients', 48)
local oneSync = GetConvar('onesync', 'off')
```

`GetConvar` and `GetConvarInt` are the server-side way to read a configuration variable that was set in `server.cfg`. The first argument is the convar name. The second argument is a fallback default: if the convar was never set, you get that value instead of `nil`. `GetConvarInt` returns a number; `GetConvar` returns a string, which is why `onesync` (whose values are the words `off`, `on`, and `legacy`) is read as a string while the port and player cap are read as integers.

The mental model: a convar is a setting outside your code that a script can read. The bound port still comes from `endpoint_add_udp` and `endpoint_add_tcp`; `qu_game_port` is only a lesson-owned label that you keep in sync. It does not introspect the socket and is not proof that a firewall rule works. The real proof is the endpoint lines, the host firewall/provider rules, and an external connection test.

### Why the port needs both TCP and UDP

This is the single most common reason a fresh server is unreachable. The connection happens in two phases on the same port number.

TCP and UDP are separate protocols, so firewalls treat `30120/TCP` and `30120/UDP` as separate rules even though the number matches. Cfx.re's default config binds both. If one is blocked, connection, file-service, discovery, or live-session behavior can fail in ways that vary by path and proxy. Open both on the same game port, then verify from outside the host network.

### Game port 30120 versus txAdmin port 40120

These get confused constantly. `30120` is the conventional **game port** in the default config. `40120` is txAdmin's **default** web-panel port, but hosts can change it. The panel is for administrators, not players: restrict it with the provider firewall, VPN, or trusted source IPs where your hosting setup supports that.

<Checkpoint question="A player says your server shows up in the list and starts connecting, but freezes at the loading screen and times out. The server console shows no errors. What is the most likely cause?">
  First confirm that both `30120/TCP` and `30120/UDP` are allowed at the OS firewall, provider firewall, and any router/NAT, then test externally. A listing entry alone does not prove either inbound path is healthy. If both are open, continue with server console errors, proxy configuration, resource downloads, and client-specific issues.
</Checkpoint>

### Telling a server problem from a network problem

When players complain about lag, there are two completely different homes for the cause, and they need opposite fixes.

**Server-side lag** is your scripts being slow. You read this with **resmon** (type `resmon` in F8 on a client, or watch the server thread time in txAdmin). Each resource shows a per-frame millisecond cost. If one resource is over budget, the whole server hitches for everyone at once, and no amount of network tuning helps. Fix the script.

**Network-side lag** is the connection between players and your machine: high ping, packet loss, jitter. You read this in-game with the **netgraph**. Turn it on with the client command `netgraph 1` (`netgraph true` also works) and watch the ping and packet-loss numbers. For a per-frame perf and packet-loss overlay, use `cl_drawperf 1`. If the server thread time is healthy in txAdmin but the netgraph shows high ping or dropped packets, the problem is the network, not your code, and that is when distance, bandwidth, and attacks come into play.

The diagnostic order is always the same: check resmon and the server thread first, because that is the cause you can fix yourself. Only when the server side is healthy do you look at the network.

### How OneSync and the player cap drive bandwidth

```lua
local maxClients = GetConvarInt('sv_maxClients', 48)
local oneSync = GetConvar('onesync', 'off')
```

These two are not just settings; they are your bandwidth budget. `sv_maxClients` is how many players the server allows at once. **OneSync** is the modern state-synchronization mode that lets a server go past the old 32-slot ceiling and have the server, not the clients, own the authoritative game state.

The link to bandwidth is direct: every additional player is more entity and event traffic the server must replicate to everyone else. OneSync `on` is required once you cross the old limits (32-plus slots need OneSync, and above 64 you must run `onesync on`), and a higher `sv_maxClients` with OneSync on is a heavier, steadier outbound load on your connection. This matters because a server that is comfortable at 24 players can saturate its uplink at 100, and the symptom looks exactly like network lag even though nobody is attacking you. Knowing your cap and your sync mode tells you how much bandwidth you are signing up for.

Routing buckets, instancing, and other advanced OneSync features all require OneSync to be enabled in the first place, so this single setting is the gate to most modern multiplayer behavior.

### What DDoS protection and a proxy actually do

A DDoS attack floods your connection with junk traffic so that real players' packets cannot get through. The defense is not on your server at all; it is upstream, in the network.

A **scrubbing center** or **filtered IP** is a machine that sits in front of your server and absorbs the flood. Concretely, at the packet level: all traffic for your advertised address arrives at the scrubbing center first, which drops the malicious packets and forwards only the clean UDP gameplay traffic on to your real machine, the **origin**. Players connect to the protected front; the attack hits the front and dies there.

A **proxy** adds the second half: it keeps your origin IP private. If attackers know the raw IP of your actual machine, they can route the flood straight at it and bypass the protection entirely. A proxy means the only address the public ever sees is the protected front, and your origin stays hidden behind it. The combination is "filtered IP plus hidden origin", and it is why a leaked origin IP undoes everything.

This is almost entirely a **hosting feature**. A beginner does not hand-build a scrubbing center. The practical move is to choose a host whose plan includes filtered or anti-DDoS networking, and to never let your raw origin IP leak. Note that the FiveM convar `sv_endpointPrivacy` hides player IP addresses from the server's public reports; it does not hide your server's own origin IP, so do not mistake it for DDoS protection.

## Try it yourself

<Exercise>
  Prove your own port is actually reachable, end to end. First, confirm the server is listening: restart `qu_networking_basics`, run `netinfo`, and read back your real `30120`, `sv_maxClients`, and `onesync` values from the console. Then test the UDP path specifically, which is the one beginners forget.

  Test from outside your network by connecting a FiveM client to the public address while watching the server console. Generic web port-check sites often test TCP only and cannot reliably prove a stateful UDP game service. Confirm the OS firewall, provider firewall, and router/NAT each allow both protocols on your configured game port, then record which layer was blocking the connection.

  Write down two lines: which ports your server reports listening on, and the result of the external UDP check (open or closed, and what you changed to fix it).
</Exercise>

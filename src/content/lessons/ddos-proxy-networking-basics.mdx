<Callout variant="fivem" title="Two ports, two jobs">
 A standard FiveM config opens both TCP and UDP on the same game port. The default is `30120`. Your host firewall must allow both. Do not reduce this to "TCP is the handshake, UDP is gameplay". FXServer runs several services over both protocols. Separately, txAdmin uses its own web port. The default is `40120`, and that one can be changed.
</Callout>

## Build it

We are going to make the networking config concrete. This is a server-only resource. It reads your live network settings and prints them to the console. The ports, the OneSync mode, and the player cap stop being words in a guide and become numbers you can see. Follow the names exactly. Once it prints, the next section takes it apart and explains what each number controls.

<StepGroup>
 <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
 A **resource** is a folder of code FiveM loads. Inside your server's `resources` folder, create this folder:

 ```text
 resources/qu_networking_basics
 ```
 </Step>

 <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
 Create this exact file layout:

 ```text
resources/qu_networking_basics/
 fxmanifest.lua
 server.lua
 ```
 </Step>

 <Step num={3} title="Set the endpoints in server.cfg" outcome="The server binds TCP and UDP on the same port.">
 Open `server.cfg`. Near the top, where your other networking lines live, declare both endpoints on the same port:

 ```cfg
 endpoint_add_tcp "0.0.0.0:30120"
 endpoint_add_udp "0.0.0.0:30120"
 set qu_game_port 30120
 ```

 `0.0.0.0` means "listen on every network interface on this machine". `30120` is the port players connect to. The extra `qu_game_port` line is a **convar**, a setting outside your code that a script can read. This one is lesson-owned so the probe below can report the number. Keep it equal to the endpoint port. Cfx.re's default config binds both protocols, so allow both through the firewall.
 </Step>

 <Step num={4} title="Write fxmanifest.lua" outcome="FiveM knows which file to load.">
 `fxmanifest.lua` is the manifest file in every resource. It tells FiveM what the resource is and which files to load. Open it and paste this:

 ```lua
 fx_version 'cerulean'
 game 'gta5'

 server_script 'server.lua'
 ```

 `cerulean` is the current manifest format version. `server_script` tells FiveM this file runs on the server, not on players' machines.
 </Step>

 <Step num={5} title="Write the lesson code" outcome="The topic is now a runnable probe of your real config.">
 Open `server.lua` and paste this:

 ```lua
 RegisterCommand('netinfo', function()
 local port = GetConvarInt('qu_game_port', 0)
 local maxClients = GetConvarInt('sv_maxClients', 48)
 local oneSync = GetConvar('onesync', 'off')

 print('[qu_networking_basics] game port (TCP+UDP): ' .. port)
 print('[qu_networking_basics] txAdmin panel port: check txAdmin host settings (default 40120)')
 print('[qu_networking_basics] sv_maxClients: ' .. maxClients)
 print('[qu_networking_basics] onesync: ' .. oneSync)
 end, true)
 ```
 </Step>

 <Step num={6} title="Start and test it" outcome="Your real networking numbers appear in the server console.">
 Add this line to `server.cfg`. `ensure` starts a resource and restarts it if it was already running:

 ```text
 ensure qu_networking_basics
 ```

 Save, then run this in the txAdmin Live Console:

 ```text
 restart qu_networking_basics
 ```

 Then run the test command in the same console:

 ```text
 netinfo
 ```

 <ExpectedOutput>
 [qu_networking_basics] game port (TCP+UDP): 30120
 [qu_networking_basics] txAdmin panel port: check txAdmin host settings (default 40120)
 [qu_networking_basics] sv_maxClients: 48
 [qu_networking_basics] onesync: on
 </ExpectedOutput>

 Your `sv_maxClients` and `onesync` values will be whatever your own `server.cfg` sets. The point is that they are now readable, not memorized.
 </Step>
</StepGroup>

## How it works

You printed the four numbers that decide whether players can reach you, how many fit, and how much bandwidth each one costs. Now take the code apart. Each line maps to a real decision instead of a magic value.

### Reading your live config

```lua
local port = GetConvarInt('qu_game_port', 0)
local maxClients = GetConvarInt('sv_maxClients', 48)
local oneSync = GetConvar('onesync', 'off')
```

`GetConvar` and `GetConvarInt` read a convar that was set in `server.cfg`. The first argument is the convar name. The second argument is a fallback default. If the convar was never set, you get that value instead of `nil`. `GetConvarInt` returns a number. `GetConvar` returns a string. That is why `onesync` is read as a string, since its values are the words `off`, `on`, and `legacy`, while the port and player cap are read as integers.

Here is the mental model. A convar is a setting outside your code that a script can read. The bound port still comes from `endpoint_add_udp` and `endpoint_add_tcp`. The `qu_game_port` line is only a label you keep in sync. It does not inspect the socket. It is not proof that a firewall rule works. The real proof is the endpoint lines, the host firewall and provider rules, and an external connection test.

### Why the port needs both TCP and UDP

This is the single most common reason a fresh server is unreachable.

TCP and UDP are separate protocols. Firewalls treat `30120/TCP` and `30120/UDP` as separate rules even though the number matches. Cfx.re's default config binds both. If one is blocked, the connection, file service, discovery, or live session can fail in ways that vary. Open both on the same game port. Then verify from outside the host network.

### Game port 30120 versus txAdmin port 40120

These get confused constantly.

`30120` is the conventional **game port** in the default config. It is for players.

`40120` is txAdmin's **default** web-panel port, and hosts can change it. It is for administrators, not players. Restrict it with the provider firewall, a VPN, or trusted source IPs.

<Checkpoint question="A player says your server shows up in the list and starts connecting, but freezes at the loading screen and times out. The server console shows no errors. What is the most likely cause?">
 First confirm that both `30120/TCP` and `30120/UDP` are allowed at the OS firewall, the provider firewall, and any router or NAT. Then test externally. A listing entry alone does not prove either inbound path is healthy. If both are open, continue with server console errors, proxy configuration, resource downloads, and client-specific issues.
</Checkpoint>

### Telling a server problem from a network problem

When players complain about lag, there are two completely different homes for the cause. They need opposite fixes.

**Server-side lag** is your scripts being slow.

You read it with **resmon**, the in-game resource monitor. It shows each resource's per-frame millisecond cost. If one resource is over budget, the whole server hitches for everyone at once, and no amount of network tuning helps. Fix the script.

Heads up. `resmon` and `netgraph` are developer commands. On a normal FiveM client the **F8 console**, the in-game command line you open with the F8 key, answers `Access denied for command resmon`. Enable developer mode first. Launch FiveM with the extra argument `+set moo 31337`. Add it to your FiveM shortcut, then reopen F8. Only then will `resmon` open. With no dev mode, watch the server thread time in txAdmin instead.

**Network-side lag** is the connection between players and your machine. High ping, packet loss, jitter.

You read it in-game with the **netgraph**. These are client commands. Join the server, press F8 to open the FiveM client console (not the server or txAdmin console), and type `netgraph true` (`netgraph 1` also works) to watch the ping and packet-loss numbers. For an FPS, ping, and packet-loss overlay in the screen corner, type `cl_drawperf true` in the same F8 console.

If the server thread time is healthy in txAdmin but the netgraph shows high ping or dropped packets, the problem is the network, not your code. That is when distance, bandwidth, and attacks come into play.

The diagnostic order is always the same. Check resmon and the server thread first, because that is the cause you can fix yourself. Only when the server side is healthy do you look at the network.

### How OneSync and the player cap drive bandwidth

```lua
local maxClients = GetConvarInt('sv_maxClients', 48)
local oneSync = GetConvar('onesync', 'off')
```

These two are not just settings. They are your bandwidth budget.

`sv_maxClients` is how many players the server allows at once.

**OneSync** is the modern state-synchronization mode. The server, not the clients, owns the authoritative game state. **`onesync on` is the modern default.** The vanilla cap is 32 slots. To go above 32, up to 2048, you enable `onesync on`. Only `onesync legacy` and `onesync off` are legacy modes. Do not treat `onesync on` as old or optional.

The link to bandwidth is direct. Every additional player is more entity and event traffic the server must replicate to everyone else. A higher `sv_maxClients` with OneSync on is a heavier, steadier outbound load on your connection. This matters because a server that is comfortable at 24 players can saturate its uplink at 100, and the symptom looks exactly like network lag even though nobody is attacking you. Knowing your cap and your sync mode tells you how much bandwidth you are signing up for.

Routing buckets, instancing, and other advanced features all require OneSync to be on in the first place. This one setting is the gate to most modern multiplayer behavior.

### What DDoS protection and a proxy actually do

A DDoS attack floods your connection with junk traffic so that real players' packets cannot get through. The defense is not on your server at all. It is upstream, in the network.

A **scrubbing center** or **filtered IP** is a machine that sits in front of your server and absorbs the flood. At the packet level, all traffic for your advertised address arrives at the scrubbing center first. It drops the malicious packets and forwards only the clean UDP gameplay traffic on to your real machine, the **origin**. Players connect to the protected front. The attack hits the front and dies there.

A **proxy** adds the second half. It keeps your origin IP private. If attackers know the raw IP of your actual machine, they can route the flood straight at it and bypass the protection entirely. A proxy means the only address the public ever sees is the protected front. Your origin stays hidden behind it. The combination is "filtered IP plus hidden origin", and it is why a leaked origin IP undoes everything.

This is almost entirely a **hosting feature**. A beginner does not hand-build a scrubbing center. The practical move is to choose a host whose plan includes filtered or anti-DDoS networking, and to never let your raw origin IP leak.

One trap to avoid. The FiveM convar `sv_endpointPrivacy` hides player IP addresses from the server's public reports. It does not hide your server's own origin IP. Do not mistake it for DDoS protection.

## Try it yourself

<Exercise>
 Prove your own port is actually reachable, end to end. First, confirm the server is listening. Restart `qu_networking_basics`, run `netinfo`, and read back your real `30120`, `sv_maxClients`, and `onesync` values from the console. Then test the UDP path specifically, which is the one beginners forget.

 Test from outside your network by connecting a FiveM client to the public address while watching the server console. Generic web port-check sites often test TCP only and cannot reliably prove a stateful UDP game service. Confirm the OS firewall, the provider firewall, and the router or NAT each allow both protocols on your configured game port. Then record which layer was blocking the connection.

 Write down two lines. Which ports your server reports listening on, and the result of the external UDP check (open or closed, and what you changed to fix it).
</Exercise>

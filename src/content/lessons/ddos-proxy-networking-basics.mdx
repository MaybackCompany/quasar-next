<Callout variant="fivem" title="Two ports, two jobs">
 Your game runs on one port number, but two protocols. The default FiveM config binds both TCP and UDP on the same game port (`30120`), and your host firewall has to allow both. Resist the tempting shortcut that "TCP is just the handshake, UDP is just gameplay": FXServer (the FiveM server program) runs several TCP and UDP services on that port, and how it gets listed publicly has changed over the years. One more port to know: txAdmin (the web dashboard you manage the server from) runs on its own port, `40120` by default, and you can change it.
</Callout>

## Build it

We are going to turn your networking settings into numbers you can see. This is a server-only resource that reads your live config and prints it to the console, so the ports, the OneSync mode, and the player cap stop being words in a guide. Follow the names exactly. Once it prints, the next section takes it apart line by line.

<StepGroup>
 <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
 Inside your server's `resources` folder, create this folder:

 ```text
 resources/qu_networking_basics
 ```
 </Step>

 <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
 Create this exact file layout:

 ```text
resources/qu_networking_basics/
 fxmanifest.lua
 server.lua
 ```
 </Step>

 <Step num={3} title="Set the endpoints in server.cfg" outcome="The server binds TCP and UDP on the same port.">
 Open `server.cfg`. Near the top, where your other networking lines live, make sure both endpoints are declared on the same port:

 ```cfg
 endpoint_add_tcp "0.0.0.0:30120"
 endpoint_add_udp "0.0.0.0:30120"
 set qu_game_port 30120
 ```

 Plain version: `0.0.0.0` means "listen on every network connection this machine has," and `30120` is the port players connect to. The extra `qu_game_port` line is just a label we own so the probe below can print the number back to you; keep it equal to the endpoint port. The default config binds both protocols, so allow both through the firewall.
 </Step>

 <Step num={4} title="Write fxmanifest.lua" outcome="FiveM knows which file to load.">
 Open `fxmanifest.lua` and paste this:

 ```lua
 fx_version 'cerulean'
 game 'gta5'

 server_script 'server.lua'
 ```

 No `lua54` line. As of June 2025 that directive is deprecated and ignored, because Lua 5.4 is now the only Lua runtime.
 </Step>

 <Step num={5} title="Write the lesson code" outcome="The topic is now a runnable probe of your real config.">
 Open `server.lua` and paste this:

 ```lua
 RegisterCommand('netinfo', function()
 local port = GetConvarInt('qu_game_port', 0)
 local maxClients = GetConvarInt('sv_maxClients', 48)
 local oneSync = GetConvar('onesync', 'off')

 print('[qu_networking_basics] game port (TCP+UDP): ' .. port)
 print('[qu_networking_basics] txAdmin panel port: check txAdmin host settings (default 40120)')
 print('[qu_networking_basics] sv_maxClients: ' .. maxClients)
 print('[qu_networking_basics] onesync: ' .. oneSync)
 end, true)
 ```
 </Step>

 <Step num={6} title="Start and test it" outcome="Your real networking numbers appear in the server console.">
 Add this line to `server.cfg`:

 ```text
 ensure qu_networking_basics
 ```

 Save, then run:

 ```text
 restart qu_networking_basics
 ```

 Run this test from the txAdmin Live Console:

 ```text
 netinfo
 ```

 <ExpectedOutput>
 [qu_networking_basics] game port (TCP+UDP): 30120
 [qu_networking_basics] txAdmin panel port: check txAdmin host settings (default 40120)
 [qu_networking_basics] sv_maxClients: 48
 [qu_networking_basics] onesync: on
 </ExpectedOutput>

 Your `sv_maxClients` and `onesync` values will be whatever your own `server.cfg` sets. The point is that they are now readable, not memorized.
 </Step>
</StepGroup>

## How it works

You just printed the four numbers that decide whether players can reach you, how many fit, and how much bandwidth each one costs. Now take the code apart so every line maps to a real decision instead of a magic value.

### Reading your live config

```lua
local port = GetConvarInt('qu_game_port', 0)
local maxClients = GetConvarInt('sv_maxClients', 48)
local oneSync = GetConvar('onesync', 'off')
```

These two functions let a script read a setting from `server.cfg`. A convar is just that: a setting that lives outside your code, which a script can look up. The first argument is the convar name. The second is a fallback used if the convar was never set, so you get that value instead of `nil`. `GetConvarInt` hands you a number; `GetConvar` hands you a string. That is why `onesync` (whose values are the words `off`, `on`, and `legacy`) is read as a string, while the port and player cap come back as integers.

One thing to be clear on: this only reports the label. The port the server actually binds still comes from `endpoint_add_udp` and `endpoint_add_tcp`; `qu_game_port` is just a label you keep in sync. Printing it does not prove a firewall rule works. The real proof is the endpoint lines, your host or provider firewall rules, and a connection test from outside your network.

### Why the port needs both TCP and UDP

This is the single most common reason a fresh server is unreachable. The connection happens in two phases on the same port number.

TCP and UDP are separate protocols, so a firewall treats `30120/TCP` and `30120/UDP` as two different rules even though the number is the same. The default config binds both. Block one and things break in confusing ways: connecting, downloading files, getting listed, or staying in the session can each fail depending on which path is cut. So open both on the same game port, then verify from outside the host network.

### Game port 30120 versus txAdmin port 40120

People mix these two up all the time. `30120` is the **game port** in the default config, the one players connect through. `40120` is txAdmin's **default** web-panel port, though hosts can change it. The panel is for you, the admin, not for players. Lock it down: restrict it to a VPN or trusted IPs with your provider firewall wherever your setup allows.

<Checkpoint question="A player says your server shows up in the list and starts connecting, but freezes at the loading screen and times out. The server console shows no errors. What is the most likely cause?">
 First confirm that both `30120/TCP` and `30120/UDP` are allowed at the OS firewall, provider firewall, and any router/NAT, then test externally. A listing entry alone does not prove either inbound path is healthy. If both are open, continue with server console errors, proxy configuration, resource downloads, and client-specific issues.
</Checkpoint>

### Telling a server problem from a network problem

"Lag" has two completely different causes, and they need opposite fixes. Figure out which one you have before you change anything.

**Server-side lag** is your scripts being slow. You read it with **resmon**, which shows each resource's per-frame cost in milliseconds. If one resource blows its budget, the whole server hitches for everyone at the same instant, and no amount of network tuning saves you. You fix the script. Heads up: `resmon` and `netgraph` are *developer* commands. On a normal FiveM client the F8 console just says `Access denied for command resmon`. Turn on developer mode first by launching FiveM with the extra argument `+set moo 31337` (add it to your FiveM shortcut, then reopen F8), and only then does `resmon` open. No dev mode? Watch the server thread time in txAdmin instead.

**Network-side lag** is the connection between players and your machine: high ping, packet loss, jitter. You read it in-game with the **netgraph**. These are CLIENT commands: join the server, press F8 to open the FiveM client console (not the server or txAdmin console), and type `netgraph true` (`netgraph 1` also works) to watch ping and packet loss. Want an FPS/ping/packet-loss overlay in the screen corner too? Type `cl_drawperf true` in the same F8 console. If txAdmin shows a healthy server thread time but the netgraph shows high ping or dropped packets, the problem is the network, not your code. That is when distance, bandwidth, and attacks come into play.

The diagnostic order is always the same: check resmon and the server thread first, because that is the cause you can fix yourself. Only when the server side is healthy do you look at the network.

### How OneSync and the player cap drive bandwidth

```lua
local maxClients = GetConvarInt('sv_maxClients', 48)
local oneSync = GetConvar('onesync', 'off')
```

Think of these two as your bandwidth budget, not just settings. `sv_maxClients` is how many players you allow at once. **OneSync** is the modern sync mode that lets a server blow past the old 32-slot ceiling and makes the server, not the players' games, the source of truth for the game world.

The link to bandwidth is direct: every extra player is more entity and event traffic the server has to copy out to everyone else. You need `onesync on` once you cross the old limits (over 32 slots needs OneSync, and above 64 you must run `onesync on`), and a higher player cap with OneSync on means a heavier, steadier outbound load on your connection. Here is why that bites: a server that feels great at 24 players can max out its upload at 100, and the symptom looks exactly like network lag even though nobody is attacking you. Knowing your cap and your sync mode tells you how much bandwidth you just signed up for.

Routing buckets, instancing, and the other advanced OneSync features all need OneSync turned on first, so this one setting is the gate to most modern multiplayer behavior.

### What DDoS protection and a proxy actually do

A DDoS attack floods your connection with junk traffic so real players' packets can't get through. Here is the key insight: the defense does not live on your server at all. It lives upstream, out in the network.

A **scrubbing center** (also called a **filtered IP**) is a machine that sits in front of your server and soaks up the flood. At the packet level: all traffic for your advertised address hits the scrubbing center first, it throws away the malicious packets, and it forwards only the clean UDP gameplay traffic on to your real machine, the **origin**. Players connect to the protected front, the attack slams into that front, and it dies there.

A **proxy** adds the second half: it keeps your origin IP private. If attackers learn the raw IP of your actual machine, they aim the flood straight at it and skip the protection entirely. A proxy means the only address the public ever sees is the protected front, with your origin hidden behind it. The combo is "filtered IP plus hidden origin," and it is why one leaked origin IP undoes all of it.

This is almost entirely a **hosting feature**. You do not hand-build a scrubbing center as a beginner. The move is to pick a host whose plan includes filtered or anti-DDoS networking, and to never leak your raw origin IP. Two gotchas to keep straight. First, the FiveM convar `sv_endpointPrivacy` hides *player* IP addresses from the server's public reports; it does not hide your server's own origin IP, so do not mistake it for DDoS protection. Second, a strict firewall or DDoS filter can knock you off the server browser: Cfx.re polls your server for player counts and listing data from a fixed set of IPs published at `https://static.cfx.re/ips.txt`, and if your firewall blocks those, your server stops showing up. Allow every IP in that list, and check it now and then since new ones get added.

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| "Connection refused" when joining by IP | The game port (30120) isn't open or isn't forwarded | Forward port 30120 TCP+UDP on your router. Allow it in Windows Firewall and any VPS firewall |
| Server works on localhost but not from the internet | You're connecting to `localhost` but friends need your PUBLIC IP | Use your public IP (find at whatismyip.com) or a Cfx.re join code |
| Everyone lags or times out under load | Your internet upload speed is the bottleneck | Check speedtest.net. If upload is under 10 Mbps, upgrade your plan or use a VPS |
| txAdmin page loads but server browser shows "Offline" | The game port isn't open even though the web port works | txAdmin uses port 40120 (web). The game uses port 30120 (game). Both need to be open |
| "endpoint_add_tcp" not working | Wrong format or placed after resources start | Put endpoint lines at the very top of server.cfg, before any `ensure` lines |

## Try it yourself

<Exercise>
 Prove your own port is actually reachable, end to end. First, confirm the server is listening: restart `qu_networking_basics`, run `netinfo`, and read back your real `30120`, `sv_maxClients`, and `onesync` values from the console. Then test the UDP path specifically, which is the one beginners forget.

 Test from outside your network by connecting a FiveM client to the public address while watching the server console. Generic web port-check sites often test TCP only and cannot reliably prove a stateful UDP game service. Confirm the OS firewall, provider firewall, and router/NAT each allow both protocols on your configured game port, then record which layer was blocking the connection.

 Write down two lines: which ports your server reports listening on, and the result of the external UDP check (open or closed, and what you changed to fix it).
</Exercise>

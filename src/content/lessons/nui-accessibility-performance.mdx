<Callout variant="fivem" title="The focus trap that ruins UIs">
  `SetNuiFocus` controls whether the cursor and keyboard go to your UI or the game. Set focus on when the UI opens and OFF the instant it closes. A UI that grabs focus and never releases it leaves the player frozen, unable to move, the single most common NUI complaint.
</Callout>

## Make it good, not just visible

<StepGroup>
  <Step num={1} title="Release focus on every close path" outcome="The player is never stuck with a hidden UI holding focus.">
    Call `SetNuiFocus(false, false)` on every way the UI can close: button, escape key, and any server-driven close. Test closing it five different ways.
  </Step>

  <Step num={2} title="Scale with viewport units, not fixed pixels" outcome="The UI looks right at 1080p, 1440p, and ultrawide.">
    NUI renders at the player's full game resolution, so `vw` and `vh` map straight to their screen. Size with relative units (`vw`, `vh`, `rem`, `%`) and `clamp()`, not hardcoded pixel widths. A panel fixed at 600px is small on a 1440p screen and oversized on 1080p.
  </Step>

  <Step num={3} title="Stop rendering when hidden" outcome="The UI costs nothing when closed.">
    Hide the page (`display: none`) when closed and stop any animation loops or timers. A NUI that keeps a `requestAnimationFrame` loop running while hidden burns frames for no reason.
  </Step>

  <Step num={4} title="Make it readable" outcome="Players can actually use it under pressure.">
    Sufficient contrast, legible font sizes, clear focus states on buttons, and not relying on color alone for meaning. People read your UI mid-firefight, not in a calm museum.
  </Step>
</StepGroup>

## How it works

The four steps above are advice until you see them in code. Here are the three patterns that carry the weight: one close path, one idle-render fix, and one scaling rule. Wire these into the resource you already have.

<Callout variant="info" title="Where these files live and how to apply changes">
  Open your resource folder (inside your server's `resources/` directory) in VS Code. The browser files sit in the resource's `html/` folder (for example `html/script.js` and the CSS file the page loads); `client.lua` sits in the resource root. After you save an edit, switch to your FXServer console (or the txAdmin Live Console) and run `restart <your-resource-name>` so the new code loads. Then rejoin or re-open the UI to test.
</Callout>

### One close path, called from everywhere

The frozen-player bug happens because focus is released on some close paths but not others. The fix is to never release focus inline. Funnel every close through a single `closeUI()` function, then call that one function from the X button, the Escape key, and any server-driven close. If focus release lives in one place, no path can forget it.

```js
// html/script.js
function closeUI() {
  document.body.style.display = 'none';   // step 3: stop showing it
  fetch(`https://${GetParentResourceName()}/close`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({})
  });
}

// X button click
document.getElementById('close-btn').addEventListener('click', closeUI);

// Escape key
document.addEventListener('keyup', function (e) {
  if (e.key === 'Escape') closeUI();
});

// Server-driven close: Lua sends { type: 'forceClose' }
window.addEventListener('message', function (e) {
  if (e.data.type === 'forceClose') closeUI();
});
```

Use `https://${GetParentResourceName()}/<endpoint>` so renaming the resource does not break callbacks. `GetParentResourceName()` returns the resource hosting the page. On the Lua side, that endpoint releases focus:

```lua
-- client.lua

-- OPEN path (shown for context): grab focus AND show the page.
-- Without this there is nothing to release, so the freeze bug can't even happen.
RegisterCommand('openui', function()
    SetNuiFocus(true, true)               -- keyboard + cursor go to the UI
    SendNUIMessage({ type = 'open' })     -- tell the page to show itself
end)

-- CLOSE path: release focus. cb('ok') MUST be called or the fetch hangs.
RegisterNUICallback('close', function(data, cb)
    SetNuiFocus(false, false)             -- give keyboard + cursor back to the game
    cb('ok')
end)
```

`SetNuiFocus(true, true)` is what routes the keyboard and mouse to your page in the first place; `SetNuiFocus(false, false)` is what gives them back. The freeze bug is releasing the second one on only some close paths.

Now Escape, the X button, and a server `forceClose` all release focus through the same line. Test all three.

<Callout variant="warning" title="Treat every callback payload as hostile">
  The NUI page runs in a browser the player fully controls, so anything it sends to a `RegisterNUICallback` is untrusted client input. A `close` callback carrying no data is harmless, but the moment a callback carries an item id, an amount, or a target player, you MUST re-check it server-side (does the player own the item? is the amount within range? are they allowed?). Never trust a NUI payload to move money, grant permissions, or spawn anything. The browser can send any value it likes.
</Callout>

### Stop the loop when the page is hidden

An animation loop that keeps running while the UI is closed is invisible work. It still schedules a frame, recalculates, and repaints every tick even though nobody can see it. Gate the loop on a visible flag and let it die when you close.

```js
// Before: the loop runs forever, even while display: none
function animate() {
  spinner.style.transform = 'rotate(' + (angle += 2) + 'deg)';
  requestAnimationFrame(animate);   // keeps scheduling, hidden or not
}
animate();

// After: the loop early-returns once the UI is hidden, so it stops
let visible = false;
function animate() {
  if (!visible) return;             // closed: do nothing, do not reschedule
  spinner.style.transform = 'rotate(' + (angle += 2) + 'deg)';
  requestAnimationFrame(animate);
}

// Start the loop only when the UI opens
window.addEventListener('message', function (e) {
  if (e.data.type === 'open') {
    visible = true;
    document.body.style.display = 'block';
    requestAnimationFrame(animate);
  }
});
```

In `closeUI()` you set `visible = false` and hide the page, so the next frame returns immediately and the chain stops. The same idea applies to `setInterval`: keep the timer id and call `clearInterval` on close. The rule is simple. If the player cannot see it, it should not be rendering.

### Why fixed px breaks across resolutions

NUI has no responsive viewport meta tweaks to lean on. It renders at the player's actual screen resolution, so a width set in pixels is a fixed fraction of a 1080p screen and a smaller fraction of a 1440p or ultrawide one. The panel that looked right while you built it on 1080p shrinks into a corner on a wider monitor. Size against the viewport instead, and use `clamp()` to set a floor and ceiling so it never gets unusable at the extremes.

```css
/* Before: locked to one resolution */
.panel { width: 600px; }

/* After: scales with the screen, clamped so it stays sane */
.panel {
  width: clamp(360px, 40vw, 720px);  /* min, preferred, max */
  font-size: clamp(14px, 1.4vh, 20px);
}
```

`40vw` means forty percent of the screen width, so the panel keeps the same visual share at every resolution. The `clamp()` floor stops it shrinking below 360px on small displays, and the ceiling stops it sprawling on ultrawide. Open the resource on every resolution you can reach, including a downscaled window, and confirm the panel holds its proportions.

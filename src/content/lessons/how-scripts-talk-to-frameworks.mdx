## The one idea: your script asks, the framework answers

A plain FiveM resource knows almost nothing about a player. It can read a ped with `PlayerPedId`, but it has no idea that ped is "John, a cop, with $4,200 in the bank." That knowledge lives in the framework. So every framework hands your script a **getter**: you pass it a server id, and it hands back a **player object**, a table that holds that player's money, job, identifier, and a set of functions to change them.

```text
your script              framework                database
-----------              ---------                --------
GetPlayer(source)  --->  look up player    --->   read row
                   <---  player object (table)

Player.addMoney(100) --> framework updates --->   write row
```

The whole skill is three moves, in this order:

1. Get a reference to the framework itself (the "core object" or its exports).
2. Use it to get the **player object** for one `source`.
3. Call functions on that player object to read or change money and job.

Every framework follows that exact shape. Only the names change. That is why, once you can read one, you can read all three.


  </Step>

  <Step num={4} title="Write the detect-and-act code" outcome="The topic is now runnable code that works on whichever framework you run.">
    Open `server.lua` and paste this. It detects the framework once, then a `/fwtest` command gets the player, gives $100, and prints the job.

    ```lua
-- Detect the framework ONCE when the resource starts.
-- GetResourceState returns 'started' only for a resource that is actually running.
local Framework = nil
local FrameworkName = 'none'

if GetResourceState('es_extended') == 'started' then
    Framework = exports['es_extended']:getSharedObject()
    FrameworkName = 'esx'
elseif GetResourceState('qbx_core') == 'started' then
    FrameworkName = 'qbox'
elseif GetResourceState('qb-core') == 'started' then
    Framework = exports['qb-core']:GetCoreObject()
    FrameworkName = 'qbcore'
end

print(('[qu_framework_talk] detected framework: %s'):format(FrameworkName))

RegisterCommand('fwtest', function(source)
    local src = source
    if src == 0 then
        print('[qu_framework_talk] run this in-game, not from the console (it needs a real player)')
        return
    end

    if FrameworkName == 'esx' then
        local xPlayer = Framework.GetPlayerFromId(src)
        if not xPlayer then return end
        xPlayer.addMoney(100)
        print(('[qu_framework_talk] ESX gave $100, job is %s'):format(xPlayer.getJob().name))

    elseif FrameworkName == 'qbcore' then
        local Player = Framework.Functions.GetPlayer(src)
        if not Player then return end
        Player.Functions.AddMoney('cash', 100)
        print(('[qu_framework_talk] QBCore gave $100, job is %s'):format(Player.PlayerData.job.name))

    elseif FrameworkName == 'qbox' then
        local Player = exports.qbx_core:GetPlayer(src)
        if not Player then return end
        Player.Functions.AddMoney('cash', 100)
        print(('[qu_framework_talk] Qbox gave $100, job is %s'):format(Player.PlayerData.job.name))

    else
        print('[qu_framework_talk] no supported framework is running on this server')
    end
end, false)
    ```
  </Step>

## How it works

You ran it and your cash went up. Now take the calls apart so the next time you open a framework's docs you are deciding, not copying. There are only four things happening: get the core, get the player, change money, read job. The rest is the detection wrapper that lets one file serve three frameworks.

### Getting the framework: export style vs event style

The very first thing every framework script needs is a reference to the framework itself. There are two historical ways to get it, and knowing both is what lets you read old and new scripts.

```lua
-- The EXPORT style (modern, preferred everywhere in 2026)
local ESX = exports['es_extended']:getSharedObject()
local QBCore = exports['qb-core']:GetCoreObject()
```

An **export** is a function one resource publishes for other resources to call. `exports['es_extended']` reaches into the running `es_extended` resource and `:getSharedObject()` calls the function it published under that name. You get back the big `ESX` table on the spot. QBCore does the identical thing with `:GetCoreObject()`. This is a direct, synchronous call: you ask, you receive, on the same line.

```lua
-- The EVENT style (older ESX scripts you will still meet in the wild)
ESX = nil
TriggerEvent('esx:getSharedObject', function(obj) ESX = obj end)
```

An **event** is a message. Here your script shouts `esx:getSharedObject` into the resource and hands over a callback function. ESX runs that callback and passes the object back into it, which assigns it to your `ESX` variable. It does the same job as the export, but indirectly, through the event bus. You will see this pattern in tutorials written before 2021. It still works on ESX Legacy, but the export style is cleaner and is what you should write today. Qbox never shipped a shared-object event at all; it is exports only, which is why the Qbox branch in your code calls `exports.qbx_core` directly and never stores a core table.

The mental model: **export is a phone call, event is a letter.** A phone call gives you the answer while you wait. A letter gets answered later, when the other side gets around to reading it. For grabbing the framework object, you want the phone call.

### Getting the player object

Once you hold the framework, you ask it for one player by their `source`, the server id of the connected player.

```lua
local xPlayer = Framework.GetPlayerFromId(src)   -- ESX
local Player  = Framework.Functions.GetPlayer(src) -- QBCore
local Player  = exports.qbx_core:GetPlayer(src)    -- Qbox
```

Look at the shape. All three take the same input, a `source` number, and all three return the same kind of thing, a **player object** table. The only differences are the path you call it through:

- **ESX** exposes it flat on the shared object as `GetPlayerFromId`. The object it returns is called `xPlayer` by convention.
- **QBCore** nests its functions under a `.Functions` sub-table, so it is `GetPlayer`. The object is called `Player`.
- **Qbox** skips the core table entirely and exposes `GetPlayer` straight off the `qbx_core` export.

That `if not Player then return end` line under each call matters. `GetPlayer` returns `nil` when the framework has no loaded player for that source yet, for example a player who is connecting but has not finished loading their character. Indexing a `nil` value is the single most common framework crash, the `attempt to index a nil value` error. Guarding for it is not optional politeness; it is the difference between a script that survives a busy server and one that errors every time someone joins.

### Changing money

With the player object in hand, you call its functions to change state.

```lua
xPlayer.addMoney(100)                  -- ESX
Player.Functions.AddMoney('cash', 100) -- QBCore
Player.Functions.AddMoney('cash', 100) -- Qbox
```

Here the dialects diverge in a way worth noticing. ESX's `addMoney(amount)` takes just a number and moves the player's main money account. QBCore and Qbox take a **money type** first, then the amount, because they track several accounts separately: `'cash'`, `'bank'`, and often `'crypto'`. So `AddMoney('cash', 100)` means add 100 to the cash account specifically. If you passed `'bank'` instead, the cash in their pocket would not move at all. Qbox is a community continuation of QBCore, so it inherits the same `'cash'` and `'bank'` account model and the same `Player.Functions.AddMoney` shape. To take money away you call `xPlayer.removeMoney(100)` on ESX and `Player.Functions.RemoveMoney('cash', 100)` on QBCore and Qbox.

### Reading job

```lua
xPlayer.getJob().name        -- ESX, returns the job name string
Player.PlayerData.job.name   -- QBCore
Player.PlayerData.job.name   -- Qbox
```

ESX hands you the job through a **function**, `getJob()`, which returns a table, and you read `.name` off it. QBCore and Qbox instead expose the job as plain **data** sitting on `Player.PlayerData.job`, so you read `.name` directly without calling anything. Both styles carry the same fields underneath: `name` (like `police`), `label` (the pretty `Police Officer`), and a `grade` table with the rank. The default job a fresh character holds is `unemployed`, which is exactly what your `/fwtest` printed.

### Why real scripts detect the framework first

Look again at the top of the file. Before any command, the script decides once which framework it is running on:

```lua
if GetResourceState('es_extended') == 'started' then
    Framework = exports['es_extended']:getSharedObject()
    FrameworkName = 'esx'
elseif GetResourceState('qbx_core') == 'started' then
    FrameworkName = 'qbox'
elseif GetResourceState('qb-core') == 'started' then
    Framework = exports['qb-core']:GetCoreObject()
    FrameworkName = 'qbcore'
end
```

`GetResourceState('es_extended')` is a FiveM native that returns the string `'started'` only when that resource is actually running on this server. So this block is your script looking around the server and asking "which framework is home?" It sets `FrameworkName` once, and from then on every command just branches on that string. This is exactly how published, multi-framework scripts work: a popular robbery or job script does not ship three separate downloads, it ships one that **detects then branches**, so it drops into an ESX server or a QBCore server unchanged.

Notice the order. Qbox uses `qbx_core` as its core resource, while QBCore uses `qb-core`. Some Qbox servers also keep a compatibility `qb-core` running, so you must check for `qbx_core` **before** `qb-core`, or a Qbox server would be misread as QBCore. Order of detection is not cosmetic; it decides correctness on hybrid servers.

<Checkpoint question="On a Qbox server that also runs a qb-core compatibility shim, why does checking qbx_core before qb-core matter?">
  Because `GetResourceState` would return `'started'` for both `qbx_core` and `qb-core` on that server. If you checked `qb-core` first, the script would set `FrameworkName = 'qbcore'` and then try to call `exports['qb-core']:GetCoreObject()` and read job and money through QBCore's paths, when it should be using Qbox's `exports.qbx_core:GetPlayer`. The two are close cousins but not identical, and the resource that defines the *real* player data is `qbx_core`. Checking the more specific resource first makes the branch land on the framework that actually owns the player, not the shim that happens to be running beside it.
</Checkpoint>

### When you reach for this pattern

Getting the player object is the opening move of nearly every server-side feature you will ever build:

- **Any shop, job, or reward.** Selling fish, paying a mechanic, a daily login bonus: all of them start with `GetPlayer(source)` and end with `AddMoney` or `RemoveMoney`.
- **Any job-gated action.** A police-only `/cuff`, a mechanic-only repair: you read `job.name` and compare it before letting the action run.
- **Any script meant to be shared or sold.** The detect-then-branch wrapper is what makes a script "framework agnostic." It is the difference between code that works on one server and code that works on thousands.

## Try it yourself

<Exercise>
  Turn `/fwtest` into a real job-gated paycheck command, `/paycop`, that only pays players whose job is `police`, and pays them an amount you type. Read the job, compare it, then add the money only if it matches.

  ```lua
  RegisterCommand('paycop', function(source, args)
      local src = source
      local amount = tonumber(args[1])
      if not amount then
          print('[qu_framework_talk] usage: /paycop <amount>')
          return
      end

      -- 1. Get the player object for your framework (reuse the branch from the lesson).
      -- 2. Read their job name into a variable called jobName.
      -- 3. if jobName ~= 'police' then print a 'not a cop' line and return.
      -- 4. Otherwise add `amount` to their cash and print a confirmation.
  end, false)
  ```

  Fill in the four steps using the right calls for your framework. `tonumber(args[1])` turns the typed amount into a number, and the job check is just a string compare against `'police'`. Test it twice: once as an unemployed player, where it should refuse, and once after running `/setjob` (ESX) or `/setjob` in QBCore/Qbox to make yourself police, where it should pay out. You have now written the exact gate every job script in FiveM uses: read job, compare, then act.
</Exercise>

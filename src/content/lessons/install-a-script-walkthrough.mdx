<Callout variant="fivem" title="Read the order, then run it">
 The single skill in this lesson is reading what a script needs before you start it, then installing those needs in dependency order. We will install a fictional but realistic script called `qs_lockpick` that depends on `oxmysql`, `ox_lib`, and a framework. Follow the order exactly. Once it boots clean, the next section explains why each step has to happen when it does, and what breaks if you swap two of them.
</Callout>

## The mental model: install bottom-up

A script almost never stands alone. It calls into other resources, and those resources must already be running and have registered their exports before your script asks for them. So you do not install top-down (the script you want first). You install **bottom-up**: the things at the base of the stack first, the script that sits on top of them last.

Here is the order, drawn as a stack. Read it from the bottom.

```text
 ┌─────────────────────────────┐
 install LAST │ qs_lockpick │ the script you actually want
 │ needs: oxmysql, ox_lib, │
 │ a framework │
 ├─────────────────────────────┤
 │ es_extended / qb-core │ framework (player data, jobs)
 │ needs: oxmysql, ox_lib │
 ├─────────────────────────────┤
 │ ox_lib │ shared library (UI, callbacks)
 ├─────────────────────────────┤
 install FIRST │ oxmysql │ database layer -> MariaDB
 └─────────────────────────────┘
 loads top-down in server.cfg,
 so write deps ABOVE the script
```

In `server.cfg` the load order is literally the top-to-bottom order of your `ensure` lines. So the resource that everyone depends on, `oxmysql`, gets `ensure`d first, and your script gets `ensure`d last. If you flip that, your script boots before `oxmysql` exists and you get `attempt to call a nil value` the instant it starts. That one rule prevents most install failures.

Note the names. `oxmysql`, `ox_lib`, `ox_target`, and `ox_inventory` are **Overextended** community resources, not Cfx.re built-ins. Your database engine is **MariaDB**. QBCore, Qbox (`qbx_core`), and ESX (`es_extended`) have separate official repositories and compatibility requirements. Install the framework and exact dependencies named by the script's current documentation.

## Build it

<StepGroup>
 <Step num={1} title="Read the README and the fxmanifest" outcome="You have a written list of what this script depends on.">
 Before you touch a single folder, open the script's `README.md` and its `fxmanifest.lua`. You are hunting for three things: its dependencies, whether it ships a `.sql` file, and which framework it targets.

 The README usually states it in plain English near the top. The `fxmanifest.lua` states it in machine-readable form with the `dependencies` key:

 ```lua
fx_version 'cerulean'
game 'gta5'

author 'SomeDev'
description 'Lockpick minigame'
version '1.0.0'

shared_script '@ox_lib/init.lua' -- the @ prefix means: load from another resource
client_script 'client.lua'
server_script {
 '@oxmysql/lib/MySQL.lua', -- pulls oxmysql's query API into this script
 'server.lua'
}

-- These resources MUST be started BEFORE this one:
dependencies {
 'oxmysql',
 'ox_lib'
}
 ```

 Write the list down. From the manifest above plus the README, this script needs: `oxmysql`, `ox_lib`, and a framework (say `es_extended`). The `@oxmysql/...` and `@ox_lib/...` paths are a second, louder signal: a script that loads files **from** another resource cannot run unless that resource is already started.

 <Callout variant="fivem" title="What the @ prefix tells you">
 Any `'@resource/path.lua'` entry in a manifest is a hard dependency. The script is borrowing code from that resource at load time. If you see `@oxmysql` or `@ox_lib`, those two are non-negotiable installs, even if the README forgot to mention them.
 </Callout>
 </Step>

 <Step num={2} title="Install the dependencies FIRST" outcome="oxmysql, ox_lib, and your framework are sitting in resources, before the script.">
 Download each dependency from its official source and place each one in your `resources` folder. The community convention is to group them under bracketed category folders. A folder whose name is wrapped in `[brackets]` is treated by FiveM as a grouping container, not a resource.

 Put the shared base in `[framework]` and standalone libraries in `[standalone]`:

 ```text
resources/
 [framework]/
 oxmysql/ # database layer (Overextended)
 ox_lib/ # shared library (Overextended)
 es_extended/ # framework, OR qbx_core / qb-core
 [standalone]/
 # other shared deps go here as you add them
 [scripts]/
 # qs_lockpick will go here in the next step
 ```

 Do not rename these folders. FiveM identifies a resource by its folder name, and your script's `dependencies` list and `@oxmysql/...` paths reference those exact names. Rename `oxmysql` to `oxmysql-v2` and every `@oxmysql` reference breaks.
 </Step>

 <Step num={3} title="Drop the resource into resources/[scripts]" outcome="The script folder exists exactly once, with its fxmanifest at the top.">
 Unzip the script and place its folder inside `[scripts]`. The folder must contain the `fxmanifest.lua` directly, not nested one level too deep.

 ```text
resources/[scripts]/
 qs_lockpick/
 fxmanifest.lua # this must be here, NOT in a subfolder
 client.lua
 server.lua
 lockpick.sql # the schema this script needs in the database
 README.md
 ```

 A common unzip mistake produces `qs_lockpick/qs_lockpick/fxmanifest.lua`, a folder inside a folder. FiveM will not find the manifest and the resource silently does nothing. If you ever see a resource that refuses to appear, check that the `fxmanifest.lua` is at the top level of its folder.
 </Step>

 <Step num={4} title="Import its .sql into MariaDB" outcome="The tables the script queries now exist in your database.">
 This script ships `lockpick.sql`, which means it expects tables in your database. If you skip this, the script will boot fine and then crash the first time it runs a query, with a `no such table` error. Import the schema now, before the first run.

 Open HeidiSQL, connect to your MariaDB, and select your FiveM database in the left tree so the import lands in the right place. Then use **File -> Run SQL file**, pick `lockpick.sql`, and set the encoding to **UTF-8** (not Auto detect, or framework characters corrupt).


 </Mockup>

 After it runs, confirm the table is really there. Open a Query tab and run:

 ```sql
USE fivem;
SHOW TABLES LIKE 'lockpick%';
 ```

 <MockTable
 head={["Tables_in_fivem (lockpick%)"]}
 rows={[["lockpick_attempts"]]}
 caption="HeidiSQL result grid: the table the script needs now exists."
 />

 Seeing the table name in the grid is your proof the import worked. No row here means the import ran against the wrong database, or the file never ran.
 </Step>

 <Step num={5} title="Ensure it AFTER its dependencies in server.cfg" outcome="The load order in server.cfg matches the dependency stack.">
 Now wire the load order. Open `server.cfg` and add the `ensure` lines so every dependency starts **above** the script that needs it. Top to bottom is load order.

 ```cfg
# --- Database + libraries (the base of the stack) ---
ensure oxmysql # must start before anything that queries the DB
ensure ox_lib # shared library used by the framework and scripts

# --- Framework (depends on oxmysql + ox_lib, so it comes after them) ---
ensure es_extended

# --- Your script (depends on all of the above, so it comes LAST) ---
ensure qs_lockpick
 ```

 Your database connection string must also be set, or `oxmysql` cannot connect. Confirm this line exists earlier in the file, pointing at the MariaDB database you imported into:

 ```cfg
set mysql_connection_string "mysql://root:yourpassword@localhost:3306/fivem"
 ```

 Treat this line as a live credential. Replace `root`/`yourpassword` with your real MariaDB user and password (on a public server, create a dedicated database user instead of using `root`), and never paste your `server.cfg` into Discord, a forum, or a screenshot — anyone who reads this line can connect to your database.

 Prefer `ensure` over `start`. `ensure` starts the resource if it is stopped and restarts it if it is already running, so it is the safe, idempotent directive. `start` only starts a resource that is currently stopped, so running it again on an already-started resource does nothing useful (and won't pick up your changes) — `ensure` is the one to reach for every time.
 </Step>

 <Step num={6} title="Restart and read the console" outcome="The script boots clean, and you can prove it in the console.">
 You added a brand-new resource folder while the server may be live, so FiveM has not scanned it yet. In the txAdmin **Live Console** command box at the bottom of the page (or the FXServer console window itself), type `refresh` first so the manifest is detected, then start the script — type each line and press Enter:

 ```text
 refresh
 ensure qs_lockpick
 ```

 Now read the txAdmin Live Console. A clean install prints the script's own start banner and no red error lines:

 <ExpectedOutput>
 Started resource qs_lockpick
 </ExpectedOutput>

 Reading the console is not optional. It is the script telling you, in its own words, whether it found its database, loaded its config, and started. If those lines appear and no red traceback follows, the install is done.
 </Step>
</StepGroup>

## How it works

You installed it in a specific order and it booted clean. Now take the order apart, because the *why* behind each step is what lets you install any script, not this one.

### Why dependencies go in first

A resource registers its exports and events only once it has started. `oxmysql` registers its query API when it boots; `ox_lib` registers `lib.*` when it boots; your framework registers the player object when it boots. Your script calls all of those the moment it starts. If the thing it calls has not started yet, the call resolves to `nil`, and Lua throws `attempt to call a nil value`.

So the rule is not a style preference. It is cause and effect. The dependency must be **running** before the dependent asks it for anything. Installing bottom-up, and ordering `server.cfg` top-down with deps above the script, is how you guarantee that.

### Why the .sql import is its own step

A script's code and a script's database are two separate things you must both provide. The folder you dropped in is the code. The `.sql` file is the shape of the data the code expects: tables, columns, keys. FiveM starts the code regardless of whether the tables exist, because it has no idea what your script will query. The mismatch only surfaces at runtime, the first time the code runs a `SELECT` or `INSERT` against a table that was never created.

That is why `no such table` is a database error, not a code error. The connection worked. The script ran. The table was never imported. Importing the `.sql` before the first run closes that gap. If a script ships no `.sql` file, do not assume it stores nothing: it may reuse framework tables, create its schema in code, use KVP storage, or call an external service. Read its official installation notes to learn what persistence it uses.

### Why load order is the order of the lines

There is no separate "priority" setting in `server.cfg`. The order resources start in is exactly the order their `ensure` lines appear, read top to bottom. Bracketed `[category]` folders like `[framework]` and `[scripts]` only *group* resources for tidiness; they do not control fine-grained order. So two scripts where one depends on the other must still be ordered by hand in `server.cfg`, dependency first.

<Checkpoint question="Your script errors with 'attempt to call a nil value' on the oxmysql export the instant it starts, even though oxmysql is installed and working. What is almost certainly wrong?">
 The load order in `server.cfg` is backwards: your script is `ensure`d **above** `ensure oxmysql`, so it starts before oxmysql has registered its exports. At that moment `exports.oxmysql` is still `nil`, the call fails, and you get `attempt to call a nil value`. The fix is not in the code. Move the `ensure oxmysql` line above the `ensure qs_lockpick` line, restart, and the export exists by the time your script asks for it. This is why the install order and the `server.cfg` order have to match the dependency stack.
</Checkpoint>

### The repeatable checklist

Strip away the specific names and you have a checklist that works for every script you will ever install:

1. **Read.** Open the README and the `fxmanifest.lua`. List the `dependencies`, note every `@resource/...` path, find any `.sql` file, and identify the target framework.
2. **Dependencies first.** Install `oxmysql`, `ox_lib`, the framework, and anything else on the list, into `resources`, with their real folder names.
3. **Drop the resource.** Put the script in `resources/[scripts]`, with `fxmanifest.lua` at the top level of its folder.
4. **Import the SQL.** If it ships a `.sql`, run it into your MariaDB database with HeidiSQL, then confirm the tables exist.
5. **Order it.** Add `ensure` lines in `server.cfg` with every dependency above the script.
6. **Restart and read.** Run `refresh` then `ensure <script>`, and read the console for its start banner and for red.

Six steps, same order, every time. When step six shows red instead of a clean banner, that is your cue to switch from installing to diagnosing.

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| Script folder has `__resource.lua` instead of `fxmanifest.lua` | The script was made before 2019 and uses the old manifest format | It still works. Treat `__resource.lua` the same as `fxmanifest.lua` |
| Script depends on a framework you don't have | The script needs ESX/QBCore/Qbox to function | Check the README. Some scripts have a "standalone" config option. If not, you need that framework |
| Script starts but says "nil value" for framework functions | Framework not loaded before this script | Move `ensure your_script` BELOW `ensure your_framework` in server.cfg |
| SQL file import fails with syntax error | The .sql file has conflicting table names or old syntax | Open the .sql in a text editor. Check table names don't conflict with existing tables. Run each statement separately |

## Try it yourself

<Exercise>
 Take a real free script you have never installed (pick one from the Cfx.re releases forum that lists `ox_lib` as a dependency) and install it cold using only the six-step checklist, writing nothing down except what you read out of its own README and fxmanifest. Before you touch `server.cfg`, answer these three on paper:

 - What does its `dependencies` block list, and does the manifest load any `@resource/...` files that the README did not mention?
 - Does it ship a `.sql` file? If yes, which tables will it create, and which database must you import them into?
 - Given its dependencies, what is the exact top-to-bottom order of `ensure` lines you will add?

 Then install it, restart, and read the console. If you get a clean start banner on the first try, you have internalized the order. If you get red, do not change code: re-check your three answers against the failure, because the mistake will almost always be a missing dependency, a skipped import, or a backwards load order, in that frequency.
</Exercise>

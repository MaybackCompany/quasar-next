  An export lets one resource borrow a function from another and get an answer back. That is the whole idea. Events fire and forget; exports are real function calls, so they run instantly and hand you a return value. This is how every framework wires its pieces together: `exports.ox_inventory:GetItemCount`, `exports.es_extended:getSharedObject`. Here is the deal: you will publish a function in one resource and then call it from a second one, so you actually see the cross-resource call this lesson is named for.

## Build it

<StepGroup>
  <Step num={1} title="Make the two resource folders" outcome="The server has one folder per resource.">
    Inside your server's `resources` folder, create both of these folders:

    ```text
    resources/qu_money
    resources/qu_money_consumer
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in each manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_money/
  fxmanifest.lua
  server.lua
resources/qu_money_consumer/
  fxmanifest.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write qu_money/fxmanifest.lua" outcome="FiveM knows which files to load for the provider.">
    Open `qu_money/fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'
    ```
  </Step>

  <Step num={4} title="Write qu_money/server.lua" outcome="The provider exposes two functions as exports.">
    Open `qu_money/server.lua` and paste this:

    ```lua
local balances = {}

local function getBalance(src)
    balances[src] = balances[src] or 0
    return balances[src]
end

local function addBalance(src, amount)
    balances[src] = getBalance(src) + amount
    return balances[src]
end

exports('GetBalance', getBalance)
exports('AddBalance', addBalance)
    ```
  </Step>

  <Step num={5} title="Write qu_money_consumer/fxmanifest.lua" outcome="The consumer declares it depends on the provider.">
    Open `qu_money_consumer/fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

dependencies {
    'qu_money'
}

server_script 'server.lua'
    ```
  </Step>

  <Step num={6} title="Write qu_money_consumer/server.lua" outcome="The consumer calls the provider's export by name.">
    Open `qu_money_consumer/server.lua` and paste this:

    ```lua
RegisterCommand('checkmoney', function(src)
    exports.qu_money:AddBalance(src, 50)
    local balance = exports.qu_money:GetBalance(src)
    print('[qu_money_consumer] player ' .. src .. ' balance ' .. balance)
end, true)
    ```
  </Step>

  <Step num={7} title="Start and test it" outcome="The cross-resource call prints the expected proof.">
    Open `server.cfg` and add both lines, provider first:

    ```text
    ensure qu_money
    ensure qu_money_consumer
    ```

    Save. If this is the first time you are loading these resources, start them with `ensure` (provider first); `restart` only works on a resource that is already running. Run these in the txAdmin Live Console:

    ```text
    ensure qu_money
    ensure qu_money_consumer
    ```

    Later, after you edit a file, re-load with `restart qu_money` then `restart qu_money_consumer`.

    Run this test from the txAdmin Live Console:

    ```text
    checkmoney
    ```

    <ExpectedOutput>
      [qu_money_consumer] player 0 balance 50
    </ExpectedOutput>

    The `0` is the server console's source id (the console is always player 0). The `50` is the value that came *back* across the resource boundary from `qu_money`. Run `checkmoney` again and the balance climbs to `100`, then `150`. The number persists because both calls read and write the same `balances` table living inside `qu_money`.
  </Step>

## How it works

You just called a function that lives in `qu_money` from a totally separate resource and got a number back. That is an export in one sentence: one resource publishes a function, another calls it like a normal method and gets a value back. Now take the two files apart, one half at a time.

### Registering the export: `exports('name', fn)`

In `qu_money/server.lua` these two lines are what make the functions reachable from outside:

```lua
exports('GetBalance', getBalance)
exports('AddBalance', addBalance)
```

`exports('GetBalance', getBalance)` means: when any resource asks me for `GetBalance`, run my `getBalance` function and hand back whatever it returns. Here `exports` is being *called as a function*. First argument is the public name, a string, that other resources will type. Second argument is the actual Lua function to run. The public name and the local function name do not have to match. That is the trick that lets you keep your real functions `local` and still expose one clean, deliberate name to the outside.

There is an older way to do the same thing, declared in the manifest instead of in code:

```lua
-- older manifest-declared form, shown for recognition only
server_export 'GetBalance'
```

That form requires `GetBalance` to be a *global* function (no `local`), because the manifest can only point at a global by name. The modern `exports('name', fn)` runtime form is preferred: it lets you keep your functions `local`, register them explicitly, and rename the public surface without touching globals. You will see the old `export 'fn'` and `server_export 'fn'` lines in legacy resources, so recognize them, but write the runtime form.

### Why the functions are `local`, and the trap if they are not

Keep your functions `local` so nothing can accidentally clobber them. Notice `getBalance` and `addBalance` use `local function`. The old manifest-export style forced you to make them global instead, and globals all live in one shared namespace inside a resource's Lua VM (the sandbox your scripts run in). So if two of your scripts both define a global `getBalance`, the second one silently overwrites the first, and you get a bug that is miserable to track down. Going `local` plus `exports('name', fn)` means the only thing the outside world can touch is the name you chose to publish. Everything else stays private. That is the safe default.

### Calling the export: `exports.qu_money:GetBalance(src)`

This is the line the whole lesson is named for, from `qu_money_consumer/server.lua`:

```lua
local balance = exports.qu_money:GetBalance(src)
```

Read it left to right. `exports.qu_money` picks the resource you want, by its folder name. The colon then calls its published `GetBalance` function, passing `src`. Do not drop the colon: `exports.qu_money:GetBalance(src)` is the method-call form FiveM expects. If a resource name has a character Lua does not allow in a plain identifier (a dash, say), switch to the bracket form, `exports['qu_money']:GetBalance(src)`. Same thing, just spelled differently.

The key word is **synchronous**, which just means it happens right now, in order, with no waiting. `GetBalance` runs immediately, on the server, in the same tick, and its answer lands in `balance` on that very line. No callback, no event round trip. It is exactly like calling a normal function, except the function happens to live in another resource's folder. Compare that to `TriggerServerEvent`, which throws a message over the wall and returns nothing; to hear an answer you would have to register a whole second event going back.

### Same-side only, and why your dependency line matters

An export only works on the side it was registered on. `qu_money` registered its exports inside a `server_script`, so they are server exports, reachable only from server code. A client script calling `exports.qu_money:GetBalance` gets nothing, because that function simply does not exist in the client's Lua VM. To get the data to a client, the server calls the export and then ships the result down with `TriggerClientEvent`.

The `dependencies` block in the consumer manifest is what guarantees correctness here:

```lua
dependencies {
    'qu_money'
}
```

This tells FiveM to start `qu_money` first, every time. If `qu_money` were not running yet when the consumer called `exports.qu_money:GetBalance`, the call would error, because the export would not exist yet. Use the plural key, `dependencies { '...' }`. You will see the singular `dependency 'x'` around and FiveM tolerates it, but write the plural form. Declaring the dependency is the difference between a resource that starts reliably and one that only works when the load order happens to get lucky.

<Checkpoint question="Why is exports.qu_money:GetBalance(src) able to return a value, while TriggerServerEvent cannot?">
  Because an export is a real, synchronous function call, not a fire-and-forget message. When you write `exports.qu_money:GetBalance(src)`, FiveM runs `qu_money`'s registered function right then, in the same tick, on the same side, and the function's `return` value comes straight back into your variable. `TriggerServerEvent` only delivers a message to a handler and immediately moves on; the handler has no way to hand a value back to the caller, so to get an answer you would need a second event going the other direction. That is the core trade: exports give you a return value but only work same-side; events cross the client/server boundary but give you nothing back.
</Checkpoint>

### When to reach for an export versus an event

Use this decision rule.

- **Reach for an export when you need a return value and the caller is on the same side.** Reading a balance, looking up an item count, asking a framework for a player object. You want the answer in a variable on the next line. That is an export.
- **Reach for an event when you are crossing the client/server boundary, broadcasting to many listeners, or you genuinely do not need a reply.** Telling every client the server is restarting, telling the server a player pressed a key. That is `TriggerClientEvent` or `TriggerServerEvent`.
- **Reach for a server callback (covered in the callbacks lesson) when you need a return value but the caller and the function are on opposite sides.** That is the only case an export cannot serve, because exports are same-side only.

Frameworks live on the first reason: `exports.ox_inventory:GetItemCount(src, 'water')` reads a number and you use it on the next line. Lock in one rule, "exports return values, events do not," and most of FiveM's resource wiring stops being a mystery.

<Callout variant="warning" title="Security note">
  Two things here are security-relevant. The final `true` on `RegisterCommand` makes the command **restricted**: it only runs for a caller that holds the matching ACE permission (the server console always does, which is why the test works from the Live Console). Second, an export like `AddBalance`/`RemoveBalance` mutates a balance and can be called by **any** server-side resource with no check. In a real money system, keep mutation exports private to a trusted set of resources and validate the caller/arguments; do not expose an unguarded "add money" function.
</Callout>

## Try it yourself

<Exercise>
  Extend the provider so a withdrawal cannot overdraw. Add a third export to `qu_money` named `RemoveBalance` that refuses to go negative and reports whether it succeeded:

  ```lua
  local function removeBalance(src, amount)
      if getBalance(src) < amount then
          return false, getBalance(src)
      end
      balances[src] = getBalance(src) - amount
      return true, balances[src]
  end

  exports('RemoveBalance', removeBalance)
  ```

  Then add a `withdraw` command to `qu_money_consumer` that calls it and prints both returned values:

  ```lua
  RegisterCommand('withdraw', function(src)
      local ok, balance = exports.qu_money:RemoveBalance(src, 30)
      print('[qu_money_consumer] ok ' .. tostring(ok) .. ' balance ' .. balance)
  end, true)
  ```

  Restart both resources (provider first), run `checkmoney` once to fund the account to `50`, then run `withdraw`. Confirm it prints `ok true balance 20`. Run `withdraw` again and confirm it prints `ok false balance 20`, because `20` is less than `30` and the export refused. You just proved that an export can return more than one value, and that the *provider* is the right place to enforce a rule, since every consumer gets the check for free.
</Exercise>

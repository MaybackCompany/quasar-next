>
 A **convar** (configuration variable) is a key/value pair you set in `server.cfg` or at runtime, then read from Lua. Convars are how you externalize the things that *change per server*: a debug flag, a database name, a starting balance, an owner ID. You tune them without editing Lua and without rebuilding the resource. The catch that trips up everyone is replication: the prefix you use to declare a convar decides whether a client can ever read it. Get that wrong and your client silently reads the default forever. This lesson makes the difference visible.
</LessonHero>

## Build it

<StepGroup>
 <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
 Inside your server's `resources` folder, create this folder:

 ```text
 resources/qu_convars_server_cfg
 ```
 </Step>

 <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
 Create this exact file layout:

 ```text
resources/qu_convars_server_cfg/
 fxmanifest.lua
 server.lua
 client.lua
 ```
 </Step>

 <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
 Open `fxmanifest.lua` and paste this:

 ```lua
fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'
client_script 'client.lua'
 ```

 Older tutorials add a `-- Lua 5.4 is the default runtime in 2026; the lua54 directive is no longer needed` line here. As of June 2025 that setting is deprecated and ignored: Lua 5.4 is now the only Lua runtime, so you leave it out.
 </Step>

 <Step num={4} title="Declare the convars in server.cfg" outcome="The server has two config values it can read.">
 Open `server.cfg` and add these two lines **above** the line that ensures this resource. You will add the `ensure` line in Step 6.

 `server.cfg` is NOT inside the `resources` folder. It lives in your server's base data directory, the same folder you launch FXServer from, one level above `resources`. (If you set the server up with txAdmin, you can also open it from the txAdmin **Settings** page.)

 ```text
set qu_convars_cash 750
setr qu_convars_debug true
 ```

 The two prefixes are not interchangeable. `set` is server-only. `setr` is replicated, which means it is also sent to every client. You will see in a moment why the client half of this lesson reads the `setr` one and not the `set` one.
 </Step>

 <Step num={5} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
 Open `server.lua` and paste this:

 ```lua
RegisterCommand('cashcfg', function()
 local cash = GetConvarInt('qu_convars_cash', 500)
 print('[qu_convars_server_cfg] server reads cash ' .. cash)
end, false)
 ```

 Open `client.lua` and paste this:

 ```lua
RegisterCommand('debugcfg', function()
 local debug = GetConvar('qu_convars_debug', 'false')
 print('[qu_convars_server_cfg] client reads debug ' .. debug)
end, false)
 ```
 </Step>

 <Step num={6} title="Start and test it" outcome="The expected proof appears in the correct console.">
 Add this line to `server.cfg`, below the two convar lines from Step 4:

 ```text
 ensure qu_convars_server_cfg
 ```

 Save, then restart the whole server so FXServer re-reads `server.cfg`. txAdmin is the web control panel that manages your server; open it in a browser at `http://<your-server-ip>:40120` (use `http://localhost:40120` if you are on the machine running the server) and log in. In the left sidebar open the txAdmin dashboard and click **Restart** (the server restart action, not a single-resource restart). A resource restart alone does NOT re-execute changed `server.cfg` lines, so the new convar values would not load.

 Run the first test in the txAdmin Live Console:

 ```text
 cashcfg
 ```

 Join the server. While you are in-game, press the **F8** key to open the FiveM developer console (a text overlay that drops down over the game). This is where client-side `print` output appears and where you type client commands. With it open, run the second test there:

 ```text
 debugcfg
 ```

 <ExpectedOutput>
 [qu_convars_server_cfg] server reads cash 750
F8: [qu_convars_server_cfg] client reads debug true
 </ExpectedOutput>
 </Step>

## How it works

It worked, so now make it make sense. Two convars, two read functions, two consoles. The reason the lesson is split that way, and not dumped into one file, is the single most important fact about convars: the prefix you declared each one with decided whether the client could ever see it.

### The three prefixes, and who can read each

When you write a convar in `server.cfg` you choose one of three commands. They look almost identical and they behave completely differently.

- `set name value` declares a **server-only** convar. The server can read it. Clients cannot. If a client tries, it gets the fallback default you passed, not the real value. This is the right choice for anything a player should never see: database strings, API keys, owner IDs.
- `setr name value` declares a **replicated** convar. The `r` is for replicated. The server reads it, and the value is also pushed to every connected client, so client Lua can read the real value too. Use this only for config that is safe for players to know and that the client genuinely needs, like a debug flag that turns on extra on-screen logging.
- `sets name value` declares a **server-info convar** exposed in the server's public info, such as `sets tags "roleplay, economy"`. Treat every `sets` value as public and never place a secret in it. Use `set`/`setr` for application configuration so the intended visibility is obvious.

That is the whole rule. `set` stays on the server, `setr` rides along to clients, `sets` is for the browser. In this lesson `qu_convars_cash` is a `set` because the starting balance is a server decision a player has no business reading, and `qu_convars_debug` is a `setr` because the client console needs to know whether debug logging is on.

<Checkpoint question="If you change qu_convars_debug from setr to set and restart, what does the F8 line print, and why?">
 It prints `client reads debug false`. With plain `set`, the convar is server-only and never replicates to clients, so on the client side `GetConvar('qu_convars_debug', 'false')` cannot find a value and returns the fallback you supplied, the string `'false'`. The server console would still read the real value, but the client is permanently blind to a non-replicated convar. This is the number one convar bug: a value that "works on the server" but silently reads the default on every client because it was declared with `set` instead of `setr`.
</Checkpoint>

### The four read functions

You read a convar with one of four natives. They differ only in the type they return, and every one of them takes the same two arguments: the convar name, and a fallback default to use when the convar is missing or unreadable.

- `GetConvar(name, default)` returns a **string**. Everything is a string at heart in `server.cfg`, so this is the most general reader.
- `GetConvarInt(name, default)` returns an **integer**. It parses the stored text into a whole number for you, which is why the server side of this lesson uses it on `qu_convars_cash`: a balance is a number, not text.
- `GetConvarFloat(name, default)` returns a **floating-point number**, for decimals like a price multiplier.
- `GetConvarBool(name, default)` returns a **boolean** `true` or `false`.

The default argument is not optional decoration. It is your safety net. If the convar line is missing from `server.cfg`, misspelled, or (the case you met) declared with the wrong prefix so the client cannot see it, the read returns the default instead of crashing. That is exactly why the F8 line prints `false` when the convar does not reach the client: `'false'` is the default you handed `GetConvar`.

### The restricted flag on the command

Both commands here end with `false`:

```lua
RegisterCommand('cashcfg', function()
 -- ...
end, false)
```

That last argument is the **restricted flag**. With `false`, anyone can run the command, including the server console and any connected player. With `true`, the command requires an ace permission, so only the server console (which holds all aces) and explicitly authorized admins can run it. The original version of this lesson used `true` on the server command, which is why a beginner typing it from anywhere but the console got silence. For a teaching command you want to fire freely while learning, `false` is correct. When you later wrap a real admin action like granting cash, you would switch to `true` and grant the ace, so players cannot run it on themselves.

### Runtime versus server.cfg

One more model to lock in. Editing the `server.cfg` **file** does not execute the changed line on a running server. Perform a full server restart, or deliberately re-run the relevant cfg with `exec`, before testing the new value. Restarting only the resource does not re-read `server.cfg`. The Lua `GetConvar*` calls read the current convar value each time they execute. You can also set a convar live from the server console:

```text
setr qu_convars_debug false
```

A live `set`/`setr` takes effect immediately, no restart needed, and a `setr` change is pushed to connected clients right away. The cfg line is the value the convar starts at on boot.

### When you reach for this pattern

Convars are how you stop hard-coding the things that differ between your test server and a live server. Three real cases:

- **A debug switch.** Wrap your noisy `print` calls in `if GetConvarBool('myresource_debug', false) then`. Flip `setr myresource_debug true` while you investigate, flip it off in production, and you never ship a build to change logging.
- **Per-server tuning.** Starting balance, max stack size, payout rates. One resource, many servers, each with its own numbers in its own `server.cfg`.
- **Secrets that must stay server-side.** A Tebex secret, a Discord webhook, a database connection string. Always a plain `set`, never `setr`. A `setr` convar is copied to every connected client, where the value lives in client memory and can be read or extracted by any player, so declaring a secret with `setr` is a real credential leak, not a cosmetic one. Keep secrets as `set` so they never leave the server.

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| ConVar change doesn't take effect after restart | The resource sets the value in code after startup, overwriting server.cfg | Resources can call `SetConvar()` at runtime. server.cfg sets the initial value, but code wins if it runs after |
| `set` vs `setr` vs `sets` confusion | Each does something different | `set` = server-only value, `setr` = replicated to all clients, `sets` = publicly exposed (server browser and info.json). Never put secrets in `sets` |
| License key in a public GitHub repo | You committed server.cfg with the key in it | Rotate the key immediately at portal.cfx.re. Add server.cfg to .gitignore |
| ConVar value won't change no matter what you do | It's a restricted engine ConVar (like `sv_licenseKey`) | Engine-protected ConVars can only be set in server.cfg at startup. They cannot be changed at runtime |

## Try it yourself

<Exercise>
 Add a third convar and prove the replication rule with your own hands. Put a server-only secret next to the existing lines in `server.cfg`:

 ```text
 set qu_convars_secret supersecret
 ```

 Now try to read it from **both** sides. Add to `server.lua`:

 ```lua
 RegisterCommand('secretcfg_server', function()
 print('[qu_convars_server_cfg] server secret ' .. GetConvar('qu_convars_secret', 'hidden'))
 end, false)
 ```

 And add the matching client command to `client.lua`:

 ```lua
 RegisterCommand('secretcfg_client', function()
 print('[qu_convars_server_cfg] client secret ' .. GetConvar('qu_convars_secret', 'hidden'))
 end, false)
 ```

 Fully restart the server so it re-reads `server.cfg`. Run `secretcfg_server` in txAdmin and `secretcfg_client` in F8. The server prints `server secret supersecret`. The client prints `client secret hidden`, because a plain `set` convar never crosses to the client and the read falls back to your default. You have demonstrated, in your own console, why every secret stays a `set` and never a `setr`.
</Exercise>

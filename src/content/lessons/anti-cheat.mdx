    Now open `server.cfg` and add this line near your other config values:

  <Step num={4} title="Write client.lua" outcome="The client can send a reward intent to the server.">
    Open `client.lua` and paste this. Notice the client sends no amount. It only sends the intent to claim:

    ```lua
RegisterCommand('claimreward', function()
    TriggerServerEvent('qu_anticheat:claimDailyReward')
end, false)

RegisterNetEvent('qu_anticheat:rewardResult', function(ok, message)
    print(('[qu_anticheat] %s: %s'):format(ok and 'approved' or 'denied', message))
end)
    ```
  </Step>

  <Step num={5} title="Write server.lua" outcome="The server owns every decision: amount, rate, and logging.">
    Open `server.lua` and paste this:

    ```lua
local DAILY_REWARD = 250
local COOLDOWN_SECONDS = 10

local lastClaim = {}
local webhook = GetConvar('qu_ac_webhook', '')

local function logViolation(src, reason)
    local name = GetPlayerName(src) or 'unknown'
    print(('[qu_anticheat] blocked %s (id %d): %s'):format(name, src, reason))

    if webhook ~= '' then
        PerformHttpRequest(webhook, function() end, 'POST', json.encode({
            username = 'qu_anticheat',
            content = ('Blocked %s (id %d): %s'):format(name, src, reason)
        }), { ['Content-Type'] = 'application/json' })
    end
end

RegisterNetEvent('qu_anticheat:claimDailyReward', function()
    local src = source
    local now = os.time()

    if lastClaim[src] and now - lastClaim[src] < COOLDOWN_SECONDS then
        logViolation(src, 'reward spam')
        TriggerClientEvent('qu_anticheat:rewardResult', src, false, 'on cooldown')
        return
    end

    lastClaim[src] = now
    local amount = DAILY_REWARD
    print(('[qu_anticheat] reward approved for %s (id %d): %d (demo only)'):format(GetPlayerName(src), src, amount))
    TriggerClientEvent('qu_anticheat:rewardResult', src, true, ('approved %d (demo only)'):format(amount))
end)

AddEventHandler('playerDropped', function()
    lastClaim[source] = nil
end)
    ```
  </Step>

## How it works

You proved that this event's amount and short cooldown decision live on the server. You did not build a complete daily reward: no money is granted, the state resets on resource restart, and disconnecting clears the source-keyed cooldown.

### Why `source` is trustworthy but the payload is not

Look at the first two lines of the event handler:

```lua
RegisterNetEvent('qu_anticheat:claimDailyReward', function()
    local src = source
```

`RegisterNetEvent` registers a networked event. FiveM sets the server-side `source` to the connection that sent it; the payload does not choose another sender slot. That makes `source` useful for identifying this request, but it is a temporary, reusable server slot, not a durable person or character id.

The payload is the opposite. Anything the client passes as an argument is just data the client chose to send, and a cheater rewrites their own client freely. That is why this resource sends **no amount at all**. The old, broken version of this lesson had the client send `amount` and the server merely range-checked it. That is not authority, that is a suggestion. Here the client sends only the intent `qu_anticheat:claimDailyReward`, and the server reads `DAILY_REWARD` from its own constant. There is no amount field for a cheater to inflate, because the amount never left the server.

Lock this in as a rule: use `source` to identify the sending connection, resolve durable identity through the framework when needed, and never trust the payload. The client may say "I want to claim my daily reward." The server must decide whether that connection's active character is entitled and what the reward is.

<Checkpoint question="A teammate sends amount from the client and clamps it to 1..500 on the server. Why is that still cheatable, and what is the fix?">
  Because the client controls the payload, the cheater simply sends the maximum the clamp allows, every single time, as fast as the rate limit permits. The range check only bounds the damage per call. It does not establish who decides the value. The fix is the one in this lesson: the client sends an intent with no amount, and the server derives the amount itself from server-side state (a constant here, a database row in production). When there is no amount field on the wire, there is nothing to forge.
</Checkpoint>

### The rate limit, line by line

```lua
local lastClaim = {}
local COOLDOWN_SECONDS = 10
```

`lastClaim` is a table that maps a player's server ID to the last time they successfully claimed. It starts empty. Inside the handler:

```lua
local now = os.time()

if lastClaim[src] and now - lastClaim[src] < COOLDOWN_SECONDS then
    logViolation(src, 'reward spam')
    TriggerClientEvent('qu_anticheat:rewardResult', src, false, 'on cooldown')
    return
end
```

`lastClaim[src]` is `nil` on a player's first claim, so the `and` short-circuits and the guard is skipped. On a repeat claim, `now - lastClaim[src]` is the number of seconds since the last success. If that is under ten, the call is spam: log it, tell the client no, and `return` before any reward logic runs. Only when the guard passes do we record the new timestamp with `lastClaim[src] = now`.

A note on the clock. `os.time()` returns whole seconds, so its resolution is one second. For a ten-second cooldown that is fine. If you needed a sub-second window, for example "no more than one shot fired per 200 milliseconds," `os.time()` is too coarse and you would reach for `GetGameTimer()`, which returns milliseconds since the server started. Pick the clock that matches the window you are guarding.

### Clearing state on disconnect

```lua
AddEventHandler('playerDropped', function()
    lastClaim[source] = nil
end)
```

This is the line the old lesson was missing, and leaving it out is a real bug. Server IDs are **reused**. When a player disconnects, their ID goes back into the pool and the next player who connects can be handed it. If you never clear `lastClaim`, two things go wrong. First the table grows forever, because every player who ever connected leaves a stale entry behind, a slow memory leak on a long-running server. Second, a reconnecting player can inherit a cooldown they never earned, or skip one they should have, because the slot still holds the previous occupant's timestamp. `playerDropped` fires on the server the moment a player disconnects, and it is the correct hook to release per-player state. Whenever you key a table by `source`, clear that key on `playerDropped`.

### Why the webhook must never be replicated

```lua
local webhook = GetConvar('qu_ac_webhook', '')
```

This reads the convar you set with `set qu_ac_webhook ...`. The second argument is the fallback when the convar is unset, so an empty webhook simply skips the Discord post instead of crashing. The security point is the word `set`. A plain `set` convar lives only on the server and is never sent to clients. If you had written `setr` (the replicated variant), FiveM would push that value to every connected client, and any player could read your webhook URL straight out of their own game and spam your Discord channel with garbage. An audit webhook is a secret. It belongs in a plain `set` convar, or better still an environment variable, and never in a `setr` convar, a `client_script`, or a `shared_script`. Treat the URL like a password: if it ever leaks (committed to a public repo, pasted in chat), delete the webhook in Discord and create a new one -- changing the convar alone does not revoke a leaked URL.

### The audit log

```lua
local function logViolation(src, reason)
    local name = GetPlayerName(src) or 'unknown'
    print(('[qu_anticheat] blocked %s (id %d): %s'):format(name, src, reason))

    if webhook ~= '' then
        PerformHttpRequest(webhook, function() end, 'POST', json.encode(...))
    end
end
```

Every rejection records who and why. The `print` gives you a local trail in the server console. `PerformHttpRequest` posts the same line to Discord so you see attacks in near real time without watching the console. The empty callback `function() end` is deliberate: you are firing and forgetting, you do not need Discord's response. `GetPlayerName(src) or 'unknown'` guards the rare case where the player has already dropped by the time you log, so the format string never breaks on a `nil` name.

### When you reach for these four patterns

You combine these checks on every event where a player gains something. In production the handler verifies stored entitlement, persists a last-claimed timestamp against a stable framework id, and grants the reward through the server framework API only after the durable write succeeds. A source-keyed memory table is a burst limiter, not a once-per-day guarantee.

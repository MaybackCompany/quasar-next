## How it works

One command, two printed lines. That is the whole lesson in one observation. You fired a single event and two separate handlers reacted to it. Now take the code apart so you understand the rule behind it, because this exact mechanism is how every resource on your server reacts to things without ever touching each other's variables.

### An event is a named message

Look at the string `'qu_trigger_event:shout'`. That is an event name. It is not a function, not a variable, not a file. It is just a label that two sides agree on. One side fires the label with `TriggerEvent`, and any side that called `AddEventHandler` with the matching label runs its code. The colon in the middle is not special to FiveM. It is a naming convention: `resourceName:action`. Namespacing your event names this way stops a clash the day another resource also wants an event called `shout`.

The reason events exist at all is the rule from the intro: resources do not share variables. Your `qu_trigger_event` resource cannot reach into another resource and read its tables, and it should not be able to. Each resource is its own sandbox. Events are the seam between those sandboxes. One resource announces that something happened, and any resource that cares has registered a handler to react. That decoupling is the point. The firing side does not know or care who is listening, and the listening side does not know or care who fired.

### Two handlers, one event, both run

Here are the two listeners:

```lua
AddEventHandler('qu_trigger_event:shout', function(message)
    print('[qu_trigger_event] handler one ' .. message)
end)

AddEventHandler('qu_trigger_event:shout', function(message)
    print('[qu_trigger_event] handler two ' .. message)
end)
```

`AddEventHandler` says: when the event named in the first argument fires, run this function. Both calls use the same name, so both handlers run. Do not build logic that depends on listener order: the official event documentation promises matching handlers, not a stable ordering contract between independent resources.

This is the part the old version of this lesson never said out loud, and it is the entire reason the demo prints twice. One event can have many listeners. That is not a bug to design around, it is the feature. A `playerDied` event in a real server might be heard by the medical resource, the scoreboard resource, and the anticheat resource all at once, none of them aware of the others.

### Firing the event

```lua
RegisterCommand('shout', function(_, args)
    TriggerEvent('qu_trigger_event:shout', table.concat(args, ' '))
end, false)
```

`TriggerEvent` is the line that fires the message. It takes the event name, then any values you want to pass to the handlers. Here you pass one value, the reconstructed message string, and both handlers receive it as their `message` parameter.

The documented property to keep is that `TriggerEvent` is **same-side only**. This code is a `client_script`, so only client-side handlers for that name hear it; it never crosses the network to the server. Handlers may yield or schedule other work, so do not use undocumented timing or listener-order assumptions as a coordination mechanism.

### Why args is a table, and what table.concat does

The command handler signature is `function(_, args)`. FiveM hands a command handler three values: the source (who ran it), an `args` table, and the raw text. The first underscore is a throwaway name for the source, which we do not use here. The second, `args`, is the interesting one.

FiveM splits whatever you typed after the command on spaces and gives you the pieces as a **table** (a list). Type `/shout hello world` and `args` is not the string `"hello world"`, it is a two-element list: `args[1]` is `"hello"` and `args[2]` is `"world"`. FiveM cannot know whether you meant one phrase or two separate arguments, so it always hands you the split pieces and lets you decide.

That is why `table.concat(args, ' ')` is there. `table.concat` glues the elements of a table back into one string, putting the separator you pass (here a single space) between each piece. So the list `{ "hello", "world" }` becomes the string `"hello world"`. Without it, you would only ever pass `args[1]` and `/shout hello world` would shout just `hello`. Concatenating with a space rebuilds the full sentence the player typed.

### The distinction that protects your server: AddEventHandler vs RegisterNetEvent

This is the most important paragraph in the lesson. There are two ways to register a handler in FiveM, and choosing the wrong one is either a silent failure or a security hole.

- `AddEventHandler` registers a handler for **local, same-side events only**. The event must be fired on the same side, by `TriggerEvent`. That is what you used here.
- `RegisterNetEvent` registers a handler that is allowed to receive an event **from across the network**, fired by `TriggerServerEvent` (client to server) or `TriggerClientEvent` (server to client). That is the next lesson.

Here is why mixing them up bites you. If you register a networked event with plain `AddEventHandler` and a client tries to trigger it across the network, FiveM blocks it and your handler silently never runs. You will swear the code is correct and lose an hour. Conversely, if you use `RegisterNetEvent` for everything out of habit, you have just told FiveM that any connected player is allowed to fire that event with any arguments they invent. A networked handler that trusts its inputs is the single most common way FiveM servers get exploited. A player can fire your `giveMoney` event from a cheat menu with `amount = 999999999` if you registered it as net and never checked the values.

The rule to carry forward: use `AddEventHandler` when the event stays on one side, and reach for `RegisterNetEvent` only when an event genuinely needs to cross the network, and then validate every argument as if a cheater sent it, because one might have.

<Checkpoint question="Why did one /shout print two lines, and what would happen if you changed both AddEventHandler calls to RegisterNetEvent?">
  Two lines printed because both `AddEventHandler` calls registered a handler for the same event name, `qu_trigger_event:shout`. The event being local and same-side does not limit how many handlers can listen, only where it can be fired from.

  Switching both to `RegisterNetEvent` would still print two lines here, because `TriggerEvent` fires locally and `RegisterNetEvent` handlers also catch local triggers. But it would be the wrong choice: you would have flagged a purely local event as network-capable, which means any client could now fire `qu_trigger_event:shout` across the wire with arguments you never sanitized. Nothing in this demo needs that, so `AddEventHandler` is correct and safer.
</Checkpoint>

### When you reach for this pattern

Same-side events are how you keep one resource's parts loosely joined so you can change one without breaking the others. Three real cases:

- **Fan-out within a resource.** One thing happens (a job ends, a round finishes) and several independent systems need to react. Fire one event, let each system register its own handler. You can add a fourth reaction later without touching the firing code.
- **A clean seam for testing.** Wrap the real trigger condition behind an event, then fire that event by hand from a command while building. You exercise the reaction without setting up the real situation.
- **Built-in FiveM events.** The platform fires events like `onResourceStart` and `playerSpawned` on your side for free. You listen for them with `AddEventHandler`, exactly like you listened for your own `shout`. You did not fire those, FiveM did, but the listening half is identical.

## Try it yourself

<Exercise>
  Make the two handlers do different work so you can prove that each handler runs independently. Right now both just print. Change handler two so it reacts to the content of the message instead of echoing it:

  ```lua
  AddEventHandler('qu_trigger_event:shout', function(message)
      print('[qu_trigger_event] handler one ' .. message)
  end)

  AddEventHandler('qu_trigger_event:shout', function(message)
      print('[qu_trigger_event] handler two heard ' .. #message .. ' characters')
  end)
  ```

  Restart `qu_trigger_event`, then run `/shout hello world` in chat or `shout hello world` in F8. Handler one should echo `hello world`, and handler two should report `11 characters`. The `#` operator on this ASCII string returns its byte length. You just proved the two handlers are independent: they receive the same value but do different things with it.
</Exercise>

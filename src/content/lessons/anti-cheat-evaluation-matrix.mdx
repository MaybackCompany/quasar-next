## The layers that actually stop cheaters

The order here is deliberate. Each layer catches a class of attack the one above it cannot, and a product anti-cheat only earns its keep once the first four are in place.

- **Server authority.** The client may request, only the server grants, after a check. This is the floor. A client that asks for money gets recomputed by the server, never trusted.
- **Rate limits.** Cap how often an action can happen so spam and automation are blocked. A legitimate player robs a store every few minutes; a bot tries every frame.
- **Integrity checks.** Detect tampered resources and unexpected events fired by an injected script.
- **Logging and review.** Catch patterns after the fact and ban with evidence, not vibes.
- **Anti-cheat product.** A layer for injected menus and known cheat signatures, sitting on top of all of the above. Never under it.

## How it works

The reason this lesson exists is that the words above stay abstract until you know what each layer is actually fighting. Two threats get blurred together constantly, and the difference decides which layer addresses them.

### Injected menus versus resource tampering

An **injected menu** (often called an external menu or mod menu) is a separate program the cheater runs alongside the game. It hooks into the game client's memory and calls game functions directly, or fires fake network events at your server. Nothing on your server gave it permission. It exists entirely inside the player's machine, which the cheater controls completely.

**Resource tampering** is different. Here the cheater modifies a resource your server actually sent them, or runs an extra client resource that talks to your scripts. The attack rides on top of legitimate resources rather than bypassing them. An integrity layer is aimed at this case: it looks for client state that does not match what the server shipped.

Knowing which is which tells you which layer does the work. Server authority and rate limits defend against both, because they live on the server and judge requests no matter where they came from. Integrity checks and a product anti-cheat are aimed mostly at the injected and tampered client.

### Why client-side detection can be faked

This is the mechanism behind never trust the client. The cheater owns the client process. Anything that runs there, including a detection script that is supposed to report cheating, runs inside a sandbox the cheater controls. They can patch it, feed it false data, or stop it from ever sending its report.

So a check that runs on the client and sends a clean here, nothing wrong message to the server is worthless on its own. A cheater simply makes that message always say clean. The same logic, run on the server against a request the client cannot forge, holds, because the server decides the outcome and the client cannot edit the server. That is the entire reason server authority sits at the bottom of the stack: it is the only place a check cannot be tampered out of existence.

### What an integrity check actually inspects

Integrity check is a buzzword until you know what it reads. A real one inspects concrete signals on the server side, where the client cannot rewrite them:

- **Unexpected event names.** Your resources fire a known set of net events. An injected script firing an event your code never registers, or firing one far more often than your UI ever could, is a flag.
- **Resource hashes and lists.** The set of client resources is known. A client running a resource that was never sent, or whose contents do not match what the server shipped, is tampering.
- **Impossible triggers.** A reward event that should only follow a verified server-side action, arriving without that action having happened, did not come from your code path.

None of these trust a self-report from the client. Each one watches what the client does from a vantage point the cheater cannot reach, which is exactly why it works and why a client-side scanner does not.

## The layers that actually stop cheaters

The order here is deliberate. Each layer catches a class of attack the one above it cannot, and a product anti-cheat only earns its keep once the first four are in place.

- **Server authority.** The client may request, only the server grants, after a check. This is the floor. A client that asks for money gets recomputed by the server, never trusted.
- **Rate limits.** Cap how often an action can happen so spam and automation are blocked. A legitimate player robs a store every few minutes; a bot tries every frame.
- **Server-side invariants.** Reject impossible state transitions, invalid event arguments, out-of-range actions, and requests that do not follow the expected server-owned sequence.
- **Logging and review.** Catch patterns after the fact and ban with evidence, not vibes. In practice this means recording suspicious requests server-side (to your server console, the txAdmin Live Console, or a logging resource) so a ban can point to a logged event rather than a hunch.
- **Anti-cheat product.** A layer for injected menus and known cheat signatures, sitting on top of all of the above. Never under it.

## How it works

The reason this lesson exists is that the words above stay abstract until you know what each layer is actually fighting. Two threats get blurred together constantly, and the difference decides which layer addresses them.

### Injected menus versus resource tampering

An **injected menu** (often called an external menu or mod menu) is a separate program the cheater runs alongside the game. It hooks into the game client's memory and calls game functions directly, or fires fake network events at your server. Nothing on your server gave it permission. It exists entirely inside the player's machine, which the cheater controls completely.

**Resource tampering** is different. Here the cheater modifies a resource your server actually sent them, or runs an extra client resource that talks to your scripts. The attack rides on top of legitimate resources rather than bypassing them. An integrity layer is aimed at this case: it looks for client state that does not match what the server shipped. (As the next section notes, ordinary Lua cannot reliably hash every client file, so treat any product's integrity claims as something to verify, not assume.)

Knowing which is which tells you which layer does the work. Server authority and rate limits defend against both, because they live on the server and judge requests no matter where they came from. Integrity checks and a product anti-cheat are aimed mostly at the injected and tampered client.

### Why client-side detection can be faked

This is the mechanism behind never trust the client. The cheater owns the client process. Anything that runs there, including a detection script that is supposed to report cheating, runs inside a sandbox the cheater controls. They can patch it, feed it false data, or stop it from ever sending its report.

So a check that runs on the client and sends a clean here, nothing wrong message to the server is worthless on its own. A cheater makes that message always say clean. The same logic, run on the server against a request the client cannot forge, holds, because the server decides the outcome and the client cannot edit the server. That is the entire reason server authority sits at the bottom of the stack: it is the only place a check cannot be tampered out of existence.

### What the server can actually inspect

Do not promise that ordinary Lua can reliably enumerate a cheater's private injected code or prove every client file hash. Focus first on behavior and state the server can observe and own:

- **Invalid event arguments.** Reject wrong types, impossible quantities, invalid entity ownership, and coordinates outside the allowed interaction area.
- **Rate and sequence violations.** A purchase arriving before the server opened a transaction, or hundreds of requests inside a legitimate cooldown, is server-observable evidence.
- **Impossible state transitions.** A reward request without the server-recorded prerequisite, or movement/value changes outside allowed bounds, should be rejected and logged.

None of these trust a clean/dirty self-report from the client. They validate the request against server-owned state. A commercial anti-cheat may add platform-specific detections, but evaluate those claims with evidence and do not replace these controls.

## Common failures

| What you see | What it means | How to fix it |
|---|---|---|
| Anti-cheat blocks legitimate players | The anti-cheat is too aggressive or poorly configured | Check the anti-cheat's config. Lower sensitivity on false-positive triggers (e.g., speed hacks on fast cars) |
| Anti-cheat doesn't catch obvious cheaters | It's not configured for your framework or missing basic protections | Even the best anti-cheat needs server-side validation. Never trust the client to report its own cheats |
| Anti-cheat causes server lag | It's doing too many checks per tick or scanning too many entities | Reduce scan frequency, limit entity range, or disable non-critical checks. Performance over paranoia |
| "I bought an anti-cheat but the cheater still got in" | Anti-cheats are detection tools, not prevention. A good one catches most, none catch all | Layer your defenses: ACE permissions + server-side validation + anti-cheat. No single tool is enough |

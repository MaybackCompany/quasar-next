## 1. The core diagnosis

Strong code does not automatically create a strong store. A product can work perfectly and still fail because buyers cannot find it, understand it, trust its compatibility, or see evidence that support exists. Diagnose the store in five areas: trust, traffic, packaging, offer structure, and presentation.

Do not guess which one is weak. Review package-page questions, support tickets, delivery failures, refund reasons, and the statistics available in the Tebex Control Panel.

## 2. Trust is the foundation

Before a buyer pays, the page must answer five questions without a Discord ticket:

1. What does this product improve for my server or players?
2. Does it support my framework, inventory, target system, database, and artifact requirements?
3. What exactly will Tebex deliver?
4. How do I install, update, and get support?
5. What happens if it does not work or I need a refund?

Trust comes from evidence: a working demo, current documentation, precise compatibility, real support terms, visible release history, and honest buyer feedback. Do not manufacture testimonials, exchange products for guaranteed positive reviews, or claim usage numbers you cannot prove.

## 3. You are not selling code

A server owner buys a player or operator outcome: a cleaner dealership, easier staff workflow, stronger roleplay loop, less configuration work, or reliable support. Lead with that result. Put technical details directly below it so an informed buyer can verify compatibility.

Compare these two openings:

```text
Feature-first: Advanced vehicle shop with oxmysql, exports, categories, and callbacks.

Outcome-first: Give players a clear vehicle-buying experience that installs without
replacing your existing garage. Tested on the listed ESX and Qbox versions.
```

The outcome earns attention. The compatibility statement earns trust. You need both.

## 4. Tebex store optimization

<StepGroup>
  <Step num={1} title="Create and prepare the Tebex project" outcome="the project is ready for review before products are promoted.">
    1. Create the Creator account through `creator.tebex.io` and verify the email address.
    2. Complete the identity, business, wallet, and project details requested by the current onboarding flow.
    3. Connect the FiveM game-server integration from the Tebex Control Panel.
    4. Add a support page, contact route, privacy information, and clear package terms.
    5. Submit the project for Tebex review and resolve every requested change.

    Store review and identity checks can change. Follow the current dashboard checklist instead of copying screenshots from an older tutorial.

    ```md
    # Tebex launch record
    Creator email verified: yes / no
    Identity verification: pending / complete
    Wallet configured: yes / no
    FiveM integration connected: yes / no
    Support page URL:
    Policies reviewed on:
    Store review status:
    ```
  </Step>

  <Step num={2} title="Define the customer result" outcome="the package leads with a concrete server outcome instead of a feature dump.">
    Write one sentence using this shape:

    ```text
    Help [server owner] create [player or operator result] without [specific setup pain].
    ```

    Example:

    ```text
    Help ESX and Qbox server owners give players a clear vehicle-shopping flow
    without replacing their current garage or editing protected files.
    ```

    Then write the proof that supports the promise: supported frameworks, measured idle usage, installation time from a clean test, configuration surface, and a short real-server demonstration.
  </Step>

  <Step num={3} title="Create one complete package" outcome="a buyer can judge, install, and support the product from the listing.">
    In **Packages**, create a category and one package. Tebex currently supports one-time, recurring, or combined pricing plus multiple delivery types. Choose the delivery method that actually matches the product.

    Use this listing order:

    ```md
    # Product name and result
    One sentence describing the server or player outcome.

    ## See it working
    Short video and readable screenshots from a real test server.

    ## What is included
    Exact resources, editable files, protected files, configs, and documentation.

    ## Compatibility
    Framework versions, inventory, target, database, artifact or game-build needs,
    OneSync assumptions, and known incompatibilities.

    ## Installation
    Dependencies first, SQL or migration steps, ensure order, configuration, and proof.

    ## Performance and security
    Measured test conditions, server-side validation boundaries, and stored data.

    ## Support and updates
    Support channel, required evidence, response target, update access, and changelog.

    ## Terms
    Delivery, refund route, license scope, and links to the current policies.
    ```

    Tebex package pages support images and YouTube video. Keep the short showcase focused on the result. Put installation and troubleshooting in a separate practical walkthrough so buyers can evaluate both the experience and the setup.
  </Step>

  <Step num={4} title="Test delivery without a real purchase" outcome="the package reaches the correct account and every delivery action is recorded.">
    Use Tebex's documented test path before making the store public:

    1. Keep the store private.
    2. Enable **Test Mode** under the current checkout settings.
    3. Complete the purchase with the test customer.
    4. Confirm the transaction, webhook behavior, FiveM asset or file delivery, Discord action, and customer email as applicable.
    5. Test refund or subscription-expiry behavior only through the supported path for that delivery type.
    6. Disable Test Mode before launch.

    Tebex also supports manual payments, but its documentation notes that manual payments do not test the same webhook and statistics path. Use Test Mode when you need to prove the checkout flow. FiveM assets have special revocation limits for test and manual payments, so follow the current FiveM package-testing instructions exactly.

    <ExpectedOutput>
      The test customer receives the intended deliverable, no unrelated item or role is granted, the support link works, and the package can be installed from the published instructions.
    </ExpectedOutput>
  </Step>
</StepGroup>

Tebex currently documents regional pricing and supported currencies, coupons and sales, gift cards, abandoned-basket recovery, purchase follow-ups, upselling, offers, and store statistics. Availability can depend on the current plan, project type, region, and dashboard configuration.

Use a measurement record for every feature:

```md
## Growth experiment
Feature:
Buyer problem it addresses:
Packages included:
Start date:
End or review date:
Baseline metric:
Success metric:
Result:
Keep / change / disable:
```

- **Regional pricing and currencies:** use the current country-based controls. Check settlement and conversion behavior before changing prices.
- **Abandoned-basket recovery:** use a clear reminder, sensible frequency limits, and an optional discount only when margin allows it.
- **Purchase follow-ups:** send installation, support, changelog, or genuinely related product information. Do not turn support email into spam.
- **Coupons and sales:** set a real reason, start, and end. Do not leave every package permanently discounted.

Do not copy conversion percentages from another store. Your statistics page is the source for your own visits, payments, and average transaction value.

## 5. The value ladder

Do not start with a wall of unrelated packages. Group products around buyer jobs and keep the choice small:

| Offer level | Purpose | Example |
| --- | --- | --- |
| Free proof | Demonstrates documentation, code quality, and support process. | A useful standalone utility with the same install quality as paid work. |
| Focused package | Solves one clear server problem. | Vehicle shop, dispatch add-on, or staff tool. |
| Outcome bundle | Combines products that solve one larger job. | A police operations pack with explicitly listed compatibility. |
| Recurring package | Funds real ongoing value that continues after purchase. | Maintained library, continuing releases, or defined support access. |

Each level must be useful on its own and make the next choice understandable. A free product should demonstrate the same documentation and support quality as paid work, not act as broken bait.

## 6. Bundles: sell solutions

Bundle products that solve one larger job and have been tested together. A police operations bundle might contain dispatch, evidence, and custody tools, but only if their framework, inventory, target, and database requirements can coexist.

Publish the included products, shared dependencies, install order, duplicate-purchase handling, and bundle discount. Do not call a random collection a solution.

## 7. Upsells and downsells

An upsell offers a compatible next step, such as a tested bundle containing the current package. A lower-cost alternative can help a buyer who does not need the full product. Both must be relevant, clearly different, and honest about what is omitted.

Never hide a required dependency as an upsell. If a package cannot function without another paid product, say so before checkout.

## 8. Membership that works

Do not create a subscription only to make billing recur. State what new value continues each billing period, how cancellation works, what access ends, and how existing licenses behave. Ongoing releases, maintained libraries, defined premium support, and genuine early-access programs can support recurring value. A static download with no continuing service usually cannot.

Avoid fake urgency, permanent countdowns, and "best value" labels with no defensible comparison.

## 9. Organic traffic channels

Publish where actual buyers already look: your support Discord, YouTube, the Cfx.re community when its rules allow the post, and the channels you can maintain consistently. Answer useful questions, share release notes, and link to real documentation. Do not paste the same sales message into unrelated communities.

Choose two channels you can maintain for 90 days. Consistency and searchable proof matter more than opening six abandoned profiles.

## 10. Video strategy

Create two kinds of product media:

1. **Short showcase:** 30 to 90 seconds showing the feature in a believable server situation.
2. **Practical walkthrough:** installation, compatibility, configuration, and the expected result on a clean server.

The short showcase earns attention. The practical walkthrough answers compatibility and setup questions. Put both on the package page when the current template supports them.

## 11. Storytelling in everything

Show the product inside a believable server moment. Establish the player's goal, show the friction, demonstrate the product, and finish with the result. A vehicle-shop showcase should show a player arriving, browsing, purchasing, receiving the vehicle, and driving away. A floating menu on an empty test map proves much less.

Keep the story truthful. Do not imply multiplayer behavior, performance, or compatibility that the test did not prove.

## 12. Brand and commercial packaging

Use consistent thumbnails, readable screenshots, one naming system, and the same compatibility format across every listing. Keep the store landing page focused on products and useful categories. Make price, included items, support, and next action easy to find.

The package should feel complete before purchase: listing, demonstration, documentation, changelog, support route, and policy links. Code is one part of the product.

## 13. Language and voiceover workarounds

Language should not block clear product education. Record naturally in the language you speak best, then add reviewed English captions, a translated script, or a properly disclosed dubbed track. Verify technical terms and product claims after translation.

Do not publish an unreviewed generated voiceover that misstates compatibility, prices, policies, or installation steps. Clear captions and on-screen labels are better than inaccurate narration.

## 14. Your 30-day action plan

```md
# 30-day Tebex plan

## Week 1: foundation
[ ] Complete project, identity, wallet, FiveM integration, policy, and review steps
[ ] Create one category and one complete package
[ ] Test checkout and delivery privately

## Week 2: proof
[ ] Publish one short showcase
[ ] Publish one installation walkthrough
[ ] Add compatibility, FAQ, changelog, and support evidence to the package

## Week 3: offer structure
[ ] Add one genuinely related bundle or upsell
[ ] Review regional pricing and supported currencies without changing them blindly
[ ] Configure one measured follow-up or recovery experiment

## Week 4: learn
[ ] Review store statistics and support questions
[ ] Fix the biggest source of buyer confusion
[ ] Publish the result and next release note
```

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_heal_command
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_heal_command/
  fxmanifest.lua
  server.lua
  client.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

client_script 'client.lua'
server_script 'server.lua'
    ```
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `server.lua` and paste this:

    ```lua
local cooldown = {}

RegisterCommand('heal', function(src)
    local now = os.time()
    if cooldown[src] and now - cooldown[src] < 60 then
        TriggerClientEvent('qu_heal_command:deny', src, 60 - (now - cooldown[src]))
        return
    end
    cooldown[src] = now
    TriggerClientEvent('qu_heal_command:heal', src)
    print('[qu_heal_command] approved heal for ' .. GetPlayerName(src))
end, false)
    ```

    Open `client.lua` and paste this:

    ```lua
RegisterNetEvent('qu_heal_command:heal', function()
    local ped = PlayerPedId()
    SetEntityHealth(ped, GetEntityMaxHealth(ped))
    print('[qu_heal_command] healed')
end)

RegisterNetEvent('qu_heal_command:deny', function(seconds)
    print('[qu_heal_command] wait ' .. seconds .. ' seconds')
end)
    ```
  </Step>

  <Step num={5} title="Start and test it" outcome="The expected proof appears in the correct console.">
    Open `server.cfg` and add this line:

    ```text
    ensure qu_heal_command
    ```

    Save. Now start the resource in the live console (txAdmin Live Console or the FXServer console window) by typing:

    ```text
    ensure qu_heal_command
    ```

    Use `ensure`, not `restart`, the first time: the line you just added to `server.cfg` is only read when the whole server boots, so on an already-running server the resource has not started yet. `restart` only works on a resource that is already running, so it would do nothing here. `ensure` starts the resource if it is stopped and restarts it if it is already running, so it always works.

    Join your own server, wait until your character has spawned in, then open the in-game chat box (press the chat key, by default `T`) and type the command, then press Enter:

    ```text
    /heal
    ```

    <ExpectedOutput>
      Server: [qu_heal_command] approved heal for yourName
F8: [qu_heal_command] healed
    </ExpectedOutput>
  </Step>
</StepGroup>

## How it works

You just built a command that crosses the network twice. The chat input lands on the server, the server decides whether this resource's cooldown passed, and the answer travels back to the client to apply health. This protects the command rule. It does not stop a modified client from changing its own health by another route, so a server-approved heal command is not an anti-cheat. Here is what every line does.

### The manifest is just a list of files

```lua
fx_version 'cerulean'
game 'gta5'

client_script 'client.lua'
server_script 'server.lua'
```

`fx_version 'cerulean'` is the manifest format version. `cerulean` is the current one, so always use it. `game 'gta5'` says this resource runs on GTA V. The two `*_script` lines tell FiveM where each file runs. `server.lua` runs once on the server machine, `client.lua` runs on every connected player's PC. That split is the whole reason this command is safe. Notice there is no `lua54 'yes'` line. That directive was removed in June 2025. Lua 5.4 is now the only runtime, so the line does nothing and you should not add it.

### server.lua, the part that says yes

```lua
local cooldown = {}
```

This is a Lua table used as a lookup. The key is a player's server id, the value is the last time that player healed. Because it lives in `server.lua`, it exists once on the server, so no player can see it or tamper with it. A client variable would be reset or faked instantly by anyone running a cheat menu. Keeping the rule on the server is the entire point.

```lua
RegisterCommand('heal', function(src)
```

`RegisterCommand` tells FiveM that the text `/heal` should run this function. The first argument the function receives is `src`, the server id of the player who typed the command. On the server this number is trustworthy. The player cannot lie about who they are, because FiveM fills in `src` itself. That is why the heal logic belongs here and not on the client.

```lua
    local now = os.time()
    if cooldown[src] and now - cooldown[src] < 60 then
        TriggerClientEvent('qu_heal_command:deny', src, 60 - (now - cooldown[src]))
        return
    end
```

`os.time()` returns the current time in whole seconds. The `if` reads as: if this player has healed before, and fewer than 60 seconds have passed since then, refuse. When it refuses, it fires `qu_heal_command:deny` back to that one player and passes the seconds remaining, then `return` stops the function so the heal never happens. The second argument to `TriggerClientEvent`, the `src`, is the target. The event goes only to the player who asked, not to everyone.

```lua
    cooldown[src] = now
    TriggerClientEvent('qu_heal_command:heal', src)
    print('[qu_heal_command] approved heal for ' .. GetPlayerName(src))
end, false)
```

If the code reaches here, the request is approved. It stamps the current time into `cooldown[src]` so the next attempt is measured from now, fires `qu_heal_command:heal` to that one player, and prints a server log so you can see the approval. The `false` at the very end is the `restricted` argument of `RegisterCommand`. `false` means any player may run the command. If you passed `true`, only players with an ace permission could run it, which is how you would build an admin only command later.

### client.lua, the part that does the work

```lua
RegisterNetEvent('qu_heal_command:heal', function()
    local ped = PlayerPedId()
    SetEntityHealth(ped, GetEntityMaxHealth(ped))
    print('[qu_heal_command] healed')
end)
```

`RegisterNetEvent` listens for a networked event, here the approval the server just sent. `PlayerPedId()` returns the entity handle of your character, the body the camera follows. `GetEntityMaxHealth(ped)` reads the health ceiling for that ped, and `SetEntityHealth` writes it back, so the player refills to full no matter what the max is. Health natives like these only exist on the client because the ped is a client side entity. The server cannot call `SetEntityHealth` on your ped directly, which is exactly why the work is split. The server approves, the client acts.

```lua
RegisterNetEvent('qu_heal_command:deny', function(seconds)
    print('[qu_heal_command] wait ' .. seconds .. ' seconds')
end)
```

This second listener handles the refusal. It receives the `seconds` the server calculated and prints them to F8. In a finished resource you would replace this print with a chat message or a notification, but as a proof a print is enough to confirm the cooldown path fired.

### The mental model

Picture two rooms. The server room holds the rulebook and never lets players in. The client room is the player's own PC, where the character actually lives. A command is a request slipped under the server door. The server reads its rulebook, the cooldown table, decides yes or no, and slides an answer back. Only on a yes does the client room refill the health. Whenever a script changes something a player should not be able to control, money, health, items, admin powers, the decision lives on the server and the visible effect lives on the client. Events are the slot under the door.

When would you reach for this pattern again? Any time you write a command that grants something. A `/giveweapon`, a `/repair`, a `/revive`, a shop purchase. They all share this shape: the server validates, then the client or the database applies. Learn it once here and you will reuse it for the rest of your scripting.

## Now revive someone else

Your `/heal` always acts on you, the person who typed it, so it never had to ask "who?". A `/revive` is the next rung up: an admin types `/revive 3` and player 3 stands back up. That one change, acting on someone other than the caller, forces two new jobs the heal command never needed. First, the command now carries an argument (the target id) that the player typed, and anything a player types is untrusted, so the server must read it and check it before acting. Second, because reviving another player is a power, not a self-service convenience, the command must be locked to admins. We will add a `/revive` to the same `qu_heal_command` resource and reuse the exact same spine: the server validates, then the client acts.

<Callout variant="warning" title="A targeted command is a privilege, not a feature">
  `/heal` healing yourself is harmless. `/revive <id>` lets one player change another player's state, which is grief bait if any player can run it. So this command is registered `restricted` (the third `RegisterCommand` argument is `true`), which means only an identity with the right ace permission may run it. The server console always bypasses the restriction, so you can test from the live console even before you grant yourself the ace.
</Callout>

Add this to the bottom of `server.lua`:

```lua
RegisterCommand('revive', function(src, args)
    local targetId = tonumber(args[1])
    if not targetId then
        print('[qu_heal_command] usage: /revive <id>')
        return
    end

    if GetPlayerName(targetId) == nil then
        print('[qu_heal_command] no player with id ' .. targetId)
        return
    end

    TriggerClientEvent('qu_heal_command:revive', targetId)
    print('[qu_heal_command] revived ' .. GetPlayerName(targetId))
end, true)
```

Add this to the bottom of `client.lua`:

```lua
RegisterNetEvent('qu_heal_command:revive', function()
    local ped = PlayerPedId()
    if IsEntityDead(ped) then
        SetEntityHealth(ped, GetEntityMaxHealth(ped))
        ClearPedTasksImmediately(ped)
        print('[qu_heal_command] revived')
    end
end)
```

In the live console (txAdmin Live Console or the FXServer window) type `ensure qu_heal_command` to reload the resource so both new blocks load. Then, from that same server console, type `revive 1` (use your own server id, which `status` will show in the live console). Because the server console bypasses the `restricted` flag, the revive runs without any ace grant.

<ExpectedOutput>
Server: [qu_heal_command] revived yourName
F8: [qu_heal_command] revived
</ExpectedOutput>

If you instead type `/revive 1` in the in-game chat box as a normal player, nothing happens and the server console prints an access-denied line. That is the `restricted` flag doing its job. To let a specific person run it from chat, add an ace in `server.cfg` (read at boot), for example `add_ace identifier.fivem:1234567 command.revive allow`, then `ensure qu_heal_command`.

### Why every new line earns its place

```lua
RegisterCommand('revive', function(src, args)
```

This time the function keeps a second parameter, `args`. FiveM splits whatever the player typed after `/revive` on spaces and hands it over as a Lua table (an array). `/revive 3` gives `args[1] == "3"`. `src` is still the trustworthy server id of the caller, filled in by FiveM, but `args` is raw player text and is trusted by nothing.

```lua
    local targetId = tonumber(args[1])
    if not targetId then
        print('[qu_heal_command] usage: /revive <id>')
        return
    end
```

`args[1]` is always a string, or `nil` if the player typed `/revive` with nothing after it. `tonumber` converts `"3"` to the number `3`, and returns `nil` for anything that is not a number, including the empty case. So this one check catches both "they typed nothing" and "they typed garbage". When `targetId` is `nil` we print the usage hint to the server console and `return`, so the command never proceeds on bad input. This is the validate-at-the-boundary habit: the boundary is the moment untrusted text enters your server code.

```lua
    if GetPlayerName(targetId) == nil then
        print('[qu_heal_command] no player with id ' .. targetId)
        return
    end
```

A number that parses cleanly can still point at nobody, for example `/revive 99` when no player 99 is connected. `GetPlayerName` returns the player's name for a connected server id and `nil` for an id that is not in the game, so this is a reliable presence test you can trust. Do not use ping for this, an existence check on ping is a guess, not a fact. If the id is empty we print why and `return`, so the revive event is only ever sent to a real, connected target.

```lua
    TriggerClientEvent('qu_heal_command:revive', targetId)
    print('[qu_heal_command] revived ' .. GetPlayerName(targetId))
end, true)
```

Here is the whole point of a targeted command. The first heal command always passed `src` as the target, the caller. This one passes `targetId`, so the event lands on the player who was named, not the admin who typed the command. The trailing `true` is the `restricted` argument, the same slot that held `false` in `/heal`. `true` means an ace permission is required, which is why a state-changing command that affects other players must use it. The print goes to the server console, where your admin actions belong as an audit trail.

```lua
RegisterNetEvent('qu_heal_command:revive', function()
    local ped = PlayerPedId()
    if IsEntityDead(ped) then
        SetEntityHealth(ped, GetEntityMaxHealth(ped))
        ClearPedTasksImmediately(ped)
        print('[qu_heal_command] revived')
    end
end)
```

This runs on the target's machine, because health and ped state are client-side, exactly as in the heal command. The new piece is the guard `IsEntityDead(ped)`: a revive only makes sense on a downed player, so we refill health and then `ClearPedTasksImmediately(ped)` to cancel the death animation and ragdoll so the body actually gets up instead of lying there at full health. `print` here goes to the target's F8 console, not the server console, because client scripts print to F8. If you want the admin to see confirmation, the server-side print above already gives them that.

<Checkpoint question="Why is the cooldown table in server.lua instead of client.lua?">
  Because a client-side cooldown can be reset by the player. Keeping it on the server means calls through this command receive a real yes or no. It does not prevent a modified client from healing outside this resource.
</Checkpoint>

  items={[
    { symptom: "approved heal prints in the server console but F8 shows nothing", fix: "You are reading the wrong console for that line, or you are not in game. The healed line only appears in the F8 client console. Join the server and run /heal again." },

<Exercise>
  Right now a denied heal only prints to F8, so the player never sees why nothing happened. Make the refusal visible in chat. In `client.lua`, change the deny listener to send a chat message instead of a print:

  ```lua
  RegisterNetEvent('qu_heal_command:deny', function(seconds)
      TriggerEvent('chat:addMessage', {
          args = { 'Heal', 'Wait ' .. seconds .. ' more seconds.' }
      })
  end)
  ```

  Restart `qu_heal_command`, run `/heal` twice within a minute, and confirm the second attempt shows a chat line counting down instead of healing. Bonus: add a matching `chat:addMessage` to the heal listener so a successful heal confirms itself in chat too.
</Exercise>

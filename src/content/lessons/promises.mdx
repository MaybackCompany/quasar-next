    ```lua
-- Wrap the callback API PerformHttpRequest into a promise.
-- It returns the promise immediately; the promise settles later,
-- when FiveM fires the callback with the HTTP result.
local function httpGet(url)
    local p = promise.new()

    PerformHttpRequest(url, function(status, body, headers)
        if status == 200 then
            p:resolve(body)
        else
            p:reject('HTTP ' .. status .. ' for ' .. url)
        end
    end, 'GET')

    return p
end

RegisterCommand('promisetest', function()
    -- Await needs a scheduler-managed coroutine. An explicit thread makes
    -- that execution context unambiguous for this first example.
    Citizen.CreateThread(function()
        print('[qu_promises] start')

        -- Three async calls, read top to bottom. No nesting.
        local a = Citizen.Await(httpGet('https://httpbin.org/uuid'))
        local b = Citizen.Await(httpGet('https://httpbin.org/uuid'))
        local c = Citizen.Await(httpGet('https://httpbin.org/uuid'))

        print('[qu_promises] got 3 responses, total bytes: ' .. (#a + #b + #c))

        -- Now prove :reject() surfaces as a Lua error you can catch.
        local ok, err = pcall(function()
            return Citizen.Await(httpGet('https://httpbin.org/status/500'))
        end)
        print('[qu_promises] reject path: ok=' .. tostring(ok) .. ' err=' .. tostring(err))

        print('[qu_promises] done')
    end)
end, true)
    ```

## How it works

You ran it and four lines appeared in order, even though every HTTP call took real network time. Between `start` and the byte count, the server did not freeze. That is the whole point of this lesson, and now you will see exactly how three pieces make it happen: `promise.new()`, `:resolve()` / `:reject()`, and `Citizen.Await`.

### What a promise actually is

A promise is a small object that stands in for a value you do not have yet. Think of it as a claim ticket. When you call an async API like `PerformHttpRequest`, the answer is not ready: the request has to leave your server, travel the network, and come back. You cannot return the answer, because it does not exist. So instead you hand back a ticket that says "the answer will land here later." That ticket is the promise.

```lua
local p = promise.new()
```

`promise.new()` creates one fresh, unsettled promise. Right now it holds nothing. It is in the pending state. A promise has exactly three states and it only ever moves once: pending to **resolved** (it got a value), or pending to **rejected** (it got an error). After it settles, it is frozen. You cannot resolve a rejected promise or change a resolved one.

### `:resolve()` and `:reject()` fill the ticket

```lua
PerformHttpRequest(url, function(status, body, headers)
    if status == 200 then
        p:resolve(body)
    else
        p:reject('HTTP ' .. status .. ' for ' .. url)
    end
end, 'GET')
```

`PerformHttpRequest` is the classic callback API. You hand it a function, and FiveM calls that function later, when the response arrives. Inside that callback you make a decision and settle the promise:

- `p:resolve(body)` means the answer arrived and here it is. Whatever value you pass becomes the resolved value of the promise.
- `p:reject(message)` means it failed. The value you pass becomes the error.

This callback runs at some unknown future moment, not when `httpGet` returns. That is the gap a promise bridges. The function `httpGet` returns the pending promise *immediately*, on the same line. The `resolve` or `reject` happens seconds later, inside the callback, when the network replies.

<Checkpoint question="The httpGet function has a return p at the bottom and a p:resolve inside the callback. Which one runs first, and why does that order matter?">
  `return p` runs first, the instant you call `httpGet`. The promise is still pending at that moment because the HTTP request has not come back yet. `p:resolve(body)` runs later, when FiveM fires the callback. This is the heart of the pattern: you can hand back the ticket before the answer exists, and someone fills it in later. If you tried to return `body` directly instead of a promise, you could not, because `body` does not exist yet when `httpGet` returns.
</Checkpoint>

### `Citizen.Await` collects the value

```lua
local a = Citizen.Await(httpGet('https://httpbin.org/uuid'))
```

`Citizen.Await(p)` does one job: it pauses the current thread until the promise `p` settles, then it gives you the result. If the promise resolves, `Await` returns the resolved value, so `a` becomes the response body. If the promise rejects, `Await` raises a Lua error carrying the reject message.

The word that matters is **pauses**, not **blocks**. A blocking call would freeze the entire server: no other resource could run, no player could move, until the network replied. `Await` does not do that. It yields the current coroutine back to the FiveM scheduler. The scheduler is then free to run every other thread on the server while your thread sits parked, waiting for its promise. When the promise settles, the scheduler wakes your thread back up exactly where it left off and hands you the value. That is why the line "feels synchronous while staying non-blocking": you read it top to bottom like normal code, but under the hood your thread stepped aside and let the server keep working.

### Why Await needs a scheduler context

Look at where the awaits sit:

```lua
RegisterCommand('promisetest', function()
    Citizen.CreateThread(function()
        local a = Citizen.Await(httpGet('https://httpbin.org/uuid'))
        -- ...
    end)
end, true)
```

`Citizen.Await` yields the current coroutine, so the code must run in a context owned by the Cfx scheduler. `CreateThread`, `SetTimeout`, and registered event/command callbacks are scheduler contexts in the current Lua runtime. This example creates an explicit thread because it is easy to recognize and keeps the command callback small. The actual runtime assertion is about being outside the scheduler, not about command handlers specifically.

### Flattening the pyramid

Here is the payoff. Without promises, three dependent HTTP calls nest, each result trapped inside the previous callback:

```lua
PerformHttpRequest(url1, function(s1, b1)
    PerformHttpRequest(url2, function(s2, b2)
        PerformHttpRequest(url3, function(s3, b3)
            -- only here can you see b1, b2, and b3 together
        end, 'GET')
    end, 'GET')
end, 'GET')
```

That is the pyramid: indentation marching to the right, error handling duplicated at every level, the actual logic buried three deep. With promises and `Await`, the same three calls read as a straight list:

```lua
local a = Citizen.Await(httpGet(url1))
local b = Citizen.Await(httpGet(url2))
local c = Citizen.Await(httpGet(url3))
-- a, b, and c are all in scope, side by side
```

Each line waits for its call, returns the value into a normal local variable, and moves on. There is no nesting, the variables share one flat scope, and you handle errors in one place instead of three. The shape of the code now matches the shape of the task: do this, then this, then this.

### Handling failure with reject

A rejected promise turns into a raised error when you await it, so you catch it the standard Lua way, with `pcall`:

```lua
local ok, err = pcall(function()
    return Citizen.Await(httpGet('https://httpbin.org/status/500'))
end)
```

`httpbin.org/status/500` always returns HTTP 500, so the callback hits the `else` branch and calls `p:reject(...)`. When you `Await` that rejected promise, it raises. `pcall` runs the function in protected mode: it returns `false` plus the error message instead of crashing your thread. That is why the output shows `ok=false` and the reject message. If you await a promise that can reject and you do not wrap it in `pcall`, the raised error stops the rest of your thread, which is one of the ways a promise-based script "dies silently" mid-task.

### When you reach for this pattern

You wrap a callback API in a promise whenever you have async work whose result you need before the next step can run. The three you will hit constantly in FiveM:

- **HTTP requests.** `PerformHttpRequest`, exactly as built here. Calling an external API, a webhook, a license check. Security: this runs on the server, so never build the URL from player-supplied input without validating it (an attacker who controls the URL can make your server hit internal/private addresses — SSRF). Treat the response body as untrusted: validate or json-decode it before using it, and never concatenate it into a SQL query. Keep webhook URLs and API keys in convars (GetConvar), never hardcoded in the resource.
- **Database queries.** Modern libraries like oxmysql already return promises, so you can `Await` them directly. Older callback-style MySQL wrappers you wrap yourself, the same way you wrapped `PerformHttpRequest`.
- **NUI callbacks and any "do this, get a result later" native.** If it hands you a callback instead of a return value, it is a candidate for promise-wrapping.

The rule of thumb: if two async steps depend on each other, promises plus `Await` will keep them readable. If they are independent, you may not even need to wait between them.

## Try it yourself

<Exercise>
  Right now the three HTTP calls run one after another, each waiting for the last to finish, even though none of them depends on the others. That is wasteful: three sequential one-second calls take three seconds. Refactor so the three requests fire at once and you await all of them together.

  ```lua
  Citizen.CreateThread(function()
      -- Start all three immediately. httpGet returns a pending promise,
      -- so the requests are now in flight in parallel.
      local p1 = httpGet('https://httpbin.org/uuid')
      local p2 = httpGet('https://httpbin.org/uuid')
      local p3 = httpGet('https://httpbin.org/uuid')

      -- Now collect the results.
      local a = Citizen.Await(p1)
      local b = Citizen.Await(p2)
      local c = Citizen.Await(p3)

      print('[qu_promises] parallel bytes: ' .. (#a + #b + #c))
  end)
  ```

  Notice the difference from Step 4: there you called `Citizen.Await(httpGet(...))` on one line, which started a request and immediately waited for it before starting the next. Here you start all three first, then await. The total time drops to roughly the slowest single call instead of the sum of all three. Add a `restricted` second command named `paralleltest` for this version, restart `qu_promises`, run both, and watch which one finishes faster in the console. You have now controlled not just *whether* code waits, but *what it waits on*.
</Exercise>

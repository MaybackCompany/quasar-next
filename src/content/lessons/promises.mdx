>
  FiveM is full of callback APIs: `PerformHttpRequest`, the older `MySQL-Async`, NUI callbacks. Three of them in a row and your code looks like a pyramid. Promises flatten the pyramid. Combined with `Citizen.Await`, you get top-to-bottom code that *feels* synchronous while staying non-blocking.

## Build it

<StepGroup>
  <Step num={1} title="Make the resource folder" outcome="The server has one folder for this lesson.">
    Inside your server's `resources` folder, create this folder:

    ```text
    resources/qu_promises
    ```
  </Step>

  <Step num={2} title="Create the files" outcome="Every file named in the manifest exists.">
    Create this exact file layout:

    ```text
resources/qu_promises/
  fxmanifest.lua
  server.lua
    ```
  </Step>

  <Step num={3} title="Write fxmanifest.lua" outcome="FiveM knows which files to load.">
    Open `fxmanifest.lua` and paste this:

    ```lua
fx_version 'cerulean'
game 'gta5'

server_script 'server.lua'
    ```
  </Step>

  <Step num={4} title="Write the lesson code" outcome="The topic is now represented by runnable code.">
    Open `server.lua` and paste this:

    ```lua
local function afterOneSecond(label)
    local p = promise.new()
    SetTimeout(1000, function() p:resolve(label .. ' done') end)
    return Citizen.Await(p)
end

RegisterCommand('promisetest', function()
    print('[qu_promises] start')
    print('[qu_promises] ' .. afterOneSecond('job'))
end, true)
    ```
  </Step>

  <Step num={5} title="Start and test it" outcome="The expected proof appears in the correct console.">
    Open `server.cfg` and add this line:

    ```text
    ensure qu_promises
    ```

    Save, then run:

    ```text
    restart qu_promises
    ```

    Run this test:

    ```text
    promisetest
    ```

    <ExpectedOutput>
      [qu_promises] start
[qu_promises] job done
    </ExpectedOutput>
  </Step>
</StepGroup>

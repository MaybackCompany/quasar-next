## Script Ladder

Do the five tests in order. Each one adds exactly one new responsibility on top of the previous one, which is the safest way for a total beginner to build confidence without guessing. The last two are intentionally the bridge from website study into real local practice.

<StepGroup>
  <Step num="Test 1" title="/hello" outcome="you can register a command and print a clear success message.">
      From a blank file, rebuild a command called `/hello` that prints `Hello Quasar`.

      - **Happy path:** running `/hello` prints the exact line once.
      - **Guard check:** the command name is correct and does not silently do nothing.
  </Step>

  <Step num="Test 2" title="/wallet" outcome="you can read state, branch on it, and print a human-readable result.">
      Make a command called `/wallet`. It should print one of two messages: `wallet: healthy` if cash is 100 or more, or `wallet: low` if cash is below 100.

      - **Happy path:** with 500 cash, the script prints `wallet: healthy`.
      - **Guard path:** with 50 cash, the script prints `wallet: low` and does not move money.
  </Step>

  <Step num="Test 3" title="/ping" outcome="you can trigger a server event and give the learner a visible success line.">
      Rebuild the Day 3 command from memory. The command should fire `TriggerServerEvent('qu:ping', 'pong')` and print `sent pong to server`.

      - **Happy path:** the event name and argument are exact.
      - **Guard path:** if you accidentally use `TriggerEvent` instead, fix it before you move on.
  </Step>

  <Step num="Test 4" title="/healme" outcome="you can use one client native and one command in the same tiny script.">
      Make a client command called `/healme`. It should get the local ped, set health to full, and print `healed`. This one assumes a real local client file and a running FiveM client; it is not a browser-only exercise anymore.

      - **Happy path:** your health changes and the message prints.
      - **Guard path:** the code lives in `client/main.lua`, not on the server side.
  </Step>

  <Step num="Test 5" title="/deposit" outcome="you can rebuild the guard-first money pattern without copying.">
      Pick one framework first, ESX, QBCore, or Qbox, and rebuild `/deposit`. Only after the single-framework version works should you try the multi-framework version.

      - **Happy path:** 500 cash / 1000 bank becomes 400 cash / 1100 bank.
      - **Guard path:** with less than 100 cash, the function returns early and nothing moves.
      - **Stretch:** add the runtime framework detection from Day 7.
  </Step>
</StepGroup>

<Checkpoint question="The real graduation check">
  You are ready for the main curriculum if all of these are true:

  <Callout variant="info" title="The five green lights">

    - I know that the client asks and the server decides.
    - I can read a small framework API example without freezing.
    - I can explain what my script does in plain English before I run it.

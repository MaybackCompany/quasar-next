  Five queries you reuse in every FiveM resource that touches the database. Always with `?` placeholders. Never with string concatenation.
</LessonHero>

<Callout variant="warn" title="One SQL rule">
  Use `?` placeholders for any value that came from a player. Never build SQL by joining strings. The whole point of parameterized queries is that the database parses the values, not the query string.
</Callout>

## Setup

Add oxmysql as a server script in `fxmanifest.lua` and declare a dependency. Configure your connection string in oxmysql's `convars`.

```
-- fxmanifest.lua
server_scripts {
  '@oxmysql/lib/MySQL.lua',
  'server/*.lua',
}

dependencies {
  'oxmysql',
}

-- server.cfg
set mysql_connection_string "mysql://user:pass@127.0.0.1:3306/fivem"
```

## Insert

```
-- await: returns insertId
local insertId = MySQL.insert.await(
  'INSERT INTO licenses (citizenid, type, issued_at) VALUES (?, ?, NOW())',
  { citizenid, 'driver' }
)

-- callback variant
MySQL.insert(
  'INSERT INTO licenses (citizenid, type) VALUES (?, ?)',
  { citizenid, 'driver' },
  function(insertId)
    print('new license id:', insertId)
  end
)
```

## Select one row

```
local row = MySQL.single.await(
  'SELECT id, money, job FROM users WHERE citizenid = ? LIMIT 1',
  { citizenid }
)

if not row then
  return                                            -- no match
end

print(row.money, row.job)

-- single value
local money = MySQL.scalar.await(
  'SELECT money FROM users WHERE citizenid = ?',
  { citizenid }
)
```

<Callout variant="tip" title="scalar vs single">
  `scalar` returns the first column of the first row. `single` returns the whole row as a table. Use `scalar` when you only need one number or string.
</Callout>

## Select many rows

```
local rows = MySQL.query.await(
  'SELECT plate, model FROM player_vehicles WHERE citizenid = ? ORDER BY id DESC',
  { citizenid }
)

for _, v in ipairs(rows) do
  print(v.plate, v.model)
end

print('count:', #rows)
```

## Update

```
local affected = MySQL.update.await(
  'UPDATE users SET money = money + ? WHERE citizenid = ?',
  { 500, citizenid }
)

if affected == 0 then
  print('no row matched')
end
```

<Callout variant="tip" title="Always check affectedRows">
  If `UPDATE` returns `0`, the WHERE clause matched nothing. Either the row does not exist or you targeted the wrong key.
</Callout>

## Delete

```
local affected = MySQL.update.await(
  'DELETE FROM player_vehicles WHERE plate = ? AND citizenid = ?',
  { plate, citizenid }
)

if affected == 0 then
  return false                                      -- nothing deleted, abort
end
```

<Callout variant="warn" title="Always include the owner">
  Never delete by plate alone. A malicious client could pass a plate they do not own. Always include the citizenid (or whatever identifier ties the row to the player) in the WHERE clause.
</Callout>

## Why parameterized queries

```
-- WRONG: string concatenation
local name = source.name                            -- player-controlled
MySQL.query.await(
  "SELECT * FROM users WHERE name = '" .. name .. "'"
)
-- if name is:  ' OR 1=1 --
-- the query becomes:  SELECT * FROM users WHERE name = '' OR 1=1 --'
-- every row returns

-- RIGHT: ? placeholder
MySQL.query.await(
  'SELECT * FROM users WHERE name = ?',
  { name }
)
-- the database treats name as a value, no matter what is in it
```

<Callout variant="warn" title="One bypass and you are done">
  SQL injection is not a theoretical concern. A single concatenated query lets a player rewrite your database, dump tables, or wipe accounts. There is no excuse to skip the placeholder.
</Callout>

## Useful patterns

```
-- transaction (multiple queries atomically)
MySQL.transaction({
  { query = 'UPDATE users SET money = money - ? WHERE id = ?', values = { 500, fromId } },
  { query = 'UPDATE users SET money = money + ? WHERE id = ?', values = { 500, toId } },
}, function(success)
  if not success then
    print('transfer failed, rolled back')
  end
end)

-- IN clause with a list
local ids = { 1, 5, 9 }
local placeholders = string.rep('?,', #ids):sub(1, -2)
local rows = MySQL.query.await(
  'SELECT * FROM users WHERE id IN (' .. placeholders .. ')',
  ids
)

-- check exists without fetching the row
local exists = MySQL.scalar.await(
  'SELECT 1 FROM users WHERE citizenid = ? LIMIT 1',
  { citizenid }
) ~= nil
```

## Common mistakes

<Callout variant="warn" title="Loading oxmysql on the client">
  `@oxmysql/lib/MySQL.lua` belongs in `server_scripts`. On the client it throws *attempt to index a nil value (global 'MySQL')*.
</Callout>

<Callout variant="warn" title="Mixing .await inside a tight loop">
  One `MySQL.query.await` per iteration serializes every request. For batches, build one query with an `IN` clause or a `VALUES (?,?), (?,?)` insert.
</Callout>

<Callout variant="warn" title="source after await">
  After `MySQL.query.await`, the global `source` may have changed. Snapshot `local src = source` on the first line of every event handler.
</Callout>

<Callout variant="warn" title="Skipping LIMIT 1">
  For single-row lookups, add `LIMIT 1`. The database short-circuits as soon as one row matches, instead of scanning the rest.
</Callout>

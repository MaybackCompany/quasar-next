Owner rule

If you cannot explain what a new player does in the first hour, you are not ready to install scripts. Plan the first hour first, then buy only the systems that support it.

## The sample server concept

Use this worked example as the shape of your own brief. The point is not to copy the theme. The point is to see how a server idea turns into install decisions.

      Do not start with "we need gangs, shops, drugs, apartments, and ten jobs." Start with what a brand-new player sees and does.

      ```
      # First-hour loop
      1. Player joins and spawns at city hall.
      2. Player creates a character and receives an ID.
      3. Player takes a starter route: taxi, delivery, or sanitation.
      4. Player earns first legal money.
      5. Player meets one staff guide or another player at a public hub.
      6. Player buys food, rents a basic vehicle, and learns where to return tomorrow.
      ```

      A staff member can read the six lines and explain where the player goes without asking you in Discord.
  </Step>
</StepGroup>

## Must-have now vs later

This is where owners save money. A must-have system is required for the first-hour loop. A later system can be exciting and still be wrong for v1.

| Decision | Harbor County example | Why |
| --- | --- | --- |
| Must have now | Character creation, identity, spawn, legal job, banking, inventory, vehicle rental, basic police | These support the first hour and data persistence. |
| Must have now | txAdmin roles, test accounts, database backup, rollback notes | These protect the server before public players arrive. |
| Later | Custom housing, full businesses, complex crime, casino, racing, advanced phone apps | They add dependencies and staff load before the core loop is proven. |
| Not for this concept | Military roleplay, zombie survival, heavy gang wars | They fight the grounded civilian promise. |

<StepGroup>
  <Step num={3} title="Make dependency and budget decisions" outcome="every planned paid script has a reason and a risk note.">
      Before purchase, ask three questions: does it support the first hour, what framework does it require, and who will maintain it when it breaks?

      ```
      # Budget and dependency decisions
      Framework: QBCore
      Reason: beginner-friendly ecosystem, broad script support, easier to hire help.

      Database: MariaDB + oxmysql
      Reason: common FiveM persistence path.

      Buy now:
      - Multi-character: required for identity flow.
      - Inventory: required for food, work items, and starter loop.
      - Phone: only if it supports QBCore and does not require a full economy rewrite.

      Wait:
      - Housing: not needed for first public beta.
      - Racing: fun, but not part of the first-hour loop.
      - Advanced crime: increases police, economy, and staff complexity.

      Monthly budget cap:
      - Hosting:
      - Scripts:
      - Developer hours:
      - Emergency reserve:
      ```

      <Callout variant="warn" title="Owner safety rule">
        Never buy a script only because a showcase video looks good. Require framework support, install docs, dependency list, update history, and a rollback plan.
      </Callout>
  </Step>
</StepGroup>

## Completed example

This is the minimum quality bar for your own brief. It is specific enough for a developer to estimate and narrow enough for staff to test.

```
# Harbor County RP brief
Audience: adults who want relaxed civilian roleplay.
Promise: first legal money and first social contact in 60 minutes.
Framework: QBCore.
Hosting path: hosted VPS first, local staging for experiments.

First hour:
1. Spawn at city hall.
2. Create character.
3. Receive ID and starter cash.
4. Choose taxi, delivery, or sanitation.
5. Earn first money.
6. Meet a guide at the diner.
7. Rent a vehicle and learn the return route.

Must-have v1:
- Multi-character
- Inventory
- Banking
- Starter jobs
- Basic police and EMS
- Vehicle rental
- Staff permissions
- Daily database backup

Later:
- Housing
- Player-owned businesses
- Racing
- Advanced crime
- Casino

Dependency risks:
- Phone requires ox_lib and framework bridge.
- Inventory may replace framework default inventory.
- Vehicle keys must match garage and rental scripts.

Pass condition:
A new test player can finish the first-hour loop with no staff teleporting them.
```

## Questions to ask a developer

1. Which framework best supports this first-hour loop: QBCore, ESX, or Qbox?
2. Which systems replace default framework behavior, such as inventory, vehicles, housing, or phone?
3. Which scripts require SQL imports?
4. Which scripts must be installed before others?
5. What can be staged locally before touching production?
6. What is the rollback command or restore step if the script fails?

## Failure recovery

<Callout variant="info" title="The concept is still too broad">
  <Callout variant="danger">
    Cut the v1 list until it has one player loop, one economy route, one staff route, and one rollback plan. Anything that does not support those four goes to later.
  </Callout>
</Callout>

<Callout variant="info" title="Staff disagree about the server direction">
  <Callout variant="danger">
    Use the first-hour loop as the tie-breaker. The system that improves the first hour wins. The system that only sounds exciting waits.
  </Callout>
</Callout>

<Callout variant="info" title="A bought script does not fit the framework">
  <Callout variant="danger">
    Do not force it into production. Put it in `later`, ask the seller about framework support, and keep the server on the planned stack.
  </Callout>
</Callout>

## Pass condition

You are done when `server-brief.md` names the audience, first-hour loop, framework direction, must-have v1 list, later list, budget cap, dependency risks, and pass condition. To confirm, the completed `server-brief.md` is open in VS Code with every section filled.

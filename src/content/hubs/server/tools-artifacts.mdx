Install first

Install the editor, archive tool, file transfer app, database, database viewer, and planning board. Then download and pin an FXServer artifact instead of trusting an old video link.

## Vocabulary

| Term | Description |
| --- | --- |
| Artifact | The FXServer build files that run the server. It is the server engine, not your scripts. |
| Pin | A recorded artifact build number that you can reinstall if a newer build breaks your server. |
| Rollback | A known-good copy or version you can return to after a bad install or update. |
| SFTP | A secure way to move files between your PC and a hosted server. |

<StepGroup>
  <Step num={1} title="Create the owner toolkit folder" outcome="every proof file has a home.">
      1. Open File Explorer.
      2. Create `Documents\FiveM-Owner\`.
      3. Inside it create `tools`, `artifacts`, `backups`, `planning`, and `proof`.
      4. Create `tools-checklist.md` in the root folder.

      To confirm, File Explorer shows the `FiveM-Owner` folders.
  </Step>

  <Step num={2} title="Install VS Code and prove folder editing works" outcome="you can safely edit server.cfg, Lua, and Markdown files.">
      1. Download VS Code from [code.visualstudio.com/download](https://code.visualstudio.com/download).
      2. Windows: run the User Installer. Tick both **Open with Code** checkboxes.
      3. macOS for planning only: install the Universal build, then drag Visual Studio Code into Applications.
      4. Open `Documents\FiveM-Owner` in VS Code.
      5. Edit `tools-checklist.md` and add `VS Code: installed`.

      Right-clicking a folder shows **Open with Code** on Windows, and VS Code displays `tools-checklist.md`.
  </Step>

  <Step num={3} title="Install an archive tool and show file extensions" outcome="you can extract .7z, .zip, and .rar files and see real names.">
      1. Windows option A: install WinRAR from [win-rar.com](https://www.win-rar.com/).
      2. Windows option B: install 7-Zip from [7-zip.org](https://www.7-zip.org/).
      3. Open File Explorer → **View** → **Show** → turn on **File name extensions**.
      4. Also turn on **Hidden items**.
      5. Create `proof\extension-test.cfg`.

      File Explorer shows `extension-test.cfg`, not `extension-test.cfg.txt`.

      <Callout variant="gotcha" title="Recovery">
        If you see `.txt`, rename the file after extensions are visible. FiveM will not read `server.cfg.txt`.
      </Callout>
  </Step>

  <Step num={4} title="Install WinSCP and prove you can open a login profile" outcome="you are ready for hosted file transfers, even if you are not hosted yet.">
      1. Download WinSCP from [winscp.net](https://winscp.net/eng/download.php).
      2. Install with the default Commander interface.
      3. Open WinSCP.
      4. Create a saved site named `FiveM hosting`.
      5. Leave host blank if you do not have hosting yet. Save the profile name only.

      WinSCP opens to the login window and shows your saved site in the list.
  </Step>

  <Step num={5} title="Install MariaDB and prove the service runs" outcome="MariaDB listens on port 3306.">
      1. Download MariaDB Server from [mariadb.org/download](https://mariadb.org/download/).
      2. Windows: choose the 64-bit MSI. Keep port `3306`. Install as a Windows service.
      3. Linux VPS: use the host panel database page when available, or install through the host's documented package manager path.
      4. Save the root or database password in a password manager, not in Discord.
      5. Windows proof: open PowerShell and run `Get-Service MariaDB*`.

      ```
      Status   Name               DisplayName
      ------   ----               -----------
      Running  MariaDB            MariaDB
      ```

      <Callout variant="danger" title="Recovery">
        If the service is stopped, open Windows Services, find MariaDB, click **Start**, then rerun the check. If port `3306` is already used, another MySQL/MariaDB install is running.
      </Callout>
  </Step>

  <Step num={6} title="Install HeidiSQL and prove a database connection" outcome="HeidiSQL can see a database named fivem.">
      1. Download HeidiSQL from [heidisql.com/download.php](https://www.heidisql.com/download.php).
      2. Create a session named `local FiveM`.
      3. Use host `127.0.0.1`, user `root`, your password, and port `3306`.
      4. Click **Open**.
      5. Right-click the connection → **Create new** → **Database**.
      6. Name it `fivem` and use `utf8mb4_unicode_ci` if asked.

      The left tree shows `fivem`. To confirm, HeidiSQL is connected with the `fivem` database visible.
  </Step>

  <Step num={7} title="Create and protect your Cfx.re key" outcome="your license key is ready and private.">
      1. Sign in at [portal.cfx.re](https://portal.cfx.re/).
      2. Open **Server Management**.
      3. Create a new server license key for your server.
      4. Save it in your password manager or a private local file named `license-key-private.txt`.
      5. Write `Cfx.re key created: yes` in `tools-checklist.md`. Do not paste the key itself.

      <Callout variant="warn" title="Secret handling">
        Blur the license key in screenshots. If it leaks, revoke it in the Cfx.re portal and create a new one.
      </Callout>
  </Step>

  <Step num={8} title="Download, pin, and archive an FXServer artifact" outcome="you have one current build and one recorded rollback build.">
      1. Open the official artifact page for your operating system.
      2. Windows: [Windows artifacts](https://runtime.fivem.net/artifacts/fivem/build_server_windows/master/).
      3. Linux hosted/VPS: [Linux artifacts](https://runtime.fivem.net/artifacts/fivem/build_proot_linux/master/).
      4. Choose a build, download `server.7z`, and copy the build number from the URL or page label.
      5. Save it as `artifacts\server-BUILDNUMBER.7z`.
      6. Write the build number, date, OS, and source URL in `artifact-pin.md`.

      ```
      # FXServer artifact pin
      OS: Windows / Linux
      Build number:
      Downloaded date:
      Source URL:
      Why this build:
      Known-good test:
      Rollback build:
      Rollback file:
      ```

      <Callout variant="tip" title="Latest is not a rollback plan">
        For a fresh install, a current official build is fine. For an existing server, record the working build first. Update only after you can return to the previous build.
      </Callout>
  </Step>
</StepGroup>

## Artifact rollback procedure

Use this when a new artifact causes startup problems after an update.

1. Stop the server in txAdmin or the hosting panel.
2. Rename the current artifact folder to `server_bad_BUILDNUMBER`.
3. Extract the previous known-good `server-BUILDNUMBER.7z` into the active server folder.
4. Keep `server-data` and `resources` unchanged unless your host stores artifacts and data together.
5. Start the server.
6. Run `status` in txAdmin and connect once.
7. Record the result in `artifact-pin.md`.

```
Rollback date:
Bad build:
Restored build:
Reason:
Server started: yes / no
Direct connect worked: yes / no
Console errors after rollback:
Decision:
```

## Pass condition

You are done when `tools-checklist.md` has a proof line for each tool, HeidiSQL connects to `fivem`, MariaDB service is running, your Cfx.re key is private, and `artifact-pin.md` names both the chosen build and rollback procedure.

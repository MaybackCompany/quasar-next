## Outcome

You will read player data on the server and avoid client authority.

<Callout variant="tip" title="Why it matters">
  Money and jobs must be trusted server data. Client values are only requests.
</Callout>

## Vocabulary

| Term | Description |
| --- | --- |
| Player object | Framework data plus helper functions. Example: Player.PlayerData.job.name. |
| Server write | A change made by the server after validation. Example: AddMoney runs server-side. |
| Client request | A message asking the server to act. Example: a client asks to claim a reward. |

## Minimal example

```lua
local MODULE = 'framework'
local QBCore = exports['qb-core']:GetCoreObject()

RegisterNetEvent('quasar:framework:jobCheck', function()
    local src = source
    local Player = QBCore.Functions.GetPlayer(src)
    if not Player then
        print(('[QUIET][%s][missing] source=%s'):format(MODULE, src))
        return
    end

    local job = Player.PlayerData.job.name
    print(('[QUIET][%s][job] source=%s job=%s'):format(MODULE, src, job))
end)
```

<ExpectedOutput>
  Expected output:

  [QUIET][framework][job] prints the server-owned job name.

## Production checklist

1. Confirm the code is on the correct side.
2. Validate every payload before acting.
3. Log with `[QUIET][module][action]`.
4. Add one cleanup or rollback path.
5. Copy the expected output into your test note.

| Task | Note | QBCore | ESX | ox_lib |
| --- | --- | --- | --- | --- |
| Get the player object on the server | Use the FiveM source (player id) to load the framework's player record. Always check it is not nil before reading fields. | local QBCore = exports['qb-core']:GetCoreObject() local Player = QBCore.Functions.GetPlayer(source) if not Player then return end local cid = Player.PlayerData.citizenid | ESX = exports['es_extended']:getSharedObject() local xPlayer = ESX.GetPlayerFromId(source) if not xPlayer then return end local id = xPlayer.identifier | -- ox_core (optional). With QBCore/ESX, -- use the framework's player object. local player = exports.ox_core:GetPlayer(source) |
| Get the player object on the client | Client-side player data mirrors a subset of server data. Treat it as read-only. | local QBCore = exports['qb-core']:GetCoreObject() local PlayerData = QBCore.Functions.GetPlayerData() local cid = PlayerData.citizenid | ESX = exports['es_extended']:getSharedObject() local xPlayer = ESX.GetPlayerData() local job = xPlayer.job and xPlayer.job.name | n/a. Use the underlying framework or ox_core's client API. |
| Stable player identifier | The id you save in the database. Different frameworks use different shapes, so saved data does not auto-migrate between them. | -- citizenid: 8-char string, e.g. "ABCD1234" local cid = Player.PlayerData.citizenid | -- identifier: license, steam, or fivem hex local id = xPlayer.identifier | -- ox_core uses charId (number). -- For raw FiveM, GetPlayerIdentifierByType local lic = GetPlayerIdentifierByType(source, 'license') |
| Add money to a player | Always run money changes on the server. Never trust the client to ask for money. | Player.Functions.AddMoney('cash', 250, 'reward-quest-1') | xPlayer.addMoney(250) -- accounts: xPlayer.addAccountMoney('bank', 250) | n/a. Money is a framework concern. Use QBCore/ESX/ox_core money API. |
| Remove money from a player | Check the balance first. The framework will not stop you spending below zero in older forks. | if Player.PlayerData.money.cash >= 100 then Player.Functions.RemoveMoney('cash', 100, 'shop-buy') end | if xPlayer.getMoney() >= 100 then xPlayer.removeMoney(100) end | n/a. Use the framework money API. |
| Read a player's job | Both frameworks expose name and grade. Names differ slightly, so always read with a guard. | local job = Player.PlayerData.job local name = job and job.name -- "police" local grade = job and job.grade.level -- 0..N | local job = xPlayer.job local name = job and job.name -- "police" local grade = job and job.grade -- number | n/a. Read from QBCore/ESX/ox_core. |
| Set a player's job | Server-side. Make sure the job exists in shared/jobs.lua before calling this. | Player.Functions.SetJob('police', 0) | xPlayer.setJob('police', 0) | n/a. ox_core has its own group/job system. |
| Show a quick notification | Each framework ships a default notify. ox_lib's is the most flexible. Pick one and use it everywhere for visual consistency. |  |  |  |
| Notify from the server to a client | From the server, send a TriggerClientEvent to one source. Do not loop the whole player list unless you mean to broadcast. | TriggerClientEvent('QBCore:Notify', source, 'Saved.', 'success') | TriggerClientEvent('esx:showNotification', source, 'Saved.') | TriggerClientEvent('ox_lib:notify', source, \{ title = 'Saved', type = 'success' \}) |
| Open a simple input dialog | For text/number entry. ox_lib's input is the cleanest cross-framework choice. |  |  | local input = lib.inputDialog('Name plate', \{ \{ type = 'input', label = 'Plate', required = true \}, \}) if not input then return end local plate = input[1] |
| Show a context menu | For action lists like a phone menu, garage menu, or admin tool. |  |  | lib.registerContext(\{ id = 'garage_menu', title = 'Garage', options = \{ \{ title = 'Cars', event = 'garage:list' \} \} \}) lib.showContext('garage_menu') |
| Add an item to a player | Server-side. The item must exist in the inventory's items table. ox_inventory is recommended for new servers. |  |  |  |
| Check if a player has an item | Always check on the server. Clients can lie about their inventory. | local item = Player.Functions.GetItemByName('bread') if item and item.amount > 0 then -- has bread end | local item = xPlayer.getInventoryItem('bread') if item and item.count > 0 then -- has bread end |  |
| Use-item callback (server) | Run code when a player uses an item from their inventory. Each system registers callbacks differently. | QBCore.Functions.CreateUseableItem('bread', function(source, item) -- consume + apply effect end) | ESX.RegisterUsableItem('bread', function(source) local xPlayer = ESX.GetPlayerFromId(source) xPlayer.removeInventoryItem('bread', 1) -- apply effect end) |  |
| Add an interaction zone on an entity | Each target system has slightly different option shapes. ox_target is the cross-framework standard. |  | n/a. ESX does not ship a target system. Most servers run ox_target or qb-target. |  |
| Server callback (client asks server, gets answer) | For things like "do I have permission" or "give me my saved data". Always validate the request on the server. | -- server QBCore.Functions.CreateCallback('shop:can-buy', function(source, cb, item) cb(Player.PlayerData.money.cash >= Items[item].price) end) -- client QBCore.Functions.TriggerCallback('shop:can-buy', function(ok) if ok then ... end, 'bread') | -- server ESX.RegisterServerCallback('shop:can-buy', function(source, cb, item) local xPlayer = ESX.GetPlayerFromId(source) cb(xPlayer.getMoney() >= Items[item].price) end) -- client ESX.TriggerServerCallback('shop:can-buy', function(ok) ... end, 'bread') | -- server lib.callback.register('shop:can-buy', function(source, item) return hasMoney(source, Items[item].price) end) -- client (await) local ok = lib.callback.await('shop:can-buy', false, 'bread') |
| Progress bar / circle while doing an action | Lock the player into a short action so they cannot move or shoot during it. ox_lib's progressCircle is the most polished. |  | n/a. ESX has no built-in progress bar. Use ox_lib.progressCircle or progress addon. | if lib.progressCircle(\{ duration = 5000, label = 'Lockpicking...', position = 'bottom', disable = \{ move = true, combat = true \}, \}) then -- success end |
| Listen for player loaded (after spawn) | Run setup once after the player is fully in the world. Use this for spawning gear, restoring state, or sending welcome data. | RegisterNetEvent('QBCore:Client:OnPlayerLoaded', function() -- ready end) | RegisterNetEvent('esx:playerLoaded') AddEventHandler('esx:playerLoaded', function(xPlayer) -- ready end) |  |

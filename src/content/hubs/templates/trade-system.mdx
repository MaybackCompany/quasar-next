>
  A drop-in trade resource: one player runs a command, the other sees an NUI menu, and the server moves the item between their inventories. The item-handling functions are framework-agnostic, so the same code runs on ESX, QB, ox_inventory, or qs-inventory by flipping a config value.
</LessonHero>

<Callout variant="fivem" title="How the flow works">
  Player A runs `/tradeRequest`. The server tells Player B's client to open the menu, pre-filled with A's player ID, the item, and the count. When B clicks the button, the NUI posts back to the client, the client asks the server to confirm, and the server checks B's inventory, removes the item from B, and gives it to A. The whole transfer happens server-side, which is the only place item amounts can be trusted.
</Callout>

<Callout variant="danger" title="Read this before you ship it">
  This template is a teaching base, not a hardened release. It does not check distance between the two players, does not stop a player from trading with themselves, and trusts the item name and count that arrive in the server event. Before production, validate that both players exist and are near each other, clamp the count to a positive integer server-side, and never let the client decide which inventory function runs. Treat every value that crosses the net event boundary as hostile.
</Callout>

## Files

- **fxmanifest.lua**, declares the load order: shared config and locales, then client and server, then the NUI page.
- **config/config.lua**, picks the language, the inventory system, and the framework.
- **server/server.lua**, the command, the confirm event, and the framework-agnostic item functions.
- **client/client.lua**, receives the request, opens the NUI, relays the accept callback, and shows notifications.
- **html/index.html**, **css/index.css**, **js/index.js**, the trade menu interface.
- **locales/en.lua**, **locales/es.lua**, the translated message strings.

## fxmanifest.lua

The manifest sets the load order. Config and locales are shared so both sides can read them, the inventory logic stays server-only, and the three NUI files are registered under both `ui_page` and `files` so the game can serve them.

## config/config.lua

Three switches drive the whole resource. `Config.Locale` picks the language table, `Config.Inventory` selects which inventory exports run, and `Config.Framework` tells the server which core object to load. Set these once to match your server.

```
Config = Config or {}
Locales = Locales or {}

Config.Locale = 'en'

Config.Inventory = 'qs' -- 'esx', 'ox', 'qs', 'qb'

Config.Framework = 'esx' -- 'esx', 'qb'
```

## server/server.lua

This is the heart of the resource. It registers the trade command, listens for the confirm event, and exposes three item functions that branch on `Config.Inventory`. Because the transfer runs entirely here, the client can never fake a successful trade.

### 2.1 The trade command

It reads three arguments, rejects anything missing or a non-positive count, then tells the target player's client to open the menu and confirms back to the sender.

```
Core = {}
PlayerTrades = {}

RegisterCommand('tradeRequest', function(source, args)
    local targetId = tonumber(args[1])
    local itemName = args[2]
    local itemCount = tonumber(args[3])

    if not targetId or not itemName or not itemCount or itemCount <= 0 then
        TriggerClientEvent('trade:client:notify', source, _U('usage_trade_request'))
        return
    end

    TriggerClientEvent('trade:client:receiveRequest', targetId, source, itemName, itemCount)
    TriggerClientEvent('trade:client:notify', source, string.format(_U('trade_request_sent'), targetId))
end)
```

### 2.2 Confirming the trade

When the accepting player confirms, the server checks their inventory, and only if the item is there does it remove from one side and give to the other. Both players get a notification about the outcome.

```
RegisterNetEvent('trade:server:confirmTrade')
AddEventHandler('trade:server:confirmTrade', function(sourcePlayer, itemName, itemCount)
    local source = source
    local targetPlayer = sourcePlayer

    if HasItem(targetPlayer, itemName, itemCount) then
        RemoveItem(targetPlayer, itemName, itemCount)
        GiveItem(source, itemName, itemCount)
        TriggerClientEvent('trade:client:notify', targetPlayer, _U('trade_success'))
        TriggerClientEvent('trade:client:notify', source, _U('trade_received'))
    else
        TriggerClientEvent('trade:client:notify', source, _U('trade_failed'))
    end
end)
```

### 2.3 Loading the framework

Based on the config value, the server grabs the right shared object so the ESX and QB inventory branches have something to call.

```
if Config.Framework == 'esx' then
    ESX = exports['es_extended']:getSharedObject()
elseif Config.Framework == 'qb' then
    QBCore = exports['qb-core']:GetCoreObject()
end
```

### 2.4 The item functions

These three functions are the part you reuse everywhere. Each one branches on `Config.Inventory` and calls the matching exports for qs-inventory, ox_inventory, ESX, or QB. Swap inventories without touching the trade logic above.

```
function HasItem(playerId, itemName, itemCount)
    local playerId = tonumber(playerId) -- Convert playerId
    local itemCount = tonumber(itemCount) -- Convert itemCount
    if not playerId or not itemName or not itemCount or itemCount <= 0 then
        return false
    end

    if Config.Inventory == 'qs' then
        local totalAmount = exports['qs-inventory']:GetItemTotalAmount(playerId, itemName)
        return totalAmount >= itemCount
    elseif Config.Inventory == 'ox' then
        local itemCountOX = exports.ox_inventory:GetItemCount(playerId, itemName)
        return itemCountOX >= itemCount
    elseif Config.Inventory == 'esx' then
        local player = ESX.GetPlayerFromId(playerId)
        local item = player.getInventoryItem(itemName)
        return item and item.count >= itemCount
    elseif Config.Inventory == 'qb' then
        local player = QBCore.Functions.GetPlayer(playerId)
        local item = player.Functions.GetItemByName(itemName)
        return item and item.amount >= itemCount
    else
        return false
    end
end

function RemoveItem(playerId, itemName, itemCount)
    if Config.Inventory == 'qs' then
        exports['qs-inventory']:RemoveItem(playerId, itemName, itemCount)
    elseif Config.Inventory == 'ox' then
        exports.ox_inventory:RemoveItem(playerId, itemName, itemCount)
    elseif Config.Inventory == 'esx' then
        local player = ESX.GetPlayerFromId(playerId)
        player.removeInventoryItem(itemName, itemCount)
    elseif Config.Inventory == 'qb' then
        local player = QBCore.Functions.GetPlayer(playerId)
        player.Functions.RemoveItem(itemName, itemCount)
    end
end

function GiveItem(playerId, itemName, itemCount)
    if Config.Inventory == 'qs' then
        exports['qs-inventory']:AddItem(playerId, itemName, itemCount)
    elseif Config.Inventory == 'ox' then
        exports.ox_inventory:AddItem(playerId, itemName, itemCount)
    elseif Config.Inventory == 'esx' then
        local player = ESX.GetPlayerFromId(playerId)
        player.addInventoryItem(itemName, itemCount)
    elseif Config.Inventory == 'qb' then
        local player = QBCore.Functions.GetPlayer(playerId)
        player.Functions.AddItem(itemName, itemCount)
    end
end
```

### 2.5 The locale helper

A tiny helper that looks up a key in the active language table and falls back to the raw key if no translation exists, so a missing string never crashes the resource.

```
function _U(str)
    if Locales and Locales[Config.Locale] and Locales[Config.Locale][str] then
        return Locales[Config.Locale][str]
    else
        return str
    end
end
```

<Callout variant="warn" title="The ESX and QB branches assume a valid player">
  In the ESX and QB cases, `HasItem` calls methods on `player` directly. If the target ID is offline or invalid, that lookup returns nil and the call errors. Add a nil check on `player` before reading the inventory item if your server lets stale IDs reach this code.
</Callout>

## client/client.lua

The client is a thin relay. It receives the request, opens the NUI with focus, forwards the accept callback to the server, and draws notifications using the native game UI. It holds no trade logic of its own, which is exactly what you want.

```
RegisterNetEvent('trade:client:receiveRequest')
AddEventHandler('trade:client:receiveRequest', function(requesterId, itemName, itemCount)
    TriggerEvent('trade:client:showMenu', requesterId, itemName, itemCount)
end)

RegisterNetEvent('trade:client:showMenu')
AddEventHandler('trade:client:showMenu', function(requesterId, itemName, itemCount)
    SetNuiFocus(true, true)
    SendNUIMessage({
        action = 'showMenu',
        requesterId = requesterId,
        itemName = itemName,
        itemCount = itemCount
    })
end)

RegisterNetEvent('trade:client:notify')
AddEventHandler('trade:client:notify', function(message)
    SetNotificationTextEntry("STRING")
    AddTextComponentString(message)
    DrawNotification(false, true)
end)

RegisterNUICallback('acceptRequest', function(data, cb)
    TriggerServerEvent('trade:server:confirmTrade', data.playerId, data.itemName, data.itemCount)
    cb({ success = true })
    SetNuiFocus(false, false)
end)
```

<Callout variant="fivem" title="Always release NUI focus">
  The callback calls `SetNuiFocus(false, false)` after sending the event. If you add early returns or error paths, make sure focus is released on every one of them, otherwise the player gets stuck with a frozen cursor and no way to close the menu.
</Callout>

## html/js/index.js

The script listens for the `showMenu` message from the client, fills the inputs, and disables them so the values cannot be edited by hand. On button click it posts the data to the `acceptRequest` callback and hides the menu when the server replies. The inline comments are Spanish in the source, preserved here.

```
window.addEventListener('message', (event) => {
    const data = event.data;
    if (typeof data !== 'object' || !data.action) return;

    switch (data.action) {
        case 'showMenu':
            ShowRequestEventMenu(data.requesterId, data.itemName, data.itemCount);
            break;
    }
});

function ShowRequestEventMenu(requesterId, itemName, itemCount) {
    const menu = document.getElementById('RequestTradeMenu');
    menu.style.display = 'block';

    const playerIdInput = document.getElementById('RequestTradeMenuPlayerId');
    const itemNameInput = document.getElementById('RequestTradeMenuItemName');
    const itemCountInput = document.getElementById('RequestTradeMenuItemCount');

    // Rellenar los campos con los datos recibidos
    playerIdInput.value = requesterId;
    itemNameInput.value = itemName;
    itemCountInput.value = itemCount;

    // Deshabilitar los campos para evitar modificaciones accidentales
    playerIdInput.disabled = true;
    itemNameInput.disabled = true;
    itemCountInput.disabled = true;
}

document.addEventListener('DOMContentLoaded', () => {
    document.getElementById('AcceptTradeButton').addEventListener('click', () => {
        const playerId = document.getElementById('RequestTradeMenuPlayerId').value;
        const itemName = document.getElementById('RequestTradeMenuItemName').value;
        const itemCount = document.getElementById('RequestTradeMenuItemCount').value;

        // Enviar la solicitud al servidor para aceptar la solicitud de intercambio
        fetch(`http://${GetParentResourceName()}/acceptRequest`, {
            method: 'POST',
            headers: {
                'Content-Type': 'application/json'
            },
            body: JSON.stringify({ playerId, itemName, itemCount })
    })
        .then(response => {
            if (!response.ok) {
                throw new Error('Network response was not ok');
            }
            return response.json();
        })
        .then(data => {
            if (data.success) {
                document.getElementById('RequestTradeMenu').style.display = 'none';
            } else {
                console.error('Trade request failed:', data.message);
            }
        })

    });
});
```

<Callout variant="tip" title="Disabled inputs are a nice touch">
  After filling the fields, the script disables them so the accepting player cannot quietly change the player ID, item, or count before confirming. It is a small guard, but it keeps the menu honest. The real enforcement still happens server-side in `HasItem`.
</Callout>

## Install

1. Create `resources/[local]/trade-system/` and copy all nine files into it, keeping the folder layout (`client/`, `server/`, `config/`, `locales/`, `html/`).
2. Open `config/config.lua` and set `Config.Framework` and `Config.Inventory` to match your server. Set `Config.Locale` to `'en'` or `'es'`.
3. Add `ensure trade-system` to `server.cfg` after your framework and your inventory resource.
4. Restart the server.

## Test it

1. Get two players (or two characters) on the server and note their IDs.
2. As player A, run `/tradeRequest [B's id] [itemName] [count]`, for example `/tradeRequest 2 water 1`.
3. Player B sees the menu pre-filled. Click the button to confirm.
4. Check both inventories: the item should move from B to A, and both players should get a notification.

## Make it yours

The fastest wins are in three spots. First, set the three values in `config/config.lua` to your stack. Second, fix the button id mismatch between the HTML and the CSS if you want the styled button. Third, before going live, harden the server: confirm both players exist and are close, block self-trades, and clamp the count to a positive whole number inside `trade:server:confirmTrade` so the menu cannot send junk. Once that is in place, this is a solid base for a two-way trade where both players have to approve.
